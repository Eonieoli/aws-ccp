## 10.1 긴밀한 결합 vs 느슨한 결합 — 모놀리식 vs 마이크로서비스, 디커플링의 가치

> 💡 **한 줄 요약**: 시스템 구성 요소 간의 직접적 의존을 줄이는 "느슨한 결합"은 확장성, 내결함성, 유지보수성을 획기적으로 높이는 아키텍처 설계 원칙이다.

### 직관적 이해

회사에서 점심 주문을 하는 상황을 떠올려 봅시다.

**방법 1 — 긴밀한 결합 방식**: 팀원 A가 식당에 직접 전화해서 주문합니다. 그런데 식당 전화가 통화 중이면? A는 전화가 연결될 때까지 기다려야 합니다. 식당이 문을 닫았다면? A는 주문 자체를 할 수 없습니다. A의 업무가 식당의 상태에 완전히 종속되어 있습니다.

**방법 2 — 느슨한 결합 방식**: 팀원 A가 배달 앱에 주문을 넣습니다. 식당이 지금 바쁘더라도 주문은 앱에 저장되어 있으므로, A는 바로 자기 업무로 돌아갈 수 있습니다. 식당이 잠시 쉬는 시간이더라도 주문이 사라지지 않습니다. 나중에 식당이 준비되면 주문을 처리합니다.

이것이 바로 **긴밀한 결합(Tight Coupling)**과 **느슨한 결합(Loose Coupling)**의 차이입니다. 배달 앱이라는 "중간 매개체"가 주문하는 사람과 식당 사이를 분리해 주는 것처럼, 클라우드 아키텍처에서도 서비스 사이에 중간 매개체를 두어 각 구성 요소가 독립적으로 작동할 수 있게 합니다.

**왜 이것이 필요한가?** 현대의 애플리케이션은 하나의 거대한 프로그램이 아니라, 수십~수백 개의 작은 서비스가 협력하여 동작합니다. 이 서비스들이 서로 직접 연결되어 있으면, 하나가 멈추는 순간 연쇄적으로 전체 시스템이 마비될 수 있습니다. 느슨한 결합은 이 위험을 제거합니다.

### 핵심 개념 설명

> 📝 **긴밀한 결합(Tight Coupling)**: 시스템의 구성 요소들이 서로 직접 통신하며, 한 구성 요소의 변경이나 장애가 다른 구성 요소에 즉시 영향을 미치는 구조.

> 📝 **느슨한 결합(Loose Coupling)**: 시스템의 구성 요소들이 중간 매개체(메시지 큐, 이벤트 버스 등)를 통해 간접적으로 통신하며, 각 구성 요소가 독립적으로 작동·확장·장애 복구할 수 있는 구조.

> 📝 **디커플링(Decoupling)**: 긴밀하게 결합된 구성 요소들을 분리하여 느슨한 결합으로 전환하는 과정.

#### 긴밀한 결합의 문제점

> 🔗 **연결**: Chapter 5에서 배운 EC2 인스턴스와 Lambda 함수를 떠올려 봅시다. 이들이 서로 직접 호출하는 구조를 생각해 보겠습니다.

긴밀하게 결합된 시스템을 그림으로 표현하면 다음과 같습니다:

```
[주문 서비스(EC2)] ──직접 호출──→ [결제 서비스(EC2)] ──직접 호출──→ [배송 서비스(Lambda)]
```

이 구조의 문제점은:

1. **장애 전파**: 결제 서비스가 다운되면, 주문 서비스도 응답을 받지 못해 멈춥니다. 고객은 "주문이 안 돼요"라는 에러를 봅니다.
2. **확장의 어려움**: 주문이 갑자기 10배 늘어나면, 주문 서비스뿐 아니라 결제 서비스, 배송 서비스까지 동시에 확장해야 합니다. 하나라도 느리면 전체가 느려집니다.
3. **배포의 복잡성**: 결제 서비스를 업데이트하려면, 주문 서비스의 호출 방식과 완벽히 맞아야 합니다. 작은 변경도 전체 시스템 테스트가 필요합니다.

#### 느슨한 결합의 해결 방식

중간에 **메시지 큐(Message Queue)**를 배치하면:

```
[주문 서비스] ──메시지 전송──→ [메시지 큐] ──메시지 수신──→ [결제 서비스]
                                    │
                                    └──메시지 수신──→ [배송 서비스]
```

이제 각 서비스는 독립적입니다:

1. **장애 격리**: 결제 서비스가 다운되어도, 주문 서비스는 메시지를 큐에 넣고 정상 응답합니다. 결제 서비스가 복구되면 큐에 쌓인 메시지를 처리합니다.
2. **독립적 확장**: 주문이 폭증해도 메시지가 큐에 쌓이므로, 결제 서비스는 자신의 속도에 맞춰 처리합니다. 필요하면 결제 서비스만 추가로 확장하면 됩니다.
3. **독립적 배포**: 결제 서비스를 업데이트할 때, 메시지 형식만 유지하면 주문 서비스는 변경할 필요가 없습니다.

#### 모놀리식 아키텍처 vs 마이크로서비스 아키텍처

> 📝 **모놀리식 아키텍처(Monolithic Architecture)**: 모든 기능이 하나의 애플리케이션 안에 긴밀하게 통합된 구조. 하나의 코드베이스, 하나의 배포 단위로 구성됨.

> 📝 **마이크로서비스 아키텍처(Microservices Architecture)**: 각 기능을 독립적인 소규모 서비스로 분리하고, 이들이 API나 메시지를 통해 느슨하게 통신하는 구조.

| 특성 | 모놀리식 | 마이크로서비스 |
|------|----------|----------------|
| 구조 | 하나의 큰 애플리케이션 | 여러 개의 작은 서비스 |
| 결합도 | 긴밀한 결합 | 느슨한 결합 |
| 배포 | 전체를 한 번에 배포 | 각 서비스를 독립 배포 |
| 확장 | 전체를 함께 확장 | 필요한 서비스만 확장 |
| 장애 영향 | 하나의 장애가 전체에 영향 | 장애가 해당 서비스에 국한 |
| 적합한 상황 | 소규모 애플리케이션, 초기 개발 | 대규모·복잡한 시스템 |

AWS의 애플리케이션 통합 서비스들(SQS, SNS, EventBridge 등)은 바로 이 **마이크로서비스 간의 느슨한 결합**을 구현하기 위한 도구입니다.

### 상세 예시

> 🔍 **예시 10.1.1: 온라인 쇼핑몰의 주문 처리**
>
> **상황**: 고객이 상품을 주문하면, (1) 주문 접수, (2) 결제 처리, (3) 재고 차감, (4) 배송 요청이 순차적으로 이루어져야 합니다.
>
> **긴밀한 결합 방식**: 주문 서비스가 결제 → 재고 → 배송 서비스를 차례로 직접 호출합니다. 결제 서비스의 응답 시간이 3초로 느려지면, 고객은 3초 이상 기다려야 합니다. 재고 서비스가 다운되면, 주문 자체가 실패합니다.
>
> **느슨한 결합 방식**: 주문 서비스는 "주문 접수됨"이라는 메시지를 큐에 넣고 고객에게 즉시 "주문이 접수되었습니다"를 응답합니다. 결제, 재고, 배송 서비스는 각자의 속도로 큐에서 메시지를 꺼내 처리합니다. 재고 서비스가 일시 정지되어도 메시지는 큐에 보관되므로, 복구 후 처리됩니다.
>
> **핵심 포인트**: 느슨한 결합은 고객 경험(빠른 응답)과 시스템 안정성(장애 격리)을 동시에 개선합니다.

> 🔍 **예시 10.1.2: 일상 속 긴밀한 결합과 느슨한 결합**
>
> **상황**: 친구에게 중요한 소식을 전달해야 합니다.
>
> **긴밀한 결합 = 전화 통화**: 친구가 전화를 받아야만 소식을 전할 수 있습니다. 친구가 회의 중이면? 기다려야 합니다. 친구의 폰이 꺼져 있으면? 전달 자체가 불가능합니다.
>
> **느슨한 결합 = 문자 메시지**: 메시지를 보내면 끝입니다. 친구가 회의 중이든, 폰이 꺼져 있든, 나중에 확인하면 됩니다. 메시지는 통신사 서버에 저장되어 있으므로 사라지지 않습니다.
>
> **핵심 포인트**: 느슨한 결합의 본질은 "보내는 쪽"과 "받는 쪽"의 시간과 상태를 분리하는 것입니다. 이를 **비동기(Asynchronous) 통신**이라고 합니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: "느슨한 결합이 항상 더 좋다"는 것은 아닙니다.
>
> - **단순한 애플리케이션**에서는 모놀리식이 더 효율적일 수 있습니다. 서비스가 3~4개뿐인데 메시지 큐를 도입하면 불필요한 복잡성이 추가됩니다.
> - **실시간 응답이 반드시 필요한 경우**(예: 결제 결과를 즉시 보여줘야 할 때)에는 동기식 직접 호출이 적합할 수 있습니다.
> - CCP 시험에서는 "느슨한 결합"과 "디커플링"이라는 키워드가 나오면 SQS, SNS, EventBridge를 떠올리면 됩니다.

### Section 요약

- **긴밀한 결합**은 구성 요소가 직접 통신하는 구조로, 장애 전파·확장 어려움·배포 복잡성 문제가 있다
- **느슨한 결합**은 중간 매개체(메시지 큐, 이벤트 버스)를 통해 구성 요소를 분리하여, 독립적 확장·장애 격리·독립 배포를 가능하게 한다
- **디커플링**은 긴밀한 결합을 느슨한 결합으로 전환하는 과정이며, AWS의 통합 서비스(SQS, SNS, EventBridge)가 이를 지원한다
- 모놀리식 아키텍처는 소규모에 적합하고, 마이크로서비스 아키텍처는 대규모·복잡한 시스템에 적합하다

## 10.2 Amazon SQS — 메시지 큐, 표준 대기열 vs FIFO 대기열

> 💡 **한 줄 요약**: Amazon SQS는 완전관리형 메시지 큐 서비스로, 애플리케이션 구성 요소 간에 메시지를 안전하게 전달하고 보관하여 느슨한 결합을 구현한다.

### 직관적 이해

맛집 앞의 대기 줄을 떠올려 봅시다.

식당에 빈자리가 없으면 고객은 돌아가야 할까요? 아닙니다. **대기 명부(웨이팅 리스트)**에 이름을 적어두면, 자리가 나는 순서대로 호출됩니다. 고객(메시지를 보내는 쪽)은 대기 명부에 이름을 적고 근처에서 다른 일을 할 수 있고, 식당(메시지를 받는 쪽)은 자신의 처리 속도에 맞춰 고객을 안내합니다.

**Amazon SQS(Simple Queue Service)**는 바로 이 "대기 명부"의 클라우드 버전입니다. 한쪽에서 메시지를 보내면(대기 명부에 이름을 적으면), 다른 쪽에서 준비가 되었을 때 메시지를 꺼내 처리합니다(자리가 나면 호출합니다).

**왜 SQS가 필요한가?** Section 10.1에서 배운 느슨한 결합을 실제로 구현하는 가장 기본적이고 강력한 방법이 바로 메시지 큐입니다. SQS는 AWS에서 가장 오래된 서비스 중 하나로(2004년 베타 출시, S3보다 먼저 나왔습니다!), AWS 아키텍처의 근간이 되는 핵심 서비스입니다.

### 핵심 개념 설명

> 📝 **메시지 큐(Message Queue)**: 메시지를 임시로 저장하는 버퍼 역할의 서비스. 보내는 쪽(Producer)과 받는 쪽(Consumer)이 동시에 활성 상태가 아니어도 메시지가 유실되지 않도록 보장한다.

> 📝 **Producer(생산자)**: 메시지를 큐에 보내는 애플리케이션 또는 서비스.

> 📝 **Consumer(소비자)**: 큐에서 메시지를 꺼내 처리하는 애플리케이션 또는 서비스.

#### SQS의 동작 흐름

```
[Producer] ──메시지 전송──→ [SQS 대기열] ──메시지 수신──→ [Consumer]
   (보내기)                    (보관)                      (처리 후 삭제)
```

1. **Producer**가 SQS 대기열에 메시지를 보냅니다
2. 메시지는 SQS 대기열에 **안전하게 저장**됩니다 (여러 서버에 복제)
3. **Consumer**가 대기열에서 메시지를 가져옵니다 (pull 방식)
4. Consumer가 메시지를 성공적으로 처리한 후, 대기열에서 해당 메시지를 **삭제**합니다

> 💬 **참고**: SQS는 "pull(당기기)" 방식입니다. Consumer가 능동적으로 메시지를 가져갑니다. 이는 Section 10.3에서 배울 SNS의 "push(밀어내기)" 방식과 대비됩니다.

#### SQS의 핵심 특성

| 특성 | 설명 |
|------|------|
| **완전관리형** | 서버 관리, 패치, 확장을 AWS가 모두 처리 |
| **무제한 처리량** | 초당 보낼 수 있는 메시지 수에 제한 없음 (표준 대기열) |
| **메시지 보존** | 기본 4일, 최대 14일간 메시지를 보관 |
| **메시지 크기** | 최대 256KB (더 큰 데이터는 S3에 저장하고 참조만 전달) |
| **자동 확장** | 트래픽에 따라 자동으로 처리 용량이 확장됨 |

#### 표준 대기열 vs FIFO 대기열

SQS는 두 가지 유형의 대기열을 제공합니다:

> 📝 **표준 대기열(Standard Queue)**: 최대 처리량을 제공하며, 메시지가 "최소 1회" 전달되고, 순서가 "최선 노력(best-effort)"으로 유지되는 대기열.

> 📝 **FIFO 대기열(First-In-First-Out Queue)**: 메시지가 보낸 순서대로 정확히 1회만 전달되는 것을 보장하는 대기열. 대기열 이름이 반드시 `.fifo`로 끝나야 한다.

| 특성 | 표준 대기열 | FIFO 대기열 |
|------|------------|-------------|
| **처리량** | 무제한 (초당 수만~수십만 건) | 초당 300건 (배치 시 3,000건) |
| **전달 보장** | 최소 1회 (중복 가능) | 정확히 1회 (중복 제거) |
| **순서 보장** | 최선 노력 (순서 바뀔 수 있음) | 엄격한 선입선출 순서 |
| **적합한 상황** | 높은 처리량이 필요하고 순서/중복이 크게 상관없을 때 | 순서가 중요하고 중복 처리가 문제될 때 |
| **예시** | 이미지 처리, 로그 수집 | 은행 거래, 주문 처리 |

#### 가시성 타임아웃(Visibility Timeout)

Consumer가 메시지를 가져간 후 처리하는 동안, 다른 Consumer가 같은 메시지를 가져가면 중복 처리가 발생합니다. 이를 방지하기 위해 SQS는 **가시성 타임아웃**이라는 메커니즘을 사용합니다.

```
1. Consumer A가 메시지를 가져감
2. 해당 메시지는 "보이지 않는 상태"가 됨 (가시성 타임아웃 시작)
3. Consumer A가 처리를 완료하고 메시지를 삭제함 → 정상 완료
   또는
3. 가시성 타임아웃이 만료되면 메시지가 다시 "보이는 상태"가 됨 → 다른 Consumer가 재처리
```

기본 가시성 타임아웃은 30초이며, 0초에서 12시간까지 설정할 수 있습니다.

#### 배달 못한 편지 대기열(Dead-Letter Queue, DLQ)

> 📝 **배달 못한 편지 대기열(Dead-Letter Queue)**: 정해진 횟수만큼 처리에 실패한 메시지가 이동하는 별도의 대기열. 실패한 메시지를 격리하여 분석·디버깅할 수 있게 해준다.

메시지가 계속 처리에 실패하면 무한히 재시도하는 것은 비효율적입니다. DLQ에 이동된 메시지를 분석하면 어떤 유형의 메시지가 왜 실패하는지 파악할 수 있습니다.

### 상세 예시

> 🔍 **예시 10.2.1: 이미지 썸네일 생성**
>
> **상황**: 사용자가 웹사이트에 사진을 업로드하면, 다양한 크기의 썸네일(소형, 중형, 대형)을 자동으로 생성해야 합니다.
>
> **풀이/설명**:
> 1. 웹 서버(EC2)가 업로드된 이미지 정보를 SQS **표준 대기열**에 메시지로 보냅니다
> 2. 썸네일 생성 서비스(Lambda)가 대기열에서 메시지를 가져와 썸네일을 생성합니다
> 3. 업로드가 갑자기 100배 늘어나도, 메시지는 큐에 안전하게 쌓이고 Lambda가 점진적으로 처리합니다
> 4. 표준 대기열을 사용한 이유: 썸네일 생성 순서는 중요하지 않고, 높은 처리량이 필요하기 때문
>
> **핵심 포인트**: SQS는 트래픽 급증을 흡수하는 "버퍼" 역할을 합니다. Producer와 Consumer의 처리 속도가 달라도 문제없습니다.

> 🔍 **예시 10.2.2: 은행 이체 처리**
>
> **상황**: 은행 계좌 간 이체 요청이 접수되면, 반드시 요청 순서대로 처리되어야 합니다. 같은 이체가 두 번 실행되면 안 됩니다.
>
> **풀이/설명**:
> 1. 이체 요청 서비스가 이체 정보를 SQS **FIFO 대기열**에 메시지로 보냅니다
> 2. 이체 처리 서비스가 순서대로 메시지를 꺼내 처리합니다
> 3. FIFO 대기열이므로 "A→B 10만원 이체" 후 "B→C 5만원 이체"가 반드시 이 순서로 처리됩니다
> 4. 중복 제거 기능으로 네트워크 오류 등으로 같은 메시지가 두 번 전송되어도 한 번만 처리됩니다
>
> **핵심 포인트**: 금융 거래처럼 순서와 정확성이 중요한 경우에는 FIFO 대기열을 사용합니다. 처리량 제한이 있지만 정확성을 보장합니다.

### 미니 실습

> 🧪 **Mini Lab 10.2: SQS 대기열 생성 및 메시지 송수신**
>
> **목표**: SQS 표준 대기열을 생성하고, 메시지를 보내고 받아보며 메시지 큐의 동작을 직접 체험합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (SQS는 월 100만 건 무료)
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `SQS`를 입력하고 **Amazon SQS** 서비스를 클릭합니다.
>
> **Step 2**: **대기열 생성(Create queue)** 버튼을 클릭합니다.
>
> **Step 3**: 대기열 설정을 입력합니다:
> - **유형**: "표준(Standard)"을 선택합니다
> - **이름**: `my-first-queue`를 입력합니다
> - 나머지 설정은 기본값 그대로 둡니다
> - 하단의 **대기열 생성(Create queue)** 버튼을 클릭합니다
>
> **Step 4**: 대기열이 생성되면 대기열 상세 페이지로 이동됩니다. 우측 상단의 **메시지 전송 및 수신(Send and receive messages)** 버튼을 클릭합니다.
>
> **Step 5**: 메시지를 보냅니다:
> - **메시지 본문(Message body)**에 `Hello, SQS! 첫 번째 메시지입니다.`를 입력합니다
> - **메시지 전송(Send message)** 버튼을 클릭합니다
> - "메시지가 전송되었습니다" 확인 메시지가 표시됩니다
> - 같은 방법으로 2~3개의 메시지를 더 보내봅니다
>
> **Step 6**: 메시지를 받습니다:
> - 같은 페이지 아래쪽의 **메시지 수신(Receive messages)** 섹션에서 **메시지 폴링(Poll for messages)** 버튼을 클릭합니다
> - 방금 보낸 메시지가 목록에 표시됩니다
> - 메시지 하나를 클릭하면 본문 내용을 확인할 수 있습니다
>
> **Step 7**: 메시지를 삭제합니다:
> - 수신된 메시지 옆의 체크박스를 선택하고 **삭제(Delete)** 버튼을 클릭합니다
> - 실제 애플리케이션에서는 Consumer가 처리 완료 후 자동으로 삭제합니다
>
> **확인**: 메시지를 보내고, 받고, 삭제하는 전체 흐름을 확인했습니다. 대기열 상세 페이지로 돌아가면 "사용 가능한 메시지(Messages available)" 수가 변하는 것을 볼 수 있습니다.
>
> 🧹 **정리**: 왼쪽 메뉴의 **대기열(Queues)**로 이동하여, `my-first-queue`를 선택하고 **삭제(Delete)** 버튼을 클릭합니다. 확인 입력란에 `delete`를 입력하고 삭제를 완료합니다. (통합 실습에서 새로 생성할 예정입니다.)

### 흔한 실수와 주의사항

> ⚠️ **주의**: SQS에서 자주 혼동되는 사항들
>
> - **메시지는 자동으로 삭제되지 않습니다**: Consumer가 메시지를 가져간 후 명시적으로 삭제해야 합니다. 삭제하지 않으면 가시성 타임아웃 후 다시 대기열에 나타나 중복 처리될 수 있습니다.
> - **SQS는 "pull" 방식입니다**: Consumer가 능동적으로 메시지를 가져가야 합니다. 메시지가 도착했다고 SQS가 알려주지 않습니다. (알림이 필요하면 SNS를 사용합니다.)
> - **표준 대기열에서 메시지 순서를 기대하지 마세요**: "최선 노력 순서"이므로 대부분 순서가 유지되지만, 보장되지는 않습니다. 순서가 중요하면 반드시 FIFO를 사용하세요.
> - **메시지 크기 제한(256KB)**: 대용량 데이터는 S3에 저장하고, SQS 메시지에는 S3 위치만 포함하는 패턴을 사용합니다.

### Section 요약

- **Amazon SQS**는 완전관리형 메시지 큐 서비스로, Producer와 Consumer를 분리하여 느슨한 결합을 구현한다
- **표준 대기열**은 무제한 처리량을 제공하며, 순서 보장 없이 최소 1회 전달한다
- **FIFO 대기열**은 엄격한 순서와 정확히 1회 전달을 보장하지만, 처리량에 제한이 있다
- 가시성 타임아웃과 Dead-Letter Queue는 안정적인 메시지 처리를 위한 핵심 메커니즘이다
- SQS는 트래픽 급증을 흡수하는 버퍼 역할을 하여, 시스템의 내결함성과 확장성을 높인다

## 10.3 Amazon SNS — 알림 서비스, 토픽 기반 Pub/Sub 메시징

> 💡 **한 줄 요약**: Amazon SNS는 하나의 메시지를 여러 구독자에게 동시에 전달하는 Pub/Sub(발행/구독) 방식의 완전관리형 알림 서비스이다.

### 직관적 이해

학교의 방송 시스템을 떠올려 봅시다.

교장 선생님이 전교생에게 공지사항을 전달할 때, 모든 학생을 한 명씩 찾아가서 말하지 않습니다. **교내 방송**을 통해 한 번 말하면, 모든 교실의 스피커에서 동시에 들립니다. 특정 학년만 해당하는 공지라면? "3학년 학생들"이라는 **채널(토픽)**을 통해 전달합니다.

**Amazon SNS(Simple Notification Service)**는 이 교내 방송 시스템의 클라우드 버전입니다. 메시지를 **토픽(Topic)**에 발행하면, 해당 토픽을 **구독(Subscribe)**하고 있는 모든 수신자에게 동시에 전달됩니다.

**SQS와의 핵심 차이**: SQS는 "대기 줄에서 한 명씩 가져가는" 방식(1:1, pull)이고, SNS는 "방송으로 모두에게 전달하는" 방식(1:N, push)입니다.

### 핵심 개념 설명

> 📝 **Pub/Sub(Publish/Subscribe) 패턴**: 메시지 발행자(Publisher)가 특정 주제(Topic)에 메시지를 발행하면, 해당 주제를 구독(Subscribe)하고 있는 모든 수신자에게 메시지가 자동으로 전달되는 메시징 패턴.

> 📝 **토픽(Topic)**: SNS에서 메시지의 "채널" 역할을 하는 논리적 접근 포인트. Publisher는 토픽에 메시지를 발행하고, Subscriber는 토픽을 구독한다.

> 📝 **구독(Subscription)**: 특정 토픽의 메시지를 받기 위한 등록. 구독할 때 메시지를 받을 엔드포인트(이메일, SMS, Lambda 함수, SQS 대기열 등)를 지정한다.

#### SNS의 동작 흐름

```
                              ┌──→ [이메일 구독자]
                              │
[Publisher] ──발행──→ [SNS 토픽] ──→ [SMS 구독자]
                              │
                              ├──→ [Lambda 함수]
                              │
                              └──→ [SQS 대기열]
```

1. **Publisher**가 SNS **토픽**에 메시지를 발행(publish)합니다
2. SNS가 해당 토픽의 모든 **구독자**에게 메시지를 **동시에 push**합니다
3. 각 구독자는 자신의 유형(이메일, SMS, HTTP, Lambda, SQS 등)에 맞는 형태로 메시지를 수신합니다

#### SNS가 지원하는 구독 엔드포인트

| 엔드포인트 유형 | 설명 | 사용 예시 |
|------------------|------|-----------|
| **이메일/이메일-JSON** | 이메일로 알림 전송 | 시스템 경고 알림 |
| **SMS** | 문자 메시지로 알림 전송 | 긴급 장애 알림 |
| **HTTP/HTTPS** | 웹 엔드포인트에 POST 요청 | 외부 시스템 연동 |
| **AWS Lambda** | Lambda 함수를 트리거 | 자동 처리 로직 실행 |
| **Amazon SQS** | SQS 대기열에 메시지 전달 | 비동기 처리를 위한 버퍼링 |
| **Amazon Kinesis Data Firehose** | 데이터 스트림에 전달 | 대량 데이터 수집 |
| **플랫폼 애플리케이션** | 모바일 푸시 알림 | 앱 푸시 알림 |

#### SNS + SQS: Fan-out 패턴

SNS와 SQS를 결합하면 매우 강력한 **Fan-out(팬아웃)** 패턴을 구현할 수 있습니다.

> 📝 **Fan-out 패턴**: 하나의 메시지를 여러 SQS 대기열에 동시에 전달하여, 각 대기열의 Consumer가 독립적으로 같은 메시지를 처리하는 패턴.

```
                              ┌──→ [SQS: 주문처리 큐] ──→ [주문 서비스]
                              │
[주문 발생] ──→ [SNS 토픽] ──→├──→ [SQS: 재고관리 큐] ──→ [재고 서비스]
                              │
                              └──→ [SQS: 알림발송 큐] ──→ [알림 서비스]
```

이 패턴이 강력한 이유: SNS의 "동시 전달(push)"과 SQS의 "안전한 보관·독립적 처리(pull)"를 결합하여, 하나의 이벤트를 여러 서비스가 독립적이고 안정적으로 처리할 수 있습니다.

#### SNS 메시지 필터링

모든 구독자가 모든 메시지를 받을 필요는 없습니다. **구독 필터 정책(Subscription Filter Policy)**을 설정하면, 구독자는 자신에게 필요한 메시지만 수신할 수 있습니다.

예를 들어, "주문" 토픽에서:
- 배송팀은 `order_type: "delivery"` 메시지만 받도록 필터링
- 매장팀은 `order_type: "pickup"` 메시지만 받도록 필터링

> ⚠️ **주의**: SQS와 SNS를 혼동하지 마세요!
>
> | 비교 항목 | SQS (메시지 큐) | SNS (알림 서비스) |
> |-----------|-----------------|-------------------|
> | **통신 방식** | Pull (Consumer가 가져감) | Push (구독자에게 전달) |
> | **수신자 수** | 하나의 Consumer가 메시지 처리 | 여러 구독자에게 동시 전달 |
> | **메시지 보존** | 처리 전까지 보관 (최대 14일) | 전달 후 보관하지 않음 |
> | **사용 패턴** | 작업 분배, 비동기 처리 | 알림, 팬아웃 |
>
> - **CCP 시험 팁**: "디커플링", "비동기 처리", "메시지 큐"가 나오면 → **SQS**. "알림", "여러 구독자에게 동시에", "Pub/Sub"이 나오면 → **SNS**.
> - SNS 이메일 구독은 반드시 이메일로 전송되는 **확인 링크를 클릭**해야 활성화됩니다. 확인하지 않으면 메시지가 전달되지 않습니다.
> - SNS는 메시지를 보관하지 않습니다. 구독자가 수신에 실패하면 메시지가 유실될 수 있습니다. 이를 방지하려면 SNS → SQS 패턴(Fan-out)을 사용하세요.

### Section 요약

- **Amazon SNS**는 Pub/Sub 방식의 완전관리형 알림 서비스로, 하나의 메시지를 여러 구독자에게 동시에 push한다
- **토픽(Topic)**은 메시지의 채널이고, **구독(Subscription)**은 토픽의 메시지를 받기 위한 등록이다
- 이메일, SMS, Lambda, SQS 등 다양한 엔드포인트로 메시지를 전달할 수 있다
- **Fan-out 패턴**: SNS + SQS를 결합하면 하나의 메시지를 여러 서비스가 독립적으로 처리할 수 있다
- SNS는 메시지를 보관하지 않으므로, 안정적인 처리가 필요하면 SQS와 함께 사용한다

## 10.4 Amazon EventBridge — 이벤트 기반 아키텍처, 이벤트 버스와 규칙

> 💡 **한 줄 요약**: Amazon EventBridge는 AWS 서비스, 자체 애플리케이션, SaaS 파트너의 이벤트를 감지하고 자동으로 적절한 대상에 라우팅하는 서버리스 이벤트 버스 서비스이다.

### 직관적 이해

공항의 항공편 안내 시스템을 상상해 봅시다.

공항에서는 다양한 이벤트가 끊임없이 발생합니다: 항공편 도착, 출발, 지연, 게이트 변경 등. 이 이벤트들은 공항의 **중앙 시스템(이벤트 버스)**으로 모입니다. 중앙 시스템에는 **규칙**이 설정되어 있습니다:
- "항공편 지연" 이벤트가 발생하면 → 해당 승객에게 SMS 전송, 라운지 할인 쿠폰 발행, 안내 데스크에 알림
- "항공편 도착" 이벤트가 발생하면 → 수하물 시스템 가동, 도착 게이트 안내판 업데이트

**Amazon EventBridge**는 이 공항 중앙 시스템의 클라우드 버전입니다. 다양한 소스에서 이벤트가 발생하면, 사전에 정의된 규칙에 따라 적절한 대상(Target)으로 자동 라우팅합니다.

**SNS와의 차이점**: SNS는 "내가 보내는 메시지를 구독자에게 전달"하는 것이 핵심이라면, EventBridge는 "시스템에서 자동으로 발생하는 이벤트를 감지하고, 규칙에 따라 라우팅"하는 것이 핵심입니다. 특히 **AWS 서비스에서 자동으로 발생하는 이벤트**(예: EC2 인스턴스가 중지됨, S3에 파일이 업로드됨)를 처리하는 데 탁월합니다.

### 핵심 개념 설명

> 📝 **이벤트(Event)**: 시스템에서 발생한 상태 변화를 나타내는 메시지. 예: "EC2 인스턴스가 중지됨", "S3에 파일이 업로드됨", "사용자가 로그인함" 등.

> 📝 **이벤트 버스(Event Bus)**: 이벤트를 수신하고 규칙에 따라 라우팅하는 파이프라인. EventBridge의 핵심 구성 요소.

> 📝 **규칙(Rule)**: 특정 조건(이벤트 패턴)에 맞는 이벤트를 감지하고, 하나 이상의 대상(Target)으로 전달하는 조건문.

> 📝 **대상(Target)**: 규칙에 의해 이벤트를 전달받아 처리하는 AWS 서비스. Lambda, SQS, SNS, Step Functions 등이 될 수 있다.

#### EventBridge의 동작 흐름

```
[이벤트 소스]                [이벤트 버스]         [규칙 + 대상]
                                │
[AWS 서비스] ──이벤트──→        │     ┌── [규칙 1: EC2 중지됨] ──→ [Lambda: 자동 재시작]
                                │     │
[자체 앱] ──이벤트──→    [이벤트 버스] ──┤── [규칙 2: S3 업로드됨] ──→ [SQS: 처리 큐]
                                │     │
[SaaS 파트너] ──이벤트──→       │     └── [규칙 3: 매일 09시] ──→ [Lambda: 일일 보고서]
```

1. 다양한 **소스**에서 이벤트가 발생합니다
2. 이벤트가 **이벤트 버스**로 전달됩니다
3. 이벤트 버스에 정의된 **규칙**이 이벤트 패턴을 평가합니다
4. 조건에 맞는 이벤트가 규칙의 **대상(Target)**으로 라우팅됩니다

#### 세 가지 유형의 이벤트 버스

| 유형 | 설명 |
|------|------|
| **기본 이벤트 버스(Default)** | AWS 계정에 기본으로 존재. AWS 서비스에서 발생하는 이벤트가 자동으로 전달됨 |
| **사용자 정의 이벤트 버스(Custom)** | 자체 애플리케이션의 이벤트를 위해 직접 생성하는 이벤트 버스 |
| **SaaS 파트너 이벤트 버스(Partner)** | Zendesk, Datadog 등 SaaS 파트너의 이벤트를 수신하는 이벤트 버스 |

#### EventBridge의 규칙 유형

EventBridge 규칙은 두 가지 방식으로 트리거됩니다:

1. **이벤트 패턴(Event Pattern)**: 특정 패턴에 맞는 이벤트가 발생하면 트리거
   - 예: `"source": "aws.ec2"`, `"detail-type": "EC2 Instance State-change Notification"`, `"detail": {"state": "stopped"}`
   - → EC2 인스턴스가 중지될 때마다 자동으로 반응

2. **스케줄(Schedule)**: 정해진 시간에 정기적으로 트리거
   - 예: `rate(5 minutes)` — 5분마다 실행
   - 예: `cron(0 9 * * ? *)` — 매일 오전 9시에 실행

#### EventBridge vs SNS 비교

| 특성 | EventBridge | SNS |
|------|------------|-----|
| **이벤트 소스** | AWS 서비스, 자체 앱, SaaS 파트너 | 자체 애플리케이션이 직접 발행 |
| **라우팅 방식** | 규칙 기반 패턴 매칭 | 토픽 기반 구독 |
| **필터링** | 이벤트 내용 기반 정교한 필터링 | 구독 필터 정책 |
| **스케줄링** | cron/rate 기반 스케줄 지원 | 지원 안 함 |
| **SaaS 통합** | 네이티브 파트너 통합 | 수동 설정 필요 |
| **적합한 상황** | AWS 서비스 이벤트 반응, SaaS 통합 | 애플리케이션 알림, Fan-out |

> 💬 **참고**: EventBridge는 SNS의 상위 호환이 아닙니다. 각각 잘하는 영역이 다릅니다. 단순한 Pub/Sub 알림에는 SNS가, AWS 서비스 이벤트에 대한 자동 반응이나 복잡한 이벤트 라우팅에는 EventBridge가 적합합니다.

### 상세 예시

> 🔍 **예시 10.4.1: EC2 인스턴스 자동 모니터링**
>
> **상황**: 개발 환경의 EC2 인스턴스가 예기치 않게 중지되면, 운영팀에 즉시 알림을 보내야 합니다.
>
> **풀이/설명**:
> 1. EventBridge **기본 이벤트 버스**에 규칙을 생성합니다
> 2. 이벤트 패턴: "소스가 aws.ec2이고, 상태가 stopped로 변경된 이벤트"
> 3. 대상: SNS 토픽(운영팀 이메일 구독) 설정
> 4. EC2 인스턴스가 중지되면 → EventBridge가 이벤트를 감지 → 규칙이 매칭 → SNS 토픽에 전달 → 운영팀 이메일 수신
>
> **핵심 포인트**: AWS 서비스에서 발생하는 이벤트를 별도의 코드 없이 EventBridge 규칙만으로 자동 감지하고 반응할 수 있습니다. 이것이 "이벤트 기반 아키텍처"의 핵심입니다.

> 🔍 **예시 10.4.2: 정기 작업 스케줄링**
>
> **상황**: 매일 오전 9시에 전날의 시스템 사용 현황을 요약한 보고서를 자동으로 생성하고 싶습니다.
>
> **풀이/설명**:
> 1. EventBridge에 **스케줄 규칙**을 생성합니다
> 2. 스케줄: `cron(0 9 * * ? *)` (매일 09:00 UTC)
> 3. 대상: 보고서 생성 Lambda 함수
> 4. 매일 오전 9시에 EventBridge가 자동으로 Lambda를 호출하여 보고서를 생성합니다
>
> **핵심 포인트**: EventBridge의 스케줄 규칙은 cron 작업의 서버리스 대안입니다. 별도의 서버를 관리할 필요 없이 정기적인 작업을 실행할 수 있습니다.

### 미니 실습

> 🧪 **Mini Lab 10.4: EventBridge 스케줄 규칙 생성**
>
> **목표**: EventBridge에서 스케줄 규칙을 생성하여, 정해진 시간에 대상(SNS 토픽)이 트리거되는 것을 확인합니다.
> **예상 소요 시간**: 15분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS Management Console 상단 검색창에 `EventBridge`를 입력하고 **Amazon EventBridge**를 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **규칙(Rules)**을 클릭한 후, **규칙 생성(Create rule)** 버튼을 클릭합니다.
>
> **Step 3**: 규칙 세부 정보를 입력합니다:
> - **이름**: `my-first-rule`
> - **이벤트 버스**: "default" (기본값)
> - **규칙 유형(Rule type)**: "일정(Schedule)"을 선택합니다
> - **다음(Next)** 버튼을 클릭합니다
>
> **Step 4**: 스케줄 패턴을 설정합니다:
> - **일정 패턴(Schedule pattern)**: "일정한 비율(Rate expression)"을 선택합니다
> - **값(Value)**: `5`, **단위(Unit)**: "분(Minutes)"을 설정합니다 (5분마다 실행)
> - **다음(Next)** 버튼을 클릭합니다
>
> **Step 5**: 대상을 설정합니다:
> - **대상 유형(Target types)**: "AWS 서비스"를 선택합니다
> - **대상 선택(Select a target)**: "CloudWatch 로그 그룹(CloudWatch log group)"을 선택합니다
> - **로그 그룹(Log group)**: `/aws/events/my-first-rule`을 입력합니다
> - **다음(Next)** 버튼을 클릭합니다
>
> **Step 6**: 태그는 건너뛰고 **다음(Next)**을 클릭합니다. 검토 화면에서 설정을 확인한 후 **규칙 생성(Create rule)** 버튼을 클릭합니다.
>
> **Step 7**: 5분 정도 기다린 후, CloudWatch 서비스로 이동합니다. 왼쪽 메뉴에서 **로그 그룹(Log groups)**을 클릭하고 `/aws/events/my-first-rule`을 찾습니다. 로그 스트림이 생성되어 있다면 규칙이 정상적으로 작동한 것입니다.
>
> **확인**: EventBridge 스케줄 규칙이 5분 간격으로 자동 실행되어 CloudWatch 로그에 이벤트가 기록되는 것을 확인합니다.
>
> 🧹 **정리**: EventBridge **규칙(Rules)** 목록에서 `my-first-rule`을 선택하고 **삭제(Delete)**합니다. CloudWatch **로그 그룹** 목록에서 `/aws/events/my-first-rule`을 선택하고 **삭제(Delete)**합니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: EventBridge 관련 혼동 포인트
>
> - **EventBridge는 과거 "CloudWatch Events"로 불렸습니다**: 기존 CloudWatch Events는 EventBridge로 통합되었습니다. 콘솔에서 두 이름이 혼용될 수 있으나 같은 서비스입니다.
> - **기본 이벤트 버스와 사용자 정의 이벤트 버스를 혼동하지 마세요**: AWS 서비스 이벤트는 기본 이벤트 버스에 자동으로 전달됩니다. 자체 앱 이벤트는 사용자 정의 이벤트 버스를 만들어 사용합니다.
> - **CCP 시험 팁**: "AWS 서비스에서 자동으로 발생하는 이벤트에 반응", "이벤트 기반 아키텍처", "스케줄 기반 자동화"가 나오면 EventBridge를 떠올리세요.

### Section 요약

- **Amazon EventBridge**는 서버리스 이벤트 버스 서비스로, 다양한 소스의 이벤트를 규칙에 따라 대상으로 라우팅한다
- **이벤트 버스**는 이벤트를 수신하는 파이프라인, **규칙**은 이벤트를 필터링하고 대상으로 전달하는 조건문이다
- 이벤트 패턴 규칙(특정 이벤트에 반응)과 스케줄 규칙(정기 실행)을 모두 지원한다
- AWS 서비스 이벤트, 자체 애플리케이션 이벤트, SaaS 파트너 이벤트를 모두 처리할 수 있다
- SNS가 단순 Pub/Sub에 강하다면, EventBridge는 복잡한 이벤트 라우팅과 AWS 서비스 통합에 강하다

## 10.5 Amazon Kinesis와 MQ — 실시간 데이터 스트리밍, 레거시 메시지 브로커 호환

> 💡 **한 줄 요약**: Amazon Kinesis는 대규모 실시간 데이터 스트리밍을, Amazon MQ는 기존 온프레미스 메시지 브로커와의 호환을 위한 서비스이다.

### 직관적 이해

#### Kinesis: 고속도로의 CCTV 영상 분석

고속도로에 설치된 수천 대의 CCTV를 떠올려 봅시다. 이 카메라들은 **쉬지 않고 연속적으로** 영상 데이터를 보냅니다. 하루치를 모아서 분석하면 늦습니다. 사고가 발생하면 **즉시** 감지하여 대응해야 합니다.

이것이 **스트리밍 데이터**의 특성입니다: 끊임없이 생성되며, 실시간으로 처리해야 가치가 있습니다. **Amazon Kinesis**는 이런 대량의 스트리밍 데이터를 실시간으로 수집·처리·분석하는 서비스입니다.

**SQS/SNS와의 차이점**: SQS/SNS는 개별 "메시지"를 전달하는 데 초점을 맞추지만, Kinesis는 **연속적인 데이터 흐름(스트림)**을 실시간으로 처리하는 데 초점을 맞춥니다.

#### Amazon MQ: 기존 건물의 배관 시스템

새 건물을 지을 때는 최신 배관 시스템을 설치하면 됩니다. 하지만 오래된 건물을 리모델링할 때는 기존 배관과 호환되는 시스템이 필요합니다.

**Amazon MQ**는 바로 이 "기존 배관과 호환되는 시스템"입니다. 온프레미스에서 Apache ActiveMQ나 RabbitMQ 같은 메시지 브로커를 사용하던 기업이 AWS로 마이그레이션할 때, 기존 코드를 거의 변경하지 않고 클라우드로 전환할 수 있게 해줍니다.

### 핵심 개념 설명

#### Amazon Kinesis

> 📝 **스트리밍 데이터(Streaming Data)**: 수천 개의 소스에서 끊임없이 생성되는 연속적인 데이터. 예: IoT 센서 데이터, 웹사이트 클릭 로그, 소셜 미디어 피드, 금융 거래 데이터 등.

> 📝 **Amazon Kinesis**: 실시간 스트리밍 데이터를 수집, 처리, 분석하기 위한 완전관리형 서비스 제품군.

Kinesis는 여러 하위 서비스로 구성됩니다:

| 서비스 | 역할 | 비유 |
|--------|------|------|
| **Kinesis Data Streams** | 실시간 데이터 스트림 수집·처리 | 고속도로의 실시간 교통 데이터 파이프 |
| **Kinesis Data Firehose** | 데이터를 S3, Redshift 등 목적지로 자동 전달 | 데이터를 자동으로 창고(S3)에 적재하는 컨베이어 벨트 |
| **Kinesis Data Analytics** | 스트리밍 데이터에 실시간 SQL/Apache Flink 분석 수행 | 컨베이어 벨트 위에서 바로 분석하는 장치 |
| **Kinesis Video Streams** | 비디오 스트림 수집·처리 | CCTV 영상 실시간 분석 시스템 |

> 💬 **참고**: CCP 시험에서는 Kinesis의 세부 서비스를 깊이 있게 묻지 않습니다. "실시간 스트리밍 데이터 처리 = Kinesis"라는 핵심 연결만 기억하면 충분합니다. 데이터 분석 서비스에 대해서는 Chapter 11에서 더 자세히 다룹니다.

#### SQS vs Kinesis 비교

| 특성 | SQS | Kinesis Data Streams |
|------|-----|---------------------|
| **데이터 패턴** | 개별 메시지 (주문, 요청 등) | 연속 데이터 스트림 (로그, 센서) |
| **처리 방식** | Consumer가 메시지를 처리 후 삭제 | 데이터가 보존되어 여러 Consumer가 읽을 수 있음 |
| **데이터 보존** | 기본 4일, 최대 14일 | 기본 24시간, 최대 365일 |
| **순서 보장** | FIFO에서만 보장 | 파티션(샤드) 내에서 보장 |
| **처리량 관리** | 자동 (관리 불필요) | 샤드 수로 직접 관리 |
| **적합한 상황** | 서비스 간 비동기 메시징 | 대규모 실시간 데이터 분석 |

#### Amazon MQ

> 📝 **Amazon MQ**: Apache ActiveMQ와 RabbitMQ를 위한 관리형 메시지 브로커 서비스. 기존 온프레미스의 표준 메시징 프로토콜(JMS, AMQP, MQTT 등)을 사용하는 애플리케이션을 코드 변경 없이 AWS로 마이그레이션할 수 있게 해준다.

**언제 Amazon MQ를 사용하는가?**

- 기존에 ActiveMQ나 RabbitMQ를 사용하던 애플리케이션을 AWS로 이전할 때
- JMS, AMQP, MQTT 등의 표준 프로토콜을 반드시 사용해야 할 때
- 새로운 프로젝트라면? **SQS/SNS를 사용하세요.** Amazon MQ는 레거시 호환을 위한 서비스입니다.

> ⚠️ **주의**: 새로운 클라우드 네이티브 애플리케이션을 설계할 때는 Amazon MQ보다 SQS/SNS를 선택하는 것이 권장됩니다. SQS/SNS는 서버리스이고 자동 확장되지만, Amazon MQ는 브로커 인스턴스를 관리해야 합니다.

### 상세 예시

> 🔍 **예시 10.5.1: 실시간 주식 거래 데이터 분석**
>
> **상황**: 주식 거래 플랫폼에서 초당 수만 건의 거래 데이터가 발생합니다. 이 데이터를 실시간으로 분석하여 비정상 거래 패턴을 감지하고, 동시에 장기 분석을 위해 데이터를 저장해야 합니다.
>
> **풀이/설명**:
> 1. **Kinesis Data Streams**로 초당 수만 건의 거래 데이터를 실시간 수집합니다
> 2. **Kinesis Data Analytics**로 스트림 데이터를 실시간 분석하여 비정상 패턴을 감지합니다
> 3. **Kinesis Data Firehose**로 같은 데이터를 S3에 자동 저장하여 장기 분석에 활용합니다
> 4. 비정상 패턴이 감지되면 SNS로 운영팀에 알림을 보냅니다
>
> **핵심 포인트**: Kinesis의 핵심 가치는 "대량의 데이터를 실시간으로 처리"하는 것입니다. SQS로는 이런 대규모 연속 스트리밍 처리가 적합하지 않습니다.

> 🔍 **예시 10.5.2: 레거시 시스템 마이그레이션**
>
> **상황**: 온프레미스에서 ActiveMQ를 사용하여 주문 시스템과 배송 시스템을 연동하고 있는 기업이 AWS로 마이그레이션하려 합니다. 수십 개의 서비스가 JMS 프로토콜로 ActiveMQ에 연결되어 있습니다.
>
> **풀이/설명**:
> - 모든 서비스의 코드를 SQS/SNS용으로 변경하는 것은 시간과 비용이 많이 듭니다
> - **Amazon MQ**를 사용하면, ActiveMQ 호환 브로커를 AWS에서 관리형으로 실행할 수 있습니다
> - 기존 서비스 코드의 연결 엔드포인트만 변경하면 됩니다 (프로토콜과 API는 동일)
> - 이후 점진적으로 서비스를 SQS/SNS로 전환할 수 있습니다
>
> **핵심 포인트**: Amazon MQ는 "마이그레이션 경로"로서 가치가 있습니다. 새로 시작하는 프로젝트에서는 SQS/SNS가 더 적합합니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> - **Kinesis와 SQS를 혼동하지 마세요**: "실시간 스트리밍", "대규모 연속 데이터"가 나오면 Kinesis, "메시지 큐", "비동기 처리"가 나오면 SQS입니다.
> - **Amazon MQ는 새 프로젝트용이 아닙니다**: "기존 메시지 브로커를 마이그레이션", "ActiveMQ/RabbitMQ 호환"이 키워드입니다. 새 프로젝트는 SQS/SNS입니다.
> - **CCP 시험 팁**: Kinesis는 "실시간(real-time)"과 "스트리밍(streaming)"이라는 키워드가 핵심입니다. Amazon MQ는 "마이그레이션"과 "기존 메시지 브로커"가 키워드입니다.

### Section 요약

- **Amazon Kinesis**는 대규모 실시간 스트리밍 데이터를 수집·처리·분석하는 서비스 제품군이다
- Kinesis Data Streams(수집·처리), Firehose(목적지 전달), Data Analytics(실시간 분석), Video Streams(비디오)로 구성된다
- **Amazon MQ**는 ActiveMQ/RabbitMQ 호환 관리형 메시지 브로커로, 레거시 시스템 마이그레이션에 사용된다
- 새 프로젝트는 SQS/SNS, 실시간 스트리밍은 Kinesis, 레거시 호환은 Amazon MQ로 선택한다

## 10.6 Amazon SES와 Pinpoint — 이메일 전송 서비스, 마케팅 커뮤니케이션 플랫폼

> 💡 **한 줄 요약**: Amazon SES는 대규모 이메일 전송에 특화된 서비스이고, Amazon Pinpoint는 이메일·SMS·푸시 알림 등 다채널 마케팅 커뮤니케이션을 위한 플랫폼이다.

### 직관적 이해

#### SES: 우체국의 대량 우편 발송 서비스

기업이 100만 명의 고객에게 매달 청구서를 보내야 한다면, 편지를 한 통씩 직접 보내지 않습니다. 우체국의 **대량 우편 발송 서비스**를 이용합니다. 대량으로 보내므로 한 통당 비용이 매우 저렴하고, 우체국이 배달 과정을 관리합니다.

**Amazon SES(Simple Email Service)**는 이메일 버전의 대량 우편 발송 서비스입니다. 트랜잭션 이메일(주문 확인, 비밀번호 재설정)이나 마케팅 이메일을 대규모로 저렴하게 발송할 수 있습니다.

#### Pinpoint: 맞춤형 마케팅 에이전시

SES가 "이메일을 보내는 도구"라면, **Amazon Pinpoint**는 "누구에게, 어떤 채널로, 언제, 어떤 내용을 보낼지"를 종합적으로 관리하는 **마케팅 커뮤니케이션 플랫폼**입니다. 고객 세분화, A/B 테스트, 캠페인 관리, 성과 분석까지 모두 포함합니다.

### 핵심 개념 설명

#### Amazon SES (Simple Email Service)

> 📝 **Amazon SES(Simple Email Service)**: 대규모 이메일 전송을 위한 클라우드 기반 이메일 서비스. 트랜잭션 이메일, 마케팅 이메일, 대량 이메일 발송을 경제적으로 처리한다.

SES의 주요 기능:

| 기능 | 설명 |
|------|------|
| **이메일 전송** | API 또는 SMTP를 통한 대량 이메일 발송 |
| **이메일 수신** | 수신 이메일을 S3에 저장하거나 Lambda로 처리 |
| **발신자 인증** | SPF, DKIM, DMARC를 통한 이메일 신뢰성 확보 |
| **반송/불만 처리** | 반송(bounce) 및 불만(complaint) 자동 관리 |
| **전용 IP** | 발신 평판을 위한 전용 IP 주소 제공 |

**SES의 사용 사례**:
- 회원가입 확인 이메일
- 주문 확인/배송 알림 이메일
- 비밀번호 재설정 이메일
- 뉴스레터 발송
- 마케팅 프로모션 이메일

**SES vs SNS 이메일**: SNS로도 이메일을 보낼 수 있지만, SNS 이메일은 단순 텍스트만 가능하고 커스터마이징이 제한적입니다. SES는 HTML 이메일, 첨부 파일, 발신자 인증 등 전문적인 이메일 기능을 제공합니다.

#### Amazon Pinpoint

> 📝 **Amazon Pinpoint**: 이메일, SMS, 푸시 알림, 음성 메시지 등 다양한 채널을 통한 마케팅 커뮤니케이션을 관리하는 플랫폼. 고객 세분화, 캠페인 관리, A/B 테스트, 분석 기능을 제공한다.

Pinpoint의 주요 기능:

| 기능 | 설명 |
|------|------|
| **다채널 메시징** | 이메일, SMS, 푸시 알림, 음성을 하나의 플랫폼에서 관리 |
| **고객 세분화** | 사용자 행동, 속성 기반으로 대상 그룹 생성 |
| **캠페인 관리** | 마케팅 캠페인 생성, 스케줄링, 자동화 |
| **A/B 테스트** | 메시지 변형을 테스트하여 최적의 버전 선택 |
| **분석** | 전달률, 오픈율, 클릭율 등 캠페인 성과 분석 |
| **여정(Journey)** | 사용자 행동에 따른 자동화된 커뮤니케이션 시퀀스 |

#### SES vs Pinpoint 비교

| 특성 | SES | Pinpoint |
|------|-----|----------|
| **주요 목적** | 이메일 전송 | 다채널 마케팅 커뮤니케이션 |
| **채널** | 이메일만 | 이메일, SMS, 푸시 알림, 음성 |
| **고객 세분화** | 없음 (직접 구현 필요) | 내장된 세분화 기능 |
| **캠페인 관리** | 없음 | 캠페인 생성·스케줄·자동화 |
| **분석** | 기본 통계 (전달, 반송) | 상세 마케팅 분석 |
| **적합한 상황** | 트랜잭션 이메일, 대량 이메일 발송 | 마케팅 자동화, 고객 참여 관리 |

### 상세 예시

> 🔍 **예시 10.6.1: 전자상거래 사이트의 이메일 시스템**
>
> **상황**: 온라인 쇼핑몰에서 회원가입 확인, 주문 확인, 배송 알림 이메일을 자동으로 발송해야 합니다. 하루 평균 10만 건의 이메일을 발송합니다.
>
> **풀이/설명**:
> 1. **Amazon SES**를 이메일 발송 엔진으로 사용합니다
> 2. 주문 서비스(Lambda 또는 EC2)가 주문 처리 시 SES API를 호출하여 확인 이메일을 발송합니다
> 3. HTML 템플릿으로 브랜드에 맞는 이메일 디자인을 적용합니다
> 4. SES 비용: EC2에서 발송 시 월 62,000건 무료, 이후 1,000건당 $0.10
>
> **핵심 포인트**: SES는 **트랜잭션 이메일**(시스템이 자동으로 보내는 이메일)에 최적화되어 있습니다. 대량 발송에도 안정적이고 비용 효율적입니다.

> 🔍 **예시 10.6.2: 모바일 앱의 마케팅 캠페인**
>
> **상황**: 모바일 앱 운영자가 "지난 30일간 앱을 사용하지 않은 사용자"에게 복귀 유도 메시지를 이메일과 푸시 알림으로 동시에 보내고 싶습니다.
>
> **풀이/설명**:
> 1. **Amazon Pinpoint**에서 사용자 세그먼트를 생성합니다: "최근 30일 비활성 사용자"
> 2. 캠페인을 생성합니다: "복귀 유도 캠페인"
> 3. 메시지 채널: 이메일 + 푸시 알림을 동시 설정합니다
> 4. A/B 테스트: 메시지 문구 2가지를 테스트하여 더 효과적인 버전을 확인합니다
> 5. 캠페인 성과(오픈율, 앱 재방문율)를 Pinpoint 대시보드에서 분석합니다
>
> **핵심 포인트**: Pinpoint는 **마케팅 관점**에서 "누구에게, 무엇을, 어떻게 보낼지"를 종합적으로 관리합니다. 단순 발송(SES)을 넘어 전략적 커뮤니케이션을 지원합니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> - **SES, SNS, Pinpoint를 혼동하지 마세요**:
>   - **SNS**: 시스템 알림 (Pub/Sub 방식, 이메일은 단순 텍스트)
>   - **SES**: 전문적인 이메일 발송 (HTML, 첨부 파일, 대량 발송)
>   - **Pinpoint**: 다채널 마케팅 커뮤니케이션 플랫폼
> - **CCP 시험 팁**: "대량 이메일 발송", "트랜잭션 이메일"이 나오면 SES. "마케팅 캠페인", "고객 세분화", "다채널 메시징"이 나오면 Pinpoint.
> - SES는 처음에 "샌드박스 모드"로 시작하여 확인된 이메일 주소로만 발송할 수 있습니다. 프로덕션 사용을 위해서는 AWS에 샌드박스 해제를 요청해야 합니다.

### Section 요약

- **Amazon SES**는 대규모 이메일 전송에 특화된 서비스로, 트랜잭션 이메일과 마케팅 이메일 발송에 적합하다
- **Amazon Pinpoint**는 이메일·SMS·푸시 알림 등 다채널 마케팅 커뮤니케이션을 위한 통합 플랫폼이다
- SES는 "이메일 발송 엔진", Pinpoint는 "마케팅 커뮤니케이션 관리 플랫폼"으로 구분할 수 있다
- SNS의 이메일은 단순 알림용이고, SES는 전문적인 이메일 기능을 제공한다

# Part C: 통합 및 정리 (Integration Block)

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 AWS의 애플리케이션 통합과 메시징 서비스를 학습했습니다.

- **10.1 긴밀한 결합 vs 느슨한 결합**: 시스템 구성 요소 간의 직접적 의존을 줄이는 느슨한 결합(디커플링)은 확장성, 내결함성, 유지보수성을 높이는 핵심 설계 원칙이다. AWS의 통합 서비스들이 이를 구현하는 도구이다.

- **10.2 Amazon SQS**: 완전관리형 메시지 큐 서비스로, Producer-Consumer 패턴의 비동기 메시징을 구현한다. 표준 대기열(최대 처리량)과 FIFO 대기열(순서·중복 보장)을 선택할 수 있다.

- **10.3 Amazon SNS**: Pub/Sub 방식의 알림 서비스로, 하나의 메시지를 여러 구독자에게 동시에 push한다. SQS와 결합한 Fan-out 패턴이 자주 사용된다.

- **10.4 Amazon EventBridge**: 서버리스 이벤트 버스로, AWS 서비스·자체 앱·SaaS 파트너의 이벤트를 규칙에 따라 대상으로 라우팅한다. 스케줄 기반 자동화도 지원한다.

- **10.5 Amazon Kinesis와 MQ**: Kinesis는 대규모 실시간 스트리밍 데이터 처리, Amazon MQ는 레거시 메시지 브로커(ActiveMQ/RabbitMQ) 마이그레이션을 위한 서비스이다.

- **10.6 Amazon SES와 Pinpoint**: SES는 대규모 이메일 전송 서비스, Pinpoint는 다채널 마케팅 커뮤니케이션 플랫폼이다.

## 자기 점검 질문 (Self-Check Questions)

**Q1.** SQS는 Consumer에게 메시지를 자동으로 push하는 서비스이다. (O/X)

<details>
<summary>정답 확인</summary>

**X** — SQS는 **pull** 방식입니다. Consumer가 능동적으로 메시지를 가져가야 합니다. Push 방식은 SNS입니다.
</details>

**Q2.** 메시지의 순서가 반드시 보장되어야 하는 경우, SQS의 어떤 대기열 유형을 사용해야 하는가?

<details>
<summary>정답 확인</summary>

**FIFO 대기열** — First-In-First-Out 대기열은 메시지가 보낸 순서대로 정확히 1회만 전달되는 것을 보장합니다. 대기열 이름이 `.fifo`로 끝나야 합니다.
</details>

**Q3.** SNS 토픽에 메시지를 발행한 후, 해당 메시지는 SNS에 얼마나 오래 보관되는가?

<details>
<summary>정답 확인</summary>

**보관되지 않습니다** — SNS는 메시지를 구독자에게 전달한 후 보관하지 않습니다. 안정적인 처리가 필요하면 SNS → SQS (Fan-out) 패턴을 사용합니다.
</details>

**Q4.** 기존 온프레미스에서 RabbitMQ를 사용하던 애플리케이션을 AWS로 마이그레이션하려면, 어떤 AWS 서비스를 사용하는 것이 가장 적합한가?

<details>
<summary>정답 확인</summary>

**Amazon MQ** — Amazon MQ는 ActiveMQ와 RabbitMQ 호환 관리형 메시지 브로커로, 기존 코드를 거의 변경하지 않고 마이그레이션할 수 있습니다.
</details>

**Q5.** EventBridge는 과거 어떤 AWS 서비스의 이름으로 불렸는가?

<details>
<summary>정답 확인</summary>

**CloudWatch Events** — EventBridge는 CloudWatch Events를 확장·발전시킨 서비스입니다. 기존 CloudWatch Events의 모든 기능을 포함하며, SaaS 파트너 통합 등이 추가되었습니다.
</details>

**Q6.** 대규모 마케팅 이메일을 HTML 형식으로 발송해야 할 때, SNS보다 적합한 AWS 서비스는?

<details>
<summary>정답 확인</summary>

**Amazon SES (Simple Email Service)** — SNS의 이메일은 단순 텍스트만 가능하지만, SES는 HTML 이메일, 첨부 파일, 대량 발송, 발신자 인증 등 전문적인 이메일 기능을 제공합니다.
</details>

**Q7.** IoT 센서에서 초당 수만 건의 데이터가 연속적으로 발생하며, 이를 실시간으로 수집·분석해야 합니다. 가장 적합한 AWS 서비스는? (SQS / SNS / Kinesis / EventBridge)

<details>
<summary>정답 확인</summary>

**Amazon Kinesis** — Kinesis는 대규모 실시간 스트리밍 데이터의 수집·처리·분석에 특화된 서비스입니다. "실시간", "스트리밍", "연속 데이터"가 키워드입니다.
</details>

# Part C-2: 통합 실습 시나리오 (Integrated Hands-on Lab)

## 🧪 통합 실습: SNS와 SQS를 활용한 주문 알림 시스템 구축

> **시나리오**: 당신은 온라인 쇼핑몰의 클라우드 엔지니어입니다. 고객이 주문하면, (1) 주문 처리 시스템이 비동기적으로 주문을 처리하고, (2) 관리자에게 이메일 알림이 전송되는 시스템을 구축해야 합니다. SNS + SQS의 Fan-out 패턴을 직접 구현해 봅니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | SNS 토픽에 메시지를 발행하면, SQS 대기열과 이메일 구독자에게 동시에 전달되는 Fan-out 패턴 구현 |
| **사용 서비스** | Amazon SNS, Amazon SQS |
| **활용 개념** | Section 10.1 느슨한 결합, Section 10.2 SQS 메시지 큐, Section 10.3 SNS Pub/Sub, Fan-out 패턴 |
| **선수 실습** | 없음 (Mini Lab에서 생성한 리소스는 정리한 상태) |
| **예상 소요 시간** | 20분 |
| **비용** | Free Tier 범위 내 |

### 아키텍처 다이어그램

```
                                    ┌──→ [SQS: order-processing-queue] (주문 처리 시스템 역할)
                                    │
[주문 발생 (수동 메시지 발행)] ──→ [SNS: order-topic] ──┤
                                    │
                                    └──→ [이메일 구독자] (관리자 알림)
```

### 실습 단계

> **Phase 1: SNS 토픽 생성** (활용 개념: Section 10.3)
>
> **Step 1**: AWS Management Console에서 **SNS** 서비스로 이동합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **주제(Topics)** → **주제 생성(Create topic)**을 클릭합니다.
>
> **Step 3**: 토픽을 설정합니다:
> - **유형**: "표준(Standard)"
> - **이름**: `order-topic`
> - **주제 생성(Create topic)** 클릭
>
> **✅ 체크포인트**: `order-topic`이 생성되고, ARN(Amazon Resource Name)이 표시됩니다. 이 ARN을 메모해 둡니다.

> **Phase 2: SQS 대기열 생성 및 SNS 구독 연결** (활용 개념: Section 10.2, 10.3)
>
> **Step 1**: 새 탭에서 **SQS** 서비스로 이동합니다.
>
> **Step 2**: **대기열 생성(Create queue)**을 클릭합니다.
>
> **Step 3**: 대기열을 설정합니다:
> - **유형**: "표준(Standard)"
> - **이름**: `order-processing-queue`
> - 나머지 설정은 기본값 그대로
> - **대기열 생성(Create queue)** 클릭
>
> **Step 4**: 대기열 상세 페이지 하단의 **SNS 구독(SNS subscriptions)** 탭을 클릭합니다.
>
> **Step 5**: **Amazon SNS 주제 구독(Subscribe to Amazon SNS topic)** 버튼을 클릭합니다.
>
> **Step 6**: 드롭다운에서 Phase 1에서 생성한 `order-topic`을 선택하고 **저장(Save)** 버튼을 클릭합니다.
>
> **✅ 체크포인트**: SNS 구독 탭에 `order-topic`이 표시됩니다. 이제 `order-topic`에 발행된 메시지가 `order-processing-queue`에도 전달됩니다.

> **Phase 3: 이메일 구독 추가** (활용 개념: Section 10.3)
>
> **Step 1**: SNS 서비스로 돌아가 `order-topic`을 클릭합니다.
>
> **Step 2**: **구독 생성(Create subscription)**을 클릭합니다.
>
> **Step 3**: 구독을 설정합니다:
> - **프로토콜**: "이메일(Email)"
> - **엔드포인트**: 본인의 이메일 주소
> - **구독 생성(Create subscription)** 클릭
>
> **Step 4**: 이메일 수신함에서 확인 메일의 **Confirm subscription** 링크를 클릭합니다.
>
> **✅ 체크포인트**: SNS 토픽의 구독 목록에 SQS 구독(상태: 확인됨)과 이메일 구독(상태: 확인됨) 두 개가 표시됩니다.

> **Phase 4: 메시지 발행 및 Fan-out 확인** (활용 개념: Section 10.1, 10.2, 10.3)
>
> **Step 1**: SNS 서비스에서 `order-topic`의 **메시지 게시(Publish message)**를 클릭합니다.
>
> **Step 2**: 메시지를 작성합니다:
> - **제목**: `새 주문 알림`
> - **메시지 본문**:
>   ```
>   주문번호: ORD-2026-001
>   상품: AWS 클라우드 입문서
>   수량: 1
>   금액: 35,000원
>   ```
> - **메시지 게시(Publish message)** 클릭
>
> **Step 3**: 이메일 수신함을 확인합니다. "새 주문 알림" 이메일이 도착해야 합니다.
>
> **Step 4**: SQS 서비스로 이동합니다. `order-processing-queue`를 클릭하고 **메시지 전송 및 수신(Send and receive messages)**을 클릭합니다.
>
> **Step 5**: **메시지 폴링(Poll for messages)** 버튼을 클릭합니다. SNS에서 전달된 메시지가 표시됩니다. 메시지를 클릭하면 주문 내용이 포함된 본문을 확인할 수 있습니다.
>
> **✅ 체크포인트**: **하나의 메시지를 SNS 토픽에 발행했는데, 이메일 구독자와 SQS 대기열 모두에서 같은 메시지를 수신**했습니다. 이것이 Fan-out 패턴입니다!

### 최종 확인

- SNS 토픽 `order-topic`에 메시지를 발행하면:
  - 이메일로 관리자 알림이 전달됩니다 (SNS → 이메일)
  - SQS `order-processing-queue`에 메시지가 저장됩니다 (SNS → SQS)
- 두 경로가 독립적으로 동작하므로, 이메일 전송이 실패해도 SQS의 메시지 처리에 영향이 없습니다
- 이것이 Section 10.1에서 배운 **느슨한 결합**의 실제 구현입니다

### 🧹 리소스 정리 (Clean-up)

> ⚠️ **비용 방지**: 실습이 끝나면 아래 리소스를 삭제하세요. (순서가 중요합니다!)
>
> 1. **SNS 구독 삭제**: SNS → `order-topic` → 구독 탭에서 모든 구독을 선택하고 **삭제(Delete)**
> 2. **SNS 토픽 삭제**: SNS → 주제(Topics) → `order-topic` 선택 → **삭제(Delete)** (확인 입력란에 `delete me` 입력)
> 3. **SQS 대기열 삭제**: SQS → 대기열(Queues) → `order-processing-queue` 선택 → **삭제(Delete)** (확인 입력란에 `delete` 입력)

### 트러블슈팅 (Troubleshooting)

> **Q: 이메일 구독 확인 메일이 오지 않아요.**
> A: 스팸/정크 메일함을 확인하세요. AWS SNS에서 보내는 확인 메일이 스팸으로 분류될 수 있습니다. 이메일 주소에 오타가 없는지도 확인하세요. 확인 메일은 구독 생성 후 수 분 내에 도착합니다.

> **Q: SQS에서 메시지 폴링을 했는데 메시지가 보이지 않아요.**
> A: 메시지가 전달되기까지 몇 초가 걸릴 수 있습니다. 잠시 기다린 후 다시 폴링해 보세요. 또한, SQS 대기열이 SNS 토픽에 올바르게 구독되어 있는지 확인하세요 (SQS 대기열의 "SNS 구독" 탭에서 확인).

> **Q: SQS 대기열의 SNS 구독을 설정할 때, 토픽이 드롭다운에 보이지 않아요.**
> A: SNS 토픽과 SQS 대기열이 같은 리전에 있는지 확인하세요. 서로 다른 리전에 생성했다면 드롭다운에 표시되지 않습니다.

## D2. Exercise (응용 문제)

> **Exercise 10.1**: 다음 시나리오를 읽고, 가장 적합한 AWS 통합 서비스 조합을 제시하고 그 이유를 설명하세요.
>
> **시나리오**: 전자상거래 플랫폼에서 주문이 접수되면, (1) 결제 서비스가 결제를 처리하고, (2) 재고 서비스가 재고를 차감하고, (3) 알림 서비스가 고객에게 주문 확인 이메일을 보내야 합니다. 세 서비스는 서로 독립적으로 동작해야 하며, 하나의 서비스가 실패해도 나머지 서비스에 영향이 없어야 합니다.
>
> **힌트**: Fan-out 패턴을 떠올려 보세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 하나의 이벤트(주문 접수)를 여러 서비스가 독립적으로 처리해야 하므로, SNS + SQS Fan-out 패턴이 적합합니다.
>
> **아키텍처 설계**:
> ```
>                              ┌──→ [SQS: 결제 큐] ──→ [결제 서비스]
>                              │
> [주문 서비스] ──→ [SNS: 주문 토픽] ──→ [SQS: 재고 큐] ──→ [재고 서비스]
>                              │
>                              └──→ [SQS: 알림 큐] ──→ [알림 서비스] ──→ [SES: 이메일 발송]
> ```
>
> **서비스 선택 이유**:
> 1. **SNS (주문 토픽)**: 하나의 주문 메시지를 세 서비스에 동시에 전달 (Fan-out)
> 2. **SQS (각 서비스 앞)**: 각 서비스가 독립적으로 메시지를 처리. 서비스 장애 시 메시지가 큐에 보관되어 복구 후 처리 가능
> 3. **SES**: 고객에게 HTML 형식의 주문 확인 이메일 발송
>
> **느슨한 결합의 효과**:
> - 결제 서비스가 다운되어도, 재고 차감과 이메일 발송은 정상적으로 동작
> - 각 서비스를 독립적으로 확장 가능 (블랙 프라이데이에 결제 서비스만 3배로 확장)
> - 새로운 서비스(예: 분석 서비스)를 추가하려면 SNS 구독만 추가하면 됨
>
> **보충 설명**: 이메일 발송에 SNS의 이메일 구독을 직접 사용할 수도 있지만, SES를 사용하면 HTML 이메일, 브랜딩, 첨부 파일 등 전문적인 이메일 기능을 활용할 수 있습니다.
> </details>

> **Exercise 10.2**: 다음 요구사항을 읽고, SQS 표준 대기열과 FIFO 대기열 중 어떤 것을 사용해야 하는지 각각 판단하고 근거를 설명하세요.
>
> (a) 소셜 미디어 플랫폼에서 사용자가 업로드한 이미지의 썸네일을 자동 생성하는 시스템
> (b) 은행의 계좌 이체 요청을 처리하는 시스템
> (c) IoT 센서에서 수집한 온도 데이터를 데이터베이스에 저장하는 시스템
> (d) 티켓 예매 시스템에서 좌석 예약 요청을 처리하는 시스템
>
> <details>
> <summary>모범 답안</summary>
>
> **(a) 이미지 썸네일 생성 → 표준 대기열**
> - 썸네일 생성 순서는 중요하지 않음 (어떤 이미지를 먼저 처리해도 무방)
> - 같은 이미지가 두 번 처리되어도 결과가 동일하므로 중복이 큰 문제가 아님
> - 높은 처리량이 필요할 수 있음 (수만 장의 이미지 동시 업로드 시)
>
> **(b) 은행 계좌 이체 → FIFO 대기열**
> - 이체 순서가 중요함 (계좌 A에서 10만원 출금 → 5만원 입금의 순서가 바뀌면 잔고 오류)
> - 같은 이체가 두 번 실행되면 금전적 손실 발생 → 정확히 1회 전달 필수
>
> **(c) IoT 온도 데이터 저장 → 표준 대기열**
> - 센서 데이터는 대량으로 발생하므로 높은 처리량이 필요
> - 같은 온도 데이터가 중복 저장되어도 분석에 큰 영향이 없음
> - 순서가 바뀌어도 타임스탬프가 있으므로 나중에 정렬 가능
>
> **(d) 좌석 예약 요청 → FIFO 대기열**
> - 예약 순서가 중요함 (먼저 요청한 사람이 좌석을 확보해야 함)
> - 같은 예약이 두 번 처리되면 이중 결제 발생 → 정확히 1회 전달 필수
>
> **보충 설명**: FIFO 대기열의 처리량 제한(초당 300건)이 문제가 되는 경우, 표준 대기열을 사용하되 애플리케이션 레벨에서 멱등성(idempotency)과 순서 관리를 구현할 수도 있습니다. 하지만 CCP 수준에서는 "순서/중복이 중요하면 FIFO"로 기억하면 충분합니다.
> </details>

## D3. Problem (심화 문제)

> **Problem 10.1**: 다음 시나리오를 읽고, 전체 시스템 아키텍처를 설계하세요. 어떤 AWS 서비스를 어떤 역할로 사용할지, 그리고 왜 그 서비스를 선택했는지를 설명해야 합니다.
>
> **시나리오**: 글로벌 뉴스 미디어 기업이 다음 요구사항을 가진 시스템을 구축하려 합니다:
>
> 1. 전 세계 기자들이 기사를 작성하면, 편집팀에게 실시간 알림이 전달되어야 합니다
> 2. 기사가 발행되면, (a) 번역 서비스가 다국어 번역을 수행하고, (b) 이미지 처리 서비스가 썸네일을 생성하고, (c) 검색 인덱스 서비스가 검색 데이터를 업데이트해야 합니다. 세 작업은 서로 독립적입니다
> 3. 웹사이트 방문자의 클릭 데이터가 초당 수만 건 발생하며, 이를 실시간으로 분석하여 인기 기사 순위를 업데이트해야 합니다
> 4. 매일 아침 8시에 전날의 기사 통계 보고서가 자동 생성되어야 합니다
> 5. 기사 발행 시 구독자에게 이메일 뉴스레터가 발송되어야 합니다
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 각 요구사항을 분석하여 가장 적합한 통합 서비스를 매칭하고, 전체를 느슨하게 결합된 아키텍처로 설계합니다.
>
> **풀이 전략**: 요구사항별로 적합한 서비스를 선택합니다.
>
> **상세 풀이**:
>
> **(1) 기자 → 편집팀 실시간 알림**: **Amazon SNS**
> - 기사 작성 이벤트를 "기사-작성" SNS 토픽에 발행
> - 편집팀이 이메일, SMS 등으로 구독하여 실시간 알림 수신
> - 이유: 1:N 즉시 알림에 SNS가 적합
>
> **(2) 기사 발행 → 번역/이미지/검색 독립 처리**: **SNS + SQS Fan-out**
> - "기사-발행" SNS 토픽에 발행
> - 3개의 SQS 대기열(번역 큐, 이미지 큐, 검색 큐)이 각각 구독
> - 각 서비스가 자신의 큐에서 독립적으로 메시지를 처리
> - 이유: 독립적·비동기 처리에 Fan-out 패턴이 최적
>
> **(3) 클릭 데이터 실시간 분석**: **Amazon Kinesis**
> - Kinesis Data Streams로 초당 수만 건의 클릭 데이터 수집
> - Kinesis Data Analytics로 실시간 인기 기사 순위 계산
> - 이유: 대규모 연속 스트리밍 데이터의 실시간 처리에 Kinesis가 적합 (SQS는 이런 실시간 스트리밍에 적합하지 않음)
>
> **(4) 매일 아침 8시 보고서 자동 생성**: **Amazon EventBridge (스케줄 규칙)**
> - EventBridge에 cron 규칙 생성: `cron(0 8 * * ? *)`
> - 대상: 보고서 생성 Lambda 함수
> - 이유: 정기적 스케줄 실행에 EventBridge 스케줄 규칙이 적합
>
> **(5) 구독자 이메일 뉴스레터 발송**: **Amazon SES** (또는 Pinpoint)
> - 기사 발행 시 뉴스레터 발송 Lambda가 SES API를 호출하여 HTML 이메일 발송
> - 이 Lambda는 (2)의 SNS → SQS 패턴에 추가 큐를 만들어 트리거할 수 있음
> - 이유: 대량 HTML 이메일 발송에 SES가 적합. 마케팅 세분화가 필요하면 Pinpoint 사용
>
> **전체 아키텍처 다이어그램**:
> ```
> [기자 앱]
>    │
>    ├──기사 작성──→ [SNS: 기사-작성 토픽] ──→ [편집팀 알림]
>    │
>    └──기사 발행──→ [SNS: 기사-발행 토픽] ──┬→ [SQS: 번역 큐] → [번역 서비스]
>                                             ├→ [SQS: 이미지 큐] → [이미지 서비스]
>                                             ├→ [SQS: 검색 큐] → [검색 서비스]
>                                             └→ [SQS: 뉴스레터 큐] → [Lambda] → [SES]
>
> [웹사이트] ──클릭 데이터──→ [Kinesis Data Streams] → [Kinesis Analytics] → [인기 순위 DB]
>
> [EventBridge: cron 08:00] ──→ [Lambda: 보고서 생성]
> ```
>
> **확장 생각**: 이 아키텍처에서 모든 서비스가 느슨하게 결합되어 있으므로, 새로운 요구사항(예: 소셜 미디어 자동 공유)이 추가되면 SNS 토픽에 구독을 추가하기만 하면 됩니다. 이것이 느슨한 결합 아키텍처의 최대 장점입니다.
> </details>

> **Problem 10.2**: SQS, SNS, EventBridge는 모두 "서비스 간 통신"을 돕는 도구이지만, 각각의 설계 철학과 적합한 영역이 다릅니다. 세 서비스를 "통신 방식", "메시지 보존 여부", "이벤트 소스의 다양성", "사용자가 신경 써야 하는 것" 네 가지 기준으로 비교 분석하고, 각 서비스가 **유일하게 최적인 시나리오**를 하나씩 제시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 세 서비스의 근본적인 차이를 이해하면, CCP 시험뿐 아니라 실무에서도 올바른 선택을 할 수 있습니다.
>
> **비교 분석**:
>
> | 기준 | SQS | SNS | EventBridge |
> |------|-----|-----|-------------|
> | **통신 방식** | Pull (Consumer가 가져감) | Push (구독자에게 전달) | Push (대상으로 라우팅) |
> | **메시지 보존** | O (최대 14일) | X (전달 후 삭제) | X (전달 후 삭제) |
> | **이벤트 소스** | 자체 앱만 (Producer가 직접 전송) | 자체 앱만 (Publisher가 직접 발행) | AWS 서비스 + 자체 앱 + SaaS 파트너 |
> | **사용자가 신경 쓸 것** | 메시지 처리·삭제 로직 | 구독자 관리, 필터 정책 | 이벤트 패턴·규칙 설계 |
>
> **각 서비스가 유일하게 최적인 시나리오**:
>
> **SQS만 최적**: "주문 처리 서비스가 다운되었을 때, 주문 요청이 유실되지 않고 복구 후 자동으로 처리되어야 한다"
> → 메시지 보존이 핵심. SNS/EventBridge는 전달 실패 시 메시지가 유실될 수 있지만, SQS는 메시지를 안전하게 보관
>
> **SNS만 최적**: "하나의 결제 완료 이벤트를 고객 이메일, 내부 Slack 알림, 회계 시스템, 포인트 적립 시스템에 동시에 전달해야 한다"
> → 다수의 이종(異種) 구독자에게 동시 전달이 핵심. EventBridge도 가능하지만, 단순 Fan-out에는 SNS가 더 직관적이고 가볍다
>
> **EventBridge만 최적**: "S3 버킷에 새 파일이 업로드되면, 파일 확장자에 따라 다른 Lambda 함수를 자동으로 호출해야 한다. 또한 매일 새벽 2시에 정리 작업을 실행해야 한다"
> → AWS 서비스의 자동 이벤트를 규칙 기반으로 라우팅 + 스케줄 기능. SQS/SNS로는 AWS 서비스 이벤트를 직접 수신할 수 없다
>
> **확장 생각**: 실무에서는 이 세 서비스를 함께 사용하는 경우가 많습니다. 예를 들어, EventBridge가 AWS 서비스 이벤트를 감지 → SNS 토픽으로 전달 → SQS 대기열로 Fan-out하여 각 서비스가 안정적으로 처리하는 패턴이 일반적입니다. 서비스를 배타적으로 선택하기보다, 각각의 강점을 조합하는 것이 좋습니다.
> </details>

> **Problem 10.3**: 한 기업이 모놀리식 아키텍처에서 마이크로서비스 아키텍처로 전환하려고 합니다. 현재 시스템은 하나의 EC2 인스턴스에서 주문, 결제, 재고, 배송, 알림 기능이 모두 동작하고 있습니다. 이 전환을 **단계적으로** 수행하는 마이그레이션 계획을 수립하세요. 각 단계에서 어떤 AWS 통합 서비스를 어떤 목적으로 도입하는지 설명하고, 각 단계의 위험 요소와 완화 방안도 제시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: "빅뱅" 전환(한 번에 모두 전환)은 위험합니다. 단계적으로 하나씩 분리하면서, 각 단계에서 느슨한 결합을 점진적으로 도입합니다.
>
> **풀이 전략**: 가장 독립적인 기능부터 분리하고, 점차 핵심 기능을 분리합니다.
>
> **상세 풀이**:
>
> **1단계: 알림 서비스 분리 (가장 안전한 시작점)**
> - 알림 기능을 Lambda로 분리하고, 기존 모놀리스와 **Amazon SNS**로 연결
> - 모놀리스에서 알림이 필요한 시점에 SNS 토픽에 메시지를 발행
> - 위험: 낮음 (알림이 실패해도 핵심 비즈니스에 영향 없음)
> - 완화: SNS → SQS → Lambda 패턴으로 메시지 유실 방지
>
> **2단계: 재고 서비스 분리**
> - 재고 관리를 별도 서비스로 분리하고, **Amazon SQS**로 연결
> - 모놀리스가 재고 변경 요청을 SQS에 전송, 재고 서비스가 비동기로 처리
> - 위험: 중간 (재고 정합성 문제 가능)
> - 완화: FIFO 대기열로 순서 보장, Dead-Letter Queue로 실패 메시지 추적
>
> **3단계: 배송 서비스 분리**
> - 배송 서비스를 분리하고, SNS + SQS Fan-out 패턴 도입
> - 주문 완료 시 SNS 토픽에 발행 → 재고 큐 + 배송 큐로 Fan-out
> - 위험: 중간 (배송 지연 가능)
> - 완화: SQS의 메시지 보존으로 배송 서비스 장애 시에도 요청 유실 방지
>
> **4단계: 결제 서비스 분리 (가장 신중하게)**
> - 결제는 비즈니스 핵심이므로 마지막에 분리
> - 결제 요청을 SQS FIFO 대기열로 전달하여 순서·중복 보장
> - **EventBridge**로 결제 상태 변경 이벤트를 모니터링
> - 위험: 높음 (결제 실패 = 매출 손실)
> - 완화: 기존 결제 로직과 새 서비스를 병렬 실행하여 결과를 비교하는 "섀도잉" 기간 운영
>
> **5단계: 모놀리스 폐기**
> - 모든 기능이 분리되면 기존 모놀리스 EC2 인스턴스 제거
> - EventBridge로 전체 시스템의 이벤트를 모니터링하는 중앙 관제 구성
>
> **최종 아키텍처**:
> ```
> [API Gateway]
>    │
>    ├──→ [주문 서비스] ──→ [SNS: 주문 토픽] ──┬→ [SQS] → [결제 서비스]
>    │                                          ├→ [SQS] → [재고 서비스]
>    │                                          ├→ [SQS] → [배송 서비스]
>    │                                          └→ [SQS] → [알림 서비스] → [SES]
>    │
>    └──→ [EventBridge] → 모니터링·스케줄 작업
> ```
>
> **확장 생각**: 이 문제의 핵심은 "기술적 선택"보다 "리스크 관리"입니다. 실무에서 마이크로서비스 전환은 기술 문제가 아니라 조직·프로세스 문제인 경우가 많습니다. 각 단계에서 충분한 테스트와 모니터링을 수행하며, 문제 발생 시 이전 상태로 롤백할 수 있는 계획이 필수적입니다.
> </details>
