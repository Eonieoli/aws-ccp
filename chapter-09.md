## 9.1 컨테이너 기초 — 컨테이너 vs 가상 머신, Docker의 핵심 개념

> 💡 **한 줄 요약**: 컨테이너는 애플리케이션과 실행 환경을 하나로 묶어, 어디서든 동일하게 실행되도록 보장하는 경량 가상화 기술이다.

### 직관적 이해

여러분이 친구에게 요리를 선물한다고 상상해 봅시다.

**방법 1 — 가상 머신(EC2) 방식**: 요리를 전달하기 위해 주방 전체를 통째로 옮겨줍니다. 냉장고, 가스레인지, 싱크대, 조리도구, 식재료, 그리고 완성된 요리까지 모두 포함됩니다. 확실히 요리를 만들 수 있지만, 엄청나게 크고 무겁습니다.

**방법 2 — 컨테이너 방식**: 요리와 레시피, 그리고 필요한 최소한의 양념·도구만 도시락 박스에 담아 전달합니다. 친구 집의 주방(운영체제)을 공유하되, 도시락 안의 내용물은 독립적으로 격리되어 있습니다.

> 🔗 **연결**: Chapter 5에서 EC2 인스턴스는 "AWS 클라우드에 있는 가상 서버"라고 배웠습니다. EC2는 운영체제(OS)부터 시작하는 완전한 가상 머신입니다. 컨테이너는 이 가상 머신보다 더 가볍고 빠른 실행 단위입니다.

**왜 컨테이너가 필요해졌을까?**

EC2 같은 가상 머신은 강력하지만, 몇 가지 현실적인 문제가 있었습니다:

- **"내 컴퓨터에서는 되는데..."**: 개발자의 노트북에서는 잘 작동하던 프로그램이, 서버에 올리면 작동하지 않는 문제가 빈번했습니다. 운영체제 버전, 라이브러리 버전, 설정 파일 등 환경이 조금만 달라도 프로그램이 깨질 수 있기 때문입니다.
- **무거운 시작**: 가상 머신은 부팅에 수십 초에서 수 분이 걸립니다. 운영체제 전체를 시작해야 하니까요.
- **자원 낭비**: 간단한 웹 서버 하나를 실행하기 위해 운영체제 전체를 띄우는 것은 비효율적입니다.

컨테이너는 이런 문제를 해결하기 위해 등장했습니다. "애플리케이션이 실행되는 데 필요한 모든 것을 하나의 패키지로 묶어서, 어디서든 동일하게 실행되게 하자"는 것이 컨테이너의 핵심 아이디어입니다.

### 핵심 개념 설명

> 📝 **컨테이너(Container)**: 애플리케이션 코드, 런타임(실행 환경), 시스템 도구, 라이브러리, 설정 등을 하나의 패키지로 묶은 경량 실행 단위. 호스트 운영체제의 커널을 공유하면서도, 각 컨테이너는 서로 격리된 환경에서 독립적으로 실행된다.

> 📝 **Docker**: 컨테이너를 만들고, 배포하고, 실행하기 위한 가장 널리 사용되는 오픈소스 플랫폼. 컨테이너 기술 자체를 대중화시킨 도구이다.

> 📝 **컨테이너 이미지(Container Image)**: 컨테이너를 실행하기 위한 모든 것(코드, 런타임, 라이브러리, 설정 등)이 담긴 읽기 전용 템플릿. 이미지를 실행하면 컨테이너가 된다.

**가상 머신 vs 컨테이너 — 구조적 차이**

```
[ 가상 머신 (EC2) 구조 ]              [ 컨테이너 구조 ]

┌──────────┐ ┌──────────┐       ┌──────────┐ ┌──────────┐
│  App A   │ │  App B   │       │  App A   │ │  App B   │
├──────────┤ ├──────────┤       ├──────────┤ ├──────────┤
│ Bins/Libs│ │ Bins/Libs│       │ Bins/Libs│ │ Bins/Libs│
├──────────┤ ├──────────┤       └──────────┘ └──────────┘
│ Guest OS │ │ Guest OS │       ┌──────────────────────┐
│ (Linux)  │ │(Windows) │       │   Container Runtime  │
└──────────┘ └──────────┘       │     (e.g., Docker)   │
┌──────────────────────┐       └──────────────────────┘
│     Hypervisor       │       ┌──────────────────────┐
└──────────────────────┘       │      Host OS         │
┌──────────────────────┐       └──────────────────────┘
│      Host OS         │       ┌──────────────────────┐
└──────────────────────┘       │     Hardware         │
┌──────────────────────┐       └──────────────────────┘
│     Hardware         │
└──────────────────────┘
```

핵심 차이를 정리하면:

| 비교 항목 | 가상 머신 (EC2) | 컨테이너 |
|-----------|----------------|----------|
| **격리 수준** | 완전한 격리 (별도 OS) | 프로세스 수준 격리 (OS 커널 공유) |
| **크기** | 수 GB (OS 포함) | 수 MB ~ 수백 MB |
| **시작 시간** | 수십 초 ~ 수 분 | 수 초 이내 |
| **자원 효율성** | OS마다 자원 소모 | OS 공유로 효율적 |
| **이식성** | AMI에 종속적 | 어디서든 동일하게 실행 |
| **사용 사례** | 완전한 서버 환경 필요 시 | 마이크로서비스, 빠른 배포 |

> 🔗 **연결**: Chapter 5에서 배운 AMI(Amazon Machine Image)는 EC2 인스턴스를 만들기 위한 템플릿이었습니다. 컨테이너 이미지는 AMI와 비슷한 역할을 하지만, 운영체제 전체가 아닌 애플리케이션과 그 의존성만 포함하므로 훨씬 가볍습니다.

**Docker의 핵심 개념**

Docker를 이해하기 위한 핵심 용어들을 살펴봅시다:

1. **Dockerfile**: 컨테이너 이미지를 만들기 위한 "레시피". 어떤 기본 이미지를 사용하고, 어떤 파일을 복사하고, 어떤 명령을 실행할지 텍스트로 기술합니다.

2. **이미지(Image)**: Dockerfile을 기반으로 만들어진 읽기 전용 패키지. "설계도"에 비유할 수 있습니다.

3. **컨테이너(Container)**: 이미지를 실행한 인스턴스. "설계도로 지은 실제 건물"에 비유할 수 있습니다. 하나의 이미지로 여러 컨테이너를 동시에 실행할 수 있습니다.

4. **레지스트리(Registry)**: 컨테이너 이미지를 저장하고 공유하는 저장소. 앱 스토어처럼, 필요한 이미지를 다운로드(pull)하거나 내가 만든 이미지를 업로드(push)할 수 있습니다.

```
[Dockerfile] --빌드-→ [이미지] --실행-→ [컨테이너]
                         │
                         ▼
                    [레지스트리에 저장]
                    (ECR, Docker Hub 등)
```

### 상세 예시

> 🔍 **예시 9.1.1: 일상적 비유 — 앱 설치와 컨테이너**
>
> **상황**: 스마트폰에 앱을 설치하는 것을 생각해 봅시다.
> **설명**: 앱 스토어에서 앱(이미지)을 다운로드하면, 폰에 설치된 앱(컨테이너)이 됩니다. 앱은 필요한 모든 것을 패키지 안에 가지고 있어서, 어떤 폰에서든 동일하게 작동합니다. 폰의 운영체제(Android/iOS)는 공유하지만, 각 앱은 독립적으로 실행되고, 다른 앱의 데이터에 접근할 수 없습니다.
> **핵심 포인트**: 컨테이너도 마찬가지입니다. 필요한 모든 것을 이미지 안에 담고, 호스트 OS를 공유하면서 독립적으로 실행됩니다.

> 🔍 **예시 9.1.2: 전문적 예시 — EC2에서 컨테이너로의 전환**
>
> **상황**: 한 스타트업이 3개의 마이크로서비스(사용자 관리, 주문 처리, 결제)를 운영합니다. 처음에는 각 서비스마다 EC2 인스턴스를 하나씩 할당했습니다.
> **설명**:
> - EC2 방식: 3개의 인스턴스 × (Linux OS + 런타임 + 서비스 코드) = 각 인스턴스가 최소 8GB 메모리 사용, 부팅에 30초~1분, 운영비용 3배
> - 컨테이너 방식: 1개의 EC2 인스턴스 위에 3개의 컨테이너를 실행. 각 컨테이너는 200MB, 시작에 2초, OS 오버헤드 1번만 발생
> **핵심 포인트**: 컨테이너를 사용하면 자원 효율성이 크게 높아지고, 배포와 확장이 빨라집니다. 특히 마이크로서비스 아키텍처에서 컨테이너의 가치가 빛납니다.

> 📝 **컨테이너 오케스트레이션(Container Orchestration)**: 다수의 컨테이너를 자동으로 배포, 관리, 확장, 네트워킹하는 시스템. 컨테이너가 수십~수백 개로 늘어나면, 어떤 서버에 어떤 컨테이너를 배치할지, 장애가 나면 어떻게 복구할지를 사람이 일일이 관리할 수 없기 때문에 오케스트레이션 도구가 필요하다.

일상적으로 비유하면, 오케스트레이션은 오케스트라의 지휘자와 같습니다. 수십 명의 연주자(컨테이너)가 각자의 악기를 연주하지만, 지휘자(오케스트레이터)가 전체를 조율해야 아름다운 음악(안정적인 서비스)이 됩니다.

AWS에서 컨테이너 오케스트레이션을 담당하는 대표 서비스가 **Amazon ECS**와 **Amazon EKS**이며, 이어지는 Section에서 자세히 살펴보겠습니다.

> ⚠️ **주의**
>
> - **컨테이너 ≠ Docker**: Docker는 컨테이너를 다루는 도구 중 하나일 뿐, 컨테이너 기술 자체는 Docker 이전에도 존재했습니다(예: Linux LXC). AWS CCP 시험에서는 "컨테이너"라는 개념과 AWS의 컨테이너 서비스를 이해하는 것이 중요합니다.
> - **컨테이너가 항상 최선은 아닙니다**: 단일 애플리케이션을 하나의 서버에서 실행하는 간단한 경우에는 EC2 하나로 충분합니다. 컨테이너는 특히 여러 서비스를 독립적으로 배포하고 확장해야 하는 마이크로서비스 아키텍처에서 진가를 발휘합니다.

### Section 요약

- 컨테이너는 애플리케이션과 실행 환경을 하나로 묶은 경량 실행 단위로, 호스트 OS 커널을 공유하면서 격리된 환경을 제공한다
- 가상 머신(EC2)에 비해 크기가 작고(MB 단위), 시작이 빠르며(초 단위), 자원 효율성이 높다
- Docker는 컨테이너를 만들고 실행하는 대표적인 도구이며, Dockerfile → 이미지 → 컨테이너의 흐름으로 작동한다
- 컨테이너가 많아지면 오케스트레이션(자동 배포·관리·확장)이 필요하며, AWS에서는 ECS와 EKS가 이 역할을 담당한다

## 9.2 Amazon ECS와 ECR — AWS의 컨테이너 오케스트레이션 서비스

> 💡 **한 줄 요약**: Amazon ECS는 AWS가 자체 개발한 컨테이너 오케스트레이션 서비스이고, ECR은 컨테이너 이미지를 저장하는 AWS의 전용 레지스트리이다.

### 직관적 이해

Section 9.1에서 컨테이너를 도시락에 비유했습니다. 이제 도시락이 수십, 수백 개가 되었다고 상상해 봅시다.

- 어떤 도시락을 어느 테이블(서버)에 놓을지 결정해야 합니다
- 도시락 하나가 상해서 못 쓰게 되면 새 도시락으로 교체해야 합니다
- 점심시간에 수요가 몰리면 도시락 수를 늘려야 합니다

이런 관리를 자동으로 해 주는 것이 **Amazon ECS(Elastic Container Service)** 입니다. 그리고 도시락의 레시피(컨테이너 이미지)를 안전하게 보관하는 냉장고가 **Amazon ECR(Elastic Container Registry)** 입니다.

**왜 ECS가 필요한가?**

컨테이너 하나를 EC2 위에서 직접 실행하는 것은 어렵지 않습니다. 하지만 실제 서비스 운영에서는:
- 수십~수백 개의 컨테이너를 동시에 실행해야 합니다
- 트래픽 변화에 따라 컨테이너 수를 늘리거나 줄여야 합니다
- 컨테이너가 비정상 종료되면 자동으로 재시작해야 합니다
- 컨테이너 간 네트워크 통신을 관리해야 합니다

이 모든 것을 수동으로 하는 것은 비현실적이며, ECS가 이를 자동화합니다.

### 핵심 개념 설명

> 📝 **Amazon ECS(Elastic Container Service)**: AWS가 자체 개발한 완전 관리형 컨테이너 오케스트레이션 서비스. Docker 컨테이너를 쉽게 실행, 중지, 관리할 수 있다.

> 📝 **Amazon ECR(Elastic Container Registry)**: AWS의 완전 관리형 Docker 컨테이너 이미지 저장소. 컨테이너 이미지를 안전하게 저장, 관리, 배포할 수 있다.

**ECS의 핵심 구성 요소**

ECS를 이해하려면 4가지 핵심 구성 요소를 알아야 합니다:

```
[ECS 클러스터]
    │
    ├── [서비스 A]
    │       │
    │       ├── [태스크 1] ← 태스크 정의 A
    │       └── [태스크 2] ← 태스크 정의 A
    │
    └── [서비스 B]
            │
            └── [태스크 3] ← 태스크 정의 B
```

1. **클러스터(Cluster)**: 컨테이너를 실행하는 논리적 그룹. 하나의 프로젝트나 환경(개발, 운영 등)을 위한 공간입니다.

2. **태스크 정의(Task Definition)**: 컨테이너를 어떻게 실행할지 정의한 설계도(JSON 파일). "어떤 이미지를 사용할지, CPU와 메모리를 얼마나 할당할지, 어떤 포트를 열지" 등을 지정합니다.

3. **태스크(Task)**: 태스크 정의를 기반으로 실제 실행 중인 컨테이너(들)의 인스턴스. 하나의 태스크에 1개 이상의 컨테이너가 포함될 수 있습니다.

4. **서비스(Service)**: 지정한 수의 태스크가 항상 실행되도록 관리하는 구성. "이 태스크를 항상 3개 유지해줘"라고 설정하면, 하나가 죽어도 자동으로 새 태스크를 시작합니다.

**ECS의 시작 유형 (Launch Type)**

ECS에서 컨테이너를 실행할 때, 컨테이너가 실행되는 인프라를 선택할 수 있습니다:

| 시작 유형 | 설명 | 특징 |
|-----------|------|------|
| **EC2 시작 유형** | 직접 관리하는 EC2 인스턴스 위에서 컨테이너 실행 | 서버를 직접 관리, 세밀한 제어 가능, 비용 최적화 가능 |
| **Fargate 시작 유형** | AWS가 인프라를 관리, 서버리스로 컨테이너 실행 | 서버 관리 불필요, 태스크 단위 과금, 운영 부담 최소화 |

> 🔗 **연결**: Chapter 5에서 EC2 인스턴스를 직접 관리하는 것과 Lambda처럼 서버 관리가 필요 없는 서버리스의 차이를 배웠습니다. ECS의 EC2 시작 유형은 "서버를 직접 관리"하는 방식이고, Fargate 시작 유형은 "서버리스" 방식입니다. Fargate에 대해서는 Section 9.4에서 자세히 다룹니다.

**ECR — 컨테이너 이미지 저장소**

ECR은 Docker Hub와 같은 역할을 하는 AWS의 컨테이너 이미지 레지스트리입니다.

- **Private Registry**: 이미지를 비공개로 저장하여 조직 내에서만 접근 가능
- **ECS와 자연스러운 통합**: ECS 태스크 정의에서 ECR의 이미지를 직접 참조 가능
- **이미지 스캔**: 저장된 이미지의 보안 취약점을 자동으로 검사
- **수명 주기 정책**: 오래된 이미지를 자동으로 정리하여 스토리지 비용 절감

```
[개발자 PC]                    [AWS Cloud]
    │                              │
    │  docker push               ┌──────────┐
    │ ─────────────────────────→ │   ECR    │
    │                            │ (이미지   │
    │                            │  저장소)  │
    │                            └────┬─────┘
    │                                 │ 이미지 pull
    │                            ┌────▼─────┐
    │                            │   ECS    │
    │                            │(컨테이너  │
    │                            │  실행)    │
    │                            └──────────┘
```

### 상세 예시

> 🔍 **예시 9.2.1: 일상적 비유 — 식당 프랜차이즈**
>
> **상황**: 인기 있는 김밥 레시피(컨테이너 이미지)를 여러 지점(서버)에서 동일하게 제공하고 싶습니다.
> **설명**:
> - **ECR**: 본사의 레시피 금고. 표준 레시피를 안전하게 보관하고, 인가된 지점만 열람 가능
> - **ECS 태스크 정의**: "이 레시피로, 밥 200g, 김 1장을 사용하여 제조하라"는 표준 매뉴얼
> - **ECS 태스크**: 실제로 김밥을 만드는 각 지점의 주방
> - **ECS 서비스**: "항상 3개 지점을 운영하고, 한 곳이 문을 닫으면 새 지점을 연다"는 운영 정책
> - **ECS 클러스터**: "서울 지역 지점 그룹"이라는 관리 단위
> **핵심 포인트**: ECS는 컨테이너의 "표준화된 배포와 운영 관리"를 자동화합니다.

> 🔍 **예시 9.2.2: 전문적 예시 — 웹 서비스 배포**
>
> **상황**: 전자상거래 회사가 주문 처리 서비스를 ECS로 운영합니다.
> **설명**:
> 1. 개발팀이 주문 서비스 코드를 Docker 이미지로 빌드 → ECR에 push
> 2. ECS 태스크 정의에 "ECR의 order-service:v2.1 이미지, CPU 256, 메모리 512MB, 포트 8080" 지정
> 3. ECS 서비스를 "항상 태스크 3개 유지"로 설정
> 4. 블랙 프라이데이에 트래픽이 급증하면 → Auto Scaling이 태스크를 3개 → 10개로 확장
> 5. 태스크 하나가 비정상 종료 → ECS가 자동으로 새 태스크를 시작하여 항상 지정된 수를 유지
> **핵심 포인트**: ECS는 컨테이너의 배포, 확장, 복구를 자동화하여 안정적인 서비스 운영을 지원합니다.

### 미니 실습

> 🧪 **Mini Lab 9.2: ECS 클러스터 생성 및 탐색**
>
> **목표**: ECS 클러스터를 생성하고 ECS 콘솔의 구성 요소를 직접 확인한다
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (클러스터 생성 자체는 무료, 컨테이너를 실행하지 않으면 비용 없음)
>
> **Step 1**: AWS 콘솔에 로그인 → 상단 검색창에 "ECS" 입력 → **Elastic Container Service** 클릭
>
> **Step 2**: 왼쪽 메뉴에서 **Clusters(클러스터)** 클릭 → **Create cluster(클러스터 생성)** 버튼 클릭
>
> **Step 3**: 클러스터 설정
> - **Cluster name(클러스터 이름)**: `my-first-cluster` 입력
> - **Infrastructure(인프라)**: **AWS Fargate (serverless)** 가 기본 선택되어 있는지 확인 (체크 상태 유지)
> - 나머지 설정은 기본값 유지
>
> **Step 4**: **Create(생성)** 버튼 클릭 → 클러스터가 생성됩니다 (약 10~30초 소요)
>
> **Step 5**: 생성된 `my-first-cluster`를 클릭하여 내부 탐색
> - **Services(서비스)** 탭: 현재 비어 있음 — 아직 서비스를 배포하지 않았기 때문
> - **Tasks(태스크)** 탭: 현재 비어 있음 — 아직 태스크를 실행하지 않았기 때문
> - **Infrastructure(인프라)** 탭: Fargate가 활성화되어 있는 것을 확인
>
> **Step 6**: 왼쪽 메뉴에서 **Task definitions(태스크 정의)** 클릭 → 태스크 정의가 어떤 형태인지 확인 (현재 비어 있음)
>
> **확인**: `my-first-cluster`가 **Active(활성)** 상태로 표시되고, Services와 Tasks 탭이 존재하는 것을 확인할 수 있습니다.
>
> **🧹 정리**: 클러스터 자체는 비용이 발생하지 않으므로 지금 삭제하지 않아도 됩니다. 다만 더 이상 사용하지 않을 경우: `my-first-cluster` 선택 → **Delete cluster(클러스터 삭제)** → 확인란에 `delete my-first-cluster` 입력 → **Delete** 클릭

> ⚠️ **주의**
>
> - **ECS와 EC2를 혼동하지 마세요**: EC2는 가상 서버(인스턴스) 서비스이고, ECS는 컨테이너 관리(오케스트레이션) 서비스입니다. ECS가 EC2 위에서 컨테이너를 실행할 수 있지만, ECS 자체가 서버는 아닙니다.
> - **태스크 정의 vs 태스크**: 태스크 정의는 "설계도"이고, 태스크는 "실제 실행 중인 것"입니다. 하나의 태스크 정의로 여러 태스크를 실행할 수 있습니다 (AMI와 EC2 인스턴스의 관계와 유사합니다).
> - **ECR 저장 비용**: ECR에 이미지를 저장하면 스토리지 비용이 발생합니다. Free Tier에서 매월 500MB까지 무료이므로, 실습용 이미지는 이 범위 내에서 충분합니다.

### Section 요약

- Amazon ECS는 AWS의 자체 컨테이너 오케스트레이션 서비스로, 컨테이너의 배포·확장·복구를 자동화한다
- ECS의 핵심 구성 요소: 클러스터 → 서비스 → 태스크 → 태스크 정의
- ECS는 EC2 시작 유형(서버 직접 관리)과 Fargate 시작 유형(서버리스)을 선택할 수 있다
- Amazon ECR은 컨테이너 이미지를 저장하는 AWS 관리형 레지스트리로, ECS와 자연스럽게 통합된다

## 9.3 Amazon EKS — Kubernetes 기반 컨테이너 관리

> 💡 **한 줄 요약**: Amazon EKS는 오픈소스 Kubernetes를 AWS에서 쉽게 실행할 수 있게 해주는 완전 관리형 서비스이다.

### 직관적 이해

Section 9.2에서 ECS가 AWS의 자체 오케스트레이션 서비스라고 배웠습니다. 그런데 컨테이너 오케스트레이션 세계에는 또 다른 거대한 플레이어가 있습니다 — 바로 **Kubernetes(쿠버네티스)** 입니다.

비유하면, ECS는 "삼성 스마트폰의 운영체제(One UI)"와 같습니다. 삼성이 만들었고, 삼성 기기에서 최적화되어 작동합니다. 반면 Kubernetes는 "Android 오픈소스"와 같습니다. Google이 처음 만들었지만, 이제는 전 세계 누구나 사용하고 기여하는 오픈소스 표준이 되었습니다. 삼성, LG, 샤오미 등 어느 제조사에서든 사용할 수 있죠.

**왜 EKS가 필요한가?**

많은 기업이 이미 Kubernetes를 사용하고 있거나, 특정 클라우드에 종속되지 않기 위해(멀티 클라우드 전략) Kubernetes를 선택합니다. 하지만 Kubernetes를 직접 설치하고 운영하는 것은 매우 복잡합니다. Amazon EKS는 이 복잡한 설치와 운영을 AWS가 대신 처리해 줍니다.

### 핵심 개념 설명

> 📝 **Kubernetes(쿠버네티스, K8s)**: Google이 개발하고 오픈소스로 공개한 컨테이너 오케스트레이션 플랫폼. 컨테이너의 배포, 확장, 관리를 자동화하는 업계 표준이다. "K8s"라는 약칭은 K와 s 사이에 8글자가 있다는 뜻이다.

> 📝 **Amazon EKS(Elastic Kubernetes Service)**: AWS에서 Kubernetes를 쉽게 실행할 수 있도록 하는 완전 관리형 서비스. Kubernetes의 컨트롤 플레인(관리 영역)을 AWS가 운영하고, 사용자는 컨테이너 배포에 집중할 수 있다.

**ECS vs EKS — 언제 무엇을 선택하는가?**

| 비교 항목 | Amazon ECS | Amazon EKS |
|-----------|-----------|-----------|
| **개발사** | AWS 자체 개발 | 오픈소스 Kubernetes 기반 |
| **학습 난이도** | 상대적으로 쉬움 | 상대적으로 어려움 (Kubernetes 학습 필요) |
| **AWS 통합** | AWS 서비스와 깊이 통합 | AWS 서비스와 통합 + Kubernetes 생태계 |
| **이식성** | AWS에 종속 | 다른 클라우드(GCP, Azure)로 이전 용이 |
| **적합한 경우** | AWS만 사용, 빠르게 시작하고 싶은 경우 | 멀티 클라우드, 이미 K8s 사용 중인 경우 |
| **비용** | 추가 비용 없음 (인프라 비용만) | 클러스터당 시간당 $0.10 + 인프라 비용 |

> 💬 **참고**: CCP 시험에서는 ECS와 EKS의 깊은 기술적 차이보다는, "ECS는 AWS 자체 오케스트레이션, EKS는 Kubernetes 기반 오케스트레이션"이라는 구분과 각각의 사용 시나리오를 이해하는 것이 중요합니다.

**EKS도 Fargate를 지원한다**

EKS 역시 ECS처럼 두 가지 방식으로 컨테이너를 실행할 수 있습니다:
- **EC2 노드**: Kubernetes 워커 노드로 EC2 인스턴스를 직접 관리
- **Fargate**: 서버리스로 컨테이너 실행 (노드 관리 불필요)

### 상세 예시

> 🔍 **예시 9.3.1: 일상적 비유 — 국제 표준 vs 자체 규격**
>
> **상황**: 두 건설 회사가 건물을 짓습니다.
> **설명**:
> - **A사 (ECS 방식)**: 자체 설계 규격으로 건물을 짓습니다. 자사 건설 장비에 최적화되어 빠르고 효율적이지만, A사 장비로만 유지보수 가능합니다.
> - **B사 (EKS/Kubernetes 방식)**: 국제 건축 표준(ISO)에 따라 건물을 짓습니다. 어느 회사의 장비로든 유지보수 가능하고, 다른 나라에서도 같은 방식으로 건물을 지을 수 있습니다. 다만, 국제 표준을 익히는 데 시간이 더 걸립니다.
> **핵심 포인트**: ECS는 AWS에 최적화된 자체 솔루션, EKS(Kubernetes)는 업계 표준 기반 솔루션입니다. 선택은 조직의 상황에 따라 달라집니다.

> 🔍 **예시 9.3.2: 전문적 예시 — 멀티 클라우드 전략**
>
> **상황**: 대기업 C사는 AWS와 Google Cloud를 동시에 사용하는 멀티 클라우드 전략을 채택했습니다.
> **설명**: C사는 Amazon EKS(AWS)와 Google GKE(Google Cloud) 위에서 동일한 Kubernetes 설정을 사용하여 서비스를 운영합니다. 개발팀은 Kubernetes라는 하나의 표준만 익히면 양쪽 클라우드에서 컨테이너를 관리할 수 있습니다. 만약 한쪽 클라우드에 장애가 발생해도 다른 쪽으로 트래픽을 전환할 수 있습니다.
> **핵심 포인트**: Kubernetes의 최대 장점은 이식성입니다. EKS를 선택하면 AWS에서 Kubernetes를 쉽게 운영하면서도, 필요시 다른 클라우드로의 이전 가능성을 열어둘 수 있습니다.

> ⚠️ **주의**
>
> - **EKS = Kubernetes를 "쉽게" 운영해주는 것이지, Kubernetes 자체를 "쉽게" 만들어주는 것은 아닙니다**: Kubernetes 자체의 개념(Pod, Deployment, Service, Namespace 등)은 여전히 배워야 합니다. EKS가 해주는 것은 Kubernetes 클러스터의 설치, 업그레이드, 패치 등의 운영 부담을 줄여주는 것입니다.
> - **CCP 시험 수준에서는**: "EKS = Kubernetes 관리형 서비스"라는 것과, "Kubernetes를 사용하고 싶거나, 멀티 클라우드를 원하면 EKS를 선택"한다는 것을 이해하면 충분합니다.

### Section 요약

- Kubernetes(K8s)는 Google이 개발한 오픈소스 컨테이너 오케스트레이션 플랫폼으로, 업계 표준이다
- Amazon EKS는 Kubernetes를 AWS에서 완전 관리형으로 실행할 수 있게 해주는 서비스이다
- ECS는 AWS 전용 솔루션(간단, AWS 최적화), EKS는 Kubernetes 표준 기반(이식성, 멀티 클라우드)이다
- EKS도 EC2 노드와 Fargate 중 선택하여 컨테이너를 실행할 수 있다

## 9.4 AWS Fargate와 App Runner — 서버리스 컨테이너 실행 환경, 간소화된 컨테이너 배포

> 💡 **한 줄 요약**: Fargate는 서버 관리 없이 컨테이너를 실행하는 서버리스 컴퓨팅 엔진이고, App Runner는 컨테이너 배포를 극단적으로 간소화한 서비스이다.

### 직관적 이해

지금까지 ECS와 EKS를 배우면서, 컨테이너를 "어디서" 실행할지에 대해 EC2 인스턴스를 직접 관리하는 방식을 살펴봤습니다. 하지만 이렇게 하면 여전히 서버(EC2)를 관리해야 합니다 — 패치, 용량 계획, 보안 업데이트 등을 신경 써야 하죠.

비유하면:
- **EC2 시작 유형**: 직접 차를 사서 운전하는 것. 정비, 보험, 주차 모두 내가 관리
- **Fargate**: 택시를 타는 것. 목적지만 말하면 됨. 차량 관리는 택시 회사(AWS)가 담당
- **App Runner**: 택시 앱에서 버튼 한 번으로 호출하는 것. 더 간편하지만, 세밀한 경로 선택은 어려움

> 🔗 **연결**: Chapter 5에서 배운 AWS Lambda를 떠올려 봅시다. Lambda는 "코드만 올리면 AWS가 서버를 알아서 관리해주는" 서버리스 서비스였습니다. Fargate는 Lambda의 이 "서버리스" 철학을 컨테이너에 적용한 것입니다.

### 핵심 개념 설명

> 📝 **AWS Fargate**: Amazon ECS 및 Amazon EKS와 함께 사용할 수 있는 서버리스 컴퓨팅 엔진. 서버를 프로비저닝하거나 관리할 필요 없이, 컨테이너를 실행할 수 있다. 실행한 컨테이너의 CPU와 메모리 사용량에 따라 과금된다.

> 📝 **AWS App Runner**: 컨테이너화된 웹 애플리케이션이나 API를 빠르게 배포할 수 있는 완전 관리형 서비스. 소스 코드 또는 컨테이너 이미지를 제공하면, 빌드, 배포, 확장, 로드 밸런싱까지 자동으로 처리한다.

**Fargate — EC2 시작 유형과의 비교**

| 비교 항목 | EC2 시작 유형 | Fargate |
|-----------|--------------|---------|
| **서버 관리** | 직접 관리 (패치, 보안 등) | AWS가 관리 (서버리스) |
| **용량 계획** | EC2 인스턴스 수와 크기를 계획 | 태스크별 CPU/메모리만 지정 |
| **비용 모델** | EC2 인스턴스 시간당 과금 | 태스크의 CPU/메모리 사용량 기반 |
| **확장** | EC2 Auto Scaling 설정 필요 | 태스크 수만 조절하면 됨 |
| **적합한 경우** | 비용 최적화, GPU 필요, 세밀한 제어 | 운영 부담 최소화, 빠른 시작 |

**Fargate가 해결하는 문제**

```
[ EC2 시작 유형을 사용할 때 ]

개발자가 고민해야 할 것:
✗ EC2 인스턴스 유형 선택 (t3.medium? m5.large?)
✗ 인스턴스 수 결정 (몇 대가 필요하지?)
✗ OS 패치 및 보안 업데이트
✗ 컨테이너 배치 최적화 (어떤 인스턴스에 어떤 컨테이너?)
✗ 인스턴스 모니터링 및 교체

[ Fargate를 사용할 때 ]

개발자가 고민해야 할 것:
✓ 컨테이너에 필요한 CPU와 메모리 양만 지정
→ 나머지는 모두 AWS가 처리!
```

**App Runner — 가장 간단한 컨테이너 배포**

App Runner는 Fargate보다 한 단계 더 추상화된 서비스입니다:

```
[개발자가 해야 할 것]

ECS + Fargate:
  이미지 준비 → 태스크 정의 작성 → 서비스 구성 → 네트워크 설정 → 로드 밸런서 구성 → 배포

App Runner:
  이미지 또는 소스 코드 제공 → 배포 (끝!)
```

App Runner가 자동으로 처리하는 것들:
- 빌드 및 배포
- HTTPS 인증서 설정
- 로드 밸런싱
- 오토 스케일링 (트래픽에 따라 자동 확장/축소)
- 헬스 체크 및 자동 복구

**언제 무엇을 선택하는가?**

```
"서버를 직접 관리하고 싶다 / GPU가 필요하다"
    → ECS 또는 EKS + EC2 시작 유형

"서버 관리는 싫지만, 세밀한 컨테이너 설정이 필요하다"
    → ECS 또는 EKS + Fargate

"웹 앱/API를 가장 빠르고 간단하게 배포하고 싶다"
    → App Runner

"코드만 실행하면 된다 (컨테이너가 필요 없다)"
    → Lambda
```

### 상세 예시

> 🔍 **예시 9.4.1: 일상적 비유 — 주거 형태와 관리 수준**
>
> **상황**: 서울에서 살 곳을 고르는 상황을 생각해 봅시다.
> **설명**:
> - **EC2 시작 유형 = 자가 주택**: 내 집이라 자유롭게 꾸밀 수 있지만, 지붕 수리, 정원 관리, 보일러 정비 등 모두 직접 해야 합니다.
> - **Fargate = 아파트**: 건물 관리(보안, 엘리베이터, 주차장)는 관리사무소가 처리합니다. 집 안의 인테리어(컨테이너 설정)는 직접 합니다.
> - **App Runner = 호텔**: 체크인만 하면 됩니다. 청소, 시설 관리, 조식까지 모두 포함. 다만, 벽지를 바꾸거나 가구를 재배치하는 것은 어렵습니다.
> **핵심 포인트**: 관리 편의성과 제어 수준은 트레이드오프 관계입니다. 필요에 맞는 수준을 선택하면 됩니다.

> 🔍 **예시 9.4.2: 전문적 예시 — 스타트업의 서비스 선택**
>
> **상황**: 3인 스타트업이 MVP(최소 기능 제품) 웹 API를 빠르게 출시해야 합니다.
> **설명**: 소규모 팀에서 인프라 관리에 시간을 쓸 여유가 없습니다. App Runner를 선택하면 Docker 이미지를 ECR에 올리고 App Runner에 연결하기만 하면, HTTPS, 로드 밸런싱, 오토 스케일링이 자동으로 설정됩니다. 나중에 서비스가 성장하여 더 세밀한 제어가 필요해지면, ECS + Fargate로 전환할 수 있습니다.
> **핵심 포인트**: App Runner는 빠른 시작과 간편한 운영이 최우선인 경우에 적합합니다. 복잡한 마이크로서비스 아키텍처보다는 단일 웹 앱이나 API 서비스에 어울립니다.

> 🔬 **Deep Dive: Fargate의 과금 구조**
>
> _이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다._
>
> Fargate는 태스크가 실행되는 동안 사용한 vCPU와 메모리에 대해 초 단위로 과금됩니다 (최소 1분).
>
> - **vCPU**: 시간당 약 $0.04048 (서울 리전 기준)
> - **메모리**: GB당 시간당 약 $0.004445
>
> 예를 들어, 0.25 vCPU / 0.5GB 메모리 태스크를 1시간 실행하면:
> - vCPU: $0.04048 × 0.25 = $0.01012
> - 메모리: $0.004445 × 0.5 = $0.00222
> - **합계: 약 $0.012/시간**
>
> EC2 시작 유형에서 t3.micro(2 vCPU, 1GB)를 사용하면 시간당 약 $0.013이지만, 컨테이너가 자원을 다 쓰지 않아도 전체 인스턴스 비용을 지불합니다. Fargate는 실제 사용한 만큼만 과금되므로, 간헐적 워크로드에서 비용 효율적일 수 있습니다.

> ⚠️ **주의**
>
> - **Fargate ≠ Lambda**: 둘 다 서버리스이지만, Lambda는 "코드(함수)"를 실행하고, Fargate는 "컨테이너"를 실행합니다. Lambda는 최대 15분 실행 제한이 있고, Fargate는 장시간 실행되는 컨테이너에 적합합니다.
> - **App Runner는 웹 앱/API 전용**: App Runner는 HTTP(S) 요청을 처리하는 웹 애플리케이션과 API에 최적화되어 있습니다. 배치 처리나 백그라운드 작업에는 ECS + Fargate가 더 적합합니다.

### Section 요약

- AWS Fargate는 ECS/EKS와 함께 사용하는 서버리스 컴퓨팅 엔진으로, 서버 관리 없이 컨테이너를 실행한다
- Fargate는 태스크 단위로 CPU/메모리를 지정하고, 사용한 만큼만 과금된다
- AWS App Runner는 가장 간소화된 컨테이너 배포 서비스로, 소스 코드나 이미지만 제공하면 빌드·배포·확장을 자동 처리한다
- 선택 기준: EC2(세밀한 제어) → Fargate(서버리스 컨테이너) → App Runner(최대 간편함) → Lambda(함수 단위 서버리스)

## 9.5 Amazon API Gateway — RESTful API 생성 및 관리

> 💡 **한 줄 요약**: Amazon API Gateway는 개발자가 API를 쉽게 생성하고, 게시하고, 관리할 수 있게 해주는 완전 관리형 서비스이다.

### 직관적 이해

레스토랑에 갔다고 상상해 봅시다. 손님(클라이언트)이 직접 주방(백엔드 서비스)에 들어가서 요리를 주문하지는 않습니다. 대신 **웨이터(API Gateway)** 가 주문을 받아 주방에 전달하고, 완성된 요리를 손님에게 가져다줍니다.

웨이터는 단순히 주문을 전달하는 것만이 아닙니다:
- 메뉴에 없는 주문은 거부합니다 (요청 검증)
- VIP 손님과 일반 손님을 구분합니다 (인증/인가)
- 주문이 너무 많으면 대기열을 관리합니다 (트래픽 관리)
- 몇 번 주문이 들어왔는지 기록합니다 (모니터링)

이것이 바로 **API Gateway**가 하는 일입니다.

**왜 API Gateway가 필요한가?**

> 📝 **API(Application Programming Interface)**: 서로 다른 소프트웨어가 통신하기 위한 약속된 인터페이스. 특히 웹 API는 HTTP 프로토콜을 통해 데이터를 주고받는 방식을 정의한다.

현대의 애플리케이션은 프론트엔드(웹, 모바일 앱)와 백엔드(서버, 데이터베이스)가 분리되어 있습니다. 프론트엔드가 백엔드에 데이터를 요청할 때 API를 사용합니다. 그런데 백엔드 서비스가 여러 개로 나뉘어 있다면(마이크로서비스), 프론트엔드가 각 서비스의 주소를 모두 알아야 할까요? 그것은 복잡하고 보안에도 좋지 않습니다.

API Gateway는 **단일 진입점(Single Entry Point)** 역할을 합니다. 클라이언트는 API Gateway의 주소 하나만 알면 되고, API Gateway가 요청을 적절한 백엔드 서비스로 라우팅합니다.

### 핵심 개념 설명

> 📝 **Amazon API Gateway**: RESTful API와 WebSocket API를 생성, 게시, 유지 관리, 모니터링, 보안 관리할 수 있는 완전 관리형 서비스. 어떤 규모에서든 API를 생성하고, 백엔드 서비스(Lambda, EC2, 기타 AWS 서비스)에 대한 "정문" 역할을 한다.

**API Gateway의 핵심 기능**

```
[클라이언트]                    [API Gateway]                    [백엔드]
(웹, 모바일앱)                      │
     │                              │
     │  GET /users/123              │
     │ ───────────────────────→     │
     │                              │──→ 인증 확인 ✓
     │                              │──→ 요청 검증 ✓
     │                              │──→ 트래픽 제한 확인 ✓
     │                              │──→ 캐싱 확인
     │                              │
     │                              │  요청 전달
     │                              │ ───────────→  [Lambda 함수]
     │                              │                    또는
     │                              │               [EC2 인스턴스]
     │                              │                    또는
     │                              │               [ECS 컨테이너]
     │                              │ ←───────────  응답 반환
     │                              │
     │  { "name": "홍길동", ... }   │
     │ ←───────────────────────     │──→ 로깅/모니터링 기록
     │                              │
```

1. **API 생성 및 라우팅**: URL 경로별로 어떤 백엔드 서비스가 처리할지 지정
2. **인증 및 인가**: API 키, IAM 역할, Cognito 등을 통한 접근 제어
3. **트래픽 관리**: 요청 속도 제한(throttling)으로 백엔드 과부하 방지
4. **모니터링**: API 호출 횟수, 지연 시간, 오류율 등을 자동 기록
5. **API 버전 관리**: 여러 버전의 API를 동시에 운영 (v1, v2 등)

**API Gateway + Lambda = 서버리스 API**

API Gateway와 Lambda의 조합은 가장 대표적인 서버리스 패턴입니다:

> 🔗 **연결**: Chapter 5에서 Lambda는 "이벤트가 발생하면 코드를 실행하는 서버리스 컴퓨팅 서비스"라고 배웠습니다. API Gateway는 HTTP 요청이라는 이벤트를 Lambda에 전달하는 역할을 합니다.

```
[사용자 요청]                                          [응답]
     │                                                   ▲
     ▼                                                   │
┌──────────────┐     ┌──────────────┐     ┌─────────────┐
│ API Gateway  │────→│   Lambda     │────→│ DynamoDB    │
│ (요청 수신   │     │ (비즈니스    │     │ (데이터     │
│  및 라우팅)  │←────│  로직 실행)  │←────│  저장/조회) │
└──────────────┘     └──────────────┘     └─────────────┘
     서버리스             서버리스             서버리스
```

이 구성에서는 서버를 단 하나도 관리할 필요가 없습니다. 모든 것이 서버리스입니다.

### 상세 예시

> 🔍 **예시 9.5.1: 일상적 비유 — 건물의 안내 데스크**
>
> **상황**: 대형 오피스 빌딩에 방문한 상황입니다.
> **설명**: 빌딩 1층의 안내 데스크(API Gateway)에서 방문증을 발급받고(인증), 방문 목적에 맞는 층과 사무실로 안내받습니다(라우팅). 안내 데스크는 출입이 허가된 사람만 통과시키고(인가), 방문 기록을 남기며(로깅), 동시에 너무 많은 사람이 몰리면 대기하게 합니다(트래픽 관리).
> **핵심 포인트**: API Gateway는 백엔드 서비스의 "정문"으로, 보안·라우팅·관리를 일원화합니다.

> 🔍 **예시 9.5.2: 전문적 예시 — 모바일 앱의 API 구성**
>
> **상황**: 음식 배달 앱의 백엔드 API를 구성합니다.
> **설명**:
> - `GET /restaurants` → Lambda 함수 A (레스토랑 목록 조회)
> - `POST /orders` → Lambda 함수 B (주문 생성)
> - `GET /orders/{id}` → Lambda 함수 C (주문 상태 조회)
> - `POST /payments` → ECS 서비스 (결제 처리 — 장시간 실행)
>
> API Gateway가 URL 경로에 따라 적절한 백엔드로 요청을 라우팅합니다. 모바일 앱은 API Gateway의 주소(`https://api.delivery.com`)만 알면 됩니다.
> **핵심 포인트**: API Gateway는 다양한 백엔드 서비스를 하나의 통합된 API 인터페이스 뒤에 숨길 수 있습니다.

### 미니 실습

> 🧪 **Mini Lab 9.5: API Gateway + Lambda 서버리스 API 만들기**
>
> **목표**: API Gateway와 Lambda를 연동하여, HTTP 요청을 보내면 응답을 반환하는 간단한 서버리스 API를 만든다
> **예상 소요 시간**: 15분

> **Step 1**: AWS 콘솔 상단 검색창에 "Lambda" 입력 → **Lambda** 클릭
>
> **Step 2**: **Create function(함수 생성)** 클릭

> import json
>
> def lambda_handler(event, context):
>     return {
>         'statusCode': 200,
>         'headers': {'Content-Type': 'application/json'},
>         'body': json.dumps({
>             'message': 'Hello from Lambda!',

> - **API endpoint type(API 엔드포인트 유형)**: **Regional(리전)** 
> - **Create API(API 생성)** 클릭
>
> **Step 7**: **Create method(메서드 생성)** 클릭
> - **Method type(메서드 유형)**: **GET** 선택
> - **Integration type(통합 유형)**: **Lambda function(Lambda 함수)** 선택
> - **Lambda function**: `my-api-handler` 선택
> - **Create method(메서드 생성)** 클릭
>
> **Step 8**: **Deploy API(API 배포)** 클릭
> - **Stage name(스테이지 이름)**: `test` 입력
> - **Deploy(배포)** 클릭
>
> **Step 9**: 배포 후 표시되는 **Invoke URL(호출 URL)** 을 복사합니다

>
> **Step 10**: 웹 브라우저 주소창에 복사한 URL을 붙여넣고 Enter

> ```

>

> 1. **API Gateway**: `my-first-api` 선택 → **Delete API(API 삭제)** → API 이름 입력하여 확인

> ⚠️ **주의**
>
> - **API Gateway ≠ 백엔드 서비스 자체**: API Gateway는 요청을 받아 백엔드로 전달하는 "통로"입니다. 실제 비즈니스 로직은 Lambda, EC2, ECS 등의 백엔드에서 실행됩니다.
> - **비용 모니터링**: API Gateway는 Free Tier에서 12개월간 월 100만 건의 REST API 호출이 무료입니다. 이를 초과하면 요청 건당 과금되므로, 테스트용 API를 외부에 공개하지 않도록 주의하세요.

### Section 요약

- Amazon API Gateway는 API를 생성·관리하는 완전 관리형 서비스로, 백엔드 서비스의 "정문" 역할을 한다
- 인증/인가, 트래픽 관리, 라우팅, 모니터링 등을 제공한다
- API Gateway + Lambda 조합은 가장 대표적인 서버리스 API 패턴이다
- API Gateway는 Lambda뿐 아니라 EC2, ECS, 다른 AWS 서비스와도 연동할 수 있다

## 9.6 AWS Step Functions과 서버리스 아키텍처 — 워크플로 오케스트레이션, 서버리스 패턴 설계

> 💡 **한 줄 요약**: AWS Step Functions는 여러 AWS 서비스를 시각적 워크플로로 연결하고, 서버리스 아키텍처의 복잡한 비즈니스 로직을 조율하는 오케스트레이션 서비스이다.

### 직관적 이해

온라인 쇼핑몰에서 주문이 들어왔을 때 일어나는 과정을 생각해 봅시다:

1. 재고 확인 → 2. 결제 처리 → 3. 재고 차감 → 4. 배송 요청 → 5. 확인 이메일 발송

각 단계를 Lambda 함수로 만들 수 있습니다. 하지만 문제가 있습니다:
- 2단계(결제)가 실패하면 어떻게 하지?
- 3단계에서 오류가 나면 2단계의 결제를 취소해야 하는데?
- 이 모든 흐름을 어떻게 관리하지?

하나의 Lambda 함수 안에 이 모든 로직을 넣으면 코드가 복잡해지고, 오류 처리가 매우 어려워집니다. **Step Functions**는 이 문제를 해결합니다 — 각 단계를 독립된 서비스로 두고, 단계 간의 흐름(순서, 조건 분기, 오류 처리, 재시도 등)을 시각적으로 정의할 수 있습니다.

비유하면, Step Functions는 **연극의 대본(Script)** 과 같습니다. 각 배우(Lambda 함수, ECS 태스크 등)가 언제 등장하고, 어떤 대사를 하고, 다음에 누가 나오는지를 대본이 정해줍니다. 배우(서비스) 하나하나가 뛰어나도, 대본 없이는 연극이 될 수 없습니다.

### 핵심 개념 설명

> 📝 **AWS Step Functions**: 시각적 워크플로를 사용하여 여러 AWS 서비스를 조율하는 서버리스 오케스트레이션 서비스. 워크플로의 각 단계(State)를 정의하고, 단계 간의 순서, 조건 분기, 병렬 실행, 오류 처리를 관리한다.

> 📝 **상태 머신(State Machine)**: Step Functions에서 워크플로를 정의하는 단위. 하나 이상의 상태(State)로 구성되며, 상태 간의 전환 규칙을 정의한다.

**Step Functions의 핵심 개념**

```
[상태 머신 (State Machine)]
    │
    ▼
┌─────────────┐
│  시작 상태   │
└──────┬──────┘
       │
       ▼
┌─────────────┐     성공     ┌─────────────┐
│ 재고 확인    │ ──────────→ │ 결제 처리    │
│ (Lambda)    │             │ (Lambda)    │
└──────┬──────┘             └──────┬──────┘
       │ 재고 없음                  │
       ▼                          ▼ 
┌─────────────┐            ┌─────────────┐
│ 품절 알림    │            │ 배송 요청    │
│ (SNS)       │            │ (Lambda)    │
└─────────────┘            └──────┬──────┘
                                  │
                                  ▼
                           ┌─────────────┐
                           │ 확인 이메일  │
                           │ (SES)       │
                           └─────────────┘
```

Step Functions의 상태(State) 유형:

| 상태 유형 | 역할 | 예시 |
|-----------|------|------|
| **Task** | 작업 수행 (Lambda 호출, ECS 태스크 실행 등) | Lambda 함수로 결제 처리 |
| **Choice** | 조건에 따른 분기 | 재고가 있으면 A로, 없으면 B로 |
| **Parallel** | 여러 작업을 동시에 실행 | 이메일 발송과 SMS 발송을 동시에 |
| **Wait** | 지정된 시간만큼 대기 | 5분 후 다음 단계 실행 |
| **Pass** | 입력을 그대로 출력으로 전달 | 데이터 변환 또는 테스트용 |
| **Succeed/Fail** | 워크플로 성공/실패 종료 | 최종 결과 반환 |

**서버리스 아키텍처 패턴 종합**

이 Chapter에서 배운 서비스들을 조합하여 완전한 서버리스 아키텍처를 구성할 수 있습니다:

```
[완전한 서버리스 아키텍처 예시]

[사용자] → [API Gateway] → [Lambda] → [DynamoDB]
                              │
                              ▼
                        [Step Functions]
                              │
                    ┌─────────┼─────────┐
                    ▼         ▼         ▼
              [Lambda A] [Lambda B] [Lambda C]
                    │         │         │
                    ▼         ▼         ▼
              [S3 저장]  [SNS 알림]  [SQS 큐]
```

서버리스 아키텍처의 핵심 장점:
1. **서버 관리 불필요**: 모든 인프라를 AWS가 관리
2. **자동 확장**: 트래픽에 따라 자동으로 스케일 업/다운
3. **사용한 만큼만 과금**: 요청이 없으면 비용도 없음
4. **빠른 개발**: 인프라 구성 대신 비즈니스 로직에 집중

> 💬 **참고**: SQS(메시지 큐), SNS(알림 서비스)는 Chapter 10에서 자세히 다룹니다. 여기서는 "서버리스 아키텍처에서 서비스 간 연결에 사용되는 도구"라는 정도로 이해하면 됩니다.

### 상세 예시

> 🔍 **예시 9.6.1: 일상적 비유 — 자동 택배 분류 시스템**
>
> **상황**: 택배 물류센터의 자동 분류 시스템을 생각해 봅시다.
> **설명**: 택배가 도착하면(이벤트 발생):
> 1. 바코드 스캔 → 무게 측정 (Task)
> 2. 무게가 30kg 이상이면 대형 택배 라인으로, 미만이면 일반 라인으로 (Choice)
> 3. 일반 라인에서 지역별로 동시에 분류 (Parallel)
> 4. 분류 완료 후 배송 차량에 적재 (Task)
> 
> 이 전체 프로세스를 관리하는 컨트롤러가 Step Functions입니다. 각 단계(스캔, 분류, 적재)는 독립적인 장비(서비스)이지만, 전체 흐름을 조율하는 시스템이 필요합니다.
> **핵심 포인트**: Step Functions는 여러 독립적인 서비스(Lambda, ECS 등)를 하나의 비즈니스 프로세스로 엮어줍니다.

> 🔍 **예시 9.6.2: 전문적 예시 — 이미지 처리 파이프라인**
>
> **상황**: 사용자가 프로필 사진을 업로드하면 자동으로 처리하는 시스템을 구축합니다.
> **설명**: Step Functions 워크플로:
> 1. **Task**: Lambda 함수가 S3에 업로드된 원본 이미지를 가져옴
> 2. **Parallel**: 3가지 작업을 동시에 실행
>    - Lambda A: 썸네일(작은 이미지) 생성 → S3에 저장
>    - Lambda B: 이미지에서 부적절한 콘텐츠 검출 (Amazon Rekognition)
>    - Lambda C: 이미지 메타데이터를 DynamoDB에 저장
> 3. **Choice**: 부적절한 콘텐츠가 감지되었는가?
>    - Yes → 이미지 삭제 + 관리자 알림(SNS)
>    - No → 프로필 업데이트 완료
>
> **핵심 포인트**: Step Functions를 사용하면 복잡한 비즈니스 로직을 시각적으로 설계하고, 오류 처리와 재시도를 자동화할 수 있습니다.

### 미니 실습

> 🧪 **Mini Lab 9.6: Step Functions 워크플로 탐색**
>
> **목표**: Step Functions 콘솔에서 워크플로가 어떻게 구성되는지 시각적으로 확인한다
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (매월 4,000회 상태 전환 무료)
>
> **Step 1**: AWS 콘솔 상단 검색창에 "Step Functions" 입력 → **Step Functions** 클릭
>
> **Step 2**: **Create state machine(상태 머신 생성)** 클릭
>
> **Step 3**: 템플릿 선택 화면에서 **Blank(빈 템플릿)** 을 선택합니다.
>
> **Step 4**: Workflow Studio(워크플로 스튜디오)가 열립니다. 왼쪽 패널에서 사용 가능한 상태 유형들을 확인합니다:
> - **Flow** 탭: Choice, Parallel, Wait, Map 등 흐름 제어 상태
> - **Actions** 탭: Lambda Invoke, DynamoDB, SNS, SQS 등 AWS 서비스 호출
>
> **Step 5**: **Actions** 탭에서 **AWS Lambda: Invoke** 를 드래그하여 캔버스의 "Start" 아래에 놓습니다
>
> **Step 6**: **Flow** 탭에서 **Choice** 를 드래그하여 Lambda 아래에 놓습니다
>
> **Step 7**: 상단의 **Config** 모드와 **Code** 모드를 전환하며, 시각적 워크플로와 JSON 정의가 어떻게 대응되는지 확인합니다
>
> **확인**: 시각적 워크플로에서 Lambda → Choice 순서로 연결된 흐름이 보이고, 오른쪽 JSON 코드에 "States" 정의가 포함되어 있는 것을 확인합니다.
>
> **🧹 정리**: 상태 머신을 저장(생성)하지 않았다면 비용이 발생하지 않습니다. 페이지를 그냥 닫아도 됩니다. 만약 저장했다면: **State machines** 목록에서 해당 상태 머신 선택 → **Delete(삭제)** 클릭

> ⚠️ **주의**
>
> - **Step Functions ≠ Lambda**: Lambda는 개별 함수(작업)를 실행하고, Step Functions는 여러 Lambda 함수(또는 다른 서비스)의 실행 순서와 흐름을 관리합니다. "Lambda가 연주자라면, Step Functions는 지휘자"입니다.
> - **Express vs Standard**: Step Functions에는 두 가지 유형이 있습니다. Standard(최대 1년 실행, 상태별 과금)와 Express(최대 5분, 실행 시간/횟수 기반 과금). CCP 시험에서는 이 구분까지 깊이 묻지 않지만, "짧은 고빈도 워크플로에는 Express"라는 점을 알아두면 좋습니다.

### Section 요약

- AWS Step Functions는 여러 AWS 서비스를 시각적 워크플로로 연결하는 서버리스 오케스트레이션 서비스이다
- 상태 머신(State Machine)으로 워크플로를 정의하며, Task, Choice, Parallel 등의 상태 유형을 사용한다
- 서버리스 아키텍처(API Gateway + Lambda + Step Functions + DynamoDB 등)는 서버 관리 없이 자동 확장되고 사용한 만큼만 과금된다
- 복잡한 비즈니스 프로세스의 흐름 제어, 오류 처리, 재시도를 자동화할 수 있다

- **Section 9.6**: Step Functions는 여러 서비스의 실행 흐름을 시각적으로 조율하고, 서버리스 아키텍처의 복잡한 로직을 관리한다

## 자기 점검 질문 (Self-Check Questions)

**1. 컨테이너와 가상 머신의 가장 큰 차이점은 무엇인가?**

<details>
<summary>정답 확인</summary>

컨테이너는 **호스트 OS의 커널을 공유**하면서 프로세스 수준으로 격리되는 반면, 가상 머신은 **각각 별도의 OS**를 가집니다. 이 때문에 컨테이너는 가상 머신보다 가볍고(MB vs GB), 시작이 빠르며(초 vs 분), 자원 효율성이 높습니다.
</details>

**2. (O/X) Amazon ECS와 Amazon EKS는 동일한 기술을 기반으로 한다.**

<details>
<summary>정답 확인</summary>

**X**. ECS는 AWS가 자체 개발한 오케스트레이션 서비스이고, EKS는 오픈소스 **Kubernetes** 기반의 오케스트레이션 서비스입니다. 두 서비스 모두 컨테이너를 관리하지만, 기반 기술이 다릅니다.
</details>

**3. AWS Fargate의 핵심 장점은 무엇인가?**

<details>
<summary>정답 확인</summary>

Fargate는 **서버(인프라)를 관리할 필요 없이** 컨테이너를 실행할 수 있는 서버리스 컴퓨팅 엔진입니다. EC2 인스턴스의 프로비저닝, 패치, 보안 업데이트 등을 AWS가 처리하며, 사용자는 컨테이너의 CPU/메모리 요구 사항만 지정하면 됩니다.
</details>

**4. (O/X) API Gateway는 백엔드에서 비즈니스 로직을 직접 실행한다.**

<details>
<summary>정답 확인</summary>

**X**. API Gateway는 요청을 받아 백엔드 서비스(Lambda, EC2, ECS 등)로 **라우팅**하는 "정문" 역할을 합니다. 비즈니스 로직은 백엔드 서비스에서 실행됩니다.
</details>

**5. Lambda와 Fargate는 모두 서버리스인데, 어떤 차이가 있는가?**

<details>
<summary>정답 확인</summary>

Lambda는 **코드(함수)**를 실행하는 서버리스 서비스로, 최대 실행 시간이 15분입니다. Fargate는 **컨테이너**를 실행하는 서버리스 서비스로, 장시간 실행되는 애플리케이션에 적합합니다. Lambda는 이벤트 기반의 단기 작업에, Fargate는 지속적으로 실행되는 서비스에 주로 사용됩니다.
</details>

**6. (O/X) AWS Step Functions는 Lambda 함수만 오케스트레이션할 수 있다.**

<details>
<summary>정답 확인</summary>

**X**. Step Functions는 Lambda 외에도 **ECS 태스크, DynamoDB, SNS, SQS, API Gateway** 등 200개 이상의 AWS 서비스와 통합하여 워크플로를 구성할 수 있습니다.
</details>

**7. 멀티 클라우드 전략을 사용하는 기업이 컨테이너 오케스트레이션 서비스를 선택할 때, ECS와 EKS 중 어떤 것이 더 적합한가?**

<details>
<summary>정답 확인</summary>

**EKS**가 더 적합합니다. EKS는 오픈소스 Kubernetes를 기반으로 하므로, AWS뿐 아니라 Google Cloud(GKE), Azure(AKS) 등 다른 클라우드에서도 동일한 Kubernetes 설정을 사용할 수 있어 **이식성**이 높습니다. ECS는 AWS 전용 서비스이므로 다른 클라우드로의 이전이 어렵습니다.
</details>

## 🧪 통합 실습: 서버리스 API로 "나의 첫 번째 클라우드 명함" 만들기

> **시나리오**: 당신은 프리랜서 개발자입니다. 자기소개 정보를 반환하는 간단한 API를 서버리스로 구축하고, 누구든 URL을 통해 당신의 정보를 확인할 수 있게 만들고 싶습니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | API Gateway + Lambda + Step Functions를 연동하여 서버리스 API 구축 |
| **사용 서비스** | Amazon API Gateway, AWS Lambda, AWS Step Functions |
| **활용 개념** | Section 9.5 API Gateway, Section 9.6 Step Functions, Chapter 5 Lambda |
| **선수 실습** | 없음 (이 Chapter 내에서 완결) |
| **예상 소요 시간** | 25분 |
| **비용** | Free Tier 범위 내 |

### 아키텍처 다이어그램

```
[사용자 브라우저]
       │
       │  GET /card
       ▼
┌──────────────┐
│ API Gateway  │  ← REST API "my-card-api"
│ (정문)       │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│Step Functions│  ← 상태 머신 "card-workflow"
│ (워크플로)   │
└──────┬───────┘
       │
       ├──→ [Lambda: get-greeting]    → "안녕하세요! 저는 ○○○입니다"
       │
       └──→ [Lambda: get-skills]      → "기술 스택: Python, AWS, ..."
       │
       ▼
  [응답 조합 및 반환]
```

### 실습 단계

> **Phase 1: Lambda 함수 2개 생성** (활용 개념: Chapter 5 Lambda)
>
> **Step 1**: AWS 콘솔 → 상단 검색창에 "Lambda" 입력 → Lambda 클릭
>
> **Step 2**: 첫 번째 함수 생성 — **Create function** 클릭
> - Function name: `get-greeting`
> - Runtime: Python 3.12
> - **Create function** 클릭
>
> **Step 3**: 함수 코드를 다음으로 교체:
> ```python
> import json
> 
> def lambda_handler(event, context):
>     name = event.get('name', '클라우드 학습자')
>     return {
>         'greeting': f'안녕하세요! 저는 {name}입니다. 반갑습니다!'
>     }
> ```
> → **Deploy** 클릭
>
> **Step 4**: 두 번째 함수 생성 — Lambda 목록으로 돌아가서 **Create function** 클릭
> - Function name: `get-skills`
> - Runtime: Python 3.12
> - **Create function** 클릭
>
> **Step 5**: 함수 코드를 다음으로 교체:
> ```python
> import json
> 
> def lambda_handler(event, context):
>     return {
>         'skills': ['AWS Cloud', 'Python', 'Serverless Architecture'],
>         'certification': 'AWS Cloud Practitioner (학습 중)'
>     }
> ```
> → **Deploy** 클릭
>
> **✅ 체크포인트**: Lambda 함수 목록에 `get-greeting`과 `get-skills` 두 개의 함수가 표시됩니다.

> **Phase 2: Step Functions 상태 머신 생성** (활용 개념: Section 9.6)
>
> **Step 6**: 상단 검색창에 "Step Functions" 입력 → Step Functions 클릭
>
> **Step 7**: **Create state machine** 클릭 → **Blank** 템플릿 선택
>
> **Step 8**: Workflow Studio에서 시각적으로 워크플로를 구성합니다:
> 1. 왼쪽 **Flow** 탭에서 **Parallel**을 캔버스로 드래그
> 2. Parallel 상태의 첫 번째 브랜치에 **Actions** 탭 → **AWS Lambda: Invoke**를 드래그
>    - 오른쪽 설정 패널에서 **Function name**: `get-greeting` 선택
>    - **State name**을 `GetGreeting`으로 변경
> 3. Parallel 상태의 두 번째 브랜치에 **AWS Lambda: Invoke**를 드래그
>    - **Function name**: `get-skills` 선택
>    - **State name**을 `GetSkills`로 변경
>
> **Step 9**: 상단의 **Config** 버튼을 클릭하여 상태 머신 설정:
> - **State machine name**: `card-workflow`
> - **Permissions**: **Create a new role** (새 역할 생성) 선택
>
> **Step 10**: **Create(생성)** 클릭
>
> **Step 11**: 생성 후, 상태 머신 상세 페이지에서 **Start execution(실행 시작)** 클릭
> - Input(입력)에 다음을 입력:
> ```json
> {
>     "name": "홍길동"
> }
> ```
> - **Start execution** 클릭
>
> **✅ 체크포인트**: 실행 결과에서 Parallel의 두 브랜치가 모두 **Succeeded(성공)** 으로 표시되고, 각 브랜치의 출력에 greeting과 skills 정보가 포함되어 있어야 합니다.

> **Phase 3: API Gateway 연동** (활용 개념: Section 9.5)
>
> **Step 12**: 상단 검색창에 "API Gateway" 입력 → API Gateway 클릭
>
> **Step 13**: **Create API** → **REST API** 의 **Build** 클릭
> - API name: `my-card-api`
> - Endpoint type: Regional
> - **Create API** 클릭
>
> **Step 14**: **Create resource(리소스 생성)** 클릭
> - Resource name: `card`
> - **Create resource** 클릭
>
> **Step 15**: `/card` 리소스가 선택된 상태에서 **Create method(메서드 생성)** 클릭
> - Method type: **GET**
> - Integration type: **AWS Service**
> - AWS Region: `ap-northeast-2` (사용 중인 리전)
> - AWS Service: **Step Functions**
> - HTTP method: **POST**
> - Action: **StartSyncExecution**
> - Execution role: Step Functions를 호출할 수 있는 IAM 역할 ARN 입력
>
> > 💬 **참고**: IAM 역할 설정이 복잡할 수 있습니다. 이 단계에서 어려움이 있다면, Phase 2의 Step Functions 직접 실행 결과로 통합 실습의 목표를 달성한 것으로 간주해도 됩니다. IAM에 대해서는 Chapter 13에서 자세히 다룹니다.
>
> **Step 16**: (Phase 3이 성공한 경우) API를 배포합니다
> - **Deploy API** 클릭
> - Stage name: `demo`
> - **Deploy** 클릭
>
> **Step 17**: 배포된 URL + `/card` 경로를 브라우저에서 확인
>
> **✅ 체크포인트**: 브라우저에서 인사말과 기술 스택 정보가 포함된 JSON 응답이 표시됩니다.

>

> 2. **Step Functions**: Step Functions 콘솔 → `card-workflow` 선택 → **Delete** → 확인
> 3. **Lambda 함수**: Lambda 콘솔 → `get-greeting` 선택 → **Actions** → **Delete function** (get-skills도 동일)
> 4. **IAM 역할**: (자동 생성된 역할이 있다면) IAM 콘솔 → Roles → Step Functions 관련 역할 삭제
>
> 💬 **참고**: Lambda 함수와 API Gateway는 호출하지 않으면 비용이 발생하지 않지만, 정리 습관을 들이는 것이 좋습니다.

### 트러블슈팅 (Troubleshooting)

> **Q: Step Functions 실행 시 "Lambda.ResourceNotFoundException" 오류가 발생합니다**
> A: Lambda 함수 이름이 정확한지 확인하세요. 또한 Step Functions와 Lambda가 **같은 리전**에 있어야 합니다. 콘솔 상단의 리전 선택을 확인하세요.

> **Q: Step Functions 실행 시 "AccessDeniedException" 오류가 발생합니다**
> A: Step Functions의 실행 역할(IAM Role)에 Lambda를 호출할 수 있는 권한이 없는 것입니다. 상태 머신 생성 시 "Create a new role"을 선택했다면 자동으로 권한이 부여되지만, 기존 역할을 사용한 경우 `lambda:InvokeFunction` 권한을 추가해야 합니다.

> **Q: API Gateway에서 Step Functions 연동이 어렵습니다**
> A: API Gateway와 Step Functions의 직접 연동은 IAM 역할 설정이 필요하여 초보자에게 어려울 수 있습니다. 대안으로 API Gateway → Lambda → Step Functions 호출 방식을 사용할 수도 있습니다. IAM에 대해서는 Chapter 13에서 상세히 다루므로, 지금은 Phase 2까지의 결과로 충분합니다.

>
> **요구사항**:
> 1. 주문이 들어오면 재고를 확인한다
> 2. 재고가 있으면 결제를 처리하고, 없으면 품절 알림을 보낸다
> 3. 결제가 완료되면, 배송 준비와 확인 이메일 발송을 동시에 진행한다
> 4. 모든 과정이 완료되면 "주문 처리 완료"로 종료한다
>
> **힌트**: Task, Choice, Parallel, Succeed 상태 유형을 사용합니다.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 요구사항을 Step Functions의 상태 유형에 매핑합니다.
>
> ```
> [Start]
>    │
>    ▼
> [Task: 재고 확인]
>    │
>    ▼
> [Choice: 재고 있음?]
>    │           │
>    │ Yes       │ No
>    ▼           ▼
> [Task:      [Task: 품절
>  결제 처리]   알림 발송]
>    │              │
>    ▼              ▼
> [Parallel]    [Succeed:
>    │  │        주문 실패]
>    │  │
>    ▼  ▼
> [Task:  [Task:
>  배송    확인
>  준비]   이메일]
>    │  │
>    ▼  ▼
> [Succeed: 주문 처리 완료]
> ```
>
> **상태별 설명**:
> 1. **Task (재고 확인)**: Lambda 함수가 데이터베이스에서 재고 수량을 조회
> 2. **Choice (재고 있음?)**: 재고 수량 > 0이면 결제로, 아니면 품절 알림으로 분기
> 3. **Task (결제 처리)**: Lambda 함수가 결제 API 호출
> 4. **Parallel**: 배송 준비(Lambda)와 이메일 발송(Lambda + SES)을 동시에 실행
> 5. **Succeed**: 워크플로 정상 종료
>
> **보충 설명**: 실제 운영에서는 결제 실패 시 재시도(Retry) 로직이나, 오류 발생 시 보상 트랜잭션(결제 취소 등)도 추가해야 합니다. Step Functions는 각 Task에 Retry와 Catch 설정을 지원합니다.
> </details>

> B) 트래픽에 따라 자동으로 확장된다

> **풀이 전략**:
>
> 1. **사용자 관리 서비스**: 트래픽은 적지만 항상 실행 → **ECS + Fargate**
>    - 항상 실행되어야 하므로 Lambda(이벤트 기반)보다 적합
>    - 운영 부담 최소화를 위해 EC2 대신 Fargate 선택
>    - 트래픽이 적으므로 소규모 태스크(0.25 vCPU, 0.5GB)로 비용 절감
>
> 2. **상품 카탈로그 서비스**: 읽기 요청 많음, 빠른 응답 → **API Gateway + Lambda**
>    - 읽기 요청은 보통 실행 시간이 짧아 Lambda에 적합
>    - API Gateway의 캐싱 기능으로 반복 요청에 대한 응답 속도 향상
>    - 사용한 만큼만 과금되어 비용 효율적
>
> 3. **주문 처리 서비스**: 이벤트 기반, 다단계 프로세스 → **API Gateway + Step Functions + Lambda**
>    - 주문 시에만 실행 → 서버리스(Lambda)로 비용 절감
>    - 여러 단계(재고 → 결제 → 배송) → Step Functions로 워크플로 관리
>    - 오류 처리, 재시도 → Step Functions의 내장 기능 활용
>
> 4. **Kubernetes 미경험** → EKS 대신 ECS 선택 (학습 비용 절감)
>
> **아키텍처 다이어그램**:
> ```
> [사용자/클라이언트]
>        │
>        ▼
> ┌──────────────┐
> │ API Gateway  │  (단일 진입점)
> └──────┬───────┘
>        │
>    ┌───┼───────────────┐
>    │   │               │
>    ▼   ▼               ▼
> [ECS+  [Lambda:      [Step Functions]
> Fargate: 카탈로그     ┌───┼───┐
>  사용자  조회]       │   │   │
>  관리]              [λ]  [λ] [λ]
>                   재고  결제 배송
> ```
>
> **확장 생각**: 서비스가 성장하면:
> - 상품 카탈로그에 ElastiCache(Redis)를 추가하여 캐싱 계층 강화
> - 주문 처리에 SQS를 추가하여 비동기 처리로 응답 속도 개선
> - 전체 서비스 앞에 CloudFront(CDN)를 배치하여 글로벌 응답 속도 향상
> </details>

> **Problem 9.2**: "서버리스와 컨테이너는 경쟁 관계인가, 보완 관계인가?"라는 질문에 대해 논술하세요.
>
> 다음 관점을 반드시 포함하세요:
> - Lambda(서버리스)와 Fargate(서버리스 컨테이너)의 관계
> - 각 기술이 잘 맞는 워크로드 유형
> - 하나의 애플리케이션 내에서 두 기술을 함께 사용하는 사례
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 서버리스와 컨테이너는 경쟁 관계가 아니라 **보완 관계**입니다. 오히려 Fargate처럼 두 개념이 결합된 형태도 존재합니다.
>
> **풀이 전략**:
>
> **1. 둘은 다른 문제를 해결한다**
> - **Lambda(서버리스 함수)**: "이 코드를 이벤트가 발생할 때만, 짧게 실행하고 싶다" → 이벤트 기반, 단기 실행, 함수 단위
> - **컨테이너(ECS/EKS)**: "이 애플리케이션을 일관된 환경에서, 지속적으로 실행하고 싶다" → 장시간 실행, 복잡한 런타임, 이식성
>
> **2. Fargate는 두 개념의 융합이다**
> - Fargate는 컨테이너를 서버리스로 실행합니다. 컨테이너의 이식성·유연성과 서버리스의 관리 편의성을 모두 제공합니다.
> - 이것은 두 기술이 대립하는 것이 아니라, 서로의 장점을 결합할 수 있음을 보여줍니다.
>
> **3. 하나의 애플리케이션에서 함께 사용하는 사례**
> - 웹 서버(상시 실행) → ECS + Fargate (컨테이너)
> - 이미지 업로드 처리(이벤트 기반) → Lambda (서버리스)
> - 배치 데이터 처리(야간 실행) → Lambda + Step Functions (서버리스 워크플로)
> - API 계층 → API Gateway (서버리스)
>
> 즉, 워크로드의 특성에 따라 같은 애플리케이션 내에서도 컨테이너와 서버리스 함수를 적재적소에 배치하는 것이 최적의 아키텍처입니다.
>
> **확장 생각**: 클라우드 아키텍처의 핵심 원칙은 "하나의 기술로 모든 문제를 해결하려 하지 않는 것"입니다. 이것은 Chapter 5에서 배운 "적합한 컴퓨팅 서비스 선택"의 연장선에 있으며, Chapter 24의 Well-Architected Framework에서 다시 다루게 됩니다.
> </details>

> **Problem 9.3**: 아래 기업 상황을 읽고, 현재 아키텍처의 문제점을 분석하고 개선안을 제시하세요.
>
> **현재 상황**: D사는 온라인 교육 플랫폼을 운영합니다. 현재 아키텍처:
> - 하나의 대형 EC2 인스턴스(r5.2xlarge)에서 모든 서비스(웹 서버, 동영상 처리, 사용자 인증, 결제)를 실행
> - 동영상 업로드 시 인코딩 처리에 10~30분 소요
> - 시험 기간에 트래픽이 10배로 급증하지만, 평소에는 인스턴스 사용률이 15% 미만
> - 월 EC2 비용: 약 $400
>
> 이 Chapter에서 배운 서비스들을 활용하여 아키텍처를 개선하세요. 예상 비용 절감 효과도 정성적으로 분석하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 모놀리식 아키텍처를 마이크로서비스로 분리하고, 각 워크로드 특성에 맞는 서비스를 선택합니다.
>
> **현재 문제점 분석**:
> 1. **자원 낭비**: 평소 사용률 15% → 85%의 자원이 낭비됨
> 2. **확장성 문제**: 시험 기간 10배 트래픽 → 하나의 인스턴스로는 감당 어려움
> 3. **단일 장애점**: 하나의 인스턴스가 모든 것을 담당 → 하나의 장애가 전체 서비스 중단으로 이어짐
> 4. **비용 비효율**: 트래픽이 적은 시간에도 대형 인스턴스 비용 지불
>
> **개선 아키텍처**:
>
> ```
> [사용자]
>    │
>    ▼
> [API Gateway] ← 단일 진입점, 트래픽 관리
>    │
>    ├──→ [ECS + Fargate: 웹 서버]    (상시 실행, 자동 확장)
>    │
>    ├──→ [Lambda: 사용자 인증]        (이벤트 기반, 짧은 실행)
>    │
>    ├──→ [Lambda: 결제 처리]          (이벤트 기반)
>    │
>    └──→ [Step Functions + Lambda:    (장시간 워크플로)
>          동영상 처리 파이프라인]
>              │
>              ├─ 업로드 수신 (Lambda)
>              ├─ 인코딩 (ECS + Fargate — 대용량 처리)
>              └─ 완료 알림 (Lambda → SNS)
> ```
>
> **서비스별 선택 이유**:
> - **웹 서버**: 상시 실행이 필요하므로 ECS + Fargate. 시험 기간에 태스크 수를 자동 확장
> - **인증/결제**: 요청 시에만 실행, 짧은 실행 시간 → Lambda가 최적
> - **동영상 처리**: 10~30분 소요로 Lambda의 15분 제한 초과 가능 → ECS + Fargate 태스크로 처리. Step Functions로 전체 파이프라인 관리
>
> **비용 절감 효과 (정성적)**:
> - **웹 서버**: 평소 소규모 Fargate 태스크(~$0.01/시간) → 시험 기간에만 확장 → 평소 비용 대폭 절감
> - **인증/결제**: Lambda Free Tier 범위 내 또는 최소 비용
> - **동영상 처리**: 인코딩 시에만 Fargate 태스크 실행 → 유휴 비용 없음
> - **예상 절감**: 월 $400 → $100~150 수준으로 60~75% 절감 가능
>
> **확장 생각**: 이 아키텍처는 Chapter 7에서 배운 CloudFront(CDN)를 추가하면 동영상 스트리밍 성능을 더욱 개선할 수 있고, Chapter 10에서 배울 SQS를 사용하면 동영상 처리 요청을 큐에 넣어 안정적으로 처리할 수 있습니다.
> </details>
