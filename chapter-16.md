## 16.1 AWS 규정 준수 프로그램과 Artifact

> 💡 **한 줄 요약**: AWS는 전 세계 수십 가지 보안·규제 인증을 보유하고 있으며, AWS Artifact를 통해 이 인증 보고서에 직접 접근할 수 있습니다.

---

### 직관적 이해 (Intuitive Understanding)
여러분이 식당을 운영한다고 상상해 봅시다. 손님들에게 "우리 식당은 위생적입니다"라고 말하는 것만으로는 신뢰를 얻기 어렵습니다. 대신, 보건소의 위생 검사 합격증을 벽에 걸어두면 손님들은 안심하고 식사할 수 있습니다. 이 합격증이 바로 **규정 준수 인증(Compliance Certification)**의 역할입니다.

클라우드 환경도 마찬가지입니다. 기업이 AWS에 데이터를 저장하고 서비스를 운영할 때, 고객, 감사인, 규제 기관은 "AWS가 정말 안전한가?"라고 질문합니다. AWS는 이에 대한 답으로, 독립적인 제3자 감사 기관이 검증한 다양한 **규정 준수 프로그램(Compliance Program)**을 운영합니다.

하지만 왜 이것이 AWS 사용자인 *우리*에게 중요할까요?

> 🔗 **연결**: Chapter 12에서 배운 **공동 책임 모델**을 떠올려 봅시다. AWS는 "클라우드의 보안(Security **of** the Cloud)"을, 고객은 "클라우드 안에서의 보안(Security **in** the Cloud)"을 책임집니다.

규정 준수도 마찬가지입니다. AWS가 인프라 수준의 인증을 획득해 놓았기 때문에, 우리는 그 위에서 *우리의 책임 영역*만 추가로 충족하면 됩니다. 예를 들어, 병원이 AWS에서 환자 데이터를 처리한다면, AWS가 이미 HIPAA 관련 인프라 인증을 보유하고 있으므로, 병원은 자신의 애플리케이션 수준에서 HIPAA 요구사항만 추가로 충족하면 됩니다. AWS의 인증이 없다면 병원은 데이터센터부터 모든 것을 직접 인증받아야 했을 것입니다.

---

### 핵심 개념 설명 (Core Explanation)
#### 규정 준수란 무엇인가?

> 📝 **규정 준수(Compliance)**: 조직이 법률, 규제, 산업 표준, 내부 정책 등의 요구사항을 충족하고 있음을 의미합니다. 클라우드 환경에서는 데이터 보호, 접근 제어, 감사 추적 등에 관한 각종 규정을 지키는 것을 말합니다.

규정 준수가 중요한 이유는 명확합니다:

- **법적 의무**: 많은 산업에서 특정 규정 준수는 법적 요구사항입니다. 위반 시 막대한 벌금이 부과될 수 있습니다.
- **고객 신뢰**: 인증을 통해 고객에게 데이터가 안전하게 관리됨을 증명할 수 있습니다.
- **비즈니스 기회**: 많은 대기업과 정부 기관은 인증을 보유한 서비스 제공자만 선택합니다.

#### AWS의 주요 규정 준수 프로그램

AWS는 전 세계적으로 100개 이상의 보안 규정 준수 인증을 보유하고 있습니다. AWS CCP 시험에서 자주 등장하는 핵심 인증을 살펴봅시다.

##### 1. SOC 보고서 (System and Organization Controls)

> 📝 **SOC 보고서**: 독립적인 감사 법인(예: 회계법인)이 AWS의 내부 통제(보안, 가용성, 기밀성 등)를 검증한 보고서입니다.

SOC 보고서에는 세 가지 유형이 있습니다:

| 유형 | 목적 | 대상 |
|------|------|------|
| **SOC 1** | 재무 보고에 영향을 미치는 내부 통제 검증 | 고객의 재무 감사인 |
| **SOC 2** | 보안, 가용성, 처리 무결성, 기밀성, 개인정보 보호에 대한 통제 검증 | 고객, 규제 기관 |
| **SOC 3** | SOC 2의 요약 버전 — 일반 대중에게 공개 가능 | 일반 대중 |

일상적 비유로 설명하면, SOC 1은 "이 식당의 회계가 정확한가?"를 확인하는 것이고, SOC 2는 "이 식당의 위생·안전·서비스가 기준을 충족하는가?"를 상세히 검증하는 것이며, SOC 3는 "이 식당은 검증을 통과했습니다"라는 간단한 요약 인증서입니다.

##### 2. ISO 27001

> 📝 **ISO 27001**: 국제표준화기구(ISO)가 정한 **정보보안 관리 체계(ISMS)** 표준입니다. 조직이 체계적으로 정보보안을 관리하고 있음을 인증합니다.

ISO 27001은 전 세계적으로 가장 널리 인정받는 보안 표준 중 하나입니다. AWS가 ISO 27001 인증을 보유하고 있다는 것은, AWS가 물리적 보안부터 데이터 접근 제어까지 포괄적인 보안 관리 체계를 운영하고 있음을 의미합니다. 마치 국제적으로 인정받는 "품질 보증 마크"를 받은 것과 같습니다.

##### 3. HIPAA (Health Insurance Portability and Accountability Act)

> 📝 **HIPAA**: 미국의 의료 정보 보호법으로, **보호 대상 건강 정보(PHI: Protected Health Information)**의 수집, 저장, 전송에 관한 엄격한 보안 요구사항을 규정합니다.

AWS는 **HIPAA 적격 서비스(HIPAA Eligible Services)**를 제공합니다. 이는 AWS가 HIPAA의 기술적 보호 요구사항을 충족하는 서비스를 별도로 지정하고, 고객이 해당 서비스를 사용할 때 **BAA(Business Associate Agreement, 사업 제휴 계약)**를 체결할 수 있음을 의미합니다.

> ⚠️ **주의**: AWS가 HIPAA 적격 서비스를 제공한다고 해서, AWS 위에서 운영하는 모든 시스템이 자동으로 HIPAA를 준수하는 것은 아닙니다. 공동 책임 모델에 따라, 고객은 자신의 애플리케이션과 데이터 관리 측면에서 HIPAA 요구사항을 별도로 충족해야 합니다.

##### 4. GDPR (General Data Protection Regulation)

> 📝 **GDPR**: 유럽연합(EU)의 일반 데이터 보호 규정으로, EU 시민의 **개인 데이터(Personal Data)**를 처리하는 모든 조직에 적용됩니다. 세계에서 가장 엄격한 개인정보 보호 규정 중 하나입니다.

GDPR의 핵심 요구사항은 다음과 같습니다:
- **데이터 최소화**: 필요한 최소한의 개인 데이터만 수집
- **목적 제한**: 명시된 목적으로만 데이터 사용
- **데이터 주체 권리**: 개인이 자신의 데이터 삭제, 이동, 열람을 요구할 수 있음
- **위반 시 벌금**: 전 세계 연매출의 최대 4% 또는 2,000만 유로 중 큰 금액

AWS는 GDPR을 준수하는 데이터 처리 부록(DPA: Data Processing Addendum)을 제공하며, EU 리전을 통해 데이터가 EU 내에 저장되도록 보장할 수 있습니다.

##### 5. PCI DSS (Payment Card Industry Data Security Standard)

> 📝 **PCI DSS**: 신용카드 정보를 처리, 저장, 전송하는 모든 조직이 준수해야 하는 보안 표준입니다. Visa, MasterCard 등 주요 카드사가 공동으로 제정했습니다.

온라인 쇼핑몰을 운영한다면, 고객의 카드 정보를 다루게 되므로 PCI DSS 준수가 필수적입니다. AWS는 PCI DSS Level 1(가장 높은 수준) 인증을 보유하고 있어, 이 위에서 결제 시스템을 구축할 때 인프라 수준의 요구사항은 이미 충족됩니다.

#### 주요 규정 준수 프로그램 비교

| 인증/규정 | 분야 | 주요 대상 | 핵심 요구사항 |
|-----------|------|----------|-------------|
| **SOC 1/2/3** | 내부 통제 | 금융, SaaS 등 전반 | 보안·가용성·기밀성 통제 검증 |
| **ISO 27001** | 정보보안 | 전 산업 | 체계적 보안 관리 체계(ISMS) |
| **HIPAA** | 의료 | 미국 의료·헬스케어 | 보호 대상 건강 정보(PHI) 보호 |
| **GDPR** | 개인정보 | EU 시민 데이터 처리자 | 개인 데이터 보호 및 처리 투명성 |
| **PCI DSS** | 결제 | 카드 정보 처리 조직 | 카드 소유자 데이터 보호 |

#### AWS Artifact

자, AWS가 이렇게 많은 인증을 보유하고 있다는 것은 알겠는데, 우리가 이 인증 보고서를 실제로 확인하려면 어떻게 해야 할까요? 이메일로 요청해야 할까요? AWS 영업 담당자에게 연락해야 할까요?

훨씬 간단합니다. AWS는 **Artifact**라는 셀프서비스 포털을 제공합니다.

> 📝 **AWS Artifact**: AWS의 규정 준수 보고서, 인증서, 계약서에 온디맨드로 접근할 수 있는 **무료** 셀프서비스 포털입니다.

AWS Artifact는 두 가지 주요 기능을 제공합니다:

1. **Artifact Reports**: SOC 보고서, ISO 인증서, PCI DSS 적합성 증명서(AOC) 등 제3자 감사 보고서를 다운로드할 수 있습니다.
2. **Artifact Agreements**: BAA(HIPAA 관련), DPA(GDPR 관련) 등의 계약서를 검토하고 수락할 수 있습니다.

일상적 비유로 설명하면, Artifact는 도서관의 "참고 자료실"과 같습니다. 감사인이 "AWS가 정말 SOC 2 인증을 받았나요?"라고 물으면, Artifact에 들어가서 해당 보고서를 바로 다운로드하여 제출할 수 있습니다. 별도의 요청이나 대기 시간이 필요 없습니다.

### 상세 예시 (Detailed Examples)
> 🔍 **예시 16.1.1: 스타트업의 규정 준수 요구사항**
>
> **상황**: 핀테크 스타트업 A사가 AWS에서 모바일 결제 서비스를 운영합니다. 대형 은행과의 파트너십을 위해 보안 감사를 받아야 합니다.
>
> **풀이/설명**:
> 1. 은행 감사인이 A사에 "인프라의 보안 통제 검증 보고서를 제출하세요"라고 요청합니다.
> 2. A사는 AWS Artifact에서 **SOC 2 보고서**를 다운로드하여 제출합니다 — 이것으로 AWS 인프라의 보안이 검증됨을 증명합니다.
> 3. 카드 결제를 처리하므로, **PCI DSS AOC(적합성 증명서)**도 함께 다운로드하여 제출합니다.
> 4. A사는 자신의 애플리케이션 수준에서 추가적으로 필요한 보안 통제를 문서화하여 제출합니다.
>
> **핵심 포인트**: AWS Artifact를 통해 인프라 수준의 규정 준수 증명을 즉시 확보할 수 있으므로, 고객은 자신의 책임 영역에 집중할 수 있습니다.

> 🔍 **예시 16.1.2: 의료 서비스의 HIPAA 준수**
>
> **상황**: 의료 스타트업 B사가 AWS에서 환자 건강 기록을 저장하는 시스템을 구축하려 합니다. HIPAA를 준수해야 합니다.
>
> **풀이/설명**:
> 1. B사는 AWS Artifact에서 **AWS BAA(Business Associate Agreement)**를 검토하고 수락합니다.
> 2. BAA를 체결함으로써, AWS가 HIPAA의 "Business Associate(사업 동반자)" 역할의 의무를 이행함을 계약적으로 확인합니다.
> 3. B사는 HIPAA 적격 서비스(예: Amazon S3, Amazon RDS 등 지정된 서비스)만을 사용하여 PHI를 처리합니다.
> 4. B사 자체적으로 데이터 암호화, 접근 로깅, 최소 권한 원칙 등 HIPAA의 기술적 요구사항을 자신의 애플리케이션 수준에서 구현합니다.
>
> **핵심 포인트**: HIPAA 준수는 BAA 체결과 적격 서비스 사용만으로 완료되지 않습니다. 공동 책임 모델에 따라, 고객의 영역(애플리케이션, 데이터 관리)에서도 HIPAA 요구사항을 충족해야 합니다.

### 미니 실습 (Hands-on Mini Lab)
> 🧪 **Mini Lab 16.1: AWS Artifact에서 규정 준수 보고서 탐색하기**
>
> **목표**: AWS Artifact 포털에 접근하여 어떤 규정 준수 보고서와 계약서가 제공되는지 직접 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (무료)
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `Artifact`를 입력하고 **AWS Artifact**를 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **Reports**를 클릭합니다. 다양한 규정 준수 보고서 목록이 표시됩니다.
>
> **Step 3**: 검색창에 `SOC`를 입력하여 SOC 보고서를 검색합니다. SOC 1, SOC 2, SOC 3 등 다양한 SOC 보고서가 나타나는 것을 확인합니다.
>
> **Step 4**: 아무 SOC 보고서를 하나 클릭하여 상세 정보를 확인합니다. 보고서를 다운로드하려면 NDA(비밀유지계약) 동의가 필요할 수 있습니다 — 실습에서는 동의하지 않아도 됩니다.
>
> **Step 5**: 다시 검색창에서 `ISO`를 검색하여 ISO 27001 인증서를 확인합니다.
>
> **Step 6**: 왼쪽 메뉴에서 **Agreements**를 클릭합니다. BAA, DPA 등 계약 유형을 확인할 수 있습니다. "Account agreements"와 "Organization agreements" 탭이 있는 것을 확인합니다.
>
> **확인**: Reports 섹션에서 SOC, ISO, PCI 등 다양한 보고서를 검색할 수 있고, Agreements 섹션에서 BAA 등의 계약서를 확인할 수 있으면 성공입니다.
>
> 🧹 **정리**: AWS Artifact는 정보를 조회하는 서비스이므로, 별도로 삭제하거나 비활성화할 리소스가 없습니다.

### 깊이 있는 이해 (Deep Dive)
> 🔬 **Deep Dive: 규정 준수 프로그램의 계층 구조**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 규정 준수 프로그램은 크게 세 가지 계층으로 분류할 수 있습니다:
>
> 1. **법률/규제(Regulation)**: 정부가 제정한 법적 의무 — HIPAA, GDPR 등
>    - 위반 시 **법적 제재**(벌금, 소송)가 따름
>    - 해당 산업/지역에서는 선택이 아닌 의무
>
> 2. **산업 표준(Industry Standard)**: 업계가 자율적으로 제정한 표준 — PCI DSS 등
>    - 법적 의무는 아닐 수 있으나, 사실상 필수(카드 결제를 하려면 PCI DSS 준수 필수)
>    - 위반 시 **비즈니스 제재**(카드 결제 처리 권한 박탈)
>
> 3. **인증 프레임워크(Certification Framework)**: 자발적 인증 — SOC, ISO 27001 등
>    - 법적 의무는 아니지만, 시장에서의 **신뢰와 경쟁력**을 위해 취득
>    - 고객이나 파트너가 요구하는 경우가 많음
>
> AWS는 이 모든 계층에 걸쳐 인증을 보유하고 있으며, 이를 통해 고객의 규정 준수 부담을 인프라 수준에서 크게 줄여줍니다.

---

### 흔한 실수와 주의사항 (Common Pitfalls)
> ⚠️ **주의: "AWS가 인증을 받았으니, 우리도 자동으로 인증된다" — 가장 흔한 오해**
>
> AWS의 규정 준수 인증은 **AWS의 인프라와 서비스**에 대한 것입니다. 고객이 AWS 위에서 운영하는 **애플리케이션과 데이터 관리**는 고객의 책임 영역입니다. 공동 책임 모델에 따라, 인프라 수준의 인증을 AWS가 제공하고, 그 위의 영역은 고객이 별도로 인증을 획득해야 합니다.

> ⚠️ **주의: SOC 2와 SOC 3의 혼동**
>
> SOC 2는 상세한 보고서로 NDA 하에 공유되며, SOC 3는 일반 대중에게 공개 가능한 요약 버전입니다. 시험에서 "고객이 일반에 공개할 수 있는 보고서"를 물으면 답은 **SOC 3**입니다.

---

### Section 요약 (Section Summary)

- **규정 준수(Compliance)**는 법률, 산업 표준, 인증 프레임워크에 따라 요구되는 보안·데이터 보호 의무를 충족하는 것이다.
- AWS는 SOC, ISO 27001, HIPAA, GDPR, PCI DSS 등 100개 이상의 인증을 보유하고 있다.
- **AWS Artifact**는 이러한 규정 준수 보고서와 계약서에 무료로 셀프서비스 접근할 수 있는 포털이다.
- 공동 책임 모델에 따라, AWS의 인증이 고객의 애플리케이션까지 자동으로 확장되지는 않는다.

## 16.2 AWS Config

> 💡 **한 줄 요약**: AWS Config는 AWS 리소스의 구성(설정)을 지속적으로 기록하고, 변경 사항을 추적하며, 원하는 구성 규칙에 부합하는지 자동으로 평가합니다.

---

### 직관적 이해 (Intuitive Understanding)
여러분의 집에 CCTV가 있다고 상상해 봅시다. CCTV는 집 안의 상태를 24시간 녹화합니다. 누군가 가구를 옮기면, "몇 시에 소파가 이동했다"는 기록이 남습니다. 만약 "거실 소파는 항상 벽 옆에 있어야 한다"는 규칙을 설정해 두면, 소파가 옮겨졌을 때 "규칙 위반"이라고 알려줄 수도 있습니다.

**AWS Config**는 바로 이런 역할을 합니다 — AWS 리소스 세계의 CCTV입니다. EC2 인스턴스, 보안 그룹, S3 버킷 등 AWS 리소스의 "구성(Configuration)"을 지속적으로 기록하고, 변경이 발생하면 추적하며, 미리 정의한 규칙에 위반되면 알려줍니다.

왜 이것이 필요할까요? 실제 기업 환경에서는 수백, 수천 개의 AWS 리소스가 존재합니다. 누군가 보안 그룹의 인바운드 규칙을 변경하거나, S3 버킷의 퍼블릭 접근을 허용하면, 보안 사고로 이어질 수 있습니다. AWS Config를 사용하면 이런 변경을 실시간으로 감지하고, 규정 준수 위반 여부를 자동으로 판단할 수 있습니다.

---

### 핵심 개념 설명 (Core Explanation)
#### AWS Config의 핵심 기능

AWS Config는 세 가지 핵심 기능을 제공합니다:

**1. 리소스 구성 기록 (Configuration Recording)**

> 📝 **구성 항목(Configuration Item, CI)**: 특정 시점에서 AWS 리소스의 속성, 관계, 구성 상태를 기록한 스냅샷입니다. 리소스의 "신분증"과 같습니다.

AWS Config는 지원되는 AWS 리소스의 구성 항목을 지속적으로 기록합니다. 예를 들어, EC2 인스턴스의 경우 인스턴스 유형, 보안 그룹, VPC, AMI ID 등의 정보가 기록됩니다.

**2. 변경 추적 (Change Tracking)**

리소스의 구성이 변경될 때마다 새로운 구성 항목이 생성됩니다. 이를 통해 **구성 타임라인(Configuration Timeline)**을 확인할 수 있습니다 — 마치 리소스의 "변경 이력서"를 보는 것과 같습니다.

예: "이 보안 그룹은 1월 15일에 포트 22를 열었고, 2월 3일에 포트 80을 추가했으며, 3월 10일에 포트 22를 다시 닫았다."

**3. 규정 준수 규칙 평가 (Config Rules)**

> 📝 **Config Rules(구성 규칙)**: AWS 리소스가 원하는 구성 상태를 유지하고 있는지 자동으로 평가하는 규칙입니다. 규칙에 부합하면 "준수(Compliant)", 위반하면 "비준수(Non-compliant)"로 표시됩니다.

AWS Config Rules에는 두 가지 유형이 있습니다:

| 유형 | 설명 | 예시 |
|------|------|------|
| **AWS 관리형 규칙** | AWS가 미리 정의한 규칙 (300개 이상) | `s3-bucket-public-read-prohibited`: S3 버킷이 퍼블릭 읽기를 허용하지 않는지 확인 |
| **사용자 지정 규칙** | Lambda 함수를 사용하여 직접 만든 규칙 | "모든 EC2 인스턴스에 'Environment' 태그가 있어야 한다" |

규칙 평가는 두 가지 방식으로 실행됩니다:
- **변경 트리거(Configuration change)**: 관련 리소스가 변경될 때 자동 평가
- **주기적 트리거(Periodic)**: 일정 간격(1시간, 3시간, 6시간, 12시간, 24시간)으로 평가

---

#### AWS Config의 작동 흐름

```
[AWS 리소스 변경 발생]
        │
        ▼
[AWS Config가 변경 감지]
        │
        ├──▶ [구성 항목(CI) 기록] ──▶ [S3 버킷에 저장]
        │
        ├──▶ [Config Rules 평가]
        │         │
        │         ├── 준수(Compliant) ✅
        │         └── 비준수(Non-compliant) ❌ ──▶ [SNS 알림 전송 가능]
        │
        └──▶ [구성 타임라인 업데이트]
```

---

#### 거버넌스에서 AWS Config의 역할

> 📝 **거버넌스(Governance)**: 조직의 IT 자원이 정책과 규정에 따라 관리되고 있는지 감독하고 통제하는 체계입니다. "올바른 방식으로 운영되고 있는가?"를 지속적으로 확인하는 과정입니다.

AWS Config는 거버넌스의 핵심 도구입니다:

- **지속적 모니터링**: "모든 S3 버킷이 암호화되어 있는가?", "모든 보안 그룹이 SSH를 제한하고 있는가?" 같은 질문에 실시간으로 답할 수 있습니다.
- **감사 증거 제공**: 감사인이 "3개월 전 이 리소스의 상태는 어떠했나요?"라고 물으면, 구성 타임라인을 통해 정확한 답을 제공할 수 있습니다.
- **규정 준수 대시보드**: Config Rules의 준수/비준수 현황을 대시보드로 한눈에 확인할 수 있습니다.

---

### 상세 예시 (Detailed Examples)
> 🔍 **예시 16.2.1: 보안 그룹 변경 추적**
>
> **상황**: 보안팀에서 "누가 웹 서버 보안 그룹에 포트 22(SSH)를 전 세계에 개방했는가?"라는 보안 사고 조사를 진행합니다.
>
> **풀이/설명**:
> 1. AWS Config 콘솔에서 해당 보안 그룹 리소스를 검색합니다.
> 2. **구성 타임라인(Configuration Timeline)**을 확인합니다.
> 3. 타임라인에서 인바운드 규칙이 변경된 시점을 찾습니다.
> 4. 해당 시점의 구성 항목을 비교하면, "이전에는 포트 22가 특정 IP만 허용했는데, 이 시점 이후 0.0.0.0/0(전체)으로 변경됨"을 확인할 수 있습니다.
>
> **핵심 포인트**: AWS Config의 구성 타임라인을 통해 "무엇이, 언제, 어떻게 변경되었는지"를 정확히 추적할 수 있습니다. (누가 변경했는지는 CloudTrail에서 확인합니다 — Section 16.3 참고.)

> 🔍 **예시 16.2.2: Config Rules를 활용한 자동 규정 준수 확인**
>
> **상황**: 회사 보안 정책에 따라 "모든 S3 버킷은 서버 측 암호화가 활성화되어야 한다"는 규칙을 적용하려 합니다.
>
> **풀이/설명**:
> 1. AWS Config에서 AWS 관리형 규칙 `s3-bucket-server-side-encryption-enabled`를 활성화합니다.
> 2. 이 규칙은 계정 내 모든 S3 버킷을 평가하여, 서버 측 암호화가 활성화되지 않은 버킷을 **비준수(Non-compliant)**로 표시합니다.
> 3. 새로운 S3 버킷이 생성될 때마다 자동으로 평가가 실행됩니다.
> 4. 비준수 리소스가 발견되면 SNS를 통해 보안팀에 알림을 보내도록 설정할 수 있습니다.
>
> **핵심 포인트**: Config Rules를 사용하면 수동으로 하나하나 확인하지 않아도, 규정 준수 여부가 자동으로 평가되고 관리됩니다.

### 미니 실습 (Hands-on Mini Lab)
> 🧪 **Mini Lab 16.2: AWS Config 구성 기록 살펴보기**
>
> **목표**: AWS Config를 활성화하고, 리소스의 구성 기록과 Config Rules 대시보드를 직접 확인합니다.
> **예상 소요 시간**: 15분
> **비용**: ⚠️ 비용 발생 (구성 항목당 $0.003, 실습 규모에서 $1 미만)
>

>
> **Step 4**: 리소스 목록에서 아무 리소스(예: 보안 그룹, IAM 역할 등)를 하나 클릭합니다. 해당 리소스의 **구성 타임라인(Configuration timeline)**과 현재 구성 상태를 확인할 수 있습니다.
>
> **Step 5**: 왼쪽 메뉴에서 **Rules**를 클릭한 후, **Add rule**을 클릭합니다. AWS 관리형 규칙 목록이 표시됩니다. 검색창에 `s3-bucket`을 입력하여 S3 관련 규칙들을 탐색합니다. (규칙을 실제로 추가하지 않아도 됩니다 — 어떤 규칙이 있는지 확인만 합니다.)
>
> **Step 6**: 왼쪽 메뉴에서 **Dashboard**를 클릭하여 전체 규정 준수 현황 대시보드를 확인합니다.
>
> **확인**: Resources에서 리소스 목록과 구성 타임라인을 확인할 수 있고, Rules에서 다양한 관리형 규칙을 탐색할 수 있으면 성공입니다.
>
> 🧹 **정리**: AWS Config는 활성화된 동안 지속적으로 구성 항목을 기록하여 비용이 발생합니다. 실습이 끝나면 다음과 같이 비활성화합니다:
> 1. AWS Config 콘솔 왼쪽 메뉴에서 **Settings**를 클릭합니다.
> 2. **Recording** 섹션에서 **Edit**를 클릭합니다.
> 3. **Enable recording** 토글을 끄고(비활성화) **Save**합니다.
> 4. (선택) Config가 생성한 S3 버킷(`config-bucket-xxxxx`)도 삭제하려면, S3 콘솔에서 해당 버킷을 비운 후 삭제합니다.

---

### 흔한 실수와 주의사항 (Common Pitfalls)
> ⚠️ **주의: AWS Config와 CloudTrail의 혼동**
>
> 초보자가 가장 자주 혼동하는 부분입니다:
> - **AWS Config**: 리소스의 **구성 상태(무엇이 어떻게 설정되어 있는가)**를 기록합니다. → "이 보안 그룹의 현재/과거 설정은 무엇인가?"
> - **AWS CloudTrail**: **API 활동(누가 무엇을 했는가)**을 기록합니다. → "누가 이 보안 그룹을 변경했는가?"
>
> 둘은 보완적인 서비스입니다. Config는 "무엇이 변했는지"를, CloudTrail은 "누가 변경했는지"를 알려줍니다.

> ⚠️ **주의: AWS Config 비용**
>
> AWS Config는 무료가 아닙니다. 기록된 구성 항목당, 평가된 Config Rule당 과금됩니다. 학습 실습 후에는 반드시 기록을 비활성화하세요.

---

### Section 요약 (Section Summary)

- **AWS Config**는 AWS 리소스의 구성을 지속적으로 기록하고, 변경을 추적하며, 규칙 기반으로 규정 준수 여부를 자동 평가하는 서비스이다.
- **구성 항목(CI)**은 특정 시점의 리소스 상태 스냅샷이며, **구성 타임라인**으로 변경 이력을 확인할 수 있다.
- **Config Rules**를 사용하면 "모든 S3 버킷이 암호화되어 있는가?" 같은 규정 준수 질문에 자동으로 답할 수 있다.
- AWS Config는 거버넌스의 핵심 도구로, 감사 증거 제공과 지속적 규정 준수 모니터링에 활용된다.

## 16.3 AWS CloudTrail 소개

> 💡 **한 줄 요약**: AWS CloudTrail은 AWS 계정에서 발생하는 모든 API 호출을 기록하여, "누가, 언제, 무엇을, 어디서" 했는지를 추적하는 감사 서비스입니다.

---

### 직관적 이해 (Intuitive Understanding)
건물 출입 기록을 생각해 봅시다. 사무실 빌딩에는 출입 카드 시스템이 있어서, 누가 몇 시에 어떤 문을 열고 들어왔는지 모든 기록이 남습니다. 만약 야간에 서버실에 무단 침입이 발생하면, 이 출입 기록을 통해 누가 그 시간에 서버실에 접근했는지 확인할 수 있습니다.

**AWS CloudTrail**은 AWS 계정의 "출입 기록 시스템"입니다. AWS에서 일어나는 거의 모든 활동은 **API 호출**로 이루어집니다 — 콘솔에서 버튼을 클릭하든, CLI로 명령어를 실행하든, SDK로 코드를 실행하든, 내부적으로는 모두 AWS API를 호출합니다. CloudTrail은 이 모든 API 호출을 기록합니다.

왜 이것이 필요할까요?

- **보안 사고 조사**: "누가 이 EC2 인스턴스를 삭제했지?"
- **규정 준수 감사**: "지난 90일간 IAM 정책을 변경한 기록을 보여주세요."
- **운영 문제 해결**: "왜 이 Lambda 함수의 권한이 갑자기 바뀌었지?"

---

### 핵심 개념 설명 (Core Explanation)
#### CloudTrail 이벤트란?

> 📝 **CloudTrail 이벤트(Event)**: AWS 계정에서 발생한 단일 API 호출 또는 활동의 기록입니다. 누가, 언제, 어디서(소스 IP), 무엇을(API 이름), 어떤 리소스에 대해 작업했는지를 담고 있습니다.

CloudTrail 이벤트에 포함되는 주요 정보:

| 필드 | 설명 | 예시 |
|------|------|------|
| **누가 (Who)** | 요청을 수행한 IAM 사용자/역할 | `arn:aws:iam::123456789012:user/Alice` |
| **언제 (When)** | 이벤트 발생 시각 | `2026-04-08T09:15:30Z` |
| **무엇을 (What)** | 호출된 API 이름 | `RunInstances` (EC2 시작) |
| **어디서 (Where)** | 요청 소스 IP 주소 | `203.0.113.50` |
| **대상 (Target)** | 대상 리소스 | `i-0abcdef1234567890` |
| **결과 (Result)** | 성공 또는 실패 | `Success` 또는 `AccessDenied` |

> 🔗 **연결**: Chapter 13에서 배운 **IAM 사용자, 역할, 정책**을 떠올려 봅시다. CloudTrail은 이 IAM 자격 증명이 *실제로 어떤 활동을 수행했는지*를 기록합니다. IAM이 "누가 무엇을 *할 수 있는가*"를 정의한다면, CloudTrail은 "누가 무엇을 *했는가*"를 기록합니다.

---

#### CloudTrail의 이벤트 유형

CloudTrail은 세 가지 유형의 이벤트를 기록합니다:

| 이벤트 유형 | 설명 | 예시 | 기본 기록 |
|------------|------|------|----------|
| **관리 이벤트 (Management Events)** | AWS 리소스에 대한 관리 작업 | EC2 인스턴스 생성, IAM 정책 변경, VPC 생성 | ✅ 기본 활성화 |
| **데이터 이벤트 (Data Events)** | 리소스 *내부*의 데이터 작업 | S3 객체 읽기/쓰기, Lambda 함수 실행 | ❌ 별도 활성화 필요 |
| **인사이트 이벤트 (Insights Events)** | 비정상적인 API 활동 패턴 감지 | 갑자기 API 호출 빈도가 급증 | ❌ 별도 활성화 필요 |

> 💬 **참고**: 데이터 이벤트는 양이 매우 많을 수 있어(예: S3에 파일을 업로드할 때마다 기록) 별도로 활성화해야 하며, 추가 비용이 발생합니다.

---

#### CloudTrail의 기본 동작

AWS CloudTrail에 대해 알아야 할 핵심 사항:

1. **기본 활성화**: 모든 AWS 계정에는 CloudTrail이 기본적으로 활성화되어 있습니다. 별도 설정 없이도 최근 90일간의 관리 이벤트를 확인할 수 있습니다.

2. **이벤트 기록 (Event History)**: CloudTrail 콘솔의 "Event history"에서 최근 90일간의 관리 이벤트를 무료로 조회, 검색, 다운로드할 수 있습니다.

3. **추적 (Trail)**: 90일 이상 보관하거나, 데이터 이벤트를 기록하거나, 로그를 S3에 저장하려면 **추적(Trail)**을 생성해야 합니다.

> 📝 **추적(Trail)**: CloudTrail 이벤트를 지속적으로 S3 버킷에 저장하도록 설정하는 구성입니다. 장기 보관과 상세 분석이 필요할 때 생성합니다.

```
[AWS 계정의 모든 API 호출]
        │
        ▼
[CloudTrail이 기록]
        │
        ├──▶ [이벤트 기록 (Event History)] ── 최근 90일, 무료
        │
        └──▶ [추적 (Trail)] ──▶ [S3 버킷에 장기 저장]
                                       │
                                       └──▶ [감사, 분석, 규정 준수 증거]
```

---

#### 감사에서 CloudTrail의 역할

CloudTrail은 규정 준수와 감사의 핵심 증거입니다:

- **감사 추적(Audit Trail)**: 모든 API 활동의 불변 기록을 제공하여, 감사인이 요구하는 "증거"를 갖추게 합니다.
- **이상 활동 탐지**: 비정상적인 API 호출 패턴을 식별하여 보안 위협을 조기에 감지할 수 있습니다.
- **책임 추적(Accountability)**: 특정 변경이 누구에 의해 이루어졌는지 정확히 식별할 수 있습니다.

> 💬 **참고**: CloudTrail의 심화 활용(로그 분석, Athena와의 연동, 조직 수준 추적 등)은 Chapter 18에서 자세히 다룹니다. 이 Section에서는 CloudTrail의 개념과 감사에서의 역할에 집중합니다.

### 상세 예시 (Detailed Examples)
> 🔍 **예시 16.3.1: 삭제된 EC2 인스턴스 추적**
>
> **상황**: 월요일 아침, 개발팀이 "주말 동안 프로덕션 EC2 인스턴스가 삭제되었다"고 보고합니다. 누가 삭제했는지 알아내야 합니다.
>
> **풀이/설명**:
> 1. CloudTrail 콘솔의 **Event history**에 접근합니다.
> 2. 필터에서 **Event name**을 `TerminateInstances`로 설정합니다.
> 3. 시간 범위를 지난 주말로 설정하여 검색합니다.
> 4. 결과에서 해당 인스턴스 ID와 일치하는 이벤트를 찾습니다.
> 5. 이벤트 상세 정보에서 **User name**(IAM 사용자), **Source IP address**(접속 IP), **Event time**(정확한 시각)을 확인합니다.
>
> **핵심 포인트**: CloudTrail의 이벤트 기록을 통해 "누가, 언제, 어디서" 리소스를 삭제했는지 정확히 추적할 수 있습니다.

> 🔍 **예시 16.3.2: Config와 CloudTrail의 상호 보완**
>
> **상황**: S3 버킷의 퍼블릭 접근이 갑자기 허용된 것을 AWS Config가 "비준수"로 탐지했습니다.
>
> **풀이/설명**:
> 1. **AWS Config**에서 해당 S3 버킷의 구성 타임라인을 확인합니다 → "4월 5일에 Block Public Access 설정이 비활성화됨"을 발견합니다.
> 2. **AWS CloudTrail**에서 해당 시간대에 `PutBucketPublicAccessBlock` API 호출을 검색합니다 → "사용자 'DevUser01'이 4월 5일 14:30에 해당 설정을 변경함"을 확인합니다.
> 3. 원인: 개발자가 테스트를 위해 퍼블릭 접근을 임시로 열었지만, 다시 닫는 것을 잊은 것으로 파악됩니다.
>
> **핵심 포인트**: Config는 **"무엇이 변했는지"**, CloudTrail은 **"누가 변경했는지"**를 알려줍니다. 두 서비스를 함께 사용하면 완전한 감사 추적이 가능합니다.

### 미니 실습 (Hands-on Mini Lab)
> 🧪 **Mini Lab 16.3: CloudTrail 이벤트 기록 탐색하기**
>
> **목표**: CloudTrail의 이벤트 기록(Event History)에서 최근 API 활동을 직접 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (무료 — 이벤트 기록 조회는 무료)
>
> **Step 1**: AWS Management Console 상단 검색창에 `CloudTrail`을 입력하고 **AWS CloudTrail**을 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **Event history**를 클릭합니다. 최근 90일간의 관리 이벤트 목록이 시간순으로 표시됩니다.
>
> **Step 3**: 이벤트 목록을 스크롤하며 살펴봅니다. 각 이벤트에서 **Event name**(API 이름), **User name**(수행자), **Event time**(시각), **Event source**(서비스)를 확인합니다.
>
> **Step 4**: 필터 드롭다운에서 **Event name**을 선택하고, `ConsoleLogin`을 입력합니다. 이것은 AWS 콘솔에 로그인한 기록입니다. 본인의 로그인 기록이 표시되는 것을 확인합니다.
>
> **Step 5**: 아무 이벤트를 하나 클릭하여 상세 정보를 펼칩니다. JSON 형식으로 된 전체 이벤트 레코드를 확인합니다. `userIdentity`, `sourceIPAddress`, `eventTime`, `eventName` 등의 필드를 살펴봅니다.
>
> **Step 6**: 필터를 **Event name**으로 변경하고, `CreateBucket`이나 `RunInstances` 등 다른 API 이름으로 검색하여, 어떤 활동들이 기록되어 있는지 탐색합니다.
>
> **확인**: Event history에서 최근 API 활동 목록을 확인할 수 있고, 각 이벤트의 상세 정보(수행자, 시각, 소스 IP 등)를 볼 수 있으면 성공입니다.
>
> 🧹 **정리**: CloudTrail 이벤트 기록은 기본 제공 기능으로, 별도로 삭제하거나 비활성화할 것이 없습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)
> ⚠️ **주의: "CloudTrail은 실시간 방어 도구다" — 오해**
>
> CloudTrail은 **기록(로깅)** 서비스이지, **차단(방어)** 서비스가 아닙니다. 누군가 리소스를 삭제하면, CloudTrail은 그 사실을 *기록*하지만 삭제를 *막지*는 않습니다. 방어를 위해서는 IAM 정책으로 권한을 제한하거나, SCP(서비스 제어 정책)로 통제해야 합니다 — 이는 Section 16.4에서 다룹니다.

> ⚠️ **주의: 기본 90일 기록의 한계**
>
> CloudTrail의 Event History는 최근 90일간의 관리 이벤트만 보관합니다. 규정 준수 감사에서는 1년 이상의 기록을 요구하는 경우가 많으므로, **Trail을 생성하여 S3에 장기 보관**하는 것이 실무에서 필수적입니다.

---

### Section 요약 (Section Summary)

- **AWS CloudTrail**은 AWS 계정에서 발생하는 모든 API 호출을 기록하는 감사 서비스이다.
- 모든 AWS 계정에 기본 활성화되어 있으며, 최근 90일간의 관리 이벤트를 무료로 조회할 수 있다.
- CloudTrail 이벤트에는 누가(IAM 자격 증명), 언제, 무엇을(API), 어디서(소스 IP) 했는지의 정보가 포함된다.
- AWS Config가 "무엇이 변했는지"를 알려준다면, CloudTrail은 "누가 변경했는지"를 알려주는 보완적 서비스이다.
- 장기 보관이 필요하면 **추적(Trail)**을 생성하여 S3에 저장한다.

## 16.4 AWS Organizations와 SCP

> 💡 **한 줄 요약**: AWS Organizations는 여러 AWS 계정을 하나로 묶어 중앙에서 관리하며, 서비스 제어 정책(SCP)을 통해 각 계정이 할 수 있는 작업의 최대 범위를 제한합니다.

---

### 직관적 이해 (Intuitive Understanding)
대기업의 조직 구조를 떠올려 봅시다. 본사 아래 여러 사업부(개발, 마케팅, 재무)가 있고, 각 사업부 아래 여러 팀이 있습니다. 본사에서는 전사적인 규칙(예: "모든 부서는 매월 경비 보고서를 제출해야 한다")을 정하고, 각 사업부는 그 범위 안에서 자율적으로 운영합니다.

AWS에서도 마찬가지입니다. 기업이 성장하면 하나의 AWS 계정으로는 부족해집니다:

- **개발 환경**과 **프로덕션 환경**을 분리하고 싶다
- **부서별로 비용을 추적**하고 싶다
- **보안 사고의 영향 범위(blast radius)**를 제한하고 싶다

이때 **AWS Organizations**를 사용하여 여러 AWS 계정을 체계적으로 관리합니다.

### 핵심 개념 설명 (Core Explanation)
#### AWS Organizations의 구조

> 📝 **AWS Organizations**: 여러 AWS 계정을 하나의 조직으로 묶어 중앙에서 관리할 수 있는 서비스입니다. 결제 통합, 정책 관리, 계정 자동 생성 등의 기능을 제공합니다.

AWS Organizations의 핵심 구성 요소:

| 구성 요소 | 설명 | 비유 |
|----------|------|------|
| **관리 계정 (Management Account)** | 조직의 최상위 계정. 다른 계정을 생성·초대하고, 정책을 관리 | 본사 CEO |
| **멤버 계정 (Member Account)** | 조직에 속한 개별 AWS 계정 | 각 사업부/팀 |
| **루트 (Root)** | 조직의 최상위 컨테이너 | 회사 전체 |
| **조직 단위 (OU: Organizational Unit)** | 계정을 그룹으로 묶는 컨테이너. OU 안에 하위 OU를 만들 수 있음 | 사업부, 부서 |

조직 구조의 예시:

```
[Root] ─── 조직의 최상위
   │
   ├── [관리 계정] ── 정책 관리, 결제 통합
   │
   ├── [OU: Production] ── 프로덕션 환경
   │       ├── 계정: prod-web
   │       └── 계정: prod-api
   │
   ├── [OU: Development] ── 개발 환경
   │       ├── 계정: dev-team-a
   │       └── 계정: dev-team-b
   │
   └── [OU: Security] ── 보안/감사
           └── 계정: security-audit
```

#### 서비스 제어 정책 (SCP)

> 📝 **서비스 제어 정책(SCP: Service Control Policy)**: 조직 내 계정이 사용할 수 있는 AWS 서비스와 작업의 **최대 범위(Permission Boundary)**를 정의하는 정책입니다. SCP는 권한을 *부여*하지 않고, 권한의 *상한선*을 설정합니다.

이 개념을 이해하는 것이 매우 중요합니다. 비유로 설명하겠습니다:

> 🔗 **연결**: Chapter 13에서 배운 **IAM 정책**을 떠올려 봅시다. IAM 정책이 "이 사용자는 S3에 접근할 수 있다"라고 *허용*하더라도, SCP가 "이 계정에서는 S3를 사용할 수 없다"라고 *제한*하면, 해당 사용자는 S3에 접근할 수 없습니다.

IAM 정책과 SCP의 관계를 일상 비유로 설명하면:

- **SCP**는 건물의 **출입문**과 같습니다. "이 건물에서는 흡연이 금지"라는 규칙이 있으면, 건물 안의 모든 사람에게 적용됩니다.
- **IAM 정책**은 각 방의 **개별 열쇠**와 같습니다. 건물에 들어올 수 있더라도, 특정 방의 열쇠가 없으면 그 방에 들어갈 수 없습니다.

둘의 관계를 공식처럼 표현하면:

```
실제 허용 권한 = IAM 정책 허용 ∩ SCP 허용 (교집합)
```

IAM 정책이 허용하더라도 SCP가 차단하면 불가능하고, SCP가 허용하더라도 IAM 정책이 없으면 역시 불가능합니다.

---

#### SCP의 주요 특징

1. **관리 계정에는 적용되지 않음**: SCP는 멤버 계정에만 적용됩니다. 관리 계정은 SCP에 의해 제한되지 않습니다.
2. **상속(Inheritance)**: 상위 OU에 적용된 SCP는 하위 OU와 그 안의 모든 계정에 상속됩니다.
3. **루트 사용자에게도 적용**: 멤버 계정의 루트 사용자조차 SCP에 의해 제한될 수 있습니다.
4. **허용 목록 vs 거부 목록**: SCP는 허용 목록(Allowlist) 방식 또는 거부 목록(Denylist) 방식으로 작성할 수 있습니다.

---

#### SCP 활용 예시

**거부 목록 방식** (가장 일반적):

기본적으로 모든 서비스를 허용하되, 특정 서비스나 작업을 명시적으로 차단합니다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": [
        "ec2:RunInstances"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": ["ap-northeast-2", "us-east-1"]
        }
      }
    }
  ]
}
```

위 SCP는 "서울(ap-northeast-2)과 버지니아(us-east-1) 리전 외의 리전에서는 EC2 인스턴스를 시작할 수 없다"는 의미입니다. 데이터 주권이나 비용 관리를 위해 사용하는 리전을 제한하는 일반적인 패턴입니다.

---

#### 통합 결제 (Consolidated Billing)

AWS Organizations의 또 다른 핵심 기능은 **통합 결제**입니다:

- 모든 멤버 계정의 비용이 관리 계정으로 통합되어 하나의 청구서로 발행됩니다.
- **볼륨 할인** 혜택: 여러 계정의 사용량이 합산되어 더 높은 할인 티어에 도달할 수 있습니다.
- 각 계정별 비용을 별도로 추적할 수 있어, 부서별 비용 배분이 가능합니다.

### 상세 예시 (Detailed Examples)
> 🔍 **예시 16.4.1: 다중 계정 거버넌스 전략**
>
> **상황**: 중견 기업 C사가 AWS 도입을 확대하면서 10개의 프로젝트 팀이 각각 AWS를 사용합니다. 보안팀은 "모든 팀이 ap-northeast-2(서울) 리전만 사용하고, 프로덕션 계정에서는 IAM 사용자를 직접 생성할 수 없도록" 하고 싶습니다.
>
> **풀이/설명**:
> 1. **AWS Organizations**를 설정하고, 다음 OU 구조를 만듭니다:
>    - `Root` → `OU: Production`, `OU: Development`, `OU: Sandbox`
> 2. **Root 수준에 SCP 적용**: 허용 리전을 `ap-northeast-2`로 제한하는 SCP를 Root에 적용합니다. 이 정책은 모든 OU와 계정에 상속됩니다.
> 3. **Production OU에 추가 SCP 적용**: `iam:CreateUser` 작업을 거부하는 SCP를 Production OU에 적용합니다. 프로덕션에서는 IAM 사용자 대신 역할(Role)만 사용하도록 강제합니다.
> 4. 결과: 모든 팀은 서울 리전에서만 작업할 수 있고, 프로덕션 팀은 추가로 IAM 사용자 생성이 차단됩니다.
>
> **핵심 포인트**: SCP는 OU 계층 구조를 통해 상속되므로, 조직 전체에 일관된 정책을 적용하면서도, 특정 환경에 추가 제한을 둘 수 있습니다.

> 🔍 **예시 16.4.2: SCP와 IAM 정책의 상호작용**
>
> **상황**: Development OU에 속한 계정의 IAM 사용자 "DevUser"에게 `AmazonS3FullAccess` IAM 정책이 할당되어 있습니다. 그러나 해당 OU에 "S3 버킷 삭제 금지" SCP가 적용되어 있습니다. DevUser가 S3 버킷을 삭제할 수 있을까요?
>
> **풀이/설명**:
> 1. DevUser의 IAM 정책: `AmazonS3FullAccess` → S3의 모든 작업 허용 (삭제 포함)
> 2. OU의 SCP: `Deny s3:DeleteBucket` → S3 버킷 삭제 차단
> 3. 실제 권한 = IAM 허용 ∩ SCP 허용 = S3 전체 **에서** 버킷 삭제를 **제외**
> 4. **결론**: DevUser는 S3의 다른 모든 작업(읽기, 쓰기 등)은 가능하지만, **버킷 삭제는 불가능**합니다.
>
> **핵심 포인트**: SCP의 Deny는 IAM 정책의 Allow보다 우선합니다. SCP는 권한의 "천장(ceiling)"을 설정하며, IAM 정책이 아무리 넓은 권한을 부여해도 SCP의 한계를 넘을 수 없습니다.

### 미니 실습 (Hands-on Mini Lab)
> 🧪 **Mini Lab 16.4: AWS Organizations 콘솔 탐색하기**
>
> **목표**: AWS Organizations 콘솔에서 조직 구조, OU, 정책 기능을 탐색합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (무료 — AWS Organizations 자체는 무료)
>
> **Step 1**: AWS Management Console 상단 검색창에 `Organizations`를 입력하고 **AWS Organizations**를 클릭합니다.
>
> **Step 2**: 아직 조직을 생성하지 않은 경우, **Create an organization** 버튼이 표시됩니다. 클릭하여 조직을 생성합니다. (현재 계정이 관리 계정이 됩니다.)
>
> **Step 3**: 조직이 생성되면, 왼쪽 메뉴에서 **AWS accounts**를 클릭합니다. 현재 계정(관리 계정)이 Root 아래에 표시됩니다.
>
> **Step 4**: **Root**를 클릭한 후, **Actions** → **Organizational unit** → **Create new**를 선택합니다. OU 이름을 `Test-OU`로 입력하고 생성합니다. (이것은 빈 OU이므로 비용이 발생하지 않습니다.)
>
> **Step 5**: 왼쪽 메뉴에서 **Policies**를 클릭합니다. **Service control policies**를 선택합니다. SCP 기능이 비활성화된 경우 **Enable service control policies**를 클릭하여 활성화합니다.
>
> **Step 6**: SCP 목록에서 **FullAWSAccess**라는 기본 정책을 확인합니다. 이 정책은 모든 AWS 서비스와 작업을 허용하며, 기본적으로 Root에 연결되어 있습니다.
>
> **확인**: 조직 구조에서 Root, 관리 계정, 생성한 Test-OU를 확인할 수 있고, SCP 메뉴에서 FullAWSAccess 정책을 확인할 수 있으면 성공입니다.
>
> 🧹 **정리**:
> 1. 생성한 `Test-OU`를 삭제합니다: **AWS accounts** → `Test-OU` 클릭 → **Actions** → **Delete organizational unit**.
> 2. (선택) 조직 자체를 삭제하려면: **Settings** → **Delete organization**. 단, 멤버 계정이 없어야 삭제 가능합니다.
> 3. 💬 **참고**: 조직을 유지해도 비용은 발생하지 않습니다. 이후 학습에서 활용할 수 있으므로 유지해도 무방합니다.

### 깊이 있는 이해 (Deep Dive)
> 🔬 **Deep Dive: 다중 계정 전략(Multi-Account Strategy)의 모범 사례**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> AWS는 다중 계정 전략에서 다음과 같은 OU 구조를 권장합니다:
>
> ```
> Root
>  ├── Security OU ── 보안/감사 전용 계정 (로그 아카이브, 보안 도구)
>  ├── Infrastructure OU ── 공유 인프라 (네트워크 허브, DNS 등)
>  ├── Workloads OU
>  │     ├── Production OU ── 프로덕션 워크로드
>  │     └── SDLC OU ── 개발/스테이징 환경
>  ├── Sandbox OU ── 실험/PoC 용도
>  └── Suspended OU ── 비활성화된 계정
> ```
>
> 이 구조의 핵심 원칙:
> - **최소 권한**: 각 OU에 필요한 최소한의 서비스만 SCP로 허용
> - **환경 분리**: 프로덕션과 개발 환경을 별도 OU로 분리하여 영향 범위 제한
> - **중앙 보안**: 로그와 보안 도구를 별도 계정에 집중하여 변조 방지
>
> 이 구조는 Chapter 16.5에서 다루는 AWS Control Tower가 자동으로 설정해 주는 기본 구조의 기반이 됩니다.

---

### 흔한 실수와 주의사항 (Common Pitfalls)
> ⚠️ **주의: SCP는 관리 계정에 적용되지 않는다**
>
> 가장 자주 시험에 나오는 포인트입니다. SCP는 **멤버 계정에만** 적용됩니다. 관리 계정의 루트 사용자나 IAM 사용자는 SCP의 제한을 받지 않습니다. 따라서 관리 계정은 최소한의 용도(조직 관리, 결제)로만 사용하고, 실제 워크로드는 멤버 계정에서 실행하는 것이 모범 사례입니다.

> ⚠️ **주의: SCP는 권한을 "부여"하지 않는다**
>
> SCP가 "EC2 허용"이라고 되어 있어도, 해당 계정의 IAM 사용자에게 EC2 접근 권한이 IAM 정책으로 부여되어 있지 않으면 접근할 수 없습니다. SCP는 권한의 상한선만 설정하며, 실제 권한은 IAM 정책으로 부여해야 합니다.

---

### Section 요약 (Section Summary)

- **AWS Organizations**는 여러 AWS 계정을 하나의 조직으로 묶어 중앙 관리하는 서비스이다.
- **OU(조직 단위)**를 사용하여 계정을 그룹으로 분류하고, 계층적으로 관리할 수 있다.
- **SCP(서비스 제어 정책)**는 계정이 사용할 수 있는 서비스/작업의 최대 범위를 설정하며, IAM 정책과의 교집합이 실제 권한이 된다.
- SCP는 멤버 계정에만 적용되며, 관리 계정에는 적용되지 않는다.
- **통합 결제**를 통해 여러 계정의 비용을 하나로 관리하고, 볼륨 할인 혜택을 받을 수 있다.

## 16.5 AWS Control Tower

> 💡 **한 줄 요약**: AWS Control Tower는 AWS Organizations 기반의 다중 계정 환경을 모범 사례에 따라 자동으로 설정하고, 가드레일로 지속적으로 관리하는 서비스입니다.

---

### 직관적 이해 (Intuitive Understanding)
새 아파트 단지를 건설한다고 상상해 봅시다. 각 세대(계정)를 하나하나 직접 설계하고 배관·전기·소방 설비를 설치할 수도 있지만, "표준 설계도(Blueprint)"가 있으면 모든 세대가 안전 기준을 충족하면서도 빠르게 건설할 수 있습니다. 또한 입주 후에도 "소방 점검"이 정기적으로 이루어져 안전 기준이 유지됩니다.

**AWS Control Tower**는 바로 이 "표준 설계도 + 정기 점검" 역할을 합니다:

- **랜딩 존(Landing Zone)**: 모범 사례에 따라 자동으로 설정되는 다중 계정 환경의 "표준 설계도"
- **가드레일(Guardrails)**: 환경이 모범 사례를 벗어나지 않도록 지속적으로 감시하는 "안전 장치"

Section 16.4에서 AWS Organizations로 OU를 만들고, SCP를 작성하고, 하나하나 설정하는 과정을 보았습니다. Control Tower는 이 모든 것을 **자동화**합니다.

---

### 핵심 개념 설명 (Core Explanation)
#### 랜딩 존 (Landing Zone)

> 📝 **랜딩 존(Landing Zone)**: AWS Control Tower가 모범 사례에 따라 자동으로 프로비저닝하는 다중 계정 환경입니다. Organizations, IAM Identity Center, 로그 아카이브 계정, 감사 계정 등이 포함됩니다.

Control Tower가 자동으로 설정하는 항목:

| 자동 설정 항목 | 설명 |
|--------------|------|
| **AWS Organizations** | 조직 구조와 기본 OU 생성 |
| **로그 아카이브 계정** | CloudTrail, Config 로그를 중앙 저장하는 전용 계정 |
| **감사(Audit) 계정** | 보안팀이 모든 계정을 감사할 수 있는 전용 계정 |
| **IAM Identity Center** | SSO(Single Sign-On)로 모든 계정에 통합 접근 |
| **기본 가드레일** | 필수 보안 가드레일 자동 적용 |

---

#### 가드레일 (Guardrails)

> 📝 **가드레일(Guardrail)**: Control Tower가 관리하는 사전 정의된 거버넌스 규칙으로, 다중 계정 환경이 모범 사례를 유지하도록 보장합니다.

가드레일은 두 가지 방식으로 동작합니다:

| 유형 | 동작 방식 | 사용 기술 | 예시 |
|------|----------|----------|------|
| **예방적(Preventive)** | 규칙을 위반하는 작업을 사전에 **차단** | SCP | "루트 사용자의 접근 키 생성 금지" |
| **탐지적(Detective)** | 규칙 위반을 **탐지**하고 알림 | AWS Config Rules | "MFA가 활성화되지 않은 루트 사용자 탐지" |

가드레일은 또한 적용 수준에 따라 분류됩니다:

| 수준 | 설명 |
|------|------|
| **필수(Mandatory)** | Control Tower가 자동으로 적용하며 비활성화 불가 |
| **강력 권장(Strongly recommended)** | AWS 모범 사례에 기반하며, 활성화를 강력히 권장 |
| **선택적(Elective)** | 특정 요구사항에 따라 선택적으로 활성화 |

---

#### Control Tower와 Organizations의 관계

```
AWS Control Tower
    │
    ├── AWS Organizations (자동 구성)
    │       ├── Root
    │       │    ├── Security OU (자동 생성)
    │       │    │    ├── 로그 아카이브 계정 (자동 생성)
    │       │    │    └── 감사 계정 (자동 생성)
    │       │    └── Sandbox OU (자동 생성)
    │       │
    │       └── SCP (가드레일의 예방적 규칙으로 자동 적용)
    │
    ├── AWS Config Rules (가드레일의 탐지적 규칙으로 자동 적용)
    │
    └── IAM Identity Center (SSO 자동 구성)
```

즉, Control Tower는 Organizations, Config, SCP 등 이 Chapter에서 배운 여러 서비스를 **통합하고 자동화**하는 상위 서비스입니다.

---

### 상세 예시 (Detailed Examples)
> 🔍 **예시 16.5.1: Control Tower 없이 vs Control Tower로 다중 계정 구성**
>
> **상황**: 50인 규모 스타트업이 개발, 스테이징, 프로덕션 환경을 분리하여 AWS 다중 계정 환경을 구축하려 합니다.
>
> **풀이/설명**:
>
> | 단계 | Control Tower 없이 (수동) | Control Tower 사용 |
> |------|-------------------------|-------------------|
> | 1. 조직 생성 | Organizations에서 수동 생성 | 자동 생성 |
> | 2. OU 구조 | 직접 설계하고 생성 | 모범 사례 구조 자동 생성 |
> | 3. 보안 계정 | 로그/감사 계정 직접 생성·구성 | 자동 생성·구성 |
> | 4. SCP 작성 | JSON으로 직접 작성·적용 | 가드레일로 자동 적용 |
> | 5. Config 규칙 | 각 계정에 수동 설정 | 탐지적 가드레일로 자동 설정 |
> | 6. SSO 구성 | IAM Identity Center 직접 설정 | 자동 구성 |
> | 7. 신규 계정 | 수동 생성 후 정책 수동 적용 | Account Factory로 표준화된 계정 자동 생성 |
>
> **핵심 포인트**: Control Tower는 수동으로 수일이 걸릴 수 있는 다중 계정 설정을 자동화하여, 실수 없이 모범 사례를 적용합니다.

> 🔍 **예시 16.5.2: 가드레일의 예방적 vs 탐지적 동작**
>
> **상황**: "모든 S3 버킷에 서버 측 암호화가 적용되어야 한다"는 규칙을 적용하려 합니다.
>
> **풀이/설명**:
> - **예방적 가드레일**: SCP를 사용하여 암호화 없이 S3 버킷을 생성하는 API 호출 자체를 차단합니다. → 위반이 발생하기 **전에** 막음
> - **탐지적 가드레일**: Config Rule을 사용하여 암호화가 없는 S3 버킷이 존재하는지 주기적으로 검사합니다. → 위반이 발생한 **후에** 탐지하여 알림
>
> 두 가지를 함께 사용하면 더욱 강력한 거버넌스를 구현할 수 있습니다.
>
> **핵심 포인트**: 예방적 가드레일은 "잘못된 일이 일어나지 않게" 하고, 탐지적 가드레일은 "잘못된 일이 일어났을 때 알려줍니다."

---

### 흔한 실수와 주의사항 (Common Pitfalls)
> ⚠️ **주의: Control Tower ≠ Organizations**
>
> Control Tower는 Organizations을 **기반으로** 작동하지만, 별개의 서비스입니다. Organizations는 계정 관리와 SCP를 제공하는 기본 서비스이고, Control Tower는 그 위에 자동화된 설정, 가드레일, Account Factory 등을 추가한 상위 서비스입니다.

> ⚠️ **주의: Control Tower 활성화 시 주의**
>
> Control Tower를 활성화하면 여러 리소스(계정, Config 규칙, CloudTrail 추적 등)가 자동으로 생성됩니다. 학습 목적으로는 콘솔에서 기능을 탐색하는 것만으로 충분하며, 실제 활성화는 비용과 구성 영향을 이해한 후 진행해야 합니다.

---

### Section 요약 (Section Summary)

- **AWS Control Tower**는 다중 계정 환경을 모범 사례에 따라 자동으로 설정하고 관리하는 서비스이다.
- **랜딩 존**은 Control Tower가 자동으로 프로비저닝하는 안전한 다중 계정 환경이다.
- **가드레일**은 예방적(SCP 기반 차단)과 탐지적(Config Rules 기반 탐지) 두 가지 유형이 있다.
- Control Tower는 Organizations, Config, SCP 등 여러 서비스를 통합·자동화하는 상위 서비스이다.

## 16.6 AWS Audit Manager

> 💡 **한 줄 요약**: AWS Audit Manager는 AWS 사용에 대한 감사 증거를 자동으로 수집하고, 규정 준수 프레임워크에 맞는 감사 보고서를 생성하는 서비스입니다.

---

### 직관적 이해 (Intuitive Understanding)
세무 감사를 받는 상황을 상상해 봅시다. 감사관이 "지난 1년간의 모든 거래 내역, 영수증, 계약서를 보여주세요"라고 요청합니다. 이때 영수증을 서랍에서 하나하나 찾아 정리하는 것은 엄청나게 번거로운 일입니다. 하지만 처음부터 회계 프로그램이 모든 거래를 자동으로 분류하고, 필요한 증빙을 감사 항목별로 정리해 놓았다면 어떨까요? 감사관에게 보고서를 바로 제출할 수 있습니다.

**AWS Audit Manager**는 바로 이런 "자동 회계 프로그램" 역할을 합니다. AWS 환경에서의 활동 증거를 자동으로 수집하고, SOC 2, HIPAA, GDPR, PCI DSS 등의 규정 준수 프레임워크에 맞춰 증거를 정리합니다.

---

### 핵심 개념 설명 (Core Explanation)
#### Audit Manager의 핵심 기능

> 📝 **AWS Audit Manager**: AWS 환경의 감사 증거를 지속적으로 자동 수집하고, 규정 준수 프레임워크에 매핑하여 감사 준비를 간소화하는 서비스입니다.

Audit Manager의 작동 흐름:

```
[규정 준수 프레임워크 선택]        [AWS 환경의 활동]
(예: SOC 2, HIPAA)                    │
        │                              │
        ▼                              ▼
[Audit Manager가 프레임워크의    [CloudTrail, Config, Security Hub 등
 각 통제 항목을 매핑]             에서 증거를 자동 수집]
        │                              │
        └──────────┬───────────────────┘
                   │
                   ▼
        [통제 항목별로 증거 정리]
                   │
                   ▼
        [감사 보고서 생성/내보내기]
```

핵심 구성 요소:

| 구성 요소 | 설명 |
|----------|------|
| **프레임워크(Framework)** | 감사 기준 (SOC 2, HIPAA 등). AWS가 사전 제공하거나 사용자 지정 가능 |
| **통제(Control)** | 프레임워크 내 개별 요구사항 (예: "암호화가 적용되어야 한다") |
| **증거(Evidence)** | 통제 항목 충족을 증명하는 데이터 (Config 기록, CloudTrail 로그 등) |
| **평가(Assessment)** | 특정 프레임워크에 대한 감사 과정. 증거를 수집하고 보고서를 생성 |

---

#### Audit Manager와 다른 서비스의 연계

Audit Manager는 이 Chapter에서 배운 여러 서비스의 데이터를 활용합니다:

- **AWS CloudTrail**: API 활동 기록 → "누가 무엇을 했는가"에 대한 증거
- **AWS Config**: 리소스 구성 상태 → "리소스가 규정에 부합하는가"에 대한 증거
- **AWS Security Hub**: 보안 검사 결과 → 보안 통제 충족 여부에 대한 증거

이렇게 여러 서비스에서 산재된 증거를 Audit Manager가 **하나의 프레임워크에 맞춰 통합 정리**하는 것이 핵심 가치입니다.

---

### 상세 예시 (Detailed Examples)
> 🔍 **예시 16.6.1: SOC 2 감사 준비**
>
> **상황**: SaaS 기업 D사가 고객의 요청으로 SOC 2 감사를 받아야 합니다. 감사인이 "접근 제어, 변경 관리, 로깅에 관한 증거를 제출하세요"라고 요청합니다.
>
> **풀이/설명**:
> 1. AWS Audit Manager에서 **SOC 2** 프레임워크를 선택하여 새 평가(Assessment)를 시작합니다.
> 2. Audit Manager가 SOC 2의 각 통제 항목(접근 제어, 변경 관리, 로깅 등)에 해당하는 증거를 자동 수집합니다:
>    - 접근 제어 → IAM 정책, CloudTrail의 인증 이벤트
>    - 변경 관리 → Config의 구성 변경 기록
>    - 로깅 → CloudTrail 활성화 여부, 로그 무결성 검증
> 3. 수집된 증거는 각 통제 항목에 매핑되어 대시보드에 표시됩니다.
> 4. 감사 시점에 보고서를 생성하여 감사인에게 제출합니다.
>
> **핵심 포인트**: Audit Manager는 증거 수집을 자동화하여, 수동으로 각 서비스에서 스크린샷과 로그를 모으는 번거로움을 없앱니다.

> 🔍 **예시 16.6.2: 지속적 감사 준비 vs 단발성 감사**
>
> **상황**: 기존에 연 1회 감사 시점에 급하게 증거를 모았던 E사가, 이제 Audit Manager를 도입하여 상시 감사 준비 체계를 구축하려 합니다.
>
> **풀이/설명**:
>
> | 항목 | 기존 방식 (수동) | Audit Manager 도입 후 |
> |------|----------------|---------------------|
> | 증거 수집 | 감사 직전 수주간 수동 수집 | 365일 자동 수집 |
> | 증거 형식 | 스크린샷, 엑셀, 이메일 혼재 | 표준화된 형식으로 통합 관리 |
> | 누락 위험 | 필요한 증거를 놓칠 수 있음 | 프레임워크 기반 체계적 수집 |
> | 감사 대응 시간 | 수주 | 수일 |
>
> **핵심 포인트**: Audit Manager를 통해 "감사 대응"에서 "지속적 감사 준비(Continuous Audit Readiness)"로 전환할 수 있습니다.

---

### 흔한 실수와 주의사항 (Common Pitfalls)
> ⚠️ **주의: Audit Manager ≠ 자동 규정 준수 달성**
>
> Audit Manager는 증거를 자동으로 *수집하고 정리*하는 도구이지, 규정 준수를 자동으로 *달성*해주는 도구가 아닙니다. 증거를 보고 비준수 항목이 발견되면, 이를 해결하는 것은 여전히 사용자의 책임입니다.

> ⚠️ **주의: Audit Manager와 AWS Artifact의 차이**
>
> - **AWS Artifact**: AWS *자체*의 규정 준수 보고서를 다운로드 (AWS의 인증 증명)
> - **AWS Audit Manager**: *고객 환경*의 감사 증거를 수집하고 보고서 생성 (고객의 규정 준수 관리)

---

### Section 요약 (Section Summary)

- **AWS Audit Manager**는 감사 증거를 자동으로 수집하고, 규정 준수 프레임워크에 맞춰 정리하는 서비스이다.
- SOC 2, HIPAA, GDPR, PCI DSS 등 다양한 사전 정의 프레임워크를 제공하며, 사용자 지정 프레임워크도 만들 수 있다.
- CloudTrail, Config, Security Hub 등에서 증거를 자동 수집하여 감사 준비를 간소화한다.
- Audit Manager(고객 환경의 감사)와 Artifact(AWS 자체의 인증 보고서)는 다른 서비스이다.

## 🧪 통합 실습: 스타트업의 규정 준수 기반 구축하기

> **시나리오**: 당신은 핀테크 스타트업의 첫 번째 클라우드 엔지니어입니다. 곧 있을 SOC 2 감사를 대비하여, AWS 환경에서 기본적인 규정 준수 체계를 구축해야 합니다. 오늘의 목표는 규정 준수 보고서를 확인하고, 리소스 변경 추적을 설정하며, API 활동 로그를 검토하는 것입니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | AWS Artifact에서 보고서 확인, AWS Config로 리소스 추적, CloudTrail로 API 활동 검토 |
| **사용 서비스** | AWS Artifact, AWS Config, AWS CloudTrail |
| **활용 개념** | Section 16.1 규정 준수 프로그램, Section 16.2 Config, Section 16.3 CloudTrail |
| **선수 실습** | 없음 (신규 설정) |
| **예상 소요 시간** | 25분 |
| **비용** | AWS Config 부분에서 소액 발생 가능 ($1 미만). Artifact와 CloudTrail Event History는 무료 |

### 아키텍처 다이어그램

```
[AWS Management Console]
        │
        ├──▶ [AWS Artifact] ── 규정 준수 보고서 확인
        │
        ├──▶ [AWS Config] ── 리소스 구성 기록 활성화
        │       │
        │       └──▶ [S3 버킷] ── 구성 기록 저장
        │
        └──▶ [AWS CloudTrail] ── API 활동 로그 검토
                │
                └──▶ [Event History] ── 최근 활동 확인
```

### 실습 단계

> **Phase 1: 규정 준수 보고서 확인** (활용 개념: Section 16.1)
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `Artifact`를 입력하고 **AWS Artifact**를 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **Reports**를 클릭합니다.
>
> **Step 3**: 검색창에 `SOC 2`를 입력합니다. SOC 2 관련 보고서 목록을 확인합니다. "이것이 감사인에게 제출할 수 있는 AWS의 인프라 보안 검증 보고서다"라는 것을 인지합니다.
>
> **Step 4**: 왼쪽 메뉴에서 **Agreements**를 클릭합니다. **Account agreements** 탭에서 어떤 계약 유형이 제공되는지 확인합니다.
>
> **✅ 체크포인트**: Reports에서 SOC 2 보고서를 검색할 수 있고, Agreements에서 계약 유형을 확인할 수 있으면 Phase 1 완료.

---

> **Phase 2: AWS Config로 리소스 변경 추적 설정** (활용 개념: Section 16.2)
>
> **Step 1**: 상단 검색창에 `Config`를 입력하고 **AWS Config**를 클릭합니다.
>
> **Step 2**: 처음 사용하는 경우, **1-click setup**을 선택하여 기본 설정으로 시작합니다. (이 과정에서 Config가 구성 기록을 저장할 S3 버킷이 자동 생성됩니다.)
>
> **Step 3**: 설정 완료 후, 왼쪽 메뉴에서 **Dashboard**를 클릭합니다. 현재 규정 준수 현황 대시보드를 확인합니다.
>
> **Step 4**: 왼쪽 메뉴에서 **Resources**를 클릭합니다. 계정 내 기록된 리소스 목록을 확인하고, 아무 리소스를 클릭하여 구성 상세 정보를 살펴봅니다.
>
> **✅ 체크포인트**: Config Dashboard가 표시되고, Resources에서 리소스 목록과 구성 정보를 확인할 수 있으면 Phase 2 완료.

---

> **Phase 3: CloudTrail로 API 활동 검토** (활용 개념: Section 16.3)
>
> **Step 1**: 상단 검색창에 `CloudTrail`을 입력하고 **AWS CloudTrail**을 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **Event history**를 클릭합니다.
>
> **Step 3**: 필터에서 **Event name**을 선택하고 `PutObject`를 검색합니다. 방금 Phase 2에서 AWS Config가 S3에 구성 기록을 저장한 활동이 기록되어 있을 수 있습니다. 이것은 Config가 S3에 데이터를 저장하는 API 호출이 CloudTrail에 기록된 것입니다.
>
> **Step 4**: 필터를 변경하여 `ConsoleLogin`을 검색합니다. 본인의 콘솔 로그인 기록을 확인합니다. 이벤트를 클릭하여 `sourceIPAddress`, `eventTime` 등을 확인합니다.
>
> **Step 5**: Config와 CloudTrail의 보완 관계를 직접 확인해 봅니다:
> - Config: "리소스가 어떤 상태인지" 기록
> - CloudTrail: "누가 어떤 API를 호출했는지" 기록
>
> **✅ 체크포인트**: Event History에서 API 활동을 검색하고, 각 이벤트의 상세 정보를 확인할 수 있으면 Phase 3 완료.

### 최종 확인

- AWS Config가 활성화되어 리소스 구성을 기록하고 있다.
- CloudTrail Event History에서 최근 API 활동을 조회할 수 있다.
- Config("무엇이 변했는지")와 CloudTrail("누가 했는지")의 보완 관계를 이해했다.

### 트러블슈팅 (Troubleshooting)

> **Q: AWS Config 설정 시 "Insufficient delivery policy" 오류가 발생합니다.**
> A: Config가 S3 버킷에 기록을 저장하기 위한 권한이 부족한 경우입니다. 1-click setup을 사용하면 S3 버킷과 권한이 자동 생성되므로, 1-click setup으로 다시 시도해 보세요.

> **Q: CloudTrail Event History에서 이벤트가 보이지 않습니다.**
> A: CloudTrail 이벤트가 표시되기까지 최대 15분이 소요될 수 있습니다. 리전이 올바른지 확인하고 (ap-northeast-2), 잠시 후 다시 확인해 보세요.

> **Q: AWS Artifact에서 보고서 다운로드 시 NDA 동의 화면이 나옵니다.**
> A: 일부 보고서는 NDA(비밀유지계약) 동의가 필요합니다. 학습 목적이라면 동의 후 다운로드해도 되고, 목록 확인만으로 충분합니다.

>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 실제 권한 = IAM 정책 허용 ∩ SCP 허용 (교집합) 원칙을 적용

>

> - IAM 정책 허용: EC2 Full, RDS Full
> - 실제 권한:
>   - EC2: IAM 허용 ✅ + SCP 허용 ✅ = **사용 가능**
>   - RDS: IAM 허용 ✅ + SCP Deny ❌ = **사용 불가** (SCP Deny가 우선)
>   - S3: SCP 허용 ✅ + IAM 없음 ❌ = **사용 불가**

>

> - 계정 A: EC2(전체), S3(읽기만)
> - 계정 B: EC2(전체)만 사용 가능
>
> **보충 설명**: SCP는 권한의 "천장"을 설정하고, IAM 정책은 실제 권한을 "부여"합니다. SCP가 허용하더라도 IAM 정책이 없으면 사용할 수 없고, IAM 정책이 허용하더라도 SCP가 거부하면 사용할 수 없습니다.
