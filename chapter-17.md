## 17.1 AWS Systems Manager — 인프라 운영 허브

> 💡 **한 줄 요약**: AWS Systems Manager는 AWS 인프라를 한곳에서 조회하고, 접속하고, 패치하고, 설정을 관리할 수 있는 **통합 운영 허브**입니다.

### 직관적 이해 (Intuitive Understanding)

아파트 관리사무소를 떠올려 봅시다. 관리사무소에서는 아파트 전체 동의 현황을 한눈에 파악하고, 엘리베이터 점검 일정을 잡고, 공용 시설을 관리하며, 주민에게 공지를 보냅니다. 각 세대를 일일이 방문하지 않아도 관리사무소에서 전체 아파트의 상태를 모니터링하고 관리할 수 있습니다.

**AWS Systems Manager(SSM)** 는 바로 이런 "클라우드 인프라의 관리사무소"입니다. EC2 인스턴스가 10대든 1,000대든, Systems Manager 콘솔 하나에서 서버 목록을 보고, 원격으로 접속하고, 운영 체제를 일괄 업데이트하고, 설정 값을 중앙에서 관리할 수 있습니다.

**왜 이것이 필요할까요?**

서버가 한 대라면 직접 SSH로 접속해서 관리하면 됩니다. 하지만 서버가 수십, 수백 대로 늘어나면 어떻게 될까요? 하나하나 접속해서 패치를 적용하고, 설정을 변경하는 것은 비현실적입니다. 또한 SSH 접속을 위해 22번 포트를 열어두는 것은 보안 위험이 됩니다. Systems Manager는 이런 문제를 해결하기 위해 만들어졌습니다.

> 🔗 **연결**: Chapter 13에서 배운 **IAM 역할**을 떠올려 봅시다. Systems Manager가 EC2 인스턴스를 관리하려면, 해당 인스턴스에 Systems Manager와 통신할 수 있는 IAM 역할이 부여되어 있어야 합니다. 이것이 바로 IAM 역할의 대표적인 활용 사례 중 하나입니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **AWS Systems Manager (SSM)**: AWS 리소스(주로 EC2 인스턴스와 온프레미스 서버)의 운영 작업을 자동화하고 중앙에서 관리하는 서비스. 서버 접속, 패치 관리, 설정 관리, 인벤토리 수집 등 다양한 운영 기능을 하나의 인터페이스에서 제공합니다.

Systems Manager는 하나의 서비스이지만, 그 안에 여러 가지 **기능(capability)** 이 포함되어 있습니다. 마치 스위스 군용 칼처럼, 하나의 도구 안에 다양한 기능이 들어있는 것입니다. CCP 시험에서 자주 출제되는 핵심 기능을 살펴봅시다.

#### SSM Agent — Systems Manager의 비밀 요원

Systems Manager가 EC2 인스턴스를 관리하려면, 인스턴스 내부에 **SSM Agent**라는 소프트웨어가 설치되어 있어야 합니다. SSM Agent는 아파트에 설치된 인터폰과 같습니다 — 관리사무소(Systems Manager)와 각 세대(EC2 인스턴스)를 연결해주는 통신 수단입니다.

> 📝 **SSM Agent**: EC2 인스턴스(또는 온프레미스 서버)에 설치되는 소프트웨어로, Systems Manager 서비스와 인스턴스 간의 통신을 담당합니다.

다행히 Amazon Linux 2, Amazon Linux 2023, Ubuntu Server 등 주요 AMI에는 SSM Agent가 **기본으로 설치**되어 있으므로, 별도 설치 없이 바로 사용할 수 있는 경우가 많습니다. SSM Agent가 작동하려면 두 가지가 필요합니다:

1. **SSM Agent 소프트웨어**가 인스턴스에 설치되어 실행 중이어야 합니다.
2. **IAM 역할**이 인스턴스에 연결되어 있어야 합니다 — 구체적으로 `AmazonSSMManagedInstanceCore`라는 AWS 관리형 정책이 포함된 역할이 필요합니다.

> 🔗 **연결**: Chapter 13에서 배운 **AWS 관리형 정책**이 여기서도 활용됩니다. `AmazonSSMManagedInstanceCore`는 AWS가 미리 만들어 놓은 관리형 정책으로, Systems Manager와 통신하는 데 필요한 최소한의 권한을 포함하고 있습니다.

#### Session Manager — SSH 없이 서버에 접속하기

> 📝 **Session Manager**: SSH 키나 별도의 인바운드 포트(22번) 없이, AWS 콘솔이나 CLI를 통해 EC2 인스턴스에 원격으로 접속(셸 세션)할 수 있는 기능입니다.

전통적으로 Linux 서버에 접속하려면 SSH 프로토콜을 사용하고, 이를 위해 보안 그룹에서 22번 포트를 열어야 했습니다. Session Manager는 이 방식을 근본적으로 바꿉니다:

| 비교 항목 | 전통적 SSH 접속 | Session Manager |
|-----------|----------------|-----------------|
| 포트 개방 | 22번 포트를 인바운드로 열어야 함 | **인바운드 포트를 열 필요 없음** |
| SSH 키 관리 | SSH 키 페어를 생성·배포·관리해야 함 | **SSH 키가 필요 없음** |
| 접근 제어 | IP 기반(보안 그룹) + SSH 키 | **IAM 정책**으로 세밀하게 제어 |
| 접속 기록 | 별도 설정 필요 | **자동으로 세션 로그 기록** |
| 접속 방법 | SSH 클라이언트 필요 | **AWS 콘솔 웹 브라우저**에서 바로 가능 |

Session Manager의 가장 큰 장점은 **보안성**입니다. SSH 포트를 열지 않아도 되므로 공격 표면(attack surface)이 줄어들고, 모든 접속이 IAM으로 통제되며, 누가 언제 어떤 명령을 실행했는지 기록이 남습니다.

> 🔍 **예시 17.1.1: 개발자의 서버 접속 시나리오**
>
> **상황**: 개발자 A가 운영 서버에 접속하여 로그 파일을 확인해야 합니다. 회사는 보안 정책상 SSH 포트(22번)를 외부에 개방하지 않습니다.
> **풀이/설명**:
> 1. 개발자 A는 AWS 콘솔에 IAM 사용자로 로그인합니다.
> 2. Systems Manager > Session Manager로 이동합니다.
> 3. 접속하려는 EC2 인스턴스를 선택하고 "Start session"을 클릭합니다.
> 4. 웹 브라우저에서 바로 서버의 셸(터미널)이 열립니다.
> 5. 필요한 로그 파일을 확인하고 세션을 종료합니다.
> **핵심 포인트**: SSH 키도, 포트 개방도 없이, IAM 권한만으로 안전하게 서버에 접속할 수 있습니다. 세션 기록은 자동으로 저장되어 감사에 활용할 수 있습니다.

> 🔍 **예시 17.1.2: 관리사무소 비유로 이해하는 Session Manager**
>
> **상황**: 아파트 관리인이 301호의 누수를 확인해야 합니다.
> **풀이/설명**:
> - **전통적 방법(SSH)**: 관리인이 301호의 열쇠(SSH 키)를 가지고 있어야 하고, 301호 현관문이 열려 있어야(포트 개방) 들어갈 수 있습니다. 열쇠를 분실하거나 복제되면 보안 문제가 생깁니다.
> - **Session Manager 방법**: 관리인은 관리사무소(AWS 콘솔)에서 인터폰(SSM Agent)으로 301호에 연결합니다. 관리인 신분증(IAM 인증)만 확인되면, 현관문을 열지 않아도 내부 상태를 확인하고 조치할 수 있습니다. 모든 통화 내용(세션 로그)은 자동으로 녹음됩니다.
> **핵심 포인트**: 물리적 열쇠(SSH 키) 대신 신분증 기반(IAM) 접근으로, 보안과 관리 편의성을 동시에 확보합니다.

#### Patch Manager — 운영 체제를 일괄 업데이트

> 📝 **Patch Manager**: EC2 인스턴스의 운영 체제 및 소프트웨어 패치를 자동화하는 기능. 패치 기준선(Patch Baseline)을 정의하고, 유지 관리 기간(Maintenance Window)을 설정하여 일괄적으로 패치를 적용합니다.

서버 100대의 운영 체제에 보안 패치가 나왔다고 가정해 봅시다. 100대에 하나씩 접속해서 `yum update`나 `apt upgrade`를 실행하는 것은 시간도 오래 걸리고 실수할 위험도 큽니다. Patch Manager는 이 과정을 자동화합니다:

1. **패치 기준선(Patch Baseline)**: "어떤 패치를 적용할 것인가?"를 정의합니다. 예를 들어, "심각도가 Critical과 Important인 보안 패치만 자동 적용"과 같은 규칙을 설정합니다.
2. **유지 관리 기간(Maintenance Window)**: "언제 패치를 적용할 것인가?"를 정합니다. 예를 들어, "매주 일요일 새벽 2시~4시"처럼 업무에 영향이 적은 시간대를 지정합니다.
3. **패치 그룹(Patch Group)**: "어떤 서버에 적용할 것인가?"를 태그로 분류합니다. 개발 서버와 운영 서버에 다른 패치 정책을 적용할 수 있습니다.

#### Parameter Store — 설정 값을 안전하게 보관

> 📝 **Parameter Store**: 설정 데이터(데이터베이스 연결 문자열, API 키, 비밀번호 등)를 중앙에서 안전하게 저장하고 관리하는 기능. 암호화를 지원하며, IAM으로 접근을 제어합니다.

애플리케이션은 동작하기 위해 다양한 설정 값이 필요합니다 — 데이터베이스의 주소, API 키, 비밀번호 등. 이런 값들을 코드에 직접 적어 놓으면 보안상 매우 위험합니다(코드가 유출되면 비밀번호도 유출됩니다). Parameter Store는 이런 설정 값을 안전하게 보관하는 "금고" 역할을 합니다.

- **일반 파라미터(String, StringList)**: 암호화가 필요 없는 일반 설정 값
- **보안 파라미터(SecureString)**: 비밀번호, API 키 등 암호화가 필요한 민감한 값. AWS KMS(Key Management Service)로 자동 암호화됩니다.

> 🔍 **예시 17.1.3: Parameter Store 활용 — 데이터베이스 비밀번호 관리**
>
> **상황**: 웹 애플리케이션이 데이터베이스에 접속하기 위해 비밀번호가 필요합니다. 이 비밀번호를 소스 코드에 직접 적는 것은 보안상 위험합니다.
> **풀이/설명**:
> 1. Parameter Store에 `/myapp/db/password`라는 이름으로 SecureString 파라미터를 생성합니다.
> 2. 비밀번호 값은 KMS로 자동 암호화되어 저장됩니다.
> 3. 애플리케이션은 실행 시 Parameter Store에서 이 값을 조회하여 사용합니다.
> 4. 비밀번호를 변경해야 할 때, Parameter Store의 값만 업데이트하면 됩니다.
> **핵심 포인트**: 민감한 설정 값을 코드 밖에서 안전하게 관리하고, IAM 정책으로 누가 이 값에 접근할 수 있는지 통제합니다.

> 🔍 **예시 17.1.4: 스위스 군용 칼로 이해하는 Systems Manager**
>
> **상황**: Systems Manager의 여러 기능을 일상적 비유로 이해합니다.
> **풀이/설명**:
> | Systems Manager 기능 | 일상 비유 |
> |---------------------|----------|
> | Session Manager | 관리사무소 인터폰 — 현관문(포트)을 열지 않아도 연결 |
> | Patch Manager | 아파트 전체 일괄 방역 서비스 — 일정에 맞춰 자동 수행 |
> | Parameter Store | 관리사무소 금고 — 중요한 서류(설정 값)를 안전하게 보관 |
> | Inventory | 입주민 현황표 — 각 세대(서버)에 무엇이 설치되어 있는지 파악 |
> | Run Command | 관리사무소 방송 — 모든 세대에 동시에 안내(명령) 전달 |
> **핵심 포인트**: Systems Manager는 "하나의 서비스, 여러 기능"입니다. 각 기능이 독립적으로 사용될 수 있지만, 모두 같은 관리 콘솔에서 통합되어 운영 효율을 높입니다.

> 🧪 **Mini Lab 17.1: Systems Manager 콘솔 탐색하기**
>
> **목표**: Systems Manager 콘솔에 접속하여 주요 기능 메뉴를 탐색하고, 관리형 인스턴스 목록 화면을 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `Systems Manager`를 입력하고 클릭합니다.
> **Step 2**: Systems Manager 대시보드가 표시됩니다. 왼쪽 탐색 메뉴를 살펴봅시다. 다음과 같은 카테고리가 보입니다:
>   - **Node Management**: Fleet Manager, Session Manager, Run Command, Patch Manager 등
>   - **Application Management**: Parameter Store, AppConfig 등
>   - **Operations Management**: Explorer, OpsCenter 등
> **Step 3**: 왼쪽 메뉴에서 **Node Management > Fleet Manager**를 클릭합니다. 현재 SSM Agent가 설치되고 IAM 역할이 연결된 EC2 인스턴스가 있다면 목록에 표시됩니다. (인스턴스가 없다면 빈 목록이 표시됩니다 — 정상입니다.)
> **Step 4**: 왼쪽 메뉴에서 **Application Management > Parameter Store**를 클릭합니다. 아직 파라미터를 생성하지 않았으므로 빈 화면이 표시됩니다. "Create parameter" 버튼이 있는 것을 확인합니다.
> **Step 5**: 왼쪽 메뉴에서 **Node Management > Session Manager**를 클릭합니다. Session Manager 시작 페이지가 표시됩니다.
>
> **확인**: Systems Manager의 다양한 기능이 왼쪽 탐색 메뉴에 카테고리별로 정리되어 있으며, 각 기능 페이지에 접근할 수 있는 것을 확인했습니다.
> **🧹 정리**: 이 실습에서는 리소스를 생성하지 않았으므로 삭제할 것이 없습니다.

> 🔬 **Deep Dive: Systems Manager의 하이브리드 환경 지원**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> Systems Manager의 흥미로운 특징 중 하나는 **AWS 밖의 서버도 관리할 수 있다**는 것입니다. 온프레미스(회사 데이터센터)에 있는 서버나 다른 클라우드의 서버에도 SSM Agent를 설치하면, AWS Systems Manager에서 함께 관리할 수 있습니다. 이를 **하이브리드 환경 관리**라고 합니다.
>
> 이 기능은 클라우드로의 전환 과정에서 특히 유용합니다. 회사가 아직 온프레미스 서버와 AWS를 함께 운영하고 있다면, Systems Manager 하나로 두 환경을 통합 관리할 수 있어 운영 복잡성을 줄일 수 있습니다.
>
> 💬 **참고**: 하이브리드 환경과 마이그레이션에 대해서는 Chapter 20에서 자세히 다룹니다.

> ⚠️ **주의**
>
> - **SSM Agent vs. Systems Manager**: SSM Agent는 EC2 인스턴스에 설치되는 소프트웨어이고, Systems Manager는 이를 통해 인스턴스를 관리하는 AWS 서비스입니다. 둘을 혼동하지 마세요.
> - **IAM 역할 누락**: EC2 인스턴스가 Systems Manager에 표시되지 않는 가장 흔한 원인은 IAM 역할이 연결되지 않은 것입니다. `AmazonSSMManagedInstanceCore` 정책이 포함된 역할이 반드시 필요합니다.
> - **Systems Manager ≠ 모니터링 도구**: Systems Manager는 "운영 관리" 도구이지, 성능 지표를 수집하고 경보를 설정하는 "모니터링" 도구가 아닙니다. 모니터링은 Chapter 18에서 배울 CloudWatch의 역할입니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 17.1.5: 대규모 보안 패치 긴급 적용**
>
> **상황**: 금요일 오후, Linux 커널에서 심각한 보안 취약점(CVE)이 발표되었습니다. 회사의 서울 리전에 운영 중인 EC2 인스턴스 120대에 긴급 패치를 적용해야 합니다.
>
> **풀이/설명**:
> 1. Patch Manager에서 해당 CVE에 대응하는 보안 패치를 포함한 **패치 기준선(Patch Baseline)**을 업데이트합니다.
> 2. `PatchGroup: Production` 태그가 붙은 인스턴스 그룹을 대상으로 지정합니다.
> 3. 유지 관리 기간(Maintenance Window)을 "오늘 밤 11시~새벽 2시"로 설정하여, 트래픽이 적은 시간대에 일괄 적용합니다.
> 4. 패치 적용 후, Patch Manager 대시보드에서 120대 모두 "Compliant(준수)" 상태인지 확인합니다.
>
> **핵심 포인트**: 120대의 서버에 하나하나 접속하지 않고, Patch Manager로 한 번에 패치를 적용하고 결과까지 확인할 수 있습니다. 긴급 보안 패치 상황에서 Systems Manager의 가치가 극대화됩니다.

> 🔍 **예시 17.1.6: 멀티 환경 설정 관리 — Parameter Store 활용**
>
> **상황**: 웹 애플리케이션이 개발(dev), 스테이징(staging), 운영(prod) 세 환경에서 실행됩니다. 각 환경마다 데이터베이스 접속 정보가 다릅니다.
>
> **풀이/설명**:
> 1. Parameter Store에 계층 구조로 파라미터를 생성합니다:
>    - `/myapp/dev/db/host` → 개발 DB 주소
>    - `/myapp/staging/db/host` → 스테이징 DB 주소
>    - `/myapp/prod/db/host` → 운영 DB 주소 (SecureString으로 암호화)
> 2. 각 환경의 EC2 인스턴스에 부여된 IAM 역할에 따라, 해당 환경의 파라미터만 읽을 수 있도록 접근을 제한합니다.
> 3. 운영 DB 비밀번호를 분기마다 변경할 때, Parameter Store의 값만 업데이트하면 애플리케이션 코드 변경 없이 적용됩니다.
>
> **핵심 포인트**: Parameter Store의 계층 구조와 IAM 정책을 결합하면, 환경별 설정을 안전하게 분리·관리할 수 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의: "Systems Manager = Session Manager"로 좁게 이해하는 실수**
>
> Systems Manager는 Session Manager, Patch Manager, Parameter Store, Run Command, Inventory 등 **여러 기능의 집합체**입니다. 시험에서 "원격 접속" 시나리오만 나오는 것이 아니라, "일괄 패치", "설정 관리" 등 다양한 시나리오로 출제됩니다.

- **SSM Agent가 설치되어 있는데 인스턴스가 보이지 않는 경우**: 가장 흔한 원인은 **IAM 역할 미연결**입니다. `AmazonSSMManagedInstanceCore` 정책이 포함된 IAM 역할이 인스턴스에 반드시 연결되어 있어야 합니다.
- **Parameter Store와 Secrets Manager를 혼동하지 마세요**: 두 서비스 모두 민감한 값을 저장하지만, Secrets Manager는 **자동 비밀번호 교체(Rotation)** 기능이 있고 비용이 발생합니다. Parameter Store의 기본 기능은 무료입니다.
- **Patch Manager는 "패치 설치"만 하는 것이 아닙니다**: 패치 기준선에 **맞지 않는** 인스턴스를 탐지하는 **스캔 전용 모드**도 있습니다. "설치"와 "스캔"을 구분하세요.
- **Systems Manager는 EC2 전용이 아닙니다**: 온프레미스 서버에도 SSM Agent를 설치하면 Systems Manager로 관리할 수 있습니다. 시험에서 "하이브리드 환경 관리"가 나오면 Systems Manager를 떠올리세요.

---

### Section 요약 (Section Summary)

- **AWS Systems Manager**는 EC2 인스턴스를 중앙에서 관리하는 **통합 운영 허브**이다.
- **Session Manager**를 사용하면 SSH 포트를 열지 않고도 안전하게 서버에 접속할 수 있다.
- **Patch Manager**로 수백 대의 서버에 운영 체제 패치를 자동으로 일괄 적용할 수 있다.
- **Parameter Store**에 비밀번호, API 키 등 민감한 설정 값을 암호화하여 안전하게 보관할 수 있다.
- Systems Manager를 사용하려면 EC2 인스턴스에 **SSM Agent**가 설치되어 있고 **IAM 역할**이 연결되어 있어야 한다.

## 17.2 AWS Trusted Advisor — 모범 사례 자동 점검

> 💡 **한 줄 요약**: AWS Trusted Advisor는 당신의 AWS 계정을 자동으로 검사하여, 비용 절감·보안 강화·성능 개선·복원력 향상·운영 우수성·서비스 한도 등 6가지 관점의 개선 사항을 추천하는 **자동 컨설턴트**입니다.

### 직관적 이해 (Intuitive Understanding)

건강 검진을 떠올려 봅시다. 우리는 정기적으로 병원에 가서 혈압, 혈당, 콜레스테롤 등 다양한 수치를 검사받습니다. 의사는 검사 결과를 보고 "혈압이 좀 높으니 염분 섭취를 줄이세요", "콜레스테롤은 정상이니 잘 유지하세요"와 같이 조언해 줍니다.

**AWS Trusted Advisor**는 AWS 계정의 "건강 검진" 서비스입니다. 당신의 AWS 환경을 여러 관점에서 자동으로 점검하고, 모범 사례(Best Practices)에 비추어 개선이 필요한 항목을 알려줍니다. 마치 AWS 전문 컨설턴트가 옆에서 "이 부분은 이렇게 바꾸면 더 좋습니다"라고 조언해 주는 것과 같습니다.

**왜 이것이 필요할까요?**

AWS를 처음 사용하는 사람이든, 오랜 경험자든, "내가 놓치고 있는 것이 없을까?"라는 걱정은 항상 있습니다. 불필요한 리소스에 비용을 낭비하고 있지는 않은지, 보안 설정에 구멍이 있지는 않은지, 서비스 한도에 근접하고 있지는 않은지 — 이런 것들을 일일이 확인하는 것은 매우 번거롭습니다. Trusted Advisor는 이 모든 것을 자동으로 점검해 줍니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **AWS Trusted Advisor**: AWS 계정의 리소스를 자동으로 검사하여, 비용 최적화·성능·보안·내결함성(복원력)·운영 우수성·서비스 한도 등 6가지 카테고리에서 모범 사례 위반 사항과 개선 권장 사항을 제공하는 서비스입니다.

#### 6가지 점검 카테고리

Trusted Advisor는 다음 **6가지 카테고리(Pillar)** 로 점검을 수행합니다. 각 카테고리가 무엇을 점검하는지 살펴봅시다:

**1. 비용 최적화 (Cost Optimization)** 💰

"돈을 낭비하고 있지는 않은가?"를 점검합니다.

- 사용하지 않는 EC2 인스턴스가 켜져 있지 않은지
- 연결되지 않은 EBS 볼륨(데이터 저장소)이 비용을 발생시키고 있지 않은지
- 예약 인스턴스(Reserved Instance)를 활용하면 절약할 수 있는 부분이 있는지

**2. 성능 (Performance)** ⚡

"리소스가 효율적으로 사용되고 있는가?"를 점검합니다.

- EC2 인스턴스의 사양이 실제 사용량에 비해 과다하거나 부족하지 않은지
- 데이터 전송 경로가 최적인지

**3. 보안 (Security)** 🔒

"보안 구멍이 있지 않은가?"를 점검합니다.

- 보안 그룹(Security Group)에서 무제한 접근(0.0.0.0/0)을 허용하고 있지 않은지
- IAM 사용자에게 MFA가 설정되어 있는지
- S3 버킷이 실수로 퍼블릭 접근이 가능한 상태가 아닌지
- 루트 사용자에게 MFA가 설정되어 있는지

> 🔗 **연결**: Chapter 13에서 배운 **MFA(다중 인증)** 설정을 기억하시나요? Trusted Advisor의 보안 점검에서 "루트 계정 MFA"와 "IAM 사용자 MFA" 항목은 가장 기본적인 점검 항목 중 하나입니다.

**4. 내결함성/복원력 (Fault Tolerance / Resilience)** 🛡️

"장애에 대비하고 있는가?"를 점검합니다.

- 중요한 데이터의 백업이 설정되어 있는지
- 서비스가 단일 가용 영역(AZ)에만 배포되어 있지 않은지

**5. 운영 우수성 (Operational Excellence)** ⚙️

"운영이 효율적이고 자동화되어 있는가?"를 점검합니다.

- 로깅과 모니터링이 적절히 설정되어 있는지
- 운영 자동화가 가능한 부분이 있는지

**6. 서비스 한도 (Service Limits)** 📊

"서비스 사용량이 한도에 근접하고 있지 않은가?"를 점검합니다.

- 각 AWS 서비스에는 기본 한도(예: 리전당 EC2 인스턴스 최대 수, VPC 최대 수 등)가 있습니다.
- 이 한도의 80% 이상을 사용하고 있다면 경고를 표시합니다.

> 🔍 **예시 17.2.1: 건강 검진 결과표로 이해하는 Trusted Advisor**
>
> **상황**: 김 대리는 회사의 AWS 계정을 관리하고 있습니다. 최근 Trusted Advisor를 처음 확인했더니 다음과 같은 결과가 나왔습니다.
> **풀이/설명**:
> | 카테고리 | 점검 항목 | 상태 | 의미 |
> |---------|----------|------|------|
> | 💰 비용 최적화 | 사용률 낮은 EC2 인스턴스 | 🟡 경고 | 지난 14일간 CPU 사용률이 10% 미만인 인스턴스 2개 발견 |
> | 🔒 보안 | 루트 계정 MFA | 🔴 위험 | 루트 계정에 MFA가 설정되지 않음 |
> | 🔒 보안 | 보안 그룹 — 무제한 접근 | 🟡 경고 | 포트 22(SSH)에 0.0.0.0/0 허용 중인 보안 그룹 1개 |
> | 📊 서비스 한도 | VPC 수 | 🟢 정상 | 5개 중 2개 사용 (40%) |
>
> 김 대리는 가장 시급한 🔴 항목(루트 MFA)부터 조치하기로 했습니다.
> **핵심 포인트**: Trusted Advisor는 🟢(정상), 🟡(주의/경고), 🔴(위험/조치 필요)로 상태를 표시하여, 우선순위를 쉽게 판단할 수 있게 합니다.

> 🔍 **예시 17.2.2: 무료 점검 vs. 유료 점검**
>
> **상황**: "Trusted Advisor의 모든 점검을 무료로 사용할 수 있는가?"
> **풀이/설명**:
> 아닙니다. Trusted Advisor의 점검 항목 중 **전체 점검 기능**을 사용하려면 **Business 지원 플랜 이상**이 필요합니다.
>
> | 지원 플랜 | 사용 가능한 Trusted Advisor 점검 |
> |----------|-------------------------------|
> | Basic / Developer | **핵심 보안 점검**(7개)과 **서비스 한도** 점검만 가능 |
> | Business 이상 | **6가지 카테고리의 전체 점검** 사용 가능 |
>
> 무료(Basic/Developer)로 사용할 수 있는 핵심 보안 점검 예시:
> - S3 버킷 퍼블릭 접근 여부
> - 보안 그룹의 무제한 포트 개방 여부
> - IAM 사용자의 MFA 설정 여부
> - 루트 계정 MFA 설정 여부
> - EBS 퍼블릭 스냅샷 여부
> - RDS 퍼블릭 스냅샷 여부
> - 서비스 한도 점검
>
> **핵심 포인트**: CCP 시험에서 "Trusted Advisor의 전체 기능을 사용하려면 어떤 지원 플랜이 필요한가?"라는 문제가 자주 출제됩니다. 정답은 **Business 지원 플랜 이상**입니다.

> 💬 **참고**: AWS 지원 플랜(Basic, Developer, Business, Enterprise On-Ramp, Enterprise)에 대한 자세한 내용은 Chapter 23에서 다룹니다. 여기서는 Trusted Advisor와의 관계만 기억해 두세요.

> 🧪 **Mini Lab 17.2: Trusted Advisor 점검 결과 확인하기**
>
> **목표**: Trusted Advisor에 접속하여 무료로 제공되는 핵심 보안 점검 결과를 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `Trusted Advisor`를 입력하고 클릭합니다.
> **Step 2**: Trusted Advisor 대시보드가 표시됩니다. 6가지 카테고리(비용 최적화, 성능, 보안, 내결함성, 운영 우수성, 서비스 한도)별 요약 상태가 표시됩니다.
> **Step 3**: **"Security"(보안)** 카테고리를 클릭합니다. Basic/Developer 플랜에서도 확인 가능한 핵심 보안 점검 항목들이 표시됩니다.
> **Step 4**: 점검 항목 중 하나(예: "MFA on Root Account")를 클릭하여 상세 내용을 확인합니다.
>   - 🟢 녹색이면: 루트 계정에 MFA가 설정된 상태입니다.
>   - 🔴 빨간색이면: 루트 계정에 MFA가 미설정 상태이므로 Chapter 4에서 배운 MFA 설정을 권장합니다.
> **Step 5**: **"Service Limits"(서비스 한도)** 카테고리를 클릭하여 현재 계정의 서비스 사용률을 확인합니다. 각 서비스별로 현재 사용량과 한도가 표시됩니다.
>
> **확인**: Trusted Advisor가 AWS 계정의 보안 상태와 서비스 한도를 자동으로 점검하고, 색상 코드(🟢🟡🔴)로 상태를 표시하는 것을 확인했습니다.
> **🧹 정리**: 이 실습에서는 리소스를 생성하지 않았으므로 삭제할 것이 없습니다.

> ⚠️ **주의**
>
> - **Trusted Advisor ≠ 실시간 모니터링**: Trusted Advisor는 "현재 상태의 스냅샷"을 기반으로 점검합니다. 실시간 지표 감시와 경보는 CloudWatch(Chapter 18)의 역할입니다.
> - **지원 플랜 혼동 주의**: "Trusted Advisor의 전체 점검에는 Business 이상이 필요하다"는 것은 CCP 시험의 빈출 포인트입니다. Basic/Developer 플랜에서는 핵심 보안 점검과 서비스 한도 점검만 가능합니다.
> - **추천 = 자동 조치가 아님**: Trusted Advisor는 "이렇게 하면 좋겠습니다"라고 추천만 할 뿐, 자동으로 리소스를 변경하지는 않습니다. 실제 조치는 사용자가 직접 해야 합니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 17.2.3: 비용 절감을 위한 Trusted Advisor 활용**
>
> **상황**: 스타트업 C사의 월 AWS 청구서가 $3,000에서 $5,000으로 갑자기 증가했습니다. CTO가 원인을 파악하고 비용을 절감하고 싶습니다.
>
> **풀이/설명**:
> 1. CTO가 Trusted Advisor의 **"비용 최적화"** 카테고리를 확인합니다 (Business 지원 플랜 이상 필요).
> 2. 🟡 "사용률 낮은 EC2 인스턴스" 경고: CPU 사용률이 5% 미만인 m5.xlarge 인스턴스 3개 발견 → t3.medium으로 변경하면 월 $800 절약 가능.
> 3. 🟡 "연결되지 않은 Elastic IP" 경고: 사용하지 않는 Elastic IP 2개가 시간당 과금 중 → 해제하면 월 $15 절약.
> 4. 🟢 "예약 인스턴스 최적화" 권장: 24시간 실행 중인 인스턴스에 대해 예약 인스턴스를 구매하면 월 $600 추가 절약 가능.
>
> **핵심 포인트**: Trusted Advisor는 구체적인 금액까지 포함하여 절감 방안을 제시합니다. 정기적으로 확인하면 불필요한 비용 낭비를 방지할 수 있습니다.

> 🔍 **예시 17.2.4: 서비스 한도 초과 방지**
>
> **상황**: 개발팀이 새로운 마이크로서비스를 배포하기 위해 EC2 인스턴스를 대량으로 생성하려 합니다. 그런데 인스턴스 생성이 실패합니다.
>
> **풀이/설명**:
> 1. Trusted Advisor의 **"서비스 한도"** 카테고리를 확인합니다.
> 2. 🔴 "EC2 On-Demand 인스턴스 한도" 항목이 위험 상태: 현재 한도의 95%를 사용 중.
> 3. AWS Support에 **서비스 한도 증가 요청(Service Quota increase request)**을 제출합니다.
> 4. 향후 이런 상황을 예방하기 위해, Trusted Advisor의 서비스 한도 점검을 정기적으로 확인하도록 알림을 설정합니다.
>
> **핵심 포인트**: 서비스 한도에 도달하면 리소스를 더 이상 생성할 수 없습니다. Trusted Advisor의 서비스 한도 점검은 **무료(Basic) 플랜**에서도 사용 가능하므로, 반드시 활용하세요.

---

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의: "Trusted Advisor는 무료로 전부 사용할 수 있다"는 오해**
>
> Trusted Advisor의 **전체 점검**(6가지 카테고리 모두)을 사용하려면 **Business 지원 플랜 이상**이 필요합니다. Basic/Developer 플랜에서는 핵심 보안 점검(7개)과 서비스 한도 점검만 가능합니다.

- **Trusted Advisor와 AWS Config를 혼동하지 마세요**: Trusted Advisor는 "모범 사례에 따른 **권장 사항을 제공**"하고, AWS Config는 "리소스 구성의 **변경 이력을 기록**"합니다. 목적이 다릅니다.
- **Trusted Advisor와 Health Dashboard를 혼동하지 마세요**: Trusted Advisor는 "내 계정 설정이 모범 사례를 따르는가?"를 점검하고, Health Dashboard는 "AWS 서비스에 장애가 있는가?"를 알려줍니다.
- **점검 결과를 맹신하지 마세요**: Trusted Advisor가 "사용률 낮은 인스턴스"라고 표시해도, 해당 인스턴스가 **배치 작업용**(특정 시간에만 높은 사용률)일 수 있습니다. 비즈니스 맥락을 고려하여 판단하세요.
- **"추천"과 "자동 조치"를 혼동하지 마세요**: Trusted Advisor는 개선 사항을 **추천만** 할 뿐, 리소스를 자동으로 변경하지 않습니다. 실제 조치는 사용자가 직접 해야 합니다.

---

### Section 요약 (Section Summary)

- **AWS Trusted Advisor**는 AWS 계정을 **6가지 카테고리**(비용 최적화, 성능, 보안, 내결함성, 운영 우수성, 서비스 한도)로 자동 점검하는 서비스이다.
- 점검 결과는 🟢(정상), 🟡(경고), 🔴(위험) 색상 코드로 표시된다.
- **Basic/Developer 플랜**에서는 핵심 보안 점검과 서비스 한도 점검만 가능하고, **전체 점검**에는 **Business 이상** 플랜이 필요하다.
- Trusted Advisor는 추천만 할 뿐 자동으로 리소스를 변경하지 않는다.

## 17.3 AWS Health Dashboard — 서비스 상태 확인

> 💡 **한 줄 요약**: AWS Health Dashboard는 AWS 서비스의 전반적인 상태와 **내 계정에 영향을 미치는** 이벤트를 실시간으로 확인할 수 있는 상태 페이지입니다.

### 직관적 이해 (Intuitive Understanding)

지하철을 타고 출근한다고 가정해 봅시다. 지하철 앱에는 두 가지 정보가 있습니다:

1. **전체 노선 운행 상태**: "2호선 정상 운행 중", "3호선 지연 운행 중" — 모든 노선의 현재 상태
2. **내 즐겨찾기 노선 알림**: "당신이 이용하는 2호선 강남역에서 5분 지연이 발생했습니다" — 나에게 직접 영향을 미치는 정보

AWS Health Dashboard도 이와 같은 두 가지 레벨의 정보를 제공합니다.

**왜 이것이 필요할까요?**

"내 웹사이트가 갑자기 느려졌는데, 이게 내 코드 문제인지 AWS 인프라 문제인지 모르겠어!" — 이런 상황에서 가장 먼저 확인해야 할 곳이 바로 AWS Health Dashboard입니다. AWS 서비스 자체에 장애가 발생한 것이라면, 내 코드를 아무리 수정해도 소용이 없겠죠.

### 핵심 개념 설명 (Core Explanation)

AWS Health Dashboard는 두 가지 대시보드로 구성됩니다:

#### AWS Health Dashboard — Service health (서비스 상태)

> 📝 **Service Health (구: AWS Service Health Dashboard)**: AWS의 모든 서비스가 각 리전에서 정상적으로 작동하고 있는지를 보여주는 **공개** 대시보드입니다. AWS 계정이 없어도 누구나 확인할 수 있습니다.

이 대시보드는 전체 AWS 서비스의 "교통 상황판"과 같습니다:
- 각 리전별, 서비스별 현재 운영 상태를 표시합니다.
- 과거 장애 이력도 확인할 수 있습니다.
- 서비스 장애, 성능 저하, 유지보수 등의 이벤트를 공지합니다.

#### AWS Health Dashboard — Your account health (계정 상태)

> 📝 **Your Account Health (구: Personal Health Dashboard, PHD)**: **내 AWS 계정**의 리소스에 직접 영향을 미치는 이벤트만을 선별하여 보여주는 **개인화된** 대시보드입니다. AWS 계정에 로그인해야 확인할 수 있습니다.

이것이 Health Dashboard의 진짜 가치입니다. 수백 개의 AWS 서비스 중에서, **내가 실제로 사용하고 있는 서비스**에 영향을 미치는 이벤트만 필터링해서 보여줍니다.

| 비교 항목 | Service Health | Your Account Health |
|-----------|---------------|-------------------|
| 대상 | AWS 전체 서비스 상태 | **내 계정**에 영향을 미치는 이벤트 |
| 접근 | 누구나 (로그인 불필요) | AWS 계정 로그인 필요 |
| 범위 | 모든 리전, 모든 서비스 | 내가 사용하는 리전과 서비스만 |
| 알림 | 없음 (직접 확인) | **이벤트 알림** 설정 가능 |
| 예시 | "US-East-1 리전의 EC2에 장애 발생" | "서울 리전에서 당신이 사용 중인 EC2 인스턴스 i-0abc123에 영향" |

> 🔍 **예시 17.3.1: 웹사이트 장애 시 Health Dashboard 활용**
>
> **상황**: 이 대리가 운영하는 웹사이트가 갑자기 502 에러를 표시합니다. 서울 리전(ap-northeast-2)의 EC2 인스턴스에서 서비스하고 있습니다.
> **풀이/설명**:
> 1. 이 대리는 먼저 AWS 콘솔에 로그인하여 **Health Dashboard > Your account health**를 확인합니다.
> 2. "서울 리전의 EC2 서비스에서 네트워크 연결 이슈가 발생했습니다"라는 알림이 표시되어 있습니다.
> 3. 이 대리는 "내 코드 문제가 아니라 AWS 인프라 이슈구나"라고 판단합니다.
> 4. 이벤트 상세 정보에서 AWS가 제공하는 예상 복구 시간과 권장 조치를 확인합니다.
> 5. 고객에게 "AWS 인프라 이슈로 인한 일시적 장애이며, AWS에서 복구 작업 중"이라고 안내합니다.
> **핵심 포인트**: Health Dashboard는 장애의 원인이 내 쪽인지 AWS 쪽인지를 빠르게 판단하는 첫 번째 도구입니다.

> 🔍 **예시 17.3.2: 예방적 유지보수 알림**
>
> **상황**: AWS에서 하드웨어 유지보수를 위해 특정 EC2 인스턴스의 호스트 서버를 교체할 예정입니다.
> **풀이/설명**:
> - **Service Health**에서는 이 정보가 표시되지 않습니다 — 이것은 전체 서비스 장애가 아니라, 특정 인스턴스에만 해당하는 이벤트이기 때문입니다.
> - **Your Account Health**에서는 "당신의 인스턴스 i-0abc123이 2026년 4월 15일에 유지보수 예정입니다. 그 전에 인스턴스를 직접 재시작하면 자동으로 새 호스트로 이전됩니다"라는 알림이 표시됩니다.
> **핵심 포인트**: Your Account Health는 내 리소스에 "앞으로 영향을 미칠 수 있는" 예정된 이벤트도 사전에 알려줍니다.

> 🧪 **Mini Lab 17.3: AWS Health Dashboard 확인하기**
>
> **목표**: Service Health와 Your Account Health 두 가지 대시보드를 각각 확인하고, 차이를 체험합니다.
> **예상 소요 시간**: 5분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `Health`를 입력하고 **AWS Health Dashboard**를 클릭합니다.
> **Step 2**: Health Dashboard가 열리면, 왼쪽 메뉴에서 **"Your account health"** 를 클릭합니다.
>   - 내 계정에 영향을 미치는 이벤트가 있으면 목록으로 표시됩니다.
>   - 이벤트가 없으면 "No recent issues" 또는 유사한 메시지가 표시됩니다 — 이것은 좋은 소식입니다!
> **Step 3**: 왼쪽 메뉴에서 **"Service health"** 를 클릭합니다.
>   - AWS 전체 서비스의 현재 상태가 표시됩니다.
>   - 리전별, 서비스별 상태를 확인할 수 있습니다.
> **Step 4**: 두 화면의 차이를 비교합니다: Service Health는 "AWS 전체"의 상태를, Your Account Health는 "내 계정"에 영향을 미치는 이벤트만 보여줍니다.
>
> **확인**: Service Health는 AWS 전체 서비스 상태를, Your Account Health는 내 계정에 관련된 이벤트만을 선별하여 보여주는 것을 확인했습니다.
> **🧹 정리**: 이 실습에서는 리소스를 생성하지 않았으므로 삭제할 것이 없습니다.

> ⚠️ **주의**
>
> - **이름 변경에 주의**: 과거에는 "AWS Service Health Dashboard"와 "Personal Health Dashboard(PHD)"라는 별도 서비스로 존재했지만, 현재는 **AWS Health Dashboard** 하나로 통합되어 그 안에 "Service health"와 "Your account health"가 있습니다. CCP 시험에서 두 이름 모두 출제될 수 있으니 양쪽 다 기억해 두세요.
> - **Health Dashboard ≠ Trusted Advisor**: Health Dashboard는 "AWS 서비스에 장애나 이벤트가 있는가?"를 알려주고, Trusted Advisor는 "내 계정이 모범 사례를 따르고 있는가?"를 점검합니다. 목적이 다릅니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 17.3.3: 서비스 장애와 내 계정 영향 구분하기**
>
> **상황**: 운영팀이 관리하는 서울 리전의 RDS 데이터베이스가 갑자기 연결 오류를 발생시킵니다. 애플리케이션 로그에는 "Connection refused" 에러가 반복됩니다.
>
> **풀이/설명**:
> 1. 운영팀이 **Health Dashboard > Your account health**를 확인합니다.
> 2. "서울 리전 RDS 서비스에서 일부 인스턴스에 연결 문제가 발생하고 있습니다"라는 알림이 표시됩니다.
> 3. 알림의 상세 내용에 영향을 받는 RDS 인스턴스의 식별자와 예상 복구 시간이 명시되어 있습니다.
> 4. 운영팀은 이것이 AWS 인프라 수준의 문제임을 확인하고, 자체 코드 수정 대신 AWS의 복구를 기다리기로 결정합니다.
> 5. 한편 **Service health**를 확인하니, RDS 서비스 전체가 아닌 특정 가용 영역에서만 이슈가 발생한 것으로 나타납니다.
>
> **핵심 포인트**: Your account health는 "내 리소스에 영향을 미치는" 이벤트만 보여주므로, 장애 대응 시 가장 먼저 확인해야 하는 도구입니다. Service health와 비교하면 영향 범위를 더 정확히 파악할 수 있습니다.

> 🔍 **예시 17.3.4: EventBridge와 연동한 자동 알림 설정**
>
> **상황**: DevOps 팀이 AWS 유지보수 알림을 이메일이 아닌 Slack 채널로 자동 전송하고 싶습니다.
>
> **풀이/설명**:
> 1. **Amazon EventBridge**에서 AWS Health 이벤트를 소스로 하는 규칙을 생성합니다.
> 2. 이벤트 패턴을 "서울 리전의 EC2, RDS 관련 이벤트"로 제한합니다.
> 3. 대상(Target)으로 SNS 토픽을 지정하고, 해당 토픽에 Slack Webhook을 연동합니다.
> 4. EC2 인스턴스 유지보수가 예정되면, Slack 채널에 자동으로 알림이 전송됩니다.
>
> **핵심 포인트**: Health Dashboard는 콘솔에서 수동으로 확인하는 것뿐 아니라, EventBridge와 연동하여 **자동화된 알림 파이프라인**을 구축할 수 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의: "Service Health Dashboard"와 "Personal Health Dashboard"는 과거 명칭입니다**
>
> 현재는 **AWS Health Dashboard** 하나로 통합되어, 그 안에 "Service health"와 "Your account health"가 있습니다. CCP 시험에서 과거 명칭으로 출제될 수 있으므로, 두 이름 모두 기억하세요.

- **Health Dashboard에 아무 이벤트도 없다고 해서 내 시스템에 문제가 없는 것은 아닙니다**: Health Dashboard는 **AWS 인프라 수준의 이벤트**만 표시합니다. 애플리케이션 코드 버그나 잘못된 설정으로 인한 장애는 여기에 나타나지 않습니다.
- **Service health와 Your account health의 차이를 혼동하지 마세요**: Service health는 "AWS 전체 서비스 상태"(누구나 접근 가능), Your account health는 "내 계정에 영향을 미치는 이벤트만 필터링"(로그인 필요)입니다.
- **Health Dashboard에서 알림을 받으려면 별도 설정이 필요합니다**: 기본적으로 알림은 자동 전송되지 않습니다. Amazon EventBridge 또는 Health API를 활용하여 알림을 설정해야 합니다.
- **Health Dashboard는 장애 "알림" 도구이지, 장애 "해결" 도구가 아닙니다**: Health Dashboard는 이슈를 알려줄 뿐, 자동으로 문제를 해결하지 않습니다. 문제 해결은 사용자의 조치 또는 AWS의 복구를 기다려야 합니다.

---

### Section 요약 (Section Summary)

- **AWS Health Dashboard**는 AWS 서비스 상태를 확인하는 두 가지 뷰를 제공한다: **Service health**(전체 서비스 상태)와 **Your account health**(내 계정에 영향을 미치는 이벤트).
- **Your Account Health**가 더 실용적이다 — 내가 사용하는 서비스에 대한 이벤트만 필터링하여 보여주고, 알림 설정도 가능하다.
- 서비스 장애 발생 시, 원인이 AWS 인프라인지 판단하는 **첫 번째 확인 도구**로 활용된다.
- 과거 명칭(Service Health Dashboard, Personal Health Dashboard)도 함께 기억해 두어야 한다.

## 17.4 태그와 리소스 관리

> 💡 **한 줄 요약**: **태그(Tag)** 는 AWS 리소스에 붙이는 이름표로, 리소스를 팀별·프로젝트별·환경별로 분류하고 비용을 추적하며 자동화 정책을 적용하는 데 핵심적인 역할을 합니다.

### 직관적 이해 (Intuitive Understanding)

도서관을 상상해 봅시다. 도서관에 책이 10권이라면 별도의 분류 체계 없이도 원하는 책을 찾을 수 있습니다. 하지만 책이 10만 권이라면? **분류 체계(장르별, 저자별, 출판 연도별 등)와 레이블(라벨)** 이 없으면 원하는 책을 찾는 것이 불가능에 가깝습니다.

AWS 리소스도 마찬가지입니다. 처음에는 EC2 인스턴스 한두 개, S3 버킷 하나로 시작하지만, 시간이 지나면 수십, 수백 개의 리소스가 만들어집니다. "이 EC2 인스턴스는 어떤 팀이 사용하는 거지?", "이 비용은 어떤 프로젝트에서 발생한 거지?", "개발 환경과 운영 환경의 리소스를 어떻게 구분하지?" — 이런 질문에 답할 수 있게 해주는 것이 바로 **태그(Tag)** 입니다.

**왜 이것이 필요할까요?**

태그가 없으면 다음과 같은 문제가 발생합니다:
- **비용 미스터리**: 이번 달 AWS 청구서가 예상보다 높은데, 어떤 팀의 어떤 프로젝트에서 비용이 발생했는지 파악할 수 없습니다.
- **리소스 미아**: EC2 인스턴스가 30개 있는데, 어떤 것이 운영 중이고 어떤 것이 테스트용인지 알 수 없습니다.
- **관리 혼란**: 보안 패치를 적용해야 하는데, 운영 서버만 선별적으로 업데이트하고 싶은데 구분할 방법이 없습니다.

### 핵심 개념 설명 (Core Explanation)

#### 태그(Tag)란?

> 📝 **태그(Tag)**: AWS 리소스에 부여하는 **키-값(Key-Value) 쌍**의 메타데이터(정보 라벨). 예를 들어 `Environment: Production`, `Team: Backend`, `Project: WebApp`과 같은 형태입니다.

태그는 매우 간단한 구조입니다:

```
태그 키(Key)     :  태그 값(Value)
─────────────       ──────────────
Environment     :  Production
Team            :  Backend
Project         :  WebApp
CostCenter      :  CC-1234
```

하나의 리소스에 여러 개의 태그를 부여할 수 있습니다 (최대 50개). 태그는 단순한 이름표이지만, AWS의 여러 기능과 결합하면 강력한 관리 도구가 됩니다.

#### 태그의 활용 분야

**1. 리소스 분류 및 식별**

수백 개의 리소스 중에서 특정 조건에 맞는 리소스를 빠르게 찾을 수 있습니다.

```
예: "Environment: Production" 태그가 붙은 모든 리소스 조회
    → 운영 환경의 EC2, RDS, S3 등이 한눈에 표시됨
```

**2. 비용 할당 (Cost Allocation)**

태그를 **비용 할당 태그(Cost Allocation Tag)** 로 활성화하면, AWS 청구서에서 태그별로 비용을 분류하여 볼 수 있습니다.

```
예: "Project: WebApp" 태그를 비용 할당 태그로 활성화
    → 월별 청구서에서 WebApp 프로젝트에 발생한 비용만 따로 확인 가능
```

> 📝 **비용 할당 태그(Cost Allocation Tag)**: AWS 청구서에서 태그 값별로 비용을 분류할 수 있도록 활성화한 태그. AWS 생성 태그(예: `aws:createdBy`)와 사용자 정의 태그 두 종류가 있습니다. Billing 콘솔에서 활성화해야 비용 보고서에 반영됩니다.

**3. 접근 제어**

> 🔗 **연결**: Chapter 13에서 배운 **IAM 정책의 Condition(조건)** 요소를 기억하시나요? IAM 정책에서 태그를 조건으로 사용하면, "특정 태그가 붙은 리소스에만 접근 허용"과 같은 세밀한 접근 제어가 가능합니다.

```
예: IAM 정책에 "Team: Backend 태그가 있는 EC2 인스턴스만 시작/중지 가능" 조건 설정
    → Backend 팀 멤버는 자기 팀의 서버만 관리할 수 있음
```

**4. 자동화**

Systems Manager의 Patch Manager에서 "PatchGroup: WebServers" 태그가 붙은 인스턴스에만 패치를 적용하는 등, 자동화 작업의 대상을 태그로 지정할 수 있습니다.

#### 태그 전략 (Tagging Strategy)

효과적인 태그 관리를 위해서는 조직 전체가 따르는 **태그 전략**을 수립해야 합니다.

| 태그 키 | 용도 | 예시 값 |
|---------|------|---------|
| `Environment` | 환경 구분 | `Production`, `Staging`, `Development` |
| `Team` | 소유 팀 | `Backend`, `Frontend`, `DevOps` |
| `Project` | 프로젝트 | `WebApp`, `MobileAPI`, `DataPipeline` |
| `CostCenter` | 비용 부서 코드 | `CC-1001`, `CC-2003` |
| `Owner` | 담당자 | `kim@company.com` |
| `ManagedBy` | 관리 방식 | `Manual`, `Terraform`, `CloudFormation` |

> 🔍 **예시 17.4.1: 태그 없는 세계 vs. 태그 있는 세계**
>
> **상황**: 회사에 EC2 인스턴스가 20개 있습니다. 이번 달 EC2 비용이 예상보다 $500 더 나왔습니다.
> **풀이/설명**:
> - **태그 없는 세계**: "어떤 인스턴스가 추가 비용을 발생시킨 거지?" — 20개를 하나하나 확인해야 합니다. 누가 만들었는지, 어떤 용도인지, 아직 필요한지도 알 수 없습니다.
> - **태그 있는 세계**: `Project: WebApp`으로 필터링하니 8개, `Project: MobileAPI`로 필터링하니 10개, 태그가 없는 것이 2개 발견됩니다. 비용 할당 태그 보고서를 확인하니, `Project: MobileAPI`에서 $450 증가가 확인됩니다. Mobile 팀에 문의하니, 지난주에 부하 테스트를 위해 인스턴스를 추가로 실행하고 끄지 않았던 것이 원인이었습니다.
> **핵심 포인트**: 태그가 있으면 "누가, 왜, 얼마나" 비용을 사용했는지 **빠르게 추적**할 수 있습니다.

> 🔍 **예시 17.4.2: 도서관 라벨링 시스템으로 이해하는 태그**
>
> **상황**: 도서관의 분류 체계를 AWS 태그와 대응시켜 봅시다.
> **풀이/설명**:
> | 도서관 | AWS 태그 |
> |--------|---------|
> | 장르 라벨: "소설" | `Environment: Production` |
> | 저자 라벨: "김작가" | `Team: Backend` |
> | 출판년도 라벨: "2024" | `Project: WebApp` |
> | "추천도서" 표시 | `Priority: High` |
> | 대출 가능 여부 표시 | `Status: Active` |
>
> 도서관에서 "2024년에 출판된 소설 중 추천도서"를 찾으려면, 세 가지 라벨을 조합하여 검색합니다. 마찬가지로 AWS에서 "운영 환경의 Backend 팀 리소스 중 우선순위가 높은 것"을 찾으려면 태그를 조합합니다.
> **핵심 포인트**: 태그는 AWS 리소스의 "도서관 분류 체계"이며, 여러 태그를 조합하면 원하는 리소스를 정확히 찾을 수 있습니다.

#### Resource Groups — 태그 기반 리소스 묶기

> 📝 **AWS Resource Groups**: 공통 태그를 가진 리소스들을 하나의 그룹으로 묶어 한눈에 조회하고 일괄 관리할 수 있는 서비스입니다.

태그를 개별 리소스에 붙이는 것이 "책에 라벨을 붙이는 것"이라면, Resource Groups는 "같은 라벨이 붙은 책을 한 서가에 모아 놓는 것"입니다.

예를 들어, "Project: WebApp" 태그가 붙은 모든 리소스(EC2 인스턴스 3개, RDS 데이터베이스 1개, S3 버킷 2개)를 하나의 Resource Group으로 묶으면, 이 그룹 하나를 조회하는 것만으로 WebApp 프로젝트의 전체 리소스를 파악할 수 있습니다.

#### Tag Editor — 태그를 대량으로 관리

**Tag Editor**는 여러 리소스의 태그를 한꺼번에 조회하고 편집할 수 있는 도구입니다. EC2 인스턴스 100개에 `CostCenter: CC-1001` 태그를 한 번에 추가하거나, 특정 태그 값을 일괄 변경할 때 유용합니다.

> 🧪 **Mini Lab 17.4: Tag Editor로 태그 현황 확인하기**
>
> **목표**: Tag Editor에 접속하여 현재 AWS 계정의 리소스 태그 현황을 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS Management Console에 로그인한 후, 상단 검색창에 `Resource Groups`를 입력하고 **Resource Groups & Tag Editor**를 클릭합니다.
> **Step 2**: 왼쪽 메뉴에서 **"Tag Editor"** 를 클릭합니다.
> **Step 3**: "Find resources to tag" 영역에서:
>   - **Regions**: `ap-northeast-2` (서울)을 선택합니다.
>   - **Resource types**: `All supported resource types`를 선택합니다.
>   - **Tags (optional)**: 비워둡니다 (전체 리소스를 조회하기 위해).
>   - **"Search resources"** 버튼을 클릭합니다.
> **Step 4**: 서울 리전에 있는 리소스 목록이 표시됩니다. 각 리소스에 어떤 태그가 붙어 있는지(또는 태그가 없는지) 확인합니다.
> **Step 5**: (선택) 태그가 없는 리소스를 선택하고 **"Manage tags of selected resources"** 를 클릭하면, 선택한 리소스에 태그를 추가할 수 있는 화면이 표시됩니다. 실제로 태그를 추가하지 않아도, 이 기능이 어떻게 동작하는지 확인합니다.
>
> **확인**: Tag Editor를 통해 리전 내 전체 리소스의 태그 현황을 한눈에 조회할 수 있고, 여러 리소스에 일괄적으로 태그를 추가/수정할 수 있는 것을 확인했습니다.
> **🧹 정리**: 태그를 추가하지 않았다면 정리할 것이 없습니다. 태그를 추가했다면, 불필요한 태그를 삭제하여 원래 상태로 되돌려 놓습니다.

> ⚠️ **주의**
>
> - **태그는 대소문자를 구분합니다**: `Environment`와 `environment`는 서로 다른 태그 키입니다. 조직 전체에서 일관된 대소문자 규칙을 정해야 합니다.
> - **태그는 자동으로 붙지 않습니다**: 리소스를 생성할 때 태그를 지정하지 않으면 태그 없이 생성됩니다. 태그 없는 리소스는 비용 추적과 관리가 어렵습니다. AWS Organizations의 **태그 정책(Tag Policy)** 을 사용하면 필수 태그를 강제할 수 있습니다.
> - **비용 할당 태그는 활성화해야 작동합니다**: 태그를 리소스에 붙이는 것만으로는 청구서에서 비용이 분류되지 않습니다. Billing 콘솔에서 해당 태그를 **비용 할당 태그로 활성화**해야 합니다. 또한 활성화 이후 발생하는 비용부터 적용되며, 소급 적용되지 않습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 17.4.3: 태그 기반 비용 추적으로 예산 초과 원인 파악**
>
> **상황**: 회사의 AWS 월 청구서가 전달보다 40% 증가했습니다. CFO가 "어떤 부서에서 비용이 증가한 것인지 보고해 달라"고 요청합니다.
>
> **풀이/설명**:
> 1. 모든 리소스에 `Department` 태그가 적용되어 있고, 이 태그가 **비용 할당 태그로 활성화**된 상태입니다.
> 2. AWS Cost Explorer에서 `Department` 태그별로 비용을 필터링합니다.
> 3. 분석 결과: 마케팅 부서(`Department: Marketing`)에서 비용이 전달 대비 300% 증가한 것을 발견합니다.
> 4. 세부 내역을 확인하니, 마케팅 팀이 대규모 데이터 분석 작업을 위해 고사양 EC2 인스턴스를 다수 실행했고, 작업 완료 후 인스턴스를 종료하지 않은 것이 원인이었습니다.
>
> **핵심 포인트**: 태그와 비용 할당 태그를 체계적으로 운영하면, "누가(부서), 무엇에(프로젝트), 얼마를(금액)" 사용했는지 정확히 추적할 수 있습니다.

> 🔍 **예시 17.4.4: Resource Groups를 활용한 프로젝트 리소스 일괄 조회**
>
> **상황**: 프로젝트 매니저가 "WebApp 프로젝트에 현재 어떤 AWS 리소스가 사용되고 있는지 전체 목록을 보고 싶다"고 요청합니다.
>
> **풀이/설명**:
> 1. AWS Resource Groups에서 `Project: WebApp` 태그 기반의 리소스 그룹을 생성합니다.
> 2. 리소스 그룹을 조회하면 EC2 인스턴스 5개, RDS 데이터베이스 1개, S3 버킷 3개, Lambda 함수 8개가 한 화면에 표시됩니다.
> 3. 프로젝트 매니저는 이 목록을 통해 전체 인프라 구성을 파악하고, 불필요한 리소스가 있는지 검토합니다.
>
> **핵심 포인트**: Resource Groups는 서로 다른 서비스에 분산된 리소스를 **하나의 논리적 그룹**으로 묶어, 프로젝트 단위의 리소스 관리를 가능하게 합니다.

---

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의: "태그를 붙이기만 하면 비용이 자동으로 분류된다"는 오해**
>
> 태그를 리소스에 붙이는 것과 **비용 할당 태그로 활성화**하는 것은 별개의 작업입니다. Billing 콘솔에서 해당 태그를 비용 할당 태그로 명시적으로 활성화해야만 청구서에서 비용이 분류됩니다. 또한 활성화 이후 발생하는 비용부터 적용되며, **소급 적용되지 않습니다**.

- **태그 키의 대소문자를 일관되게 유지하세요**: `Environment`와 `environment`는 **서로 다른 태그 키**입니다. 팀원 A가 `Env`를, 팀원 B가 `Environment`를 사용하면 비용 보고서가 분리됩니다. 조직 전체의 태그 명명 규칙을 문서화하세요.
- **태그는 자동으로 붙지 않습니다**: 리소스 생성 시 태그를 명시적으로 지정하지 않으면 태그 없이 생성됩니다. AWS Organizations의 **태그 정책(Tag Policy)**이나 **SCP**를 활용하여 필수 태그를 강제하세요.
- **모든 서비스가 태그를 지원하는 것은 아닙니다**: 대부분의 주요 서비스는 태그를 지원하지만, 일부 서비스나 리소스 유형은 태그가 불가능합니다. 태그 전략을 수립할 때 이 점을 고려하세요.
- **태그 값을 너무 세분화하지 마세요**: 태그 값이 지나치게 세분화되면(예: 개인 이름 단위 프로젝트) 관리 복잡성이 증가하고 비용 보고서의 가독성이 떨어집니다. 적절한 수준의 분류 체계를 선택하세요.

---

### Section 요약 (Section Summary)

- **태그(Tag)** 는 AWS 리소스에 붙이는 **키-값 쌍의 이름표**로, 리소스 분류, 비용 추적, 접근 제어, 자동화에 활용된다.
- **비용 할당 태그**를 활성화하면 AWS 청구서에서 태그별 비용을 분류할 수 있다.
- **Resource Groups**로 공통 태그를 가진 리소스를 그룹으로 묶어 관리할 수 있다.
- **Tag Editor**로 여러 리소스의 태그를 한꺼번에 조회·편집할 수 있다.
- 효과적인 태그 관리를 위해 조직 전체의 **태그 전략(Tagging Strategy)** 을 수립해야 한다.

## 17.5 Service Catalog와 License Manager

> 💡 **한 줄 요약**: AWS Service Catalog는 조직이 승인한 IT 제품만 사용하게 하는 "사내 IT 쇼핑몰"이고, AWS License Manager는 소프트웨어 라이선스 사용량을 추적하는 "라이선스 관리 대장"입니다.

### 직관적 이해 (Intuitive Understanding)

큰 회사의 사무용품 구매를 생각해 봅시다. 직원들이 각자 원하는 곳에서 마음대로 사무용품을 사면 어떻게 될까요? 누군가는 비싼 만년필을 사고, 누군가는 호환되지 않는 프린터를 사고, 예산은 통제 불능이 됩니다. 그래서 회사에는 **사내 구매 시스템(사내몰)** 이 있습니다. 회사가 승인한 품목만 등록되어 있고, 직원은 그 안에서만 선택합니다.

**AWS Service Catalog**는 이런 "사내 IT 쇼핑몰"입니다. AWS 관리자가 "이 조직에서 사용해도 되는 AWS 아키텍처"를 미리 등록해 두면, 일반 사용자는 그 목록 안에서만 선택하여 사용합니다. 보안 설정이 미흡하거나 비용이 과다한 아키텍처를 실수로 사용하는 것을 방지합니다.

**라이선스 관리**의 비유도 비슷합니다. 회사에서 Microsoft Office 라이선스를 100개 구매했는데, 현재 몇 개가 사용 중인지 파악하지 못하면 불필요한 추가 구매를 하거나, 반대로 라이선스 위반 상태가 될 수 있습니다. **AWS License Manager**는 이런 소프트웨어 라이선스의 사용 현황을 추적하는 도구입니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS Service Catalog

> 📝 **AWS Service Catalog**: AWS 관리자가 승인한 IT 서비스(CloudFormation 템플릿으로 정의된 아키텍처)를 **제품(Product)** 으로 등록하고, 이를 **포트폴리오(Portfolio)** 로 묶어 특정 사용자에게 제공하는 서비스입니다. 사용자는 카탈로그에서 선택만 하면 승인된 아키텍처가 자동으로 배포됩니다.

Service Catalog의 핵심 구성 요소:

- **제품(Product)**: 하나의 배포 가능한 IT 서비스 단위. 예를 들어 "보안 규정을 준수하는 웹 서버 환경"을 하나의 제품으로 정의합니다.
- **포트폴리오(Portfolio)**: 제품들의 모음. 예를 들어 "개발팀용 포트폴리오"에는 개발팀이 사용할 수 있는 제품들만 모아 놓습니다.
- **프로비저닝(Provisioning)**: 사용자가 카탈로그에서 제품을 선택하면, 미리 정의된 설정대로 AWS 리소스가 자동 생성됩니다.

> 🔍 **예시 17.5.1: Service Catalog를 활용한 표준 환경 배포**
>
> **상황**: 회사의 클라우드 팀이 개발자들에게 표준화된 개발 환경을 제공하고 싶습니다. 하지만 개발자들이 각자 EC2 인스턴스를 만들 때마다 보안 그룹 설정이 제각각이고, 비싼 인스턴스 유형을 선택하는 경우도 있습니다.
> **풀이/설명**:
> 1. 클라우드 팀이 "표준 개발 서버"라는 제품을 Service Catalog에 등록합니다.
>    - 인스턴스 유형: t3.micro (비용 효율적)
>    - 보안 그룹: 회사 VPN에서만 접근 가능
>    - 태그: `Environment: Development`, `ManagedBy: ServiceCatalog` 자동 부여
> 2. "개발팀 포트폴리오"에 이 제품을 추가하고, 개발팀 IAM 그룹에 접근 권한을 부여합니다.
> 3. 개발자 A가 Service Catalog에 접속하면, "표준 개발 서버" 제품이 보입니다.
> 4. "Launch product"를 클릭하면, 미리 정의된 설정대로 EC2 인스턴스가 생성됩니다.
> **핵심 포인트**: 개발자는 복잡한 설정을 알 필요 없이 "클릭 한 번"으로 표준화된 환경을 받고, 관리자는 승인된 구성만 사용되도록 통제합니다.

> 💬 **참고**: Service Catalog의 제품은 AWS CloudFormation 템플릿으로 정의됩니다. CloudFormation에 대해서는 Chapter 19에서 자세히 다룹니다. 여기서는 "미리 정의된 설정대로 리소스가 자동 생성된다"는 개념만 이해하면 충분합니다.

#### AWS License Manager

> 📝 **AWS License Manager**: 소프트웨어 라이선스를 중앙에서 관리하고, 라이선스 사용량을 추적하며, 라이선스 초과 사용을 방지하는 서비스입니다. Microsoft, Oracle, SAP 등의 상용 소프트웨어 라이선스를 관리합니다.

License Manager의 주요 기능:

- **라이선스 규칙 설정**: "이 소프트웨어는 최대 50개의 vCPU까지만 사용 가능"과 같은 규칙을 정의합니다.
- **사용량 추적**: 현재 라이선스가 몇 개 사용 중인지 실시간으로 추적합니다.
- **초과 방지**: 라이선스 한도를 초과하는 인스턴스 실행을 자동으로 차단하거나 경고합니다.

> 🔍 **예시 17.5.2: License Manager로 라이선스 초과 방지**
>
> **상황**: 회사에서 Windows Server 라이선스를 vCPU 100개 기준으로 구매했습니다. 현재 다양한 팀에서 EC2 Windows 인스턴스를 사용하고 있습니다.
> **풀이/설명**:
> 1. License Manager에서 "Windows Server" 라이선스 구성을 생성하고, 한도를 vCPU 100개로 설정합니다.
> 2. "한도 초과 시 인스턴스 실행 차단" 규칙을 활성화합니다.
> 3. 현재 vCPU 사용량이 85개라는 것이 대시보드에 표시됩니다.
> 4. 개발팀이 vCPU 20개짜리 Windows 인스턴스를 추가하려 하면, 한도(100개)를 초과하므로 실행이 차단됩니다.
> **핵심 포인트**: License Manager는 라이선스 사용 현황을 실시간으로 파악하고, 한도 초과를 자동으로 방지하여 라이선스 위반 위험을 줄입니다.

> ⚠️ **주의**
>
> - **Service Catalog ≠ AWS Marketplace**: Service Catalog는 **조직 내부**에서 승인한 IT 서비스 목록을 관리하는 것이고, AWS Marketplace는 외부 판매자가 제공하는 소프트웨어를 구매하는 **외부 마켓**입니다. 둘의 목적이 다릅니다.
> - **License Manager의 CCP 출제 범위**: CCP 시험에서 License Manager는 깊은 이해보다는 "라이선스를 추적하고 관리하는 서비스가 무엇인가?"라는 수준의 질문이 출제됩니다. "License Manager"라는 이름과 역할을 기억하면 충분합니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 17.5.3: Service Catalog로 신입 개발자 온보딩 환경 자동화**
>
> **상황**: 매달 신입 개발자가 입사하는데, 매번 VPC, EC2, 보안 그룹, IAM 역할을 수동으로 설정하다 보니 1인당 환경 구성에 하루가 걸립니다. 또한 설정이 팀마다 달라서 보안 기준이 일관되지 않습니다.
>
> **풀이/설명**:
> 1. 클라우드 팀이 **"신입 개발 환경"** 제품을 Service Catalog에 등록합니다. 이 제품에는 표준 VPC, t3.micro EC2 인스턴스, 필수 보안 그룹, 읽기 전용 IAM 역할이 미리 정의되어 있습니다.
> 2. 신입 개발자의 매니저가 Service Catalog에 접속하여 "신입 개발 환경"을 선택하고 "Launch"를 클릭합니다.
> 3. 10분 만에 표준화된 개발 환경이 자동 생성됩니다.
> 4. 보안 팀은 모든 신입 개발 환경이 동일한 보안 기준을 만족함을 보장받습니다.
>
> **핵심 포인트**: Service Catalog는 환경 구성의 **표준화**와 **자동화**를 통해, 구성 시간을 단축하고 보안 일관성을 유지합니다.

> 🔍 **예시 17.5.4: License Manager로 감사 대비 라이선스 보고서 생성**
>
> **상황**: 연말 IT 감사에서 "현재 사용 중인 Windows Server 라이선스 수와 구매한 라이선스 수를 비교한 보고서를 제출하세요"라는 요청이 들어왔습니다.
>
> **풀이/설명**:
> 1. License Manager의 대시보드에서 Windows Server 라이선스 구성을 확인합니다.
> 2. **구매 수량**: vCPU 200개 기준 라이선스
> 3. **현재 사용량**: vCPU 165개 사용 중 (82.5%)
> 4. License Manager의 보고서 기능을 통해 월별 사용 추이를 포함한 라이선스 보고서를 생성하여 감사인에게 제출합니다.
>
> **핵심 포인트**: License Manager는 라이선스 사용 현황을 실시간으로 추적하므로, 감사 시 즉시 정확한 보고서를 제출할 수 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의: "Service Catalog = AWS Marketplace"로 혼동하는 실수**
>
> Service Catalog는 **조직 내부**에서 관리자가 승인한 아키텍처를 사용자에게 제공하는 것이고, AWS Marketplace는 **외부 판매자**가 소프트웨어를 판매하는 마켓입니다. 시험에서 "내부 표준화"와 관련된 시나리오가 나오면 Service Catalog, "외부 소프트웨어 구매"가 나오면 Marketplace를 선택하세요.

- **Service Catalog의 "제품"은 CloudFormation 템플릿입니다**: Service Catalog를 사용하려면 AWS CloudFormation에 대한 기본적인 이해가 필요합니다. 그러나 CCP 시험에서는 구현 세부사항보다 **"승인된 제품을 표준화하여 배포하는 서비스"**라는 개념만 기억하면 충분합니다.
- **License Manager는 AWS 서비스의 라이선스가 아닙니다**: License Manager는 **서드파티 상용 소프트웨어**(Microsoft, Oracle, SAP 등)의 라이선스를 관리하는 도구입니다. AWS 서비스 자체의 사용량이나 과금과는 관련이 없습니다.
- **Service Catalog의 "포트폴리오(Portfolio)"를 잊지 마세요**: 제품(Product)을 생성한 것만으로는 사용자에게 노출되지 않습니다. 포트폴리오에 제품을 추가하고, 해당 포트폴리오에 IAM 사용자/그룹 접근 권한을 부여해야 합니다.
- **License Manager로 라이선스를 "구매"할 수 없습니다**: License Manager는 이미 보유한 라이선스의 **사용량을 추적하고 관리**하는 도구이며, 라이선스 구매 기능은 제공하지 않습니다.

---

### Section 요약 (Section Summary)

- **AWS Service Catalog**는 관리자가 승인한 IT 서비스만 사용자에게 제공하는 "사내 IT 쇼핑몰"이다.
- Service Catalog의 핵심 구성: **제품(Product)**, **포트폴리오(Portfolio)**, **프로비저닝(Provisioning)**.
- **AWS License Manager**는 소프트웨어 라이선스 사용량을 추적하고 초과를 방지하는 서비스이다.
- Service Catalog와 AWS Marketplace는 목적이 다르다: 전자는 내부 통제, 후자는 외부 구매이다.

**Q4.** AWS Health Dashboard에서 "Your account health"가 보여주는 정보와 "Service health"가 보여주는 정보의 차이를 한 문장으로 설명하세요.

<details>
<summary>정답 및 해설</summary>

**정답**: Service health는 AWS 전체 서비스의 운영 상태를 보여주고, Your account health는 내 계정의 리소스에 직접 영향을 미치는 이벤트만 선별하여 보여줍니다.

**해설**: 실무에서는 Your account health가 더 유용합니다. 내가 사용하지 않는 서비스의 장애 정보는 필터링되고, 내 리소스에 영향을 미치는 이벤트만 알려주기 때문입니다.
</details>

**Q5.** AWS 리소스에 태그를 붙이면 자동으로 AWS 청구서에서 태그별 비용이 분류된다. (O/X)

<details>
<summary>정답 및 해설</summary>

**정답**: X

**해설**: 태그를 리소스에 붙이는 것만으로는 비용이 분류되지 않습니다. Billing 콘솔에서 해당 태그를 **비용 할당 태그(Cost Allocation Tag)** 로 별도 활성화해야 합니다. 또한 활성화 이후 발생하는 비용부터 적용되며, 소급 적용되지 않습니다.
</details>

**Q6.** AWS Service Catalog와 AWS Marketplace의 차이를 간단히 설명하세요.

<details>
<summary>정답 및 해설</summary>

**정답**: Service Catalog는 **조직 내부**에서 관리자가 승인한 IT 서비스를 사용자에게 제공하는 내부 카탈로그이고, AWS Marketplace는 **외부 판매자**가 제공하는 소프트웨어를 구매하는 외부 마켓입니다.

**해설**: Service Catalog의 목적은 "조직 내 표준화와 거버넌스"이고, AWS Marketplace의 목적은 "타사 소프트웨어 검색과 구매"입니다.
</details>

**Q7.** EC2 인스턴스가 AWS Systems Manager에 관리되려면 두 가지가 필요합니다. 무엇일까요?

<details>
<summary>정답 및 해설</summary>

**정답**: (1) SSM Agent가 인스턴스에 설치되어 실행 중이어야 하고, (2) `AmazonSSMManagedInstanceCore` 정책이 포함된 **IAM 역할**이 인스턴스에 연결되어 있어야 합니다.

**해설**: SSM Agent는 Amazon Linux, Ubuntu 등 주요 AMI에 기본 설치되어 있지만, IAM 역할은 사용자가 직접 연결해야 합니다. 인스턴스가 Systems Manager에 나타나지 않는 가장 흔한 원인은 IAM 역할 누락입니다.
</details>

## 🧪 통합 실습: 태그 전략 수립과 관리 도구 탐색

> **시나리오**: 당신은 스타트업의 클라우드 관리자입니다. 회사에서 AWS를 사용하기 시작한 지 3개월이 되었고, 리소스가 점점 늘어나면서 "어떤 리소스가 어떤 팀의 것인지", "비용이 어디서 발생하는지" 파악하기 어려워졌습니다. CEO로부터 "리소스를 체계적으로 관리하고, 현재 AWS 계정의 상태를 점검하라"는 요청을 받았습니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | 태그 전략을 수립하여 리소스에 적용하고, Trusted Advisor와 Health Dashboard로 계정 상태를 점검한다 |
| **사용 서비스** | AWS Resource Groups & Tag Editor, AWS Trusted Advisor, AWS Health Dashboard, AWS Systems Manager |
| **활용 개념** | Section 17.1 Systems Manager 콘솔 탐색, Section 17.2 Trusted Advisor 점검, Section 17.3 Health Dashboard 확인, Section 17.4 태그 전략과 Tag Editor |
| **선수 실습** | Chapter 4의 AWS 계정 생성 완료 |
| **예상 소요 시간** | 25분 |
| **비용** | Free Tier 범위 내 |

### 아키텍처 다이어그램

```
[AWS Management Console]
        │
        ├── [Tag Editor] ─── 리소스 태그 현황 조회 & 태그 적용
        │
        ├── [Resource Groups] ─── 태그 기반 리소스 그룹 생성
        │
        ├── [Trusted Advisor] ─── 6가지 카테고리 점검 확인
        │
        ├── [Health Dashboard] ─── 서비스 상태 & 계정 이벤트 확인
        │
        └── [Systems Manager] ─── 관리 도구 콘솔 탐색
```

### 실습 단계

> **Phase 1: 태그 전략 수립 및 적용** (활용 개념: Section 17.4)
>
> **Step 1**: 먼저 이 회사의 태그 전략을 수립합니다. 다음 표를 참고하여 우리 회사에서 사용할 표준 태그를 정합니다:
>
> | 태그 키 | 필수 여부 | 설명 | 예시 값 |
> |---------|----------|------|---------|
> | `Environment` | 필수 | 환경 구분 | `Production`, `Development`, `Staging` |
> | `Team` | 필수 | 소유 팀 | `Backend`, `Frontend`, `Data` |
> | `Project` | 필수 | 프로젝트명 | `MainWebsite`, `InternalTool` |
> | `CostCenter` | 권장 | 비용 부서 코드 | `CC-001`, `CC-002` |
>
> **Step 2**: AWS 콘솔에서 상단 검색창에 `Resource Groups`를 입력하고 **Resource Groups & Tag Editor**를 클릭합니다.
> **Step 3**: 왼쪽 메뉴에서 **"Tag Editor"** 를 클릭합니다.
> **Step 4**: Regions에서 `ap-northeast-2`(서울)를 선택하고, Resource types에서 `All supported resource types`를 선택한 뒤 **"Search resources"** 를 클릭합니다.
> **Step 5**: 검색 결과에서 태그가 없는 리소스가 있는지 확인합니다. (계정에 리소스가 있다면) 태그가 없는 리소스를 선택하고, Phase 1에서 정한 태그 전략에 따라 적절한 태그를 추가합니다.
>
> **✅ 체크포인트**: Tag Editor에서 리전 내 리소스를 조회하고, 태그 현황을 확인할 수 있어야 합니다.

> **Phase 2: Resource Group 생성** (활용 개념: Section 17.4)
>
> **Step 1**: 왼쪽 메뉴에서 **"Create Resource Group"** 을 클릭합니다.
> **Step 2**: Group type으로 **"Tag based"** 를 선택합니다.
> **Step 3**: Grouping criteria에서:
>   - Tags에 `Environment` 키를 입력하고, 값으로 `Development`를 입력합니다.
>   - **"Preview group resources"** 를 클릭하여 해당 태그가 붙은 리소스를 미리 확인합니다.
> **Step 4**: Group details에서:
>   - Group name: `Dev-Environment-Resources`를 입력합니다.
>   - Description: `개발 환경의 모든 리소스`를 입력합니다.
> **Step 5**: **"Create group"** 을 클릭합니다.
>
> **✅ 체크포인트**: "Dev-Environment-Resources"라는 Resource Group이 생성되고, 해당 태그가 붙은 리소스가 그룹에 포함되어 표시됩니다. (태그가 부여된 리소스가 없으면 빈 그룹이 됩니다 — 정상입니다.)

> **Phase 3: Trusted Advisor 점검 확인** (활용 개념: Section 17.2)
>
> **Step 1**: AWS 콘솔 상단 검색창에 `Trusted Advisor`를 입력하고 클릭합니다.
> **Step 2**: 대시보드에서 6가지 카테고리의 요약 상태를 확인합니다.
> **Step 3**: **"Security"** 카테고리를 클릭하여 보안 점검 항목을 확인합니다.
>   - 🔴 빨간색 항목이 있다면 내용을 확인하고 메모합니다.
>   - 특히 "MFA on Root Account" 항목의 상태를 확인합니다.
> **Step 4**: **"Service Limits"** 카테고리를 클릭하여 서비스 한도 사용률을 확인합니다.
>
> **✅ 체크포인트**: Trusted Advisor에서 보안 점검 결과와 서비스 한도를 확인할 수 있어야 합니다.

> **Phase 4: Health Dashboard 확인** (활용 개념: Section 17.3)
>
> **Step 1**: AWS 콘솔 상단 검색창에 `Health`를 입력하고 **AWS Health Dashboard**를 클릭합니다.
> **Step 2**: **"Your account health"** 에서 내 계정에 영향을 미치는 이벤트가 있는지 확인합니다.

### 트러블슈팅 (Troubleshooting)

> **Q: Tag Editor에서 "Search resources"를 클릭했는데 아무 리소스도 표시되지 않습니다.**
> A: AWS 계정에 해당 리전에 리소스가 생성되어 있지 않은 경우입니다. 이전 Chapter의 실습에서 리소스를 모두 삭제했다면 정상입니다. 다른 리전을 선택해 보거나, 간단한 리소스(예: S3 버킷)를 하나 생성한 후 다시 시도해 보세요.

> **Q: Trusted Advisor에서 일부 카테고리가 "Upgrade your support plan"으로 표시됩니다.**
> A: Basic/Developer 지원 플랜에서는 핵심 보안 점검과 서비스 한도 점검만 가능합니다. 나머지 카테고리의 전체 점검은 Business 이상 플랜에서만 사용할 수 있습니다. 이 실습에서는 사용 가능한 점검 항목만 확인하면 됩니다.

> **Q: Resource Group을 생성했는데 "0 resources" 라고 표시됩니다.**
> A: 지정한 태그(예: `Environment: Development`)가 붙은 리소스가 해당 리전에 없는 경우입니다. Phase 1에서 리소스에 태그를 추가한 후, 같은 태그 조건으로 Resource Group을 생성하면 리소스가 포함됩니다.

> **Exercise 17.1**: 당신은 회사의 클라우드 관리자입니다. CEO가 다음과 같은 요구사항을 제시했습니다:

> - "서버에 접속할 때 SSH 키 관리가 번거로운데, 더 안전하고 편한 방법이 없을까?"

>
> 각 요구사항에 대해 AWS Systems Manager의 어떤 기능을 활용할 수 있는지 설명하고, 각 기능이 왜 적합한지 이유를 서술하세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: CEO의 세 가지 요구사항을 각각 Systems Manager의 핵심 기능에 매핑합니다.
>
> **풀이 과정**:
>
> **요구 1: "EC2 서버에 보안 패치를 매주 자동으로 적용"**
> → **Patch Manager** 활용
> - 패치 기준선(Patch Baseline)에서 "Critical과 Important 보안 패치만 자동 적용" 규칙을 설정합니다.
> - 유지 관리 기간(Maintenance Window)을 "매주 일요일 새벽 2시"로 설정하여 업무 시간에 영향이 없게 합니다.
> - 적합한 이유: 수동으로 서버에 접속할 필요 없이 중앙에서 일괄적으로 패치가 적용되므로 운영 효율과 보안이 동시에 향상됩니다.
>
> **요구 2: "SSH 키 관리 없이 안전하게 서버 접속"**
> → **Session Manager** 활용
> - SSH 포트(22번)를 열지 않고, IAM 인증으로 브라우저에서 직접 서버에 접속합니다.
> - 적합한 이유: SSH 키 생성·배포·폐기 과정이 불필요하며, 모든 세션이 자동으로 기록되어 감사에도 유용합니다. 포트를 열지 않으므로 보안 위험도 줄어듭니다.
>
> **요구 3: "데이터베이스 비밀번호를 안전하게 관리"**
> → **Parameter Store** 활용 (SecureString 타입)
> - 비밀번호를 SecureString으로 저장하면 KMS로 자동 암호화됩니다.
> - 적합한 이유: 소스 코드에서 비밀번호를 분리하여 코드 유출 시에도 비밀번호가 노출되지 않습니다. 비밀번호 변경 시 Parameter Store 값만 업데이트하면 되므로 관리도 편리합니다.
>
> **보충 설명**: 세 기능 모두 Systems Manager라는 하나의 서비스 안에 있으므로, 콘솔 하나에서 통합 관리할 수 있다는 것이 추가적인 장점입니다.
> </details>

> **Problem 17.1**: 당신은 50명의 개발자가 있는 중견 기업의 AWS 관리자입니다. 현재 다음과 같은 문제가 있습니다:

>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 이 문제는 "태그 전략 + Systems Manager + Trusted Advisor + Service Catalog + IAM"을 종합적으로 활용해야 하는 실전 시나리오입니다.
>
> **풀이 전략**: 4가지 문제를 각각 분석하고, 해당 문제를 해결하는 도구와 전략을 매핑합니다.
>
> **상세 풀이**:
>
> **문제 1: 인스턴스 소유 파악 불가 → 태그 전략 수립**
> - 필수 태그를 정의합니다: `Team`, `Project`, `Environment`, `Owner`, `CostCenter`
> - Tag Editor를 사용하여 기존 150개 인스턴스에 태그를 일괄 부여합니다.

> - `Team`과 `Project` 태그를 Billing 콘솔에서 비용 할당 태그로 활성화합니다.

> - Systems Manager의 Fleet Manager에서 전체 인스턴스의 패치 준수율(Compliance)을 한눈에 확인할 수 있습니다.
>

> - Service Catalog에 승인된 인스턴스 유형만 포함하는 제품을 등록합니다(예: t3.micro, t3.small, t3.medium만 허용).

> **Problem 17.3**: 다음은 가상의 회사 "CloudStart Inc."의 AWS 리소스 현황입니다:

> (2) "운영 환경(Production)에 있는 모든 리소스"를 Resource Group으로 묶으려 할 때, 현재 태그로는 어떤 문제가 발생하는지 설명하세요.
> (3) 이 회사를 위한 개선된 태그 전략과, 이를 강제하기 위해 활용할 수 있는 AWS 서비스를 제안하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 태그의 일관성이 없으면, 태그 기반 관리(Resource Groups, 비용 할당, 자동화)가 모두 무너집니다.
>
> **풀이 전략**: 태그의 키 일관성, 값 일관성, 태그 누락을 체계적으로 점검합니다.
>
> **상세 풀이**:
>
> **(1) 문제점**:
>
> - **키 불일치 — 환경 태그**: `Env` (web-server-1, main-db) vs. `Environment` (web-server-2). 같은 목적의 태그인데 키가 다릅니다.
> - **키 불일치 — 팀 태그**: `Team` (Lambda) vs. `team` (web-server-2) vs. `Team` (web-server-1에서는 소문자). AWS는 대소문자를 구분하므로 `Team`과 `team`은 서로 다른 키입니다.
> - **값 불일치 — 환경 태그**: `prod` (web-server-1) vs. `Production` (web-server-2). 같은 운영 환경을 다른 값으로 표현하고 있습니다.
> - **값 불일치 — 팀 태그**: `backend` (web-server-1) vs. `Backend` (web-server-2).
> - **태그 누락**: test-server에는 태그가 전혀 없습니다.
> - **태그 불완전**: 각 리소스마다 태그의 종류가 다릅니다 (어떤 것은 Team만, 어떤 것은 Project만).
>
> **(2) Resource Group 문제**:
>
> "운영 환경의 모든 리소스"를 묶으려면 태그 조건을 설정해야 하는데:
> - `Environment: Production`으로 검색하면 → web-server-2만 검색됩니다.
> - `Env: prod`로 검색하면 → web-server-1과 main-db만 검색됩니다.
> - 두 조건을 모두 설정해도, test-server는 태그가 없어 빠지며, app-data-bucket과 data-processor에도 환경 태그가 없어 빠집니다.
> - 결론: 현재 태그로는 **운영 환경의 전체 리소스를 하나의 Resource Group으로 묶는 것이 불가능**합니다.
>
> **(3) 개선된 태그 전략 및 강제 방안**:
>
> **태그 전략**:
> | 태그 키 (필수) | 허용 값 | 규칙 |
> |--------------|---------|------|
> | `Environment` | `Production`, `Staging`, `Development` | PascalCase, 전체 단어 |
> | `Team` | `Backend`, `Frontend`, `Data`, `DevOps` | PascalCase |
> | `Project` | `WebApp`, `Analytics`, `Internal` | PascalCase |
> | `Owner` | 이메일 주소 | 소문자 |
>
> **강제 방안**:
> - **AWS Organizations의 태그 정책(Tag Policy)**: 필수 태그 키와 허용되는 값을 정의하여, 조직 내 모든 계정에 일관된 태그 규칙을 적용합니다.
> - **AWS Config**(Chapter 16에서 배운 내용): `required-tags` 규칙을 설정하여 필수 태그가 없는 리소스를 자동으로 탐지합니다.
> - **Tag Editor**: 기존 리소스의 태그를 일괄 수정하여 새로운 태그 전략에 맞게 정리합니다.
>
> **확장 생각**: 태그 전략은 기술적 도구만으로 완성되지 않습니다. 조직 전체가 합의한 명명 규칙 문서를 작성하고, 신규 직원 온보딩 시 태그 가이드를 교육하며, 정기적으로 태그 준수율을 점검하는 **프로세스**가 함께 필요합니다.
> </details>
