## 20.1 클라우드 도입 프레임워크(CAF) — 6가지 관점, 조직 변화 관리

> 💡 **한 줄 요약**: AWS Cloud Adoption Framework(CAF)은 조직이 클라우드를 도입할 때 고려해야 할 6가지 관점(비즈니스, 사람, 거버넌스, 플랫폼, 보안, 운영)을 체계적으로 정리한 프레임워크입니다.

### 직관적 이해

회사를 이사한다고 생각해봅시다. 새 사무실로 옮기려면 단순히 짐을 나르는 것만으로는 부족합니다.

- **경영진**은 "이사 비용 대비 효과가 있을까?"를 따집니다 → 비즈니스 관점
- **직원들**은 "새 사무실에서 일하는 법을 배워야 하나?"를 걱정합니다 → 사람 관점
- **관리부서**는 "예산 초과 없이, 규정에 맞게 이사할 수 있을까?"를 점검합니다 → 거버넌스 관점
- **시설팀**은 "네트워크, 전기, 서버실은 어떻게 구성할까?"를 설계합니다 → 플랫폼 관점
- **보안팀**은 "이사 중에 중요 서류가 분실되지 않을까?"를 확인합니다 → 보안 관점
- **운영팀**은 "이사 후에 업무가 중단 없이 돌아갈까?"를 계획합니다 → 운영 관점

클라우드 도입도 마찬가지입니다. 기술적으로 서버를 옮기는 것만이 아니라, 조직 전체가 준비되어야 합니다. **AWS CAF**는 이 복잡한 전환 과정을 체계적으로 안내하는 나침반과 같습니다.

### 핵심 개념 설명

> 📝 **AWS Cloud Adoption Framework(CAF)**: AWS가 제공하는 클라우드 도입 가이드라인으로, 조직이 클라우드로 전환할 때 **기술적·비기술적 측면을 모두 포괄하는 6가지 관점(Perspective)**을 통해 체계적인 준비를 돕는 프레임워크입니다.

CAF는 클라우드 도입을 단순한 기술 프로젝트가 아닌, **조직 전체의 디지털 전환(Digital Transformation)**으로 바라봅니다. 6가지 관점은 크게 **비즈니스 역량**과 **기술 역량** 두 그룹으로 나뉩니다.

#### 비즈니스 역량 (Business Capabilities)

| 관점 | 영문 | 핵심 질문 | 주요 이해관계자 |
|------|------|-----------|----------------|
| **비즈니스** | Business | 클라우드 투자가 비즈니스 성과로 이어지는가? | CxO, 사업부 리더 |
| **사람** | People | 조직 구성원이 클라우드 환경에서 일할 준비가 되었는가? | HR, 교육 담당자 |
| **거버넌스** | Governance | 리스크 관리, 예산 통제, 규정 준수가 가능한가? | CIO, CFO, 프로그램 관리자 |

#### 기술 역량 (Technical Capabilities)

| 관점 | 영문 | 핵심 질문 | 주요 이해관계자 |
|------|------|-----------|----------------|
| **플랫폼** | Platform | 클라우드 인프라를 어떻게 설계·구축하는가? | CTO, 아키텍트, 엔지니어 |
| **보안** | Security | 데이터와 워크로드를 어떻게 보호하는가? | CISO, 보안 엔지니어 |
| **운영** | Operations | 클라우드 서비스를 어떻게 안정적으로 운영하는가? | 운영팀, SRE, IT 서비스 관리자 |

> 💬 **참고**: CAF는 AWS가 수천 건의 클라우드 도입 사례에서 축적한 경험을 정리한 것입니다. 실제로는 각 관점별로 더 세부적인 역량(Capability)과 실행 항목이 있지만, CCP 시험에서는 **6가지 관점의 이름과 각각의 핵심 초점**을 아는 것이 중요합니다.

#### CAF의 4단계 전환 여정

CAF는 클라우드 도입을 4단계로 나눕니다:

1. **구상(Envision)**: 클라우드 도입으로 달성할 비즈니스 목표를 정의합니다
2. **정렬(Align)**: 6가지 관점에서 현재 상태와 목표 상태의 차이(Gap)를 파악합니다
3. **실행(Launch)**: 파일럿 프로젝트를 시작하여 실제 마이그레이션을 수행합니다
4. **확장(Scale)**: 파일럿의 성공 경험을 조직 전체로 확대합니다

### 상세 예시

> 🔍 **예시 20.1.1: 중견 제조업체의 클라우드 도입 — 6가지 관점 적용**
>
> **상황**: 직원 500명의 제조업체가 자체 서버실의 ERP(전사적 자원관리) 시스템을 AWS로 이전하려고 합니다.
>
> **각 관점에서의 고려 사항**:
> - **비즈니스**: "서버 운영 비용이 연간 3억 원인데, 클라우드로 옮기면 얼마나 절감되는가?" → ROI(투자 대비 수익) 분석
> - **사람**: "IT팀 10명 중 클라우드 경험자가 1명뿐이다" → AWS 교육 프로그램 필요
> - **거버넌스**: "제조업 규정상 생산 데이터는 국내에 보관해야 한다" → 서울 리전 사용 필수
> - **플랫폼**: "현재 Oracle DB + Java 앱 구조를 AWS에서 어떻게 구현할까?" → 아키텍처 설계
> - **보안**: "고객 주문 데이터의 암호화 기준은?" → 보안 정책 수립
> - **운영**: "ERP가 1시간만 멈춰도 생산 라인이 정지된다" → 고가용성 운영 계획
>
> **핵심 포인트**: CAF는 "기술 이전"이 아니라 "조직 변화"로서의 클라우드 도입을 강조합니다. 기술팀만 준비해서는 안 되고, 경영진·인사·재무·보안 등 조직 전체가 함께 준비해야 합니다.

> 🔍 **예시 20.1.2: 비즈니스 관점과 거버넌스 관점의 차이**
>
> **상황**: "클라우드에 얼마를 쓸 것인가?"라는 질문에 두 관점이 다르게 접근합니다.
>
> **비즈니스 관점의 접근**: "클라우드에 투자하면 신제품 출시 속도가 3개월에서 1개월로 단축된다. 이것이 가져올 매출 증가를 계산해보자." → **가치(Value)** 중심
>
> **거버넌스 관점의 접근**: "클라우드 예산이 매달 초과되지 않도록 알림을 설정하고, 분기별 비용 리뷰 프로세스를 만들자." → **통제(Control)** 중심
>
> **핵심 포인트**: 비즈니스 관점은 "왜 투자하는가(Why)", 거버넌스 관점은 "어떻게 관리하는가(How)"에 초점을 둡니다. CCP 시험에서 이 둘을 혼동하지 않도록 주의하세요.

> ⚠️ **주의**
>
> - CAF의 6가지 관점 이름을 외울 때 **비사거 / 플보운**(비즈니스·사람·거버넌스 / 플랫폼·보안·운영)으로 그룹화하면 기억하기 쉽습니다.
> - "사람(People)" 관점을 "HR 관점"으로만 좁게 생각하면 안 됩니다. 사람 관점에는 조직 문화 변화, 기술 교육, 변화 관리(Change Management)가 모두 포함됩니다.
> - CCP 시험에서는 "특정 상황에서 어떤 관점이 관련되는가?"를 묻는 시나리오 문제가 출제됩니다. 각 관점의 **이해관계자**와 **핵심 질문**을 연결지어 기억하세요.

### Section 요약

- **AWS CAF**는 클라우드 도입을 조직 전체의 변화로 바라보는 프레임워크이다
- 6가지 관점은 **비즈니스 역량**(비즈니스, 사람, 거버넌스)과 **기술 역량**(플랫폼, 보안, 운영)으로 나뉜다
- 각 관점에는 고유한 **이해관계자**와 **핵심 질문**이 있다
- CAF의 전환 여정은 **구상 → 정렬 → 실행 → 확장** 4단계로 진행된다

## 20.2 6R 마이그레이션 전략 — Rehost, Replatform, Refactor 등 전략 비교

> 💡 **한 줄 요약**: 6R 전략은 기존 애플리케이션을 클라우드로 이전할 때 선택할 수 있는 6가지 접근 방식으로, 각 애플리케이션의 특성에 따라 최적의 전략이 달라집니다.

### 직관적 이해

오래된 집에서 새 집으로 이사한다고 생각해봅시다. 집 안의 모든 물건을 똑같은 방식으로 옮기지는 않습니다:

- 냉장고는 **그대로 옮깁니다** (Rehost) — 잘 작동하니 새 집에 그냥 가져가면 됩니다
- 오래된 커튼은 **새 집 창문 크기에 맞게 수선합니다** (Replatform) — 약간의 조정만 합니다
- 낡은 부엌은 **새 집에 맞게 완전히 새로 설계합니다** (Refactor) — 처음부터 다시 만듭니다
- 세탁기는 **새로 구입합니다** (Repurchase) — 기존 것을 버리고 최신 모델로 바꿉니다
- 지하실의 와인 셀러는 **그대로 둡니다** (Retain) — 아직 옮길 필요가 없습니다
- 고장난 에어컨은 **버립니다** (Retire) — 더 이상 필요 없습니다

이것이 바로 **6R 마이그레이션 전략**입니다. 기존 시스템 하나하나를 분석하여, 각각에 가장 적합한 이전 방법을 선택하는 것입니다.

### 핵심 개념 설명

> 📝 **6R 마이그레이션 전략**: 기존 애플리케이션을 클라우드로 이전할 때 선택할 수 있는 6가지 전략으로, 각 전략은 이전의 복잡도, 소요 시간, 비용, 그리고 클라우드 최적화 수준이 다릅니다.

#### 1. Rehost (리호스트) — "Lift and Shift"

기존 애플리케이션을 **변경 없이 그대로** 클라우드로 옮기는 방식입니다.

- **비유**: 이삿짐을 포장도 안 풀고 그대로 새 집에 넣는 것
- **장점**: 가장 빠르고 간단한 이전 방법
- **단점**: 클라우드의 장점(자동 확장, 서버리스 등)을 충분히 활용하지 못함
- **적합한 경우**: 빠르게 데이터센터를 폐쇄해야 할 때, 우선 이전 후 최적화할 계획일 때

> 🔗 **연결**: Chapter 5에서 배운 EC2를 떠올려 봅시다. 온프레미스 서버에서 돌리던 애플리케이션을 동일한 설정의 EC2 인스턴스에 그대로 올리는 것이 Rehost의 대표적인 예입니다.

#### 2. Replatform (리플랫폼) — "Lift, Tinker, and Shift"

애플리케이션의 핵심 아키텍처는 유지하면서, **일부 구성 요소만 클라우드 서비스로 교체**하는 방식입니다.

- **비유**: 냉장고는 그대로 가져가되, 새 집에 맞는 전원 어댑터를 달아주는 것
- **장점**: 비교적 적은 노력으로 클라우드 이점(관리형 서비스 활용)을 일부 누릴 수 있음
- **단점**: 완전한 클라우드 최적화는 아님
- **예시**: 자체 관리하던 MySQL DB를 Amazon RDS로 교체 (애플리케이션 코드는 거의 변경하지 않지만 DB 관리 부담이 줄어듦)

#### 3. Refactor / Re-architect (리팩터) — "재설계"

애플리케이션을 **클라우드 네이티브 아키텍처로 완전히 재설계**하는 방식입니다.

- **비유**: 낡은 집을 허물고 새 집을 짓는 것
- **장점**: 클라우드의 모든 장점을 극대화, 장기적으로 가장 효율적
- **단점**: 시간과 비용이 가장 많이 들며, 높은 기술 역량이 필요
- **예시**: 모놀리식 애플리케이션을 마이크로서비스로 분해하고, 서버리스(Lambda) 아키텍처로 전환

> 🔗 **연결**: Chapter 5에서 배운 Lambda(서버리스)를 기억하시나요? Refactor 전략을 적용하면, EC2에서 돌리던 애플리케이션을 Lambda 기반으로 재설계하여 서버 관리 부담을 완전히 제거할 수 있습니다.

#### 4. Repurchase (리퍼처스) — "Drop and Shop"

기존 애플리케이션을 **버리고, 클라우드 기반의 새로운 SaaS(Software as a Service) 제품을 구매**하여 대체하는 방식입니다.

- **비유**: 오래된 세탁기를 수리하는 대신 최신 모델을 구매하는 것
- **장점**: 유지보수 부담이 사라지고, 최신 기능을 바로 활용 가능
- **단점**: 기존 시스템에 맞춤화(customization)가 많았다면 SaaS 전환이 어려울 수 있음
- **예시**: 자체 개발한 CRM(고객 관리 시스템)을 Salesforce로 교체, 자체 이메일 서버를 Microsoft 365로 교체

#### 5. Retain (유지) — "Keep As-Is"

현재 시점에서는 **이전하지 않고 기존 환경에 그대로 유지**하는 방식입니다.

- **비유**: 당장 옮기기 어려운 물건은 창고에 보관해두는 것
- **이유**: 최근에 대규모 업그레이드를 했거나, 복잡한 의존성이 있어 당장 이전이 어렵거나, 곧 폐기 예정인 경우
- **주의**: "마이그레이션하지 않기로 결정하는 것"도 전략입니다. 모든 것을 한꺼번에 옮길 필요는 없습니다.

#### 6. Retire (폐기) — "Get Rid of It"

더 이상 필요하지 않은 애플리케이션을 **제거**하는 방식입니다.

- **비유**: 이사를 기회로 안 쓰는 물건을 정리하는 것
- **장점**: 유지보수 비용 절감, 보안 리스크 감소
- **예시**: 아무도 사용하지 않는 레거시 시스템, 중복된 기능을 가진 시스템

#### 6R 전략 비교 종합표

| 전략 | 복잡도 | 소요 시간 | 클라우드 최적화 | 핵심 키워드 |
|------|--------|----------|----------------|------------|
| **Rehost** | 낮음 | 빠름 | 낮음 | "그대로 옮기기" |
| **Replatform** | 중간 | 중간 | 중간 | "일부 교체" |
| **Refactor** | 높음 | 느림 | 높음 | "완전 재설계" |
| **Repurchase** | 중간 | 중간 | 해당 없음 | "새로 구매" |
| **Retain** | 없음 | 없음 | 없음 | "현 상태 유지" |
| **Retire** | 없음 | 빠름 | 해당 없음 | "폐기" |

### 상세 예시

> 🔍 **예시 20.2.1: 전자상거래 기업의 마이그레이션 전략 결정**
>
> **상황**: 온라인 쇼핑몰을 운영하는 기업이 자체 데이터센터에서 AWS로 이전합니다. 다음 시스템들을 어떤 전략으로 이전할지 결정해야 합니다.
>
> **풀이/설명**:
>
> | 시스템 | 전략 | 이유 |
> |--------|------|------|
> | 웹 서버 (Apache + Java) | **Rehost** | 우선 빠르게 이전하고, 이후 최적화 |
> | MySQL 데이터베이스 | **Replatform** | Amazon RDS for MySQL로 교체하면 DB 관리 부담이 줄어듦 |
> | 레거시 재고관리 시스템 | **Refactor** | 트래픽에 따라 자동 확장이 필요하므로 마이크로서비스로 재설계 |
> | 자체 개발 이메일 시스템 | **Repurchase** | Amazon SES 또는 외부 이메일 SaaS로 대체 |
> | 테스트 서버 (3개월 후 폐기 예정) | **Retain** | 곧 폐기할 것이므로 이전 불필요 |
> | 5년 전 중단된 프로모션 사이트 | **Retire** | 더 이상 사용하지 않으므로 폐기 |
>
> **핵심 포인트**: 하나의 기업 안에서도 시스템마다 다른 전략을 적용합니다. "모든 것을 한 가지 방식으로 이전"하는 것이 아닌, **각 시스템의 특성과 비즈니스 가치를 분석하여 최적의 전략을 선택**하는 것이 핵심입니다.

> 🔍 **예시 20.2.2: Rehost vs Replatform vs Refactor — 한 애플리케이션의 세 가지 시나리오**
>
> **상황**: 온프레미스에서 Java 웹 애플리케이션과 Oracle 데이터베이스를 운영하고 있습니다.
>
> **Rehost**: Java 앱을 EC2에, Oracle DB도 EC2에 직접 설치 → 변경 없이 이전. 클라우드에 있지만 기존과 동일하게 운영
>
> **Replatform**: Java 앱은 EC2에, Oracle DB는 Amazon RDS for Oracle로 교체 → 앱 코드는 거의 동일하지만 DB 백업·패치가 자동화됨
>
> **Refactor**: Java 앱을 컨테이너화하여 ECS에서 실행하고, DB는 Amazon Aurora로 전환, 일부 기능을 Lambda로 분리 → 완전히 새로운 아키텍처
>
> **핵심 포인트**: 같은 시스템이라도 **시간, 예산, 기술 역량, 비즈니스 목표**에 따라 다른 전략을 선택합니다. 정답은 없으며, 조직의 상황에 맞는 최적의 선택이 중요합니다.

> ⚠️ **주의**
>
> - CCP 시험에서 "Lift and Shift"라는 표현이 나오면 **Rehost**를 의미합니다.
> - **Replatform과 Refactor를 혼동하지 마세요**: Replatform은 "일부 조정", Refactor는 "완전 재설계"입니다. Replatform은 애플리케이션 코드를 거의 건드리지 않는 반면, Refactor는 아키텍처 자체를 바꿉니다.
> - **Retain은 "실패"가 아닙니다**: 모든 것을 한 번에 옮길 필요는 없습니다. 전략적으로 남겨두는 것도 훌륭한 결정입니다.

### Section 요약

- **6R 전략**은 마이그레이션의 6가지 접근 방식: Rehost, Replatform, Refactor, Repurchase, Retain, Retire
- 각 전략은 **복잡도, 소요 시간, 클라우드 최적화 수준**이 다르다
- 같은 조직 내에서도 시스템마다 **다른 전략을 적용**할 수 있다
- 시험에서는 시나리오를 주고 **적합한 전략을 선택**하는 문제가 자주 출제된다

## 20.3 AWS Migration Hub와 MGN — 마이그레이션 중앙 추적, Application Migration Service를 통한 리호스팅

> 💡 **한 줄 요약**: AWS Migration Hub는 마이그레이션의 진행 상황을 한곳에서 추적하는 대시보드이며, AWS MGN(Application Migration Service)은 서버를 자동으로 AWS로 리호스팅하는 도구입니다.

### 직관적 이해

대규모 이사를 한다고 상상해봅시다. 트럭 여러 대에 짐을 나눠 싣고, 서로 다른 이사업체를 이용합니다. 이때 필요한 것이 무엇일까요?

- **이사 현황판**(Migration Hub): "1호 트럭은 도착 완료, 2호 트럭은 이동 중, 3호 트럭은 적재 중" — 전체 이사 진행 상황을 한눈에 볼 수 있는 대시보드
- **이사 전문 업체**(MGN): 짐을 포장하고, 옮기고, 새 집에 배치하는 작업을 자동으로 수행하는 전문 이사 서비스

마이그레이션도 마찬가지입니다. 수십~수백 대의 서버를 이전할 때, **전체 진행 상황을 추적**하고 **실제 이전을 자동화**하는 도구가 필요합니다.

### 핵심 개념 설명

#### AWS Migration Hub

> 📝 **AWS Migration Hub**: 여러 AWS 마이그레이션 도구(MGN, DMS 등)를 사용하여 진행하는 마이그레이션의 **전체 진행 상황을 한곳에서 추적하고 관리**하는 중앙 대시보드 서비스입니다.

Migration Hub의 핵심 역할:

- **단일 대시보드**: 여러 마이그레이션 도구의 진행 상황을 하나의 화면에서 확인
- **애플리케이션 그룹화**: 관련된 서버·데이터베이스를 하나의 애플리케이션으로 묶어 추적
- **진행 상태 시각화**: 각 서버/DB의 이전 단계(발견 → 평가 → 이전 → 완료)를 시각적으로 표시

> 💬 **참고**: Migration Hub 자체는 추가 비용 없이 사용할 수 있습니다. 실제 비용은 Migration Hub가 추적하는 개별 마이그레이션 도구(MGN, DMS 등)에서 발생합니다.

#### AWS Application Discovery Service

> 📝 **AWS Application Discovery Service**: 온프레미스 환경의 서버 구성, 성능 데이터, 네트워크 의존성 등을 **자동으로 수집하고 분석**하여 마이그레이션 계획 수립을 돕는 서비스입니다.

마이그레이션의 첫 단계는 "지금 우리가 뭘 갖고 있는지"를 파악하는 것입니다. Application Discovery Service는 이 **현황 파악(Discovery)** 작업을 자동화합니다.

- **에이전트 기반 수집**: 온프레미스 서버에 에이전트를 설치하여 상세 데이터 수집 (CPU 사용률, 네트워크 연결, 프로세스 정보)
- **에이전트 없는(Agentless) 수집**: VMware 환경에서 에이전트 없이 VM 정보를 수집
- **Migration Hub 연동**: 수집된 데이터를 Migration Hub에서 조회하여 마이그레이션 전략 결정에 활용

#### AWS Application Migration Service (AWS MGN)

> 📝 **AWS Application Migration Service (AWS MGN)**: 온프레미스 서버를 AWS로 **자동으로 리호스팅(Rehost)**하는 서비스입니다. 소스 서버의 데이터를 지속적으로 복제하고, 전환(cutover) 시점에 AWS에서 동일한 서버를 실행합니다.

MGN의 동작 방식:

1. **에이전트 설치**: 이전할 소스 서버에 AWS Replication Agent를 설치합니다
2. **지속적 복제**: 에이전트가 서버의 디스크 데이터를 AWS로 지속적으로 복제합니다 (이 과정에서 소스 서버는 정상 운영됨)
3. **테스트 인스턴스 실행**: 복제된 데이터로 AWS에서 테스트 인스턴스를 실행하여 정상 작동 여부를 확인합니다
4. **전환(Cutover)**: 테스트가 완료되면, 최종 전환하여 AWS에서 본격 운영을 시작합니다

> 🔗 **연결**: Chapter 5에서 배운 EC2를 떠올려 봅시다. MGN은 온프레미스 서버의 이미지를 EC2 인스턴스로 변환하여 실행합니다. 즉, 기존 서버가 EC2 인스턴스로 "부활"하는 것입니다.

### 상세 예시

> 🔍 **예시 20.3.1: 마이그레이션 도구의 역할 분담 — 비유로 이해하기**
>
> **상황**: 100대의 서버를 AWS로 이전하는 프로젝트를 비유로 이해해봅시다.
>
> **풀이/설명**:
> - **Application Discovery Service** = **이삿짐 목록 작성**: "서버가 몇 대인지, 각각 얼마나 자원을 사용하는지, 서로 어떻게 연결되어 있는지"를 조사
> - **AWS MGN** = **이사 실행**: 목록이 완성되면, 실제로 서버를 포장(복제)하고 새 집(AWS)으로 옮기는 작업 수행
> - **Migration Hub** = **이사 현황판**: 전체 100대 중 몇 대가 이전 완료되었고, 몇 대가 진행 중인지 추적
>
> **핵심 포인트**: 세 서비스는 **발견(Discovery) → 실행(Migration) → 추적(Tracking)**이라는 마이그레이션 파이프라인을 구성합니다.

> 🔍 **예시 20.3.2: MGN을 활용한 서버 리호스팅 흐름**
>
> **상황**: 온프레미스 데이터센터에서 Linux 웹 서버를 운영하고 있습니다. 데이터센터 계약이 3개월 후 만료되어 빠르게 AWS로 이전해야 합니다.
>
> **풀이/설명**:
> 1. **Discovery**: Application Discovery Service로 웹 서버의 사양(CPU 4코어, RAM 16GB, 디스크 200GB)과 네트워크 연결(DB 서버와 통신)을 파악
> 2. **에이전트 설치**: 웹 서버에 AWS Replication Agent를 설치
> 3. **복제**: 에이전트가 200GB 디스크 데이터를 AWS로 지속적으로 복제 (이 동안 웹 서버는 정상 서비스 중)
> 4. **테스트**: AWS에서 테스트 EC2 인스턴스를 실행하여 웹 애플리케이션이 정상 작동하는지 확인
> 5. **Cutover**: 금요일 밤 유지보수 시간에 DNS를 변경하여 트래픽을 AWS EC2로 전환
> 6. **Migration Hub 확인**: 대시보드에서 이 서버의 상태가 "마이그레이션 완료"로 업데이트됨
>
> **핵심 포인트**: MGN의 핵심 장점은 **서버가 서비스를 제공하면서 동시에 복제가 진행**된다는 것입니다. 긴 다운타임 없이 마이그레이션을 수행할 수 있습니다.

> 🧪 **Mini Lab 20.3: AWS Migration Hub 대시보드 탐색**
>
> **목표**: Migration Hub의 대시보드 구성과 주요 기능을 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (콘솔 탐색만 수행)
>
> **Step 1**: AWS 콘솔 상단 검색창에 `Migration Hub`를 입력하고, **AWS Migration Hub**를 클릭합니다.
>
> **Step 2**: 왼쪽 탐색 메뉴에서 다음 항목들을 확인합니다:
> - **Dashboard**: 마이그레이션 전체 현황을 보여주는 대시보드
> - **Discover → Servers**: 발견된 서버 목록 (현재는 비어 있음)
> - **Migrate → Applications**: 마이그레이션 대상 애플리케이션 목록
>
> **Step 3**: **Discover → Servers** 화면에서 "Start discovery" 또는 "Get started" 버튼이 있는지 확인합니다. 이 버튼을 통해 Application Discovery Service와 연동하여 온프레미스 서버를 등록할 수 있습니다. (실제 클릭하지 않아도 됩니다.)
>
> **Step 4**: 왼쪽 메뉴에서 **Tools** 항목을 확인합니다. 여기에서 MGN, DMS 등 다양한 마이그레이션 도구로의 링크를 볼 수 있습니다.
>
> **확인**: Migration Hub가 마이그레이션의 "중앙 제어실" 역할을 한다는 것을 콘솔 구조를 통해 확인했습니다.
> **🧹 정리**: 콘솔 탐색만 수행했으므로 별도 리소스 삭제가 필요하지 않습니다.

> ⚠️ **주의**
>
> - **Migration Hub ≠ 실제 이전 도구**: Migration Hub는 추적·관리 도구이며, 실제 서버 이전은 MGN, 데이터 이전은 DMS가 수행합니다.
> - CCP 시험에서 "서버를 자동으로 리호스팅하는 서비스는?"이라는 질문이 나오면 정답은 **AWS MGN(Application Migration Service)**입니다. Migration Hub와 혼동하지 마세요.
> - MGN은 이전에 **Server Migration Service(SMS)**라고 불리던 서비스의 후속입니다. 오래된 자료에서 SMS로 언급된 것은 현재 MGN으로 대체되었습니다.

### Section 요약

- **AWS Migration Hub**는 마이그레이션 전체 진행 상황을 추적하는 중앙 대시보드이다
- **Application Discovery Service**는 온프레미스 환경의 서버와 의존성을 자동으로 파악한다
- **AWS MGN**은 서버를 자동으로 AWS로 리호스팅하며, 지속적 복제를 통해 다운타임을 최소화한다
- 세 서비스는 **발견 → 실행 → 추적**이라는 마이그레이션 파이프라인을 구성한다

## 20.4 데이터 전송 서비스 — DMS, DataSync, Transfer Family

> 💡 **한 줄 요약**: AWS는 데이터 유형과 전송 시나리오에 따라 DMS(데이터베이스 마이그레이션), DataSync(파일 시스템 동기화), Transfer Family(SFTP/FTP 기반 전송)를 제공합니다.

### 직관적 이해

이사를 할 때, 옮기는 물건의 종류에 따라 다른 방법을 사용합니다:

- **금고 속 귀중품**은 전문 보안 운송 업체에 맡깁니다 → **DMS** (데이터베이스처럼 정교한 데이터)
- **창고의 박스들**은 대형 택배로 빠르게 보냅니다 → **DataSync** (파일 시스템의 대량 데이터)
- **우편함으로 오는 서류**는 주소만 바꿔 새 집으로 전달받습니다 → **Transfer Family** (기존 FTP/SFTP 방식 유지)

데이터도 마찬가지입니다. 데이터베이스, 파일 시스템, FTP 서버 — 각각의 데이터 유형에 맞는 전용 전송 서비스가 있습니다.

### 핵심 개념 설명

#### AWS Database Migration Service (DMS)

> 📝 **AWS DMS (Database Migration Service)**: 데이터베이스를 AWS로 마이그레이션하는 서비스로, **소스 데이터베이스가 운영 중인 상태에서도** 데이터를 안전하게 이전할 수 있습니다.

DMS의 핵심 특징:

- **소스 DB 중단 불필요**: 기존 데이터베이스가 서비스하면서 동시에 데이터를 이전
- **동종 마이그레이션**: 같은 DB 엔진 간 이전 (예: MySQL → Amazon RDS for MySQL)
- **이기종 마이그레이션**: 다른 DB 엔진 간 이전 (예: Oracle → Amazon Aurora)
- **지속적 복제**: 일회성 이전뿐 아니라, 변경 데이터를 지속적으로 복제하는 것도 가능

> 📝 **AWS Schema Conversion Tool (SCT)**: 이기종 마이그레이션 시, 소스 DB의 스키마(테이블 구조, 저장 프로시저 등)를 **대상 DB 엔진에 맞게 자동으로 변환**하는 도구입니다.

```
[동종 마이그레이션]
MySQL (온프레미스) ──DMS──→ Amazon RDS for MySQL
                             (스키마 변환 불필요)

[이기종 마이그레이션]  
Oracle (온프레미스) ──SCT──→ 스키마 변환 ──DMS──→ Amazon Aurora (PostgreSQL 호환)
                             (스키마를 PostgreSQL 형식으로 변환)
```

#### AWS DataSync

> 📝 **AWS DataSync**: 온프레미스 스토리지(NFS, SMB 파일 서버)와 AWS 스토리지 서비스(S3, EFS, FSx) 간에 **대량의 데이터를 빠르고 안전하게 전송**하는 서비스입니다.

> 🔗 **연결**: Chapter 6에서 배운 S3, EFS, FSx를 떠올려 봅시다. DataSync는 온프레미스의 파일 시스템 데이터를 이러한 AWS 스토리지 서비스로 빠르게 옮기는 데 사용됩니다.

DataSync의 핵심 특징:

- **빠른 전송**: 네트워크 최적화와 병렬 전송으로 일반 복사 도구보다 최대 10배 빠름
- **자동 일정**: 시간·일·주 단위로 자동 동기화 설정 가능
- **데이터 무결성 검증**: 전송 과정에서 자동으로 데이터 무결성을 확인
- **사용 사례**: 온프레미스 파일 서버의 클라우드 마이그레이션, 정기적 데이터 백업, 재해 복구를 위한 데이터 복제

#### AWS Transfer Family

> 📝 **AWS Transfer Family**: 기존에 SFTP, FTPS, FTP 프로토콜로 데이터를 주고받던 워크플로를 **변경 없이 그대로 AWS S3 또는 EFS와 연동**할 수 있게 하는 관리형 파일 전송 서비스입니다.

Transfer Family가 필요한 이유는 간단합니다: 많은 기업이 이미 FTP/SFTP 기반의 데이터 교환 프로세스를 구축해 놓았기 때문입니다. 거래처, 파트너사, 내부 시스템이 모두 FTP로 파일을 주고받고 있는데, 클라우드로 옮긴다고 이 모든 시스템을 바꿀 수는 없습니다.

- **기존 워크플로 유지**: 파트너사는 이전과 동일한 FTP/SFTP 주소로 파일을 업로드
- **백엔드는 S3/EFS**: 실제 파일은 S3 버킷이나 EFS에 저장됨
- **관리형 서비스**: FTP 서버 인프라를 직접 관리할 필요 없음

#### 데이터 전송 서비스 비교

| 서비스 | 데이터 유형 | 핵심 사용 사례 | 비유 |
|--------|------------|---------------|------|
| **DMS** | 데이터베이스 | DB 마이그레이션, DB 엔진 변경 | 금고 보안 운송 |
| **DataSync** | 파일 시스템 | 대량 파일 이전, 정기 동기화 | 대형 택배 서비스 |
| **Transfer Family** | FTP/SFTP 파일 | 기존 FTP 워크플로 유지 | 우편 전달 서비스 |

### 상세 예시

> 🔍 **예시 20.4.1: 병원 시스템의 데이터베이스 마이그레이션 — DMS 활용**
>
> **상황**: 병원이 온프레미스 Oracle 데이터베이스에 저장된 환자 기록을 Amazon Aurora(PostgreSQL 호환)로 이전하려고 합니다. 병원 시스템은 24시간 운영되어 중단이 불가합니다.
>
> **풀이/설명**:
> 1. **SCT(Schema Conversion Tool)**로 Oracle 스키마를 PostgreSQL 형식으로 변환
> 2. **DMS**를 설정하여 Oracle → Aurora로 데이터 이전 시작
> 3. 기존 Oracle DB는 정상 운영 상태를 유지 (DMS가 변경 데이터를 지속적으로 복제)
> 4. 테스트 기간을 거친 후, 최종 전환(cutover) 시점에 애플리케이션이 Aurora를 바라보도록 변경
>
> **핵심 포인트**: DMS의 최대 강점은 **소스 DB 중단 없이** 마이그레이션을 수행한다는 것입니다. 24/7 운영이 필요한 시스템에 특히 적합합니다.

> 🔍 **예시 20.4.2: 미디어 회사의 파일 서버 이전 — DataSync 활용**
>
> **상황**: 미디어 회사가 온프레미스 NFS 파일 서버에 50TB의 영상 파일을 보유하고 있습니다. 이 데이터를 Amazon S3로 옮기고, 이후에도 매일 새로운 영상을 동기화하려고 합니다.
>
> **풀이/설명**:
> - **초기 이전**: DataSync를 설정하여 50TB를 S3로 전송 (DataSync의 네트워크 최적화로 빠르게 전송)
> - **지속적 동기화**: 일일 스케줄을 설정하여 매일 새로 추가된 영상 파일만 자동으로 S3에 동기화
> - **데이터 무결성**: DataSync가 전송 중 자동으로 체크섬(checksum)을 비교하여 데이터 손상이 없음을 확인
>
> **핵심 포인트**: DataSync는 일회성 이전뿐 아니라 **지속적인 동기화**에도 사용됩니다. "옮기고 끝"이 아니라 "옮기고 계속 동기화"가 필요할 때 유용합니다.

> ⚠️ **주의**
>
> - **DMS ≠ DataSync**: DMS는 **데이터베이스** 전용, DataSync는 **파일 시스템** 전용입니다. 시험에서 이 둘을 혼동하는 문제가 자주 출제됩니다.
> - **이기종 DB 마이그레이션 시 SCT가 필요합니다**: Oracle → Aurora처럼 DB 엔진이 바뀌는 경우, DMS만으로는 부족하고 SCT로 스키마를 먼저 변환해야 합니다. 같은 엔진(MySQL → RDS MySQL) 간에는 SCT가 필요하지 않습니다.
> - **Transfer Family와 DataSync를 혼동하지 마세요**: Transfer Family는 "기존 FTP/SFTP 워크플로를 유지하면서 S3에 저장"하는 것이고, DataSync는 "대량 데이터를 빠르게 이전·동기화"하는 것입니다.

### Section 요약

- **AWS DMS**는 데이터베이스를 중단 없이 마이그레이션하며, 이기종 이전 시 **SCT**로 스키마를 변환한다
- **AWS DataSync**는 파일 시스템 데이터를 S3/EFS/FSx로 빠르게 전송하고 지속적으로 동기화한다
- **AWS Transfer Family**는 기존 FTP/SFTP 워크플로를 변경 없이 AWS 스토리지와 연동한다
- 각 서비스는 **데이터 유형**(데이터베이스 / 파일 / FTP)에 따라 선택한다

## 20.5 AWS Snow Family — 대용량 오프라인 데이터 전송, 엣지 컴퓨팅

> 💡 **한 줄 요약**: AWS Snow Family는 네트워크 전송이 비현실적인 대용량 데이터를 물리적 디바이스로 이동시키며, 네트워크가 불안정한 엣지 환경에서의 컴퓨팅에도 사용됩니다.

> 🔗 **연결**: Chapter 6.8에서 Snow Family의 기본 개념(Snowcone, Snowball Edge, 사용 과정)을 배웠습니다. 이 Section에서는 마이그레이션 관점에서의 심화 내용과 엣지 컴퓨팅 활용을 다룹니다.

### 직관적 이해

1GB 파일을 인터넷으로 전송하는 데 약 10초가 걸린다고 합시다. 그렇다면:

- 1TB(1,000GB) = 약 3시간
- 100TB = 약 12일
- 1PB(1,000TB) = 약 **340일** (거의 1년!)

100Mbps 인터넷 회선으로 100TB를 전송하려면 약 **100일**이 걸립니다. 하지만 Snow Family 디바이스를 택배로 보내면? **약 1주일**이면 충분합니다.

이것이 Snow Family의 존재 이유입니다. 충분히 큰 데이터는 **네트워크보다 택배가 빠릅니다**.

### 핵심 개념 설명

Chapter 6.8에서 Snow Family의 기본 개념을 학습했으므로, 여기서는 **마이그레이션 관점의 심화 내용**에 집중합니다.

#### Snow Family 디바이스 상세 비교

| 특성 | **Snowcone / Snowcone SSD** | **Snowball Edge Storage Optimized** | **Snowball Edge Compute Optimized** |
|------|-----|-----|-----|
| **스토리지 용량** | HDD 8TB / SSD 14TB | 80TB 사용 가능 | 28TB 사용 가능 |
| **컴퓨팅** | 2 vCPU, 4GB RAM | 40 vCPU, 80GB RAM | 104 vCPU, 416GB RAM |
| **크기** | 작은 노트북 크기 (2.1kg) | 여행용 캐리어 크기 | 여행용 캐리어 크기 |
| **주요 사용 사례** | 소규모 엣지 수집, 공간 제약 환경 | 대용량 데이터 마이그레이션 | 엣지에서의 고성능 컴퓨팅 |
| **네트워크** | Wi-Fi, 유선 이더넷 | 10/25/100 Gbps | 10/25/100 Gbps |

> 💬 **참고**: AWS는 2024년에 **Snowmobile**(트럭 크기 컨테이너, 100PB 용량) 서비스를 **폐기(deprecate)**했습니다. CCP 시험에서 Snowmobile이 보기로 나온다면, 현재 사용 가능한 서비스가 아님을 기억하세요. 대신 Snowball Edge를 여러 대 사용하여 대용량 데이터를 이전하는 것이 현재 권장 방식입니다.

#### Snow Family의 마이그레이션 워크플로

```
[1. 주문]        [2. 수령]         [3. 데이터 복사]      [4. 반송]         [5. 업로드]
AWS 콘솔에서 → 디바이스가 → 온프레미스 데이터를 → AWS로 → AWS가 데이터를
디바이스 주문    배송됨        디바이스에 복사       반송     S3에 업로드
```

이 과정에서 **데이터는 자동으로 암호화**됩니다. 디바이스가 분실되더라도 256비트 암호화와 TPM(Trusted Platform Module)으로 보호되어 데이터가 유출되지 않습니다.

#### 엣지 컴퓨팅 활용

Snow Family는 단순한 데이터 운반 도구가 아닙니다. **엣지 컴퓨팅(Edge Computing)** 용도로도 사용됩니다.

> 📝 **엣지 컴퓨팅 (Edge Computing)**: 데이터가 생성되는 현장(엣지)에서 직접 컴퓨팅을 수행하는 방식입니다. 데이터를 클라우드로 보내지 않고 현장에서 바로 처리하여 지연 시간을 줄이고, 네트워크 연결이 불안정한 환경에서도 동작합니다.

Snow Family 엣지 컴퓨팅 사용 사례:

- **선박**: 바다 한가운데에서 센서 데이터를 수집·분석 (인터넷 연결 없음)
- **공장**: 생산 라인의 품질 검사 데이터를 현장에서 ML 추론
- **원격지 건설 현장**: 드론으로 촬영한 영상을 현장에서 분석
- **군사 작전**: 보안이 필요한 현장에서 데이터를 처리

> 🔗 **연결**: Chapter 3.4에서 배운 Local Zones와 Wavelength도 "사용자에게 더 가까운 곳에서 컴퓨팅"하는 개념이었습니다. Snow Family의 엣지 컴퓨팅은 이 개념을 **네트워크가 아예 없는 극한 환경**까지 확장한 것입니다.

### 상세 예시

> 🔍 **예시 20.5.1: 영화 제작사의 대용량 데이터 마이그레이션**
>
> **상황**: 영화 제작사가 온프레미스 스토리지에 500TB의 영상 원본 파일을 보유하고 있습니다. 이를 Amazon S3 Glacier로 아카이빙하려고 합니다. 회사의 인터넷 회선은 1Gbps입니다.
>
> **풀이/설명**:
> - **인터넷 전송 예상 시간**: 500TB ÷ 1Gbps ≈ **약 46일** (24시간 연속 전송 가정, 실제로는 다른 업무에도 네트워크를 사용하므로 훨씬 더 걸림)
> - **Snow Family 사용**: Snowball Edge Storage Optimized(80TB) 7대를 주문 → 데이터 복사 → 반송 → **약 1~2주** 완료
> - **추가 이점**: 마이그레이션 기간 동안 회사 네트워크에 전혀 부담을 주지 않음
>
> **핵심 포인트**: 대용량 데이터 이전에서 Snow Family는 **시간 절약**뿐 아니라 **네트워크 대역폭 보호**라는 이점도 제공합니다.

> 🔍 **예시 20.5.2: 해양 연구선에서의 Snowcone 엣지 컴퓨팅**
>
> **상황**: 해양 연구선이 30일간 태평양에서 해저 지형을 탐사합니다. 매일 약 200GB의 소나(Sonar) 데이터가 생성되며, 위성 인터넷은 속도가 매우 느립니다.
>
> **풀이/설명**:
> - **Snowcone SSD**(14TB)를 여러 대 선박에 탑재
> - 매일 수집되는 소나 데이터를 Snowcone에 저장
> - Snowcone에서 간단한 데이터 전처리(노이즈 제거, 포맷 변환) 수행 (엣지 컴퓨팅)
> - 항구 귀환 후 Snowcone을 AWS로 반송하여 S3에 업로드
> - AWS에서 본격적인 데이터 분석 수행
>
> **핵심 포인트**: Snowcone은 가장 작고 가벼워서 선박, 드론, 차량 등 **공간과 전력이 제한된 환경**에 적합합니다.

> ⚠️ **주의**
>
> - **Snow Family vs DataSync 구분**: 네트워크 대역폭이 충분하면 **DataSync**, 네트워크가 부족하거나 오프라인이면 **Snow Family**를 사용합니다.
> - **Snowball Edge에는 두 가지 유형이 있습니다**: Storage Optimized(대용량 저장)와 Compute Optimized(고성능 컴퓨팅). 시험에서 "엣지에서 ML 추론을 실행"이라는 시나리오가 나오면 **Compute Optimized**가 적합합니다.
> - Snow Family 디바이스는 **구매하는 것이 아니라 임대**하는 것입니다. 사용 후 AWS에 반송합니다.

### Section 요약

- Snow Family는 **네트워크보다 물리적 전송이 빠른** 대용량 데이터 마이그레이션에 사용된다
- **Snowcone**(소형·경량)과 **Snowball Edge**(대용량·고성능) 두 제품군이 있다
- Snow Family는 데이터 전송뿐 아니라 **엣지 컴퓨팅** 용도로도 활용된다
- 데이터는 **자동 암호화**되어 보안이 보장된다
- Snowmobile은 2024년에 폐기되었다

## 20.6 AWS Elastic Disaster Recovery — 클라우드 기반 재해 복구(DR), RPO/RTO 개념

> 💡 **한 줄 요약**: AWS Elastic Disaster Recovery(DRS)는 온프레미스 또는 다른 클라우드의 서버를 AWS에 지속적으로 복제하여, 재해 발생 시 빠르게 복구할 수 있게 하는 서비스입니다.

### 직관적 이해

중요한 문서 작업을 하고 있다고 상상해봅시다. 컴퓨터가 갑자기 고장나면 어떻게 될까요?

- **백업을 안 해뒀다면**: 작업한 내용이 모두 사라집니다 (최악의 시나리오)
- **매일 밤 백업했다면**: 어제 저녁 이후의 작업분만 잃습니다 (하루치 손실)
- **실시간으로 클라우드에 동기화했다면**: 거의 아무것도 잃지 않습니다 (최소 손실)

기업의 서버도 마찬가지입니다. 화재, 홍수, 정전 같은 재해가 발생하면 서버가 멈추고 데이터가 손실될 수 있습니다. **AWS Elastic Disaster Recovery(DRS)**는 서버를 AWS에 실시간으로 복제해두어, 재해가 발생하면 즉시 AWS에서 동일한 서버를 실행할 수 있게 합니다.

### 핵심 개념 설명

먼저 재해 복구(DR)의 두 가지 핵심 지표를 이해해야 합니다.

> 📝 **RPO (Recovery Point Objective, 복구 시점 목표)**: 재해 발생 시 **최대 허용 가능한 데이터 손실량**을 시간으로 표현한 것입니다. "최대 몇 시간/분 전의 데이터까지 복구할 수 있어야 하는가?"라는 질문에 대한 답입니다.

> 📝 **RTO (Recovery Time Objective, 복구 시간 목표)**: 재해 발생 후 **시스템이 정상 운영으로 복귀하기까지의 최대 허용 시간**입니다. "재해 후 최대 몇 시간/분 안에 서비스를 다시 시작해야 하는가?"라는 질문에 대한 답입니다.

```
                    RPO                              RTO
            ◄──────────────────►              ◄──────────────────►
            
마지막 백업 시점 ────── 재해 발생 ────── 서비스 복구 완료
            │                  │                  │
     이 시점까지의        이 시점에서        이 시점에 서비스
     데이터는 보존됨      서비스 중단         다시 시작
            
     RPO가 작을수록 → 데이터 손실이 적음 (비용↑)
     RTO가 작을수록 → 복구가 빠름 (비용↑)
```

#### AWS Elastic Disaster Recovery (DRS)

> 📝 **AWS Elastic Disaster Recovery (DRS)**: 소스 서버(온프레미스 또는 다른 클라우드)의 데이터를 AWS에 **지속적으로 복제**하고, 재해 발생 시 **분 단위(minutes)**의 RPO와 RTO로 빠르게 복구할 수 있게 하는 서비스입니다.

DRS의 동작 방식은 앞서 배운 MGN과 유사합니다:

1. **에이전트 설치**: 소스 서버에 AWS Replication Agent를 설치
2. **지속적 복제**: 서버의 디스크 데이터가 AWS의 저비용 스테이징 영역에 지속적으로 복제됨 (서브초(sub-second) 수준의 RPO)
3. **대기 상태**: 평상시에는 저비용 스테이징 영역에만 데이터가 존재 (EC2 인스턴스를 실행하지 않으므로 비용이 적음)
4. **재해 발생 시 복구**: 버튼 한 번으로 EC2 인스턴스를 실행하여 서비스를 신속하게 복구 (분 단위 RTO)
5. **원래 환경 복구 후**: 다시 원래 환경으로 돌아가는 "페일백(Failback)"도 지원

DRS의 비용 효율성:

- 기존 DR 방식: 별도의 백업 데이터센터를 구축하여 항상 운영 → **비용이 매우 높음**
- DRS 방식: 평상시에는 저비용 스토리지에 데이터만 복제, 재해 시에만 EC2 실행 → **사용한 만큼만 비용 발생**

### 상세 예시

> 🔍 **예시 20.6.1: RPO와 RTO를 일상 비유로 이해하기**
>
> **상황**: 당신은 소설을 쓰고 있습니다. 컴퓨터가 갑자기 고장났을 때의 시나리오를 비교합니다.
>
> **풀이/설명**:
>
> | 시나리오 | RPO | RTO | 결과 |
> |---------|-----|-----|------|
> | 백업 안 함 | ∞ (무한대) | ∞ | 모든 원고 손실, 복구 불가 |
> | 매일 밤 USB에 백업 | 24시간 | 수 시간 (새 PC 구매 후 복원) | 최대 하루치 작업 손실 |
> | 매 1시간마다 Dropbox 동기화 | 1시간 | 수 분 (다른 PC에서 로그인) | 최대 1시간치 작업 손실 |
> | 실시간 Google Docs | 수 초 | 수 초 (다른 기기에서 접속) | 거의 손실 없음 |
>
> AWS Elastic Disaster Recovery는 마지막 시나리오(실시간 동기화)에 해당합니다.
>
> **핵심 포인트**: RPO와 RTO가 작을수록 좋지만, **비용도 증가**합니다. 비즈니스 요구사항에 맞는 적절한 수준을 선택하는 것이 중요합니다.

> 🔍 **예시 20.6.2: 금융 회사의 재해 복구 시나리오 — DRS 활용**
>
> **상황**: 증권 거래 시스템을 운영하는 금융 회사가 있습니다. 데이터센터에 화재가 발생하면, 1분 내에 데이터 손실 없이 복구해야 합니다 (RPO: 수 초, RTO: 분 단위).
>
> **풀이/설명**:
> - 모든 거래 서버에 AWS Replication Agent를 설치
> - 거래 데이터가 실시간으로(서브초 수준) AWS 서울 리전의 스테이징 영역에 복제됨
> - 화재 발생 → 즉시 DRS 콘솔에서 "Recover" 실행
> - 수 분 내에 AWS에서 동일한 거래 서버가 EC2 인스턴스로 실행됨
> - DNS를 AWS 환경으로 전환하여 서비스 재개
> - 데이터센터 복구 후, DRS의 페일백 기능으로 다시 원래 환경으로 이전
>
> **핵심 포인트**: DRS의 핵심 가치는 **"평상시에는 저비용으로 유지하면서, 재해 시 빠르게 복구"**하는 것입니다. 별도의 DR 데이터센터를 구축하는 것보다 훨씬 경제적입니다.

> 🔬 **Deep Dive: DR 전략의 4가지 유형**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> AWS는 재해 복구 전략을 비용과 복구 속도에 따라 4단계로 분류합니다:
>
> | 전략 | RPO/RTO | 비용 | 설명 |
> |------|---------|------|------|
> | **백업 & 복원** | 시간~일 | 가장 낮음 | 정기 백업만 유지, 필요 시 복원 |
> | **파일럿 라이트** | 분~시간 | 낮음 | 핵심 시스템만 최소한으로 복제·운영 |
> | **웜 스탠바이** | 분 | 중간 | 축소된 규모로 항상 운영 |
> | **멀티사이트 액티브-액티브** | 거의 0 | 가장 높음 | 동일 규모로 동시 운영 |
>
> DRS는 주로 **파일럿 라이트**와 **웜 스탠바이** 사이의 전략을 효율적으로 구현하는 데 사용됩니다.

> ⚠️ **주의**
>
> - **DRS와 MGN을 혼동하지 마세요**: MGN은 **마이그레이션(일회성 이전)** 도구이고, DRS는 **재해 복구(지속적 보호)** 도구입니다. 기술적으로 유사하지만 목적이 다릅니다.
> - **RPO와 RTO를 헷갈리지 마세요**: RPO는 **데이터 손실**(과거를 향함), RTO는 **복구 시간**(미래를 향함)입니다.
> - CCP 시험에서 "재해 복구를 위한 AWS 서비스는?"이라는 질문이 나오면 **AWS Elastic Disaster Recovery (DRS)**를 선택하세요.

### Section 요약

- **RPO**는 허용 가능한 최대 데이터 손실 시간, **RTO**는 서비스 복구까지의 최대 시간이다
- **AWS Elastic Disaster Recovery(DRS)**는 서버를 AWS에 지속적으로 복제하여 재해 시 분 단위로 복구한다
- DRS는 평상시 저비용 스테이징 영역만 사용하므로, 별도 DR 데이터센터보다 **비용 효율적**이다
- MGN은 마이그레이션, DRS는 재해 복구 — **목적이 다르다**

## 20.7 AWS의 혁신 서비스 — VMware Cloud on AWS, 하이브리드/엣지 컴퓨팅 활용 사례

> 💡 **한 줄 요약**: AWS는 기존 VMware 환경을 그대로 AWS에서 운영할 수 있는 VMware Cloud on AWS와, 다양한 하이브리드·엣지 컴퓨팅 옵션을 제공하여 유연한 클라우드 전환을 지원합니다.

### 직관적 이해

모든 기업이 한 번에 100% 클라우드로 전환할 수 있는 것은 아닙니다. 현실적인 이유들이 있습니다:

- "우리 회사의 모든 서버가 VMware 가상화 위에서 돌아가는데, 이걸 다 바꿀 수는 없어" → **VMware Cloud on AWS**
- "공장의 로봇 제어 시스템은 지연 시간 때문에 클라우드에 올릴 수 없어" → **Outposts, Local Zones**
- "5G 게임 서비스는 기지국 근처에서 실행되어야 해" → **Wavelength**

이러한 "하이브리드" 요구를 해결하는 것이 이 Section에서 다루는 혁신 서비스들입니다.

### 핵심 개념 설명

#### VMware Cloud on AWS

> 📝 **VMware Cloud on AWS**: 온프레미스 VMware 환경(vSphere, vSAN, NSX)을 **그대로 AWS 인프라 위에서 운영**할 수 있게 하는 서비스입니다. VMware 툴과 프로세스를 변경하지 않고도 AWS 클라우드의 이점을 누릴 수 있습니다.

많은 기업이 이미 VMware 가상화 플랫폼을 사용하여 서버를 관리하고 있습니다. VMware Cloud on AWS를 사용하면:

- **기존 VMware 도구 그대로 사용**: vCenter, vSphere Client 등 익숙한 관리 도구를 계속 사용
- **AWS 네이티브 서비스 연동**: VMware 워크로드에서 S3, RDS, Lambda 등 AWS 서비스에 접근 가능
- **빠른 마이그레이션**: VMware vMotion을 사용하여 온프레미스 VM을 AWS로 라이브 마이그레이션

> 💬 **참고**: VMware Cloud on AWS는 VMware와 AWS가 공동으로 제공하는 서비스입니다. CCP 시험에서는 "VMware 환경을 AWS로 이전"하는 시나리오에서 이 서비스를 선택할 수 있어야 합니다.

#### 하이브리드/엣지 컴퓨팅 — 기존 개념과의 연결

> 🔗 **연결**: Chapter 3.4에서 Local Zones, Wavelength, Outposts의 기본 개념을 배웠습니다. 여기서는 이들이 **마이그레이션·혁신 관점에서 어떻게 활용되는지**를 살펴봅니다.

| 서비스 | 핵심 사용 사례 | Ch.3.4 비유 |
|--------|---------------|-------------|
| **AWS Outposts** | 온프레미스에서 AWS 서비스를 실행해야 할 때 (규정·지연 시간 등) | 내 집에 설치한 자판기 |
| **AWS Local Zones** | 특정 대도시 사용자에게 초저지연 서비스를 제공할 때 | 동네 슈퍼마켓 |
| **AWS Wavelength** | 5G 네트워크 엣지에서 초저지연 서비스를 제공할 때 | 5G 기지국 안의 미니 매장 |

이 서비스들이 마이그레이션과 관련되는 이유는 무엇일까요? 많은 기업이 "모든 것을 클라우드로"가 아닌 **"적재적소에 클라우드를"** 전략을 선택합니다:

- **핵심 비즈니스 로직** → AWS 리전 (완전한 클라우드)
- **지연에 민감한 워크로드** → Local Zones 또는 Wavelength
- **규정상 온프레미스 필수** → Outposts (AWS 인프라를 현장에 설치)
- **기존 VMware 환경** → VMware Cloud on AWS (점진적 전환)

### 상세 예시

> 🔍 **예시 20.7.1: 제조업체의 하이브리드 클라우드 전략**
>
> **상황**: 자동차 부품 제조업체가 클라우드를 도입하려고 합니다. 하지만 공장의 생산 라인 제어 시스템은 수 밀리초 이내의 응답 시간이 필요하고, 특정 데이터는 규정상 공장 내에 보관해야 합니다.
>
> **풀이/설명**:
> - **공장 생산 라인 제어**: **AWS Outposts**를 공장에 설치 → 지연 시간 최소화 + AWS 서비스 사용
> - **본사 ERP 시스템**: AWS 서울 리전에 **Rehost** → EC2에서 운영
> - **전사 데이터 분석**: AWS 리전의 **S3 + 분석 서비스** 사용 → 클라우드 확장성 활용
> - **VMware 가상 서버**: **VMware Cloud on AWS**로 기존 VMware 환경 이전 → 관리 도구 변경 없이
>
> **핵심 포인트**: 하나의 기업 내에서도 워크로드의 특성에 따라 **여러 인프라 옵션을 조합**하여 사용합니다. 이것이 하이브리드 클라우드 전략의 핵심입니다.

> 🔍 **예시 20.7.2: VMware Cloud on AWS를 선택하는 이유**
>
> **상황**: IT 팀이 50명인 금융 회사에서, 200대의 VMware 가상 머신을 운영하고 있습니다. 3년 후 데이터센터 임대 계약이 만료됩니다.
>
> **풀이/설명**:
> - **Refactor(재설계)**: 200대의 VM을 모두 클라우드 네이티브로 재설계 → IT 팀 50명으로는 3년 안에 불가능
> - **Rehost(MGN 사용)**: 200대를 EC2로 이전 → 가능하지만, VMware 관리 도구를 모두 버려야 함, IT팀의 재교육 필요
> - **VMware Cloud on AWS**: 기존 VMware 환경을 그대로 AWS로 이동 → 관리 도구·프로세스 변경 없이 빠르게 이전 가능
>
> **핵심 포인트**: VMware Cloud on AWS는 "VMware에 대한 투자(기술·프로세스·인력)를 보존하면서 클라우드로 이전"하려는 기업에 적합합니다.

> ⚠️ **주의**
>
> - **VMware Cloud on AWS는 "VMware를 잘 모르는" 사용자가 선택하는 서비스가 아닙니다**: 이 서비스는 이미 VMware를 사용하고 있는 기업이 AWS로 이전할 때 유용합니다.
> - CCP 시험에서 "온프레미스에서 AWS 서비스를 실행"하는 시나리오가 나오면 **Outposts**를 선택하세요. "VMware 환경을 AWS에서 실행"이면 **VMware Cloud on AWS**를 선택합니다. 이 두 서비스를 혼동하지 마세요.

### Section 요약

- **VMware Cloud on AWS**는 기존 VMware 환경을 그대로 AWS 위에서 운영하는 서비스이다
- **하이브리드·엣지 컴퓨팅**(Outposts, Local Zones, Wavelength)은 "적재적소에 클라우드를" 전략을 실현한다
- 기업은 워크로드 특성에 따라 **여러 인프라 옵션을 조합**하여 사용할 수 있다
- CCP 시험에서는 시나리오에 맞는 **서비스 선택**이 중요하다

## 자기 점검 질문 (Self-Check Questions)

**Q1.** AWS CAF의 6가지 관점 중, "조직 구성원의 교육과 변화 관리"에 해당하는 관점은?

<details>
<summary>정답</summary>

**사람(People) 관점**. 사람 관점은 조직 문화 변화, 기술 교육, 변화 관리를 다룹니다.
</details>

**Q2.** "기존 MySQL 데이터베이스를 변경 없이 Amazon RDS for MySQL로 옮기는 것"은 6R 전략 중 어디에 해당하는가?

<details>
<summary>정답</summary>

**Replatform**. 애플리케이션 코드는 거의 변경하지 않지만, 자체 관리 DB를 AWS 관리형 서비스(RDS)로 교체하는 것은 Replatform입니다. (DB 자체를 그대로 EC2에 설치하면 Rehost)
</details>

**Q3.** O/X: AWS Migration Hub는 실제로 서버를 이전하는 서비스이다.

<details>
<summary>정답</summary>

**X**. Migration Hub는 마이그레이션의 진행 상황을 **추적**하는 대시보드 서비스입니다. 실제 서버 이전은 MGN이 수행합니다.
</details>

**Q4.** Oracle 데이터베이스를 Amazon Aurora(PostgreSQL 호환)로 이전할 때, DMS와 함께 사용해야 하는 도구는?

<details>
<summary>정답</summary>

**AWS SCT (Schema Conversion Tool)**. 이기종 마이그레이션(DB 엔진이 다른 경우)에서는 SCT로 스키마를 대상 DB 형식으로 변환한 후, DMS로 데이터를 이전합니다.
</details>

**Q5.** 100TB의 데이터를 AWS로 이전해야 하는데, 네트워크 대역폭이 부족합니다. 어떤 서비스를 사용해야 하는가?

<details>
<summary>정답</summary>

**AWS Snow Family (Snowball Edge)**. 네트워크 전송이 비현실적인 대용량 데이터는 Snow Family 디바이스를 통한 물리적 전송이 효율적입니다.
</details>

**Q6.** RPO가 1시간이라는 것은 어떤 의미인가?

<details>
<summary>정답</summary>

재해 발생 시 **최대 1시간 분량의 데이터가 손실**될 수 있다는 의미입니다. 즉, 마지막 백업/복제 시점이 최대 1시간 전이라는 뜻입니다.
</details>

**Q7.** O/X: Snowmobile은 현재(2025~2026) AWS에서 사용 가능한 서비스이다.

<details>
<summary>정답</summary>

**X**. Snowmobile은 2024년에 폐기(deprecate)되었습니다. 현재 Snow Family는 Snowcone과 Snowball Edge만 사용 가능합니다.
</details>

> | Tomcat 앱 서버 | ? | ? |
> | MySQL DB (5TB) | ? | ? |
> | NFS 파일 서버 (30TB) | ? | ? |
> | 레거시 HR 시스템 | ? | ? |

> **Step 2**: CAF의 6가지 관점에서 이 마이그레이션에 대해 한 가지씩 고려 사항을 적어보세요.
>
> **✅ 체크포인트**: 아래 모범 답안과 비교하여 자신의 전략이 합리적인지 확인합니다.
>
> <details>
> <summary>모범 답안</summary>
>
> | 시스템 | 전략 | 이유 |
> |--------|------|------|
> | Apache 웹 서버 | **Rehost** | 설정이 단순하고 빠르게 이전 가능 |
> | Tomcat 앱 서버 | **Rehost** | Java 앱을 그대로 EC2에서 실행 |
> | MySQL DB | **Replatform** | RDS for MySQL로 이전하면 DB 관리 자동화 |
> | NFS 파일 서버 | **Rehost** (DataSync로 S3에 이전) | 대용량 파일을 S3로 이전하여 스토리지 비용 절감 |
> | 레거시 HR 시스템 | **Retire** | 곧 폐기 예정이므로 이전 불필요 |
>
> CAF 관점 예시:
> - **비즈니스**: 데이터센터 운영비 절감 효과 분석
> - **사람**: IT팀의 AWS 교육 필요
> - **거버넌스**: 마이그레이션 예산 및 일정 관리
> - **플랫폼**: EC2 인스턴스 타입, RDS 사양 설계
> - **보안**: 데이터 전송 시 암호화 확인
> - **운영**: 마이그레이션 후 모니터링 계획 수립
> </details>

> **Phase 2: AWS Migration Hub 탐색** (활용 개념: Section 20.3)
>
> **Step 1**: AWS 콘솔 상단 검색창에 `Migration Hub`를 입력하고 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **Dashboard**를 클릭합니다. 마이그레이션 상태를 요약하는 화면을 확인합니다 (현재는 데이터가 없습니다).
>
> **Step 3**: **Discover → Servers** 메뉴를 확인합니다. 실제 마이그레이션에서는 여기에 Application Discovery Service로 수집한 서버 목록이 표시됩니다.
>
> **Step 4**: **Migrate → Applications** 메뉴를 확인합니다. 마이그레이션할 애플리케이션을 그룹으로 관리하는 화면입니다.
>
> **✅ 체크포인트**: Migration Hub가 "추적 도구"임을 이해했습니다 — 직접 이전하는 것이 아니라, 다른 도구(MGN, DMS)의 진행 상황을 한곳에서 모니터링합니다.

> **Phase 3: AWS DMS 콘솔 탐색** (활용 개념: Section 20.4)
>
> **Step 1**: AWS 콘솔 상단 검색창에 `DMS`를 입력하고 **AWS Database Migration Service**를 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 주요 항목을 확인합니다:
> - **Replication instances**: 마이그레이션을 수행하는 인스턴스 (실제 생성하지 않습니다)
> - **Endpoints**: 소스 DB와 대상 DB의 연결 정보
> - **Database migration tasks**: 마이그레이션 작업 목록
>
> **Step 3**: **Endpoints** 메뉴를 클릭하고, "Create endpoint" 버튼을 확인합니다. "Source endpoint"와 "Target endpoint"로 나뉘어 있음을 확인합니다. (실제 생성하지 않습니다.)
>
> **✅ 체크포인트**: DMS의 구성 요소가 **복제 인스턴스 + 소스 엔드포인트 + 대상 엔드포인트 + 마이그레이션 태스크**로 이루어져 있음을 확인했습니다.

> **Phase 4: AWS Snow Family 콘솔 탐색** (활용 개념: Section 20.5)
>
> **Step 1**: AWS 콘솔 상단 검색창에 `Snow Family`를 입력하고 클릭합니다.
>
> **Step 2**: Snow Family 콘솔의 메인 화면에서 "Order an AWS Snow Family device" 버튼을 확인합니다. (실제 주문하지 않습니다.)
>
> **Step 3**: 디바이스 선택 화면에서 Snowcone과 Snowball Edge의 옵션을 확인합니다. 각 디바이스의 스토리지 용량과 컴퓨팅 사양이 표시됩니다.
>
> **Step 4**: 주문 과정의 첫 화면만 확인합니다: Job name, Job type(Import into S3 / Export from S3 / Local compute and storage), 디바이스 선택 등의 옵션이 있습니다.
>
> **✅ 체크포인트**: Snow Family 주문 과정에서 "데이터 가져오기(Import)", "데이터 내보내기(Export)", "로컬 컴퓨팅(Local compute)" 세 가지 Job 유형이 있음을 확인했습니다.

### 최종 확인

- Migration Hub에서 마이그레이션 전체를 추적하는 중앙 대시보드의 구조를 확인했습니다

- 사고 실습을 통해 6R 전략과 CAF 관점을 실제 시나리오에 적용해보았습니다

### 🧹 리소스 정리 (Clean-up)

> ⚠️ **비용 방지**: 이 실습에서는 콘솔 탐색만 수행했으므로, 생성된 리소스가 없습니다.
>
> **확인 사항**: DMS에서 Replication instance를 실수로 생성했다면 즉시 삭제하세요. DMS 인스턴스는 실행 중인 동안 시간당 비용이 발생합니다.
> - DMS 콘솔 → Replication instances → 해당 인스턴스 선택 → Actions → Delete

### 트러블슈팅 (Troubleshooting)

> **Q: Migration Hub 콘솔에서 "Access Denied" 오류가 나타납니다.**
> A: IAM 사용자에게 `AWSMigrationHubFullAccess` 정책이 연결되어 있는지 확인하세요. 루트 계정이라면 이 문제가 발생하지 않습니다.

> **Q: Snow Family 콘솔이 보이지 않습니다.**
> A: Snow Family는 모든 리전에서 사용할 수 있는 것은 아닙니다. 리전을 미국 동부(us-east-1)로 변경하여 확인해보세요. 콘솔 탐색 목적이므로 리전은 크게 중요하지 않습니다.

> **Q: DMS에서 실수로 Replication instance를 생성했습니다.**
> A: 즉시 삭제하세요. DMS 콘솔 → Replication instances → 인스턴스 선택 → Actions → Delete. 삭제에 몇 분이 소요될 수 있습니다. 인스턴스가 활성 상태인 동안 비용이 발생하므로 가능한 빨리 삭제하는 것이 좋습니다.

> - 마이그레이션 중 비즈니스 중단 최소화
> - 재해 복구 능력 확보 (RPO 1시간 이내, RTO 4시간 이내)
> - 비용 최적화
>
> (1) 각 시스템에 대한 마이그레이션 전략을 제시하세요
> (2) 사용할 AWS 서비스를 매핑하세요
> (3) 마이그레이션 순서(Phase)를 설계하세요
> (4) 재해 복구 방안을 포함하세요
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 18개월이라는 시간 제약과 VMware 환경이라는 특수성을 고려하여, 빠르고 안전한 이전을 우선합니다.
>
> **풀이 전략**:
>
> **(1) 시스템별 전략**:
>
> | 시스템 | 전략 | 이유 |
> |--------|------|------|
> | 서버 50대 (VMware) | **Rehost** (VMware Cloud on AWS 또는 MGN) | VMware 환경이므로 VMware Cloud on AWS가 가장 빠른 경로. 또는 MGN으로 EC2 이전 |
> | Oracle 메인 DB | **Replatform** (DMS → RDS for Oracle) | 운영 중단 없이 이전, 관리 부담 경감 |
> | MySQL 보조 DB 2개 | **Replatform** (DMS → RDS for MySQL) | 동종 마이그레이션, DMS로 무중단 이전 |
> | 파일 서버 200TB | **Rehost** (Snow Family + DataSync) | 200TB는 인터넷 전송 비현실적, Snowball Edge로 초기 이전 후 DataSync로 잔여 데이터 동기화 |
>
> **(2) AWS 서비스 매핑**:
> - **현황 파악**: Application Discovery Service → Migration Hub
> - **서버 이전**: VMware Cloud on AWS 또는 MGN
> - **DB 이전**: DMS (Oracle, MySQL), SCT (Oracle → 향후 Aurora 전환 시)
> - **파일 이전**: Snow Family (Snowball Edge × 3대) + DataSync
> - **재해 복구**: AWS Elastic Disaster Recovery (DRS)
> - **추적**: Migration Hub
>
> **(3) 마이그레이션 Phase**:
>
> - **Phase 1 (1~3개월)**: Discovery + 파일럿
>   - Application Discovery Service로 전체 서버 현황 파악
>   - 비핵심 서버 5대를 MGN으로 파일럿 이전
>   - Snowball Edge 주문 및 파일 서버 데이터 초기 전송 시작
>
> - **Phase 2 (4~9개월)**: 핵심 인프라 이전
>   - VMware Cloud on AWS 또는 MGN으로 나머지 서버 이전
>   - DMS로 MySQL 보조 DB 이전 (비교적 간단)
>   - DataSync로 파일 서버 지속 동기화 시작
>
> - **Phase 3 (10~15개월)**: DB 이전 + 최적화
>   - DMS로 Oracle 메인 DB 이전 (가장 위험도 높으므로 마지막)
>   - 전체 시스템 통합 테스트
>   - DRS 설정으로 재해 복구 체계 구축
>
> - **Phase 4 (16~18개월)**: 검증 + 정리
>   - 전체 시스템 운영 안정성 확인
>   - 온프레미스 데이터센터 리소스 정리
>   - 임대 계약 종료
>
> **(4) 재해 복구 방안**:
> - AWS Elastic Disaster Recovery를 모든 핵심 서버에 설정
> - RPO < 1시간: DRS의 지속적 복제로 서브초~분 수준 RPO 달성 (요구사항 충족)
> - RTO < 4시간: DRS의 즉시 복구 + DNS 전환으로 분 단위 RTO 달성 (요구사항 충족)
> - DB는 RDS Multi-AZ 배포로 추가적인 고가용성 확보
>
> **확장 생각**: 200TB 파일 서버는 단순히 S3에 올리는 것이 아니라, S3 스토리지 클래스 전략(자주 접근하는 최근 문서 → S3 Standard, 오래된 문서 → S3 Glacier)을 적용하면 장기적으로 스토리지 비용을 크게 절감할 수 있습니다.
> </details>

> **Problem 20.2**: DMS를 사용한 이기종 데이터베이스 마이그레이션에서 발생할 수 있는 **기술적 도전과 해결 전략**을 분석하세요.
>
> **상황**: Oracle 데이터베이스(10TB, 저장 프로시저 500개, 트리거 200개)를 Amazon Aurora(PostgreSQL 호환)로 이전합니다. 매일 100만 건의 트랜잭션이 발생하며, 이전 중에도 서비스를 중단할 수 없습니다.
>
> (1) 이 마이그레이션에서 예상되는 주요 도전과제 3가지를 식별하세요
> (2) 각 도전과제에 대한 해결 전략을 제시하세요
> (3) 이 상황에서 Replatform(DMS + SCT)이 아닌 다른 전략을 고려할 수 있는 상황은?
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 이기종 DB 마이그레이션은 "데이터 옮기기"보다 "호환성 확보"가 더 큰 도전입니다.
>
> **(1) 주요 도전과제**:
>
> 1. **스키마 호환성**: Oracle의 저장 프로시저 500개와 트리거 200개가 PostgreSQL 문법으로 **100% 자동 변환되지 않을 수 있음**
>    - Oracle 전용 기능(PL/SQL, Oracle 고유 함수 등)은 SCT가 변환하지 못하는 경우가 있음
>
> 2. **데이터 타입 차이**: Oracle과 PostgreSQL의 데이터 타입이 완전히 일치하지 않음
>    - 예: Oracle의 NUMBER 타입 → PostgreSQL의 NUMERIC/BIGINT 매핑, CLOB/BLOB 처리
>
> 3. **지속적 복제 성능**: 매일 100만 건의 트랜잭션이 발생하는 환경에서, DMS의 복제 지연(replication lag)이 커질 수 있음
>
> **(2) 해결 전략**:
>
> 1. **스키마 호환성 해결**:
>    - SCT로 자동 변환 가능한 항목과 수동 변환이 필요한 항목을 먼저 분류
>    - 수동 변환이 필요한 저장 프로시저/트리거를 PostgreSQL(PL/pgSQL)로 재작성
>    - 충분한 테스트 기간(최소 4~8주)을 확보
>
> 2. **데이터 타입 해결**:
>    - SCT의 데이터 타입 매핑 규칙을 사전에 검토·조정
>    - 전환 후 데이터 검증(데이터 카운트, 체크섬 비교)을 반드시 수행
>
> 3. **복제 성능 해결**:
>    - DMS 복제 인스턴스의 사양을 충분히 높게 설정
>    - 초기 전체 로드와 변경 데이터 캡처(CDC)를 분리하여 실행
>    - Cutover 직전에 복제 지연이 최소화된 시점을 선택
>
> **(3) 다른 전략을 고려할 상황**:
>
> - **Rehost (Oracle on EC2)**: SCT 수동 변환 범위가 너무 크고, 일정이 촉박한 경우. 우선 Oracle DB를 EC2에 그대로 올린 후, 이후에 Aurora로 전환
> - **Retain**: Oracle 라이선스 비용이 문제가 아니고, 마이그레이션 리스크가 비즈니스 영향보다 큰 경우
> - **Refactor**: 저장 프로시저 의존도가 너무 높아 DB만 바꿔서는 안 되고, 애플리케이션 자체를 재설계해야 하는 경우
>
> **확장 생각**: 실제 현장에서 이기종 DB 마이그레이션은 가장 어려운 마이그레이션 유형 중 하나입니다. "기술적으로 가능"한 것과 "현실적으로 안전"한 것의 간극을 파악하는 것이 아키텍트의 핵심 역량입니다. 때로는 2단계 전략(먼저 Rehost → 안정화 후 Refactor)이 가장 현명한 선택일 수 있습니다.
> </details>
