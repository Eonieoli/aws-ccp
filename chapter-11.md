## 11.1 데이터 분석 서비스 개요

> 💡 **한 줄 요약**: AWS는 데이터를 저장하는 것에서 그치지 않고, 저장된 데이터에서 가치를 추출하기 위한 다양한 분석 서비스를 제공합니다.

### 직관적 이해

여러분이 대형 도서관의 사서라고 상상해 보세요. 도서관에는 수백만 권의 책이 있습니다. 책을 보관하는 것 자체는 중요하지만, 진짜 가치는 **"누가, 어떤 책을, 언제, 얼마나 자주 빌려가는지"**를 분석하여 도서관 운영을 개선하는 데 있습니다. 가장 인기 있는 장르는 무엇인지, 어떤 시간대에 방문자가 많은지, 신간 구매 예산을 어디에 집중해야 하는지 — 이런 **인사이트(insight)**를 얻으려면 데이터를 **분석**해야 합니다.

기업도 마찬가지입니다. 웹사이트 방문 기록, 판매 데이터, 센서 수집 정보 등 매일 엄청난 양의 데이터가 쌓입니다. 하지만 이 데이터를 그대로 두면 그저 저장 비용만 발생할 뿐입니다. AWS의 데이터 분석 서비스는 이 **"쌓인 데이터를 가치 있는 정보로 변환하는 도구"**입니다.

> 🔗 **연결**: Chapter 6에서 Amazon S3가 사실상 무한한 데이터를 저장할 수 있는 객체 스토리지임을 배웠습니다. 이 Chapter에서는 S3에 저장된 데이터를 *분석하는* 서비스들을 다룹니다.

### 핵심 개념 설명

AWS의 데이터 분석 서비스는 크게 네 가지 역할로 나눌 수 있습니다:

> 📝 **Amazon Athena**: S3에 저장된 데이터를 표준 SQL로 직접 쿼리(조회)할 수 있는 **서버리스 쿼리 서비스**입니다. 별도의 서버를 구축하거나 데이터를 이동시킬 필요 없이, S3의 데이터를 바로 분석할 수 있습니다.

> 📝 **AWS Glue**: 데이터를 분석에 적합한 형태로 변환하는 **서버리스 ETL(Extract, Transform, Load)** 서비스입니다. 여러 소스에서 데이터를 추출하고, 정제·변환한 뒤, 분석 시스템에 적재합니다.

> 📝 **Amazon EMR (Elastic MapReduce)**: Apache Spark, Hadoop 등 **빅데이터 프레임워크**를 관리형으로 실행할 수 있는 서비스입니다. 대규모 데이터 처리와 분석에 사용됩니다.

> 📝 **Amazon QuickSight**: 데이터를 차트, 그래프, 대시보드로 **시각화**하는 **비즈니스 인텔리전스(BI)** 서비스입니다. 기술 지식이 없는 비즈니스 사용자도 데이터를 시각적으로 탐색할 수 있습니다.

이 네 서비스의 관계를 도서관 비유로 정리하면:

| 서비스 | 도서관 비유 | 역할 |
|--------|------------|------|
| **S3** | 도서관 서가 | 데이터(책)를 저장 |
| **Glue** | 사서의 정리 작업 | 데이터를 분류·정리·변환 |
| **Athena** | 검색 시스템 | 저장된 데이터에 질문(쿼리) |
| **EMR** | 대규모 조사 팀 | 방대한 데이터를 처리·분석 |
| **QuickSight** | 통계 보고서/차트 | 분석 결과를 시각적으로 표현 |

#### Athena — S3 위에서 SQL로 바로 분석

Athena의 가장 큰 장점은 **서버리스**라는 점입니다. 데이터베이스 서버를 별도로 구축할 필요 없이, S3에 있는 CSV, JSON, Parquet 등의 파일을 SQL 쿼리로 바로 조회할 수 있습니다.

> 🔗 **연결**: Chapter 8에서 배운 관계형 데이터베이스(RDS, Aurora)는 데이터를 *데이터베이스 내부*에 저장하고 쿼리했습니다. Athena는 데이터를 이동시키지 않고 S3에 그대로 둔 채 쿼리한다는 점이 핵심 차이입니다.

Athena의 특징:
- **서버리스**: 인프라 관리 불필요, 쿼리를 실행할 때만 비용 발생
- **과금 방식**: 스캔한 데이터 양에 따라 과금 (1TB당 약 $5)
- **표준 SQL 지원**: 이미 SQL을 아는 사람이라면 즉시 사용 가능
- **다양한 데이터 형식**: CSV, JSON, ORC, Avro, Parquet 등 지원

#### Glue — 데이터를 분석에 적합하게 준비

실세계의 데이터는 지저분합니다. 날짜 형식이 다르거나, 불필요한 필드가 있거나, 여러 시스템에 흩어져 있습니다. AWS Glue는 이런 데이터를 **추출(Extract)**, **변환(Transform)**, **적재(Load)**하는 ETL 과정을 자동화합니다.

> 📝 **ETL (Extract, Transform, Load)**: 다양한 데이터 소스에서 데이터를 가져오고(Extract), 분석에 적합한 형태로 변환하고(Transform), 목적지에 저장(Load)하는 과정입니다. 요리에 비유하면, 시장에서 재료를 사고(Extract), 씻고 손질하고(Transform), 냄비에 넣는(Load) 과정과 같습니다.

Glue의 주요 구성 요소:
- **Glue Data Catalog**: 데이터의 위치, 구조, 형식 등 메타데이터를 자동으로 탐색하여 카탈로그로 관리. 마치 도서관의 목록 시스템처럼, "어떤 데이터가, 어디에, 어떤 형식으로 있는지"를 기록합니다.
- **Glue ETL Jobs**: 데이터 변환 작업을 자동으로 실행
- **Glue Crawlers**: S3, RDS 등의 데이터 소스를 자동으로 탐색하여 Data Catalog에 등록

#### EMR — 빅데이터 프레임워크 실행

일반적인 분석 도구로는 처리하기 어려운 **페타바이트(PB) 규모**의 데이터를 다뤄야 할 때 EMR을 사용합니다. EMR은 Apache Spark, Hadoop, Hive, Presto 같은 오픈소스 빅데이터 프레임워크를 AWS의 관리형 클러스터에서 실행합니다.

> 🔗 **연결**: Chapter 5에서 배운 EC2 인스턴스의 개념을 떠올려 봅시다. EMR은 여러 대의 EC2 인스턴스를 클러스터로 묶어 빅데이터 프레임워크를 실행하는 것이라 생각할 수 있습니다.

- **사용 시나리오**: 로그 분석, 대규모 데이터 변환, 머신러닝 데이터 전처리
- **관리형**: 클러스터 프로비저닝, 구성, 튜닝을 AWS가 관리
- **비용 효율**: 작업이 끝나면 클러스터를 종료하여 비용 절감 가능

#### QuickSight — 데이터 시각화와 BI

아무리 뛰어난 분석도, 결과를 사람이 이해할 수 있는 형태로 보여주지 못하면 의미가 줄어듭니다. QuickSight는 분석 결과를 **대시보드, 차트, 그래프**로 시각화하여 비즈니스 의사결정을 지원합니다.

특징:
- **서버리스**: 인프라 관리 불필요
- **다양한 데이터 소스 연결**: S3, Athena, RDS, Redshift 등과 직접 연결
- **ML 기반 인사이트**: 데이터에서 이상 탐지, 예측 등을 자동으로 수행
- **공유 및 협업**: 대시보드를 조직 내 사용자와 공유 가능

### 상세 예시

> 🔍 **예시 11.1.1: 전자상거래 회사의 분석 파이프라인**
>
> **상황**: 온라인 쇼핑몰에서 매일 수백만 건의 주문 데이터가 S3에 CSV 형식으로 저장됩니다. 마케팅 팀은 "지난 한 달 동안 가장 많이 팔린 상품 카테고리는?"이라는 질문에 답하고 싶습니다.
>
> **풀이/설명**:
> 1. 주문 데이터는 이미 S3에 CSV로 저장되어 있음
> 2. **Athena**에서 SQL 쿼리를 작성: `SELECT category, COUNT(*) as order_count FROM orders WHERE order_date >= '2026-03-01' GROUP BY category ORDER BY order_count DESC`
> 3. 쿼리 결과를 **QuickSight**에 연결하여 막대 차트로 시각화
>
> **핵심 포인트**: 별도의 데이터베이스를 구축하지 않고도, S3의 데이터를 Athena로 SQL 쿼리하고 QuickSight로 시각화할 수 있습니다.

> 🔍 **예시 11.1.2: 일상 비유 — 가계부 분석**
>
> **상황**: 매달 가계부(데이터)를 엑셀 파일(S3)에 기록하고 있습니다. 연말에 "올해 가장 지출이 많은 항목은?"이라는 질문에 답하고 싶습니다.
>
> **풀이/설명**:
> - **S3** = 엑셀 파일이 저장된 폴더
> - **Athena** = 엑셀의 필터/피벗 테이블 기능 (데이터에 질문하기)
> - **QuickSight** = 엑셀의 차트 기능 (결과를 그래프로 보기)
> - **Glue** = 만약 가계부 형식이 월마다 다르다면, 형식을 통일해주는 작업
>
> **핵심 포인트**: AWS의 분석 서비스는 우리가 일상에서 엑셀로 하는 "데이터 정리 → 질문 → 시각화" 과정을 클라우드 규모로 확장한 것입니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **Athena vs RDS 혼동**: Athena는 S3의 데이터를 *직접 쿼리*하는 서비스이고, RDS는 데이터를 *데이터베이스 내부*에 저장하고 쿼리하는 서비스입니다. Athena는 대규모 데이터 분석에, RDS는 애플리케이션의 트랜잭션 처리에 적합합니다.
> 2. **Glue = ETL 도구**: Glue는 데이터 분석 자체를 하는 것이 아니라, 데이터를 분석에 *적합한 형태로 준비*하는 도구임을 기억하세요.
> 3. **EMR ≠ 서버리스**: Athena, Glue, QuickSight는 서버리스이지만, EMR은 EC2 클러스터를 사용합니다. EMR 클러스터를 사용 후 종료하지 않으면 지속적으로 비용이 발생합니다.

### Section 요약

- **Athena**: S3 데이터를 SQL로 직접 쿼리하는 서버리스 분석 서비스
- **Glue**: 데이터를 추출·변환·적재(ETL)하는 서버리스 데이터 통합 서비스. Data Catalog으로 메타데이터 관리
- **EMR**: Apache Spark/Hadoop 등 빅데이터 프레임워크를 관리형 클러스터에서 실행
- **QuickSight**: 데이터를 대시보드와 차트로 시각화하는 BI 서비스
- 이 네 서비스는 데이터 준비(Glue) → 쿼리/처리(Athena, EMR) → 시각화(QuickSight) 파이프라인을 구성

## 11.2 데이터 레이크와 분석 파이프라인

> 💡 **한 줄 요약**: 데이터 레이크는 모든 유형의 데이터를 원본 그대로 한 곳에 모아두는 중앙 저장소이며, AWS에서는 S3를 기반으로 구축합니다.

### 직관적 이해

**데이터 웨어하우스**가 잘 정리된 서류 캐비닛이라면, **데이터 레이크**는 거대한 호수와 같습니다. 호수에는 다양한 물이 흘러들어옵니다 — 깨끗한 계곡물, 빗물, 지하수. 마찬가지로 데이터 레이크에는 정형 데이터(표 형태), 반정형 데이터(JSON, XML), 비정형 데이터(이미지, 동영상, 로그 파일) 등 **모든 종류의 데이터가 원본 그대로** 흘러들어옵니다.

왜 이런 접근이 필요할까요? 전통적인 데이터 웨어하우스는 데이터를 저장하기 *전에* 형식을 맞추고 정리해야 했습니다(스키마 온 라이트, Schema on Write). 하지만 빅데이터 시대에는 "지금은 어떻게 쓸지 모르지만, 일단 모아두면 나중에 가치를 발견할 수 있다"는 발상이 중요해졌습니다. 데이터 레이크는 데이터를 *조회할 때* 구조를 부여합니다(스키마 온 리드, Schema on Read).

> 🔗 **연결**: Chapter 6에서 배운 Amazon S3의 특징을 떠올려 봅시다 — 사실상 무한한 저장 용량, 높은 내구성(99.999999999%), 다양한 스토리지 클래스. 이러한 S3의 특성이 데이터 레이크의 기반 저장소로 S3가 선택되는 이유입니다.

### 핵심 개념 설명

> 📝 **데이터 레이크(Data Lake)**: 정형, 반정형, 비정형 데이터를 원본 형태 그대로 저장하는 **중앙 집중식 저장소**입니다. 데이터의 규모, 형식에 관계없이 저장하고, 필요할 때 다양한 분석 도구로 처리합니다.

#### S3 기반 데이터 레이크 아키텍처

AWS에서 데이터 레이크를 구축하는 일반적인 아키텍처:

```
[데이터 소스]                    [S3 데이터 레이크]              [분석/소비]
                                      │
 IoT 센서 ──────┐              ┌──── Raw Zone ────┐
 애플리케이션 ──┼── 수집 ──→  ├── Processed Zone ─┼──→  Athena (SQL 쿼리)
 데이터베이스 ──┤      ↑      └── Curated Zone ───┘     EMR (빅데이터)
 외부 API ──────┘   Kinesis/                              QuickSight (시각화)
                    Glue                                   SageMaker (ML)
```

데이터 레이크는 보통 **세 개의 영역(Zone)**으로 구성됩니다:

| 영역 | 설명 | 비유 |
|------|------|------|
| **Raw Zone** (원시 영역) | 원본 데이터를 그대로 저장 | 시장에서 가져온 재료 그대로 |
| **Processed Zone** (가공 영역) | 정제·변환된 데이터 | 재료를 씻고 손질한 상태 |
| **Curated Zone** (큐레이션 영역) | 분석 목적에 맞게 최적화된 데이터 | 요리에 바로 투입 가능한 상태 |

이 영역 간의 데이터 이동과 변환을 **AWS Glue**가 담당합니다.

#### AWS Data Exchange

> 📝 **AWS Data Exchange**: 서드파티(3rd party) 데이터를 AWS 마켓플레이스에서 구독하여 받을 수 있는 서비스입니다. 날씨 데이터, 금융 데이터, 인구통계 데이터 등 외부 데이터를 자신의 S3에 직접 전달받아 분석에 활용할 수 있습니다.

Data Exchange는 마치 **데이터를 위한 앱스토어**와 같습니다. 직접 수집하기 어려운 외부 데이터를 구독형으로 받아, 자사 데이터와 결합하여 더 풍부한 분석을 할 수 있습니다.

### 상세 예시

> 🔍 **예시 11.2.1: 날씨 데이터와 매출 분석**
>
> **상황**: 편의점 체인이 날씨에 따른 매출 패턴을 분석하고 싶습니다. 자체 POS(판매) 데이터는 있지만, 날씨 데이터는 없습니다.
>
> **풀이/설명**:
> 1. **자체 매출 데이터**: 각 매장의 POS 시스템에서 S3 데이터 레이크의 Raw Zone에 매일 적재
> 2. **외부 날씨 데이터**: AWS Data Exchange에서 기상 데이터 구독 → S3에 자동 전달
> 3. **Glue**: 매출 데이터와 날씨 데이터를 날짜·지역 기준으로 결합(Join)하여 Processed Zone에 저장
> 4. **Athena**: "비 오는 날 우산 매출은 얼마나 증가하는가?" 같은 SQL 쿼리 실행
> 5. **QuickSight**: 결과를 대시보드로 시각화
>
> **핵심 포인트**: 데이터 레이크는 내부 데이터와 외부 데이터를 결합하여 새로운 인사이트를 만들어내는 데 적합합니다.

> 🔍 **예시 11.2.2: 일상 비유 — 사진 정리**
>
> **상황**: 스마트폰, 디지털 카메라, SNS에서 수년간 찍은 사진(데이터)을 모두 하나의 외장 하드(데이터 레이크)에 모았습니다.
>
> **풀이/설명**:
> - **Raw Zone** = 모든 사진을 날짜별 폴더에 원본 그대로 저장
> - **Processed Zone** = 중복 제거, 흐린 사진 분류, EXIF 정보 추출
> - **Curated Zone** = "가족 여행", "음식", "풍경" 등 주제별로 최종 정리
> - 나중에 "2025년에 방문한 해변 사진"을 찾을 때(분석/쿼리), 정리가 잘 되어 있으므로 빠르게 찾을 수 있음
>
> **핵심 포인트**: 데이터 레이크의 핵심은 "일단 모아두고, 필요할 때 구조화한다"는 접근 방식입니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **데이터 레이크 vs 데이터 웨어하우스 혼동**: 데이터 레이크는 원본 데이터를 그대로 저장(Schema on Read)하고, 데이터 웨어하우스는 정제된 데이터를 구조화하여 저장(Schema on Write)합니다. 둘은 대체 관계가 아니라 보완 관계입니다.
> 2. **"데이터 늪(Data Swamp)" 주의**: 데이터를 무질서하게 쌓기만 하고 관리하지 않으면, 아무도 활용할 수 없는 "데이터 늪"이 됩니다. Glue Data Catalog 같은 메타데이터 관리가 중요합니다.

### Section 요약

- **데이터 레이크**는 모든 유형의 데이터를 원본 그대로 저장하는 중앙 저장소로, AWS에서는 S3를 기반으로 구축
- Raw → Processed → Curated 영역으로 데이터를 점진적으로 정제
- **AWS Data Exchange**를 통해 외부 서드파티 데이터를 구독하여 분석에 활용 가능
- 데이터 레이크는 데이터 웨어하우스와 보완적으로 함께 사용되는 경우가 많음

## 11.3 AWS AI/ML 서비스 계층

> 💡 **한 줄 요약**: AWS는 AI/ML 전문 지식 수준에 따라 "사전 구축 AI → ML 플랫폼 → 프레임워크/인프라" 세 계층의 서비스를 제공하여, 누구나 AI를 활용할 수 있게 합니다.

### 직관적 이해

요리에 비유해 봅시다. 음식을 먹고 싶을 때 세 가지 방법이 있습니다:

1. **배달 음식 주문** (가장 쉬움): 메뉴를 고르고 주문만 하면 됩니다. 요리 지식이 전혀 없어도 됩니다.
2. **밀키트 사용** (중간): 재료와 레시피가 준비되어 있고, 여러분이 직접 조리합니다. 기본적인 요리 기술이 필요합니다.
3. **처음부터 요리** (가장 어려움): 재료를 직접 고르고, 레시피를 개발하고, 장비도 직접 관리합니다. 전문적인 요리 기술이 필요합니다.

AWS의 AI/ML 서비스도 이와 같은 **세 계층**으로 나뉩니다:

### 핵심 개념 설명

```
┌─────────────────────────────────────────────────┐
│   계층 1: 사전 구축 AI 서비스 (AI Services)      │  ← ML 지식 불필요
│   Rekognition, Comprehend, Polly, Transcribe... │     API 호출만으로 사용
├─────────────────────────────────────────────────┤
│   계층 2: ML 플랫폼 (SageMaker AI)              │  ← ML 지식 필요
│   모델 구축, 학습, 배포를 위한 통합 환경          │     데이터 과학자 대상
├─────────────────────────────────────────────────┤
│   계층 3: ML 프레임워크 & 인프라                  │  ← 깊은 ML 전문성 필요
│   TensorFlow, PyTorch + EC2/GPU 인스턴스         │     ML 연구자/엔지니어 대상
└─────────────────────────────────────────────────┘
```

#### 계층 1: 사전 구축 AI 서비스 (Pre-built AI Services)

ML(머신러닝) 지식이 없어도 **API를 호출**하는 것만으로 AI 기능을 사용할 수 있는 서비스입니다. AWS가 이미 학습시킨 모델을 사용하므로, 데이터를 보내면 결과를 받을 수 있습니다.

배달 음식처럼 — "사진을 보내면 사진 속 객체를 알려줍니다(Rekognition)", "텍스트를 보내면 음성으로 읽어줍니다(Polly)" 같은 방식입니다.

주요 서비스는 다음 Section(11.4)에서 자세히 다룹니다.

#### 계층 2: ML 플랫폼 (Amazon SageMaker AI)

> 📝 **Amazon SageMaker AI**: 머신러닝 모델을 **구축(Build)**, **학습(Train)**, **배포(Deploy)**하는 전 과정을 지원하는 통합 ML 플랫폼입니다.

사전 구축 AI 서비스가 제공하는 기능만으로는 부족하고, **자사 데이터로 맞춤 모델을 만들어야** 할 때 SageMaker AI를 사용합니다. 밀키트처럼 — 재료(데이터)와 도구(알고리즘)는 준비되어 있고, 여러분이 직접 조리(모델 학습)합니다.

SageMaker AI의 주요 기능:
- **SageMaker Studio**: 웹 기반 통합 개발 환경(IDE)
- **내장 알고리즘**: 자주 사용되는 ML 알고리즘을 미리 제공
- **자동 모델 튜닝**: 최적의 하이퍼파라미터를 자동으로 탐색
- **모델 배포**: 학습된 모델을 API 엔드포인트로 배포

#### 계층 3: ML 프레임워크 & 인프라

가장 높은 자유도를 가진 계층입니다. TensorFlow, PyTorch 같은 오픈소스 프레임워크를 사용하여, EC2의 GPU 인스턴스에서 직접 모델을 설계하고 학습시킵니다. 처음부터 요리하는 셰프처럼 — 완전한 제어권을 가지지만, 전문적인 ML 지식이 필요합니다.

> 💬 **참고**: CLF-C02 시험에서는 이 세 계층의 **구분과 각각의 대표 서비스를 아는 것**이 중요합니다. 계층 3의 세부 내용까지 깊게 다루지는 않습니다.

### 상세 예시

> 🔍 **예시 11.3.1: 쇼핑몰 리뷰 분석**
>
> **상황**: 전자상거래 사이트에서 고객 리뷰의 감정(긍정/부정)을 분석하고 싶습니다.
>
> **풀이/설명**:
> - **계층 1 접근** (가장 간단): Amazon Comprehend API에 리뷰 텍스트를 전송하면, 바로 감정 분석 결과를 반환합니다. ML 지식이 필요 없으며, 몇 줄의 코드로 구현 가능합니다.
> - **계층 2 접근** (맞춤 모델): 일반적인 감정 분석이 자사 제품 리뷰의 특수한 표현을 잘 인식하지 못한다면, SageMaker AI에서 자사 리뷰 데이터로 맞춤 모델을 학습시킬 수 있습니다.
> - **계층 3 접근** (완전 제어): 최신 연구 논문의 알고리즘을 직접 구현하거나, 초대규모 모델을 학습시켜야 한다면, GPU 인스턴스에서 PyTorch로 직접 구현합니다.
>
> **핵심 포인트**: 대부분의 비즈니스 사용 사례에서는 계층 1(사전 구축 AI)로 충분합니다. "필요한 만큼만 복잡하게" 접근하는 것이 중요합니다.

> 🔍 **예시 11.3.2: 일상 비유 — 교통수단 선택**
>
> **상황**: 서울에서 부산까지 이동해야 합니다.
>
> **풀이/설명**:
> - **계층 1 (사전 구축 AI)** = KTX: 가장 빠르고 편합니다. 운전 기술이 필요 없고, 정해진 역에서 타고 내리면 됩니다. 하지만 정해진 노선만 가능합니다.
> - **계층 2 (SageMaker AI)** = 렌터카: 원하는 경로를 선택할 수 있고, 중간에 원하는 곳에 들를 수 있습니다. 하지만 운전 기술이 필요합니다.
> - **계층 3 (프레임워크/인프라)** = 직접 자동차 제작: 완전한 맞춤이 가능하지만, 자동차 공학 지식이 필요합니다.
>
> **핵심 포인트**: 목적지(비즈니스 목표)는 같더라도, 필요한 유연성과 보유한 전문성에 따라 적합한 수단(계층)이 다릅니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **"AI = SageMaker" 라는 오해**: 시험에서 AI/ML 관련 문제가 나왔을 때, 무조건 SageMaker를 선택하지 마세요. 단순한 이미지 인식, 텍스트 분석 등에는 사전 구축 AI 서비스(Rekognition, Comprehend 등)가 정답인 경우가 많습니다.
> 2. **계층 선택 기준**: "ML 전문가 없이", "API 호출만으로", "즉시 사용" 같은 키워드가 나오면 **사전 구축 AI 서비스(계층 1)**를 떠올리세요. "맞춤 모델 학습", "자체 데이터로 훈련" 같은 키워드가 나오면 **SageMaker AI(계층 2)**를 떠올리세요.

### Section 요약

- AWS AI/ML 서비스는 **세 계층**으로 구분: 사전 구축 AI → ML 플랫폼(SageMaker AI) → 프레임워크/인프라
- **계층 1** (사전 구축 AI): ML 지식 없이 API 호출만으로 AI 기능 사용. 대부분의 일반 사용 사례에 적합
- **계층 2** (SageMaker AI): 자체 데이터로 맞춤 모델을 구축·학습·배포. 데이터 과학자 대상
- **계층 3** (프레임워크/인프라): TensorFlow, PyTorch 등으로 완전한 제어. ML 전문가 대상
- 시험에서는 각 계층의 특징과 대표 서비스를 구분하는 것이 중요

## 11.4 주요 AI/ML 서비스

> 💡 **한 줄 요약**: AWS는 이미지 인식, 자연어 처리, 음성 변환 등 다양한 AI 기능을 API 호출만으로 사용할 수 있는 사전 구축 AI 서비스를 제공합니다.

### 직관적 이해

스마트폰에는 이미 많은 AI 기능이 내장되어 있습니다 — 사진에서 얼굴을 인식하고, 음성 명령을 이해하고, 외국어를 번역합니다. AWS의 사전 구축 AI 서비스는 이와 같은 AI 기능을 **여러분의 애플리케이션에 추가**할 수 있게 해줍니다. 마치 레고 블록처럼, 필요한 AI 기능을 골라서 조합하면 됩니다.

### 핵심 개념 설명

AWS의 주요 사전 구축 AI 서비스를 **기능 영역별**로 살펴봅시다:

#### 비전 (이미지/동영상 분석)

> 📝 **Amazon Rekognition**: 이미지와 동영상에서 **객체, 사람, 텍스트, 장면, 활동**을 식별하는 컴퓨터 비전 서비스입니다.

주요 기능:
- **객체 및 장면 탐지**: "이 사진에 고양이가 있다", "해변 풍경이다"
- **얼굴 분석**: 얼굴 탐지, 감정 분석(웃는 얼굴, 놀란 얼굴), 얼굴 비교
- **텍스트 탐지**: 이미지 속 문자(간판, 번호판 등) 인식
- **콘텐츠 모더레이션**: 부적절한 콘텐츠 자동 감지
- **유명인 인식**: 이미지 속 유명인 식별

활용 사례: 소셜 미디어의 사진 자동 태깅, 보안 카메라의 이상 행동 탐지, 미디어 콘텐츠의 부적절한 이미지 필터링

#### 자연어 처리 (텍스트 분석)

> 📝 **Amazon Comprehend**: 텍스트에서 **감정, 핵심 구문, 언어, 엔티티(개체)**를 자동으로 추출하는 자연어 처리(NLP) 서비스입니다.

주요 기능:
- **감정 분석(Sentiment Analysis)**: 텍스트가 긍정적, 부정적, 중립적, 혼합적인지 판별
- **엔티티 인식**: 텍스트에서 사람, 장소, 날짜, 조직 등의 개체를 식별
- **핵심 구문 추출**: 텍스트의 핵심 주제를 파악
- **언어 탐지**: 텍스트가 어떤 언어로 쓰여졌는지 판별
- **주제 모델링**: 대량의 문서에서 공통 주제를 발견

활용 사례: 고객 리뷰 감정 분석, 지원 티켓 자동 분류, 소셜 미디어 여론 모니터링

> 📝 **Amazon Textract**: 스캔된 문서에서 **텍스트, 표, 양식 데이터**를 자동으로 추출하는 서비스입니다. 단순 OCR(광학 문자 인식)을 넘어, 문서의 구조(표, 양식의 키-값 쌍)를 이해합니다.

활용 사례: 신분증·운전면허증 정보 추출, 세금 신고서 자동 처리, 계약서 데이터 추출

#### 음성 관련 서비스

> 📝 **Amazon Polly**: 텍스트를 자연스러운 **음성으로 변환(Text-to-Speech, TTS)**하는 서비스입니다. 다양한 언어와 음성 스타일을 지원합니다.

> 📝 **Amazon Transcribe**: 음성을 **텍스트로 변환(Speech-to-Text, STT)**하는 서비스입니다. 실시간 또는 녹음된 오디오를 텍스트로 변환합니다.

이 두 서비스는 **반대 방향**의 변환을 수행합니다:
- **Polly**: 텍스트 → 음성 (읽어주기)
- **Transcribe**: 음성 → 텍스트 (받아쓰기)

활용 사례:
- Polly: 뉴스 기사 음성 읽기, 접근성 향상(시각 장애인 지원), 콜센터 자동 응답
- Transcribe: 회의 녹음 자동 전사(기록), 동영상 자막 생성, 콜센터 통화 기록 분석

#### 번역

> 📝 **Amazon Translate**: 텍스트를 다른 언어로 **자동 번역**하는 서비스입니다. 75개 이상의 언어를 지원합니다.

활용 사례: 다국어 웹사이트 콘텐츠 번역, 고객 채팅 실시간 번역, 다국어 문서 처리

#### 대화형 AI

> 📝 **Amazon Lex**: **챗봇과 음성 봇**을 구축하는 서비스입니다. Amazon Alexa(알렉사)와 동일한 대화 이해 기술을 사용합니다.

Lex는 사용자의 **의도(Intent)**를 파악하고, 필요한 정보를 수집하여, 적절한 동작을 수행하는 대화형 인터페이스를 만듭니다.

활용 사례: 고객 서비스 챗봇, 예약 시스템, FAQ 자동 응답

#### 지능형 검색

> 📝 **Amazon Kendra**: ML 기반의 **지능형 검색 서비스**입니다. 자연어로 질문하면 문서에서 정확한 답변을 찾아줍니다. 단순 키워드 매칭이 아닌, 질문의 의미를 이해하여 답변합니다.

기존 검색 엔진이 "키워드가 포함된 문서 목록"을 반환한다면, Kendra는 "질문에 대한 정확한 답변"을 반환합니다.

활용 사례: 사내 문서 검색 시스템, IT 헬프데스크, 규정/정책 문서 검색

### 상세 예시

> 🔍 **예시 11.4.1: 글로벌 고객 지원 시스템 구축**
>
> **상황**: 전 세계 고객을 대상으로 하는 기업이 다국어 고객 지원 시스템을 구축하려 합니다.
>
> **풀이/설명**:
> 1. 고객이 모국어로 채팅 문의를 보냄
> 2. **Amazon Translate**: 고객 메시지를 영어로 번역
> 3. **Amazon Comprehend**: 메시지의 감정(불만 vs 일반 문의)과 주제(결제, 배송, 반품 등)를 자동 분류
> 4. **Amazon Lex**: 간단한 FAQ는 챗봇이 자동 응답
> 5. 복잡한 문의는 담당 상담원에게 전달. 상담원의 응답을 다시 **Translate**로 고객 언어로 번역
> 6. 통화가 필요한 경우, 상담원의 응답을 **Amazon Polly**로 음성 변환
>
> **핵심 포인트**: 여러 AI 서비스를 레고 블록처럼 조합하여, ML 전문가 없이도 복잡한 AI 기능을 구현할 수 있습니다.

> 🔍 **예시 11.4.2: 일상 비유 — 스마트폰 AI 기능과의 비교**
>
> **상황**: 여러분의 스마트폰이 이미 제공하는 AI 기능을 생각해 봅시다.
>
> **풀이/설명**:
> | 스마트폰 기능 | 대응하는 AWS 서비스 |
> |---|---|
> | 사진 속 얼굴 인식 | Amazon Rekognition |
> | 음성 비서(시리, 구글 어시스턴트) | Amazon Lex + Polly + Transcribe |
> | 사진 속 텍스트 복사 | Amazon Textract |
> | 카메라 번역 | Amazon Translate |
> | 받아쓰기(딕테이션) | Amazon Transcribe |

> **핵심 포인트**: AWS의 AI 서비스는 스마트폰에 내장된 것과 유사한 AI 기능을 *여러분의 애플리케이션에* 추가할 수 있게 해줍니다.

### 미니 실습

> 🧪 **Mini Lab 11.4: Amazon Comprehend로 감정 분석 체험**
>
> **목표**: Amazon Comprehend의 실시간 분석 기능을 사용하여 텍스트의 감정을 분석해봅니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (12개월 무료 티어: 월 50,000자 무료)
>
> **Step 1**: AWS Management Console에 로그인합니다.
>
> **Step 2**: 상단 검색창에 "Comprehend"를 입력하고 **Amazon Comprehend**를 클릭합니다.
>
> **Step 3**: Comprehend 콘솔 페이지에서 **"Real-time analysis"** (실시간 분석) 버튼을 클릭합니다.
>
> **Step 4**: **"Input text"** 영역에 다음 텍스트를 입력합니다:
> ```
> I love this product! It works perfectly and the delivery was super fast. Best purchase I've made this year!
> ```
>
> **Step 5**: **"Analyze"** 버튼을 클릭합니다.
>
> **Step 6**: 결과를 확인합니다:
> - **Sentiment** (감정): "Positive"가 높은 점수를 보여야 합니다
> - **Entities** (엔티티): 텍스트에서 감지된 개체들이 표시됩니다
> - **Key phrases** (핵심 구문): 주요 구문이 추출됩니다
> - **Language** (언어): "en" (영어)로 감지됩니다
>
> **Step 7**: 이번에는 부정적인 텍스트를 입력하여 결과를 비교해봅니다:
> ```
> Terrible experience. The product arrived broken and customer service was unhelpful. Very disappointed.
> ```
>
> **확인**: 첫 번째 텍스트는 "Positive"가 높고, 두 번째 텍스트는 "Negative"가 높게 나오면 성공입니다.
>
> **🧹 정리**: Comprehend 실시간 분석은 별도의 리소스를 생성하지 않으므로 정리할 것이 없습니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **Polly vs Transcribe 혼동**: 시험에서 자주 혼동됩니다. **Polly = 텍스트→음성** (Polly는 "말하는" 서비스), **Transcribe = 음성→텍스트** (Transcribe는 "받아쓰는" 서비스)로 기억하세요.
> 2. **Textract vs Rekognition 차이**: 둘 다 이미지에서 정보를 추출하지만, **Textract는 문서 전문**(양식, 표 등 구조 이해), **Rekognition은 일반 이미지/동영상**(객체, 얼굴 등 시각적 요소)에 특화되어 있습니다.
> 3. **Lex vs Comprehend 구분**: **Lex는 대화형 봇을 만드는** 서비스이고, **Comprehend는 텍스트를 분석하는** 서비스입니다. Lex는 "대화"를 만들고, Comprehend는 텍스트를 "이해"합니다.

### Section 요약

- **Rekognition**: 이미지/동영상 분석 (객체 탐지, 얼굴 분석, 콘텐츠 모더레이션)
- **Comprehend**: 자연어 처리 (감정 분석, 엔티티 인식, 주제 모델링)
- **Textract**: 문서에서 텍스트·표·양식 데이터 추출
- **Polly**: 텍스트 → 음성(TTS), **Transcribe**: 음성 → 텍스트(STT) — 반대 방향
- **Translate**: 다국어 텍스트 번역
- **Lex**: 챗봇/음성 봇 구축 (Alexa 기술 기반)
- **Kendra**: ML 기반 지능형 검색

## 11.5 Amazon Bedrock과 생성형 AI

> 💡 **한 줄 요약**: Amazon Bedrock은 다양한 파운데이션 모델(FM)에 API로 접근하여 생성형 AI 애플리케이션을 구축할 수 있게 해주는 서비스입니다.

> 💬 **참고**: Amazon Bedrock은 CLF-C02 시험의 공식 범위에 포함되지 않을 수 있으나, 2025~2026년 현재 AWS의 핵심 전략 서비스이며 실무에서 매우 중요합니다. 시험 대비 시에는 **Amazon Q**에 더 집중하는 것을 권장합니다.

### 직관적 이해

2022년 말 ChatGPT의 등장 이후, **생성형 AI(Generative AI)**는 기술 업계의 가장 큰 화두가 되었습니다. 생성형 AI는 기존 AI가 "분류"하거나 "판별"하는 데 집중했던 것과 달리, **새로운 텍스트, 이미지, 코드, 음악 등을 창작(생성)**할 수 있는 AI입니다.

그런데 생성형 AI를 사용하려면 **파운데이션 모델(Foundation Model)**이라는 초대규모 AI 모델이 필요합니다. 이 모델을 직접 만들려면 수억 달러의 비용과 수천 대의 GPU가 필요합니다. Amazon Bedrock은 이 문제를 해결합니다 — 마치 전기를 직접 발전하지 않고 전력 회사에서 공급받는 것처럼, **이미 만들어진 다양한 파운데이션 모델을 API로 제공**합니다.

### 핵심 개념 설명

> 📝 **파운데이션 모델(Foundation Model, FM)**: 방대한 양의 데이터로 사전 학습(pre-training)된 초대규모 AI 모델입니다. 하나의 모델이 텍스트 생성, 요약, 번역, 코드 작성 등 다양한 작업에 범용적으로 활용될 수 있습니다. GPT, Claude, Llama 등이 대표적인 파운데이션 모델입니다.

> 📝 **생성형 AI(Generative AI)**: 학습한 데이터의 패턴을 바탕으로 **새로운 콘텐츠**(텍스트, 이미지, 코드 등)를 생성하는 AI 기술입니다. 기존 AI가 "이 이메일은 스팸인가?"를 *판별*했다면, 생성형 AI는 "이 주제로 이메일을 *작성*해줘"라는 요청을 수행할 수 있습니다.

> 📝 **Amazon Bedrock**: 여러 AI 기업(Anthropic, Meta, Mistral, Amazon 등)이 제공하는 **파운데이션 모델에 API로 접근**할 수 있는 완전 관리형 서비스입니다. 모델을 직접 호스팅하거나 인프라를 관리할 필요 없이, API 호출만으로 생성형 AI 기능을 사용할 수 있습니다.

Bedrock의 핵심 특징:
- **다양한 모델 선택**: 하나의 서비스에서 여러 기업의 FM에 접근 (Anthropic의 Claude, Meta의 Llama, Amazon의 Titan 등)
- **서버리스**: 인프라 관리 불필요, API 호출만으로 사용
- **커스터마이징**: 자사 데이터로 모델을 미세 조정(Fine-tuning) 가능
- **보안**: 데이터가 모델 학습에 사용되지 않으며, VPC 내에서 안전하게 사용 가능

```
[Amazon Bedrock]
    │
    ├── Anthropic Claude (텍스트 생성, 분석)
    ├── Meta Llama (텍스트 생성)
    ├── Amazon Titan (텍스트, 이미지 생성)
    ├── Stability AI (이미지 생성)
    └── 기타 모델들...
         │
         ▼
    API 호출 → 생성형 AI 기능을 내 앱에 통합
```

#### 11.3의 AI/ML 계층과의 관계

Bedrock은 11.3에서 배운 세 계층 중 어디에 해당할까요? 흥미롭게도, Bedrock은 **계층 1(사전 구축 AI)과 계층 2(ML 플랫폼) 사이**에 위치합니다:
- 계층 1처럼 API 호출만으로 사용할 수 있지만
- 계층 2처럼 자사 데이터로 모델을 커스터마이징할 수도 있습니다

### 상세 예시

> 🔍 **예시 11.5.1: 콘텐츠 플랫폼의 생성형 AI 활용**
>
> **상황**: 뉴스 플랫폼에서 기사를 요약하고, 각 기사에 대한 FAQ를 자동 생성하고 싶습니다.
>
> **풀이/설명**:
> 1. 기사 전문을 Amazon Bedrock의 Claude 모델에 API로 전송
> 2. "이 기사를 3줄로 요약해줘" → 기사 요약 생성
> 3. "이 기사를 읽은 독자가 궁금해할 질문 5개를 만들고 답해줘" → FAQ 자동 생성
> 4. 생성된 콘텐츠를 편집자가 검토한 후 게시
>
> **핵심 포인트**: Bedrock을 사용하면 ML 전문 지식 없이도 생성형 AI 기능을 애플리케이션에 통합할 수 있습니다. 하지만 AI가 생성한 콘텐츠는 사람의 검토가 필요합니다.

> 🔍 **예시 11.5.2: 일상 비유 — 전기 콘센트**
>
> **상황**: 집에서 전기를 사용하는 방법을 생각해 봅시다.
>
> **풀이/설명**:
> - **직접 발전** = 파운데이션 모델을 처음부터 학습 (비용: 수억 원, 전문 인력 필요)
> - **전력 회사 계약** = Amazon Bedrock 사용 (콘센트에 플러그를 꽂듯 API 호출)
> - **다양한 전력 회사 선택** = Bedrock에서 Claude, Llama, Titan 등 여러 모델 선택
> - **전기로 다양한 기기 작동** = API로 텍스트 생성, 요약, 번역, 코드 작성 등 다양한 작업 수행
>
> **핵심 포인트**: Bedrock은 "AI의 전력 회사"처럼 작동하여, 사용자가 모델 인프라를 걱정하지 않고 AI 기능을 사용할 수 있게 합니다.

### 깊이 있는 이해

> 🔬 **Deep Dive: 파운데이션 모델과 전통 ML의 차이**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> **전통적 ML 모델**은 특정 작업 하나를 위해 학습됩니다. 예를 들어, 스팸 분류 모델은 스팸 탐지만 할 수 있고, 감정 분석 모델은 감정 분석만 할 수 있습니다. 새로운 작업이 필요하면 새로운 모델을 처음부터 학습시켜야 합니다.
>
> **파운데이션 모델**은 인터넷의 방대한 텍스트로 "언어 자체"를 이해하도록 학습되었기 때문에, 하나의 모델로 요약, 번역, 질의응답, 코드 작성 등 다양한 작업을 수행할 수 있습니다. 이를 "창발적 능력(Emergent Abilities)"이라 합니다. 모델이 크기가 커지면서 명시적으로 학습하지 않은 능력이 나타나는 현상입니다.
>
> 이 개념은 AWS 서비스 선택에도 영향을 줍니다:
> - **특정 작업**(감정 분석만, 이미지 인식만) → 사전 구축 AI 서비스 (Comprehend, Rekognition)가 효율적
> - **범용 텍스트 생성/이해** → Bedrock의 파운데이션 모델이 적합

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **Bedrock vs SageMaker AI 혼동**: **Bedrock**은 이미 만들어진 파운데이션 모델을 *사용*하는 서비스이고, **SageMaker AI**는 자체 ML 모델을 *만드는* 플랫폼입니다.
> 2. **생성형 AI의 한계**: 생성형 AI는 "환각(Hallucination)"이라 불리는 현상 — 사실이 아닌 정보를 그럴듯하게 생성하는 것 — 이 발생할 수 있습니다. 중요한 의사결정에는 AI 출력을 반드시 검증해야 합니다.
> 3. **시험 대비 팁**: CLF-C02 시험에서 Bedrock보다는 **Amazon Q** (다음 Section에서 다룸)가 출제될 가능성이 더 높습니다. Bedrock은 개념 수준의 이해면 충분합니다.

### Section 요약

- **생성형 AI**는 새로운 콘텐츠를 생성할 수 있는 AI 기술, **파운데이션 모델**은 이를 가능하게 하는 초대규모 모델
- **Amazon Bedrock**은 여러 기업의 파운데이션 모델에 API로 접근할 수 있는 서버리스 서비스
- 모델을 직접 학습시키지 않아도, Bedrock API 호출만으로 생성형 AI 기능을 애플리케이션에 통합 가능
- 사전 구축 AI(계층 1)보다 유연하고, SageMaker AI(계층 2)보다 쉬운 중간 포지션

## 11.6 최종 사용자 컴퓨팅과 비즈니스 서비스

> 💡 **한 줄 요약**: AWS는 가상 데스크톱(WorkSpaces), 애플리케이션 스트리밍(AppStream 2.0), 클라우드 기반 고객 센터(Connect) 등 최종 사용자를 위한 서비스도 제공합니다.

### 직관적 이해

지금까지 배운 AWS 서비스 대부분은 **개발자나 엔지니어가 인프라를 구축**하기 위한 것이었습니다. 하지만 AWS는 기업의 **일반 직원(최종 사용자)**이 직접 사용하는 서비스도 제공합니다. 마치 회사가 직원에게 노트북을 지급하고, 사무실 전화를 배정하는 것처럼 — 이런 것들도 클라우드로 옮길 수 있습니다.

코로나19 팬데믹으로 재택근무가 확산되면서, "물리적 사무실에 있지 않아도 동일한 업무 환경을 제공"해야 하는 필요성이 급증했습니다. AWS의 최종 사용자 컴퓨팅 서비스는 이러한 요구에 응답합니다.

### 핵심 개념 설명

#### Amazon WorkSpaces

> 📝 **Amazon WorkSpaces**: 클라우드에서 실행되는 **가상 데스크톱(VDI, Virtual Desktop Infrastructure)** 서비스입니다. 사용자는 어떤 기기(노트북, 태블릿, 씬 클라이언트)에서든 인터넷을 통해 자신의 Windows 또는 Linux 데스크톱에 접속할 수 있습니다.

가상 데스크톱은 마치 **"클라우드에 있는 내 컴퓨터"**와 같습니다. 실제 컴퓨터는 AWS 데이터센터에서 실행되고, 여러분은 화면만 스트리밍으로 보면서 사용합니다.

WorkSpaces의 장점:
- **어디서나 접속**: 인터넷만 있으면 자신의 데스크톱에 접근
- **보안 강화**: 데이터가 로컬 기기에 저장되지 않고 AWS에 유지
- **빠른 배포**: 새 직원을 위한 데스크톱을 수 분 내에 프로비저닝
- **비용 유연성**: 월 정액 또는 시간 단위 과금 선택 가능

#### Amazon AppStream 2.0

> 📝 **Amazon AppStream 2.0**: 데스크톱 **애플리케이션**을 클라우드에서 스트리밍하여, 사용자가 웹 브라우저를 통해 접속할 수 있게 하는 서비스입니다.

WorkSpaces가 **전체 데스크톱 환경**을 제공하는 반면, AppStream 2.0은 **특정 애플리케이션만** 스트리밍합니다.

예를 들어:
- 3D 설계 소프트웨어(AutoCAD 등)를 설치 없이 브라우저에서 실행
- 특수 교육 소프트웨어를 학생들에게 브라우저로 제공
- 고사양을 요구하는 소프트웨어를 저사양 기기에서 사용

**WorkSpaces vs AppStream 2.0 비교**:

| 구분 | WorkSpaces | AppStream 2.0 |
|------|-----------|---------------|
| 제공 범위 | 전체 데스크톱 환경 | 특정 애플리케이션만 |
| 사용 시나리오 | 원격 근무, 전체 업무 환경 | 특정 소프트웨어 접근 |
| 비유 | 원격으로 내 PC 전체 사용 | Netflix처럼 앱만 스트리밍 |

#### Amazon Connect

> 📝 **Amazon Connect**: 클라우드 기반 **고객 센터(Contact Center)** 서비스입니다. 전화, 채팅, 이메일 등 다양한 채널의 고객 문의를 통합 관리합니다.

전통적인 콜센터는 비싼 전화 장비와 복잡한 인프라가 필요했습니다. Amazon Connect는 이 모든 것을 클라우드에서 제공하여, 웹 브라우저만으로 고객 센터를 운영할 수 있게 합니다. Amazon 자체의 고객 서비스 센터에서 사용하는 기술과 동일한 기술을 기반으로 합니다.

특징:
- **종량제**: 사용한 만큼만 지불 (분당 과금)
- **AI 통합**: Amazon Lex(챗봇), Comprehend(감정 분석), Transcribe(통화 기록) 등과 연동
- **빠른 설정**: 수 분 내에 고객 센터를 구축 가능
- **옴니채널**: 전화, 채팅, 이메일을 하나의 플랫폼에서 관리

### 상세 예시

> 🔍 **예시 11.6.1: 재택근무 환경 구축**
>
> **상황**: 직원 500명의 기업이 갑자기 전 직원 재택근무로 전환해야 합니다. 보안이 중요한 금융 회사라, 직원 개인 컴퓨터에 회사 데이터를 두면 안 됩니다.
>
> **풀이/설명**:
> - **Amazon WorkSpaces**: 각 직원에게 가상 데스크톱을 배정. 직원은 개인 노트북에서 WorkSpaces 클라이언트로 접속하여 업무를 수행합니다. 데이터는 AWS에만 저장되므로 보안 유지.
> - 설계팀에는 **AppStream 2.0**: CAD 소프트웨어만 브라우저로 스트리밍. 고가의 라이선스를 효율적으로 공유.
> - 고객 서비스팀에는 **Amazon Connect**: 자택에서 웹 브라우저만으로 고객 전화를 받고 채팅 상담을 수행.
>
> **핵심 포인트**: 물리적 사무실 장비(데스크톱, 전화 시스템) 없이도 클라우드 서비스로 동일한 업무 환경을 구현할 수 있습니다.

> 🔍 **예시 11.6.2: 일상 비유 — 클라우드 게이밍과의 비교**
>
> **상황**: 클라우드 게이밍(GeForce NOW, Xbox Cloud Gaming)을 사용해 본 적이 있나요?
>
> **풀이/설명**:
> - **클라우드 게이밍**: 게임은 데이터센터의 강력한 서버에서 실행되고, 여러분의 기기에는 화면만 스트리밍됩니다. 저사양 노트북이나 스마트폰에서도 고사양 게임을 즐길 수 있습니다.
> - **WorkSpaces**: 위 개념과 동일합니다. 데스크톱이 AWS에서 실행되고, 여러분의 기기에는 화면만 전송됩니다.
> - **AppStream 2.0**: 특정 게임(앱) 하나만 스트리밍하는 것과 같습니다.
>
> **핵심 포인트**: "실행은 클라우드에서, 화면은 내 기기에서" — 이 원리가 가상 데스크톱과 앱 스트리밍의 핵심입니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **WorkSpaces vs AppStream 2.0 구분**: 시험에서 "특정 애플리케이션만 제공"이라는 키워드가 나오면 **AppStream 2.0**, "전체 데스크톱 환경"이라면 **WorkSpaces**를 선택하세요.
> 2. **Connect의 위치**: Amazon Connect는 컴퓨팅 서비스가 아닌 **비즈니스 애플리케이션** 카테고리입니다. 콜센터/고객 센터와 관련된 문제에서 떠올려야 합니다.

### Section 요약

- **Amazon WorkSpaces**: 클라우드 가상 데스크톱 — 전체 Windows/Linux 데스크톱 환경을 원격으로 제공
- **Amazon AppStream 2.0**: 특정 데스크톱 애플리케이션만 브라우저로 스트리밍
- **Amazon Connect**: 클라우드 기반 고객 센터 — 전화, 채팅, 이메일 통합 관리
- 세 서비스 모두 물리적 인프라 없이 클라우드에서 최종 사용자 환경을 제공하는 것이 핵심

## 11.7 Amazon Q, IoT, 기타 서비스

> 💡 **한 줄 요약**: AWS는 생성형 AI 비서(Amazon Q), 사물인터넷(IoT), GraphQL API 관리(AppSync), 블록체인 등 다양한 전문 서비스를 제공합니다.

### 직관적 이해

AWS의 서비스 범위는 계속 확장되고 있습니다. 이 Section에서는 앞에서 분류하기 어려운, 하지만 CLF-C02 시험과 실무에서 알아야 할 추가 서비스들을 다룹니다. 이 서비스들의 공통점은 각각 **특정 기술 영역의 전문 문제를 해결**한다는 것입니다.

### 핵심 개념 설명

#### Amazon Q

> 📝 **Amazon Q**: AWS가 제공하는 **생성형 AI 기반 비서**입니다. 비즈니스 사용자와 개발자를 위한 두 가지 버전이 있습니다.

Amazon Q는 마치 **"AWS 전문 지식을 가진 AI 동료"**와 같습니다:

**Amazon Q Business**:
- 기업 내부 데이터(문서, 위키, 이메일 등)에 연결하여 직원들의 질문에 답변
- "우리 회사의 연차 사용 정책은?" 같은 사내 질문에 즉시 답변
- 데이터 소스: S3, SharePoint, Salesforce, Slack 등과 연동

**Amazon Q Developer**:
- 개발자를 위한 AI 코딩 어시스턴트
- IDE(개발 환경)에서 코드 추천, 코드 설명, 버그 수정 제안
- AWS 아키텍처 관련 질문에 답변
- 기존 코드의 보안 취약점 스캔

> 💬 **참고**: Amazon Q는 CLF-C02 시험에서 출제될 가능성이 높은 서비스입니다. "AWS의 생성형 AI 비서"라는 핵심 개념을 기억하세요.

#### AWS IoT (Internet of Things)

> 📝 **IoT (Internet of Things, 사물인터넷)**: 센서, 기기, 장비 등 **물리적 "사물(Things)"이 인터넷에 연결**되어 데이터를 수집하고 주고받는 기술입니다. 스마트 홈 기기(스마트 스피커, 온도 조절기), 산업용 센서, 웨어러블 기기 등이 IoT 기기에 해당합니다.

> 📝 **AWS IoT Core**: IoT 기기를 AWS 클라우드에 안전하게 **연결하고, 데이터를 수집하고, 관리**하는 관리형 서비스입니다. 수십억 개의 기기와 수조 개의 메시지를 처리할 수 있습니다.

IoT Core의 역할:
- **기기 연결**: IoT 기기와 클라우드 간의 안전한 통신
- **메시지 브로커**: 기기가 보내는 메시지를 수신하고 라우팅
- **규칙 엔진**: 수신된 데이터에 따라 자동 동작 실행 (예: "온도가 30℃를 초과하면 알림 전송")
- **기기 관리**: 원격으로 기기 소프트웨어 업데이트, 상태 모니터링

> 📝 **AWS IoT Greengrass**: IoT 기기에서 **로컬(기기 자체)**로 데이터 처리, ML 추론, 메시징을 수행할 수 있게 해주는 서비스입니다.

IoT Core가 "기기 → 클라우드" 연결에 집중한다면, Greengrass는 "클라우드 기능을 기기로 확장"합니다. 인터넷 연결이 불안정한 환경(공장, 농장, 선박 등)에서도 기기가 자체적으로 데이터를 처리하고 의사결정을 내릴 수 있게 합니다.

**IoT Core vs IoT Greengrass 비교**:

| 구분 | IoT Core | IoT Greengrass |
|------|----------|----------------|
| 실행 위치 | 클라우드 | 기기(엣지) |
| 핵심 역할 | 기기와 클라우드 연결 | 클라우드 기능을 기기에서 로컬 실행 |
| 인터넷 필요 | 필수 | 오프라인에서도 동작 가능 |
| 비유 | 본사에 보고하는 직원 | 현장에서 독립적으로 판단하는 매니저 |

#### AWS AppSync

> 📝 **AWS AppSync**: **GraphQL API**를 쉽게 구축하고 관리하는 완전 관리형 서비스입니다.

> 📝 **GraphQL**: API를 위한 쿼리 언어로, 클라이언트가 **필요한 데이터만 정확히 요청**할 수 있게 합니다. 기존 REST API가 서버가 정한 형태로 데이터를 반환하는 반면, GraphQL은 클라이언트가 원하는 필드만 지정하여 요청합니다.

> 🔗 **연결**: Chapter 9에서 배운 API Gateway가 REST API를 관리하는 서비스였다면, AppSync는 GraphQL API를 관리하는 서비스입니다.

AppSync의 주요 특징:
- **실시간 데이터 동기화**: WebSocket을 통한 실시간 업데이트
- **오프라인 지원**: 네트워크가 끊어진 상태에서도 데이터를 로컬에 캐시하고, 연결 복구 시 자동 동기화
- **다중 데이터 소스**: DynamoDB, Lambda, RDS, HTTP 엔드포인트 등 여러 소스의 데이터를 하나의 API로 통합

#### Amazon Managed Blockchain

> 📝 **Amazon Managed Blockchain**: **블록체인 네트워크**를 생성하고 관리하는 완전 관리형 서비스입니다.

> 📝 **블록체인(Blockchain)**: 거래 기록(트랜잭션)을 여러 참여자가 공유하는 **분산 원장(Distributed Ledger)** 기술입니다. 한 번 기록된 데이터는 변경하거나 삭제할 수 없어, 데이터의 무결성과 투명성을 보장합니다.

활용 사례: 공급망 추적(제품이 농장에서 매장까지 이동하는 과정 기록), 금융 거래 기록, 자산 소유권 관리

### 상세 예시

> 🔍 **예시 11.7.1: 스마트 팩토리(Smart Factory)**
>
> **상황**: 제조 공장에 수천 개의 센서가 설치되어 있고, 각 센서가 온도, 진동, 습도 데이터를 매초 보냅니다. 이 데이터를 수집하고, 이상 징후가 감지되면 즉시 알림을 보내고 싶습니다.
>
> **풀이/설명**:
> 1. **AWS IoT Core**: 수천 개의 센서를 AWS에 연결하고, 센서 데이터를 수집
> 2. **AWS IoT Greengrass**: 인터넷 연결이 불안정할 수 있는 공장 환경에서, 센서 데이터를 기기에서 로컬로 1차 분석. 급박한 이상(온도 과다 상승 등)은 클라우드 전송 전에 즉시 경고
> 3. 수집된 데이터는 S3 데이터 레이크에 저장 → Athena로 분석 → QuickSight로 대시보드 시각화
> 4. 공급망 이력은 **Managed Blockchain**으로 기록하여 투명성 확보
>
> **핵심 포인트**: IoT Core(클라우드 연결)와 Greengrass(엣지 처리)를 함께 사용하면, 실시간성과 안정성을 모두 확보할 수 있습니다.

> 🔍 **예시 11.7.2: 일상 비유 — 스마트 홈**
>
> **상황**: 여러분의 집에 스마트 기기들이 있다고 상상해 봅시다.
>
> **풀이/설명**:
> | 스마트 홈 상황 | AWS IoT 대응 |
> |---|---|
> | 스마트 온도계가 집 온도를 스마트폰 앱으로 전송 | IoT Core: 기기 → 클라우드 데이터 전송 |
> | 인터넷이 끊겨도 스마트 스피커가 조명을 제어 | IoT Greengrass: 로컬에서 기기 제어 |
> | "온도가 28℃ 넘으면 에어컨 자동 켜기" | IoT Core 규칙 엔진: 조건 기반 자동화 |
>
> **핵심 포인트**: IoT는 물리적 세계와 디지털 세계를 연결하는 기술이며, AWS는 이를 위한 관리형 서비스를 제공합니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
>
> 1. **Amazon Q vs Amazon Kendra 구분**: 둘 다 질문에 답하는 서비스이지만, **Kendra는 문서 검색에 특화**된 ML 기반 검색 서비스이고, **Amazon Q는 생성형 AI 기반 비서**로 더 넓은 범위의 작업(코드 작성, 질의응답, 데이터 분석 등)을 수행합니다.
> 2. **IoT Core vs IoT Greengrass**: "클라우드에 연결"이 키워드면 **IoT Core**, "엣지에서 로컬 실행"이 키워드면 **IoT Greengrass**를 선택하세요.
> 3. **AppSync vs API Gateway**: **REST API** 관리라면 API Gateway, **GraphQL API** 관리라면 AppSync입니다.

 [S3 버킷] ──→ [Athena로 SQL 쿼리] ──→ 쿼리 결과 확인

```

> **Exercise 11.1**: 한 전자상거래 회사는 다음과 같은 요구사항이 있습니다:
> - 매일 수백만 건의 거래 데이터가 CSV로 S3에 저장됨
> - 날짜 형식이 시스템마다 달라 통일이 필요함
> - 마케팅 팀이 매주 "가장 많이 팔린 상품 TOP 10" 보고서를 원함
> - 경영진은 매출 추이를 대시보드로 보고 싶어함
>
> 각 요구사항에 적합한 AWS 서비스를 선택하고, 데이터 흐름을 설명하세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 데이터 정제 → 쿼리 → 시각화 파이프라인 구성
>
> **풀이 과정**:
> 1. **데이터 저장**: 이미 S3에 CSV로 저장되어 있음 (데이터 레이크의 Raw Zone 역할)
> 2. **날짜 형식 통일**: **AWS Glue** ETL 작업으로 날짜 형식을 표준화하고, 정제된 데이터를 S3의 다른 경로(Processed Zone)에 저장
> 3. **주간 보고서 (TOP 10 상품)**: **Amazon Athena**에서 SQL 쿼리로 상위 10개 상품 조회
>    ```sql
>    SELECT product_name, SUM(quantity) as total_sold
>    FROM sales_data
>    WHERE sale_date >= date_add('week', -1, current_date)
>    GROUP BY product_name
>    ORDER BY total_sold DESC
>    LIMIT 10;
>    ```
> 4. **매출 추이 대시보드**: **Amazon QuickSight**에서 Athena를 데이터 소스로 연결하여, 일별/주별/월별 매출 추이 차트를 생성하고 경영진과 공유
>
> **데이터 흐름**:
> ```
> S3 (Raw CSV) → Glue (날짜 형식 통일) → S3 (Processed) → Athena (SQL 쿼리) → QuickSight (대시보드)
> ```
>
> **정답**: Glue(데이터 정제) → Athena(SQL 분석) → QuickSight(시각화)
> **보충 설명**: 이것은 전형적인 데이터 분석 파이프라인입니다. 각 서비스가 특정 역할을 담당하며, 서버리스로 구성되어 인프라 관리가 필요 없습니다.
> </details>

> **(b) Amazon Kendra**: ML 기반 지능형 검색 서비스. 자연어 질문의 의미를 이해하고, 문서에서 정확한 답변을 찾아줍니다. 단순 키워드 매칭이 아닌 의미 기반 검색이 핵심입니다.
>
> **(c) Amazon Lex**: 챗봇/음성 봇 구축 서비스. 사용자의 의도(Intent)를 파악하고, 대화 흐름을 관리하여 자동 응답 시스템을 구축합니다.
>
> **(d) Amazon Transcribe**: 음성을 텍스트로 변환(STT)하는 서비스. 녹음 파일을 업로드하면 텍스트로 변환해줍니다. 한국어도 지원합니다.
>
> **보충 설명**: 네 가지 모두 사전 구축 AI 서비스(계층 1)로, ML 전문 지식 없이 API 호출만으로 사용할 수 있습니다.
> </details>

> **Exercise 11.3**: 한 제조 회사가 전국 10개 공장에 온도·습도 센서를 설치했습니다. 일부 공장은 인터넷 연결이 불안정합니다. 다음 요구사항을 충족하는 아키텍처를 설계하세요:
>
> - 모든 센서 데이터를 중앙에서 모니터링
> - 인터넷이 끊어져도 현장에서 온도 이상 감지 시 즉시 경보
> - 수집된 데이터로 월간 분석 보고서 생성
>
> **힌트**: IoT 서비스 2개와 분석 서비스 2개 이상을 조합하세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: IoT 데이터 수집 → 엣지 처리 → 클라우드 분석 파이프라인 설계
>
> **풀이 과정**:
>
> 1. **센서 → AWS IoT Core**: 센서 데이터를 AWS IoT Core를 통해 클라우드에 전송. 중앙 모니터링 요구사항 충족.
>
> 2. **AWS IoT Greengrass** (각 공장에 배치): 인터넷이 끊어진 상태에서도 로컬에서 온도 데이터를 분석하고, 임계값 초과 시 즉시 현장 경보를 울림. 인터넷 복구 시 데이터를 클라우드로 전송.
>
> 3. **S3 데이터 레이크**: IoT Core를 통해 수신된 센서 데이터를 S3에 저장 (데이터 레이크 Raw Zone).
>
> 4. **Amazon Athena**: S3에 저장된 월간 센서 데이터를 SQL로 쿼리하여 분석 (공장별 평균 온도, 이상 발생 횟수 등).
>
> 5. **Amazon QuickSight**: Athena 분석 결과를 월간 보고서 대시보드로 시각화.
>
> **아키텍처**:
> ```
> [공장 센서] → [IoT Greengrass (로컬 경보)] → [IoT Core] → [S3] → [Athena] → [QuickSight]
> ```
>
> **정답**: IoT Greengrass(엣지 처리) + IoT Core(클라우드 연결) + S3(저장) + Athena(분석) + QuickSight(시각화)
> **보충 설명**: Greengrass가 오프라인 경보를 처리하므로, 인터넷 연결 불안정 상황에서도 안전이 보장됩니다. 이는 IoT Core만으로는 해결할 수 없는 요구사항입니다.
> </details>

### D3. Problem (심화 문제) — 깊이 있는 사고와 창의적 문제 해결

> **Problem 11.1**: 당신은 글로벌 뉴스 미디어 회사의 CTO입니다. 다음 프로젝트를 기획하고 있습니다:
>
> *"전 세계 10개 언어로 서비스되는 뉴스 플랫폼을 구축한다. 기자가 한 언어로 기사를 작성하면 나머지 9개 언어로 자동 번역되고, 각 기사에 대한 독자 댓글의 감정을 분석하여 '여론 온도계'를 표시한다. 또한 독자가 '이 사건의 배경은?'처럼 자연어로 질문하면 관련 과거 기사에서 답변을 생성한다."*
>
> (a) 이 프로젝트에 필요한 AWS AI/ML 서비스를 모두 나열하고, 각 서비스의 역할을 설명하세요.
> (b) 생성형 AI(Bedrock)를 활용하는 것과 개별 AI 서비스(Comprehend, Translate 등)를 조합하는 것의 장단점을 비교하세요.
> (c) 이 시스템의 데이터 흐름을 아키텍처 다이어그램으로 표현하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 여러 AI 서비스를 파이프라인으로 조합하여 복잡한 비즈니스 요구사항을 충족하는 아키텍처 설계
>
> **풀이 전략**:
>
> **(a) 필요한 AWS 서비스**:
> - **Amazon Translate**: 기사를 10개 언어로 자동 번역
> - **Amazon Comprehend**: 댓글의 감정 분석 (긍정/부정 → '여론 온도계')
> - **Amazon Kendra 또는 Amazon Bedrock**: 자연어 질문에 대해 과거 기사에서 답변 검색/생성
> - **Amazon Polly** (선택): 기사를 음성으로 읽어주는 기능 추가 가능
> - **Amazon S3**: 기사·번역 결과·댓글 데이터 저장 (데이터 레이크)
> - **Amazon Athena**: 독자 행동·여론 데이터 분석
>
> **(b) Bedrock vs 개별 AI 서비스 비교**:
>
> | 관점 | Bedrock (생성형 AI) | 개별 AI 서비스 조합 |
> |------|-------|------|
> | **유연성** | 하나의 모델로 번역, 요약, 질의응답 모두 가능 | 각 서비스가 특정 작업에 최적화 |
> | **정확도** | 일반적 성능은 좋으나, 특정 작업(감정 분석 등)에서는 전문 서비스보다 낮을 수 있음 | 각 서비스가 해당 작업에 특화되어 정확도가 높음 |
> | **비용** | 토큰 기반 과금으로 대량 처리 시 비용 증가 | 각 서비스별 최적화된 과금 모델 |
> | **환각 위험** | 사실이 아닌 내용을 생성할 수 있음 | 분석 기반이므로 환각 위험 낮음 |
> | **개발 복잡도** | 단일 API로 다양한 기능 구현 가능 (낮음) | 여러 서비스를 연동해야 함 (높음) |
>
> **추천**: 번역(Translate)과 감정 분석(Comprehend)은 전문 서비스가 더 적합하고, 과거 기사 기반 질의응답은 Bedrock이 더 자연스러운 답변을 생성할 수 있습니다. 두 접근을 혼합하는 것이 최적입니다.
>
> **(c) 아키텍처**:
> ```
> [기자: 기사 작성]
>       │
>       ▼
> [Amazon Translate] → 10개 언어로 번역 → [S3 저장]
>       │
>       ▼
> [독자 댓글] → [Amazon Comprehend] → 감정 분석 → [여론 온도계 표시]
>       │
>       ▼
> [독자 질문] → [Amazon Bedrock/Kendra] → 과거 기사 검색 → [답변 생성]
>                       ↑
>              [S3: 과거 기사 데이터 레이크]
> ```
>
> **확장 생각**: 실제 서비스에서는 번역 품질 검증(사람 편집자의 검수), 감정 분석의 문화적 맥락 차이(같은 표현이 문화에 따라 다르게 해석), Bedrock 답변의 사실 검증(Fact-checking) 등의 추가 고려사항이 있습니다.
> </details>

> **Problem 11.2**: AWS AI/ML 서비스 계층(사전 구축 AI, SageMaker AI, 프레임워크/인프라)의 선택 기준에 대해 깊이 생각해 봅시다.
>
> 한 헬스케어 스타트업이 X-ray 이미지를 분석하여 폐 질환을 자동 진단하는 시스템을 개발하려 합니다. 이 회사에는 의료 데이터 과학자 2명과 풀스택 개발자 3명이 있습니다.
>
> (a) 세 계층 중 어떤 접근 방식이 적합하며 그 이유는?
> (b) Amazon Rekognition을 사용할 수 있을까? 사용할 수 있다면/없다면 그 이유는?
> (c) 이 시스템이 실제 의료 현장에서 사용되려면, 기술 외에 어떤 고려사항이 필요할까?
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: AI/ML 계층 선택은 기술적 요구사항뿐 아니라, 팀 역량, 도메인 특수성, 규제 환경을 종합적으로 고려해야 합니다.
>
> **풀이 전략**:
>
> **(a) 적합한 계층: 계층 2 (Amazon SageMaker AI)**
>
> - **계층 1 (사전 구축 AI)이 부적합한 이유**: Rekognition은 일반 객체/얼굴 인식에 최적화되어 있지, 의료 X-ray 판독이라는 고도로 전문화된 작업에는 정확도가 부족합니다.
> - **계층 3 (프레임워크/인프라)이 과도한 이유**: 데이터 과학자가 2명뿐인 소규모 팀에서 GPU 클러스터 관리, 모델 배포 인프라 구축까지 담당하면 핵심 업무(모델 개발)에 집중하기 어렵습니다.
> - **계층 2 (SageMaker AI)가 적합한 이유**: 의료 데이터 과학자가 자체 X-ray 데이터로 맞춤 모델을 학습시키되, 인프라 관리는 SageMaker에 맡길 수 있습니다. SageMaker의 내장 알고리즘, 자동 튜닝, 관리형 배포가 소규모 팀의 생산성을 높여줍니다.
>
> **(b) Rekognition 사용 가능성**: 사용할 수는 있으나, **의료 진단에는 부적합**합니다.
> - Rekognition은 범용 이미지 인식 모델로, X-ray 이미지의 미세한 폐 질환 징후를 정확히 판별하도록 학습되지 않았습니다.
> - Rekognition Custom Labels 기능으로 자체 데이터를 학습시킬 수 있으나, 의료 영상은 매우 높은 정확도가 요구되므로 SageMaker에서 전문 의료 이미지 모델을 학습시키는 것이 더 적합합니다.
> - 의료 AI는 "99% 정확"해도 나머지 1%가 생명에 관계될 수 있으므로, 범용 서비스보다는 전문 모델이 필요합니다.
>
> **(c) 기술 외 고려사항**:
> - **의료 규제 준수**: 의료 기기 인허가(FDA, KFDA), HIPAA(의료정보 보호법) 등
> - **환자 데이터 보안**: 의료 데이터의 암호화, 접근 제어, 감사 로그
> - **모델 설명 가능성**: 의사가 "AI가 왜 이런 진단을 내렸는지" 이해할 수 있어야 함 (Explainable AI)
> - **인간 검증**: AI의 진단은 보조 수단이며, 최종 판단은 의료 전문가가 해야 함
> - **편향(Bias) 검증**: 특정 인종, 연령대에 대해 모델이 편향되지 않았는지 검증
>
> **확장 생각**: 이 문제는 "어떤 서비스를 사용할까?"를 넘어 "AI를 책임감 있게 사용하려면 무엇이 필요한가?"를 생각하게 합니다. AWS는 SageMaker AI에 모델 모니터링, 편향 감지, 설명 가능성(SageMaker Clarify) 등의 기능을 제공하여 이러한 고려사항을 지원합니다. 이 내용은 이후 보안 및 거버넌스 Chapter에서 더 자세히 다뤄집니다.
> </details>
