## 13.1 IAM 개요 — 글로벌 서비스, 인증과 인가의 차이

> 💡 **한 줄 요약**: IAM은 AWS에서 "누구인지 확인하고(인증), 무엇을 할 수 있는지 결정하는(인가)" 중앙 관리 시스템입니다.

### 직관적 이해 (Intuitive Understanding)

건물에 들어가는 상황을 떠올려 봅시다.

먼저, 건물 입구에서 신분증을 보여주고 "나는 이 건물에 출입할 수 있는 사람이다"라는 것을 증명합니다. 이것이 **인증(Authentication)** — "당신이 누구인지 확인하는 것"입니다.

신분증 확인을 통과했다고 해서 건물 내 모든 곳에 갈 수 있는 것은 아닙니다. 일반 직원은 사무실까지만 갈 수 있고, 서버실은 IT 담당자만, 금고는 재무 담당자만 들어갈 수 있습니다. 이것이 **인가(Authorization)** — "당신이 무엇을 할 수 있는지 결정하는 것"입니다.

AWS에서 이 두 가지를 모두 담당하는 서비스가 바로 **IAM(Identity and Access Management)** 입니다.

> 📝 **IAM (Identity and Access Management)**: AWS 리소스에 대한 접근을 안전하게 제어하는 서비스. "누가(Identity) 무엇을 할 수 있는가(Access)"를 관리합니다. AWS 계정을 생성하면 자동으로 활성화되며, 추가 비용 없이 사용할 수 있습니다.

**왜 IAM이 필요할까요?**

Chapter 4에서 AWS 계정을 처음 만들었을 때, 우리는 **루트 사용자(Root User)** 로 로그인했습니다. 루트 사용자는 그 계정의 "건물 주인"과 같아서, 모든 방에 들어갈 수 있고 모든 것을 할 수 있습니다. 하지만 현실에서 건물 주인이 매일 모든 업무를 직접 하지 않듯이, AWS에서도 루트 사용자로 일상 작업을 하는 것은 위험합니다. 실수로 중요한 리소스를 삭제하거나, 루트 자격 증명이 유출되면 전체 계정이 위험해집니다.

> 🔗 **연결**: Chapter 12의 **최소 권한 원칙(Principle of Least Privilege)** 을 떠올려 봅시다. "각 사용자에게는 업무 수행에 필요한 최소한의 권한만 부여해야 한다"는 원칙이었습니다. IAM은 바로 이 원칙을 AWS에서 실현하는 도구입니다.

### 핵심 개념 설명 (Core Explanation)

#### 인증(Authentication)과 인가(Authorization)의 차이

이 두 개념은 IAM을 이해하는 출발점입니다.

| 구분 | 인증 (Authentication) | 인가 (Authorization) |
|------|----------------------|---------------------|
| 질문 | "당신은 누구입니까?" | "당신은 무엇을 할 수 있습니까?" |
| 비유 | 건물 입구에서 신분증 확인 | 건물 내 각 방의 출입 권한 확인 |
| AWS에서 | 사용자 이름 + 비밀번호, 또는 접근 키로 로그인 | IAM 정책이 허용하는 작업만 수행 가능 |
| 순서 | 먼저 발생 | 인증 후에 발생 |

#### IAM은 글로벌 서비스

대부분의 AWS 서비스는 특정 **리전(Region)** 에 종속됩니다. 예를 들어, 서울 리전에서 만든 EC2 인스턴스는 서울 리전에만 존재합니다. 하지만 **IAM은 글로벌 서비스**입니다. IAM에서 사용자를 만들면, 그 사용자는 모든 리전에서 동일하게 인식됩니다. 건물의 신분증 시스템이 건물 전체에 하나만 있는 것과 같습니다.

> 📝 **글로벌 서비스 (Global Service)**: 특정 리전에 종속되지 않고 AWS 전체에 걸쳐 동작하는 서비스. IAM, Route 53, CloudFront 등이 대표적입니다. AWS 콘솔에서 리전 선택 드롭다운에 "글로벌"로 표시됩니다.

#### IAM의 네 가지 핵심 구성 요소

IAM은 크게 네 가지 구성 요소로 이루어져 있습니다:

1. **IAM 사용자 (Users)**: AWS에 접근하는 개별 사람 또는 애플리케이션
2. **IAM 그룹 (Groups)**: 사용자들의 모음 — 그룹 단위로 권한을 관리
3. **IAM 역할 (Roles)**: AWS 서비스나 외부 사용자가 임시로 사용하는 권한 집합
4. **IAM 정책 (Policies)**: "무엇을 허용하고 거부할지"를 정의한 문서 (JSON 형식)

이 네 가지는 이후 Section에서 하나씩 깊이 있게 다룹니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.1.1: 스타트업의 AWS 접근 관리**
>
> **상황**: 5명이 일하는 스타트업이 AWS를 사용합니다. 대표 1명, 개발자 2명, 디자이너 1명, 인턴 1명이 있습니다.
>
> **풀이/설명**:
> - 대표가 처음 AWS 계정을 만들면 **루트 사용자**가 생성됩니다
> - 대표는 루트 사용자 대신 자신을 위한 **IAM 사용자**를 만들고, 관리자 권한을 부여합니다
> - 개발자 2명을 위한 IAM 사용자를 만들고, "개발팀" **그룹**에 넣습니다
> - "개발팀" 그룹에 EC2, S3, RDS 접근 **정책**을 부여합니다 → 개발자 2명 모두 같은 권한을 갖습니다
> - 디자이너에게는 S3(이미지 저장)만 접근할 수 있는 정책을 부여합니다
> - 인턴에게는 읽기 전용(Read-Only) 정책을 부여합니다
>
> **핵심 포인트**: 각 사람의 역할에 맞는 최소한의 권한만 부여하여 보안을 유지합니다.

> 🔍 **예시 13.1.2: 인증과 인가가 분리되는 실제 상황**
>
> **상황**: 개발자 김 씨가 AWS 콘솔에 로그인하여 EC2 인스턴스를 생성하려 합니다.
>
> **풀이/설명**:
> 1. 김 씨가 AWS 콘솔에서 사용자 이름과 비밀번호를 입력합니다 → **인증 단계** (성공하면 콘솔에 진입)
> 2. 김 씨가 EC2 서비스로 이동하여 "인스턴스 시작"을 클릭합니다
> 3. AWS가 김 씨에게 연결된 IAM 정책을 확인합니다 → **인가 단계**
> 4. 정책에 EC2 인스턴스 생성 권한이 있으면 → 인스턴스 생성 성공
> 5. 정책에 해당 권한이 없으면 → "You are not authorized to perform this operation" 오류 발생
>
> **핵심 포인트**: 로그인(인증)에 성공했다고 해서 모든 작업이 가능한 것은 아닙니다. 각 작업마다 인가 확인이 별도로 이루어집니다.

> ⚠️ **주의**
>
> - **인증과 인가를 혼동하지 마세요.** "로그인할 수 있다 = 모든 것을 할 수 있다"가 아닙니다. AWS에서 가장 흔한 오류 중 하나가 "Access Denied"인데, 이는 인증은 성공했지만 인가가 거부된 경우입니다.
> - **루트 사용자로 일상 작업을 하지 마세요.** 루트 사용자는 계정의 모든 권한을 가지고 있어, 실수나 보안 침해 시 피해가 극대화됩니다. IAM 사용자를 만들어 사용하는 것이 AWS의 첫 번째 보안 모범 사례입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: IAM 개요에서 자주 혼동되는 사항들
>
> - **인증(Authentication)과 인가(Authorization)를 혼동하지 마세요**: 로그인에 성공(인증)했다고 모든 작업을 할 수 있는 것이 아닙니다. 각 작업에 대한 권한 확인(인가)이 별도로 이루어집니다. "Access Denied" 오류는 인증은 성공했지만 인가가 거부된 경우입니다.
> - **IAM은 리전 서비스가 아니라 글로벌 서비스입니다**: IAM에서 생성한 사용자, 그룹, 역할, 정책은 모든 AWS 리전에서 사용할 수 있습니다. CCP 시험에서 "IAM은 리전별 서비스이다"라는 함정 선지에 주의하세요.
> - **IAM은 무료 서비스입니다**: IAM 사용자, 그룹, 역할, 정책 생성에는 비용이 발생하지 않습니다. IAM 자체에 대한 요금은 없으며, IAM으로 접근하는 AWS 리소스에 대해서만 비용이 발생합니다.
> - **루트 사용자와 IAM 관리자 사용자는 다릅니다**: AdministratorAccess 정책이 연결된 IAM 사용자도 루트 사용자만 할 수 있는 일부 작업(계정 삭제, 결제 정보 변경 등)은 수행할 수 없습니다.

### Section 요약 (Section Summary)

- **IAM**은 AWS 리소스에 대한 접근을 제어하는 **무료 글로벌 서비스**입니다
- **인증(Authentication)** 은 "누구인지 확인", **인가(Authorization)** 는 "무엇을 할 수 있는지 결정"입니다
- IAM의 네 가지 핵심 구성 요소: **사용자, 그룹, 역할, 정책**
- 루트 사용자 대신 IAM 사용자를 만들어 사용하는 것이 보안의 첫걸음입니다

## 13.2 IAM 사용자와 그룹 — 사용자 생성, 그룹 기반 권한 관리

> 💡 **한 줄 요약**: IAM 사용자는 AWS에 접근하는 개별 신원이고, 그룹은 같은 권한이 필요한 사용자들을 묶어 효율적으로 관리하는 단위입니다.

### 직관적 이해 (Intuitive Understanding)

회사에 새 직원이 입사하면 어떤 일이 벌어질까요?

1. 먼저 **사원증**을 발급합니다 — 이 사람이 누구인지 식별하기 위해서
2. 그 직원을 **부서**에 배치합니다 — "개발팀", "마케팅팀" 등
3. 부서에 따라 접근할 수 있는 시스템이 정해집니다 — 개발팀은 코드 저장소, 마케팅팀은 광고 플랫폼

AWS IAM에서도 똑같은 흐름입니다:
- 사원증 = **IAM 사용자**
- 부서 = **IAM 그룹**
- 부서별 시스템 접근 권한 = 그룹에 연결된 **IAM 정책**

**왜 그룹이 필요할까요?** 직원이 5명이면 한 명씩 권한을 설정해도 괜찮겠지만, 직원이 500명이라면? 개발자가 200명이라면? 200명 한 명 한 명에게 같은 권한을 반복 설정하는 것은 비효율적이고 실수가 발생하기 쉽습니다. 그룹에 권한을 한 번 설정하면 그룹에 속한 모든 사용자가 같은 권한을 자동으로 받습니다.

### 핵심 개념 설명 (Core Explanation)

#### IAM 사용자 (IAM User)

> 📝 **IAM 사용자 (IAM User)**: AWS 서비스와 리소스에 접근하는 개체(사람 또는 애플리케이션)를 나타냅니다. 고유한 이름과 보안 자격 증명(비밀번호 또는 접근 키)을 가집니다.

IAM 사용자가 AWS에 접근하는 방법은 두 가지입니다:

| 접근 방식 | 자격 증명 | 용도 |
|-----------|-----------|------|
| **AWS 콘솔 (웹)** | 사용자 이름 + 비밀번호 | 웹 브라우저에서 AWS 서비스를 시각적으로 관리 |
| **프로그래밍 방식 (CLI/SDK)** | 접근 키 ID + 비밀 접근 키 | 명령줄이나 코드에서 AWS API를 호출 |

> 📝 **접근 키 (Access Key)**: IAM 사용자가 AWS CLI나 SDK를 통해 프로그래밍 방식으로 AWS에 접근할 때 사용하는 자격 증명. **접근 키 ID (Access Key ID)** 와 **비밀 접근 키 (Secret Access Key)** 의 쌍으로 구성됩니다. 비밀 접근 키는 생성 시 한 번만 확인할 수 있으므로 안전하게 보관해야 합니다.

한 IAM 사용자에게 두 가지 접근 방식을 모두 부여할 수도, 하나만 부여할 수도 있습니다. 예를 들어, 자동화 스크립트용 사용자는 콘솔 비밀번호 없이 접근 키만 가질 수 있습니다.

#### IAM 그룹 (IAM Group)

> 📝 **IAM 그룹 (IAM Group)**: IAM 사용자들의 모음. 그룹에 정책을 연결하면, 그룹에 속한 모든 사용자가 해당 정책의 권한을 자동으로 받습니다.

IAM 그룹의 핵심 규칙들:

- 그룹에는 **사용자만** 포함할 수 있습니다 (그룹 안에 다른 그룹을 넣을 수 없음)
- 한 사용자는 **여러 그룹**에 동시에 속할 수 있습니다 (최대 10개)
- 사용자가 반드시 어떤 그룹에 속해야 하는 것은 아닙니다 (그룹에 속하지 않은 사용자도 가능)
- 그룹 자체로는 AWS에 "로그인"할 수 없습니다 — 그룹은 권한 관리 도구일 뿐입니다

#### ARN (Amazon Resource Name)

IAM을 사용하다 보면 반드시 만나게 되는 개념이 **ARN**입니다.

> 📝 **ARN (Amazon Resource Name)**: AWS의 모든 리소스를 고유하게 식별하는 표준 형식의 이름. IAM 정책에서 "어떤 리소스에 대한 권한인지"를 지정할 때 사용합니다.

ARN의 기본 형식:

```
arn:aws:서비스:리전:계정ID:리소스유형/리소스이름
```

예시:
- IAM 사용자: `arn:aws:iam::123456789012:user/johndoe`
- S3 버킷: `arn:aws:s3:::my-bucket`
- EC2 인스턴스: `arn:aws:ec2:ap-northeast-2:123456789012:instance/i-1234567890abcdef0`

IAM은 글로벌 서비스이므로 ARN에서 리전 부분이 비어 있는 것에 주목하세요.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.2.1: 팀별 그룹 구성**
>
> **상황**: 한 회사에 개발팀 3명, 운영팀 2명, 그리고 DevOps 담당자 1명(개발과 운영 모두 수행)이 있습니다.
>
> **풀이/설명**:
> - "Developers" 그룹 생성 → 개발 관련 권한 정책 연결 (EC2, S3, CodeCommit 등)
> - "Operations" 그룹 생성 → 운영 관련 권한 정책 연결 (CloudWatch, Systems Manager 등)
> - 개발팀 3명 → "Developers" 그룹에 추가
> - 운영팀 2명 → "Operations" 그룹에 추가
> - DevOps 담당자 → "Developers" **그리고** "Operations" 두 그룹 모두에 추가
> - DevOps 담당자는 두 그룹의 권한을 합산하여 가집니다
>
> **핵심 포인트**: 한 사용자가 여러 그룹에 속하면, 모든 그룹의 권한을 합산하여 받습니다. DevOps처럼 여러 역할을 겸하는 경우에 유용합니다.

> 🔍 **예시 13.2.2: 신입 직원 온보딩과 퇴사 처리**
>
> **상황**: 개발팀에 신입 직원이 입사하고, 한 달 뒤 다른 개발자가 퇴사합니다.
>
> **풀이/설명**:
> - **입사 시**:
>   1. 신입 직원을 위한 IAM 사용자 생성 (예: `park-dev`)
>   2. 콘솔 로그인용 임시 비밀번호 설정 (첫 로그인 시 변경 필수 설정)
>   3. "Developers" 그룹에 추가 → 즉시 개발팀과 동일한 권한 획득
>
> - **퇴사 시**:
>   1. 해당 IAM 사용자의 접근 키 비활성화 (즉시 API 접근 차단)
>   2. 콘솔 비밀번호 삭제
>   3. 모든 그룹에서 제거
>   4. IAM 사용자 삭제
>
> **핵심 포인트**: 그룹을 활용하면 입사자에게 "어떤 권한을 줘야 하지?" 고민할 필요 없이, 해당 부서 그룹에 추가하기만 하면 됩니다. 퇴사 시에는 접근 키 비활성화를 가장 먼저 수행하여 보안을 확보합니다.

> ⚠️ **주의**
>
> - **IAM 사용자와 루트 사용자를 혼동하지 마세요.** 루트 사용자는 AWS 계정 자체의 소유자이며, IAM 사용자는 루트 사용자가 만든 개별 접근 계정입니다. 루트 사용자는 IAM으로 권한을 제한할 수 없습니다.
> - **그룹에 그룹을 중첩할 수 없습니다.** 다른 시스템(예: Active Directory)에서는 그룹 안에 하위 그룹을 넣는 것이 가능하지만, IAM 그룹은 이를 지원하지 않습니다.
> - **접근 키는 비밀번호처럼 관리하세요.** 접근 키를 코드에 하드코딩하거나 Git에 커밋하면 보안 사고로 이어집니다. 실제로 GitHub에 접근 키가 노출되어 수천만 원의 과금이 발생한 사례가 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: IAM 사용자와 그룹에서 자주 혼동되는 사항들
>
> - **IAM 사용자를 사람에게만 사용하는 것이 아닙니다**: IAM 사용자는 사람뿐 아니라 애플리케이션(프로그래밍 방식 접근)에도 사용됩니다. 다만 애플리케이션에는 IAM 역할을 사용하는 것이 더 권장됩니다.
> - **IAM 그룹에 그룹을 중첩할 수 없습니다**: Active Directory 등에서는 그룹 안에 하위 그룹을 넣을 수 있지만, IAM 그룹은 중첩(nesting)을 지원하지 않습니다. CCP 시험에서 이 제한사항이 출제될 수 있습니다.
> - **사용자가 여러 그룹에 속하면 모든 그룹의 권한을 합산합니다**: 개발자 그룹과 테스터 그룹에 동시에 속하면 두 그룹의 권한을 모두 받습니다. 다만 어디선가 명시적 Deny가 있으면 그것이 우선합니다.
> - **접근 키를 코드에 직접 넣지 마세요**: 접근 키가 GitHub 등에 노출되면 몇 분 내에 악용될 수 있습니다. EC2에서 AWS 서비스에 접근할 때는 접근 키 대신 IAM 역할을 사용하세요.

### Section 요약 (Section Summary)

- **IAM 사용자**는 개별 사람이나 애플리케이션의 AWS 접근 신원입니다
- **IAM 그룹**은 같은 권한이 필요한 사용자들의 모음으로, 권한을 효율적으로 관리합니다
- 사용자는 **콘솔(비밀번호)** 과 **프로그래밍 방식(접근 키)** 으로 AWS에 접근합니다
- **ARN**은 AWS 리소스를 고유하게 식별하는 표준 이름 형식입니다
- 한 사용자가 여러 그룹에 속하면, 모든 그룹의 권한을 합산하여 받습니다

## 13.3 IAM 정책 심화 — JSON 정책 구조, Effect/Action/Resource, 정책 평가 로직

> 💡 **한 줄 요약**: IAM 정책은 "누가, 어떤 서비스의, 어떤 작업을, 어떤 리소스에 대해, 허용 또는 거부하는지"를 JSON 형식으로 정의한 권한 문서입니다.

### 직관적 이해 (Intuitive Understanding)

여러분이 놀이공원의 관리자라고 상상해 봅시다. 방문객들에게 어떤 놀이기구를 탈 수 있는지 "티켓"을 발급해야 합니다. 이 티켓에는 다음이 적혀 있습니다:

- **허용/거부**: 탈 수 있다 / 탈 수 없다
- **대상 놀이기구**: 롤러코스터, 바이킹, 회전목마 등
- **조건**: 키 130cm 이상만, 오후 3시 이후만 등

IAM 정책이 바로 이 "티켓"입니다. 다만, 사람이 읽는 티켓이 아니라 AWS가 읽는 **JSON 형식의 문서**입니다.

**왜 JSON일까요?** JSON은 컴퓨터가 구조화된 데이터를 쉽게 읽을 수 있는 형식입니다. AWS가 "이 사용자가 이 작업을 할 수 있는가?"를 판단할 때, 매번 정책 문서를 읽고 해석해야 하는데, JSON 형식이면 빠르고 정확하게 처리할 수 있습니다.

### 핵심 개념 설명 (Core Explanation)

#### 정책의 종류

IAM 정책은 크게 세 가지로 분류됩니다:

| 종류 | 설명 | 사용 시기 |
|------|------|-----------|
| **AWS 관리형 정책 (AWS Managed Policy)** | AWS가 미리 만들어 제공하는 정책. 이름이 `Amazon` 또는 `AWS`로 시작 | 일반적인 권한이 필요할 때 (예: S3 읽기 전용) |
| **고객 관리형 정책 (Customer Managed Policy)** | 사용자가 직접 만드는 정책 | 조직 맞춤형 세밀한 권한이 필요할 때 |
| **인라인 정책 (Inline Policy)** | 특정 사용자, 그룹, 역할에 직접 내장된 정책 | 특정 개체에만 적용되는 일회성 권한 |

> 📝 **관리형 정책 (Managed Policy)**: 독립적으로 존재하며 여러 사용자, 그룹, 역할에 재사용하여 연결할 수 있는 정책. AWS가 만든 것(AWS 관리형)과 고객이 만든 것(고객 관리형)이 있습니다.

> 📝 **인라인 정책 (Inline Policy)**: 특정 IAM 사용자, 그룹, 역할에 직접 포함된 정책. 해당 개체가 삭제되면 정책도 함께 삭제됩니다. 재사용이 불가능하여, AWS는 대부분의 경우 관리형 정책을 권장합니다.

**어떤 정책을 사용해야 할까요?** 대부분의 경우 **AWS 관리형 정책**으로 시작합니다. AWS가 이미 잘 만들어 둔 수백 개의 정책이 있기 때문입니다. 더 세밀한 제어가 필요하면 **고객 관리형 정책**을 만듭니다. **인라인 정책**은 특수한 경우에만 사용합니다.

#### JSON 정책 구조

IAM 정책의 핵심은 JSON 구조입니다. 가장 기본적인 형태를 살펴봅시다:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

이 정책을 한 줄씩 해석하면:

| 요소 | 값 | 의미 |
|------|-----|------|
| **Version** | `"2012-10-17"` | 정책 문법의 버전 (항상 이 값 사용) |
| **Statement** | `[...]` | 권한 규칙들의 배열 (하나 이상의 규칙 가능) |
| **Effect** | `"Allow"` 또는 `"Deny"` | 이 규칙이 허용인지 거부인지 |
| **Action** | `"s3:GetObject"` | 허용/거부할 AWS 작업 (`서비스:작업` 형식) |
| **Resource** | ARN | 이 규칙이 적용되는 대상 리소스 |

> 📝 **Statement**: IAM 정책 내의 개별 권한 규칙. 하나의 정책에 여러 개의 Statement를 포함할 수 있으며, 각 Statement는 Effect, Action, Resource를 반드시 포함합니다.

#### Action의 형식

Action은 항상 `서비스명:작업명` 형식입니다:

- `s3:GetObject` — S3 버킷에서 객체 가져오기
- `ec2:StartInstances` — EC2 인스턴스 시작
- `iam:CreateUser` — IAM 사용자 생성

**와일드카드(`*`)** 를 사용하여 여러 작업을 한 번에 지정할 수 있습니다:

- `s3:*` — S3의 모든 작업
- `ec2:Describe*` — EC2의 모든 Describe 작업 (조회 작업들)
- `*` — 모든 서비스의 모든 작업 (매우 위험! 관리자에게만 사용)

#### 여러 Statement가 있는 정책

실제 정책은 대부분 여러 Statement를 가집니다:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Read",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    },
    {
      "Sid": "DenyS3Delete",
      "Effect": "Deny",
      "Action": "s3:DeleteObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

> 📝 **Sid (Statement ID)**: Statement에 붙이는 선택적 식별자. 정책을 읽기 쉽게 만들기 위한 이름표 역할을 합니다. 작성하지 않아도 동작에는 문제가 없습니다.

이 정책의 의미: "my-bucket의 객체를 읽고 목록을 조회할 수 있지만, 삭제는 절대 불가"

#### 정책 평가 로직 — Deny가 항상 이긴다

사용자에게 여러 정책이 연결되어 있을 때, AWS는 다음 순서로 권한을 평가합니다:

1. 기본값: **모든 것이 거부(Deny)** 됩니다 (명시적으로 허용하지 않으면 불가)
2. 연결된 모든 정책의 Statement를 확인합니다
3. **명시적 Deny가 하나라도 있으면 → 최종 거부** (다른 정책에서 Allow가 있어도 Deny가 이김)
4. 명시적 Deny가 없고, Allow가 있으면 → 최종 허용
5. Allow도 없으면 → 기본 거부 (묵시적 거부)

이것을 간단히 기억하세요: **"Deny는 항상 Allow를 이긴다" (Explicit Deny Always Wins)**

```
[요청 발생]
    │
    ▼
명시적 Deny가 있는가? ──Yes──▶ ❌ 거부
    │ No
    ▼
명시적 Allow가 있는가? ──Yes──▶ ✅ 허용
    │ No
    ▼
❌ 묵시적 거부 (기본값)
```

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.3.1: JSON 정책 읽기 연습**
>
> **상황**: 다음 IAM 정책이 사용자에게 연결되어 있습니다. 이 사용자가 할 수 있는 것과 없는 것을 파악해 봅시다.
>
> ```json
> {
>   "Version": "2012-10-17",
>   "Statement": [
>     {
>       "Effect": "Allow",
>       "Action": [
>         "s3:GetObject",
>         "s3:PutObject",
>         "s3:ListBucket"
>       ],
>       "Resource": [
>         "arn:aws:s3:::project-data",
>         "arn:aws:s3:::project-data/*"
>       ]
>     },
>     {
>       "Effect": "Allow",
>       "Action": "ec2:Describe*",
>       "Resource": "*"
>     }
>   ]
> }
> ```
>
> **풀이/설명**:
> - **Statement 1**: `project-data` S3 버킷에서 객체를 읽고(`GetObject`), 올리고(`PutObject`), 목록을 조회(`ListBucket`)할 수 있습니다
> - **Statement 2**: 모든(`*`) EC2 리소스에 대해 Describe(조회) 작업을 할 수 있습니다 (인스턴스 목록 보기, 보안 그룹 조회 등)
> - **할 수 있는 것**: S3 project-data 버킷에 파일 업로드/다운로드, EC2 인스턴스 목록 확인
> - **할 수 없는 것**: S3 project-data 버킷에서 파일 삭제, 다른 S3 버킷 접근, EC2 인스턴스 시작/중지/생성, 그 외 모든 AWS 서비스
>
> **핵심 포인트**: 정책에 명시되지 않은 모든 작업은 기본적으로 거부됩니다. "허용된 것만 할 수 있다"가 IAM의 기본 원칙입니다.

> 🔍 **예시 13.3.2: Deny가 Allow를 이기는 상황**
>
> **상황**: 개발자 이 씨에게 두 가지 정책이 연결되어 있습니다.
> - **정책 A** (그룹 "Developers"에 연결): `"Effect": "Allow", "Action": "s3:*", "Resource": "*"` — 모든 S3 작업 허용
> - **정책 B** (이 씨에게 직접 연결): `"Effect": "Deny", "Action": "s3:DeleteBucket", "Resource": "*"` — S3 버킷 삭제 거부
>
> **풀이/설명**:
> - 이 씨가 S3 버킷에 파일을 업로드합니다 → 정책 A에서 Allow, 정책 B에서 관련 Deny 없음 → ✅ **허용**
> - 이 씨가 S3 버킷을 삭제하려 합니다 → 정책 A에서 Allow, **정책 B에서 Deny** → ❌ **거부** (Deny가 이김)
>
> **핵심 포인트**: 여러 정책에서 충돌이 발생하면 **Deny가 항상 우선**합니다. 이 패턴은 "넓게 허용하되, 특정 위험한 작업만 명시적으로 차단"할 때 유용합니다.

> 🔬 **Deep Dive: 정책 평가 로직의 전체 흐름**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 실제 AWS의 정책 평가는 앞서 설명한 것보다 더 복잡한 계층 구조를 가집니다. 사용자에게 직접 연결된 정책 외에도, **서비스 제어 정책(SCP)**, **권한 경계(Permissions Boundary)**, **세션 정책** 등이 함께 평가됩니다.
>
> 평가 우선순위 (상위가 더 강력):
> 1. **명시적 Deny** — 어디서든 Deny가 있으면 최종 거부
> 2. **SCP (Service Control Policy)** — 조직 수준에서 계정의 최대 권한을 제한
> 3. **권한 경계 (Permissions Boundary)** — 사용자/역할의 최대 권한을 제한
> 4. **자격 증명 기반 정책** — 사용자/그룹/역할에 연결된 정책
> 5. **리소스 기반 정책** — 리소스(예: S3 버킷)에 직접 설정된 정책
>
> 💬 **참고**: SCP는 Chapter 16에서 AWS Organizations와 함께 자세히 다룹니다.

> ⚠️ **주의**
>
> - **`"Action": "*"`, `"Resource": "*"` 조합은 매우 위험합니다.** 이 조합은 "모든 서비스의 모든 작업을 모든 리소스에 대해" 허용하는 것으로, 사실상 관리자 권한과 같습니다. `AdministratorAccess` 정책이 아닌 곳에서 이 조합을 사용하지 마세요.
> - **정책의 Version은 날짜처럼 보이지만 버전 번호입니다.** `"2012-10-17"`은 가장 최신의 정책 문법 버전을 뜻합니다. 이 값을 바꾸거나 생략하면 일부 기능(예: 정책 변수)이 작동하지 않습니다.
> - **Resource에서 버킷과 버킷 내 객체는 별도로 지정해야 합니다.** `arn:aws:s3:::my-bucket`은 버킷 자체, `arn:aws:s3:::my-bucket/*`은 버킷 안의 객체들입니다. `ListBucket`은 버킷에 대한 작업이고, `GetObject`는 객체에 대한 작업이므로 두 ARN을 모두 넣어야 합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: IAM 정책에서 자주 혼동되는 사항들
>
> - **명시적 Deny와 묵시적 거부를 혼동하지 마세요**: 정책에 아무것도 명시하지 않으면 "묵시적 거부(기본 거부)"이고, `"Effect": "Deny"`를 명시하면 "명시적 거부"입니다. 명시적 Deny는 어떤 Allow보다도 항상 우선합니다.
> - **`"Action": "*"`, `"Resource": "*"` 조합은 매우 위험합니다**: 이 조합은 사실상 모든 리소스에 대한 모든 작업을 허용하는 관리자 권한입니다. 최소 권한 원칙에 따라, 꼭 필요한 Action과 Resource만 지정하세요.
> - **AWS 관리형 정책과 고객 관리형 정책을 구분하세요**: AWS 관리형 정책은 AWS가 생성·유지하고, 고객 관리형 정책은 사용자가 직접 생성·관리합니다. 인라인 정책은 특정 사용자·그룹·역할에 직접 내장되며, 재사용이 불가능합니다.
> - **정책의 Version 필드는 날짜가 아니라 정책 문법 버전입니다**: `"2012-10-17"`은 최신 정책 문법 버전을 뜻합니다. 이 값을 임의로 변경하면 정책이 정상 작동하지 않을 수 있습니다.
> - **여러 정책이 연결되면 권한이 합산됩니다**: 단, 어디서든 명시적 Deny가 하나라도 있으면 해당 작업은 거부됩니다. "Deny가 Allow를 이긴다"는 CCP 시험의 핵심 포인트입니다.

### Section 요약 (Section Summary)

- IAM 정책은 **JSON 문서**로, `Effect`, `Action`, `Resource`가 핵심 요소입니다
- **AWS 관리형, 고객 관리형, 인라인** 세 종류의 정책이 있으며, 관리형 정책 사용이 권장됩니다
- Action은 `서비스:작업` 형식이며, 와일드카드(`*`)로 여러 작업을 지정할 수 있습니다
- **명시적 Deny는 항상 Allow를 이깁니다** — 정책 평가의 가장 중요한 규칙
- 정책에 명시되지 않은 모든 작업은 기본적으로 **묵시적 거부**됩니다

## 13.4 IAM 역할 — 역할의 개념, EC2 역할, 교차 계정 접근, 서비스 연결 역할

> 💡 **한 줄 요약**: IAM 역할은 사용자가 아닌 AWS 서비스나 외부 개체가 "임시로 빌려 쓰는" 권한 집합입니다.

### 직관적 이해 (Intuitive Understanding)

호텔에서 비유를 찾아봅시다.

호텔에는 **상시 직원**이 있습니다 — 프론트 데스크 직원, 청소 직원, 매니저 등. 이들은 각자 사원증(IAM 사용자)을 가지고 있고, 자신의 업무 구역에 접근할 수 있습니다.

그런데 가끔 **외부 업체**가 옵니다 — 에어컨 수리 기사, 이벤트 진행 업체, 보안 점검 요원. 이들에게 상시 사원증을 발급하는 것은 부적절합니다. 대신, **임시 출입증**을 발급합니다. 이 임시 출입증에는:

- **접근 가능 구역**이 제한되어 있고 (정책)
- **유효 기간**이 있어 시간이 지나면 자동으로 무효화됩니다 (임시 자격 증명)

IAM 역할이 바로 이 **임시 출입증** 시스템입니다.

**왜 IAM 사용자가 아닌 역할이 필요할까요?**

핵심적인 이유가 있습니다: **AWS 서비스에게 권한을 줘야 할 때**. 예를 들어, EC2 인스턴스 위에서 실행되는 애플리케이션이 S3에 파일을 업로드해야 한다면, 이 EC2 인스턴스에 접근 키를 저장해 두는 것은 위험합니다 (키가 유출될 수 있음). 대신, EC2 인스턴스에 IAM 역할을 부여하면, 인스턴스가 자동으로 임시 자격 증명을 받아서 안전하게 S3에 접근합니다.

### 핵심 개념 설명 (Core Explanation)

#### IAM 역할 (IAM Role)

> 📝 **IAM 역할 (IAM Role)**: 특정 권한을 가진 IAM 자격 증명으로, 사용자처럼 장기 자격 증명(비밀번호, 접근 키)을 갖지 않고, 역할을 "맡는(assume)" 개체에게 **임시 보안 자격 증명**을 발급합니다. AWS 서비스, 다른 AWS 계정의 사용자, 외부 자격 증명 공급자가 역할을 맡을 수 있습니다.

IAM 역할의 핵심 구성 요소 두 가지:

1. **신뢰 정책 (Trust Policy)**: "누가 이 역할을 맡을 수 있는가?" — 역할을 사용할 수 있는 대상을 정의
2. **권한 정책 (Permissions Policy)**: "이 역할을 맡으면 무엇을 할 수 있는가?" — 역할이 부여하는 권한을 정의

```
┌─────────────────────────────┐
│         IAM 역할             │
│                              │
│  신뢰 정책: "누가 맡을 수 있나?"  │  ← EC2 서비스, 다른 계정 등
│  권한 정책: "무엇을 할 수 있나?"  │  ← S3 읽기, DynamoDB 쓰기 등
│                              │
│  → 맡으면 임시 자격 증명 발급     │
└─────────────────────────────┘
```

#### AWS STS (Security Token Service)

> 📝 **AWS STS (Security Token Service)**: IAM 역할을 맡는 개체에게 **임시 보안 자격 증명**을 발급하는 서비스. 발급된 자격 증명은 접근 키 ID, 비밀 접근 키, 그리고 **세션 토큰**으로 구성되며, 지정된 시간이 지나면 자동으로 만료됩니다.

임시 자격 증명의 장점:
- **자동 만료**: 유효 기간(기본 1시간)이 지나면 자동으로 무효화되어, 유출되어도 피해가 제한됩니다
- **별도 회수 불필요**: 장기 접근 키처럼 "삭제" 작업을 하지 않아도 됩니다
- **자동 갱신**: EC2 역할의 경우, 만료 전에 자동으로 새 자격 증명이 발급됩니다

#### 역할의 주요 사용 시나리오

**시나리오 1: AWS 서비스에 권한 부여 (가장 흔한 경우)**

EC2 인스턴스가 S3에 접근해야 할 때:

```
EC2 인스턴스 ──(역할을 맡음)──▶ IAM 역할 (S3 접근 권한)
     │
     ▼
STS가 임시 자격 증명 발급
     │
     ▼
EC2가 임시 자격 증명으로 S3에 접근
```

**시나리오 2: 교차 계정 접근 (Cross-Account Access)**

회사에 "개발 계정"과 "운영 계정" 두 개의 AWS 계정이 있을 때, 개발 계정의 사용자가 운영 계정의 특정 리소스에 접근해야 할 수 있습니다.

```
개발 계정의 사용자 ──(역할 전환)──▶ 운영 계정의 IAM 역할
                                      │
                                      ▼
                                운영 계정의 리소스에 접근
```

**시나리오 3: 외부 자격 증명 연동 (Federation)**

회사의 기존 로그인 시스템(Active Directory, Google 등)으로 인증된 사용자가 AWS에 접근할 때, IAM 사용자를 별도로 만들지 않고 역할을 통해 임시 접근 권한을 부여합니다.

#### 서비스 연결 역할 (Service-Linked Role)

> 📝 **서비스 연결 역할 (Service-Linked Role)**: 특정 AWS 서비스가 사용자 대신 작업을 수행할 수 있도록, 해당 서비스에 직접 연결된 사전 정의 역할. AWS가 자동으로 생성하고 관리하며, 사용자가 권한 정책을 수정할 수 없습니다.

예를 들어, Auto Scaling 서비스가 자동으로 EC2 인스턴스를 시작/중지할 수 있는 것은, 서비스 연결 역할이 있기 때문입니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.4.1: EC2 인스턴스에 S3 접근 권한 부여 — 잘못된 방법 vs 올바른 방법**
>
> **상황**: EC2 인스턴스에서 실행되는 웹 애플리케이션이 S3 버킷에서 이미지를 가져와야 합니다.
>
> **풀이/설명**:
>
> **❌ 잘못된 방법: 접근 키를 EC2에 저장**
> - IAM 사용자의 접근 키를 EC2 인스턴스의 설정 파일에 저장
> - 위험: 인스턴스가 해킹되면 접근 키 유출 → 장기 자격 증명이므로 교체할 때까지 악용 가능
> - 위험: 여러 인스턴스에 키를 복사하면 관리 불가
>
> **✅ 올바른 방법: IAM 역할을 EC2에 연결**
> - S3 읽기 권한이 포함된 IAM 역할을 생성
> - 신뢰 정책에 "EC2 서비스가 이 역할을 맡을 수 있다"고 설정
> - EC2 인스턴스에 이 역할을 연결 (인스턴스 프로파일)
> - EC2가 자동으로 임시 자격 증명을 받아 S3에 접근
> - 장점: 접근 키를 저장할 필요 없음, 자격 증명이 자동 갱신됨, 역할 해제 시 즉시 권한 회수
>
> **핵심 포인트**: AWS 서비스에 권한을 부여할 때는 항상 IAM 역할을 사용하세요. 접근 키를 서비스에 하드코딩하는 것은 보안 안티패턴입니다.

> 🔍 **예시 13.4.2: 교차 계정 접근 시나리오**
>
> **상황**: 회사에 "Dev" 계정(111111111111)과 "Prod" 계정(222222222222)이 있습니다. Dev 계정의 개발자가 Prod 계정의 S3 버킷에서 로그를 확인해야 합니다.
>
> **풀이/설명**:
> 1. **Prod 계정**에서 IAM 역할 `CrossAccountLogReader`를 생성합니다
>    - 신뢰 정책: "Dev 계정(111111111111)의 특정 사용자가 이 역할을 맡을 수 있다"
>    - 권한 정책: "Prod 계정의 로그 S3 버킷에서 읽기만 허용"
> 2. **Dev 계정**의 개발자가 AWS 콘솔에서 "역할 전환(Switch Role)"을 수행합니다
>    - Prod 계정 ID와 역할 이름을 입력
>    - STS가 임시 자격 증명을 발급
> 3. 개발자가 Prod 계정의 S3 로그 버킷을 조회합니다
> 4. 작업이 끝나면 "원래 역할로 돌아가기"를 클릭하여 Dev 계정으로 복귀합니다
>
> **핵심 포인트**: 교차 계정 접근은 Prod 계정에 별도의 IAM 사용자를 만들지 않고도 안전하게 수행할 수 있습니다. 임시 자격 증명이 자동 만료되므로 보안 위험이 최소화됩니다.

> ⚠️ **주의**
>
> - **IAM 역할과 IAM 사용자를 혼동하지 마세요.** 사용자는 "특정 사람/앱에 영구적으로 연결된 자격 증명"이고, 역할은 "필요할 때 잠깐 빌려 쓰는 권한 세트"입니다. 역할에는 비밀번호나 영구 접근 키가 없습니다.
> - **하나의 EC2 인스턴스에는 하나의 역할만** 연결할 수 있습니다. 여러 권한이 필요하면, 하나의 역할에 여러 정책을 연결하세요.
> - **서비스 연결 역할은 삭제하기 전에 주의하세요.** 서비스가 사용 중인 역할을 삭제하면 해당 서비스가 정상 동작하지 않을 수 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: IAM 역할에서 자주 혼동되는 사항들
>
> - **IAM 역할과 IAM 사용자를 혼동하지 마세요**: 사용자는 "특정 사람/앱에 영구적으로 연결된 자격 증명"이고, 역할은 "필요할 때 잠깐 빌려 쓰는 임시 권한 세트"입니다. 역할에는 비밀번호나 영구 접근 키가 없습니다.
> - **EC2에서 AWS 서비스에 접근할 때 접근 키 대신 IAM 역할을 사용하세요**: EC2 인스턴스에 접근 키를 저장하면 키 노출 위험이 있습니다. IAM 역할을 연결하면 STS가 임시 자격 증명을 자동 발급·갱신합니다.
> - **하나의 EC2 인스턴스에는 하나의 역할만 연결할 수 있습니다**: 여러 권한이 필요하면, 하나의 역할에 여러 정책을 연결하세요. 역할을 여러 개 동시에 연결하는 것은 불가능합니다.
> - **교차 계정 접근에서 역할의 "신뢰 정책"을 잊지 마세요**: 역할에 권한 정책(무엇을 할 수 있는지)뿐 아니라 신뢰 정책(누가 이 역할을 맡을 수 있는지)도 올바르게 설정해야 합니다.

### Section 요약 (Section Summary)

- **IAM 역할**은 장기 자격 증명 없이 **임시 보안 자격 증명**을 발급하는 권한 집합입니다
- **AWS STS**가 임시 자격 증명을 발급하며, 자동 만료로 보안성이 높습니다
- 주요 사용 시나리오: **AWS 서비스에 권한 부여**, **교차 계정 접근**, **외부 자격 증명 연동**
- EC2에 접근 키를 저장하는 대신 **IAM 역할**을 사용하는 것이 보안 모범 사례입니다
- **서비스 연결 역할**은 AWS 서비스가 자동으로 생성·관리하는 사전 정의 역할입니다

## 13.5 IAM Identity Center — 다중 계정 환경에서의 SSO, 자격 증명 연동

> 💡 **한 줄 요약**: IAM Identity Center는 여러 AWS 계정과 클라우드 애플리케이션에 하나의 로그인으로 접근할 수 있게 해주는 중앙 집중식 접근 관리 서비스입니다.

### 직관적 이해 (Intuitive Understanding)

대학 캠퍼스를 떠올려 봅시다.

학생이 도서관, 식당, 체육관, 연구실을 이용하려면, 각 건물마다 따로 회원 가입을 하고 ID/비밀번호를 만들어야 한다면 매우 불편할 것입니다. 실제로는 **학생증 하나**로 모든 시설에 출입합니다. 학생증 시스템은 학교 본부에서 중앙 관리하며, 졸업하면 모든 시설 접근이 한 번에 차단됩니다.

**IAM Identity Center(이전 이름: AWS SSO)** 가 바로 이 "학생증 시스템"입니다. 직원이 한 번 로그인하면, 여러 AWS 계정과 비즈니스 애플리케이션에 추가 로그인 없이 접근할 수 있습니다.

**왜 기본 IAM만으로는 부족할까요?**

기업이 성장하면 AWS 계정이 하나가 아닙니다. 개발, 스테이징, 운영, 보안, 로깅 등 목적별로 여러 계정을 운영합니다. 직원 200명이 10개 계정에 접근해야 한다면, 각 계정마다 IAM 사용자를 만들면 200 × 10 = 2,000개의 사용자를 관리해야 합니다. IAM Identity Center를 사용하면, 직원 계정을 한 곳에서 관리하고 필요한 계정에 대한 접근 권한만 부여하면 됩니다.

### 핵심 개념 설명 (Core Explanation)

#### IAM Identity Center

> 📝 **IAM Identity Center (구 AWS SSO)**: AWS에서 제공하는 **싱글 사인온(SSO)** 서비스. 사용자가 한 번 인증하면 여러 AWS 계정, Amazon 비즈니스 애플리케이션, SAML 2.0 지원 애플리케이션에 추가 인증 없이 접근할 수 있습니다.

> 📝 **싱글 사인온 (Single Sign-On, SSO)**: 사용자가 한 번의 로그인으로 연결된 여러 시스템에 접근할 수 있는 인증 방식. 각 시스템마다 별도의 자격 증명을 관리할 필요가 없습니다.

#### 자격 증명 소스 (Identity Source)

IAM Identity Center는 사용자 정보를 어디서 가져올지 선택할 수 있습니다:

| 자격 증명 소스 | 설명 | 적합한 경우 |
|---------------|------|-------------|
| **Identity Center 디렉터리** | IAM Identity Center 자체에서 사용자를 관리 | 소규모 조직, 별도 디렉터리가 없을 때 |
| **Active Directory** | 기존 회사의 AD를 연결 | 이미 AD로 직원 관리 중인 기업 |
| **외부 IdP** | Okta, Azure AD, Google Workspace 등 외부 자격 증명 공급자 | 기존 IdP를 사용 중인 기업 |

> 📝 **자격 증명 연동 (Federation)**: 외부 자격 증명 시스템의 사용자 정보를 AWS와 연결하여, 외부 시스템의 인증으로 AWS 리소스에 접근할 수 있게 하는 방식. IAM Identity Center는 이 연동을 쉽게 설정할 수 있게 합니다.

#### 권한 세트 (Permission Set)

> 📝 **권한 세트 (Permission Set)**: IAM Identity Center에서 사용자에게 할당하는 **AWS 계정별 권한의 템플릿**. 내부적으로 IAM 역할로 변환되어 대상 계정에 생성됩니다.

작동 흐름:

```
직원 ──(한 번 로그인)──▶ IAM Identity Center 포털
                              │
                    ┌─────────┼─────────┐
                    ▼         ▼         ▼
              Dev 계정    Staging 계정  Prod 계정
           (개발자 권한)  (개발자 권한)  (읽기 전용)
```

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.5.1: 기업의 다중 계정 접근 관리**
>
> **상황**: 중견 기업에 50명의 직원, 5개의 AWS 계정(Dev, Staging, Prod, Security, Logging)이 있습니다. 기존에는 각 계정마다 IAM 사용자를 만들어 관리했습니다.
>
> **풀이/설명**:
> - **기존 방식의 문제점**:
>   - 50명 × 5개 계정 = 최대 250개의 IAM 사용자 관리
>   - 직원 퇴사 시 5개 계정 각각에서 사용자 삭제 필요 → 하나라도 빠뜨리면 보안 구멍
>   - 비밀번호 정책을 계정마다 별도로 설정
>
> - **IAM Identity Center 도입 후**:
>   - Identity Center에 50명의 사용자 등록 (한 곳에서 관리)
>   - "개발자" 권한 세트, "읽기 전용" 권한 세트, "관리자" 권한 세트 생성
>   - 개발자 30명에게 Dev 계정은 "개발자" 권한, Prod 계정은 "읽기 전용" 권한 할당
>   - 운영팀 10명에게 Prod 계정은 "관리자" 권한 할당
>   - 퇴사 시 Identity Center에서 한 번만 비활성화 → 모든 계정 접근 즉시 차단
>
> **핵심 포인트**: 다중 계정 환경에서 IAM Identity Center는 "한 곳에서 관리, 여러 곳에 적용"을 실현합니다.

> 🔍 **예시 13.5.2: 기존 회사 디렉터리와 연동**
>
> **상황**: 한 회사에서 이미 Microsoft Active Directory로 직원의 인증과 그룹을 관리하고 있습니다. AWS를 새로 도입하면서 직원들이 또 다른 계정을 만들어야 하는 불편을 없애고 싶습니다.
>
> **풀이/설명**:
> 1. IAM Identity Center에서 자격 증명 소스를 "Active Directory"로 설정
> 2. AWS Directory Service를 통해 기존 AD와 연결
> 3. AD의 "DevTeam" 그룹 → Dev 계정 "개발자" 권한 세트 연결
> 4. AD의 "OpsTeam" 그룹 → Prod 계정 "운영자" 권한 세트 연결
> 5. 직원이 회사 AD 자격 증명(기존 사번/비밀번호)으로 AWS 포털에 로그인 가능
>
> **핵심 포인트**: 자격 증명 연동을 사용하면, 직원이 새로운 자격 증명을 외울 필요 없이 기존 회사 인증 시스템으로 AWS에 접근할 수 있습니다.

> ⚠️ **주의**
>
> - **IAM Identity Center와 IAM은 다른 서비스입니다.** IAM은 단일 계정 내에서 사용자·그룹·역할·정책을 관리하고, IAM Identity Center는 여러 계정과 애플리케이션에 걸친 SSO 접근을 관리합니다. 둘은 상호 보완적입니다.
> - **IAM Identity Center는 AWS Organizations와 함께 사용합니다.** 다중 계정을 관리하려면 AWS Organizations가 필요합니다.
>
> 💬 **참고**: AWS Organizations는 Chapter 16에서 자세히 다룹니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: IAM Identity Center에서 자주 혼동되는 사항들
>
> - **IAM Identity Center와 IAM은 별개의 서비스입니다**: IAM은 단일 계정 내 사용자·그룹·역할·정책을 관리하고, IAM Identity Center는 여러 계정과 애플리케이션에 걸친 SSO 접근을 관리합니다. 둘은 대체 관계가 아니라 상호 보완적입니다.
> - **IAM Identity Center는 AWS Organizations가 필요합니다**: 다중 계정 환경에서 SSO를 구현하려면 AWS Organizations가 먼저 설정되어 있어야 합니다.
> - **권한 세트(Permission Set)와 IAM 정책을 혼동하지 마세요**: 권한 세트는 IAM Identity Center에서 "어떤 사용자가 어떤 계정에서 어떤 권한을 가지는지"를 정의하는 템플릿입니다. 내부적으로 IAM 역할과 정책으로 변환되지만, 관리 방식이 다릅니다.
> - **SSO(Single Sign-On)가 보안을 약화시킨다고 오해하지 마세요**: SSO는 하나의 자격 증명으로 여러 시스템에 접근하므로 편리하지만, MFA와 결합하면 오히려 보안이 강화됩니다. 여러 계정마다 별도 비밀번호를 관리하는 것보다 안전할 수 있습니다.

### Section 요약 (Section Summary)

- **IAM Identity Center**는 여러 AWS 계정과 애플리케이션에 **한 번의 로그인(SSO)** 으로 접근할 수 있게 합니다
- **자격 증명 소스**로 자체 디렉터리, Active Directory, 외부 IdP를 사용할 수 있습니다
- **권한 세트(Permission Set)** 로 "어떤 사용자가 어떤 계정에서 어떤 권한을 가지는지" 관리합니다
- **자격 증명 연동(Federation)** 을 통해 기존 회사 인증 시스템과 AWS를 연결할 수 있습니다

## 13.6 AWS Resource Access Manager — 계정 간 리소스 공유, RAM의 역할과 활용

> 💡 **한 줄 요약**: AWS RAM은 여러 AWS 계정 간에 특정 리소스를 안전하게 공유하여, 리소스를 중복 생성하지 않고도 다른 계정에서 사용할 수 있게 합니다.

### 직관적 이해 (Intuitive Understanding)

아파트 단지를 떠올려 봅시다. 각 세대(AWS 계정)는 독립적인 공간이지만, **놀이터, 주차장, 피트니스센터**는 단지 전체가 공유합니다. 각 세대가 놀이터를 별도로 만들 필요 없이, 관리사무소가 공유 시설의 접근 권한을 관리합니다.

**AWS RAM(Resource Access Manager)** 이 바로 이 "관리사무소"입니다. 한 계정에서 만든 리소스를 다른 계정과 공유할 수 있습니다.

**왜 리소스를 공유해야 할까요?** 예를 들어, 회사의 네트워크팀이 VPC 서브넷을 설계했습니다. 개발팀 계정과 운영팀 계정에서 이 서브넷을 사용해야 할 때, 각 계정에서 동일한 네트워크를 다시 만드는 것은 낭비이고 일관성을 해칩니다. RAM을 사용하면 네트워크팀의 서브넷을 다른 계정에서 직접 사용할 수 있습니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **AWS Resource Access Manager (RAM)**: AWS 리소스를 다른 AWS 계정 또는 AWS Organizations 내의 계정과 안전하게 공유하는 서비스. 리소스를 복제하지 않고 원본 리소스에 대한 접근을 허용합니다.

#### RAM으로 공유할 수 있는 대표적 리소스

| 리소스 | 사용 예시 |
|--------|-----------|
| VPC 서브넷 | 중앙에서 네트워크를 설계하고 여러 계정이 같은 서브넷 사용 |
| Transit Gateway | 여러 계정의 VPC를 하나의 Transit Gateway로 연결 |
| Route 53 Resolver 규칙 | DNS 규칙을 여러 계정에서 공유 |
| License Manager 구성 | 소프트웨어 라이선스를 계정 간 공유 |
| Aurora DB 클러스터 | 데이터베이스를 여러 계정에서 읽기 가능 |

#### IAM 역할을 통한 교차 계정 접근과의 차이

앞서 13.4에서 배운 IAM 역할을 통한 교차 계정 접근과 RAM은 목적이 다릅니다:

| 비교 항목 | IAM 역할 (교차 계정) | AWS RAM |
|-----------|---------------------|---------|
| **방식** | 다른 계정의 역할을 맡아서 해당 계정에서 작업 | 리소스 자체를 공유하여 내 계정에서 사용 |
| **적합한 경우** | 다른 계정의 여러 리소스에 접근해야 할 때 | 특정 리소스를 여러 계정이 함께 사용할 때 |
| **계정 전환** | 역할 전환이 필요 | 전환 없이 자기 계정에서 사용 |

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.6.1: VPC 서브넷 공유**
>
> **상황**: 네트워크팀 계정에서 회사 전체 VPC와 서브넷을 설계했습니다. 개발팀 계정에서 EC2를 이 서브넷에 배포해야 합니다.
>
> **풀이/설명**:
> 1. 네트워크팀 계정에서 RAM을 통해 서브넷을 개발팀 계정과 공유
> 2. 개발팀 계정에서 EC2 인스턴스 생성 시, 공유된 서브넷을 선택 가능
> 3. EC2 인스턴스는 개발팀 계정 소유이지만, 네트워크팀이 관리하는 서브넷에 배치됨
> 4. 네트워크 설계와 보안 규칙은 네트워크팀이 중앙에서 관리
>
> **핵심 포인트**: RAM을 사용하면 "네트워크는 네트워크팀이 관리, 서버는 개발팀이 관리"와 같이 역할 분리를 유지하면서도 리소스를 공유할 수 있습니다.

> 🔍 **예시 13.6.2: RAM과 IAM 역할의 선택**
>
> **상황**: 보안팀 계정의 CloudTrail 로그를 감사팀 계정에서 확인해야 합니다.
>
> **풀이/설명**:
> - **IAM 역할 방식**: 감사팀 사용자가 보안팀 계정의 역할을 맡아 보안팀 계정의 CloudTrail 콘솔에서 조회 → 계정을 전환해야 하는 불편함
> - **RAM + S3 방식**: CloudTrail 로그가 저장된 S3 버킷을 RAM 또는 버킷 정책으로 공유 → 감사팀이 자기 계정에서 바로 로그 조회 가능
> - 이 경우 RAM보다는 S3 버킷 정책을 사용하는 것이 더 일반적 (S3는 자체적으로 리소스 기반 정책을 지원하므로)
>
> **핵심 포인트**: 모든 교차 계정 접근에 RAM을 쓸 필요는 없습니다. 리소스 유형과 상황에 따라 IAM 역할, RAM, 리소스 기반 정책 중 적절한 방법을 선택합니다.

> ⚠️ **주의**
>
> - **RAM으로 공유한 리소스의 소유권은 변하지 않습니다.** 공유받은 계정은 리소스를 사용할 수 있지만, 삭제하거나 공유를 재배포할 수는 없습니다.
> - **모든 AWS 리소스를 RAM으로 공유할 수 있는 것은 아닙니다.** RAM이 지원하는 리소스 유형은 정해져 있습니다.
> - **RAM은 AWS Organizations 내에서 가장 효과적입니다.** Organizations 밖의 계정과 공유할 수도 있지만, 초대를 수락하는 추가 단계가 필요합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: AWS RAM에서 자주 혼동되는 사항들
>
> - **RAM과 IAM 역할 교차 계정 접근을 혼동하지 마세요**: IAM 역할은 다른 계정의 리소스에 "접근 권한을 위임"하는 것이고, RAM은 리소스 자체를 다른 계정과 "공유"하는 것입니다. 목적과 방식이 다릅니다.
> - **RAM으로 공유한 리소스의 소유권은 변하지 않습니다**: 공유받은 계정은 리소스를 사용할 수 있지만, 소유권은 원래 계정에 남아 있으며 삭제 권한은 없습니다.
> - **모든 AWS 리소스를 RAM으로 공유할 수 있는 것은 아닙니다**: RAM이 지원하는 리소스 유형은 정해져 있습니다. 대표적으로 VPC 서브넷, Transit Gateway, Route 53 Resolver 규칙 등이 있습니다.
> - **리소스 기반 정책과 RAM의 차이를 이해하세요**: S3 버킷처럼 자체 리소스 기반 정책을 지원하는 서비스는 RAM 없이도 교차 계정 접근이 가능합니다. RAM은 리소스 기반 정책을 지원하지 않는 서비스에서 특히 유용합니다.

### Section 요약 (Section Summary)

- **AWS RAM**은 AWS 리소스를 다른 계정과 **복제 없이 안전하게 공유**하는 서비스입니다
- 가장 대표적인 사용 사례는 **VPC 서브넷 공유**, **Transit Gateway 공유**입니다
- IAM 역할 교차 계정 접근과 RAM은 **목적이 다르며 상호 보완적**입니다
- 공유받은 계정은 리소스를 **사용할 수 있지만 소유권은 원래 계정에 남습니다**

## 13.7 IAM 보안 모범 사례 — 루트 계정 보호, MFA 적용, 접근 키 관리, 정기 감사

> 💡 **한 줄 요약**: IAM 보안 모범 사례는 "루트 계정 잠그기, MFA 필수화, 최소 권한 적용, 접근 키 관리, 정기 감사"로 요약됩니다.

### 직관적 이해 (Intuitive Understanding)

은행 금고를 관리한다고 생각해 봅시다. 아무리 좋은 금고라도 관리를 소홀히 하면 무용지물입니다:

- 금고 마스터 키(루트 계정)를 매일 들고 다니면? → 분실 위험
- 비밀번호 하나로만 잠그면? → 비밀번호 유출 시 속수무책
- 모든 직원에게 마스터 키를 복사해 주면? → 누가 열었는지 추적 불가
- 퇴직한 직원의 키를 회수하지 않으면? → 보안 구멍

IAM 보안 모범 사례는 이런 "상식적인" 보안 규칙을 AWS 환경에 체계적으로 적용하는 것입니다.

> 🔗 **연결**: Chapter 12에서 배운 **최소 권한 원칙**과 **심층 방어(Defense in Depth)** 가 IAM 보안 모범 사례의 핵심 철학입니다.

### 핵심 개념 설명 (Core Explanation)

#### 모범 사례 1: 루트 사용자 보호

루트 사용자는 AWS 계정의 "슈퍼 관리자"입니다. 루트 사용자만 할 수 있는 작업이 있어 완전히 비활성화할 수는 없지만, 일상적으로 사용하면 안 됩니다.

**루트 사용자 보호 원칙:**
- 루트 사용자로 일상 작업을 하지 않습니다
- 루트 사용자에 **MFA를 반드시 활성화**합니다
- 루트 사용자의 **접근 키를 만들지 않습니다**
- 루트 사용자가 필요한 작업(계정 설정 변경, Support 플랜 변경 등)에만 사용합니다

#### 모범 사례 2: MFA (다중 인증) 적용

> 📝 **MFA (Multi-Factor Authentication, 다중 인증)**: 비밀번호(알고 있는 것) 외에 추가 인증 수단(가지고 있는 것)을 함께 요구하는 보안 방식. 비밀번호가 유출되어도 두 번째 인증 수단 없이는 접근할 수 없습니다.

AWS에서 사용할 수 있는 MFA 유형:

| MFA 유형 | 설명 | 예시 |
|----------|------|------|
| **가상 MFA 장치** | 스마트폰 앱으로 일회용 코드 생성 | Google Authenticator, Authy |
| **하드웨어 MFA** | 물리적 장치로 코드 생성 | YubiKey, Gemalto 토큰 |
| **FIDO2 보안 키** | USB/NFC 보안 키 | YubiKey 5 시리즈 |

**MFA 적용 원칙:**
- **루트 사용자**: MFA 필수 (가장 중요!)
- **관리자 IAM 사용자**: MFA 강력 권장
- **모든 IAM 사용자**: 가능하면 MFA 적용

#### 모범 사례 3: 최소 권한 원칙 적용

> 🔗 **연결**: Chapter 12에서 배운 **최소 권한 원칙(Principle of Least Privilege)** — "업무 수행에 필요한 최소한의 권한만 부여한다"

실천 방법:
- 처음에는 **권한 없이** 시작하고, 필요할 때마다 추가합니다
- `AdministratorAccess` 정책을 무분별하게 부여하지 않습니다
- AWS 관리형 정책 중 상황에 맞는 것을 선택합니다 (예: `AmazonS3ReadOnlyAccess`)
- 필요하면 **고객 관리형 정책**으로 더 세밀하게 제어합니다
- **IAM Access Analyzer**를 사용하여 실제 사용되는 권한을 확인하고, 사용하지 않는 권한을 제거합니다

#### 모범 사례 4: 접근 키 관리

접근 키는 프로그래밍 방식의 AWS 접근에 사용되므로, 유출 시 심각한 피해가 발생합니다.

**접근 키 관리 원칙:**
- **가능하면 접근 키 대신 IAM 역할을 사용합니다** (EC2, Lambda 등에서)
- 접근 키가 반드시 필요한 경우, **정기적으로 교체(rotation)** 합니다
- **사용하지 않는 접근 키는 비활성화하거나 삭제합니다**
- 코드, Git 저장소, 설정 파일에 접근 키를 절대 하드코딩하지 않습니다
- **환경 변수**나 **AWS Secrets Manager**를 통해 관리합니다

#### 모범 사례 5: 정기 감사와 모니터링

- **IAM Credential Report**: 계정 내 모든 IAM 사용자의 자격 증명 상태를 CSV로 확인
- **IAM Access Advisor**: 각 사용자/역할이 마지막으로 AWS 서비스에 접근한 시간 확인
- **AWS CloudTrail**: 모든 IAM 관련 API 호출을 기록하여 "누가, 언제, 무엇을 했는지" 추적

> 📝 **IAM Credential Report**: AWS 계정의 모든 IAM 사용자와 자격 증명 상태(비밀번호 사용 여부, 접근 키 활성 상태, MFA 활성 여부 등)를 CSV 형식으로 다운로드할 수 있는 보고서.

> 📝 **IAM Access Advisor**: IAM 사용자, 그룹, 역할의 서비스 접근 기록을 보여주는 기능. "이 사용자가 마지막으로 S3에 접근한 것은 90일 전"과 같은 정보를 통해 사용하지 않는 권한을 식별하고 제거할 수 있습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 13.7.1: Credential Report로 보안 상태 점검**
>
> **상황**: 보안 담당자가 분기별 IAM 보안 감사를 수행합니다.
>
> **풀이/설명**:
> 1. IAM 콘솔에서 Credential Report를 다운로드합니다
> 2. CSV 파일을 열어 다음을 확인합니다:
>    - `mfa_active`가 `false`인 사용자 → MFA 미설정, 즉시 조치 필요
>    - `access_key_1_last_used_date`가 90일 이전인 사용자 → 접근 키 사용 안 함, 비활성화 검토
>    - `password_last_used`가 없는 사용자 → 콘솔 로그인을 하지 않는 사용자, 비밀번호 삭제 검토
>    - `access_key_1_last_rotated`가 180일 이전인 사용자 → 접근 키 교체 필요
> 3. 발견된 문제를 수정하고 기록합니다
>
> **핵심 포인트**: Credential Report는 계정 전체의 IAM 보안 상태를 한눈에 파악할 수 있는 가장 빠른 방법입니다.

> 🔍 **예시 13.7.2: 접근 키 유출 사고 대응**
>
> **상황**: 개발자가 실수로 접근 키를 GitHub 퍼블릭 저장소에 커밋했습니다.
>
> **풀이/설명**:
> 1. **즉시**: 해당 접근 키를 IAM 콘솔에서 **비활성화** (삭제가 아닌 비활성화 먼저)
> 2. **즉시**: 해당 사용자에게 연결된 정책을 확인하여 피해 범위 파악
> 3. **30분 내**: CloudTrail에서 해당 접근 키로 수행된 작업 내역 확인
> 4. **확인 후**: 비정상 활동(낯선 리전에서 EC2 생성 등)이 있으면 해당 리소스 삭제
> 5. **후속**: 접근 키를 삭제하고, 필요하면 새 접근 키 발급 (가능하면 IAM 역할로 전환)
> 6. **재발 방지**: git-secrets 같은 도구로 접근 키 커밋을 사전 차단
>
> **핵심 포인트**: 접근 키 유출 시 "비활성화 → 피해 확인 → 삭제 → 재발 방지" 순서로 대응합니다. 삭제보다 비활성화를 먼저 하는 이유는, 조사 과정에서 키 정보가 필요할 수 있기 때문입니다.

> **Part 1: MFA 활성화 (가상 MFA 장치 사용)**
>
> **Step 1**: 스마트폰에 MFA 앱을 설치합니다 (Google Authenticator, Microsoft Authenticator, 또는 Authy 중 택 1).
>
> **Step 2**: AWS 콘솔에 루트 사용자로 로그인합니다. 오른쪽 상단의 계정 이름을 클릭하고 **"보안 자격 증명(Security credentials)"** 을 선택합니다.
>
> **Step 3**: "다중 인증(Multi-factor authentication, MFA)" 섹션에서 **"MFA 할당(Assign MFA device)"** 을 클릭합니다.
>   - 장치 이름: `my-root-mfa`
>   - MFA device: **"Authenticator app"** 선택
>   - **"다음(Next)"** 클릭
>
> **Step 4**: "QR 코드 표시(Show QR code)"를 클릭하면 QR 코드가 표시됩니다. 스마트폰의 MFA 앱으로 이 QR 코드를 스캔합니다.
>
> **Step 5**: MFA 앱에 표시되는 6자리 코드를 "MFA 코드 1"에 입력합니다. 30초 후 새로 표시되는 6자리 코드를 "MFA 코드 2"에 입력합니다. **"MFA 추가(Add MFA)"** 를 클릭합니다.
>
> **Part 2: Credential Report 생성**
>
> **Step 6**: IAM 콘솔로 이동합니다. 왼쪽 메뉴에서 **"자격 증명 보고서(Credential report)"** 를 클릭합니다.
>
> **Step 7**: **"보고서 생성(Generate report)"** → **"다운로드(.csv)(Download Report)"** 를 클릭합니다.
>
> **Step 8**: 다운로드된 CSV 파일을 열고 다음을 확인합니다:
>   - `<root_account>` 행의 `mfa_active` 열이 `true`인지 확인 (Step 5에서 설정했으므로 `true`여야 함)
>   - `lab-user-01` 행의 `mfa_active` 열이 `false`인지 확인 (MFA를 설정하지 않았으므로)
>
> **확인**: 루트 사용자의 MFA가 활성화되고, Credential Report에서 `mfa_active`가 `true`로 표시되면 성공입니다.
>
> 🧹 **정리**: MFA 설정은 보안을 위해 **유지합니다** (삭제하지 마세요). Credential Report CSV 파일은 민감 정보이므로 확인 후 삭제하는 것을 권장합니다.

> ⚠️ **주의**
>
> - **MFA 장치를 분실하면 로그인이 불가능해질 수 있습니다.** MFA를 설정한 후에는 복구 방법(복구 코드, 대체 연락처 등)을 반드시 확인하세요. 루트 사용자의 MFA 분실 시 AWS 고객 지원에 연락하여 MFA를 재설정할 수 있습니다.
> - **Credential Report CSV 파일에는 민감한 정보가 포함됩니다.** 이 파일을 이메일로 공유하거나 공유 드라이브에 올리지 마세요.
> - **접근 키가 GitHub 등에 노출되면 몇 분 내에 악용될 수 있습니다.** AWS도 자동 감지 시스템이 있지만, 발견 전에 이미 피해가 발생할 수 있습니다. 예방이 최선입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: IAM 보안 모범 사례에서 자주 혼동되는 사항들
>
> - **MFA를 루트 사용자에만 설정하는 것으로 충분하다고 생각하지 마세요**: 관리자 권한이 있는 IAM 사용자에게도 MFA를 설정하는 것이 권장됩니다. 루트 사용자뿐 아니라 높은 권한을 가진 모든 사용자에게 MFA를 적용하세요.
> - **"최소 권한 원칙"을 처음부터 완벽하게 적용하기 어렵습니다**: 처음에는 필요한 최소 권한을 추정하여 부여하고, IAM Access Advisor를 사용하여 실제 사용하지 않는 권한을 점진적으로 제거하는 방식이 현실적입니다.
> - **Credential Report와 Access Advisor의 용도를 혼동하지 마세요**: Credential Report는 모든 IAM 사용자의 자격 증명 상태(비밀번호 마지막 사용, MFA 활성화 여부 등)를 보여주고, Access Advisor는 특정 IAM 개체가 어떤 서비스에 마지막으로 접근했는지를 보여줍니다.
> - **접근 키를 정기적으로 교체(Rotate)하세요**: 접근 키는 생성 후 변경하지 않으면 노출 위험이 누적됩니다. 가능하면 접근 키 대신 IAM 역할을 사용하고, 불가피하게 접근 키를 사용해야 한다면 90일 이내 주기로 교체하는 것이 좋습니다.

### Section 요약 (Section Summary)

- **루트 사용자 보호**: 일상 작업에 사용하지 않고, MFA를 반드시 설정합니다
- **MFA**: 비밀번호 외 추가 인증 수단을 요구하여 보안을 강화합니다
- **최소 권한 원칙**: 필요한 최소한의 권한만 부여하고, 사용하지 않는 권한은 제거합니다
- **접근 키 관리**: 가능하면 IAM 역할로 대체하고, 정기적으로 교체·감사합니다
- **정기 감사**: Credential Report와 Access Advisor로 보안 상태를 주기적으로 점검합니다

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 AWS의 자격 증명 및 접근 관리 서비스인 IAM을 체계적으로 학습했습니다.

- **13.1 IAM 개요**: IAM은 AWS 리소스에 대한 접근을 제어하는 **무료 글로벌 서비스**이며, **인증(누구인가)**과 **인가(무엇을 할 수 있는가)**를 모두 담당합니다.
- **13.2 IAM 사용자와 그룹**: **IAM 사용자**는 개별 접근 신원이고, **그룹**으로 묶어 같은 권한을 효율적으로 관리합니다. 사용자는 콘솔(비밀번호)과 프로그래밍(접근 키) 방식으로 접근합니다.
- **13.3 IAM 정책 심화**: IAM 정책은 **JSON 문서**로, `Effect`, `Action`, `Resource`가 핵심입니다. **명시적 Deny는 항상 Allow를 이기며**, 명시되지 않은 모든 작업은 기본 거부됩니다.
- **13.4 IAM 역할**: 역할은 **임시 보안 자격 증명**을 발급하는 권한 집합으로, AWS 서비스에 권한 부여, 교차 계정 접근, 외부 연동에 사용됩니다. 접근 키 대신 역할을 사용하는 것이 모범 사례입니다.
- **13.5 IAM Identity Center**: 여러 AWS 계정과 애플리케이션에 **SSO(싱글 사인온)** 를 제공하며, 기존 디렉터리와 **자격 증명 연동**이 가능합니다.
- **13.6 AWS RAM**: 여러 AWS 계정 간에 특정 리소스(VPC 서브넷, Transit Gateway 등)를 **복제 없이 공유**합니다.
- **13.7 IAM 보안 모범 사례**: 루트 보호, MFA 필수화, 최소 권한 적용, 접근 키 관리, Credential Report/Access Advisor를 통한 정기 감사가 핵심입니다.

## 자기 점검 질문 (Self-Check Questions)

**Q1.** IAM은 리전별 서비스이다. (O/X)

<details>
<summary>정답 확인</summary>

**X** — IAM은 **글로벌 서비스**입니다. IAM에서 생성한 사용자, 그룹, 역할, 정책은 모든 AWS 리전에서 동일하게 적용됩니다.
</details>

**Q2.** IAM 그룹 안에 다른 IAM 그룹을 포함시킬 수 있다. (O/X)

<details>
<summary>정답 확인</summary>

**X** — IAM 그룹에는 **IAM 사용자만** 포함할 수 있습니다. 그룹 중첩(nesting)은 지원되지 않습니다.
</details>

**Q3.** IAM 정책에서 Effect의 값으로 사용할 수 있는 것은 무엇인가요?

<details>
<summary>정답 확인</summary>

**"Allow"** 와 **"Deny"** 두 가지입니다.
</details>

**Q4.** 한 사용자에게 정책 A에서 Allow, 정책 B에서 Deny가 같은 작업에 적용되면 최종 결과는?

<details>
<summary>정답 확인</summary>

**Deny (거부)** — 명시적 Deny는 항상 Allow를 이깁니다 (Explicit Deny Always Wins).
</details>

**Q5.** EC2 인스턴스가 S3에 접근해야 할 때, 접근 키를 EC2에 저장하는 것보다 권장되는 방법은 무엇인가요?

<details>
<summary>정답 확인</summary>

**IAM 역할**을 EC2 인스턴스에 연결하는 것입니다. 역할은 임시 보안 자격 증명을 자동으로 발급하고 갱신하므로, 접근 키 유출 위험이 없습니다.
</details>

**Q6.** IAM Identity Center의 이전 이름은 무엇이며, 주요 기능은 무엇인가요?

<details>
<summary>정답 확인</summary>

이전 이름은 **AWS SSO (Single Sign-On)** 입니다. 주요 기능은 여러 AWS 계정과 애플리케이션에 **한 번의 로그인(SSO)으로 접근**할 수 있게 하는 중앙 집중식 접근 관리입니다.
</details>

**Q7.** IAM 보안 모범 사례에서 루트 사용자에 대해 가장 먼저 해야 할 것은 무엇인가요?

<details>
<summary>정답 확인</summary>

**MFA(다중 인증)를 활성화**하는 것입니다. 루트 사용자는 계정의 모든 권한을 가지므로, MFA로 추가 보안 계층을 반드시 설정해야 합니다. 그 다음으로, 일상 작업에 루트 사용자를 사용하지 않고 IAM 사용자를 생성하여 사용합니다.
</details>

> **Step 1**: AWS 콘솔에서 IAM 서비스로 이동합니다. 왼쪽 메뉴에서 **"계정 설정(Account settings)"** 을 클릭합니다.

>   - ☑ 최소 하나의 소문자 필요
>   - ☑ 최소 하나의 숫자 필요
>   - ☑ 최소 하나의 영숫자가 아닌 문자(특수문자) 필요
>   - ☑ 비밀번호 만료 활성화: `90`일
>   - ☑ 비밀번호 재사용 방지: 최근 `3`개의 비밀번호 기억
>   - **"변경 사항 저장(Save changes)"** 클릭
>
> **✅ 체크포인트**: 계정 설정 페이지에서 비밀번호 정책이 "사용자 지정"으로 표시되고, 설정한 조건들이 나열되면 성공입니다.

>
> **Step 5**: 두 번째 개발자 사용자를 생성합니다.
>   - 왼쪽 메뉴 **"사용자(Users)"** → **"사용자 생성(Create user)"**
>   - 사용자 이름: `lab-user-02`

>   - 그룹: `lab-developers` 선택
>   - **"사용자 생성(Create user)"** 클릭
>
> **Step 6**: 매니저 사용자를 생성합니다.
>   - 사용자 이름: `lab-user-03`
>   - 콘솔 접근 활성화, IAM 사용자 생성, 사용자 지정 비밀번호 설정
>   - 그룹: `lab-managers` 선택
>   - **"사용자 생성(Create user)"** 클릭
>
> **✅ 체크포인트**: "사용자" 목록에 `lab-user-01`, `lab-user-02`, `lab-user-03` 세 명이 표시되고, "사용자 그룹"에 `lab-developers`(사용자 2명), `lab-managers`(사용자 1명)가 표시되면 성공입니다.

> **Phase 3: 권한 테스트** (활용 개념: Section 13.3 정책)
>
> **Step 7**: 시크릿/프라이빗 브라우저 창을 열고 `lab-user-02`로 로그인합니다.
>   - IAM 사용자 로그인 URL 사용 (Mini Lab 13.2의 Step 5에서 확인한 URL)
>   - S3 서비스로 이동 → **버킷 목록 조회 가능** (S3ReadOnlyAccess)
>   - EC2 서비스로 이동 → **인스턴스 목록 조회 가능** (EC2ReadOnlyAccess)
>   - EC2에서 "인스턴스 시작" 시도 → **권한 오류** (읽기 전용이므로)
>
> **Step 8**: 로그아웃 후 `lab-user-03`으로 로그인합니다.
>   - EC2 서비스로 이동 → **인스턴스 목록 조회 가능** (ViewOnlyAccess에 포함)
>   - EC2에서 "인스턴스 시작" 시도 → **권한 오류** (View 전용이므로)
>   - IAM 서비스로 이동 → 사용자 목록 **조회 가능** (ViewOnlyAccess에 포함)
>   - IAM에서 사용자 생성 시도 → **권한 오류** (View 전용이므로)
>
> **✅ 체크포인트**: 개발자(`lab-user-02`)는 S3와 EC2를 조회할 수 있지만 변경은 불가하고, 매니저(`lab-user-03`)는 전반적으로 조회만 가능하면 성공입니다.

> **Phase 4: Credential Report로 보안 상태 확인** (활용 개념: Section 13.7 보안 모범 사례)
>
> **Step 9**: 루트 사용자 또는 관리자 계정으로 다시 로그인합니다.
>
> **Step 10**: IAM 콘솔에서 **"자격 증명 보고서(Credential report)"** → **"보고서 생성(Generate report)"** → **"다운로드"** 를 클릭합니다.
>
> **Step 11**: CSV 파일을 열어 다음을 확인합니다:
>   - `lab-user-01`, `lab-user-02`, `lab-user-03`이 모두 존재하는지
>   - 세 사용자 모두 `mfa_active`가 `false`인지 (MFA를 설정하지 않았으므로)
>   - `password_enabled`가 `true`인지
>   - 실제 운영 환경이라면 MFA 미설정 사용자에게 MFA 활성화를 요청해야 합니다
>
> **✅ 체크포인트**: Credential Report에서 생성한 세 사용자의 정보를 확인할 수 있으면 성공입니다.

### 🧹 리소스 정리 (Clean-up)

> ⚠️ **비용 방지**: IAM 자체는 무료이지만, 불필요한 사용자와 그룹을 남겨두면 보안 위험이 됩니다. 실습이 끝나면 아래 리소스를 삭제하세요.
>
> 1. **IAM 사용자 삭제** — IAM 콘솔 → 사용자 → 각 사용자(`lab-user-01`, `lab-user-02`, `lab-user-03`) 선택 → "삭제(Delete)" 클릭 → 사용자 이름 입력하여 확인
> 2. **IAM 사용자 그룹 삭제** — IAM 콘솔 → 사용자 그룹 → 각 그룹(`lab-developers`, `lab-managers`) 선택 → "삭제(Delete)" 클릭
> 3. **비밀번호 정책 복원 (선택)** — 계정 설정에서 비밀번호 정책을 "IAM 기본값(IAM default)"으로 되돌릴 수 있습니다
>
> **삭제 순서 참고**: 사용자를 먼저 삭제한 후 그룹을 삭제합니다 (그룹에 사용자가 남아 있어도 삭제 가능하지만, 순서를 지키면 더 깔끔합니다).

### 트러블슈팅 (Troubleshooting)

> **Q: IAM 사용자로 로그인했는데 "Your account is not authorized to visit this page" 메시지가 나옵니다.**
> A: 해당 사용자에게 연결된 정책에 해당 서비스의 권한이 없는 경우입니다. 관리자 계정으로 로그인하여 해당 그룹에 필요한 정책이 연결되어 있는지 확인하세요.

> **Q: 비밀번호를 설정했는데 "Password does not conform to the account password policy" 오류가 발생합니다.**
> A: Phase 1에서 설정한 비밀번호 정책(12자 이상, 대소문자+숫자+특수문자)을 충족하지 못하는 비밀번호입니다. 정책 요건에 맞는 비밀번호를 입력하세요.

> **Q: 콘솔 로그인 URL이 기억나지 않습니다.**
> A: IAM 콘솔의 대시보드 페이지 상단에 "이 계정의 IAM 사용자를 위한 로그인 URL"이 표시됩니다. 형식은 `https://[계정ID].signin.aws.amazon.com/console`입니다. 계정 ID는 루트 사용자로 로그인 후 오른쪽 상단의 계정 메뉴에서 확인할 수 있습니다.

> **Q: Policy Simulator에서 `lab-user-01`이 보이지 않습니다.**
> A: Policy Simulator에 접근하는 계정에 IAM 읽기 권한이 있어야 합니다. 루트 사용자 또는 관리자 IAM 사용자로 Policy Simulator에 접속하세요.

### Practice (연습 문제) — 기본 개념 확인

> **Practice 13.1**: IAM에 대한 설명으로 올바른 것은?
>
> A. IAM은 리전별 서비스이므로, 서울 리전에서 만든 사용자는 도쿄 리전에서 사용할 수 없다
> B. IAM 사용은 무료이지만, 일정 수 이상의 사용자를 만들면 추가 비용이 발생한다
> C. IAM은 글로벌 서비스로, 생성한 사용자와 정책은 모든 리전에서 동일하게 적용된다
> D. IAM은 AWS 콘솔에서만 사용할 수 있으며 API로는 접근할 수 없다
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: C
> **해설**: IAM은 글로벌 서비스(Global Service)로, 특정 리전에 종속되지 않습니다. IAM에서 생성한 사용자, 그룹, 역할, 정책은 모든 AWS 리전에서 동일하게 인식되고 적용됩니다. IAM 사용은 완전히 무료이며 사용자 수 제한에 따른 과금도 없습니다.
> </details>

> **Practice 13.2**: IAM 그룹에 대한 설명으로 **틀린** 것은?
>
> A. 하나의 IAM 사용자는 여러 그룹에 동시에 속할 수 있다
> B. IAM 그룹 안에 다른 IAM 그룹을 포함(중첩)할 수 있다
> C. 그룹에 정책을 연결하면 그룹 내 모든 사용자에게 해당 권한이 적용된다
> D. IAM 사용자는 어떤 그룹에도 속하지 않을 수 있다
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: B
> **해설**: IAM 그룹에는 사용자만 포함할 수 있으며, 그룹 안에 다른 그룹을 중첩(nesting)할 수 없습니다. 이것은 다른 디렉터리 서비스(예: Active Directory)와 다른 IAM 그룹의 특징입니다. A, C, D는 모두 올바른 설명입니다.
> </details>

> **Practice 13.3**: 다음 IAM 정책 JSON에서 빈칸에 들어갈 수 있는 값의 조합으로 올바른 것은?
>
> ```json
> {
>   "Version": "2012-10-17",
>   "Statement": [
>     {
>       "Effect": "______",
>       "Action": "s3:GetObject",
>       "Resource": "arn:aws:s3:::my-bucket/*"
>     }
>   ]
> }
> ```
>
> A. Allow, Deny, Permit
> B. Allow, Deny
> C. Grant, Revoke
> D. Yes, No
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: B
> **해설**: IAM 정책의 Effect 필드는 **"Allow"** 또는 **"Deny"** 두 가지 값만 사용할 수 있습니다. "Permit", "Grant", "Revoke" 등은 유효한 값이 아닙니다.
> </details>

> **Practice 13.4**: 한 IAM 사용자에게 정책 A에서 S3 모든 작업을 Allow하고, 정책 B에서 S3 삭제 작업을 Deny했습니다. 이 사용자가 S3 객체를 삭제하려고 하면 어떻게 되는가?
>
> A. Allow가 Deny보다 우선하므로 삭제가 허용된다
> B. 먼저 적용된 정책이 우선하므로 정책 적용 순서에 따라 달라진다
> C. 명시적 Deny가 항상 Allow를 이기므로 삭제가 거부된다
> D. 충돌이 발생하므로 AWS가 사용자에게 선택을 요청한다
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: C
> **해설**: IAM 정책 평가에서 가장 중요한 규칙은 "명시적 Deny가 항상 Allow를 이긴다(Explicit Deny Always Wins)"입니다. 어떤 정책에서 Allow가 있더라도, 다른 정책에서 같은 작업에 대해 Deny가 있으면 최종 결과는 항상 거부입니다.
> </details>

> **Practice 13.5**: IAM 역할(Role)에 대한 설명으로 올바른 것은?
>
> A. IAM 역할에는 비밀번호와 접근 키가 영구적으로 할당된다
> B. IAM 역할은 사용자만 맡을 수 있으며, AWS 서비스는 맡을 수 없다

>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: C
> **해설**: IAM 역할을 맡으면(assume) AWS STS(Security Token Service)가 임시 보안 자격 증명을 발급합니다. 역할에는 영구적인 비밀번호나 접근 키가 없으며(A 틀림), AWS 서비스(EC2, Lambda 등)도 역할을 맡을 수 있고(B 틀림), 하나의 EC2에는 하나의 역할만 연결 가능합니다(D 틀림).
> </details>

> **Practice 13.6**: MFA(Multi-Factor Authentication)에 대한 설명으로 올바른 것은?

> B. MFA는 "알고 있는 것(비밀번호)"과 "가지고 있는 것(MFA 장치)"을 결합한 인증 방식이다
> C. MFA는 IAM 사용자에게만 적용 가능하며 루트 사용자에는 적용할 수 없다

>

> <summary>정답 및 해설</summary>
>
> **정답**: B
> **해설**: MFA는 비밀번호(알고 있는 것) 외에 MFA 장치(가지고 있는 것)를 추가로 요구하는 다중 인증 방식입니다. 비밀번호를 대체하는 것이 아니라 보완합니다(A 틀림). 루트 사용자에도 적용 가능하며 오히려 반드시 적용해야 합니다(C 틀림). MFA는 인증을 강화할 뿐 인가(정책)를 대체하지 않습니다(D 틀림).
> </details>

> **Practice 13.7**: IAM Credential Report에서 확인할 수 **없는** 정보는?
>
> A. 각 사용자의 MFA 활성화 여부
> B. 각 사용자의 접근 키 마지막 사용 날짜

> D. 각 사용자의 비밀번호 마지막 변경 날짜
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: C
> **해설**: 각 사용자가 마지막으로 접근한 AWS 서비스 목록은 **IAM Access Advisor**에서 확인합니다. Credential Report는 사용자의 자격 증명 상태(MFA 여부, 접근 키 상태, 비밀번호 상태 등)를 보여주며, 서비스별 접근 기록은 포함하지 않습니다.
> </details>

> 1. EC2 인스턴스 목록 조회 (`ec2:DescribeInstances`)
> 2. EC2 인스턴스 시작 (`ec2:StartInstances`)
> 3. EC2 인스턴스 종료(terminate) (`ec2:TerminateInstances`)
> 4. `project-alpha` 버킷에 파일 업로드 (`s3:PutObject`)
> 5. `project-alpha` 버킷에서 파일 삭제 (`s3:DeleteObject`)
> 6. `project-beta` 버킷에서 파일 다운로드 (`s3:GetObject`)
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 작업에 대해 (1) 명시적 Deny가 있는지, (2) 명시적 Allow가 있는지, (3) 둘 다 없으면 묵시적 Deny 순서로 평가합니다.
>
> **풀이 과정**:
>
> 1. **ec2:DescribeInstances → ✅ 허용**
>    - Statement 1의 `ec2:Describe*`에 매칭 → Allow. Deny 없음 → 최종 허용
>
> 2. **ec2:StartInstances → ✅ 허용**
>    - Statement 1에서 명시적으로 Allow → Deny 없음 → 최종 허용
>
> 3. **ec2:TerminateInstances → ❌ 거부 (묵시적 Deny)**
>    - 어떤 Statement에서도 Allow되지 않음. `Describe*`, `StartInstances`, `StopInstances`만 허용되며, `TerminateInstances`는 포함되지 않음 → 묵시적 거부
>
> 4. **s3:PutObject (project-alpha) → ✅ 허용**
>    - Statement 2의 `s3:*`에 매칭, 리소스도 `project-alpha/*`에 매칭 → Allow
>    - Statement 3의 `s3:Delete*`에 PutObject는 매칭되지 않음 → Deny 없음 → 최종 허용
>
> 5. **s3:DeleteObject (project-alpha) → ❌ 거부 (명시적 Deny)**
>    - Statement 2의 `s3:*`에 매칭 → Allow
>    - Statement 3의 `s3:Delete*`에 매칭, Resource `*` → **Deny**
>    - Deny가 Allow를 이김 → 최종 거부
>
> 6. **s3:GetObject (project-beta) → ❌ 거부 (묵시적 Deny)**
>    - Statement 2의 리소스가 `project-alpha`로 제한됨 → project-beta에는 Allow 없음
>    - Allow가 없으므로 → 묵시적 거부
>
> **보충 설명**: 5번이 핵심 포인트입니다. Statement 2에서 `s3:*`(모든 S3 작업)을 허용했지만, Statement 3에서 `s3:Delete*`을 명시적으로 Deny했으므로 삭제만 차단됩니다. 이 패턴은 "넓게 허용하되 특정 위험 작업만 차단"하는 일반적인 보안 패턴입니다.
> </details>

> **Exercise 13.2**: 당신은 50명 규모 회사의 AWS 관리자입니다. 다음 조건을 만족하는 IAM 구성을 설계하세요.

> - QA팀 (10명): EC2, S3에 대한 읽기 전용 권한만 필요
> - 재무팀 (5명): AWS 비용 관련 정보만 조회 가능해야 함

> **Exercise 13.3**: 다음 상황에서 IAM 사용자와 IAM 역할 중 어떤 것을 사용해야 하는지 판단하고 이유를 설명하세요.
>
> 1. 회사 직원 김 씨가 매일 AWS 콘솔에 로그인하여 작업합니다
> 2. EC2 인스턴스에서 실행 중인 애플리케이션이 DynamoDB에 데이터를 써야 합니다
> 3. 외부 감사 업체가 분기마다 한 번 AWS 리소스를 점검합니다
> 4. Lambda 함수가 S3 버킷에서 파일을 읽어야 합니다
> 5. 다른 AWS 계정의 개발자가 이 계정의 특정 S3 버킷에 접근해야 합니다
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: "장기적이고 개인에게 고정된 접근"이면 사용자, "임시적이거나 서비스/외부 개체의 접근"이면 역할을 사용합니다.
>
> **풀이 과정**:
>
> 1. **IAM 사용자** — 김 씨는 매일 AWS에 접근하는 고정된 개인입니다. 사용자 계정으로 콘솔 로그인 자격 증명(비밀번호 + MFA)을 부여합니다.
>
> 2. **IAM 역할** — EC2 인스턴스는 사람이 아닌 AWS 서비스입니다. 역할을 EC2에 연결하면 STS가 임시 자격 증명을 자동 발급합니다. 접근 키를 EC2에 저장하는 것은 보안 안티패턴입니다.
>
> 3. **IAM 역할** — 외부 감사 업체는 분기마다 한 번 일시적으로 접근합니다. 역할을 통해 임시 자격 증명을 발급하면, 감사 기간이 끝나면 자격 증명이 자동 만료됩니다. 매번 IAM 사용자를 만들고 삭제하는 것보다 효율적입니다.
>
> 4. **IAM 역할** — Lambda 함수는 AWS 서비스입니다. Lambda에 실행 역할(Execution Role)을 연결하여 S3 읽기 권한을 부여합니다. Lambda에는 접근 키를 설정할 수 없으므로 역할이 유일한 방법입니다.
>
> 5. **IAM 역할 (교차 계정)** — 이 계정에 IAM 역할을 만들고, 신뢰 정책에서 다른 계정을 허용합니다. 다른 계정의 개발자가 이 역할을 맡아 임시로 S3에 접근합니다.
>
> **정답**: 1번만 IAM 사용자, 나머지(2~5)는 모두 IAM 역할
>
> **보충 설명**: 일반 규칙 — **사람이 장기적으로 사용** → IAM 사용자, **서비스가 사용하거나 임시 접근** → IAM 역할. 사람이라도 외부 인원의 임시 접근은 역할이 적합합니다.
> </details>

> **Exercise 13.4**: IAM Identity Center와 기본 IAM의 차이를 설명하고, 다음 상황에서 어떤 것을 사용하는 것이 적합한지 판단하세요.
>
> 상황 A: AWS 계정이 1개인 소규모 스타트업, 직원 5명

>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 계정 수, 사용자 규모, 기존 인프라를 기준으로 판단합니다.
>
> **풀이 과정**:
>
> **IAM vs IAM Identity Center 비교**:
>
> | 항목 | IAM | IAM Identity Center |
> |------|-----|---------------------|
> | 범위 | 단일 AWS 계정 내 | 여러 AWS 계정 + 애플리케이션 |
> | 사용자 관리 | 계정별 개별 관리 | 중앙 집중 관리 |
> | 로그인 | 계정별 별도 로그인 | 한 번 로그인(SSO) |

> **상황 A**: **기본 IAM이 적합**

> - 직원 5명으로 관리 부담이 적음
> - IAM Identity Center를 설정하는 것이 오버엔지니어링
>
> **상황 B**: **IAM Identity Center가 적합**
> - 200명 × 15개 계정 = 최대 3,000개의 IAM 사용자를 관리해야 하는 문제 회피
> - 기존 Active Directory와 연동하여 직원이 기존 자격 증명으로 AWS에 접근 가능

> - 직원 퇴사 시 AD에서 비활성화하면 모든 AWS 계정 접근 즉시 차단
>
> **보충 설명**: 상황 A의 스타트업도 성장하여 계정이 늘어나면 IAM Identity Center로 전환하는 것이 좋습니다. 초기 설계 시 향후 확장을 고려하되, 현재 규모에 맞는 도구를 사용하는 것이 실용적입니다.
> </details>

### Problem (심화 문제) — 깊이 있는 사고와 창의적 문제 해결

> **Problem 13.1**: 당신은 금융 서비스 회사의 AWS 보안 아키텍트입니다. 다음 요구사항을 모두 만족하는 IAM 전략을 설계하세요.
>
> **요구사항**:
> - 고객 데이터가 저장된 S3 버킷(`customer-data-prod`)은 특정 팀(Data팀 3명)만 접근 가능
> - Data팀 3명 중에서도 삭제 권한은 팀장 1명에게만 부여
> - 모든 접근은 MFA 인증을 거친 후에만 허용
> - 외부 감사 시 감사관이 해당 버킷의 객체를 읽을 수 있되, 다운로드(GetObject)는 불가하고 목록 조회(ListBucket)만 가능해야 함
> - 이 모든 구성은 단일 AWS 계정 내에서 이루어짐
>
> 그룹, 사용자, 역할, 정책을 어떻게 조합할지 설계하고, 핵심 정책의 JSON 구조를 작성하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 다층적 권한 제어 — 그룹 정책으로 기본 접근, 개인 정책으로 예외 차단, 역할로 외부 접근, 정책 조건으로 MFA 강제.
>
> **풀이 전략**:
>
> **1. 그룹과 사용자 구성**
> - `DataTeam` 그룹 생성 → Data팀 3명 추가
> - 팀장을 포함한 3명 모두 그룹에 소속
>
> **2. 정책 설계**
>
> **(a) DataTeam 그룹 정책** — S3 읽기/쓰기 허용 + MFA 조건:
> ```json
> {
>   "Version": "2012-10-17",
>   "Statement": [
>     {
>       "Sid": "AllowS3AccessWithMFA",
>       "Effect": "Allow",
>       "Action": [
>         "s3:GetObject",
>         "s3:PutObject",
>         "s3:ListBucket"
>       ],
>       "Resource": [
>         "arn:aws:s3:::customer-data-prod",
>         "arn:aws:s3:::customer-data-prod/*"
>       ],
>       "Condition": {
>         "Bool": {
>           "aws:MultiFactorAuthPresent": "true"
>         }
>       }
>     }
>   ]
> }
> ```
>
> **(b) DataTeam 그룹의 삭제 차단 정책** — 팀원 전체에게 삭제 Deny:
> ```json
> {
>   "Version": "2012-10-17",
>   "Statement": [
>     {
>       "Sid": "DenyDeleteForAll",
>       "Effect": "Deny",
>       "Action": "s3:DeleteObject",
>       "Resource": "arn:aws:s3:::customer-data-prod/*"
>     }
>   ]
> }
> ```
>
> **(c) 팀장 개인에게 인라인 정책** — 삭제 허용 (그룹의 Deny를 우회할 수 없으므로 설계 변경 필요!):
>
> ⚠️ **여기서 핵심 통찰이 필요합니다**: 그룹 정책에서 Deny를 걸면 팀장 개인에게 Allow를 해도 **Deny가 이깁니다**. 따라서 설계를 바꿔야 합니다.
>
> **수정된 전략**: 그룹에서 삭제를 Deny하지 않고, 팀장이 아닌 2명에게만 삭제 Deny 정책을 개별 부여합니다.
>
> - DataTeam 그룹 정책: S3 전체 작업 Allow (MFA 조건 포함, 삭제 포함)
> - 팀원 2명에게 각각: `s3:DeleteObject` Deny 인라인 정책 연결
> - 팀장: 추가 정책 없음 → 그룹 정책의 Allow가 유효 → 삭제 가능
>
> **3. 외부 감사용 IAM 역할**
> - `AuditorRole` IAM 역할 생성
> - 신뢰 정책: 감사 업체의 AWS 계정 또는 IAM Identity Center 연동
> - 권한 정책: `s3:ListBucket`만 Allow, `GetObject` 없음
>
> ```json
> {
>   "Version": "2012-10-17",
>   "Statement": [
>     {
>       "Effect": "Allow",
>       "Action": "s3:ListBucket",
>       "Resource": "arn:aws:s3:::customer-data-prod"
>     }
>   ]
> }
> ```
>
> **확장 생각**: 이 문제에서 가장 중요한 통찰은 **"Deny는 항상 이긴다"는 규칙이 설계를 제약한다**는 점입니다. 그룹 전체에 Deny를 걸고 특정 사용자만 예외로 두는 것은 불가능합니다. 대신, 차단이 필요한 사용자에게만 개별적으로 Deny를 부여해야 합니다. 실제 현장에서 가장 흔한 IAM 설계 실수 중 하나입니다. 또한, `Condition`을 사용한 MFA 강제는 Chapter 12에서 배운 심층 방어 원칙의 구체적 적용입니다.
> </details>

> 3. 예상되는 비용 피해는 얼마인가요? (발견까지 약 24시간 경과 가정)
> 4. 향후 이런 사고를 방지하기 위한 IAM 차원의 대책을 3가지 이상 제시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 접근 키 유출로 인한 무단 리소스 생성 — 암호화폐 채굴 공격의 전형적 패턴
>
> **풀이 전략**:
>
> **1. 가능 원인**:
> - **접근 키 유출이 가장 유력합니다.** 개발자 A의 접근 키가 외부에 노출(GitHub 커밋, 로컬 환경 해킹, 피싱 등)되었을 가능성이 높습니다.
> - 근거: 평소 사용하지 않는 리전/인스턴스 유형, 새벽 시간대, GPU 인스턴스(암호화폐 채굴에 자주 사용) → 전형적인 크립토마이닝 공격 패턴
> - 6개월간 미교체된 접근 키는 유출 위험이 크게 증가합니다
>
> **2. 즉시 조치 (30분 이내)**:
> 1. 개발자 A의 접근 키 **즉시 비활성화** (IAM → 사용자 → 보안 자격 증명 → 접근 키 → 비활성화)
> 2. 세 리전(`us-east-1`, `eu-west-1`, `ap-southeast-1`)의 무단 생성된 EC2 인스턴스 60개 **즉시 종료(Terminate)**
> 3. CloudTrail에서 해당 접근 키로 수행된 **모든 작업** 검색 — EC2 외에 다른 리소스(S3 데이터 유출, IAM 백도어 생성 등)가 없는지 확인
> 4. 개발자 A에게 연락하여 접근 키 사용 내역 및 유출 경위 확인
>
> **3. 예상 비용 피해**:
> - `p3.2xlarge` 인스턴스: 시간당 약 $3.06 (온디맨드, us-east-1 기준)
> - 60개 인스턴스 × $3.06/시간 × 24시간 = **약 $4,406.40**
> - 추가로 네트워크 전송, EBS 볼륨 비용 등이 발생할 수 있음
> - AWS에 사고를 보고하면 비용 감면을 받을 수 있는 경우도 있으나 보장은 아님
>
> **4. 재발 방지 IAM 대책**:
> 1. **접근 키 정기 교체 의무화**: 90일마다 접근 키를 교체하도록 정책 수립 및 자동화
> 2. **IAM 역할로 전환**: 가능한 곳(EC2, Lambda 등)에서 접근 키 대신 IAM 역할 사용
> 3. **MFA 필수화**: 중요 작업(EC2 생성 등)에 MFA 조건을 정책에 추가
> 4. **접근 키 사용 모니터링**: CloudTrail + CloudWatch 알림으로 비정상 패턴(새벽 시간, 낯선 리전, 대량 생성) 감지
> 5. **리전 제한**: SCP(Service Control Policy)로 사용하지 않는 리전에서의 리소스 생성 차단
> 6. **git-secrets 등 사전 차단 도구**: 코드 커밋 시 접근 키가 포함되어 있으면 커밋을 차단
>
> **확장 생각**: 이 사고는 Chapter 12에서 배운 공동 책임 모델에서 **"클라우드에서의 보안"은 고객 책임**이라는 점을 극명하게 보여줍니다. AWS는 인프라를 안전하게 제공하지만, 접근 키 관리는 고객의 몫입니다. 또한, Chapter 15에서 배울 Amazon GuardDuty 같은 서비스가 이런 비정상 활동을 자동으로 탐지하는 데 도움이 됩니다.
> </details>

> **Problem 13.3**: IAM Identity Center, IAM 역할, AWS RAM의 세 가지 메커니즘을 비교하고, 다음 시나리오에서 각각 어떤 것을 사용하는 것이 가장 적합한지 판단하세요. 선택의 근거를 최소 권한 원칙과 운영 효율성 관점에서 설명하세요.
>
> **시나리오**: 한 기업이 다음 5개의 AWS 계정을 운영합니다.
> - Network 계정: 전사 VPC와 서브넷 관리
> - Dev 계정: 개발 환경
> - Staging 계정: 스테이징 환경
> - Prod 계정: 운영 환경
> - Security 계정: 보안 로그 중앙 관리
>
> 요구사항:
> (a) 개발자 30명이 Dev, Staging 계정에 접근하고, 그 중 선임 5명만 Prod 계정에 읽기 접근
> (b) Dev, Staging, Prod 계정의 EC2 인스턴스가 모두 Network 계정의 VPC 서브넷에 배치
> (c) 보안팀이 Security 계정에서 Prod 계정의 CloudTrail 로그를 분석
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 각 메커니즘의 강점에 맞춰 적재적소에 배치
>
> **풀이 전략**:
>
> **(a) 개발자의 다중 계정 접근 → IAM Identity Center**
>
> - **이유**: 30명의 개발자가 여러 계정에 접근하는 "사람의 다중 계정 SSO" 시나리오
> - **구성**:
>   - Identity Center에 개발자 30명 등록 (또는 기존 AD 연동)
>   - "Developer-FullAccess" 권한 세트 → Dev, Staging 계정에 할당 (30명 전원)
>   - "Developer-ReadOnly" 권한 세트 → Prod 계정에 할당 (선임 5명만)
> - **최소 권한 관점**: Prod에는 읽기만 허용, 선임만 접근 가능
> - **운영 효율성**: 한 곳에서 30명의 접근을 관리, 퇴사 시 즉시 전체 차단
> - IAM 역할로도 가능하지만, 30명 × 3개 계정의 역할 전환을 개별 관리하는 것은 비효율적
>
> **(b) VPC 서브넷 공유 → AWS RAM**
>
> - **이유**: Network 계정의 VPC 서브넷을 다른 계정이 "사용"하는 "리소스 공유" 시나리오
> - **구성**:
>   - Network 계정에서 RAM을 통해 서브넷을 Dev, Staging, Prod 계정과 공유
>   - 각 계정에서 EC2 인스턴스 생성 시 공유된 서브넷을 선택
> - **최소 권한 관점**: 각 계정은 서브넷을 사용할 수 있지만, 네트워크 설정(라우팅, NACL 등)은 변경 불가
> - **운영 효율성**: 네트워크 설계를 Network 계정에서 중앙 관리, 일관된 네트워크 아키텍처 유지
> - IAM 역할로는 "서브넷을 자기 계정에서 직접 사용"하는 것이 불가능 (역할 전환으로 Network 계정에서 작업해야 함)
>
> **(c) 보안팀의 로그 분석 → IAM 역할 (교차 계정)**
>
> - **이유**: Security 계정의 보안팀이 Prod 계정의 "특정 서비스에 접근"하는 시나리오
> - **구성**:
>   - Prod 계정에 `SecurityAuditRole` IAM 역할 생성
>   - 신뢰 정책: Security 계정의 보안팀 사용자가 역할을 맡을 수 있도록 설정
>   - 권한 정책: CloudTrail 읽기 + 관련 S3 버킷 읽기만 허용
>   - 보안팀이 필요 시 역할 전환하여 Prod 계정의 로그 분석
> - **최소 권한 관점**: CloudTrail 읽기만 허용, Prod의 다른 리소스에는 접근 불가
> - **운영 효율성**: 보안팀이 필요할 때만 역할 전환하여 사용, 임시 자격 증명으로 보안 강화
> - IAM Identity Center로도 가능하지만, 이미 Identity Center를 (a)에서 사용 중이라면 보안팀의 접근을 별도 역할로 분리하여 감사 추적을 명확히 하는 것이 보안 관점에서 유리
> - RAM은 이 경우 적합하지 않음 (CloudTrail은 RAM으로 공유 가능한 리소스 유형이 아님)
>
> **확장 생각**: 실제 기업에서는 세 가지 메커니즘을 동시에 사용하는 것이 일반적입니다. 핵심은 "사람의 다중 계정 접근 = Identity Center", "리소스 공유 = RAM", "서비스 간 또는 특수 접근 = IAM 역할"이라는 기본 원칙을 이해하고, 상황에 맞게 조합하는 것입니다.
> </details>

## 다음 단계 안내 (What's Next)

이 Chapter에서 우리는 AWS에서 "누가, 무엇을 할 수 있는가"를 관리하는 IAM의 핵심을 배웠습니다. 접근 제어의 **자격 증명(Identity)** 측면에 집중했다면, 다음 **Chapter 14: 네트워크 보안과 데이터 보호**에서는 **네트워크 수준의 접근 제어**를 다룹니다. 보안 그룹과 네트워크 ACL로 트래픽을 제어하고, 데이터를 암호화하여 보호하는 방법을 배우게 됩니다. IAM 정책이 "이 사람이 이 작업을 할 수 있는가?"를 결정한다면, 네트워크 보안은 "이 트래픽이 이 서버에 도달할 수 있는가?"를 결정합니다 — 두 계층이 결합되어 AWS의 보안 아키텍처를 완성합니다.
