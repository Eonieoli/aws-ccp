## 12.1 공동 책임 모델 개요 — "클라우드의 보안"과 "클라우드에서의 보안" 구분

> 💡 **한 줄 요약**: AWS는 클라우드 인프라 자체의 보안을 책임지고, 고객은 클라우드 위에 올려놓은 것들의 보안을 책임진다.

### 직관적 이해 (Intuitive Understanding)

아파트에 입주한다고 상상해 봅시다. 건물의 구조적 안전(내진 설계, 소방 시설, 공용 출입 통제)은 **건설사와 관리사무소**가 책임집니다. 하지만 여러분의 집 안에 있는 물건을 잘 보관하고, 현관문을 잠그고, 가스 밸브를 잠그는 것은 **입주자인 여러분**의 책임입니다.

아무리 건물이 튼튼해도 현관문을 열어 놓으면 도둑이 들 수 있고, 반대로 여러분이 아무리 조심해도 건물 자체가 부실하면 안전하지 않습니다. **양쪽 모두가 자기 역할을 해야** 비로소 안전한 거주 환경이 만들어집니다.

AWS 클라우드도 정확히 같은 구조입니다. "클라우드를 사용하면 보안은 AWS가 알아서 해주겠지?"라고 생각하기 쉽지만, 현실은 그렇지 않습니다. AWS와 고객이 **각자의 영역에서 보안 책임을 나누어 지는** 것이 바로 **공동 책임 모델(Shared Responsibility Model)**입니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **공동 책임 모델(Shared Responsibility Model)**: AWS와 고객이 보안과 규정 준수에 대한 책임을 나누어 지는 프레임워크. AWS는 "클라우드 **의** 보안(Security **of** the Cloud)"을, 고객은 "클라우드 **에서의** 보안(Security **in** the Cloud)"을 책임진다.

이 모델의 핵심은 두 가지 영어 전치사 — **of**와 **in** — 의 차이에 있습니다:

**"Security of the Cloud" (클라우드의 보안)** — AWS의 책임

AWS가 제공하는 인프라 자체를 보호하는 것입니다. 데이터센터 건물, 서버 하드웨어, 네트워크 장비, 가상화 소프트웨어(하이퍼바이저) 등이 여기에 해당합니다. 고객은 이 영역에 접근할 수 없고, 접근할 필요도 없습니다.

**"Security in the Cloud" (클라우드에서의 보안)** — 고객의 책임

고객이 AWS 위에 올려놓은 모든 것을 보호하는 것입니다. 데이터, 운영체제 설정, 네트워크 방화벽 규칙, 사용자 계정과 비밀번호 관리 등이 여기에 해당합니다.

이것을 시각적으로 표현하면 다음과 같습니다:

```
┌─────────────────────────────────────────────────┐
│              고객의 책임 (Customer)                │
│  "클라우드에서의 보안" (Security IN the Cloud)      │
│                                                 │
│  ┌─────────────────────────────────────────────┐│
│  │ 고객 데이터                                   ││
│  ├─────────────────────────────────────────────┤│
│  │ 플랫폼, 애플리케이션, 접근 관리 (IAM)          ││
│  ├─────────────────────────────────────────────┤│
│  │ 운영체제, 네트워크, 방화벽 구성                 ││
│  ├─────────────────────────────────────────────┤│
│  │ 클라이언트 측 데이터 암호화 / 서버 측 데이터     ││
│  │ 암호화 / 네트워크 트래픽 보호                   ││
│  └─────────────────────────────────────────────┘│
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│               AWS의 책임 (AWS)                    │
│  "클라우드의 보안" (Security OF the Cloud)         │
│                                                 │
│  ┌─────────────────────────────────────────────┐│
│  │ 소프트웨어: 컴퓨팅, 스토리지, DB, 네트워킹      ││
│  ├─────────────────────────────────────────────┤│
│  │ 하드웨어 / AWS 글로벌 인프라                    ││
│  ├─────────────────────────────────────────────┤│
│  │ 리전, 가용 영역, 엣지 로케이션                  ││
│  └─────────────────────────────────────────────┘│
└─────────────────────────────────────────────────┘
```

> 🔗 **연결**: Chapter 3에서 배운 리전(Region), 가용 영역(Availability Zone), 엣지 로케이션(Edge Location) — 이러한 AWS 글로벌 인프라의 물리적 보안은 전적으로 AWS의 책임입니다.

#### 상속된 제어와 공유된 제어

공동 책임 모델을 더 세밀하게 이해하기 위해, AWS는 제어(Controls)를 세 가지로 분류합니다:

> 📝 **상속된 제어(Inherited Controls)**: 고객이 AWS로부터 자동으로 물려받는 보안 제어. 데이터센터의 물리적 보안, 환경 제어(냉방, 전력) 등이 해당한다. 고객이 별도로 구현할 필요가 없다.

> 📝 **공유된 제어(Shared Controls)**: AWS와 고객이 각자의 계층에서 함께 적용해야 하는 보안 제어. 예를 들어 패치 관리 — AWS는 인프라의 패치를, 고객은 자신의 OS와 애플리케이션의 패치를 책임진다.

> 📝 **고객 전용 제어(Customer-Specific Controls)**: 전적으로 고객의 책임인 제어. 고객 데이터의 암호화 방식 선택, 서비스 내 접근 권한 설정 등이 해당한다.

| 제어 유형 | 설명 | 예시 |
|-----------|------|------|
| 상속된 제어 | AWS가 전적으로 책임 → 고객이 자동 혜택 | 물리적 보안, 환경 제어 |
| 공유된 제어 | 양쪽이 각자의 계층에서 적용 | 패치 관리, 구성 관리, 교육 |
| 고객 전용 제어 | 고객이 전적으로 책임 | 데이터 분류, 접근 권한 설정 |

### 상세 예시 (Detailed Examples)

> 🔍 **예시 12.1.1: 아파트 비유로 이해하는 공동 책임**
>
> **상황**: 당신은 아파트 15층에 거주하고 있습니다.
> **풀이/설명**:
> | 보안 항목 | 아파트 비유 | AWS 클라우드 |
> |-----------|------------|-------------|
> | 건물 구조 안전 | 건설사/관리사무소 | AWS (데이터센터 물리 보안) |
> | 공용 출입문 | 관리사무소 | AWS (네트워크 인프라 보안) |
> | 내 집 현관문 잠금 | 입주자 | 고객 (접근 관리, 비밀번호) |
> | 내 집 귀중품 보관 | 입주자 | 고객 (데이터 암호화) |
> | 화재 경보기 | 관리사무소가 설치, 입주자가 테스트 | 공유된 제어 (모니터링) |
>
> **핵심 포인트**: 건물이 아무리 안전해도 현관문을 안 잠그면 위험하고, 현관문을 잠가도 건물이 무너지면 소용없습니다. AWS와 고객의 보안도 마찬가지입니다.

> 🔍 **예시 12.1.2: 실제 보안 사고로 보는 공동 책임**
>
> **상황**: 한 기업이 AWS에서 운영하는 웹 서버의 고객 데이터가 유출되었습니다. 조사 결과, 원인은 S3 버킷의 접근 권한이 "퍼블릭(누구나 접근 가능)"으로 설정되어 있었기 때문이었습니다.
> **풀이/설명**:
> - AWS의 책임: S3 서비스 자체가 안전하게 동작하도록 보장 → **AWS는 이 책임을 다했습니다**
> - 고객의 책임: S3 버킷의 접근 권한을 적절하게 설정 → **고객이 이 책임을 이행하지 않았습니다**
> - 결론: 이 데이터 유출은 **고객의 보안 책임** 영역에서 발생한 문제입니다
>
> **핵심 포인트**: 공동 책임 모델을 이해하면, 보안 사고 발생 시 책임 소재를 명확히 파악할 수 있습니다. AWS의 서비스가 정상 동작했더라도, 고객의 설정 실수로 보안 사고가 발생할 수 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "클라우드에 올리면 AWS가 보안을 알아서 해준다"는 가장 위험한 오해입니다. AWS는 인프라의 보안만 책임지며, 고객 데이터의 보안과 접근 제어는 전적으로 고객의 몫입니다. 특히 S3 버킷 공개 설정, 보안 그룹의 과도한 허용, 루트 계정의 일상 사용 등은 고객 측 책임 영역에서 가장 자주 발생하는 보안 실수입니다.

> ⚠️ **주의**: CCP 시험에서 "이 보안 항목은 누구의 책임인가?"라는 질문이 자주 출제됩니다. **"of"와 "in"의 차이**를 확실히 기억하세요. 물리적 인프라와 기반 소프트웨어 = AWS 책임, 그 위에 올려놓는 모든 것 = 고객 책임입니다.

### Section 요약 (Section Summary)

- **공동 책임 모델**은 AWS와 고객이 보안 책임을 나누는 프레임워크이다
- AWS는 "클라우드**의** 보안" (인프라 자체)을, 고객은 "클라우드**에서의** 보안" (인프라 위의 모든 것)을 책임진다
- 제어는 상속된 제어, 공유된 제어, 고객 전용 제어로 분류된다
- 양쪽 모두 자기 영역의 책임을 다해야 비로소 안전한 환경이 만들어진다

## 12.2 AWS의 보안 책임 — 물리적 인프라, 하이퍼바이저, 관리형 서비스 보안

> 💡 **한 줄 요약**: AWS는 데이터센터 건물부터 하이퍼바이저까지, 고객이 접근할 수 없는 모든 인프라 계층의 보안을 책임진다.

### 직관적 이해 (Intuitive Understanding)

은행의 금고를 떠올려 봅시다. 고객은 자신의 금고 칸에 무엇을 보관할지 결정하고 자물쇠 비밀번호를 관리하지만, **금고실 자체의 물리적 보안** — 두꺼운 벽, 감시 카메라, 경비원, 화재 진압 시스템 — 은 은행이 책임집니다. 고객이 금고실 벽의 두께를 확인하거나 경비 시스템을 점검할 필요는 없습니다. 은행이 그것을 보장하기 때문입니다.

AWS의 보안 책임도 이와 같습니다. AWS는 전 세계에 분포한 데이터센터의 물리적 보안, 서버 하드웨어, 네트워크 인프라, 가상화 계층(하이퍼바이저) 등 **고객이 볼 수도, 만질 수도 없는 영역**의 보안을 책임집니다.

왜 이런 구조가 필요할까요? 온프레미스(자체 데이터센터)에서는 기업이 서버실 잠금장치부터 냉방 시스템까지 직접 관리해야 했습니다. 이것은 엄청난 비용과 전문 인력을 필요로 합니다. AWS를 사용하면 이 부담이 AWS로 이전됩니다 — 이것이 바로 **상속된 제어**의 핵심 혜택입니다.

### 핵심 개념 설명 (Core Explanation)

AWS의 보안 책임은 크게 네 가지 계층으로 나눌 수 있습니다:

#### 계층 1: 물리적 보안 (Physical Security)

AWS 데이터센터는 세계에서 가장 엄격한 물리적 보안 시설 중 하나입니다:

- **위치 비공개**: 데이터센터의 정확한 위치는 공개되지 않습니다. 외부에서 식별할 수 없도록 설계되어 있습니다.
- **다중 인증 출입 통제**: 직원이라도 데이터센터에 들어가려면 여러 단계의 신원 확인과 보안 검사를 통과해야 합니다. 생체 인식(지문, 홍채), 배지, PIN 등 다중 인증 방식을 사용합니다.
- **24/7 감시**: 전문 보안 요원과 감시 카메라가 365일 24시간 데이터센터를 모니터링합니다.
- **접근 기록**: 누가 언제 어디에 접근했는지 모든 기록이 남습니다.

#### 계층 2: 환경 제어 (Environmental Controls)

- **전력**: 이중화된 전력 공급 장치(UPS)와 백업 발전기로 정전에 대비합니다.
- **냉각**: 서버가 과열되지 않도록 정밀한 온도·습도 관리 시스템을 운영합니다.
- **화재 방지**: 자동 화재 감지 및 진압 시스템이 설치되어 있습니다.

> 🔗 **연결**: Chapter 3에서 배운 가용 영역(AZ)은 물리적으로 분리된 데이터센터(군)입니다. 각 AZ는 독립적인 전력, 냉각, 네트워크를 갖추고 있어, 한 곳에 문제가 생겨도 다른 곳은 영향을 받지 않습니다. 이 물리적 분리 자체가 AWS의 보안 책임에 포함됩니다.

#### 계층 3: 네트워크 인프라 보안

- **물리적 네트워크 장비**: 라우터, 스위치 등 네트워크 장비의 보안 및 유지보수
- **AWS 백본 네트워크**: 리전 간, AZ 간 연결하는 전용 고속 네트워크의 보안
- **DDoS 방어 기본 계층**: AWS 인프라 수준에서의 대규모 네트워크 공격 방어

#### 계층 4: 가상화 계층 (Hypervisor)

> 📝 **하이퍼바이저(Hypervisor)**: 하나의 물리적 서버 위에 여러 개의 가상 머신(VM)을 만들어 실행할 수 있게 해주는 소프트웨어. AWS에서 EC2 인스턴스를 실행할 때, 실제로는 하이퍼바이저 위에서 가상 머신이 돌아가는 것이다.

- AWS는 자체 개발한 **Nitro System**이라는 하이퍼바이저를 사용합니다
- 고객 A의 가상 머신이 고객 B의 가상 머신에 접근하지 못하도록 **격리(Isolation)**를 보장합니다
- 이 가상화 계층의 보안은 전적으로 AWS가 관리하며, 고객은 접근할 수 없습니다

#### AWS는 이 책임을 어떻게 증명하는가?

고객이 AWS 데이터센터에 직접 방문하여 보안을 확인할 수는 없습니다. 대신, AWS는 다음과 같은 방법으로 자신의 보안 책임 이행을 증명합니다:

- **제3자 감사 보고서**: SOC 1/2/3, ISO 27001 등 국제적으로 인정받는 보안 인증을 획득하고 유지합니다
- **AWS Artifact**: 이러한 인증서와 감사 보고서를 고객이 직접 다운로드할 수 있는 서비스입니다
- **규정 준수 프로그램**: 다양한 산업별·국가별 규정 준수 인증을 보유합니다

> 💬 **참고**: AWS Artifact와 규정 준수 프로그램에 대해서는 Chapter 16에서 자세히 다룹니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 12.2.1: 데이터센터 방문 요청**
>
> **상황**: 당신의 회사 보안 담당자가 "AWS 데이터센터에 직접 가서 물리적 보안을 확인하고 싶다"고 합니다.
> **풀이/설명**:
> - AWS는 데이터센터의 물리적 방문을 허용하지 않습니다. 이는 보안을 위한 조치입니다.
> - 대신, AWS Artifact에서 SOC 보고서를 다운로드하여 제3자 감사 기관이 검증한 보안 상태를 확인할 수 있습니다.
> - 이 보고서에는 물리적 접근 통제, 환경 제어, 데이터 폐기 절차 등에 대한 상세한 감사 결과가 포함됩니다.
>
> **핵심 포인트**: AWS의 보안을 검증하는 방법은 직접 방문이 아니라, 국제 표준 인증과 제3자 감사 보고서를 통해서입니다.

> 🔍 **예시 12.2.2: 온프레미스 vs AWS의 보안 책임 비교**
>
> **상황**: 자체 데이터센터를 운영하던 기업이 AWS로 이전을 고려하면서, 보안 관점에서 무엇이 달라지는지 알고 싶어 합니다.
> **풀이/설명**:
> | 보안 항목 | 온프레미스 | AWS 이전 후 |
> |-----------|-----------|------------|
> | 서버실 물리 보안 | 기업이 직접 관리 | AWS 책임 (상속된 제어) |
> | 전력·냉각 시스템 | 기업이 직접 관리 | AWS 책임 (상속된 제어) |
> | 서버 하드웨어 교체 | 기업이 직접 수행 | AWS 책임 |
> | 네트워크 장비 관리 | 기업이 직접 관리 | AWS 책임 (물리 계층) |
> | OS 패치 적용 | 기업이 직접 수행 | **여전히 고객 책임** (EC2의 경우) |
> | 데이터 암호화 | 기업이 직접 수행 | **여전히 고객 책임** |
>
> **핵심 포인트**: AWS로 이전하면 물리적 인프라의 보안 부담은 사라지지만, 데이터와 설정에 대한 보안 책임은 여전히 고객에게 남습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "AWS가 물리적 보안을 책임진다"는 것이 "고객은 하드웨어에 대해 아무것도 알 필요 없다"는 뜻은 아닙니다. 예를 들어 EC2 인스턴스 유형을 선택할 때는 하드웨어 성능 특성을 이해해야 하지만, 그 하드웨어의 *보안과 유지보수*는 AWS가 합니다.

### Section 요약 (Section Summary)

- AWS는 데이터센터의 물리적 보안, 환경 제어, 네트워크 인프라, 가상화 계층(하이퍼바이저)의 보안을 책임진다
- AWS 데이터센터는 비공개 위치, 다중 인증 출입, 24/7 감시 등 엄격한 물리적 보안을 갖추고 있다
- 고객은 직접 방문 대신 AWS Artifact를 통해 제3자 감사 보고서와 인증서를 확인할 수 있다
- AWS로 이전하면 물리적 보안 부담은 줄어들지만, 데이터와 설정의 보안은 여전히 고객 책임이다

## 12.3 고객의 보안 책임 — 데이터, 운영체제, 네트워크 구성, 접근 관리

> 💡 **한 줄 요약**: AWS가 아무리 안전한 인프라를 제공해도, 고객이 데이터·계정·설정을 제대로 관리하지 않으면 보안 사고는 고객의 몫이다.

### 직관적 이해 (Intuitive Understanding)

다시 아파트 비유를 확장해 봅시다. 건물 관리사무소가 최고의 보안 시스템을 갖추었더라도, 다음과 같은 상황은 전적으로 입주자의 책임입니다:

- 현관문을 열어놓고 외출한 경우
- 현관 비밀번호를 "1234"로 설정한 경우
- 모르는 사람에게 집 열쇠를 빌려준 경우
- 중요한 서류를 창문 옆에 아무렇게나 둔 경우

AWS 클라우드에서 고객의 보안 책임도 이와 유사합니다. AWS가 안전한 "건물"을 제공하지만, 그 안에서 **문을 잠그고, 귀중품을 금고에 넣고, 열쇠를 잘 관리하는 것**은 고객이 해야 합니다.

실제로 공개적으로 알려진 클라우드 보안 사고의 대부분은 AWS 인프라의 취약점이 아니라, **고객의 설정 실수나 관리 소홀**에서 비롯됩니다. 이 Section에서는 고객이 정확히 무엇을 책임지는지 알아봅니다.

### 핵심 개념 설명 (Core Explanation)

고객의 보안 책임은 다음 다섯 가지 핵심 영역으로 나눌 수 있습니다:

#### 영역 1: 고객 데이터 (Customer Data)

이것이 가장 중요한 책임입니다. AWS에 저장하는 **모든 데이터의 보안은 전적으로 고객의 책임**입니다.

- **어떤 데이터를 저장할지** 결정하는 것
- **어디에 저장할지** (어느 리전에, 어떤 서비스에) 선택하는 것
- **누가 접근할 수 있는지** 제어하는 것
- **데이터를 암호화할지**, 한다면 어떤 방식으로 할지 결정하는 것

> 📝 **저장 시 암호화(Encryption at Rest)**: 저장 장치(디스크, S3 등)에 보관된 데이터를 암호화하여, 물리적 장치가 탈취되더라도 데이터를 읽을 수 없게 하는 것.

> 📝 **전송 중 암호화(Encryption in Transit)**: 네트워크를 통해 이동하는 데이터를 암호화하여, 중간에 가로채더라도 내용을 읽을 수 없게 하는 것. HTTPS가 대표적인 예이다.

> 💬 **참고**: 암호화의 구체적인 구현 방식과 AWS KMS(Key Management Service)에 대해서는 Chapter 14에서 자세히 다룹니다.

#### 영역 2: 플랫폼과 애플리케이션 관리

- EC2 인스턴스 위에 설치하는 **운영체제(OS)의 보안 패치** 적용
- 직접 설치한 **애플리케이션의 보안 업데이트**
- 애플리케이션 코드의 **보안 취약점** 관리

#### 영역 3: 접근 관리 (Identity and Access Management)

- **AWS 계정의 루트 사용자** 보호 (강력한 비밀번호, MFA 활성화)
- **IAM 사용자, 그룹, 역할**을 적절히 생성하고 권한을 부여하는 것
- 누가 어떤 리소스에 접근할 수 있는지 **정책(Policy)**을 정의하는 것

> 💬 **참고**: IAM(Identity and Access Management)은 Chapter 13에서 자세히 다룹니다. 여기서는 "접근 관리가 고객의 핵심 보안 책임"이라는 점만 기억해 두세요.

#### 영역 4: 운영체제 및 네트워크 구성

- EC2 인스턴스의 **운영체제 방화벽(iptables 등)** 설정
- **보안 그룹(Security Group)** 규칙 설정 — 어떤 트래픽을 허용/차단할지
- **네트워크 ACL(Access Control List)** 규칙 설정

> 💬 **참고**: 보안 그룹과 네트워크 ACL의 상세한 설정 방법은 Chapter 14에서 다룹니다.

#### 영역 5: 데이터 무결성과 백업

- 데이터의 **백업 전략** 수립 및 실행
- **데이터 복구 절차** 테스트
- 서비스 장애나 실수로 인한 데이터 손실에 대비하는 것

### 상세 예시 (Detailed Examples)

> 🔍 **예시 12.3.1: S3 버킷 공개 설정 — 가장 흔한 보안 실수**
>
> **상황**: 개발자가 테스트 목적으로 S3 버킷을 "퍼블릭(Public)"으로 설정했습니다. 테스트가 끝난 후 이 설정을 되돌리는 것을 깜빡했고, 몇 달 뒤 외부에서 해당 버킷의 데이터를 발견했습니다.
> **풀이/설명**:
> - S3 서비스 자체는 정상적으로 동작했습니다 — AWS의 책임 영역에서는 문제가 없습니다
> - "퍼블릭"으로 설정한 것은 **고객의 설정 행위** — 이것은 고객의 보안 책임입니다
> - AWS는 이런 실수를 방지하기 위해 S3 퍼블릭 액세스 차단 기능, 경고 표시 등을 제공하지만, 이를 활용할지는 고객이 결정합니다
>
> **핵심 포인트**: AWS는 도구를 제공하지만, 그 도구를 올바르게 사용하는 것은 고객의 책임입니다.

> 🔍 **예시 12.3.2: EC2 인스턴스 운영체제 패치**
>
> **상황**: EC2 인스턴스에서 리눅스 운영체제를 실행하고 있는데, 중요한 보안 패치가 공개되었습니다. 이 패치를 적용하는 것은 누구의 책임일까요?
> **풀이/설명**:
> - AWS의 책임: EC2가 실행되는 **물리 서버와 하이퍼바이저**의 보안 패치 → AWS가 수행
> - 고객의 책임: EC2 인스턴스 **안에서 실행되는 운영체제(리눅스)**의 보안 패치 → **고객이 수행**
> - 이것이 "공유된 제어" 중 **패치 관리**의 전형적인 예입니다
>
> **핵심 포인트**: 패치 관리는 공유된 제어입니다. AWS는 자신의 인프라를, 고객은 자신의 OS와 애플리케이션을 각각 패치합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "AWS가 제공하는 관리형 서비스를 쓰면 고객의 보안 책임이 완전히 사라진다"고 오해하기 쉽습니다. 관리형 서비스(예: RDS)를 사용하면 OS 패치 같은 일부 책임이 AWS로 이전되지만, **데이터 보호, 접근 권한 설정, 네트워크 구성** 등의 책임은 여전히 고객에게 남습니다. 이 차이는 다음 Section(12.4)에서 자세히 다룹니다.

> ⚠️ **주의**: AWS 루트 계정(Root Account)은 계정의 모든 권한을 가집니다. 루트 계정으로 일상 작업을 하는 것은 마치 집 마스터키를 일상적으로 들고 다니는 것과 같습니다. 루트 계정은 반드시 MFA(다중 인증)를 설정하고, 일상 작업에는 권한이 제한된 별도의 사용자를 사용해야 합니다.

### Section 요약 (Section Summary)

- 고객은 데이터, 접근 관리, OS 및 네트워크 구성, 애플리케이션 보안, 백업을 책임진다
- 데이터 보호(암호화, 접근 제어)는 고객의 가장 핵심적인 보안 책임이다
- 대부분의 클라우드 보안 사고는 AWS 인프라 결함이 아닌 고객의 설정 실수에서 발생한다
- AWS가 보안 도구를 제공하지만, 이를 올바르게 활용하는 것은 고객의 몫이다

## 12.4 서비스별 책임 변화 — EC2 vs RDS vs Lambda에서의 책임 경계 비교

> 💡 **한 줄 요약**: AWS 서비스의 추상화 수준이 높아질수록(IaaS → PaaS → 서버리스) 고객의 보안 책임 범위는 줄어들고, AWS의 책임 범위는 늘어난다.

### 직관적 이해 (Intuitive Understanding)

주거 형태를 세 가지로 비교해 봅시다:

1. **단독주택 (= EC2, IaaS)**: 집의 구조부터 보안 시스템, 잔디 관리까지 집주인이 모든 것을 직접 관리합니다. 자유도가 높지만 책임도 큽니다.
2. **아파트 (= RDS, PaaS)**: 건물 관리는 관리사무소가 하고, 입주자는 집 안의 인테리어와 물건 관리만 하면 됩니다. 자유도는 줄지만 편리합니다.
3. **호텔 (= Lambda, 서버리스)**: 방 청소, 보안, 시설 관리를 호텔이 거의 다 해줍니다. 투숙객은 자신의 짐(데이터)과 방 열쇠(접근 권한)만 관리하면 됩니다.

> 🔗 **연결**: Chapter 2에서 배운 IaaS, PaaS, SaaS 서비스 모델을 떠올려 봅시다. 서비스 모델의 추상화 수준이 높아질수록 고객이 관리해야 할 영역이 줄어들었던 것처럼, 보안 책임도 동일한 패턴을 따릅니다.

### 핵심 개념 설명 (Core Explanation)

AWS에서 제공하는 서비스의 유형에 따라, 공동 책임 모델의 경계선이 이동합니다. 이것을 세 가지 대표 서비스로 비교해 봅시다.

#### EC2 (IaaS) — 고객의 책임이 가장 큰 경우

EC2는 가상 서버를 제공하는 **IaaS** 서비스입니다. AWS는 물리적 하드웨어와 하이퍼바이저까지만 책임지고, 그 위의 모든 것은 고객이 관리합니다.

| 계층 | 책임자 |
|------|--------|
| 고객 데이터 | **고객** |
| 애플리케이션 | **고객** |
| 운영체제 (Linux, Windows) | **고객** |
| 네트워크 구성 (보안 그룹) | **고객** |
| 하이퍼바이저 | AWS |
| 물리적 서버/네트워크 | AWS |
| 데이터센터 시설 | AWS |

#### RDS (PaaS) — 책임이 일부 AWS로 이전되는 경우

RDS는 관리형 데이터베이스 서비스로, **PaaS** 성격을 가집니다. AWS가 운영체제와 데이터베이스 엔진의 패치·관리를 대신 수행합니다.

| 계층 | 책임자 |
|------|--------|
| 고객 데이터 | **고객** |
| 데이터베이스 내 접근 권한 설정 | **고객** |
| 네트워크 구성 (보안 그룹) | **고객** |
| 암호화 사용 여부 결정 | **고객** |
| 데이터베이스 엔진 패치 | **AWS** |
| 운영체제 패치 | **AWS** |
| 하이퍼바이저 이하 | AWS |

EC2와 비교하면, **운영체제와 DB 엔진 패치**가 고객에서 AWS로 이동한 것을 볼 수 있습니다.

#### Lambda (서버리스) — 고객의 책임이 가장 작은 경우

Lambda는 **서버리스** 서비스입니다. 고객은 코드만 작성하면 되고, 서버, 운영체제, 런타임 환경 등은 AWS가 관리합니다.

| 계층 | 책임자 |
|------|--------|
| 고객 데이터 | **고객** |
| 함수 코드의 보안 | **고객** |
| 접근 권한 설정 (IAM 역할) | **고객** |
| 런타임 환경 | **AWS** |
| 운영체제 | **AWS** |
| 실행 환경 격리 | **AWS** |
| 하이퍼바이저 이하 | AWS |

#### 세 서비스 책임 비교 요약

```
고객 책임 범위:  넓음 ◀─────────────────────────▶ 좁음

                EC2 (IaaS)    RDS (PaaS)    Lambda (서버리스)
                ──────────    ──────────    ────────────────
고객 데이터       고객           고객           고객
애플리케이션      고객           고객           고객 (함수 코드)
접근 관리        고객           고객           고객
네트워크 구성     고객           고객           고객 (일부)
운영체제         고객           AWS            AWS
패치 관리        고객           AWS            AWS
런타임 환경       고객           AWS            AWS
하이퍼바이저      AWS            AWS            AWS
물리적 인프라     AWS            AWS            AWS
```

핵심 관찰: **어떤 서비스를 사용하든 "고객 데이터"와 "접근 관리"는 항상 고객의 책임**입니다. 추상화 수준이 올라가면서 줄어드는 것은 인프라 운영 관련 책임이지, 데이터 보호 책임이 아닙니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 12.4.1: 같은 웹 애플리케이션, 다른 책임 범위**
>
> **상황**: 한 기업이 웹 애플리케이션을 운영하려 합니다. 아키텍트가 세 가지 방안을 검토합니다: (A) EC2에 직접 배포, (B) Elastic Beanstalk(PaaS) 사용, (C) Lambda + API Gateway(서버리스) 사용.
> **풀이/설명**:
> - **(A) EC2 직접 배포**: 고객이 OS 설치, 웹 서버 구성, 보안 패치, 방화벽 설정을 모두 직접 관리. 보안 인력과 전문성이 필요.
> - **(B) Elastic Beanstalk**: AWS가 OS와 플랫폼 관리를 대행. 고객은 애플리케이션 코드와 데이터에 집중. 보안 운영 부담 감소.
> - **(C) Lambda + API Gateway**: 서버 관리가 완전히 제거됨. 고객은 함수 코드의 보안과 데이터 접근 제어만 관리. 보안 운영 부담이 가장 작음.
>
> **핵심 포인트**: 보안 전문 인력이 부족한 소규모 팀이라면, 관리형 서비스나 서버리스를 활용하여 고객 측 보안 책임 범위를 줄이는 것이 현명한 전략입니다.

> 🔍 **예시 12.4.2: "보안 패치는 누가 하나요?" — 서비스별 답이 다르다**
>
> **상황**: 중요한 리눅스 커널 보안 취약점이 발견되었습니다. 긴급 패치를 적용해야 합니다. 세 서비스에서 각각 누가 이 패치를 적용해야 할까요?
> **풀이/설명**:
> - **EC2**: **고객**이 직접 SSH로 접속하여 OS 패치를 적용해야 합니다
> - **RDS**: **AWS**가 유지 관리 기간(Maintenance Window) 중에 자동으로 패치를 적용합니다. 고객은 유지 관리 기간을 설정할 수 있습니다.
> - **Lambda**: **AWS**가 실행 환경을 관리하므로 고객은 아무 조치를 할 필요가 없습니다
>
> **핵심 포인트**: 같은 "OS 패치"라는 작업도, 사용하는 서비스에 따라 책임자가 달라집니다. 이것이 공동 책임 모델의 "서비스별 책임 변화"입니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: S3의 특수한 책임 모델**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> S3는 흥미로운 사례입니다. S3는 객체 스토리지 서비스로, 고객이 운영체제나 서버를 직접 관리하지 않지만, 단순히 "서버리스"라고 분류하기에는 고객이 설정할 수 있는 보안 항목이 매우 많습니다:
>
> - **버킷 정책(Bucket Policy)**: 누가 버킷에 접근할 수 있는지 JSON 형식으로 정의
> - **ACL(Access Control List)**: 객체 단위로 접근 권한 설정 (현재는 비권장)
> - **퍼블릭 액세스 차단**: 실수로 퍼블릭 설정을 하는 것을 방지하는 안전장치
> - **서버 측 암호화**: S3에 저장되는 데이터를 자동으로 암호화
> - **버전 관리**: 실수로 삭제한 데이터를 복구할 수 있도록 이전 버전을 보관
> - **객체 잠금**: 일정 기간 동안 데이터를 삭제/수정할 수 없도록 보호
>
> 이처럼 AWS 서비스의 책임 모델은 단순히 IaaS/PaaS/서버리스로 나누어지는 것이 아니라, **각 서비스의 특성에 따라 미세하게 달라집니다**. CCP 시험에서는 주요 서비스별 책임 경계를 정확히 아는 것이 중요합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "관리형 서비스를 쓰면 보안을 신경 쓰지 않아도 된다"는 위험한 오해입니다. RDS를 사용하면 OS 패치는 AWS가 하지만, **데이터베이스 내부의 사용자 계정 관리, 접근 권한 설정, 데이터 암호화** 등은 여전히 고객의 책임입니다. 편의가 늘어난 것이지, 보안 책임이 사라진 것이 아닙니다.

> ⚠️ **주의**: CCP 시험에서 자주 출제되는 패턴: "다음 중 고객의 책임은?" 또는 "다음 중 AWS의 책임은?" 형태의 질문에서, **서비스 유형을 먼저 파악**하는 것이 핵심입니다. EC2(IaaS)인지, RDS(관리형)인지, Lambda(서버리스)인지에 따라 답이 달라집니다. 단, **데이터 보호와 접근 관리는 항상 고객 책임**이라는 원칙은 변하지 않습니다.

### Section 요약 (Section Summary)

- 서비스의 추상화 수준이 높아질수록(IaaS → PaaS → 서버리스) 고객의 운영 보안 책임은 줄어든다
- EC2는 고객 책임이 가장 크고(OS까지 관리), Lambda는 가장 작다(코드와 접근 관리만)
- **데이터 보호와 접근 관리는 어떤 서비스를 사용하든 항상 고객의 책임**이다
- 보안 인력이 제한적이라면 관리형/서버리스 서비스를 선택하여 책임 범위를 줄이는 것이 전략적이다

## 12.5 보안 설계 원칙 — 최소 권한, 심층 방어, 자동화, 추적 가능성

> 💡 **한 줄 요약**: 완벽한 보안은 없지만, 핵심 원칙을 따르면 피해를 최소화하고 빠르게 대응할 수 있다.

### 직관적 이해 (Intuitive Understanding)

중세 성(城)의 방어를 떠올려 봅시다. 좋은 성은 하나의 거대한 벽에만 의존하지 않습니다:

- **해자(물길)**: 적이 성벽에 접근하기 어렵게 만듭니다 → 네트워크 경계 보안
- **성벽**: 내부와 외부를 분리합니다 → 방화벽
- **성문 경비**: 신원을 확인하고 허가된 사람만 통과시킵니다 → 인증과 인가
- **내성(내부 성벽)**: 외벽이 뚫려도 핵심 영역은 추가 방어됩니다 → 다중 보안 계층
- **망루의 감시병**: 적의 움직임을 관찰하고 기록합니다 → 모니터링과 로깅
- **경비병의 열쇠**: 모든 경비병이 왕의 금고 열쇠를 가지고 있지 않습니다 → 최소 권한

이러한 원칙들은 수천 년 동안 검증된 방어 전략이며, 현대 클라우드 보안에도 동일하게 적용됩니다.

### 핵심 개념 설명 (Core Explanation)

AWS가 권장하는 보안 설계 원칙은 네 가지입니다:

#### 원칙 1: 최소 권한 원칙 (Principle of Least Privilege)

> 📝 **최소 권한 원칙(Principle of Least Privilege)**: 사용자, 시스템, 프로세스에게 업무 수행에 **꼭 필요한 최소한의 권한만** 부여하는 원칙. "필요한 것만 허용하고, 나머지는 모두 차단"하는 접근 방식이다.

이것은 클라우드 보안에서 **가장 중요한 단일 원칙**입니다.

**왜 중요한가?**

회사의 모든 직원에게 CEO의 권한을 주면 어떻게 될까요? 편리할 수는 있지만, 실수나 악의적 행동으로 인한 피해가 극대화됩니다. 신입사원이 실수로 전체 시스템을 삭제할 수도 있고, 퇴사 예정인 직원이 기밀 데이터에 접근할 수도 있습니다.

최소 권한 원칙은 이러한 위험을 줄입니다:

- **실수의 영향 범위 제한**: 실수를 하더라도 해당 사용자의 권한 범위 내에서만 영향이 미칩니다
- **내부 위협 완화**: 악의적 행위자가 접근할 수 있는 범위를 최소화합니다
- **계정 탈취 피해 최소화**: 하나의 계정이 해킹되어도 피해가 제한됩니다

**AWS에서의 적용**:

- 새로운 IAM 사용자는 **권한 없이** 생성하고, 필요한 것만 하나씩 추가합니다
- 루트 계정은 일상 작업에 사용하지 않습니다
- 임시 자격 증명(역할)을 가능한 한 활용합니다

> 💬 **참고**: IAM에서 최소 권한 원칙을 구체적으로 구현하는 방법은 Chapter 13에서 자세히 다룹니다.

#### 원칙 2: 심층 방어 (Defense in Depth)

> 📝 **심층 방어(Defense in Depth)**: 하나의 보안 수단에만 의존하지 않고, **여러 겹의 보안 계층**을 중첩하여 방어하는 전략. 하나의 계층이 뚫려도 다음 계층이 방어할 수 있도록 설계한다.

**왜 중요한가?**

어떤 보안 수단도 100% 완벽하지 않습니다. 방화벽에 취약점이 발견될 수 있고, 비밀번호가 유출될 수 있고, 관리자가 실수할 수 있습니다. 하나의 방어선에만 의존하면, 그것이 뚫리는 순간 모든 것이 노출됩니다.

**AWS에서의 심층 방어 예시**:

```
[외부 인터넷]
      │
      ▼
┌─ 1계층: AWS Shield (DDoS 방어) ──────────────┐
│     │                                         │
│     ▼                                         │
│ ┌─ 2계층: 네트워크 ACL (서브넷 수준 방화벽) ──┐ │
│ │     │                                     │ │
│ │     ▼                                     │ │
│ │ ┌─ 3계층: 보안 그룹 (인스턴스 수준 방화벽) ┐│ │
│ │ │     │                                   ││ │
│ │ │     ▼                                   ││ │
│ │ │ ┌─ 4계층: OS 방화벽 ──────────────────┐ ││ │
│ │ │ │     │                               │ ││ │
│ │ │ │     ▼                               │ ││ │
│ │ │ │ ┌─ 5계층: 애플리케이션 인증 ───────┐│ ││ │
│ │ │ │ │     │                           ││ ││ │
│ │ │ │ │     ▼                           ││ ││ │
│ │ │ │ │  [암호화된 데이터]               ││ ││ │
│ │ │ │ └─────────────────────────────────┘│ ││ │
│ │ │ └───────────────────────────────────┘ ││ │
│ │ └─────────────────────────────────────────┘│ │
│ └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

> 💬 **참고**: AWS Shield, 보안 그룹, 네트워크 ACL 등 각 보안 계층의 구체적인 설정 방법은 Chapter 14와 Chapter 15에서 자세히 다룹니다.

#### 원칙 3: 보안 자동화 (Security Automation)

수동으로 수행하는 보안 작업은 실수가 발생하기 쉽고, 규모가 커지면 따라갈 수 없습니다. 가능한 한 보안을 **자동화**해야 합니다.

**왜 중요한가?**

- 사람은 피로하고 실수합니다. 자동화된 시스템은 24/7 일관되게 동작합니다
- 클라우드 환경에서는 리소스가 빠르게 생성·삭제되므로, 수동 보안 점검으로는 따라갈 수 없습니다
- 보안 위협 탐지와 대응 속도를 높일 수 있습니다

**AWS에서의 적용 예시**:

- AWS Config: 리소스 설정이 규칙을 위반하면 자동으로 감지
- Amazon GuardDuty: 의심스러운 활동을 자동으로 탐지하여 알림
- AWS Systems Manager: OS 패치를 자동으로 적용

> 💬 **참고**: 보안 자동화 서비스들은 Chapter 15(보안 서비스)와 Chapter 18(모니터링)에서 자세히 다룹니다.

#### 원칙 4: 추적 가능성 (Traceability)

> 📝 **추적 가능성(Traceability)**: 시스템에서 **누가, 언제, 무엇을, 어떻게** 했는지를 기록하고 추적할 수 있는 능력. 보안 사고 발생 시 원인 분석과 대응에 필수적이다.

**왜 중요한가?**

보안 사고가 발생했을 때, "무슨 일이 일어났는지" 모르면 대응할 수 없습니다. 로그(기록)가 있으면:

- 사고의 원인을 파악할 수 있습니다
- 영향 범위를 확인할 수 있습니다
- 향후 재발 방지 대책을 수립할 수 있습니다
- 법적·규제적 요구사항을 충족할 수 있습니다

**AWS에서의 적용**:

- **AWS CloudTrail**: AWS 계정에서 수행되는 모든 API 호출을 기록합니다. "누가 언제 어떤 작업을 했는지"에 대한 감사 로그(audit log)를 제공합니다.
- **Amazon CloudWatch**: 시스템의 성능 지표와 로그를 수집·모니터링합니다.
- **VPC Flow Logs**: 네트워크 트래픽의 흐름을 기록합니다.

> 💬 **참고**: CloudTrail과 CloudWatch는 Chapter 18에서 자세히 다룹니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 12.5.1: 최소 권한 — 인턴에게 어떤 권한을 줄까?**
>
> **상황**: 3개월간 인턴이 입사합니다. 업무는 S3에 저장된 분석 데이터를 읽어서 보고서를 작성하는 것입니다. 이 인턴에게 어떤 AWS 권한을 부여해야 할까요?
> **풀이/설명**:
> - ❌ **나쁜 예**: "편리하니까" 관리자(Admin) 권한을 부여 → 인턴이 실수로 운영 서버를 삭제할 수 있음
> - ❌ **나쁜 예**: S3 전체에 대한 읽기/쓰기 권한 부여 → 다른 프로젝트의 민감한 데이터에 접근 가능
> - ✅ **좋은 예**: **특정 S3 버킷의 읽기 전용** 권한만 부여 → 필요한 데이터만 읽을 수 있고, 삭제나 수정은 불가능
>
> **핵심 포인트**: "이 사람이 업무를 수행하는 데 꼭 필요한 권한은 무엇인가?"를 기준으로 판단합니다. 의심스러우면 더 적은 권한에서 시작하여 필요할 때 추가하는 것이 안전합니다.

> 🔍 **예시 12.5.2: 심층 방어 — 하나만 뚫리면 끝?**
>
> **상황**: 공격자가 웹 애플리케이션의 로그인 페이지에서 관리자 비밀번호를 알아냈습니다. 심층 방어가 적용된 시스템과 그렇지 않은 시스템의 결과를 비교합니다.
> **풀이/설명**:
>
> **심층 방어가 없는 경우**:
> 1. 비밀번호 입력 → 로그인 성공
> 2. 관리자로서 모든 데이터에 접근 → **즉시 데이터 유출**
>
> **심층 방어가 적용된 경우**:
> 1. 비밀번호 입력 → MFA(다중 인증) 코드 요구 → 코드가 없어 **로그인 차단**
> 2. (MFA도 뚫렸다면) 특정 IP에서만 접근 가능하도록 설정 → 외부 IP라 **접근 차단**
> 3. (IP 제한도 통과했다면) 데이터가 암호화되어 있어 복호화 키 없이는 **읽을 수 없음**
> 4. CloudTrail이 비정상 접근을 기록 → 보안 팀에 **알림 발송** → 계정 즉시 비활성화
>
> **핵심 포인트**: 심층 방어는 "하나가 뚫려도 다음 방어선이 있다"는 전략입니다. 공격자가 모든 계층을 통과하는 것은 하나를 통과하는 것보다 훨씬 어렵습니다.

### 미니 실습 (Hands-on Mini Lab)

> 🧪 **Mini Lab 12.5: IAM 보안 상태 확인하기**
>
> **목표**: AWS 계정의 보안 상태를 IAM 대시보드에서 확인하고, 보안 권장 사항을 탐색합니다.
> **예상 소요 시간**: 5분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS 관리 콘솔에 로그인합니다.
> **Step 2**: 상단 검색창에 "IAM"을 입력하고 **IAM**을 클릭합니다.
> **Step 3**: IAM 대시보드(Dashboard)가 표시됩니다. 화면의 **"Security recommendations"** (보안 권장 사항) 섹션을 확인합니다.
> **Step 4**: 다음 항목들의 상태를 확인합니다:
> - 루트 사용자에 MFA가 활성화되어 있는가?
> - 루트 사용자의 액세스 키가 생성되어 있지 않은가?
> - IAM 사용자가 생성되어 있는가?
> **Step 5**: 왼쪽 메뉴에서 **"Account settings"** (계정 설정)을 클릭하고, 비밀번호 정책(Password policy)을 확인합니다. 기본 정책과 사용자 지정 정책의 차이를 살펴봅니다.
>
> **확인**: IAM 대시보드에서 보안 권장 사항 항목들이 표시되고, 각 항목의 충족/미충족 상태를 확인할 수 있으면 성공입니다.
> **🧹 정리**: 별도 리소스를 생성하지 않았으므로 정리할 것이 없습니다. 다만 보안 권장 사항 중 미충족 항목이 있다면 조치를 권장합니다 (MFA 활성화 등).

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: 최소 권한 원칙을 "처음부터 완벽하게 적용해야 한다"고 오해하지 마세요. 실무에서는 **점진적 접근**이 현실적입니다. 초기에는 약간 넓은 권한으로 시작하되, AWS CloudTrail이나 IAM Access Analyzer 같은 도구를 활용하여 **실제 사용하지 않는 권한을 식별하고 제거**해 나갑니다.

> ⚠️ **주의**: "보안 계층이 많을수록 무조건 좋다"는 것은 아닙니다. 과도한 보안은 사용자 경험을 저해하고 생산성을 떨어뜨립니다. **적절한 수준의 보안**을 찾는 것이 중요하며, 이를 위해 보호해야 할 자산의 가치와 위협의 수준을 평가하는 **위험 평가(Risk Assessment)**가 선행되어야 합니다.

### Section 요약 (Section Summary)

- **최소 권한 원칙**: 꼭 필요한 최소한의 권한만 부여하여 실수와 악의적 행동의 영향을 제한한다
- **심층 방어**: 여러 겹의 보안 계층을 중첩하여 하나가 뚫려도 다음 계층이 방어한다
- **보안 자동화**: 수동 작업의 실수를 줄이고, 대규모 환경에서도 일관된 보안을 유지한다
- **추적 가능성**: 모든 활동을 기록하여 사고 발생 시 원인 분석과 대응을 가능하게 한다

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 AWS 클라우드 보안의 근간이 되는 **공동 책임 모델**과 **보안 설계 원칙**을 학습했습니다.

- **Section 12.1**: 공동 책임 모델은 AWS("클라우드**의** 보안")와 고객("클라우드**에서의** 보안")이 보안 책임을 나누는 프레임워크이다. 제어는 상속된 제어, 공유된 제어, 고객 전용 제어로 구분된다.
- **Section 12.2**: AWS는 데이터센터 물리적 보안, 환경 제어, 네트워크 인프라, 하이퍼바이저의 보안을 책임진다. 이를 제3자 감사 보고서(AWS Artifact)를 통해 증명한다.
- **Section 12.3**: 고객은 데이터 보호, 접근 관리, OS·네트워크 구성, 애플리케이션 보안, 백업을 책임진다. 실제 보안 사고의 대부분은 고객 측 설정 실수에서 발생한다.
- **Section 12.4**: 서비스의 추상화 수준이 높아질수록(EC2 → RDS → Lambda) 고객의 운영 보안 책임은 줄어들지만, 데이터 보호와 접근 관리는 항상 고객 책임이다.
- **Section 12.5**: 최소 권한, 심층 방어, 보안 자동화, 추적 가능성은 클라우드 보안의 핵심 설계 원칙이다.

**Q1.** AWS 공동 책임 모델에서 "Security of the Cloud"는 고객의 책임이다. (O/X)

**고객의 책임**입니다. EC2는 IaaS 서비스이므로 하이퍼바이저 위의 운영체제는 고객이 관리합니다. 반면, EC2가 실행되는 물리 서버의 하이퍼바이저 패치는 AWS의 책임입니다.
</details>

<details>

**X** — Lambda(서버리스)를 사용해도 **함수 코드의 보안, 데이터 보호, IAM 역할(접근 권한) 설정**은 여전히 고객의 책임입니다. 서버리스로 가면 인프라 운영 책임은 크게 줄지만, 데이터와 접근 관리 책임은 사라지지 않습니다.
</details>

**Q5.** "최소 권한 원칙"이란 무엇인가? 한 문장으로 설명하시오.

<details>
<summary>정답 확인</summary>

사용자, 시스템, 프로세스에게 **업무 수행에 꼭 필요한 최소한의 권한만 부여**하는 보안 원칙입니다. 필요한 것만 허용하고 나머지는 차단하여, 실수나 공격의 영향 범위를 최소화합니다.
</details>

**Q6.** AWS 데이터센터의 물리적 보안 상태를 직접 확인하려면 어떻게 해야 하는가?

<details>
<summary>정답 확인</summary>

AWS 데이터센터는 직접 방문이 불가능합니다. 대신 **AWS Artifact**에서 SOC 보고서, ISO 27001 인증서 등 **제3자 감사 보고서**를 다운로드하여 확인할 수 있습니다.
</details>

**Q7.** 심층 방어(Defense in Depth)의 핵심 아이디어는 무엇인가?

<details>
<summary>정답 확인</summary>

하나의 보안 수단에만 의존하지 않고 **여러 겹의 보안 계층을 중첩**하여 방어하는 것입니다. 하나의 계층이 뚫려도 다음 계층이 방어할 수 있으므로, 전체 시스템이 한 번에 무너지는 것을 방지합니다.
</details>

## 🧪 통합 실습: 내 AWS 계정의 보안 상태 점검하기

> **시나리오**: 당신은 스타트업의 클라우드 관리자로 첫 출근했습니다. CTO가 "우리 AWS 계정의 보안 상태를 점검하고, 공동 책임 모델 관점에서 우리가 신경 써야 할 항목을 정리해 달라"고 요청했습니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | AWS 계정의 보안 설정을 탐색하고, 공동 책임 모델에서 고객 책임에 해당하는 보안 항목의 현재 상태를 확인한다 |
| **사용 서비스** | IAM, AWS Artifact, Trusted Advisor, Security Hub (탐색) |
| **활용 개념** | Section 12.1 공동 책임 모델, Section 12.3 고객의 보안 책임, Section 12.5 보안 설계 원칙 |
| **선수 실습** | Chapter 4에서 생성한 AWS 계정 |
| **예상 소요 시간** | 20분 |
| **비용** | Free Tier 범위 내 |

### 아키텍처 다이어그램

이 실습은 인프라를 구성하는 것이 아니라, 기존 계정의 보안 상태를 **점검**하는 실습입니다.

```
[AWS 관리 콘솔]
      │
      ├── [IAM 대시보드] ──── 보안 권장 사항 확인
      │
      ├── [AWS Artifact] ──── AWS 보안 인증 보고서 탐색
      │
      └── [Trusted Advisor] ── 보안 점검 결과 확인
```

### 실습 단계

> **Phase 1: IAM 보안 상태 점검** (활용 개념: Section 12.3, 12.5)
>
> **Step 1**: AWS 관리 콘솔(https://console.aws.amazon.com)에 로그인합니다.
> **Step 2**: 상단 검색창에 "IAM"을 입력하고 **IAM**을 클릭하여 IAM 대시보드로 이동합니다.
> **Step 3**: 대시보드의 **"Security recommendations"** 섹션을 확인합니다. 다음 항목을 체크하세요:
>   - ✅ 루트 사용자에 MFA가 활성화되어 있는가?
>   - ✅ 루트 사용자의 액세스 키가 없는가? (없어야 안전)
>   - ✅ IAM 사용자가 생성되어 있는가? (일상 작업용)
> **Step 4**: 왼쪽 메뉴에서 **"Users"** (사용자)를 클릭합니다. 생성된 사용자 목록을 확인합니다.
> **Step 5**: 각 사용자를 클릭하여 **"Permissions"** (권한) 탭에서 어떤 정책이 연결되어 있는지 확인합니다. 최소 권한 원칙 관점에서 과도한 권한(예: AdministratorAccess)이 부여되어 있지 않은지 살펴봅니다.
>
> **✅ 체크포인트**: IAM 대시보드에서 보안 권장 사항을 확인하고, 사용자별 권한 현황을 파악할 수 있다.

> **Phase 2: AWS Artifact에서 보안 인증 탐색** (활용 개념: Section 12.2)
>
> **Step 1**: 상단 검색창에 "Artifact"를 입력하고 **AWS Artifact**를 클릭합니다.
> **Step 2**: 왼쪽 메뉴에서 **"Reports"** (보고서)를 선택합니다.
> **Step 3**: 보고서 목록에서 다음 유형의 문서를 찾아봅니다:
>   - **SOC 2 Report** — 보안, 가용성, 처리 무결성에 대한 감사 보고서
>   - **ISO 27001 Certificate** — 정보 보안 관리 체계 인증
>   - **PCI DSS Report** — 결제 카드 데이터 보안 표준 준수 보고서
> **Step 4**: 보고서 하나를 클릭하여 이용 약관(NDA)이 어떤 내용인지 확인합니다 (실제 동의·다운로드는 하지 않아도 됩니다).
>
> **✅ 체크포인트**: AWS Artifact에서 다양한 보안 인증 보고서가 존재하는 것을 확인하고, 이것이 AWS가 자신의 보안 책임을 증명하는 수단임을 이해한다.

> **Phase 3: Trusted Advisor로 보안 점검** (활용 개념: Section 12.5)
>
> **Step 1**: 상단 검색창에 "Trusted Advisor"를 입력하고 **AWS Trusted Advisor**를 클릭합니다.
> **Step 2**: Trusted Advisor 대시보드에서 **"Security"** (보안) 카테고리를 찾습니다.
> **Step 3**: 무료 티어(Basic/Developer 지원 플랜)에서도 확인할 수 있는 핵심 보안 점검 항목들을 살펴봅니다:
>   - **MFA on Root Account** — 루트 계정의 MFA 활성화 여부
>   - **Security Groups — Specific Ports Unrestricted** — 보안 그룹에서 위험한 포트가 모두에게 열려 있는지
>   - **Amazon S3 Bucket Permissions** — S3 버킷이 퍼블릭으로 설정되어 있는지
> **Step 4**: 각 항목의 상태(녹색=안전, 노란색=주의, 빨간색=위험)를 확인하고, 빨간색이나 노란색 항목이 있다면 어떤 조치가 필요한지 읽어봅니다.
>
> **✅ 체크포인트**: Trusted Advisor가 고객 책임 영역의 보안 상태를 자동으로 점검해 주는 도구임을 확인한다.

### 최종 확인

다음 세 가지를 확인하면 이 통합 실습이 완료됩니다:

1. IAM 대시보드에서 보안 권장 사항과 사용자 권한 현황을 확인했다
2. AWS Artifact에서 AWS의 보안 인증 보고서를 탐색했다
3. Trusted Advisor에서 보안 점검 결과를 확인하고, 각 항목이 공동 책임 모델의 어느 영역에 해당하는지 이해했다

### 🧹 리소스 정리 (Clean-up)

> 이 실습에서는 새로운 리소스를 생성하지 않았으므로, **별도의 정리 작업이 필요하지 않습니다.**
>
> 단, 점검 과정에서 발견한 보안 취약점이 있다면 조치를 권장합니다:
> - 루트 계정에 MFA가 없다면 → MFA 활성화 (Chapter 13에서 자세히 다룹니다)
> - 루트 계정에 액세스 키가 있다면 → 삭제 검토
> - IAM 사용자에 과도한 권한이 있다면 → 최소 권한 원칙에 따라 조정 검토 (Chapter 13에서 자세히 다룹니다)

### 트러블슈팅 (Troubleshooting)

> **Q: AWS Artifact에서 보고서 목록이 보이지 않습니다.**
> A: AWS Artifact는 글로벌 서비스이므로 리전에 무관하게 접근 가능합니다. 보고서가 보이지 않는다면, IAM 사용자의 권한에 `artifact:GetReport` 관련 권한이 포함되어 있는지 확인하세요. 관리자(Administrator) 권한이 있다면 접근 가능합니다.

> **Q: Trusted Advisor에서 일부 점검 항목이 비활성화되어 있습니다.**
> A: AWS 기본 지원 플랜(Basic)에서는 보안 관련 핵심 점검 항목만 무료로 제공됩니다. 전체 점검 항목을 사용하려면 Business 이상의 지원 플랜이 필요합니다. 무료 항목만으로도 이 실습의 목적은 충분히 달성할 수 있습니다.

> **Q: IAM 대시보드에서 보안 권장 사항이 표시되지 않습니다.**
> A: IAM 대시보드에 접근하려면 `iam:GetAccountSummary` 등의 권한이 필요합니다. 루트 계정이나 관리자 권한을 가진 IAM 사용자로 로그인했는지 확인하세요.

>
> - AWS로 이전되는 것: "클라우드의 보안" (물리 인프라) → 이 영역의 인력 필요성은 감소

> **보안 자동화**:
> - AWS Systems Manager를 활용하여 EC2 **OS 패치 자동화** (정기적 패치 적용)
> - AWS Config 규칙으로 보안 그룹의 위험한 설정(예: SSH 전체 개방)을 **자동 탐지**
> - S3 퍼블릭 액세스 차단을 계정 수준에서 활성화
>
> **추적 가능성**:
> - **CloudTrail 즉시 활성화** — 모든 API 호출 기록
> - CloudWatch Logs와 연동하여 비정상 활동 알림 설정
> - VPC Flow Logs 활성화로 네트워크 트래픽 모니터링
>
> **확장 생각**: 이러한 보안 개선은 한 번에 모두 적용하기보다, 위험도가 높은 항목(IAM 계정 분리, MFA 적용, CloudTrail 활성화)부터 우선 적용하고, 자동화와 모니터링을 점진적으로 구축하는 것이 현실적입니다.
> </details>
