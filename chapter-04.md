## 4.1 AWS 계정 생성

> 💡 **한 줄 요약**: AWS를 사용하기 위한 첫 단계는 계정을 만드는 것이며, 이 계정은 AWS 자원과 비용의 기본 단위가 됩니다.

### 직관적 이해 (Intuitive Understanding)

은행을 이용하려면 가장 먼저 해야 할 일이 무엇일까요? 바로 **계좌를 개설**하는 것입니다. 계좌가 있어야 입출금을 하고, 카드를 만들고, 대출을 받을 수 있습니다. AWS 계정도 마찬가지입니다. AWS의 200개 이상의 서비스를 사용하려면, 가장 먼저 **AWS 계정(Account)**을 만들어야 합니다.

은행 계좌가 "돈"이 오가는 단위라면, AWS 계정은 **"클라우드 자원"이 오가는 단위**입니다. 내가 만든 서버, 저장소, 데이터베이스는 모두 내 AWS 계정 안에 존재하며, 사용한 만큼의 비용도 이 계정으로 청구됩니다.

그런데 은행 계좌를 처음 만들 때 받는 "통장과 도장"이 있듯이, AWS 계정을 만들면 자동으로 생성되는 특별한 존재가 있습니다. 바로 **루트 사용자(Root User)**입니다. 이것이 왜 중요한지, 그리고 왜 조심해야 하는지 살펴봅시다.

### 핵심 개념 설명 (Core Explanation)

#### AWS 계정이란?

> 📝 **AWS 계정(AWS Account)**: AWS 클라우드 서비스를 사용하기 위한 기본 단위로, 하나의 고유한 12자리 계정 ID를 가지며, 이 계정 아래에서 생성한 모든 AWS 자원과 해당 비용이 관리됩니다.

AWS 계정을 생성하면 다음과 같은 것들이 자동으로 만들어집니다:

- **고유한 12자리 계정 ID** (예: `123456789012`) — 계정을 식별하는 번호
- **루트 사용자** — 이 계정의 모든 것을 할 수 있는 최고 권한 사용자
- **기본 리전 설정** — 서비스 사용 시 기본으로 적용되는 리전

> 🔗 **연결**: Chapter 3에서 배운 **리전**을 떠올려 봅시다. AWS 계정을 만들면 모든 리전의 서비스를 사용할 수 있지만, 콘솔에 로그인했을 때 기본으로 표시되는 리전이 있습니다. 이 리전은 언제든 전환할 수 있습니다.

#### 계정 생성에 필요한 것

AWS 계정을 만들기 위해서는 다음 4가지가 필요합니다:

1. **이메일 주소** — 계정의 "루트 사용자 이메일"이 됩니다. 하나의 이메일로는 하나의 AWS 계정만 만들 수 있습니다.
2. **비밀번호** — 루트 사용자의 로그인 비밀번호
3. **연락처 정보** — 이름, 주소, 전화번호
4. **결제 정보** — 신용카드 또는 체크카드 (본인 확인을 위해 $1 임시 결제가 발생하며, 곧 환불됩니다)

#### 루트 사용자 (Root User)

> 📝 **루트 사용자(Root User)**: AWS 계정 생성 시 자동으로 만들어지는 사용자로, 해당 계정의 **모든 AWS 서비스와 리소스에 대한 완전한 접근 권한**을 가집니다. 계정 생성 시 사용한 이메일 주소와 비밀번호로 로그인합니다.

루트 사용자를 일상적인 비유로 설명하면, 건물의 **마스터키**와 같습니다. 마스터키는 건물의 모든 문을 열 수 있지만, 그만큼 잃어버리면 위험합니다. 매일 건물을 돌아다닐 때 마스터키를 들고 다니지 않고, 내 방 열쇠만 들고 다니는 것처럼 — AWS에서도 루트 사용자를 일상적으로 사용하지 않는 것이 보안의 기본 원칙입니다.

**루트 사용자만 할 수 있는 작업들** (이런 경우에만 루트 사용자를 사용합니다):

- AWS 계정의 이메일 주소, 이름 변경
- 결제 정보 변경
- AWS 계정 해지(폐쇄)
- AWS 지원 플랜 변경

> ⚠️ **주의**: 루트 사용자는 **일상 업무에 사용하면 안 됩니다**. 루트 사용자의 자격 증명이 유출되면, 공격자가 계정의 모든 것을 제어할 수 있습니다 — 서버를 수백 대 생성하여 막대한 비용을 발생시키거나, 모든 데이터를 삭제할 수 있습니다.

> 💬 **참고**: 루트 사용자 대신 일상 업무에 사용할 별도의 사용자를 만드는 방법은 Chapter 13(IAM)에서 자세히 다룹니다. 이 Chapter에서는 먼저 루트 사용자를 보호하는 데 집중합니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 4.1.1: 은행 계좌와 AWS 계정의 비유**
>
> **상황**: 은행 계좌를 처음 개설하는 상황을 떠올려 봅시다.
> **비교**:
> | 은행 | AWS |
> |------|-----|
> | 계좌 개설 | AWS 계정 생성 |
> | 계좌 번호 | 12자리 계정 ID |
> | 통장 + 도장(본인 인증) | 루트 사용자 이메일 + 비밀번호 |
> | 입출금, 이체 | 서비스 생성, 삭제, 수정 |
> | 월별 사용료/이체 수수료 | AWS 사용료 (종량제) |
> | 계좌 해지 | AWS 계정 폐쇄 |
>
> **핵심 포인트**: 은행 계좌처럼 AWS 계정도 하나의 "자원과 비용의 관리 단위"입니다. 은행에서 통장과 도장을 안전하게 보관하는 것처럼, AWS에서도 루트 사용자의 자격 증명을 안전하게 보호해야 합니다.

> 🔍 **예시 4.1.2: 스타트업의 AWS 계정 생성**
>
> **상황**: "클라우드 카페"라는 스타트업이 자사 모바일 앱의 백엔드를 AWS에 구축하려고 합니다. 대표인 김개발 씨가 처음으로 AWS 계정을 만듭니다.
> **과정**:
> 1. 회사 대표 이메일(`admin@cloudcafe.kr`)로 AWS 계정 생성
> 2. 계정 ID `987654321098`이 자동 부여됨
> 3. 루트 사용자로 로그인하여 초기 설정 진행
> 4. MFA 설정 (다음 Section에서 학습)
> 5. 이후 개발팀원별로 별도의 사용자를 생성하여 업무 수행 (Chapter 13에서 학습)
>
> **핵심 포인트**: 루트 사용자의 이메일은 "개인 이메일"보다 "관리자 전용 이메일"을 사용하는 것이 좋습니다. 직원이 퇴사해도 계정 접근에 문제가 없도록 하기 위함입니다.

### 미니 실습 (Hands-on Mini Lab)

> **목표**: AWS 계정을 직접 생성하고 로그인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (카드 인증을 위해 $1 임시 결제 후 환불)

> 💬 **참고**: 이미 AWS 계정이 있다면 이 실습은 건너뛰고 다음 Section으로 진행해도 됩니다.

> **Step 1**: 웹 브라우저에서 `aws.amazon.com`에 접속합니다. 우측 상단의 **"AWS 계정 생성"** (또는 "Create an AWS Account") 버튼을 클릭합니다.
>
> **Step 2**: **루트 사용자 이메일 주소**와 **AWS 계정 이름**을 입력합니다.
> - 이메일: 본인이 사용하는 유효한 이메일 주소
> - 계정 이름: 원하는 이름 입력 (예: `my-aws-study`)
> - "인증 코드 보내기(Verify email address)"를 클릭하고, 이메일로 받은 인증 코드를 입력합니다.
>
> **Step 3**: **루트 사용자 비밀번호**를 설정합니다. 최소 8자, 대문자·소문자·숫자·특수문자를 포함하는 강력한 비밀번호를 설정합니다.
>
> **Step 4**: **연락처 정보**를 입력합니다. 계정 유형은 **"개인(Personal)"**을 선택합니다 (학습 목적).
>
> **Step 5**: **결제 정보**를 입력합니다. 신용카드 또는 체크카드 정보를 입력합니다. $1 임시 결제가 발생하며 며칠 내 환불됩니다.
>
> **Step 6**: **본인 확인**을 진행합니다. 전화번호를 입력하고 SMS 또는 음성 통화로 인증 코드를 수신하여 입력합니다.
>
> **Step 7**: **지원 플랜**을 선택합니다. **"기본 지원 — 무료(Basic Support — Free)"**를 선택합니다.
>
> **Step 8**: 계정 생성이 완료되면, `aws.amazon.com`에서 **"콘솔에 로그인(Sign In to the Console)"**을 클릭합니다. **"루트 사용자(Root user)"**를 선택하고, 계정 생성 시 사용한 이메일과 비밀번호로 로그인합니다.
>
> **확인**: 로그인에 성공하면 **AWS Management Console** 메인 화면이 표시됩니다. 우측 상단에 계정 이름이 보이면 성공입니다.

> ⚠️ **주의**: 계정 생성 직후에는 일부 서비스 활성화에 최대 24시간이 소요될 수 있습니다. 만약 특정 서비스에 접근이 안 된다면, 잠시 후 다시 시도해 보세요.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "루트 사용자로 모든 작업을 수행하는 것"
>
> 계정을 만들면 루트 사용자로 바로 작업을 시작하게 됩니다. 하지만 루트 사용자는 **계정 설정 변경 등 특수한 경우에만** 사용해야 합니다. 일상 업무에 루트 사용자를 사용하면, 자격 증명이 유출되었을 때 피해가 치명적입니다. 가능한 빨리 별도의 IAM 사용자를 생성하세요 (Chapter 13 참고).

> ⚠️ **주의**: "개인 이메일로 회사 AWS 계정을 생성하는 것"
>
> 루트 사용자 이메일은 계정의 최종 복구 수단입니다. 개인 이메일로 회사 계정을 만들면, 해당 직원이 퇴사했을 때 계정 접근이 불가능해질 수 있습니다. 회사 계정은 반드시 **관리자 전용 이메일(예: aws-admin@회사.com)**로 생성하세요.

> ⚠️ **주의**: "계정 생성 후 MFA를 설정하지 않고 방치하는 것"
>
> 계정 생성 직후에는 비밀번호만으로 로그인이 가능한 상태입니다. 이 상태에서 비밀번호가 유출되면 즉시 계정이 탈취될 수 있습니다. 계정을 만든 후 **가장 먼저 해야 할 일은 MFA 설정**입니다 (Section 4.2 참고).

> ⚠️ **주의**: "AWS 계정 비밀번호를 다른 서비스와 동일하게 설정하는 것"
>
> 다른 웹사이트에서 유출된 비밀번호로 AWS 계정에 로그인 시도하는 공격(크레덴셜 스터핑)은 매우 흔합니다. AWS 루트 사용자 비밀번호는 **다른 서비스와 절대 중복 사용하지 마세요**.

### Section 요약 (Section Summary)

- **AWS 계정**은 클라우드 자원과 비용의 기본 관리 단위이며, 고유한 12자리 계정 ID를 갖습니다.
- 계정 생성 시 **루트 사용자**가 자동으로 만들어지며, 이 사용자는 계정의 모든 권한을 가집니다.
- 루트 사용자는 **계정 설정 변경** 등 특수한 경우에만 사용하고, 일상 업무에는 사용하지 않는 것이 보안 모범 사례입니다.
- 계정 생성에는 이메일, 비밀번호, 연락처, 결제 정보가 필요합니다.

## 4.2 MFA와 초기 보안 설정

> 💡 **한 줄 요약**: MFA(다중 인증)는 비밀번호 외에 추가 인증 수단을 요구하여, 비밀번호가 유출되어도 계정을 보호하는 핵심 보안 장치입니다.

### 직관적 이해 (Intuitive Understanding)

집에 현관문 잠금장치만 있으면 안전할까요? 요즘 많은 집에서 현관 비밀번호 외에도 **지문 인식**이나 **카드키**를 함께 사용합니다. 비밀번호만 있으면, 누군가 번호를 훔쳐보고 들어올 수 있지만, 지문까지 요구하면 본인이 아니고서는 문을 열 수 없습니다.

**MFA(Multi-Factor Authentication, 다중 인증)**는 바로 이 원리를 디지털 세계에 적용한 것입니다. "내가 아는 것(비밀번호)" + "내가 가진 것(스마트폰)" 두 가지를 모두 제시해야 로그인할 수 있게 만드는 것입니다.

AWS에서 MFA가 특히 중요한 이유는, 앞서 배운 **루트 사용자**의 막강한 권한 때문입니다. 만약 루트 사용자의 비밀번호가 유출되면, 공격자가 수백 대의 고성능 서버를 생성하여 수천만 원의 청구서를 남길 수 있습니다. 실제로 이런 사고는 드물지 않게 발생합니다. MFA는 이러한 재앙을 막는 **가장 기본적이면서도 효과적인 보안 조치**입니다.

### 핵심 개념 설명 (Core Explanation)

#### MFA란?

> 📝 **MFA(Multi-Factor Authentication, 다중 인증)**: 사용자가 로그인할 때 비밀번호(1차 인증) 외에 추가 인증 수단(2차 인증)을 요구하는 보안 방식입니다. "내가 아는 것(Something I know)" + "내가 가진 것(Something I have)" 또는 "내가 인 것(Something I am)"을 조합합니다.

인증 요소(Factor)는 크게 3가지 범주로 나뉩니다:

| 인증 요소 | 설명 | 예시 |
|-----------|------|------|
| **지식 요소** (Something I know) | 사용자가 알고 있는 정보 | 비밀번호, PIN 번호 |
| **소유 요소** (Something I have) | 사용자가 가지고 있는 물건 | 스마트폰(인증 앱), 보안 키 |
| **생체 요소** (Something I am) | 사용자의 신체적 특성 | 지문, 얼굴 인식 |

MFA는 이 중 **2가지 이상의 서로 다른 범주**를 조합합니다. 비밀번호(지식 요소) + 인증 앱의 코드(소유 요소)가 가장 일반적인 조합입니다.

#### AWS에서 지원하는 MFA 장치 유형

AWS는 다음과 같은 MFA 장치를 지원합니다:

1. **가상 MFA 장치 (Virtual MFA Device)** — 가장 일반적이고 무료
   - 스마트폰의 인증 앱 (Google Authenticator, Microsoft Authenticator 등)
   - 30초마다 6자리 숫자 코드를 생성
   - **권장**: 학습 및 개인 사용에 가장 적합

2. **하드웨어 TOTP 토큰 (Hardware TOTP Token)**
   - 별도의 물리적 장치 (예: Gemalto 토큰)
   - 인터넷 연결 없이 코드 생성
   - 기업 환경에서 사용

3. **FIDO2 보안 키 (Security Key)**
   - USB 또는 NFC 방식의 물리적 키 (예: YubiKey)
   - 가장 강력한 보안 수준
   - 피싱 공격에 대한 저항력이 뛰어남

4. **패스키 (Passkey)**
   - 기기의 생체 인식(지문, 얼굴)을 활용
   - 비밀번호 없이도 인증 가능

#### 루트 사용자 보호 모범 사례

AWS는 루트 사용자를 보호하기 위한 모범 사례를 공식적으로 권장합니다:

1. **루트 사용자에 MFA를 반드시 활성화**합니다.
2. **루트 사용자의 액세스 키를 생성하지 않습니다** (액세스 키가 있다면 삭제).
3. **루트 사용자의 비밀번호를 강력하게 설정**합니다 (16자 이상, 대·소문자·숫자·특수문자 조합).
4. **일상 업무에는 루트 사용자를 사용하지 않습니다**.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 4.2.1: 현관문 보안 비유**
>
> **상황**: 집의 보안 수준을 높이고 싶습니다.
> **비교**:
> | 보안 수준 | 현관문 | AWS 계정 |
> |-----------|--------|----------|
> | 가장 약함 | 잠금장치 없음 | 비밀번호 없음 (불가능) |
> | 약함 | 비밀번호 잠금만 | 비밀번호만 사용 |
> | 보통 | 비밀번호 + 카드키 | 비밀번호 + 가상 MFA |
> | 강함 | 비밀번호 + 지문 인식 | 비밀번호 + FIDO2 보안 키 |
>
> **핵심 포인트**: 비밀번호 하나만으로는 충분하지 않습니다. 비밀번호는 유출될 수 있지만, 물리적 장치(스마트폰, 보안 키)까지 동시에 탈취하기는 훨씬 어렵습니다.

> 🔍 **예시 4.2.2: MFA가 없어서 생긴 실제 사고 시나리오**
>
> **상황**: 개발자 A 씨는 AWS 계정을 만들었지만, MFA를 설정하지 않았습니다. 몇 달 후, A 씨의 이메일 비밀번호가 데이터 유출 사건으로 인해 노출되었습니다. 이메일과 같은 비밀번호를 AWS에도 사용하고 있었습니다.
> **결과**:
> 1. 공격자가 유출된 이메일/비밀번호로 AWS 콘솔에 루트 사용자로 로그인
> 2. 전 세계 여러 리전에 고성능 EC2 인스턴스를 수십 대 생성 (암호화폐 채굴 목적)
> 3. 하루 만에 약 $5,000의 비용이 발생
> 4. A 씨는 AWS의 비용 알림 이메일을 보고서야 사고를 인지
>
> **교훈**: MFA가 설정되어 있었다면, 비밀번호가 유출되어도 스마트폰의 인증 코드 없이는 로그인할 수 없었을 것입니다.

### 미니 실습 (Hands-on Mini Lab)

> **목표**: 루트 사용자에 가상 MFA 장치(스마트폰 인증 앱)를 등록합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (비용 없음)

> **사전 준비**: 스마트폰에 인증 앱을 설치합니다.
> - Android: Google Play에서 "Google Authenticator" 또는 "Microsoft Authenticator" 설치
> - iOS: App Store에서 "Google Authenticator" 또는 "Microsoft Authenticator" 설치

> **Step 1**: AWS Management Console에 **루트 사용자**로 로그인합니다.
>
> **Step 2**: 우측 상단의 **계정 이름**을 클릭하고, 드롭다운 메뉴에서 **"보안 자격 증명(Security credentials)"**을 클릭합니다.
>
> **Step 3**: "내 보안 자격 증명(My Security Credentials)" 페이지에서 **"다중 인증(MFA)"** 섹션을 찾습니다. **"MFA 장치 할당(Assign MFA device)"** 버튼을 클릭합니다.
>
> **Step 4**: MFA 장치 이름을 입력합니다 (예: `my-phone-mfa`). MFA 장치 유형으로 **"인증 관리자 앱(Authenticator app)"**을 선택하고 **"다음(Next)"**을 클릭합니다.
>
> **Step 5**: 화면에 **QR 코드**가 표시됩니다. 스마트폰의 인증 앱을 열고 **"+" 버튼** (또는 "계정 추가")을 눌러 이 QR 코드를 스캔합니다.
>
> **Step 6**: 인증 앱에 6자리 숫자 코드가 표시됩니다. 이 코드를 **"MFA 코드 1"** 칸에 입력합니다. 30초를 기다리면 새로운 코드가 표시됩니다. 새 코드를 **"MFA 코드 2"** 칸에 입력합니다.
>
> **Step 7**: **"MFA 추가(Add MFA)"** 버튼을 클릭합니다.
>
> **확인**: "MFA 장치가 성공적으로 할당되었습니다"라는 메시지가 표시됩니다. MFA 섹션에 방금 등록한 장치가 나타나면 성공입니다.
>
> **동작 확인**: 로그아웃 후 다시 로그인해 봅니다. 이메일과 비밀번호를 입력한 후, 추가로 **MFA 코드 입력 화면**이 나타나면 MFA가 정상적으로 작동하는 것입니다.

> ⚠️ **주의**:
> - MFA 장치(스마트폰)를 분실하면 로그인이 불가능해질 수 있습니다. 인증 앱의 백업/복구 기능을 설정해 두거나, MFA 장치를 2개 이상 등록하는 것을 권장합니다.
> - QR 코드를 스크린샷으로 저장하면 보안 위험이 됩니다. QR 코드는 저장하지 마세요.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "MFA를 루트 사용자에만 설정하고, 다른 IAM 사용자에는 설정하지 않는 것"
>
> 루트 사용자에 MFA를 설정하는 것은 첫 번째 단계이지만, 이후 생성하는 **IAM 사용자(특히 관리자 권한을 가진 사용자)에도 MFA를 적용**해야 합니다. 권한이 큰 사용자일수록 MFA는 필수입니다.

> ⚠️ **주의**: "MFA 장치(스마트폰)의 백업을 하지 않는 것"
>
> 스마트폰을 분실하거나 교체할 때 인증 앱의 데이터가 사라지면, AWS 계정에 로그인할 수 없게 됩니다. Google Authenticator나 Microsoft Authenticator의 **클라우드 백업 기능을 활성화**하거나, **MFA 장치를 2개 이상 등록**해 두세요.

> ⚠️ **주의**: "MFA를 설정했으니 비밀번호는 대충 해도 된다는 착각"
>
> MFA는 추가 보안 계층이지, 약한 비밀번호를 보완하는 수단이 아닙니다. **강력한 비밀번호와 MFA를 모두 적용**하는 것이 올바른 보안 태세입니다. 비밀번호는 최소 16자 이상, 대소문자·숫자·특수문자를 조합하세요.

> ⚠️ **주의**: "QR 코드를 스크린샷으로 저장하거나 공유하는 것"
>
> MFA 설정 시 표시되는 QR 코드(비밀 키)를 캡처하면, 해당 이미지를 입수한 누구나 동일한 MFA 코드를 생성할 수 있습니다. QR 코드는 **절대 저장하거나 공유하지 마세요**.

### Section 요약 (Section Summary)

- **MFA(다중 인증)**는 비밀번호 외에 추가 인증 수단을 요구하여 보안을 강화합니다.
- AWS에서는 가상 MFA 장치(인증 앱), 하드웨어 토큰, FIDO2 보안 키, 패스키를 지원합니다.
- **루트 사용자에 MFA를 설정하는 것은 AWS 사용의 가장 첫 번째 보안 조치**입니다.
- MFA 장치 분실에 대비하여 백업 방법을 마련해 두어야 합니다.

## 4.3 AWS Management Console 탐색

> 💡 **한 줄 요약**: AWS Management Console은 웹 브라우저에서 AWS의 모든 서비스를 시각적으로 관리할 수 있는 GUI(그래픽 사용자 인터페이스)입니다.

### 직관적 이해 (Intuitive Understanding)

스마트폰을 처음 샀을 때를 떠올려 봅시다. 수백 가지 기능이 있지만, 처음에는 홈 화면에서 앱 아이콘을 찾아 터치하면서 하나씩 익혀갑니다. 설정에 들어가면 와이파이, 블루투스, 배터리 등 카테고리별로 메뉴가 나뉘어 있어서, 원하는 설정을 찾아갈 수 있습니다.

**AWS Management Console**은 AWS의 "스마트폰 홈 화면"과 같습니다. 200개 이상의 서비스가 있지만, 검색 기능과 카테고리 분류 덕분에 원하는 서비스를 빠르게 찾을 수 있습니다. 그리고 각 서비스를 클릭하면 해당 서비스의 "앱"이 열리는 것처럼, 서비스별 대시보드로 이동합니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS Management Console이란?

> 📝 **AWS Management Console**: 웹 브라우저를 통해 AWS 서비스를 시각적으로 관리할 수 있는 그래픽 사용자 인터페이스(GUI)입니다. 별도의 소프트웨어 설치 없이 브라우저만으로 AWS의 모든 서비스에 접근할 수 있습니다.

#### Console의 주요 구성 요소

콘솔에 로그인하면 보이는 화면의 주요 영역을 살펴봅시다:

**1. 상단 내비게이션 바 (Navigation Bar)**

| 영역 | 위치 | 기능 |
|------|------|------|
| **서비스 검색 바** | 상단 중앙 | 서비스 이름이나 키워드로 빠르게 검색 |
| **리전 선택기** | 상단 우측 | 현재 작업 중인 리전을 확인·변경 |
| **계정 메뉴** | 상단 우측 끝 | 계정 설정, 보안 자격 증명, 로그아웃 |
| **알림 벨** | 상단 우측 | AWS 관련 알림 확인 |
| **CloudShell 아이콘** | 상단 우측 | 브라우저 내 CLI 환경 실행 |

**2. 콘솔 홈 대시보드 (Console Home)**

콘솔 홈은 자주 사용하는 정보를 한눈에 볼 수 있는 대시보드입니다:

- **최근 방문 서비스**: 최근에 사용한 서비스 바로가기
- **즐겨찾기**: 사용자가 직접 등록한 즐겨찾기 서비스
- **AWS Health Dashboard**: 현재 AWS 인프라의 상태 요약
- **비용 및 사용량 위젯**: 현재 예상 비용 정보

**3. 서비스 메뉴**

좌측 상단의 **"서비스(Services)"** 메뉴를 클릭하면, 모든 AWS 서비스가 카테고리별로 정리되어 있습니다:

- 컴퓨팅 (Compute)
- 스토리지 (Storage)
- 데이터베이스 (Database)
- 네트워킹 및 콘텐츠 전송 (Networking & Content Delivery)
- 보안, 자격 증명 및 규정 준수 (Security, Identity & Compliance)
- 관리 및 거버넌스 (Management & Governance)
- 기타 다수의 카테고리

> 🔗 **연결**: Chapter 3에서 배운 **AWS 서비스 카테고리**를 떠올려 봅시다. 그때 개괄적으로 살펴본 카테고리들이 바로 이 서비스 메뉴에 그대로 반영되어 있습니다.

#### 리전 전환의 중요성

콘솔 우측 상단의 **리전 선택기**는 매우 중요한 요소입니다.

> 🔗 **연결**: Chapter 3에서 배운 것처럼, AWS 서비스는 **리전별로 독립적으로 운영**됩니다. 서울 리전(ap-northeast-2)에서 만든 서버는 도쿄 리전(ap-northeast-1)에서는 보이지 않습니다.

따라서 콘솔에서 작업할 때 **현재 어떤 리전이 선택되어 있는지 항상 확인**해야 합니다. "분명히 서버를 만들었는데 안 보인다"는 문제의 90%는 리전을 잘못 선택한 경우입니다.

> ⚠️ **주의**: 일부 AWS 서비스는 **글로벌 서비스**로, 특정 리전에 속하지 않습니다. 예를 들어 IAM(자격 증명 관리), Route 53(DNS), CloudFront(CDN) 등은 리전에 관계없이 동일합니다. 이런 서비스를 사용할 때는 리전 선택기에 **"글로벌(Global)"**이 표시됩니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 4.3.1: 스마트폰과 AWS Console의 비유**
>
> **상황**: AWS Console을 처음 사용하는 것은 새 스마트폰을 처음 켜는 것과 비슷합니다.
> **비교**:
> | 스마트폰 | AWS Console |
> |----------|-------------|
> | 홈 화면 | 콘솔 홈 대시보드 |
> | 앱 검색 (돋보기) | 서비스 검색 바 |
> | 설정 > 카테고리별 메뉴 | 서비스 메뉴 > 카테고리 |
> | 앱 아이콘 터치 → 앱 실행 | 서비스 클릭 → 서비스 대시보드 |
> | 알림 센터 | 알림 벨 |
> | 즐겨찾기/자주 사용 앱 | 즐겨찾기 서비스 |
>
> **핵심 포인트**: 200개 이상의 서비스가 있어도 겁먹을 필요 없습니다. 대부분의 사용자가 자주 쓰는 서비스는 10~20개 정도이며, 검색 기능을 활용하면 원하는 서비스를 즉시 찾을 수 있습니다.

> 🔍 **예시 4.3.2: 리전 전환 실수 시나리오**
>
> **상황**: 학습자 B 씨가 서울 리전(ap-northeast-2)에서 실습용 서버를 만들었습니다. 다음 날 콘솔에 로그인했더니 서버가 보이지 않습니다.
> **원인**: 로그인 후 콘솔이 버지니아 북부 리전(us-east-1)으로 설정되어 있었습니다. 서울 리전에서 만든 서버는 서울 리전을 선택해야만 보입니다.
> **해결**: 우측 상단의 리전 선택기에서 **"아시아 태평양(서울) ap-northeast-2"**를 선택하면 서버가 정상적으로 표시됩니다.
> **핵심 포인트**: 콘솔 작업 시 항상 리전 선택기를 먼저 확인하는 습관을 들이세요. 특히 실습할 때는 항상 같은 리전(서울 권장)을 사용하는 것이 혼동을 줄입니다.

### 미니 실습 (Hands-on Mini Lab)

> **목표**: 콘솔의 주요 인터페이스를 직접 탐색하고, 서비스 검색·리전 전환·즐겨찾기 기능을 사용해 봅니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (비용 없음)

> **Step 1**: AWS Management Console에 로그인합니다. 콘솔 홈 화면에서 **최근 방문 서비스**와 **AWS Health Dashboard** 위젯을 확인합니다.
>
> **Step 2** — 서비스 검색: 상단의 **검색 바**를 클릭하고 `EC2`를 입력합니다. 검색 결과에 "EC2 — Virtual Servers in the Cloud"가 나타나는지 확인합니다. 클릭하여 EC2 대시보드로 이동합니다.
>
> **Step 3** — 리전 확인 및 전환: 우측 상단의 **리전 선택기**를 클릭합니다. 현재 선택된 리전을 확인하고, **"아시아 태평양(서울) ap-northeast-2"**를 선택합니다. 선택 후 리전 선택기에 "서울"이 표시되는지 확인합니다.
>
> **Step 4** — 즐겨찾기 등록: 검색 바에 `S3`를 입력합니다. 검색 결과의 "S3" 옆에 있는 **별표(☆)** 아이콘을 클릭하여 즐겨찾기에 등록합니다. 마찬가지로 `CloudWatch`도 즐겨찾기에 등록합니다.
>
> **Step 5** — 즐겨찾기 확인: 상단 내비게이션 바에서 즐겨찾기한 서비스들이 바로가기로 표시되는지 확인합니다.
>
> **Step 6** — 서비스 카테고리 탐색: 좌측 상단의 **"서비스(Services)"** 메뉴를 클릭합니다. **"컴퓨팅(Compute)"** 카테고리를 찾아서 어떤 서비스들이 포함되어 있는지 살펴봅니다. 마찬가지로 **"스토리지(Storage)"**, **"데이터베이스(Database)"** 카테고리도 확인합니다.
>
> **확인**: 리전이 "서울"로 표시되고, 즐겨찾기에 S3와 CloudWatch가 추가되어 있으면 성공입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "리전을 확인하지 않고 작업을 시작하는 것"
>
> AWS Console에서 가장 흔한 실수입니다. "분명히 리소스를 만들었는데 보이지 않는다"는 문제의 대부분은 **다른 리전이 선택되어 있기 때문**입니다. 콘솔에서 작업을 시작하기 전에, 항상 우측 상단의 리전 선택기를 먼저 확인하세요.

> ⚠️ **주의**: "여러 리전에 리소스를 분산 생성하는 것"
>
> 학습 중에 이것저것 시도하다 보면, 서울 리전에 EC2를 만들고, 도쿄 리전에 S3를 만드는 등 리소스가 흩어지기 쉽습니다. 학습 목적이라면 **하나의 리전(서울 ap-northeast-2 권장)에서 모든 작업**을 수행하여 관리를 단순화하세요.

> ⚠️ **주의**: "글로벌 서비스와 리전 서비스를 혼동하는 것"
>
> IAM, Route 53, CloudFront 등은 리전에 속하지 않는 **글로벌 서비스**입니다. 반면 EC2, S3, RDS 등은 **리전 서비스**로, 리전을 전환하면 다른 리소스가 표시됩니다. 시험에서도 자주 출제되는 구분점이니 기억해 두세요.

> ⚠️ **주의**: "콘솔 URL을 북마크할 때 리전 정보를 무시하는 것"
>
> 콘솔 URL에는 리전 정보가 포함되어 있습니다(예: `region=ap-northeast-2`). 다른 리전이 포함된 URL을 북마크하면, 매번 엉뚱한 리전에서 작업을 시작하게 될 수 있습니다.

### Section 요약 (Section Summary)

- **AWS Management Console**은 웹 브라우저 기반의 GUI로, AWS의 모든 서비스를 시각적으로 관리할 수 있습니다.
- 상단 내비게이션 바에서 **서비스 검색**, **리전 전환**, **계정 관리** 등 핵심 기능에 접근합니다.
- **리전 선택기**는 매우 중요합니다. 현재 어떤 리전에서 작업 중인지 항상 확인해야 합니다.
- 자주 사용하는 서비스를 **즐겨찾기**에 등록하면 빠르게 접근할 수 있습니다.

## 4.4 AWS CLI와 CloudShell

> 💡 **한 줄 요약**: AWS CLI는 명령어 한 줄로 AWS 서비스를 제어하는 도구이며, CloudShell은 이를 브라우저에서 바로 사용할 수 있게 해주는 환경입니다.

### 직관적 이해 (Intuitive Understanding)

음식점에서 주문하는 두 가지 방법을 생각해 봅시다:

1. **터치스크린 키오스크** — 메뉴를 하나하나 터치하면서 선택. 직관적이고 사진도 볼 수 있지만, 여러 명이 주문하거나 복잡한 커스터마이징은 시간이 걸립니다.
2. **전화 주문** — "아메리카노 2잔, 샷 추가, 시럽 빼 주세요"라고 한 문장으로 전달. 빠르고 정확하지만, 메뉴를 알고 있어야 합니다.

**AWS Management Console**은 키오스크처럼 시각적으로 서비스를 조작하는 방식이고, **AWS CLI**는 전화 주문처럼 명령어 텍스트로 서비스를 조작하는 방식입니다. 둘 다 같은 결과를 만들 수 있지만, 상황에 따라 더 적합한 방식이 다릅니다.

그리고 **AWS CloudShell**은 "집에서 전화하려면 전화기를 사야 하지만, 식당 안에 비치된 내선 전화로 바로 주문할 수 있는 것"과 비슷합니다. 별도의 설치 없이 콘솔(브라우저) 안에서 바로 CLI를 사용할 수 있는 환경입니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS에 접근하는 3가지 방법

AWS 서비스를 제어하는 방법은 크게 3가지입니다:

| 접근 방식 | 설명 | 적합한 상황 |
|-----------|------|-------------|
| **AWS Management Console** | 웹 브라우저 기반 GUI | 학습, 시각적 확인, 일회성 작업 |
| **AWS CLI** | 명령줄 인터페이스 | 반복 작업, 스크립트 자동화, 빠른 조작 |
| **AWS SDK** | 프로그래밍 언어별 라이브러리 | 애플리케이션에서 AWS 서비스 호출 |

이 3가지 방법은 모두 내부적으로 **AWS API(Application Programming Interface)**를 호출합니다. 즉, 어떤 방법을 사용하든 같은 결과를 얻을 수 있습니다.

> 📝 **API(Application Programming Interface)**: 소프트웨어 간에 정해진 규칙에 따라 데이터를 주고받는 인터페이스입니다. AWS의 모든 서비스는 API를 통해 제어되며, Console, CLI, SDK는 모두 이 API를 호출하는 서로 다른 "통로"입니다.

#### AWS CLI (Command Line Interface)

> 📝 **AWS CLI(Command Line Interface)**: 터미널(명령 프롬프트)에서 텍스트 명령어를 입력하여 AWS 서비스를 제어하는 도구입니다. 로컬 컴퓨터에 설치하여 사용하거나, AWS CloudShell을 통해 브라우저에서 사용할 수 있습니다.

CLI의 기본 명령어 구조는 다음과 같습니다:

```
aws [서비스 이름] [작업] [옵션]
```

예를 들어:

```bash
# S3 버킷 목록 보기
aws s3 ls

# 현재 사용 중인 리전의 EC2 인스턴스 목록 보기
aws ec2 describe-instances

# 특정 리전의 서비스 상태 확인
aws ec2 describe-instances --region ap-northeast-2
```

**CLI가 Console보다 유리한 상황**:
- 같은 작업을 여러 번 반복해야 할 때 (스크립트로 자동화)
- 여러 서비스를 한 번에 조작해야 할 때
- 조작 이력을 기록으로 남기고 싶을 때
- Console에서 여러 번 클릭해야 하는 작업을 한 줄 명령으로 해결하고 싶을 때

#### AWS SDK (Software Development Kit)

> 📝 **AWS SDK(Software Development Kit)**: 특정 프로그래밍 언어에서 AWS 서비스를 호출할 수 있게 해주는 라이브러리 모음입니다. 애플리케이션 코드 안에서 AWS 서비스를 직접 사용할 수 있습니다.

AWS SDK는 다양한 언어를 지원합니다: Python(Boto3), JavaScript, Java, .NET, Go, Ruby 등.

예를 들어, Python으로 S3 버킷 목록을 조회하는 코드는 다음과 같습니다:

```python
import boto3

s3 = boto3.client('s3')
response = s3.list_buckets()

for bucket in response['Buckets']:
    print(bucket['Name'])
```

> 💬 **참고**: 이 교재에서는 AWS SDK를 직접 사용하는 실습은 하지 않습니다. SDK는 개발자가 애플리케이션을 만들 때 사용하는 도구이며, CCP 시험에서는 SDK의 **역할과 용도**를 아는 것이 중요합니다.

#### AWS CloudShell

> 📝 **AWS CloudShell**: AWS Management Console 내에서 바로 사용할 수 있는 **브라우저 기반 셸 환경**입니다. AWS CLI가 이미 설치되어 있으며, 별도의 설정 없이 현재 로그인한 사용자의 권한으로 명령어를 실행할 수 있습니다.

CloudShell의 장점:
- **설치 불필요**: 로컬 컴퓨터에 CLI를 설치하지 않아도 됩니다.
- **사전 설정 완료**: AWS CLI, Python, Node.js 등이 이미 설치되어 있습니다.
- **자격 증명 자동 적용**: 현재 콘솔에 로그인한 사용자의 권한이 자동으로 적용됩니다.
- **1GB 영구 스토리지**: 홈 디렉토리(`/home/cloudshell-user`)에 저장한 파일은 세션이 끝나도 유지됩니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 4.4.1: 주문 방식 비유 — Console vs CLI vs SDK**
>
> **상황**: 커피 100잔을 주문해야 합니다.
> **비교**:
> | 방식 | 비유 | AWS 도구 | 적합한 상황 |
> |------|------|----------|-------------|
> | 키오스크에서 하나씩 터치 | 메뉴 100번 반복 선택 | Console | 1~2잔 주문 (일회성 작업) |
> | 전화로 "아메리카노 100잔" | 한 마디로 완료 | CLI | 빠른 대량 작업 |
> | 카페 앱에서 "정기 배송 설정" | 앱이 매일 자동 주문 | SDK | 애플리케이션 내 자동화 |
>
> **핵심 포인트**: Console은 직관적이지만 반복 작업에 비효율적이고, CLI는 빠르지만 명령어를 알아야 하며, SDK는 프로그래밍으로 완전 자동화할 때 사용합니다.

> 🔍 **예시 4.4.2: 실무에서의 도구 선택**
>
> **상황**: "클라우드 카페" 스타트업의 개발팀이 AWS를 사용하는 상황입니다.
> **시나리오별 적합한 도구**:
> | 작업 | 적합한 도구 | 이유 |
> |------|-------------|------|
> | 처음으로 EC2 서버를 만들어 봄 | **Console** | 옵션을 하나씩 확인하며 설정할 수 있음 |
> | 매일 밤 12시에 개발 서버를 자동 종료 | **CLI** (스크립트) | 반복 작업을 명령어로 자동화 |
> | 사용자가 사진을 업로드하면 S3에 저장 | **SDK** | 앱 코드에서 직접 S3 API 호출 |
> | 현재 비용을 빠르게 확인 | **Console** | 대시보드에서 시각적으로 한눈에 확인 |
> | 10개 리전에서 동시에 서버 상태 확인 | **CLI** | 스크립트로 여러 리전을 순회 |
>
> **핵심 포인트**: 정답은 없습니다. 상황에 맞는 도구를 선택하는 것이 중요합니다. 학습 초기에는 Console을 주로 사용하고, 익숙해지면 CLI도 함께 활용하는 것이 일반적입니다.

### 미니 실습 (Hands-on Mini Lab)

> **목표**: CloudShell을 열고 간단한 AWS CLI 명령어를 실행하여 결과를 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (비용 없음)

> **Step 1**: AWS Management Console에 로그인합니다. 상단 내비게이션 바에서 **CloudShell 아이콘** (터미널 모양 `>_`)을 클릭합니다. 또는 검색 바에 `CloudShell`을 입력하고 선택합니다.
>
> **Step 2**: CloudShell이 초기화되는 동안 잠시 기다립니다 (처음 실행 시 약 1~2분 소요). "Welcome to AWS CloudShell" 메시지가 나타나면 준비 완료입니다.
>
> **Step 3** — 현재 사용자 확인: 다음 명령어를 입력하고 Enter를 누릅니다:
> ```bash
> aws sts get-caller-identity
> ```
> 결과로 현재 로그인한 사용자의 계정 ID, 사용자 ID, ARN이 표시됩니다. `"Account"` 필드에 12자리 계정 번호가 보이면 정상입니다.
>
> **Step 4** — 사용 가능한 리전 목록 확인: 다음 명령어를 입력합니다:
> ```bash
> aws ec2 describe-regions --query "Regions[].RegionName" --output table
> ```
> Chapter 3에서 배운 AWS 리전들의 목록이 표 형태로 출력됩니다.
>
> **Step 5** — S3 버킷 목록 확인: 다음 명령어를 입력합니다:
> ```bash
> aws s3 ls
> ```
> 새 계정이라면 아무것도 표시되지 않는 것이 정상입니다 (아직 버킷을 만들지 않았으므로).
>
> **Step 6** — CLI 버전 확인: 다음 명령어를 입력합니다:
> ```bash
> aws --version
> ```
> AWS CLI의 버전 정보가 표시됩니다.
>
> **확인**: `aws sts get-caller-identity` 명령에서 자신의 계정 ID가 정상적으로 표시되면 성공입니다. CloudShell 창은 우측 상단의 X 버튼으로 닫을 수 있습니다.

> 💬 **참고**: CloudShell은 현재 일부 리전에서만 사용 가능합니다. 서울 리전(ap-northeast-2)에서는 사용 가능합니다. 만약 CloudShell 아이콘이 보이지 않는다면, 리전을 서울 또는 버지니아 북부(us-east-1)로 전환해 보세요.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "Console, CLI, SDK가 서로 다른 결과를 만든다는 오해"
>
> 세 가지 접근 방식은 모두 **동일한 AWS API를 호출**합니다. Console에서 만든 리소스는 CLI로도 확인할 수 있고, CLI로 생성한 리소스는 Console에서도 보입니다. 방법이 다를 뿐 결과는 동일합니다.

> ⚠️ **주의**: "CloudShell에서 작업한 파일이 영구 보존된다고 과신하는 것"
>
> CloudShell의 홈 디렉토리(`/home/cloudshell-user`)에는 **1GB의 영구 스토리지**가 제공되지만, 이 한도를 초과하거나 120일 이상 CloudShell을 사용하지 않으면 데이터가 삭제될 수 있습니다. 중요한 파일은 별도로 백업하세요.

> ⚠️ **주의**: "CLI 명령어에서 리전을 지정하지 않는 것"
>
> AWS CLI는 기본 리전(default region) 설정에 따라 동작합니다. CloudShell에서는 현재 콘솔의 리전이 적용되지만, 로컬에 설치한 CLI에서는 `aws configure`로 설정한 리전이 사용됩니다. 의도한 리전에서 작업하고 있는지 `--region` 옵션이나 `aws configure list`로 확인하세요.

> ⚠️ **주의**: "AWS 액세스 키를 코드에 직접 작성하거나 공유하는 것"
>
> 로컬에서 CLI를 사용하려면 액세스 키가 필요합니다. 이 키를 소스 코드에 하드코딩하거나 GitHub 등에 업로드하면, **자동 스캐닝 봇에 의해 즉시 탈취**되어 막대한 비용이 발생할 수 있습니다. 액세스 키는 절대 코드에 포함하지 마세요.

### Section 요약 (Section Summary)

- AWS에 접근하는 3가지 방법: **Console**(GUI), **CLI**(명령줄), **SDK**(프로그래밍)
- 세 방법 모두 내부적으로 **AWS API**를 호출하며, 같은 결과를 만들 수 있습니다.
- **AWS CLI**는 `aws [서비스] [작업] [옵션]` 형식으로 명령어를 실행합니다.
- **AWS CloudShell**은 브라우저에서 바로 CLI를 사용할 수 있는 환경으로, 별도 설치가 불필요합니다.
- 학습 초기에는 Console을 주로 사용하되, CLI의 존재와 역할을 이해하고 있으면 됩니다.

## 4.5 AWS 프리 티어 활용과 비용 관리

> 💡 **한 줄 요약**: AWS 프리 티어는 3가지 유형(무료 체험, 12개월 무료, 항상 무료)으로 나뉘며, 그 범위와 한계를 정확히 이해해야 예기치 않은 비용을 방지할 수 있습니다.

### 직관적 이해 (Intuitive Understanding)

통신사에 가입하면 "첫 3개월 데이터 무제한!", "매달 기본 제공 데이터 5GB" 같은 혜택을 줍니다. 하지만 이 혜택에는 반드시 **조건**이 있습니다. 3개월이 지나면 요금이 부과되고, 5GB를 초과하면 추가 요금이 발생합니다.

**AWS 프리 티어(Free Tier)**도 마찬가지입니다. "무료"라는 단어 때문에 비용이 전혀 발생하지 않을 것이라고 생각하기 쉽지만, 프리 티어에는 **기간 제한**, **사용량 제한**, **서비스 제한**이 있습니다. 이 조건을 정확히 이해하지 못하면, 예기치 않은 청구서를 받게 될 수 있습니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS 프리 티어의 3가지 유형

> 📝 **AWS 프리 티어(Free Tier)**: AWS가 신규 사용자의 클라우드 학습과 체험을 돕기 위해 제공하는 무료 사용 혜택입니다. 3가지 유형으로 나뉘며, 각각 조건이 다릅니다.

| 유형 | 기간 | 특징 | 예시 |
|------|------|------|------|
| **무료 체험 (Free Trial)** | 서비스별 정해진 기간 (30일, 60일 등) | 특정 서비스를 처음 사용할 때부터 일정 기간 무료 | Amazon SageMaker: 처음 2개월 무료 |
| **12개월 무료 (12-Month Free)** | 계정 생성일로부터 12개월 | 가장 많은 핵심 서비스가 포함 | EC2 t2.micro: 월 750시간, S3: 5GB |
| **항상 무료 (Always Free)** | 제한 없음 (영구) | 사용량 한도 내에서 계속 무료 | Lambda: 월 100만 건 요청, DynamoDB: 25GB |

#### 12개월 무료 — 가장 중요한 유형

계정을 처음 만들고 12개월 동안 많은 핵심 서비스를 무료로 사용할 수 있습니다. 대표적인 서비스와 한도는 다음과 같습니다:

| 서비스 | 무료 한도 (월별) | 주의사항 |
|--------|-----------------|----------|
| **Amazon EC2** | t2.micro (또는 t3.micro) 750시간 | Linux + Windows 각각 750시간 |
| **Amazon S3** | 5GB 스토리지, 2만 GET, 2천 PUT | 스토리지 초과 시 과금 |
| **Amazon RDS** | db.t2.micro 750시간, 20GB | 단일 AZ 배포만 해당 |
| **Amazon CloudFront** | 1TB 데이터 전송 | - |
| **Amazon EBS** | 30GB 범용 SSD | EC2에 연결되는 스토리지 |

> ⚠️ **주의 — 12개월 무료의 함정**:
> - **"월 750시간"의 의미**: 한 달은 약 720~744시간입니다. t2.micro 인스턴스 **1개**를 24시간 내내 실행하면 750시간 이내입니다. 하지만 **2개**를 동시에 실행하면 합산 1,500시간이 되어 750시간을 초과합니다.
> - **12개월이 지나면**: 자동으로 정상 요금이 부과됩니다. AWS가 "12개월이 만료됩니다"라고 알려주지만, 실행 중인 리소스를 자동으로 중지해 주지는 않습니다.
> - **리전별 합산**: 서울에서 375시간, 도쿄에서 375시간을 사용하면 합산 750시간입니다. 리전별로 따로 계산되지 않습니다.

#### 항상 무료 — 계정 연차와 무관

일부 서비스는 계정을 만든 지 얼마나 됐든, 일정 사용량까지 영원히 무료입니다:

| 서비스 | 항상 무료 한도 |
|--------|---------------|
| **AWS Lambda** | 월 100만 건 요청, 320만 초 컴퓨팅 시간 |
| **Amazon DynamoDB** | 25GB 스토리지, 월 2억 건 읽기/쓰기 |
| **Amazon SNS** | 월 100만 건 게시 |
| **Amazon CloudWatch** | 기본 모니터링, 10개 사용자 정의 지표, 알람 10개 |

#### 비용 알림 설정 — 예기치 않은 과금 방지

프리 티어 범위를 실수로 초과하는 것을 방지하기 위해, **비용 알림**을 설정하는 것이 매우 중요합니다. AWS는 다음과 같은 비용 관리 도구를 제공합니다:

1. **프리 티어 사용량 알림**: 프리 티어 한도의 85%에 도달하면 이메일 알림
2. **예산(Budget) 알림**: 사용자가 설정한 금액을 초과하면 알림
3. **비용 탐색기(Cost Explorer)**: 서비스별, 기간별 비용을 시각적으로 분석

> 🔗 **연결**: Chapter 1에서 배운 클라우드의 **종량제(Pay-as-you-go)** 모델을 떠올려 봅시다. AWS는 사용한 만큼만 비용을 청구하지만, 반대로 말하면 잊어버리고 실행해 둔 리소스에 대해서도 비용이 계속 발생합니다. 이것이 비용 알림이 중요한 이유입니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 4.5.1: 통신사 데이터 요금제 비유**
>
> **상황**: 통신사의 데이터 요금제와 AWS 프리 티어를 비교해 봅시다.
> **비교**:
> | 통신사 요금제 | AWS 프리 티어 |
> |--------------|--------------|
> | "첫 3개월 데이터 무제한" | **무료 체험**: 서비스별 일정 기간 무료 |
> | "매달 기본 제공 5GB" (가입 후 1년) | **12개월 무료**: 계정 생성 후 1년간 한도 내 무료 |
> | "LTE 기본 통화 월 100분 무료" (영구) | **항상 무료**: 한도 내 영구 무료 |
> | 기본 제공량 초과 시 추가 요금 | 프리 티어 한도 초과 시 정상 요금 부과 |
>
> **핵심 포인트**: "무료"라는 말에 현혹되지 말고, **어떤 유형의 무료인지**, **한도가 얼마인지**, **기간이 언제까지인지**를 반드시 확인해야 합니다.

> 🔍 **예시 4.5.2: 프리 티어 초과로 인한 과금 시나리오**
>
> **상황**: 학습자 C 씨가 AWS 프리 티어 범위에서 실습을 하고 있습니다. 아래 시나리오에서 비용이 발생하는지 확인해 봅시다.
>
> | 시나리오 | 비용 발생 여부 | 이유 |
> |----------|---------------|------|
> | t2.micro 인스턴스 1개를 한 달 내내 실행 | ❌ 무료 | 약 720시간 < 750시간 한도 |
> | t2.micro 인스턴스 2개를 한 달 내내 실행 | ✅ **과금** | 약 1,440시간 > 750시간 한도 |
> | t2.small 인스턴스 1개를 한 달 내내 실행 | ✅ **과금** | t2.small은 프리 티어 대상이 아님 |
> | S3에 3GB 저장 | ❌ 무료 | 3GB < 5GB 한도 |
> | 계정 생성 13개월 후 t2.micro 실행 | ✅ **과금** | 12개월 무료 기간 만료 |
> | Lambda 함수를 월 50만 번 호출 | ❌ 무료 | 50만 < 100만 한도 (항상 무료) |
>
> **핵심 포인트**: 인스턴스 **유형**(t2.micro만 해당), **개수**(합산 750시간), **기간**(12개월 이내)을 모두 확인해야 합니다.

### 미니 실습 (Hands-on Mini Lab)

> **목표**: 프리 티어 사용량 알림을 활성화하고, 월 $5 예산 알림을 설정합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (비용 없음)

> **Step 1** — 프리 티어 사용량 알림 활성화:
> 1. AWS Management Console 상단의 검색 바에 `Billing`을 입력하고 **"Billing and Cost Management"**를 클릭합니다.
> 2. 좌측 메뉴에서 **"Billing preferences(결제 기본 설정)"**를 클릭합니다.
> 3. **"프리 티어 사용량 알림 수신(Receive Free Tier Usage Alerts)"** 옵션을 찾아 활성화합니다.
> 4. 알림을 받을 이메일 주소를 확인하고 **"저장(Save preferences)"**을 클릭합니다.
>
> **Step 2** — 예산(Budget) 생성:
> 1. Billing and Cost Management 좌측 메뉴에서 **"예산(Budgets)"**을 클릭합니다.
> 2. **"예산 생성(Create budget)"** 버튼을 클릭합니다.
> 3. 템플릿으로 **"제로 지출 예산(Zero spend budget)"**을 선택합니다 — 이 옵션은 비용이 $0.01이라도 발생하면 알림을 보냅니다.
> 4. 예산 이름을 입력합니다 (예: `my-zero-budget`).
> 5. 알림을 받을 이메일 주소를 입력합니다.
> 6. **"예산 생성(Create budget)"**을 클릭합니다.
>
> **확인**: 예산 목록에 방금 만든 `my-zero-budget`이 표시되면 성공입니다. 이제 $0.01이라도 비용이 발생하면 이메일로 알림을 받게 됩니다.

> 💬 **참고**: 예산(Budgets) 서비스 자체는 처음 2개 예산까지 무료입니다. 3개 이상부터 예산 1개당 월 약 $0.01의 비용이 발생합니다.

> ⚠️ **주의**:
> - 루트 사용자가 아닌 경우, Billing 관련 페이지에 접근하려면 별도의 권한 설정이 필요합니다. 이 실습에서는 루트 사용자로 수행합니다.
> - 비용 알림은 **실시간**이 아닙니다. 비용 데이터는 보통 **수 시간~24시간** 지연되어 반영됩니다. 따라서 알림만 믿지 말고, 실습이 끝나면 반드시 리소스를 삭제하는 습관을 들이세요.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의 — 프리 티어 관련 흔한 실수들**:
>
> 1. **"인스턴스를 중지(Stop)했으니 과금이 안 된다"는 착각**: EC2 인스턴스를 중지하면 컴퓨팅 비용은 멈추지만, 연결된 **EBS 볼륨(스토리지)**에 대한 비용은 계속 발생할 수 있습니다. 완전히 과금을 멈추려면 인스턴스를 **종료(Terminate)**하고 EBS 볼륨도 삭제해야 합니다.
> 2. **"실습에 사용한 리소스를 삭제하지 않고 방치"**: 실습 후 만든 리소스(EC2, RDS, EBS 등)를 삭제하지 않으면, 프리 티어 한도를 초과하여 과금이 발생할 수 있습니다.
> 3. **"한 리전에서만 리소스를 확인"**: 여러 리전에서 리소스를 만들어 놓고, 한 리전에서만 확인하면 다른 리전의 실행 중인 리소스를 놓칠 수 있습니다.
> 4. **"12개월이 지난 줄 몰랐다"**: 계정 생성일을 기억해 두고, 12개월 만료 전에 불필요한 리소스를 정리해야 합니다.

>
> 1. **Billing and Cost Management > 프리 티어(Free Tier)** 페이지에서 각 서비스별 프리 티어 사용량과 남은 한도를 확인할 수 있습니다.
> 2. 각 서비스별로 **현재 사용량 / 월간 한도** 비율이 막대 그래프로 표시됩니다.
> 3. "예측 사용량(Forecasted Usage)"이 한도를 초과할 것으로 예상되면 경고 아이콘이 표시됩니다.
>
> 또한 `aws.amazon.com/free`에서 모든 프리 티어 서비스와 한도를 확인할 수 있습니다. 새로운 서비스를 실습하기 전에 이 페이지에서 프리 티어 해당 여부를 먼저 확인하는 습관을 들이면 좋습니다.

### Section 요약 (Section Summary)

- AWS 프리 티어는 **무료 체험**(서비스별 일정 기간), **12개월 무료**(핵심 서비스 한도 내), **항상 무료**(영구 한도 내) 3가지 유형으로 나뉩니다.
- 12개월 무료의 핵심은 EC2 t2.micro **월 750시간**, S3 **5GB** 등이며, **인스턴스 유형, 개수, 리전 합산**에 주의해야 합니다.
- **비용 알림 설정**은 AWS 사용의 필수 초기 설정입니다. 제로 지출 예산을 설정하면 소액이라도 비용이 발생할 때 알림을 받을 수 있습니다.
- 실습 후에는 반드시 **리소스를 삭제**하고, 프리 티어 사용량을 주기적으로 확인하는 습관을 들여야 합니다.

<details>
<summary>정답 확인</summary>

**루트 사용자(Root User)**. 계정의 모든 AWS 서비스와 리소스에 대한 완전한 접근 권한을 가지며, 일상 업무에는 사용하지 않는 것이 보안 모범 사례입니다.

## 🧪 통합 실습: 나의 첫 AWS 환경 구축

> **시나리오**: 당신은 클라우드를 처음 학습하기 시작한 학생입니다. AWS 계정을 만들고, 보안을 설정하고, 콘솔을 탐색하고, 비용을 관리하는 — "학습을 위한 안전한 AWS 환경"을 처음부터 끝까지 구축합니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | 안전하고 비용 걱정 없는 학습용 AWS 환경을 완성한다 |
| **사용 서비스** | AWS 계정, IAM (MFA), Management Console, CloudShell, Billing |
| **활용 개념** | Section 4.1 계정 생성, Section 4.2 MFA, Section 4.3 콘솔 탐색, Section 4.4 CloudShell, Section 4.5 프리 티어와 비용 관리 |
| **선수 실습** | 없음 (이 Chapter가 첫 실습) |
| **예상 소요 시간** | 30분 |
| **비용** | Free Tier 범위 내 (비용 없음) |

### 아키텍처 다이어그램

```
[학습자] → [웹 브라우저]
              │
              ▼
        [AWS Management Console]
              │
              ├── [루트 사용자 + MFA] ──── 계정 보안 완료
              │
              ├── [콘솔 홈 대시보드] ──── 서비스 탐색 준비 완료
              │     ├── 즐겨찾기 서비스 등록
              │     └── 리전: 서울(ap-northeast-2) 설정
              │
              ├── [CloudShell] ──── CLI 환경 준비 완료
              │
              └── [Billing & Budgets] ──── 비용 알림 설정 완료
                    ├── 프리 티어 사용량 알림 활성화
                    └── 제로 지출 예산 설정

### 실습 단계

> **Phase 1: 계정 생성과 보안 설정** (활용 개념: Section 4.1, 4.2)
>
> **Step 1**: Mini Lab 4.1의 절차에 따라 AWS 계정을 생성합니다 (이미 계정이 있다면 건너뜁니다).
>
> **Step 2**: Mini Lab 4.2의 절차에 따라 루트 사용자에 MFA를 설정합니다 (이미 설정했다면 건너뜁니다).
>
> **Step 3**: 로그아웃 후 다시 로그인하여 MFA가 정상 작동하는지 확인합니다.
>
> **✅ 체크포인트**: 루트 사용자로 로그인할 때 비밀번호 입력 후 MFA 코드 입력 화면이 나타나면 Phase 1 완료.

> **Phase 2: 콘솔 환경 설정** (활용 개념: Section 4.3)
>
> **Step 1**: 우측 상단의 리전 선택기에서 **"아시아 태평양(서울) ap-northeast-2"**를 선택합니다.
>
> **Step 2**: 검색 바를 이용하여 다음 서비스를 즐겨찾기에 추가합니다:
> - `EC2` (다음 Chapter에서 사용 예정)
> - `S3` (Chapter 6에서 사용 예정)
> - `CloudWatch` (비용 모니터링에 유용)
> - `Billing` (비용 관리)
>
> **Step 3**: 좌측 상단의 "서비스" 메뉴를 열어 카테고리별로 어떤 서비스들이 있는지 둘러봅니다.
>
> **✅ 체크포인트**: 리전이 "서울"로 설정되고, 즐겨찾기에 4개 서비스가 추가되어 상단 바에 표시되면 Phase 2 완료.

> **Phase 3: CloudShell 확인** (활용 개념: Section 4.4)
>
> **Step 1**: 상단 바의 CloudShell 아이콘(`>_`)을 클릭하여 CloudShell을 엽니다.
>
> **Step 2**: 다음 명령어를 실행하여 환경이 정상인지 확인합니다:
> ```bash
> aws sts get-caller-identity
> ```
>
> **Step 3**: 자신의 계정 ID를 메모해 둡니다 (12자리 숫자).
>
> **Step 4**: 리전 목록을 확인합니다:
> ```bash
> aws ec2 describe-regions --query "Regions[].RegionName" --output table
> ```
>
> **✅ 체크포인트**: `get-caller-identity` 명령에서 자신의 계정 ID가 정상 출력되면 Phase 3 완료.

> **Phase 4: 비용 안전망 설정** (활용 개념: Section 4.5)
>
> **Step 1**: 검색 바에서 `Billing`을 검색하여 Billing and Cost Management로 이동합니다.
>
> **Step 2**: "결제 기본 설정(Billing preferences)"에서 프리 티어 사용량 알림을 활성화합니다.
>
> **Step 3**: "예산(Budgets)"에서 **제로 지출 예산**을 생성합니다 (Mini Lab 4.5 참조).
>
> **Step 4**: "프리 티어(Free Tier)" 페이지에서 현재 프리 티어 사용 현황을 확인합니다.
>
> **✅ 체크포인트**: 프리 티어 알림이 활성화되고, 제로 지출 예산이 생성되어 예산 목록에 표시되면 Phase 4 완료.

> A: 스마트폰의 시간 설정이 정확한지 확인하세요. MFA 코드는 시간 기반으로 생성되므로, 스마트폰의 시간이 실제 시간과 차이가 나면 코드가 유효하지 않을 수 있습니다. 스마트폰 설정에서 "자동 시간 설정"을 활성화하세요.

> **Q: CloudShell이 열리지 않거나, 아이콘이 보이지 않습니다.**
> A: CloudShell은 모든 리전에서 사용 가능한 것은 아닙니다. 리전을 서울(ap-northeast-2) 또는 버지니아 북부(us-east-1)로 전환해 보세요. 또한 브라우저의 팝업 차단이 활성화되어 있다면 해제해 주세요.

> **Q: Billing 페이지에 접근이 거부됩니다.**
> A: 루트 사용자가 아닌 다른 사용자로 로그인한 경우, Billing 접근 권한이 없을 수 있습니다. 이 실습에서는 루트 사용자로 로그인하여 수행하세요.

> **Q: 예산(Budget) 생성 시 "권한이 없습니다" 오류가 발생합니다.**
> A: 루트 사용자로 로그인했는지 확인하세요. 그래도 안 되면, 계정 생성 직후일 수 있으므로 몇 시간 후 다시 시도해 보세요.

> 1. 개발 서버로 t2.micro 인스턴스 1개를 24시간 운영
> 2. 테스트 데이터 저장을 위해 S3에 8GB의 파일을 업로드
> 3. 서버리스 테스트를 위해 Lambda 함수를 월 50만 회 호출
> 4. 데이터베이스 테스트를 위해 RDS db.t2.small 인스턴스 1개를 운영
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 서비스의 프리 티어 유형과 한도를 확인하고, 사용량과 비교합니다.

> **풀이**:
> 1. **무료** — EC2 t2.micro는 12개월 무료 프리 티어에 해당하며, 1개를 24시간 운영하면 월 약 720시간으로 750시간 한도 이내입니다. 계정 생성 후 6개월이므로 12개월 이내입니다.

> **Exercise 4.3**: 당신은 팀의 AWS 보안 담당자입니다. 새 팀원이 "루트 사용자로 매일 작업하면 안 되나요? MFA까지 설정했으니 충분히 안전하지 않나요?"라고 질문했습니다. 이 질문에 대해 MFA의 한계와 루트 사용자를 일상적으로 사용하면 안 되는 이유를 설명하세요.
>
> **힌트**: "최소 권한 원칙", "실수의 영향 범위", "감사(auditing)"를 키워드로 생각해 보세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: MFA는 "누가 로그인하는가"를 보호하지만, "로그인한 후 무엇을 할 수 있는가"에 대한 통제는 별개의 문제임을 설명합니다.
>
> **풀이**:
> MFA는 계정의 "입구"를 보호하는 훌륭한 보안 장치이지만, 루트 사용자를 일상적으로 사용하면 안 되는 이유는 다음과 같습니다:
>
> 1. **실수의 영향 범위가 무제한**: 루트 사용자는 모든 것을 할 수 있으므로, 실수로 중요한 리소스를 삭제하거나 설정을 변경하면 그 영향이 계정 전체에 미칩니다. 별도의 사용자를 만들어 필요한 권한만 부여하면, 실수의 영향 범위를 제한할 수 있습니다.
>
> 2. **최소 권한 원칙(Principle of Least Privilege)**: 보안의 기본 원칙은 "업무에 필요한 최소한의 권한만 부여"하는 것입니다. 루트 사용자는 이 원칙에 정면으로 위배됩니다.
>
> 3. **추적과 감사의 어려움**: 팀원 모두가 루트 사용자를 공유하면, 누가 어떤 작업을 했는지 구분할 수 없습니다. 별도의 사용자를 만들면 각 사용자의 행동을 추적할 수 있습니다.
>
> 4. **MFA의 한계**: MFA는 로그인 시의 인증을 강화하지만, 내부자의 의도적 또는 비의도적 오작동, 소셜 엔지니어링, MFA 장치 자체의 탈취 등의 위험을 완전히 제거하지는 못합니다.
>
> **결론**: MFA는 "입구의 자물쇠를 강화"하는 것이고, 별도 사용자 생성은 "방별로 다른 열쇠를 주는 것"입니다. 두 가지는 보완적이며, 둘 다 필요합니다.
>
> **확장 생각**: 이 개념은 Chapter 13(IAM)에서 "최소 권한 원칙"과 함께 더 깊이 다룹니다.
> </details>

> **풀이 전략**:
>
> **1. 루트 사용자 보안 강화**
> - MFA를 반드시 설정 (가능하다면 FIDO2 보안 키처럼 가장 강력한 수단)
> - 루트 사용자의 비밀번호를 16자 이상으로 설정
> - 루트 사용자 액세스 키를 절대 생성하지 않음
> - 루트 사용자 자격 증명은 CTO와 보안 담당자만 알도록 관리
> - 루트 사용자 사용은 계정 설정 변경, 결제 정보 변경 등 특수 작업에만 한정
>
> **2. 팀별·개인별 사용자 관리 (Chapter 13에서 자세히 학습)**
> - 50명 각각에 대해 개별 사용자를 생성
> - 개발팀, 운영팀, 경영지원팀별로 그룹을 만들어 권한을 관리
> - 각 그룹에 업무에 필요한 최소한의 권한만 부여 (최소 권한 원칙)
> - 모든 사용자에게 MFA 필수 적용
>
> **3. 비용 관리**
> - AWS Budgets에서 월 $1,000 예산 알림 설정
> - 추가로 $500, $750 등 단계별 알림도 설정
> - 프리 티어 사용량 알림 활성화
>
> **4. 행위 추적 (Chapter 18에서 자세히 학습)**
> - 개별 사용자를 만들면, 누가 어떤 작업을 했는지 기록이 남음
> - AWS CloudTrail이라는 서비스로 모든 API 호출을 기록 가능
>
> **확장 생각**: 실제 기업 환경에서는 AWS Organizations라는 서비스를 사용하여 여러 계정을 중앙에서 관리하기도 합니다. 팀별로 별도의 AWS 계정을 생성하여 비용과 권한을 완전히 분리하는 **멀티 계정 전략**이 대규모 조직에서는 일반적입니다. 이런 내용은 Chapter 17에서 다루게 됩니다.
> </details>

> **Problem 4.2**: AWS 프리 티어를 최대한 활용하여 다음 요구사항을 만족하는 "학습 프로젝트"의 인프라를 설계하세요. 12개월 동안 비용을 $0으로 유지하면서 최대한 많은 AWS 서비스를 체험하는 것이 목표입니다.
>
> - 간단한 웹 서버가 필요하다
> - 웹 서버에서 사용할 데이터를 저장할 공간이 필요하다
> - 간단한 자동화 작업(예: 매일 백업)이 필요하다
> - 서버 상태를 모니터링하고 싶다
>
> 각 요구사항에 적합한 AWS 서비스를 선택하고, 프리 티어 한도 내에서 어떻게 구성할지 설명하세요.
>
> **힌트**: Chapter 3에서 배운 AWS 서비스 카테고리와 이 Chapter에서 배운 프리 티어 정보를 조합하세요. 아직 자세히 배우지 않은 서비스도 있지만, 큰 그림에서 "이 서비스가 이런 역할을 할 것이다"라는 수준으로 설계하면 됩니다.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 프리 티어의 3가지 유형(12개월 무료 + 항상 무료)을 전략적으로 조합합니다.
>
> **풀이 전략**:
>
> | 요구사항 | 서비스 | 프리 티어 유형 | 구성 |
> |----------|--------|---------------|------|
> | 웹 서버 | **EC2** t2.micro | 12개월 무료 | 1개만 운영, 월 750시간 이내 |
> | 데이터 저장 | **S3** | 12개월 무료 | 5GB 이내로 유지 |
> | 자동화 | **Lambda** | 항상 무료 | 매일 1회 백업 = 월 30회 << 100만 건 |
> | 모니터링 | **CloudWatch** | 항상 무료 | 기본 모니터링 + 알람 10개 이내 |
>
> **세부 설계**:
> 1. EC2 t2.micro 인스턴스 1개에 웹 서버(예: Nginx)를 설치하여 운영
> 2. 웹 서버에서 사용할 정적 파일(이미지, CSS 등)은 S3에 저장 (5GB 이내)
> 3. Lambda 함수를 만들어 매일 EC2의 데이터를 S3로 백업 (월 30회 호출 → 항상 무료 한도 이내)
> 4. CloudWatch로 EC2 인스턴스의 CPU 사용률을 모니터링하고, 80% 이상이면 알람 발송
>
> **비용 검증**:
> - EC2: 1개 × 24시간 × 30일 = 720시간 < 750시간 ✅
> - S3: 5GB 이하 ✅
> - Lambda: 월 30회 << 100만 회 ✅
> - CloudWatch: 기본 모니터링 + 알람 1~2개 ✅
> - **총 비용: $0**
>
> **확장 생각**: 이 설계는 Chapter 5(EC2), Chapter 6(S3), Chapter 9(Lambda) 등에서 실제로 구축하게 됩니다. 지금은 큰 그림을 이해하는 것이 중요합니다. 또한, 웹 서버를 EC2 대신 Lambda + API Gateway로 구현하면 서버리스 아키텍처가 되어 "항상 무료" 한도 내에서 12개월 이후에도 무료로 유지할 수 있다는 점도 생각해 볼 수 있습니다.
> </details>
