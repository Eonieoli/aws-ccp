## 15.1 Amazon GuardDuty — 지능형 위협 탐지

> 💡 **한 줄 요약**: GuardDuty는 AWS 계정과 워크로드를 24시간 감시하며, 의심스러운 활동을 자동으로 탐지하여 알려주는 "AI 보안 경비원"입니다.

### 직관적 이해 (Intuitive Understanding)
집에 보안 카메라와 센서를 설치했다고 상상해 봅시다. 이 시스템은 단순히 영상을 녹화하는 것이 아니라, **AI가 영상을 분석**합니다. 평소에 출입하지 않던 시간에 누군가 들어오면 경고를 보내고, 낯선 차량이 주차장에 오래 머물면 알림을 울립니다. 여러분이 직접 모니터를 지켜보지 않아도, 시스템이 **"비정상적인 패턴"**을 자동으로 감지해 알려줍니다.

**Amazon GuardDuty**가 바로 이런 역할을 합니다. AWS 계정에서 발생하는 수많은 활동 로그를 AI와 머신러닝으로 분석하여, **악의적이거나 비정상적인 행동**을 자동으로 탐지합니다.

**왜 GuardDuty가 필요할까요?**

AWS 계정에서는 매일 수천, 수만 건의 활동이 발생합니다 — API 호출, 네트워크 연결, DNS 조회 등. 사람이 이 모든 로그를 일일이 확인하는 것은 불가능합니다. 해커가 유출된 IAM 자격 증명으로 몰래 접속하거나, EC2 인스턴스가 악성 코드에 감염되어 외부 서버와 은밀하게 통신하는 상황을 사람의 눈으로 잡아내기란 매우 어렵습니다.

> 🔗 **연결**: Chapter 13에서 IAM 자격 증명(사용자, 역할, 접근 키)을 배웠습니다. GuardDuty는 이러한 자격 증명이 **비정상적으로 사용되는 패턴**을 탐지합니다 — 예를 들어, 서울에서만 사용되던 접근 키가 갑자기 다른 나라에서 사용되는 경우를 잡아냅니다.

### 핵심 개념 설명 (Core Explanation)
> 📝 **Amazon GuardDuty**: AWS 계정, 워크로드, 데이터를 지속적으로 모니터링하여 악의적 활동과 비정상적 행동을 탐지하는 **지능형 위협 탐지 서비스**입니다. 머신러닝, 이상 탐지(anomaly detection), 통합 위협 인텔리전스를 사용하여 잠재적 위협을 식별합니다.

**GuardDuty가 분석하는 데이터 소스**:

GuardDuty는 여러분이 별도로 로그를 수집하거나 설정할 필요 없이, AWS가 이미 수집하는 다양한 데이터 소스를 자동으로 분석합니다:

| 데이터 소스 | 설명 | 탐지 예시 |
|------------|------|----------|
| **AWS CloudTrail 관리 이벤트** | AWS API 호출 기록 | 비정상적 IAM 활동, 의심스러운 콘솔 로그인 |
| **VPC Flow Logs** | 네트워크 트래픽 흐름 기록 | 알려진 악성 IP와의 통신, 비정상적 포트 스캔 |
| **DNS Logs** | DNS 쿼리 기록 | 악성 도메인과의 통신, 암호화폐 채굴 징후 |
| **S3 데이터 이벤트** | S3 버킷 접근 기록 | 비정상적 데이터 유출 패턴 |
| **EKS 감사 로그** | Kubernetes 클러스터 활동 | 컨테이너 환경에서의 의심스러운 행동 |

> 💬 **참고**: CloudTrail과 VPC Flow Logs는 AWS의 로깅 서비스입니다. 여기서는 "GuardDuty가 이러한 로그를 분석한다"는 점만 이해하면 충분합니다. CloudTrail은 Chapter 16에서, VPC Flow Logs는 Chapter 7에서 다루는 네트워크 개념과 연결됩니다.

**GuardDuty의 탐지 결과 — Finding(발견 사항)**:

GuardDuty가 위협을 감지하면 **Finding(발견 사항)**이라는 보고서를 생성합니다. 각 Finding에는 다음 정보가 포함됩니다:

- **심각도(Severity)**: Low(낮음, 1~3.9), Medium(중간, 4~6.9), High(높음, 7~8.9), Critical(심각, 9~10)
- **유형(Type)**: 어떤 종류의 위협인지 (예: `UnauthorizedAccess:IAMUser/MaliciousIPCaller`)
- **리소스(Resource)**: 관련된 AWS 리소스 (어떤 EC2 인스턴스, 어떤 IAM 사용자 등)
- **행위자(Actor)**: 위협을 발생시킨 IP 주소, 위치 정보 등

**GuardDuty가 탐지하는 대표적인 위협 유형**:

1. **정찰(Reconnaissance)**: 공격 전 정보 수집 행위 — 포트 스캔, API 호출로 리소스 목록 수집
2. **인스턴스 침해(Instance Compromise)**: EC2 인스턴스가 악성 코드에 감염되어 암호화폐 채굴, 외부 공격에 사용
3. **계정 침해(Account Compromise)**: 유출된 자격 증명으로 비정상적 API 호출, 평소와 다른 지역에서의 접근
4. **버킷 침해(S3 Compromise)**: S3 버킷의 비정상적 데이터 접근, 공개 설정 변경

> 🔍 **예시 15.1.1: 유출된 접근 키 탐지**
>
> **상황**: 개발자 김 씨가 실수로 GitHub 공개 저장소에 AWS 접근 키를 포함한 코드를 커밋했습니다. 1시간 후, 해외에서 이 접근 키를 사용한 API 호출이 발생합니다.
> **GuardDuty 탐지**: `UnauthorizedAccess:IAMUser/MaliciousIPCaller` Finding을 생성합니다. "IAM 사용자 `dev-kim`의 접근 키가 알려진 악성 IP(198.51.xxx.xxx, 국가: 러시아)에서 사용되었습니다"라는 내용을 담고 있습니다.
> **심각도**: High (8.0)
> **핵심 포인트**: GuardDuty는 위협 인텔리전스 데이터베이스(알려진 악성 IP 목록)와 행동 분석(평소 서울에서만 접속하던 사용자가 갑자기 해외에서 접속)을 결합하여 위협을 탐지합니다.

> 🔍 **예시 15.1.2: 암호화폐 채굴 감지**
>
> **상황**: 해커가 EC2 인스턴스에 침투하여 암호화폐 채굴 프로그램을 설치했습니다. 인스턴스는 겉보기에 정상 동작하지만, CPU 사용률이 높아지고 알려진 채굴 풀 서버와 통신을 시작합니다.
> **GuardDuty 탐지**: `CryptoCurrency:EC2/BitcoinTool.B!DNS` Finding을 생성합니다. "EC2 인스턴스 `i-0abc1234`가 비트코인 관련 도메인과 DNS 통신을 하고 있습니다"라는 내용입니다.
> **심각도**: High (8.0)
> **핵심 포인트**: GuardDuty는 DNS 로그를 분석하여 암호화폐 채굴 풀과의 통신 패턴을 자동으로 감지합니다. 관리자가 CPU 사용률을 모니터링하지 않더라도 위협을 조기에 발견할 수 있습니다.

**GuardDuty의 핵심 특징**:

- **에이전트 없음(Agentless)**: EC2 인스턴스에 별도의 소프트웨어를 설치할 필요가 없습니다. GuardDuty는 AWS 인프라 수준에서 데이터를 분석합니다.
- **원클릭 활성화**: AWS 콘솔에서 클릭 한 번으로 활성화할 수 있으며, 활성화 즉시 분석을 시작합니다.
- **자동 업데이트**: 위협 인텔리전스와 탐지 모델이 AWS에 의해 자동으로 업데이트됩니다.
- **다중 계정 지원**: AWS Organizations와 연동하여 여러 계정을 중앙에서 모니터링할 수 있습니다.

> 🧪 **Mini Lab 15.1: GuardDuty 활성화하기**
>
> **목표**: GuardDuty를 활성화하고 대시보드를 탐색하여, 위협 탐지 서비스의 구조를 직접 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: 30일 무료 평가판 (이후 과금 발생 가능)
>
> **Step 1**: AWS 콘솔에 로그인한 후, 상단 검색 바에 "GuardDuty"를 입력하고 **Amazon GuardDuty** 서비스를 클릭합니다.
>
> **Step 2**: GuardDuty 시작 페이지가 나타나면, **"Get Started"** 또는 **"GuardDuty 시작하기"** 버튼을 클릭합니다.
>
> **Step 3**: 서비스 역할 권한에 대한 안내가 나타납니다. GuardDuty가 CloudTrail, VPC Flow Logs, DNS 로그 등을 분석하기 위해 필요한 서비스 역할이 자동으로 생성됩니다. **"Enable GuardDuty"** 또는 **"GuardDuty 활성화"** 버튼을 클릭합니다.
>
> **Step 4**: 활성화가 완료되면 GuardDuty 대시보드로 이동합니다. 왼쪽 메뉴에서 다음 항목들을 탐색해 봅니다:
> - **Summary(요약)**: 발견 사항의 개요와 통계
> - **Findings(발견 사항)**: 탐지된 위협 목록 (새로운 계정이라면 비어 있을 수 있습니다)
> - **Settings(설정)**: 데이터 소스 구성, 무료 평가판 잔여 기간 확인
>
> **Step 5**: **Settings(설정)** 메뉴에서 현재 활성화된 데이터 소스를 확인합니다. CloudTrail, S3 보호, EKS 보호 등 각 보호 기능의 활성화 상태를 볼 수 있습니다.
>
> **확인**: GuardDuty 대시보드가 표시되고, Settings에서 "Free trial days remaining: 30" 등의 무료 평가판 정보를 확인할 수 있으면 성공입니다.
>
> 🧹 **정리**: GuardDuty는 이 Chapter의 통합 실습에서 계속 사용하므로 지금은 **활성화 상태를 유지**합니다. 모든 실습이 끝나면 반드시 비활성화합니다 (통합 실습의 리소스 정리 참고).

> ⚠️ **주의**
>
> - **GuardDuty는 "탐지"만 합니다**: GuardDuty는 위협을 **찾아내고 알려주는** 역할만 합니다. 자동으로 위협을 **차단하거나 치료하지는 않습니다**. 탐지 후 대응 조치(접근 키 비활성화, 인스턴스 격리 등)는 관리자가 수동으로 하거나 별도의 자동화를 구성해야 합니다.
> - **GuardDuty와 WAF/Shield의 차이**: Chapter 14에서 배운 WAF는 웹 애플리케이션에 들어오는 **요청을 필터링**하고, Shield는 **DDoS 공격을 방어**합니다. GuardDuty는 이러한 방어막을 우회하여 이미 침투한 위협이나 내부에서 발생하는 이상 행동을 **사후적으로 탐지**합니다. 이들은 서로 보완적인 관계입니다.
> - **Sample Findings로 테스트하기**: 실제 위협이 없어도 GuardDuty의 동작을 확인하고 싶다면, Settings에서 **"Generate sample findings"**를 클릭하여 샘플 Finding을 생성할 수 있습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 15.1.3: 내부자 위협 탐지 — 퇴사 예정 직원의 비정상적 데이터 접근**
>
> **상황**: 퇴사를 앞둔 직원이 평소에는 접근하지 않던 S3 버킷에서 대량의 고객 데이터를 다운로드하기 시작했습니다. 이 직원은 IAM 권한상 해당 버킷에 접근할 수 있지만, 지난 6개월간 한 번도 접근한 적이 없었습니다.
>
> **GuardDuty 탐지**: `Exfiltration:S3/AnomalousBehavior` Finding을 생성합니다. "IAM 사용자 `employee-lee`가 평소 패턴과 다르게 S3 버킷 `customer-data-prod`에서 대량의 `GetObject` API를 호출했습니다"라는 내용입니다.
>
> **핵심 포인트**: GuardDuty는 단순히 "허용된 접근인가"를 확인하는 것이 아니라, "평소 행동 패턴과 다른가"를 분석합니다. IAM 정책상 허용된 접근이라도 비정상적 패턴이면 탐지합니다. 이것이 행동 기반 탐지의 핵심입니다.

> 🔍 **예시 15.1.4: 일상 비유 — 은행의 이상 거래 탐지 시스템**
>
> **상황**: 여러분의 신용카드로 평소에는 서울에서만 결제하는데, 갑자기 해외에서 고액 결제가 발생합니다. 은행은 즉시 "본인이 결제한 것이 맞습니까?"라는 알림을 보냅니다.
>
> 은행의 이상 거래 탐지 시스템이 GuardDuty와 같은 원리입니다. 카드 비밀번호가 맞더라도(인증 성공), 평소 패턴과 다르면(행동 이상) 경고를 발생시킵니다. GuardDuty도 마찬가지로, IAM 자격 증명이 유효하더라도 사용 패턴이 비정상적이면 Finding을 생성합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: GuardDuty를 활성화하면 **자동으로 위협이 차단된다**고 오해하는 경우가 많습니다. GuardDuty는 탐지(Detection)만 수행하며, 차단(Prevention)은 별도로 구성해야 합니다. 탐지 후 자동 대응이 필요하면 Amazon EventBridge와 Lambda를 연동하여 자동화해야 합니다.

> ⚠️ **주의**: GuardDuty와 **AWS Config**를 혼동하지 마세요. GuardDuty는 "누군가 수상한 행동을 하고 있는가"를 실시간으로 탐지하고, AWS Config는 "리소스 설정이 규칙을 준수하는가"를 점검합니다. 목적이 완전히 다릅니다.

> ⚠️ **주의**: GuardDuty의 Finding이 생성되었다고 **항상 실제 공격을 의미하는 것은 아닙니다**. 오탐(False Positive)이 발생할 수 있으므로, Finding을 확인한 후 Detective 등으로 추가 조사하여 실제 위협인지 판단해야 합니다.

> ⚠️ **주의**: GuardDuty는 **리전(Region)별로 활성화**해야 합니다. 서울 리전에서만 활성화하면 다른 리전에서 발생하는 위협은 탐지되지 않습니다. 모든 사용 중인 리전에서 활성화하는 것이 권장됩니다.

### Section 요약 (Section Summary)

- **GuardDuty**는 AWS 계정과 워크로드에 대한 **지능형 위협 탐지 서비스**로, 머신러닝과 위협 인텔리전스를 활용합니다.
- CloudTrail 이벤트, VPC Flow Logs, DNS 로그, S3 데이터 이벤트 등을 **에이전트 없이 자동으로 분석**합니다.
- 위협을 감지하면 심각도가 포함된 **Finding(발견 사항)**을 생성하여 관리자에게 알립니다.
- 원클릭 활성화가 가능하며, **탐지만 수행**하고 자동 차단은 하지 않습니다 (탐지 ≠ 차단).

## 15.2 Amazon Inspector — EC2, Lambda, 컨테이너 취약점 자동 평가

> 💡 **한 줄 요약**: Inspector는 EC2 인스턴스, Lambda 함수, 컨테이너 이미지에 숨어 있는 **소프트웨어 취약점과 보안 노출**을 자동으로 찾아내는 "보안 검진 서비스"입니다.

### 직관적 이해 (Intuitive Understanding)
자동차를 오래 타면 정기 검사를 받습니다. 전문가가 브레이크 패드 마모 상태, 타이어 공기압, 엔진 오일 상태를 점검하고 "이 부분이 위험하니 교체하세요"라고 알려줍니다. 여러분이 자동차 정비에 전문가가 아니어도, 검사 보고서를 보고 어디가 문제인지 알 수 있습니다.

**Amazon Inspector**는 AWS에서 실행되는 서버(EC2)와 코드(Lambda)에 대한 **보안 정기 검사**를 수행합니다. 소프트웨어에 알려진 보안 취약점(CVE)이 있는지, 네트워크 설정이 위험하게 열려 있는지를 자동으로 점검하고 보고서를 생성합니다.

**왜 Inspector가 필요할까요?**

EC2 인스턴스에 설치된 운영체제와 소프트웨어에는 시간이 지나면서 보안 취약점이 발견됩니다. 예를 들어, 2021년 12월에 발견된 **Log4Shell** 취약점은 전 세계 수백만 대의 서버에 영향을 미쳤습니다. 이런 취약점이 발견되었을 때, "내 서버에도 이 취약점이 있나?"를 빠르게 확인할 수 있어야 합니다. Inspector가 바로 이 역할을 합니다.

> 🔗 **연결**: Chapter 14에서 보안 그룹과 네트워크 ACL을 통한 **네트워크 수준 보안**을 배웠습니다. Inspector는 네트워크 방어막 안쪽에 있는 **소프트웨어 자체의 취약점**을 점검합니다. 아무리 방화벽을 잘 설정해도, 서버 소프트웨어 자체에 구멍이 있으면 공격자가 그 구멍을 통해 침투할 수 있습니다.

### 핵심 개념 설명 (Core Explanation)
> 📝 **Amazon Inspector**: AWS 워크로드(EC2 인스턴스, Lambda 함수, 컨테이너 이미지)를 지속적으로 스캔하여 **소프트웨어 취약점(CVE)**과 **의도하지 않은 네트워크 노출**을 자동으로 발견하는 취약점 관리 서비스입니다.

> 📝 **CVE (Common Vulnerabilities and Exposures)**: 공개적으로 알려진 소프트웨어 보안 취약점에 부여되는 고유 식별 번호입니다. 예를 들어, `CVE-2021-44228`은 Log4Shell 취약점을 지칭합니다. Inspector는 이 CVE 데이터베이스를 기반으로 취약점을 찾습니다.

**Inspector가 스캔하는 대상**:

| 스캔 대상 | 점검 내용 | 동작 방식 |
|----------|----------|----------|
| **EC2 인스턴스** | OS 패키지 취약점, 네트워크 도달 가능성 | SSM Agent를 통해 설치된 패키지 정보 수집 |
| **Lambda 함수** | 함수 코드 내 라이브러리 취약점 | 배포된 코드 패키지 자동 분석 |
| **ECR 컨테이너 이미지** | 컨테이너 이미지 내 패키지 취약점 | 이미지 푸시 시 자동 스캔 |

> 💬 **참고**: ECR(Elastic Container Registry)은 Docker 컨테이너 이미지를 저장하는 서비스입니다. 컨테이너 개념은 Chapter 9에서 다루었습니다. 여기서는 "Inspector가 컨테이너 이미지도 검사할 수 있다"는 점만 알면 충분합니다.

**Inspector의 핵심 특징 — "지속적이고 자동적인 스캔"**:

이전 버전의 Inspector(Inspector Classic)는 스캔을 수동으로 실행해야 했지만, 현재의 **Amazon Inspector v2**는 **지속적인 자동 스캔**을 수행합니다:

- **자동 검색**: Inspector를 활성화하면, 계정 내 모든 EC2 인스턴스와 Lambda 함수를 **자동으로 검색**하여 스캔 대상에 포함합니다. 새로운 인스턴스가 시작되면 자동으로 스캔에 추가됩니다.
- **이벤트 기반 재스캔**: 새로운 CVE가 발표되거나, EC2 인스턴스에 새 패키지가 설치되거나, Lambda 함수가 업데이트되면 **자동으로 재스캔**합니다.
- **위험도 점수(Inspector Score)**: 각 발견 사항에 CVSS(Common Vulnerability Scoring System) 기반의 점수를 부여합니다. Inspector는 여기에 네트워크 도달 가능성 같은 환경적 요소를 고려하여 **맥락화된 점수**를 제공합니다.

> 🔍 **예시 15.2.1: 오래된 패키지의 취약점 발견**
>
> **상황**: 회사의 웹 서버(EC2 인스턴스)에 6개월 전에 설치한 OpenSSL 라이브러리가 있습니다. 최근 이 버전에서 심각한 보안 취약점(CVE-2024-XXXX)이 발견되었습니다.
> **Inspector 탐지**: Inspector가 자동으로 재스캔을 수행하여 "EC2 인스턴스 `i-0abc1234`에 설치된 `openssl 1.1.1k`에 CVE-2024-XXXX 취약점이 존재합니다. Inspector Score: 9.8 (Critical)"이라는 Finding을 생성합니다.
> **권장 조치**: "openssl을 1.1.1w 이상으로 업데이트하세요"라는 구체적인 해결 방법도 함께 제공합니다.
> **핵심 포인트**: 새로운 CVE가 발표되면 Inspector가 기존 인스턴스를 자동으로 재스캔하므로, 관리자가 "우리 서버에 이 취약점이 있나?"를 일일이 확인할 필요가 없습니다.

> 🔍 **예시 15.2.2: 네트워크 도달 가능성 분석**
>
> **상황**: 개발자가 테스트 목적으로 EC2 인스턴스의 보안 그룹에 SSH 포트(22번)를 모든 IP(0.0.0.0/0)에 개방해 놓고 닫는 것을 잊었습니다.
> **Inspector 탐지**: "EC2 인스턴스 `i-0abc1234`의 포트 22가 인터넷에서 직접 도달 가능합니다 (Network Reachability)"라는 Finding을 생성합니다.
> **핵심 포인트**: Inspector는 소프트웨어 취약점뿐 아니라, 네트워크 설정이 위험하게 열려 있는 경우도 탐지합니다. 보안 그룹 설정의 실수를 자동으로 잡아낼 수 있습니다.

> ⚠️ **주의**
>
> - **GuardDuty vs Inspector 혼동 금지**: **GuardDuty**는 "누군가 내 계정에서 수상한 **행동**을 하고 있나?"를 탐지합니다 (행동 분석). **Inspector**는 "내 서버에 알려진 보안 **구멍**이 있나?"를 점검합니다 (취약점 스캔). 비유하자면, GuardDuty는 "도둑이 들어왔다"를 알려주는 CCTV이고, Inspector는 "자물쇠가 낡아서 쉽게 열린다"를 알려주는 보안 진단사입니다.
> - **Inspector는 패치를 자동으로 적용하지 않습니다**: Inspector는 취약점을 찾아내고 해결 방법을 안내하지만, 실제로 소프트웨어를 업데이트하는 것은 관리자의 책임입니다 (자동화하려면 AWS Systems Manager Patch Manager를 활용할 수 있습니다).

### 상세 예시 (Detailed Examples)

> 🔍 **예시 15.2.3: Log4Shell 취약점 긴급 대응**
>
> **상황**: 2021년 12월, 전 세계적으로 Log4Shell(CVE-2021-44228) 취약점이 공개되었습니다. 회사는 수십 대의 EC2 인스턴스를 운영 중이며, 어떤 인스턴스에 취약한 Log4j 라이브러리가 설치되어 있는지 즉시 파악해야 합니다.
>
> **Inspector 활용**: Inspector가 이미 활성화되어 있었기 때문에, 새로운 CVE가 발표되자마자 모든 EC2 인스턴스를 자동으로 재스캔합니다. 30분 이내에 "52개 인스턴스 중 12개에서 Log4j 2.14.1이 발견되었으며, CVE-2021-44228에 취약합니다"라는 결과가 나옵니다. 각 Finding에는 영향받는 인스턴스 ID와 "Log4j를 2.17.0 이상으로 업데이트하세요"라는 권장 조치가 포함됩니다.
>
> **핵심 포인트**: Inspector 없이는 관리자가 수십 대의 서버에 일일이 접속하여 Log4j 버전을 확인해야 합니다. Inspector의 자동 재스캔 기능 덕분에, 긴급 취약점 발표 시 영향 범위를 빠르게 파악할 수 있습니다.

> 🔍 **예시 15.2.4: 일상 비유 — 자동차 리콜 알림**
>
> **상황**: 자동차 제조사가 특정 모델의 브레이크 결함을 발견하여 리콜을 발표했습니다. 차량 등록 시스템이 자동으로 해당 모델 소유자에게 "귀하의 차량이 리콜 대상입니다. 가까운 서비스 센터에서 무상 수리를 받으세요"라는 알림을 보냅니다.
>
> Inspector의 자동 재스캔이 바로 이 리콜 시스템과 같습니다. 새로운 CVE(리콜)가 발표되면, Inspector가 자동으로 모든 인스턴스(등록된 차량)를 확인하여 영향받는 대상을 찾아내고 해결 방법을 안내합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: Inspector가 취약점을 발견했다고 해서 **자동으로 패치가 적용되는 것은 아닙니다**. Inspector는 "발견"만 하며, 실제 패치 적용은 관리자가 직접 하거나 AWS Systems Manager Patch Manager로 자동화해야 합니다.

> ⚠️ **주의**: Inspector와 GuardDuty의 역할을 혼동하지 마세요. **Inspector는 "취약점(구멍)이 존재하는가"**를 점검하고, **GuardDuty는 "누군가 수상한 행동을 하고 있는가"**를 탐지합니다. Inspector는 사전 점검(예방적), GuardDuty는 실시간 감시(탐지적) 성격입니다.

> ⚠️ **주의**: Inspector v2는 **SSM Agent가 설치된 EC2 인스턴스만** OS 패키지 취약점을 스캔할 수 있습니다. SSM Agent가 없는 인스턴스는 네트워크 도달 가능성 분석만 수행되며, 소프트웨어 취약점은 확인되지 않습니다.

> ⚠️ **주의**: Inspector의 Finding 점수가 높다고 해서 반드시 즉시 위험한 것은 아닙니다. **네트워크 도달 가능성**을 함께 고려해야 합니다. 예를 들어, 프라이빗 서브넷에 있어서 인터넷에서 접근할 수 없는 인스턴스의 취약점은, 퍼블릭 서브넷 인스턴스의 동일한 취약점보다 실제 위험도가 낮습니다.

### Section 요약 (Section Summary)

- **Inspector**는 EC2, Lambda, 컨테이너 이미지의 **소프트웨어 취약점(CVE)과 네트워크 노출을 자동으로 스캔**합니다.
- 활성화하면 리소스를 자동 검색하고, 새로운 CVE 발표나 변경 사항이 있을 때 **자동으로 재스캔**합니다.
- 발견 사항에 **맥락화된 위험도 점수**를 부여하여 우선순위 판단을 돕습니다.
- **GuardDuty가 "행동 이상"을 탐지**한다면, **Inspector는 "취약점 존재"를 점검**하는 역할입니다.

## 15.3 Amazon Macie — S3 데이터에서 민감 정보 자동 탐지

> 💡 **한 줄 요약**: Macie는 S3 버킷에 저장된 데이터 중 개인정보(PII), 금융 정보 등 **민감한 데이터를 자동으로 발견하고 분류**하는 데이터 보안 서비스입니다.

### 직관적 이해 (Intuitive Understanding)
회사에서 매일 수천 개의 서류를 파일 캐비닛에 보관한다고 상상해 봅시다. 어느 날 개인정보보호 담당자가 묻습니다: "우리 캐비닛에 고객의 주민등록번호나 신용카드 번호가 적힌 서류가 어디에 있나요?" 수천 개의 서류를 하나하나 열어보며 확인하려면 며칠이 걸릴 것입니다.

**Amazon Macie**는 이 작업을 자동화합니다. S3 버킷(AWS의 파일 캐비닛)에 저장된 수백만 개의 파일을 머신러닝으로 분석하여, **민감한 데이터가 포함된 파일을 자동으로 찾아냅니다**.

**왜 Macie가 필요할까요?**

기업들은 S3에 방대한 양의 데이터를 저장합니다 — 로그 파일, 백업 데이터, 보고서, 고객 데이터 등. 이 중에는 실수로 또는 의도치 않게 **민감한 데이터**가 포함될 수 있습니다. 예를 들어:

- 개발자가 테스트용으로 실제 고객 데이터를 S3에 업로드
- 로그 파일에 실수로 사용자의 신용카드 번호가 기록됨
- 마케팅 팀이 고객 이메일 주소가 포함된 CSV 파일을 업로드

GDPR, 개인정보보호법 같은 규정에 따라 기업은 민감한 데이터가 **어디에 있는지 파악**하고 **적절히 보호**해야 합니다. Macie가 이 "데이터 지도"를 자동으로 만들어 줍니다.

> 🔗 **연결**: Chapter 14에서 배운 **암호화(KMS)**는 데이터를 보호하는 "방법"이었습니다. Macie는 "어떤 데이터를 보호해야 하는지"를 찾아주는 역할입니다. 먼저 Macie로 민감 데이터의 위치를 파악한 후, KMS 암호화 등으로 보호하는 것이 올바른 순서입니다.

### 핵심 개념 설명 (Core Explanation)
> 📝 **Amazon Macie**: S3 버킷에 저장된 데이터를 머신러닝과 패턴 매칭으로 분석하여 **민감한 데이터(PII, 금융 정보 등)**를 자동으로 검색, 분류, 보호하는 데이터 보안 및 프라이버시 서비스입니다.

> 📝 **PII (Personally Identifiable Information, 개인 식별 정보)**: 특정 개인을 식별할 수 있는 정보. 이름, 주소, 전화번호, 이메일, 주민등록번호, 여권 번호, 신용카드 번호 등이 포함됩니다.

**Macie가 탐지하는 민감 데이터 유형**:

| 카테고리 | 예시 |
|---------|------|
| **개인 식별 정보 (PII)** | 이름, 주소, 전화번호, 이메일, 주민등록번호 |
| **금융 정보** | 신용카드 번호, 은행 계좌 번호 |
| **자격 증명** | AWS 접근 키, 비밀 키, API 토큰 |
| **건강 정보** | 건강보험 번호, 의료 기록 ID |

**Macie의 두 가지 핵심 기능**:

**1. S3 버킷 보안 상태 평가**

Macie는 모든 S3 버킷의 보안 설정을 자동으로 분석하여 다음을 알려줍니다:
- 버킷이 **퍼블릭(공개)**으로 설정되어 있는지
- 버킷에 **암호화가 적용**되어 있는지
- 버킷의 **접근 정책**이 다른 AWS 계정이나 외부에 공유되어 있는지

**2. 민감 데이터 자동 검색 (Sensitive Data Discovery)**

S3 버킷의 파일 내용을 실제로 분석하여 민감한 데이터를 찾습니다:
- **관리형 데이터 식별자(Managed Data Identifiers)**: AWS가 사전에 정의한 100개 이상의 민감 데이터 패턴 (신용카드 번호 형식, 주민등록번호 형식 등)
- **사용자 정의 데이터 식별자(Custom Data Identifiers)**: 기업이 직접 정의한 데이터 패턴 (예: 사내 사원 번호 형식)

> 🔍 **예시 15.3.1: 실수로 업로드된 고객 데이터 발견**
>
> **상황**: 마케팅 팀이 캠페인 분석을 위해 `marketing-data-2024` S3 버킷에 CSV 파일을 업로드했습니다. 이 파일에는 고객 이름, 이메일, 전화번호가 포함되어 있었지만, 팀원들은 이를 인지하지 못했습니다.
> **Macie 탐지**: Macie가 정기 스캔을 수행하면서 해당 CSV 파일에서 2,340건의 이메일 주소, 1,890건의 전화번호, 1,200건의 이름 패턴을 발견합니다. "버킷 `marketing-data-2024`의 파일 `customer_campaign_q4.csv`에 PII(개인 식별 정보)가 포함되어 있습니다"라는 Finding을 생성합니다.
> **핵심 포인트**: Macie 없이는 수백 개의 S3 버킷에 흩어져 있는 수천 개의 파일에서 민감 데이터를 찾는 것이 사실상 불가능합니다. Macie는 이 과정을 자동화하여 데이터 거버넌스를 실현합니다.

> 🔍 **예시 15.3.2: 공개 S3 버킷의 민감 데이터 경고**
>
> **상황**: 개발팀이 정적 웹사이트 호스팅용으로 S3 버킷을 퍼블릭으로 설정했습니다. 이 버킷에 테스트 과정에서 실수로 데이터베이스 백업 파일이 업로드되었고, 이 파일에는 사용자 비밀번호 해시와 이메일이 포함되어 있습니다.
> **Macie 탐지**: Macie는 (1) 버킷이 퍼블릭 상태이며 (2) 버킷 내 파일에 자격 증명 정보와 PII가 포함되어 있다는 **이중 경고**를 생성합니다. 이는 "민감한 데이터가 인터넷에 공개되어 있다"는 최고 수준의 위험 상황입니다.
> **핵심 포인트**: Macie는 단순히 "이 파일에 민감 데이터가 있다"는 것만이 아니라, 그 데이터가 **얼마나 위험한 상태에 놓여 있는지**(퍼블릭 버킷, 암호화 미적용 등)까지 종합적으로 평가합니다.

> ⚠️ **주의**
>
> - **Macie는 S3만 스캔합니다**: Macie는 현재 **S3 버킷에 저장된 데이터만** 분석합니다. RDS 데이터베이스, DynamoDB 테이블 등 다른 데이터 저장소는 스캔하지 않습니다.
> - **비용에 주의하세요**: Macie의 민감 데이터 검색은 **검사한 데이터 양에 비례하여 비용**이 발생합니다. 대용량 버킷을 스캔할 경우 비용이 상당할 수 있으므로, 처음에는 중요한 버킷만 선택적으로 스캔하는 것이 좋습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 15.3.3: 로그 파일에 숨겨진 신용카드 번호 발견**
>
> **상황**: 전자상거래 회사의 결제 시스템이 디버그 모드로 운영되면서, 애플리케이션 로그에 고객의 신용카드 번호가 평문으로 기록되었습니다. 이 로그 파일은 매일 자동으로 S3 버킷 `app-logs-prod`에 업로드됩니다. 운영팀은 로그에 신용카드 정보가 포함되어 있다는 사실을 인지하지 못했습니다.
>
> **Macie 탐지**: Macie가 정기 스캔을 수행하면서 `app-logs-prod` 버킷의 로그 파일에서 신용카드 번호 패턴(16자리 숫자, Luhn 알고리즘 일치)을 15,000건 이상 발견합니다. "PCI 데이터(신용카드 번호)가 암호화되지 않은 상태로 저장되어 있습니다"라는 Critical 심각도 Finding을 생성합니다.
>
> **핵심 포인트**: 민감 데이터는 의도적으로 저장되는 것만이 아닙니다. 로그, 백업, 임시 파일 등에 **의도치 않게 포함**되는 경우가 매우 흔합니다. Macie는 이런 "숨겨진" 민감 데이터를 자동으로 찾아냅니다.

> 🔍 **예시 15.3.4: 일상 비유 — 이사할 때 짐 정리**
>
> **상황**: 이사를 앞두고 모든 상자를 열어 확인하니, 오래된 상자 속에 은행 통장, 여권 사본, 비밀번호가 적힌 메모지가 아무런 보호 없이 들어 있었습니다. 평소에는 이 상자를 열어볼 일이 없어서 몰랐던 것입니다.
>
> Macie는 S3 버킷(상자)을 하나하나 열어서 민감한 정보(통장, 여권 사본)가 보호 없이 저장되어 있는지 확인하는 "정리 전문가"와 같습니다. 데이터가 많아질수록 수동으로는 확인이 불가능하며, Macie가 이를 자동화합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: Macie는 **S3 버킷만 스캔**합니다. RDS 데이터베이스, DynamoDB, EBS 볼륨 등 다른 스토리지 서비스에 저장된 민감 데이터는 Macie로 탐지할 수 없습니다. 시험에서 "데이터베이스에 저장된 PII를 탐지하려면?"이라는 질문에 Macie를 답하면 오답입니다.

> ⚠️ **주의**: Macie를 **모든 S3 버킷에 대해 전체 스캔**하면 비용이 크게 증가할 수 있습니다. 스캔 대상 버킷을 선별하거나, 샘플링 비율을 조정하여 비용을 관리하는 것이 중요합니다.

> ⚠️ **주의**: Macie의 "S3 버킷 보안 상태 평가" 기능과 "민감 데이터 검색" 기능을 혼동하지 마세요. 보안 상태 평가는 버킷의 **설정(퍼블릭 여부, 암호화 여부)**을 확인하는 것이고, 민감 데이터 검색은 버킷 안의 **파일 내용**을 실제로 분석하는 것입니다. 두 기능은 별도로 동작합니다.

> ⚠️ **주의**: Macie가 민감 데이터를 발견했다고 해서 **자동으로 데이터를 암호화하거나 삭제하지 않습니다**. Macie는 발견 및 보고만 하며, 후속 조치(암호화 적용, 접근 제한, 데이터 삭제 등)는 관리자가 직접 수행해야 합니다.

### Section 요약 (Section Summary)

- **Macie**는 S3 버킷에 저장된 데이터에서 **민감한 정보(PII, 금융 정보, 자격 증명 등)를 자동으로 탐지**합니다.
- S3 버킷의 **보안 상태(퍼블릭 여부, 암호화 여부)**도 함께 평가합니다.
- AWS가 제공하는 100개 이상의 **관리형 데이터 식별자**와 사용자가 직접 정의하는 **사용자 정의 데이터 식별자**를 사용합니다.
- Macie는 **S3 전용** 서비스이며, 스캔 데이터량에 따라 비용이 발생합니다.

## 15.4 AWS Security Hub — 보안 발견 사항 통합, 보안 표준 점검

> 💡 **한 줄 요약**: Security Hub는 GuardDuty, Inspector, Macie 등 여러 보안 서비스의 발견 사항을 **한곳에 모아 보여주는 통합 보안 대시보드**이자, 보안 모범 사례 준수 여부를 자동으로 점검하는 서비스입니다.

### 직관적 이해 (Intuitive Understanding)
대형 건물의 통합 보안 관제실을 떠올려 봅시다. 건물에는 CCTV, 화재 감지기, 출입 통제 시스템, 침입 탐지 센서 등 여러 보안 시스템이 설치되어 있습니다. 이 시스템들이 각각 별도의 모니터와 알림을 보낸다면, 경비원은 여러 화면을 동시에 지켜봐야 하고 중요한 경보를 놓칠 수 있습니다.

**통합 관제실**은 이 모든 시스템의 정보를 **하나의 화면**에 모아 보여줍니다. "CCTV에서 1층 후문에 비정상적 움직임 감지, 동시에 출입 카드 미인증 접근 시도 2건" — 이렇게 여러 소스의 정보를 종합하면, 개별 시스템만 볼 때보다 **상황을 훨씬 정확하게 파악**할 수 있습니다.

**AWS Security Hub**가 바로 이 통합 관제실입니다.

**왜 Security Hub가 필요할까요?**

앞에서 배운 GuardDuty, Inspector, Macie는 각각 훌륭한 보안 도구이지만, 발견 사항이 **서비스별로 흩어져** 있습니다. 보안 담당자가 GuardDuty 대시보드, Inspector 대시보드, Macie 대시보드를 일일이 확인해야 한다면, 비효율적이고 중요한 위협을 놓칠 수 있습니다. Security Hub는 이 모든 정보를 **하나의 장소**에서 확인할 수 있게 해 줍니다.

### 핵심 개념 설명 (Core Explanation)
> 📝 **AWS Security Hub**: AWS 보안 서비스와 서드파티 도구의 보안 발견 사항(Finding)을 **중앙에서 수집, 정리, 우선순위 지정**하는 통합 보안 관리 서비스입니다. 또한 **보안 모범 사례 표준**에 대한 자동화된 점검을 수행합니다.

**Security Hub의 두 가지 핵심 역할**:

**역할 1: 보안 발견 사항 통합 (Finding Aggregation)**

Security Hub는 다음과 같은 서비스들로부터 Finding을 자동으로 수집합니다:

```
┌────────────┐   ┌────────────┐   ┌────────────┐
│ GuardDuty  │   │ Inspector  │   │   Macie    │
│ (위협 탐지) │   │ (취약점 스캔)│   │ (데이터 탐지)│
└─────┬──────┘   └─────┬──────┘   └─────┬──────┘
      │                │                │
      └────────┬───────┴────────────────┘
               ▼
      ┌────────────────┐
      │  Security Hub  │
      │ (통합 대시보드)  │
      │                │
      │ · 우선순위 정렬  │
      │ · 필터링/검색   │
      │ · 자동화 연동   │
      └────────────────┘
```

모든 Finding은 **ASFF(AWS Security Finding Format)**라는 표준 형식으로 정규화됩니다. 덕분에 서로 다른 서비스의 발견 사항을 동일한 기준으로 비교하고 정렬할 수 있습니다.

**역할 2: 보안 표준 자동 점검 (Security Standards)**

Security Hub는 보안 모범 사례에 따라 AWS 계정의 설정 상태를 **자동으로 점검**합니다. 주요 보안 표준:

| 보안 표준 | 설명 |
|----------|------|
| **AWS Foundational Security Best Practices (FSBP)** | AWS가 정의한 기본 보안 모범 사례 |
| **CIS AWS Foundations Benchmark** | Center for Internet Security에서 정의한 AWS 보안 기준 |
| **PCI DSS** | 신용카드 데이터 보안 표준 |

예를 들어, Security Hub는 다음과 같은 항목을 자동으로 점검합니다:
- "루트 계정에 MFA가 활성화되어 있는가?"
- "S3 버킷에 퍼블릭 접근이 차단되어 있는가?"
- "CloudTrail이 활성화되어 있는가?"
- "보안 그룹에서 불필요한 포트가 열려 있지 않은가?"

각 점검 항목은 **통과(PASSED)**, **실패(FAILED)**, **경고(WARNING)** 상태로 표시되며, 전체 점수를 백분율로 보여줍니다.

> 🔍 **예시 15.4.1: 통합 보안 현황 파악**
>
> **상황**: 보안 팀장이 월요일 아침, 지난 주말 동안 AWS 계정의 보안 상태를 확인하려 합니다.
> **Security Hub 활용**: Security Hub 대시보드에 접속하면, 주말 동안 발생한 모든 보안 이벤트를 한눈에 볼 수 있습니다:
> - GuardDuty에서 감지한 2건의 의심스러운 API 호출 (심각도: Medium)
> - Inspector에서 발견한 5건의 EC2 취약점 (심각도: High 1건, Medium 4건)
> - Macie에서 발견한 1건의 S3 민감 데이터 노출 (심각도: High)
>
> 보안 팀장은 심각도 "High"를 기준으로 필터링하여, 가장 먼저 처리해야 할 2건(Inspector 취약점 1건 + Macie 데이터 노출 1건)에 집중할 수 있습니다.
> **핵심 포인트**: Security Hub 없이는 3개 서비스의 대시보드를 각각 확인하고 직접 심각도를 비교해야 합니다. Security Hub는 **"지금 가장 급한 보안 문제가 무엇인가?"**라는 질문에 빠르게 답할 수 있게 해 줍니다.

> 🔍 **예시 15.4.2: 보안 표준 점검 결과 확인**
>
> **상황**: 회사가 새로운 AWS 계정을 생성하고 초기 설정을 완료했습니다. 보안 팀은 이 계정이 보안 모범 사례를 따르고 있는지 확인하고 싶습니다.
> **Security Hub 활용**: AWS Foundational Security Best Practices 표준을 활성화하면, 자동으로 약 200개 항목을 점검합니다. 결과:
> - 전체 점수: 72% (200개 중 144개 통과)
> - 실패 항목 예시: "루트 계정에 하드웨어 MFA가 설정되지 않음", "기본 VPC의 보안 그룹이 모든 트래픽을 허용함", "S3 버킷에 서버 측 암호화가 설정되지 않음"
>
> 각 실패 항목에는 **해결 방법(Remediation)**이 링크와 함께 제공되어, 무엇을 어떻게 고쳐야 하는지 바로 확인할 수 있습니다.
> **핵심 포인트**: Security Hub의 보안 표준 점검은 "내 계정이 보안 모범 사례를 얼마나 따르고 있는가?"를 **객관적인 점수**로 보여줍니다. 마치 건강 검진에서 각 항목의 정상/이상을 확인하는 것과 같습니다.

> 🧪 **Mini Lab 15.4: Security Hub 활성화 및 보안 점수 확인하기**
>
> **목표**: Security Hub를 활성화하고 보안 표준 점검 결과를 확인합니다.
> **예상 소요 시간**: 15분 (점검 결과가 완전히 생성되기까지는 최대 2시간이 걸릴 수 있습니다)
> **비용**: 30일 무료 평가판
>
> **Step 1**: AWS 콘솔 상단 검색 바에 "Security Hub"를 입력하고 **AWS Security Hub** 서비스를 클릭합니다.
>
> **Step 2**: "Go to Security Hub" 또는 "Security Hub로 이동" 버튼을 클릭합니다.
>
> **Step 3**: 보안 표준(Security standards) 선택 화면이 나타납니다. 다음 표준이 기본으로 선택되어 있습니다:
> - **AWS Foundational Security Best Practices v1.0.0** — 체크 유지
> - **CIS AWS Foundations Benchmark v1.2.0** — 체크 유지
> - PCI DSS 등 다른 표준은 필요에 따라 선택 (실습에서는 기본값 유지)
>
> **Step 4**: **"Enable Security Hub"** 버튼을 클릭하여 활성화합니다.
>
> **Step 5**: Security Hub 대시보드가 나타납니다. 왼쪽 메뉴에서 다음을 확인합니다:
> - **Summary(요약)**: 전체 보안 현황 개요
> - **Security standards(보안 표준)**: 활성화된 보안 표준과 현재 점수 (처음에는 점검이 진행 중이므로 점수가 낮거나 0%로 보일 수 있습니다)
> - **Findings(발견 사항)**: GuardDuty 등 다른 서비스에서 수집된 발견 사항 (이전 Mini Lab에서 GuardDuty를 활성화했다면, GuardDuty의 Finding이 여기에도 나타납니다)
>
> **Step 6**: **Security standards** 메뉴에서 "AWS Foundational Security Best Practices"를 클릭하여, 개별 점검 항목과 결과(PASSED/FAILED)를 확인합니다.
>
> **확인**: Security Hub 대시보드가 표시되고, 보안 표준에 대한 점검이 진행되고 있으면 성공입니다. 점검 결과가 모두 나타나기까지는 최대 2시간이 걸릴 수 있습니다.
>
> 🧹 **정리**: Security Hub는 이 Chapter의 통합 실습에서 계속 사용하므로 지금은 **활성화 상태를 유지**합니다.

> ⚠️ **주의**
>
> - **Security Hub는 다른 서비스를 "대체"하지 않습니다**: Security Hub를 활성화한다고 해서 GuardDuty나 Inspector를 대신하는 것이 아닙니다. Security Hub는 **수집·통합·정리** 역할이며, 실제 탐지/스캔은 각 서비스가 수행합니다. 해당 서비스를 활성화하지 않으면 Security Hub에 수집될 Finding도 없습니다.
> - **Security Hub를 먼저 활성화하고, 그 다음 개별 서비스를 활성화하는 것이 좋습니다**: 이렇게 하면 개별 서비스에서 생성되는 Finding이 자동으로 Security Hub로 수집됩니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 15.4.3: 다중 서비스 경보의 상관관계 파악**
>
> **상황**: 금요일 밤, 동시에 여러 보안 이벤트가 발생했습니다. GuardDuty에서 "비정상적 API 호출 3건", Inspector에서 "Critical 취약점 2건", Macie에서 "S3 버킷 퍼블릭 접근 1건"이 보고되었습니다. 각 서비스의 대시보드를 개별적으로 확인하면, 이 이벤트들이 서로 관련 있는지 판단하기 어렵습니다.
>
> **Security Hub 활용**: Security Hub의 통합 대시보드에서 모든 Finding을 심각도 순으로 정렬합니다. 필터링 기능으로 동일한 리소스(예: 특정 EC2 인스턴스)와 관련된 Finding을 묶어 볼 수 있습니다. 분석 결과, GuardDuty의 비정상적 API 호출과 Inspector의 Critical 취약점이 **같은 EC2 인스턴스**에서 발생한 것으로 확인되어, 해당 인스턴스가 취약점을 통해 침해되었을 가능성이 높다고 판단할 수 있습니다.
>
> **핵심 포인트**: Security Hub의 가장 큰 가치는 **여러 서비스의 Finding을 하나의 화면에서 상관관계를 파악**할 수 있다는 점입니다. 개별 서비스만으로는 보이지 않던 전체 그림이 보입니다.

> 🔍 **예시 15.4.4: 보안 점수를 활용한 경영진 보고**
>
> **상황**: CISO(최고정보보안책임자)가 분기별 보안 보고서를 작성해야 합니다. "우리 AWS 환경의 보안 수준이 얼마나 좋은가?"를 객관적인 수치로 보여줘야 합니다.
>
> **Security Hub 활용**: AWS Foundational Security Best Practices 표준의 점수가 지난 분기 68%에서 이번 분기 89%로 향상되었음을 보여줍니다. 실패 항목 중 "루트 계정 MFA 미설정"과 "S3 버킷 암호화 미적용" 2건이 남아 있으며, 다음 분기까지 해결 계획을 제시합니다.
>
> **핵심 포인트**: Security Hub의 보안 표준 점수는 보안 상태를 **정량적으로 측정하고 추적**하는 데 유용합니다. 보안 개선 노력의 성과를 객관적으로 증명할 수 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: Security Hub를 활성화하면 **GuardDuty, Inspector, Macie가 자동으로 활성화되는 것은 아닙니다**. Security Hub는 다른 서비스의 Finding을 "수집"하는 역할이므로, 각 서비스를 별도로 활성화해야 합니다. Security Hub만 활성화하고 다른 서비스를 활성화하지 않으면 수집할 Finding이 없습니다.

> ⚠️ **주의**: Security Hub의 보안 표준 점수가 100%라고 해서 **완벽하게 안전한 것은 아닙니다**. 보안 표준 점검은 "설정이 모범 사례를 따르는가"를 확인하는 것이며, 실시간 위협 탐지와는 다릅니다. 보안 점수와 위협 탐지(GuardDuty)를 함께 활용해야 합니다.

> ⚠️ **주의**: Security Hub는 **리전별로 활성화**됩니다. 여러 리전을 사용하는 경우, 각 리전에서 Security Hub를 활성화하고 **리전 간 집계(Cross-Region Aggregation)** 기능을 설정하여 하나의 리전에서 모든 Finding을 통합 확인할 수 있습니다.

> ⚠️ **주의**: Security Hub의 보안 표준 점검에서 **실패(FAILED) 항목이 나왔다고 무조건 수정해야 하는 것은 아닙니다**. 비즈니스 요구 사항에 따라 의도적으로 다른 설정을 사용하는 경우가 있을 수 있습니다. 이런 경우에는 해당 항목을 "Suppressed(억제)" 상태로 변경하여 점수에서 제외할 수 있습니다.

### Section 요약 (Section Summary)

- **Security Hub**는 GuardDuty, Inspector, Macie 등 **여러 보안 서비스의 발견 사항을 한곳에 통합**합니다.
- **ASFF** 표준 형식으로 발견 사항을 정규화하여, 서비스 간 비교와 우선순위 판단이 가능합니다.
- **보안 표준(FSBP, CIS 등) 자동 점검**을 통해 계정의 보안 준수 상태를 점수로 확인할 수 있습니다.
- Security Hub는 통합·정리 역할이며, 실제 탐지는 각 개별 보안 서비스가 수행합니다.

## 15.5 Amazon Detective와 보안 조사 — 보안 이벤트의 근본 원인 분석

> 💡 **한 줄 요약**: Detective는 GuardDuty 등에서 탐지된 보안 이벤트의 **근본 원인을 시각적으로 분석**하여, "무슨 일이 일어났고, 어떻게 일어났는지"를 파악하는 보안 조사 서비스입니다.

### 직관적 이해 (Intuitive Understanding)
범죄가 발생하면, 먼저 경보가 울립니다(GuardDuty의 역할). 그 다음 형사가 출동하여 사건을 조사합니다 — 범인이 어떤 경로로 들어왔는지, 어떤 물건을 건드렸는지, 도주 경로는 무엇인지를 추적합니다. 형사는 CCTV 영상, 출입 기록, 전화 기록 등 여러 증거를 모아서 **사건의 전체 그림**을 재구성합니다.

**Amazon Detective**는 바로 이 "형사"의 역할을 합니다. GuardDuty가 "수상한 활동이 감지되었습니다"라고 알리면, Detective는 관련된 모든 데이터를 수집·분석하여 **"정확히 무슨 일이 일어났는지"**를 시각적으로 보여줍니다.

**왜 Detective가 필요할까요?**

GuardDuty가 "IAM 사용자 `dev-kim`의 접근 키가 비정상적으로 사용되었습니다"라고 알렸다고 합시다. 보안 담당자는 이제 다음 질문에 답해야 합니다:

- 이 접근 키는 **언제부터** 비정상적으로 사용되기 시작했는가?
- 이 키를 사용하여 **어떤 API를 호출**했는가?
- 영향을 받은 **다른 리소스**는 없는가?
- 이것이 **실제 공격**인가, 아니면 **정상적인 활동의 오탐(False Positive)**인가?

이 질문에 답하려면 CloudTrail 로그, VPC Flow Logs, GuardDuty Finding 등 여러 데이터 소스를 교차 분석해야 합니다. 수작업으로 하면 몇 시간에서 며칠이 걸릴 수 있습니다. Detective는 이 분석을 **자동으로 수행**하고 **시각적인 그래프와 타임라인**으로 보여줍니다.

### 핵심 개념 설명 (Core Explanation)
> 📝 **Amazon Detective**: AWS 환경에서 발생한 보안 이벤트의 **근본 원인을 분석하고 조사**하는 서비스입니다. CloudTrail 로그, VPC Flow Logs, GuardDuty Finding 등을 자동으로 수집·가공하여 **인터랙티브한 시각화**와 **행동 분석 그래프**를 제공합니다.

**Detective의 핵심 개념 — 행동 그래프(Behavior Graph)**:

Detective는 수집한 모든 데이터를 **행동 그래프**라는 구조로 가공합니다. 이 그래프는 AWS 환경에서 관찰된 활동을 **엔터티(IP 주소, IAM 사용자, EC2 인스턴스 등)**와 그들 사이의 **관계(API 호출, 네트워크 연결 등)**로 표현합니다.

```
[보안 조사 워크플로우]

  GuardDuty Finding 발생
         │
         ▼
  "이 Finding을 조사해야 해"
         │
         ▼
  ┌──────────────────────┐
  │   Amazon Detective    │
  │                      │
  │  · 관련 엔터티 식별   │
  │  · 활동 타임라인 생성  │
  │  · 행동 패턴 분석     │
  │  · 시각적 그래프 제공  │
  └──────────────────────┘
         │
         ▼
  근본 원인 파악 & 대응 조치
```

**GuardDuty → Security Hub → Detective 연동**:

이 세 서비스는 자연스러운 **보안 워크플로우**를 형성합니다:

1. **GuardDuty가 위협을 탐지**합니다 → Finding 생성
2. **Security Hub가 Finding을 수집**하여 통합 대시보드에 표시합니다
3. 보안 담당자가 Finding을 확인하고, **Detective에서 상세 조사**를 시작합니다

GuardDuty 또는 Security Hub의 Finding에서 **"Investigate in Detective"** 링크를 클릭하면 바로 Detective로 이동하여 조사를 시작할 수 있습니다.

> 🔍 **예시 15.5.1: 비정상적 API 호출 조사**
>
> **상황**: GuardDuty가 "IAM 사용자 `admin-park`이 평소에 사용하지 않던 리전(eu-west-1)에서 EC2 인스턴스를 생성하는 API를 호출했습니다"라는 Finding을 생성했습니다.
> **Detective 분석**:
> - **타임라인**: `admin-park`의 지난 30일간 활동을 시간순으로 보여줍니다. 지난 29일간은 서울 리전(ap-northeast-2)에서만 활동했지만, 마지막 날 갑자기 아일랜드 리전(eu-west-1)에서 활동이 시작되었습니다.
> - **API 호출 분석**: eu-west-1에서 `RunInstances`(EC2 생성), `CreateSecurityGroup`(보안 그룹 생성), `AuthorizeSecurityGroupIngress`(모든 포트 개방) API가 호출되었습니다.
> - **IP 분석**: 이 API 호출의 소스 IP가 `admin-park`이 평소 사용하는 회사 IP가 아닌, 해외 VPN IP입니다.
> **결론**: 이 사용자의 자격 증명이 유출되어 해외에서 악용되고 있을 가능성이 높습니다. 즉시 접근 키를 비활성화하고 생성된 리소스를 삭제해야 합니다.
> **핵심 포인트**: Detective는 단순히 "비정상적 활동이 있었다"는 GuardDuty의 경보를 넘어서, **"구체적으로 무엇이 일어났고, 얼마나 심각한가"**를 판단하는 데 필요한 맥락을 제공합니다.

> 🔍 **예시 15.5.2: 오탐(False Positive) 확인**
>
> **상황**: GuardDuty가 "EC2 인스턴스에서 비정상적인 외부 네트워크 연결이 감지되었습니다"라는 Finding을 생성했습니다.
> **Detective 분석**: 해당 EC2 인스턴스의 네트워크 활동 히스토리를 분석한 결과, 이 인스턴스는 매월 첫째 주에 외부 소프트웨어 업데이트 서버에 접속하는 패턴이 있었습니다. 이번 연결도 동일한 패턴에 해당합니다.
> **결론**: 이것은 정상적인 소프트웨어 업데이트 활동이며, GuardDuty의 오탐입니다. Finding을 "archived(보관)"으로 표시합니다.
> **핵심 포인트**: 모든 GuardDuty Finding이 실제 위협은 아닙니다. Detective는 **역사적 맥락**을 제공하여, 보안 담당자가 실제 위협과 오탐을 구분하는 데 도움을 줍니다.

> ⚠️ **주의**
>
> - **Detective는 조사 도구이지 탐지 도구가 아닙니다**: Detective는 스스로 위협을 찾아내지 않습니다. GuardDuty나 Security Hub에서 탐지된 **기존 Finding을 심층 분석**하는 역할입니다. Detective 없이는 조사가 어렵지만, GuardDuty 없이는 조사할 대상 자체가 없습니다.
> - **Detective 사용을 위해서는 GuardDuty가 먼저 활성화되어 있어야 합니다**: Detective는 GuardDuty가 수집하는 데이터를 기반으로 분석하므로, GuardDuty가 활성화되지 않은 상태에서는 Detective를 활성화할 수 없습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 15.5.3: 다수의 리소스에 걸친 공격 경로 추적**
>
> **상황**: GuardDuty가 3건의 Finding을 생성했습니다 — (1) IAM 사용자 `dev-choi`의 비정상적 콘솔 로그인, (2) 새로운 EC2 인스턴스 생성, (3) 해당 인스턴스에서 외부 악성 IP로의 네트워크 연결. 이 3건이 관련된 하나의 공격인지, 우연히 동시에 발생한 별개의 이벤트인지 판단해야 합니다.
>
> **Detective 분석**: Detective의 행동 그래프에서 `dev-choi` 사용자의 활동 타임라인을 확인합니다. (1) 03:15에 평소 사용하지 않는 IP에서 콘솔 로그인 → (2) 03:22에 `RunInstances` API로 EC2 인스턴스 생성 → (3) 03:30에 생성된 인스턴스에서 외부 C2(Command & Control) 서버로 연결. 세 이벤트가 **동일한 소스 IP에서 시간 순서대로 발생**한 것을 확인하여, 이것이 자격 증명 유출을 통한 체계적인 공격임을 파악합니다.
>
> **핵심 포인트**: Detective는 개별 Finding을 연결하여 **공격의 전체 경로(Kill Chain)**를 재구성합니다. 개별 Finding만 보면 각각의 심각도가 Medium일 수 있지만, 연결하면 Critical 수준의 공격임을 알 수 있습니다.

> 🔍 **예시 15.5.4: 일상 비유 — 형사의 수사 보드**
>
> **상황**: 범죄 드라마에서 형사가 벽에 사진, 메모, 지도를 붙이고 실로 연결하여 사건의 전체 그림을 구성하는 장면을 본 적이 있을 것입니다. 용의자(IP 주소), 범행 장소(리소스), 시간대(타임라인), 증거(API 호출 기록)를 하나의 보드에 정리하면, 흩어진 단서가 하나의 이야기로 연결됩니다.
>
> Detective의 행동 그래프가 바로 이 "수사 보드"입니다. 여러 데이터 소스에서 수집한 정보를 시각적으로 연결하여, 보안 담당자가 사건의 전체 맥락을 빠르게 이해할 수 있게 해 줍니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: Detective를 **위협 탐지 도구로 오해**하지 마세요. Detective는 스스로 위협을 찾아내지 않습니다. GuardDuty가 "경보"를 울리면, Detective가 "조사"를 수행합니다. Detective 없이도 GuardDuty는 작동하지만, GuardDuty 없이 Detective를 사용할 수는 없습니다.

> ⚠️ **주의**: Detective는 **GuardDuty를 최소 48시간 이상 활성화한 후**에야 의미 있는 분석을 제공할 수 있습니다. Detective는 행동 그래프를 구축하기 위해 충분한 히스토리 데이터가 필요하므로, 활성화 직후에는 분석 결과가 제한적일 수 있습니다.

> ⚠️ **주의**: Detective는 **Security Hub나 GuardDuty의 대체재가 아닙니다**. 세 서비스는 보안 워크플로우에서 서로 다른 역할을 합니다 — GuardDuty(탐지) → Security Hub(통합) → Detective(조사). 각각의 역할을 정확히 구분하는 것이 시험에서 중요합니다.

> ⚠️ **주의**: 시험에서 "보안 이벤트의 **근본 원인**을 분석하려면 어떤 서비스를 사용해야 하는가?"라는 질문이 나오면, 답은 **Detective**입니다. "근본 원인 분석", "보안 조사", "행동 그래프"라는 키워드가 나오면 Detective를 떠올리세요.

### Section 요약 (Section Summary)

- **Detective**는 보안 이벤트의 **근본 원인을 시각적으로 분석**하는 보안 조사 서비스입니다.
- **행동 그래프(Behavior Graph)**를 통해 엔터티 간의 관계와 활동 타임라인을 분석합니다.
- **GuardDuty → Security Hub → Detective**는 탐지 → 통합 → 조사라는 자연스러운 보안 워크플로우를 형성합니다.
- Detective는 **GuardDuty가 활성화되어 있어야** 사용할 수 있습니다.

## 15.6 비밀 정보 관리 — Secrets Manager, Parameter Store를 활용한 자격 증명 관리

> 💡 **한 줄 요약**: AWS Secrets Manager와 Systems Manager Parameter Store는 비밀번호, API 키 같은 **민감한 정보를 안전하게 저장하고 관리**하는 서비스로, 코드에 비밀 정보를 하드코딩하는 위험을 제거합니다.

### 직관적 이해 (Intuitive Understanding)
아파트에 살면서 현관 비밀번호, 우편함 열쇠, 차고 리모컨, 와이파이 비밀번호 등 여러 비밀 정보를 관리해야 한다고 상상해 봅시다.

**가장 위험한 방법**: 모든 비밀번호를 포스트잇에 적어서 냉장고에 붙여놓기 — 누구나 볼 수 있고, 비밀번호를 바꿔야 할 때 모든 포스트잇을 찾아서 수정해야 합니다.

**안전한 방법**: 비밀번호 금고(Password Vault)를 사용하기 — 금고에 모든 비밀번호를 저장하고, 필요할 때만 금고에서 꺼내 쓰며, 주기적으로 비밀번호를 자동으로 변경합니다.

소프트웨어 개발에서도 마찬가지입니다. 애플리케이션은 데이터베이스 비밀번호, 외부 API 키, 인증 토큰 등 여러 **비밀 정보(Secret)**를 사용합니다. 이 정보를 코드 안에 직접 적어놓는 것(하드코딩)은 포스트잇을 냉장고에 붙여놓는 것과 같습니다.

> 🔗 **연결**: Chapter 13에서 IAM 접근 키(Access Key)를 배웠습니다. 접근 키 자체가 하나의 비밀 정보이며, 이 키가 코드에 하드코딩되어 유출되는 사고가 빈번합니다. 앞서 15.1의 GuardDuty 예시에서도 유출된 접근 키 사례를 보았습니다. Secrets Manager는 이런 비밀 정보를 안전하게 관리하여 유출 위험을 줄입니다.

> 🔗 **연결**: Chapter 14에서 배운 **KMS(Key Management Service)**는 데이터를 암호화하는 "키"를 관리하는 서비스였습니다. Secrets Manager는 내부적으로 **KMS를 사용하여** 저장된 비밀 정보를 암호화합니다. 즉, KMS는 암호화 키를 관리하고, Secrets Manager는 그 키로 암호화된 비밀 정보를 관리합니다.

### 핵심 개념 설명 (Core Explanation)
> 📝 **비밀 정보(Secret)**: 애플리케이션이 다른 시스템에 접근하기 위해 사용하는 민감한 정보. 데이터베이스 비밀번호, API 키, OAuth 토큰, SSH 키 등이 해당합니다. 이런 정보가 유출되면 시스템 전체의 보안이 위협받습니다.

AWS는 비밀 정보를 관리하는 두 가지 서비스를 제공합니다: **Secrets Manager**와 **Systems Manager Parameter Store**. 각각의 역할과 차이를 이해하는 것이 중요합니다.

---

#### AWS Secrets Manager

> 📝 **AWS Secrets Manager**: 비밀 정보를 안전하게 저장, 교체(rotate), 관리하는 전용 서비스입니다. 특히 **자동 교체(Automatic Rotation)** 기능이 핵심으로, 데이터베이스 비밀번호 같은 자격 증명을 주기적으로 자동 변경할 수 있습니다.

**Secrets Manager의 핵심 기능**:

1. **안전한 저장**: 비밀 정보를 KMS로 암호화하여 저장합니다. 코드에 비밀번호를 직접 넣는 대신, Secrets Manager에 저장된 비밀번호를 **API 호출로 가져옵니다**.

2. **자동 교체(Automatic Rotation)**: 비밀번호를 주기적으로 자동 변경합니다. 예를 들어, "30일마다 데이터베이스 비밀번호를 자동으로 변경"하도록 설정할 수 있습니다. Secrets Manager는 비밀번호를 변경하고, 데이터베이스에도 새 비밀번호를 적용하며, 애플리케이션은 항상 최신 비밀번호를 가져옵니다.

3. **RDS, Redshift 등 AWS 서비스와의 네이티브 통합**: 특히 **Amazon RDS** 데이터베이스의 비밀번호 자동 교체가 내장되어 있어, 별도의 코드 작성 없이 비밀번호 교체를 설정할 수 있습니다.

4. **버전 관리**: 비밀 정보의 이전 버전을 유지하여, 교체 과정에서 문제가 발생하면 이전 버전으로 롤백할 수 있습니다.

**Secrets Manager의 작동 방식**:

```
[기존 — 위험한 방법]
애플리케이션 코드:
  db_password = "MyP@ssw0rd123"     ← 코드에 비밀번호 노출!

[Secrets Manager 사용 — 안전한 방법]
애플리케이션 코드:
  db_password = secrets_manager.get("prod/database/password")
  → Secrets Manager에서 암호화된 비밀번호를 안전하게 가져옴
```

> 🔍 **예시 15.6.1: 데이터베이스 비밀번호 관리**
>
> **상황**: 웹 애플리케이션이 RDS MySQL 데이터베이스에 접속해야 합니다. 기존에는 비밀번호를 환경 변수 파일(.env)에 저장했지만, 이 파일이 실수로 Git에 커밋될 위험이 있습니다.
> **Secrets Manager 활용**:
> 1. Secrets Manager에 `prod/myapp/database`라는 이름으로 비밀 정보를 생성합니다.
> 2. 비밀 정보에 `{"username": "admin", "password": "MyP@ssw0rd123", "host": "mydb.xxx.rds.amazonaws.com"}` 형태로 저장합니다.
> 3. 애플리케이션은 시작 시 Secrets Manager API를 호출하여 비밀번호를 가져옵니다.
> 4. 자동 교체를 30일로 설정하면, Secrets Manager가 30일마다 비밀번호를 자동 변경합니다.
> **핵심 포인트**: 코드 어디에도 실제 비밀번호가 적히지 않으므로, 코드가 유출되어도 비밀번호는 안전합니다. 또한 자동 교체로 비밀번호가 주기적으로 변경되어, 오래된 비밀번호로 인한 위험도 줄어듭니다.

---

#### AWS Systems Manager Parameter Store

> 📝 **AWS Systems Manager Parameter Store**: 설정 데이터와 비밀 정보를 계층적으로 저장하는 서비스입니다. Secrets Manager보다 범용적이며, 비밀 정보뿐 아니라 **일반 설정값(Configuration)**도 함께 관리할 수 있습니다.

**Parameter Store의 파라미터 유형**:

| 유형 | 설명 | 사용 예 | 암호화 |
|------|------|--------|--------|
| **String** | 일반 텍스트 값 | 환경 이름, URL 등 | 없음 |
| **StringList** | 쉼표로 구분된 값 목록 | 허용 IP 목록 등 | 없음 |
| **SecureString** | 암호화된 비밀 값 | 비밀번호, API 키 | KMS 암호화 |

**Parameter Store의 계층적 구조**:

Parameter Store는 파일 시스템과 유사한 **경로(Path)** 구조로 파라미터를 정리할 수 있습니다:

```
/myapp/
  ├── prod/
  │   ├── db-host        → "mydb.xxx.rds.amazonaws.com" (String)
  │   ├── db-password    → "encrypted-value" (SecureString)
  │   └── api-key        → "encrypted-value" (SecureString)
  └── dev/
      ├── db-host        → "devdb.xxx.rds.amazonaws.com" (String)
      ├── db-password    → "encrypted-value" (SecureString)
      └── api-key        → "encrypted-value" (SecureString)
```

이 구조 덕분에 환경(prod/dev)별로 설정을 깔끔하게 분리하고, IAM 정책으로 **경로 단위의 접근 제어**가 가능합니다. 예를 들어, 개발자에게는 `/myapp/dev/*` 경로만 접근 권한을 주고, 운영팀에는 `/myapp/prod/*` 경로도 접근 가능하게 설정할 수 있습니다.

> 🔍 **예시 15.6.2: 환경별 설정 관리**
>
> **상황**: 애플리케이션이 개발(dev), 스테이징(staging), 운영(prod) 세 가지 환경에서 실행됩니다. 각 환경마다 데이터베이스 주소, API 엔드포인트, 기능 플래그 등의 설정이 다릅니다.
> **Parameter Store 활용**: `/myapp/{환경}/{파라미터명}` 구조로 설정을 저장합니다. 애플리케이션은 배포된 환경에 따라 해당 경로의 파라미터를 읽어옵니다. 민감한 값(비밀번호, API 키)은 SecureString으로, 일반 설정값(URL, 포트 번호)은 String으로 저장합니다.
> **핵심 포인트**: Parameter Store는 비밀 정보와 일반 설정을 **하나의 서비스에서 통합 관리**할 수 있다는 장점이 있습니다. 모든 것이 비밀 정보인 것은 아니므로, 설정값과 비밀값을 적절히 분류하여 관리하는 것이 효율적입니다.

#### Secrets Manager vs Parameter Store 비교

| 비교 항목 | Secrets Manager | Parameter Store |
|----------|-----------------|-----------------|
| **주요 용도** | 비밀 정보 전용 관리 | 설정값 + 비밀 정보 통합 관리 |
| **자동 교체(Rotation)** | ✅ 내장 (RDS 네이티브 지원) | ❌ 미지원 (별도 구현 필요) |
| **암호화** | 기본 암호화 (KMS) | SecureString 유형만 암호화 |
| **비용** | 비밀당 월 $0.40 + API 호출 비용 | Standard 무료, Advanced 유료 |
| **계층 구조** | ❌ | ✅ 경로 기반 계층 구조 |
| **버전 관리** | ✅ 내장 | ✅ 내장 |
| **교차 계정 접근** | ✅ | ✅ |

**어떤 서비스를 선택해야 할까?**

- **데이터베이스 비밀번호처럼 주기적으로 교체가 필요한 자격 증명** → **Secrets Manager**
- **일반 설정값과 비밀 정보를 함께 관리하고, 비용을 절약하고 싶을 때** → **Parameter Store**
- **RDS 비밀번호 자동 교체가 필요할 때** → **Secrets Manager** (네이티브 통합 지원)

> 🧪 **Mini Lab 15.6: Secrets Manager에 비밀 정보 저장하기**
>
> **목표**: Secrets Manager에 비밀 정보를 생성하고, 저장된 값을 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: ⚠️ 비밀당 월 $0.40 과금 발생 (실습 후 즉시 삭제하면 일할 계산으로 최소 비용)
>
> **Step 1**: AWS 콘솔 상단 검색 바에 "Secrets Manager"를 입력하고 **AWS Secrets Manager** 서비스를 클릭합니다.
>
> **Step 2**: **"Store a new secret"** 또는 **"새 보안 암호 저장"** 버튼을 클릭합니다.
>
> **Step 3**: 비밀 유형을 선택합니다:
> - **Secret type**: "Other type of secret" (다른 유형의 보안 암호)을 선택합니다.
> - **Key/Value pairs**: 아래와 같이 입력합니다:
>   - Key: `username`, Value: `my-app-user`
>   - **"+ Add row"**를 클릭하여 행을 추가합니다
>   - Key: `password`, Value: `TestP@ssw0rd123`
> - **Encryption key**: `aws/secretsmanager` (기본 KMS 키)를 유지합니다.
> - **Next**를 클릭합니다.
>
> **Step 4**: 비밀 이름과 설명을 입력합니다:
> - **Secret name**: `test/myapp/database`
> - **Description**: `Test secret for learning` (선택 사항)
> - **Next**를 클릭합니다.
>
> **Step 5**: 자동 교체 설정 화면이 나타납니다:
> - 이 실습에서는 자동 교체를 설정하지 않습니다. **"Disable automatic rotation"**이 선택된 상태에서 **Next**를 클릭합니다.
>
> **Step 6**: 검토 화면에서 내용을 확인하고 **"Store"**를 클릭하여 비밀 정보를 저장합니다.
>
> **Step 7**: 생성된 비밀 `test/myapp/database`를 클릭하여 상세 페이지로 이동합니다. **"Retrieve secret value"** 또는 **"보안 암호 값 검색"** 버튼을 클릭하면, 저장한 Key/Value 쌍이 표시됩니다.
>
> **확인**: `username: my-app-user`, `password: TestP@ssw0rd123`이 표시되면 성공입니다.
>
> 🧹 **정리**: 실습이 끝나면 이 비밀을 삭제합니다:
> 1. 비밀 상세 페이지에서 **Actions → Delete secret**을 클릭합니다.
> 2. 최소 대기 기간(7일)이 설정됩니다 (Secrets Manager는 즉시 삭제가 아닌 예약 삭제를 사용합니다). **"Schedule deletion"**을 클릭합니다.
> 3. 비밀은 7일 후에 영구 삭제되며, 이 기간 동안에도 비밀에 접근할 수 없습니다.
>
> ⚠️ **비용 참고**: 삭제 예약 후에도 7일간은 비밀이 존재하므로 일할 과금이 발생할 수 있습니다 (약 $0.09). 이는 Secrets Manager의 안전장치로, 실수로 삭제한 비밀을 복구할 수 있게 해 줍니다.

> ⚠️ **주의**
>
> - **하드코딩은 절대 금지**: 비밀번호, API 키 등을 코드에 직접 작성하지 마세요. GitHub 등에 실수로 푸시하면 즉시 유출됩니다. 이는 보안 사고의 가장 흔한 원인 중 하나입니다.
> - **Secrets Manager의 삭제 대기 기간**: Secrets Manager에서 비밀을 삭제하면 즉시 삭제되지 않고, 7~30일의 대기 기간이 있습니다. 이는 실수 방지를 위한 안전장치입니다.
> - **Parameter Store의 SecureString도 충분할 수 있습니다**: 자동 교체가 필요 없고 비용을 절약하고 싶다면, Parameter Store의 SecureString 유형으로도 비밀 정보를 안전하게 관리할 수 있습니다 (Standard 파라미터는 무료).

### 상세 예시 (Detailed Examples)

> 🔍 **예시 15.6.3: 자동 교체로 보안 사고 예방**
>
> **상황**: 회사의 운영 데이터베이스(RDS MySQL) 비밀번호가 2년 동안 변경되지 않았습니다. 이 비밀번호는 여러 팀원이 알고 있으며, 퇴사한 직원도 기억하고 있을 수 있습니다. 보안 감사에서 "비밀번호를 정기적으로 변경하라"는 권고를 받았습니다.
>
> **Secrets Manager 활용**: (1) 기존 비밀번호를 Secrets Manager에 저장합니다. (2) RDS와의 자동 교체를 30일 주기로 설정합니다. (3) 애플리케이션 코드를 수정하여 Secrets Manager API로 비밀번호를 가져오도록 변경합니다. 이후 Secrets Manager가 30일마다 자동으로 새 비밀번호를 생성하고, RDS에 적용하며, 애플리케이션은 항상 최신 비밀번호를 가져옵니다. 퇴사한 직원이 알고 있던 비밀번호는 이미 무효화됩니다.
>
> **핵심 포인트**: 자동 교체의 핵심 가치는 "사람이 개입하지 않아도 비밀번호가 주기적으로 변경된다"는 것입니다. 수동으로 비밀번호를 변경하면 애플리케이션 장애가 발생할 위험이 있지만, Secrets Manager는 비밀번호 변경과 애플리케이션 적용을 원자적으로 처리합니다.

> 🔍 **예시 15.6.4: Parameter Store를 활용한 멀티 환경 배포**
>
> **상황**: DevOps 팀이 동일한 애플리케이션을 dev, staging, prod 세 환경에 배포합니다. 각 환경마다 데이터베이스 주소, API 엔드포인트, 기능 플래그(feature flag) 등 수십 개의 설정이 다릅니다. 환경 변수 파일(.env)로 관리하면 파일이 여러 곳에 흩어지고 버전 관리가 어렵습니다.
>
> **Parameter Store 활용**: `/myapp/dev/db-host`, `/myapp/prod/db-host`처럼 경로 기반으로 설정을 저장합니다. 애플리케이션은 배포 환경에 따라 해당 경로의 파라미터를 일괄 조회(`GetParametersByPath`)합니다. IAM 정책으로 개발자는 `/myapp/dev/*`만, 운영팀은 `/myapp/prod/*`도 접근 가능하게 제어합니다.
>
> **핵심 포인트**: Parameter Store의 계층적 구조와 IAM 기반 접근 제어를 결합하면, 환경별 설정을 안전하고 체계적으로 관리할 수 있습니다. 모든 설정이 한곳에 집중되어 있으므로 변경 이력 추적도 용이합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: Secrets Manager와 Parameter Store의 **용도를 혼동**하지 마세요. 시험에서 "데이터베이스 비밀번호를 **자동으로 교체**하려면?"이라는 질문이 나오면 답은 **Secrets Manager**입니다. "설정값과 비밀 정보를 **계층적으로 통합 관리**하려면?"이라는 질문이 나오면 **Parameter Store**가 적합합니다.

> ⚠️ **주의**: Secrets Manager는 **비밀당 월 $0.40의 비용**이 발생합니다. 수백 개의 비밀 정보를 저장하면 비용이 상당할 수 있습니다. 자동 교체가 필요 없는 단순한 비밀 정보는 Parameter Store의 SecureString으로 저장하면 비용을 절약할 수 있습니다(Standard 파라미터는 무료).

> ⚠️ **주의**: Parameter Store의 **String 유형은 암호화되지 않습니다**. 비밀번호나 API 키 같은 민감한 정보는 반드시 **SecureString** 유형으로 저장해야 합니다. String 유형으로 저장하면 IAM 권한이 있는 누구나 평문으로 값을 확인할 수 있습니다.

> ⚠️ **주의**: Secrets Manager에 비밀 정보를 저장했더라도, **애플리케이션 코드에서 비밀번호를 캐시하여 메모리에 오래 유지**하면 보안 효과가 감소합니다. 비밀번호가 교체되었을 때 애플리케이션이 새 비밀번호를 가져올 수 있도록, 적절한 캐시 만료 시간을 설정해야 합니다.

> ⚠️ **주의**: Secrets Manager의 **삭제 대기 기간(7~30일)**을 잊지 마세요. 비밀을 삭제해도 즉시 삭제되지 않으며, 대기 기간 동안 비용이 계속 발생합니다. 이는 실수로 삭제한 비밀을 복구하기 위한 안전장치입니다.

### Section 요약 (Section Summary)

- **Secrets Manager**는 비밀 정보를 안전하게 저장하고 **자동 교체**하는 전용 서비스입니다. 특히 RDS 비밀번호 교체에 강점이 있습니다.
- **Parameter Store**는 일반 설정값과 비밀 정보를 **계층적 구조로 통합 관리**하는 서비스이며, Standard 파라미터는 무료입니다.
- 두 서비스 모두 **KMS 암호화**를 지원하며, 코드에 비밀번호를 하드코딩하는 위험을 제거합니다.
- **자동 교체가 필요하면 Secrets Manager**, **비용 효율과 설정값 통합 관리가 중요하면 Parameter Store**를 선택합니다.

- **Chapter 14 (네트워크 보안과 암호화)**: 보안 그룹, WAF, Shield로 구성한 **예방적 방어**를 우회하여 발생하는 위협을, 이번 Chapter의 서비스들이 **탐지하고 조사**합니다. KMS는 Secrets Manager가 비밀 정보를 암호화하는 데 사용됩니다.

**다음 Chapter와의 연결**:

- **Chapter 16 (규정 준수와 거버넌스)**: Security Hub의 보안 표준 점검은 규정 준수의 기반이 됩니다. Chapter 16에서 배울 AWS Config, CloudTrail, Organizations는 거버넌스 측면에서 보안 관리를 확장합니다.
- **Chapter 18 (모니터링과 로깅)**: GuardDuty가 분석하는 CloudTrail 로그, VPC Flow Logs 등을 더 깊이 배웁니다. 보안 서비스의 Finding을 CloudWatch와 연동하여 자동 대응하는 방법도 다룹니다.

## 자기 점검 질문 (Self-Check Questions)

**1. GuardDuty는 위협을 탐지하면 자동으로 해당 위협을 차단한다. (O/X)**

<details>
<summary>정답 확인</summary>

**X** — GuardDuty는 위협을 **탐지하고 Finding을 생성**할 뿐, 자동으로 차단하거나 치료하지는 않습니다. 대응 조치는 관리자가 수동으로 하거나 별도의 자동화를 구성해야 합니다.

</details>

**2. Inspector가 스캔하는 대상은 무엇인가요? (3가지)**

<details>
<summary>정답 확인</summary>

**EC2 인스턴스, Lambda 함수, ECR 컨테이너 이미지** — Inspector는 이 세 가지 워크로드의 소프트웨어 취약점(CVE)과 네트워크 도달 가능성을 스캔합니다.

</details>

**3. Macie는 RDS 데이터베이스의 민감 데이터도 스캔할 수 있다. (O/X)**

<details>
<summary>정답 확인</summary>

**X** — Macie는 **S3 버킷에 저장된 데이터만** 스캔합니다. RDS, DynamoDB 등 다른 데이터 저장소는 현재 지원하지 않습니다.

</details>

**4. Security Hub를 활성화하면 GuardDuty, Inspector, Macie를 별도로 활성화할 필요가 없다. (O/X)**

<details>
<summary>정답 확인</summary>

**X** — Security Hub는 발견 사항을 **수집·통합하는 역할**이며, 실제 탐지/스캔은 각 서비스가 수행합니다. 개별 서비스를 활성화하지 않으면 Security Hub에 수집될 Finding도 없습니다.

</details>

**5. Secrets Manager와 Parameter Store의 가장 큰 차이점은 무엇인가요?**

<details>
<summary>정답 확인</summary>

가장 큰 차이점은 **자동 교체(Automatic Rotation)** 기능입니다. Secrets Manager는 비밀번호 등을 주기적으로 자동 변경하는 기능이 내장되어 있지만, Parameter Store는 이 기능이 없습니다. 또한 Secrets Manager는 비밀 정보 전용이고 비용이 발생하지만, Parameter Store는 일반 설정값도 관리할 수 있으며 Standard 파라미터는 무료입니다.

</details>

**6. Amazon Detective를 사용하려면 반드시 먼저 활성화해야 하는 서비스는?**

<details>
<summary>정답 확인</summary>

**Amazon GuardDuty** — Detective는 GuardDuty가 수집하는 데이터를 기반으로 분석하므로, GuardDuty가 활성화되지 않은 상태에서는 Detective를 활성화할 수 없습니다.

</details>

**7. 다음 중 "행동 분석"을 통해 위협을 탐지하는 서비스는? (GuardDuty / Inspector / Macie)**

<details>
<summary>정답 확인</summary>

**GuardDuty** — GuardDuty는 머신러닝과 행동 분석으로 **비정상적인 활동 패턴**을 탐지합니다. Inspector는 **알려진 소프트웨어 취약점**을 스캔하고, Macie는 **민감한 데이터 패턴**을 탐지합니다.

</details>

### 트러블슈팅 (Troubleshooting)

> **Q: GuardDuty에서 "Generate sample findings" 버튼이 보이지 않습니다.**
> A: Settings(설정) 페이지의 중간 또는 하단 부분에 있습니다. 페이지를 스크롤하여 확인하세요. 리전이 올바르게 설정되어 있는지도 확인합니다.

> **Q: Security Hub에서 GuardDuty Finding이 보이지 않습니다.**
> A: Security Hub와 GuardDuty가 **동일한 리전**에서 활성화되어 있는지 확인하세요. Finding이 수집되는 데 몇 분이 걸릴 수 있으므로 잠시 기다린 후 새로고침합니다.

> **Q: Security Hub의 보안 표준 점수가 0%입니다.**
> A: 보안 표준 점검이 완료되기까지 **최대 2시간**이 소요될 수 있습니다. 시간이 지난 후 다시 확인하세요.

> **Q: Secrets Manager에서 비밀 삭제 시 "즉시 삭제" 옵션이 없습니다.**
> A: Secrets Manager는 안전장치로 **최소 7일의 삭제 대기 기간**을 강제합니다. 대기 기간 동안 비밀에 접근할 수 없으며, 7일 후 자동으로 영구 삭제됩니다. AWS CLI를 사용하면 `--force-delete-without-recovery` 옵션으로 즉시 삭제가 가능하지만, 이 실습에서는 콘솔의 기본 방법을 사용합니다.

> B) Lambda 함수
> C) S3 버킷

> **Exercise 15.3**: 다음 상황에서 Secrets Manager와 Parameter Store 중 어떤 서비스를 선택하는 것이 더 적합한지 판단하고, 이유를 설명하세요.

> (a) 프로덕션 RDS 데이터베이스의 비밀번호를 관리해야 하며, 보안 규정상 90일마다 비밀번호를 교체해야 한다.

> (c) 외부 SaaS 서비스의 API 키를 관리해야 하며, 이 키는 6개월마다 수동으로 갱신된다.

> <summary>모범 답안</summary>

>

>
> **(c) 두 서비스 모두 가능하지만, Parameter Store가 비용 효율적** — 자동 교체가 아닌 6개월 주기의 수동 갱신이므로 Secrets Manager의 자동 교체 기능이 크게 필요하지 않습니다. Parameter Store의 SecureString으로 저장하면 무료(Standard 기준)로 관리할 수 있습니다. 단, 향후 자동 교체 필요성이 예상된다면 Secrets Manager를 선택하는 것도 합리적입니다.
>
> **보충 설명**: 실무에서는 두 서비스를 함께 사용하는 경우가 많습니다. 자동 교체가 필요한 핵심 자격 증명은 Secrets Manager에, 나머지 설정값과 덜 민감한 비밀 정보는 Parameter Store에 저장하는 하이브리드 전략이 일반적입니다.
