## 8.1 데이터베이스 기초 개념

> 💡 **한 줄 요약**: 데이터베이스는 데이터를 체계적으로 저장·관리하는 시스템이며, 데이터의 성격에 따라 관계형(SQL)과 비관계형(NoSQL)으로 나뉜다.

### 직관적 이해

여러분이 도서관 사서라고 상상해 봅시다. 수만 권의 책을 관리해야 하는데, 두 가지 방법이 있습니다.

**방법 1: 엑셀 표처럼 정리하기** — 모든 책의 정보를 깔끔한 표에 넣습니다. "도서번호, 제목, 저자, 출판사, 분류" 같은 열(column)을 미리 정해두고, 모든 책이 같은 형식으로 기록됩니다. 새 책이 들어오면 정해진 칸에 맞춰 정보를 입력하고, "저자가 김철수인 책을 모두 찾아줘"라고 하면 빠르게 검색할 수 있습니다. 이것이 **관계형 데이터베이스**의 개념입니다.

**방법 2: 서류 봉투에 자유롭게 넣기** — 각 책에 대한 정보를 봉투에 자유 형식으로 넣습니다. 어떤 봉투에는 저자와 제목만, 어떤 봉투에는 독자 리뷰와 사진까지 들어 있습니다. 형식이 제각각이지만 유연하고, 봉투에 "도서번호"만 적어두면 필요한 봉투를 빠르게 꺼낼 수 있습니다. 이것이 **비관계형 데이터베이스(NoSQL)**의 개념입니다.

왜 두 가지 방법이 모두 필요할까요? 은행 계좌 거래 내역처럼 **정확한 구조와 일관성이 중요한 데이터**는 방법 1이 적합합니다. 반면, 소셜 미디어의 게시물처럼 **데이터 형태가 다양하고 대량으로 빠르게 처리해야 하는 데이터**는 방법 2가 더 효율적입니다.

### 핵심 개념 설명

#### 데이터베이스란?

> 📝 **데이터베이스(Database)**: 데이터를 체계적으로 저장, 관리, 검색할 수 있도록 구성된 데이터의 집합이다. 단순히 파일에 데이터를 저장하는 것과 달리, 동시 접근 제어, 데이터 무결성 보장, 효율적인 검색 등의 기능을 제공한다.

> 🔗 **연결**: Chapter 6에서 S3(객체 스토리지)와 EBS(블록 스토리지)를 배웠습니다. 이들은 파일이나 디스크 수준의 저장소입니다. 데이터베이스는 이런 스토리지 *위에서* 동작하며, 데이터에 **구조와 의미**를 부여하고 **빠른 검색과 관리 기능**을 제공합니다. 예를 들어, RDS는 내부적으로 EBS 볼륨을 저장소로 사용합니다.

#### 관계형 데이터베이스 (Relational Database)

> 📝 **관계형 데이터베이스(Relational Database)**: 데이터를 **테이블(table)** 형태로 저장하는 데이터베이스이다. 각 테이블은 행(row)과 열(column)로 구성되며, 테이블 간의 **관계(relationship)**를 통해 데이터를 연결한다. 데이터를 조회하고 조작하기 위해 **SQL(Structured Query Language)**이라는 표준 언어를 사용한다.

관계형 데이터베이스의 핵심 특성은 **ACID** 속성입니다:

| 속성 | 의미 | 일상적 비유 |
|------|------|-------------|
| **A**tomicity (원자성) | 트랜잭션은 전부 성공하거나 전부 실패한다 | 은행 송금: 출금과 입금이 둘 다 되거나 둘 다 안 되어야 함 |
| **C**onsistency (일관성) | 트랜잭션 전후로 데이터가 유효한 상태를 유지한다 | 계좌 잔액이 마이너스가 되면 안 됨 |
| **I**solation (격리성) | 동시에 실행되는 트랜잭션이 서로 영향을 주지 않는다 | 두 사람이 동시에 같은 계좌에서 출금해도 잔액이 정확해야 함 |
| **D**urability (지속성) | 완료된 트랜잭션의 결과는 영구적으로 보존된다 | 송금 완료 후 서버가 꺼져도 기록이 사라지지 않음 |

> 📝 **트랜잭션(Transaction)**: 데이터베이스에서 하나의 논리적 작업 단위를 이루는 연산들의 집합이다. 예를 들어, "A 계좌에서 10만원 출금 + B 계좌에 10만원 입금"은 하나의 트랜잭션이다.

> 📝 **SQL(Structured Query Language)**: 관계형 데이터베이스에서 데이터를 정의, 조작, 조회하기 위한 표준 언어이다. `SELECT`, `INSERT`, `UPDATE`, `DELETE` 등의 명령어를 사용한다.

관계형 데이터베이스는 **스키마(schema)**라는 구조를 미리 정의해야 합니다. 테이블에 어떤 열이 있는지, 각 열에 들어갈 데이터 타입은 무엇인지를 사전에 정하고, 이 규칙에 맞는 데이터만 저장할 수 있습니다. 이를 **스키마 온 라이트(Schema-on-Write)**라고 합니다 — 데이터를 쓸 때 스키마를 강제합니다.

**대표적인 관계형 데이터베이스**: MySQL, PostgreSQL, Oracle, Microsoft SQL Server, MariaDB

#### 비관계형 데이터베이스 (NoSQL)

> 📝 **비관계형 데이터베이스(NoSQL Database)**: 관계형 데이터베이스의 테이블-행-열 구조를 따르지 않는 데이터베이스의 총칭이다. "Not Only SQL"의 약자로, SQL만이 아닌 다양한 데이터 모델과 접근 방식을 사용한다.

NoSQL 데이터베이스가 등장한 배경을 이해하면 그 필요성이 명확해집니다. 2000년대 들어 인터넷 서비스가 폭발적으로 성장하면서, 관계형 데이터베이스로는 감당하기 어려운 상황이 생겨났습니다:

- **대규모 데이터**: 소셜 미디어에서 하루에 수십억 건의 게시물이 생성됨
- **유연한 구조**: 게시물마다 사진이 있을 수도, 없을 수도, 투표가 붙을 수도 있음
- **빠른 응답**: 수백만 사용자의 동시 요청에 밀리초 단위로 응답해야 함

이런 요구를 충족하기 위해 다양한 NoSQL 데이터베이스가 발전했습니다:

| NoSQL 유형 | 데이터 모델 | 대표 서비스 | 적합한 용도 |
|------------|-------------|------------|-------------|
| **키-값(Key-Value)** | 키로 값을 조회하는 단순 구조 | Amazon DynamoDB | 세션 관리, 장바구니, 캐시 |
| **문서(Document)** | JSON과 유사한 유연한 문서 구조 | Amazon DocumentDB, MongoDB | 콘텐츠 관리, 카탈로그 |
| **그래프(Graph)** | 노드와 엣지로 관계를 표현 | Amazon Neptune | 소셜 네트워크, 추천 엔진 |
| **인메모리(In-Memory)** | 메모리에 데이터를 저장하여 초고속 접근 | Amazon ElastiCache | 캐싱, 실시간 순위표 |

#### 관계형 vs 비관계형 — 언제 무엇을 선택할까?

| 비교 항목 | 관계형 데이터베이스 | 비관계형 데이터베이스 |
|-----------|-------------------|---------------------|
| **데이터 구조** | 고정된 스키마 (테이블) | 유연한 스키마 또는 스키마 없음 |
| **질의 언어** | SQL (표준화) | 서비스마다 고유한 API 또는 쿼리 언어 |
| **확장 방식** | 수직 확장 (더 큰 서버) 위주 | 수평 확장 (서버 추가) 용이 |
| **트랜잭션** | ACID 보장 (강력한 일관성) | 일부만 ACID 지원, 유연한 일관성 모델 |
| **적합한 사례** | 금융 거래, 주문 관리, ERP | 실시간 분석, IoT 데이터, 게임 상태 |

> ⚠️ **주의**: "NoSQL이 더 새롭고 좋다"는 흔한 오해입니다. 관계형과 비관계형 데이터베이스는 우열이 아니라 **용도의 차이**입니다. 실제로 대부분의 기업은 두 종류를 함께 사용합니다. AWS CCP 시험에서도 "어떤 상황에 어떤 데이터베이스가 적합한가?"를 묻는 문제가 자주 출제됩니다.

#### EC2에 직접 설치 vs 관리형 데이터베이스 서비스

> 🔗 **연결**: Chapter 5에서 EC2 인스턴스가 가상 서버임을 배웠습니다. EC2에 MySQL 같은 데이터베이스를 직접 설치할 수도 있습니다. 그렇다면 AWS의 관리형 데이터베이스 서비스는 왜 필요할까요?

> 📝 **관리형 서비스(Managed Service)**: AWS가 인프라 프로비저닝, 소프트웨어 설치, 패치 적용, 백업, 장애 복구 등의 운영 작업을 대신 처리해주는 서비스이다. 사용자는 데이터베이스 *사용*에 집중할 수 있다.

| 운영 작업 | EC2에 직접 설치 | AWS 관리형 서비스 (RDS 등) |
|-----------|----------------|--------------------------|
| OS 설치 및 패치 | 사용자가 직접 | AWS가 관리 |
| DB 소프트웨어 설치 | 사용자가 직접 | AWS가 관리 |
| 자동 백업 | 직접 스크립트 작성 | 자동 설정 (1일~35일 보관) |
| 장애 시 복구 | 직접 감지 + 수동 복구 | 자동 장애 감지 + 자동 전환 (Multi-AZ) |
| 확장 | 직접 서버 변경, 복제 구성 | 콘솔에서 클릭 몇 번으로 확장 |
| 보안 패치 | 직접 모니터링 + 적용 | AWS가 자동/예약 적용 |

관리형 서비스를 사용하면 **비차별화된 무거운 작업(undifferentiated heavy lifting)** — 즉, 비즈니스에 직접적인 가치를 주지 않는 인프라 관리 작업 — 을 AWS에 맡기고, 개발팀은 애플리케이션과 비즈니스 로직에 집중할 수 있습니다.

### 상세 예시

> 🔍 **예시 8.1.1: 온라인 쇼핑몰의 데이터베이스 선택**
>
> **상황**: 온라인 쇼핑몰을 구축하려 합니다. 관리해야 할 데이터가 여러 종류입니다.
> **풀이/설명**:
> - **주문 정보** (주문번호, 고객, 상품, 금액, 결제 상태): 정확한 구조, ACID 트랜잭션 필수 → **관계형 DB (RDS)**
> - **상품 카탈로그** (상품마다 속성이 다름 — 의류는 사이즈/색상, 전자제품은 사양/무게): 유연한 스키마 필요 → **문서형 NoSQL (DynamoDB 또는 DocumentDB)**
> - **장바구니 세션** (빠른 읽기/쓰기, 일시적 데이터): 초고속 접근 필요 → **인메모리 캐시 (ElastiCache)**
>
> **핵심 포인트**: 하나의 애플리케이션에서도 데이터 특성에 따라 여러 종류의 데이터베이스를 조합하여 사용하는 것이 일반적입니다. 이를 **다중 모델(polyglot persistence)** 접근법이라 합니다.

> 🔍 **예시 8.1.2: 관리형 데이터베이스의 실질적 이점**
>
> **상황**: 스타트업 A팀(개발자 3명)은 EC2에 PostgreSQL을 직접 설치하여 운영했습니다. 새벽 3시에 디스크가 가득 차서 서비스가 중단되었고, 백업도 2주 전 것만 남아 있었습니다. 스타트업 B팀(개발자 3명)은 Amazon RDS를 사용했습니다. RDS가 자동으로 매일 백업을 생성하고, 스토리지 자동 확장 기능으로 디스크 부족 문제를 사전에 방지했습니다.
> **풀이/설명**: A팀은 DB 운영에 주당 약 10시간을 쓰는 반면, B팀은 거의 0시간을 씁니다. B팀의 개발자는 그 시간에 새 기능을 개발합니다.
> **핵심 포인트**: 관리형 서비스는 비용이 약간 더 들 수 있지만, 운영 부담 감소와 안정성 향상이라는 측면에서 특히 소규모 팀에 큰 이점을 제공합니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
> - **"NoSQL은 SQL을 전혀 사용하지 않는다"는 오해**: NoSQL은 "Not Only SQL"의 약자입니다. 일부 NoSQL 데이터베이스(예: DynamoDB의 PartiQL)는 SQL과 유사한 쿼리 언어를 지원합니다.
> - **"관계형 DB는 확장이 안 된다"는 오해**: 관계형 DB도 읽기 전용 복제본(Read Replica)으로 읽기 성능을 확장할 수 있으며, Aurora는 매우 높은 수준의 확장성을 제공합니다. 다만, 쓰기 확장은 NoSQL에 비해 제한적입니다.
> - **시험 팁**: AWS CCP 시험에서 "ACID 트랜잭션이 필요한 경우"라는 조건이 나오면 관계형 데이터베이스(RDS, Aurora)를, "밀리초 단위의 지연 시간이 필요한 키-값 데이터"라는 조건이 나오면 DynamoDB를 선택하세요.

### Section 요약

- **데이터베이스**는 데이터를 체계적으로 저장·검색하는 시스템이며, S3/EBS 같은 스토리지 위에서 구조와 검색 기능을 제공한다
- **관계형 DB**는 테이블 기반 구조, SQL 사용, ACID 보장이 특징이며 정형 데이터와 트랜잭션 처리에 적합하다
- **비관계형 DB(NoSQL)**는 유연한 스키마와 수평 확장이 장점이며 대규모 비정형 데이터 처리에 적합하다
- **관리형 서비스**는 인프라 운영을 AWS에 위임하여 개발에 집중할 수 있게 해준다
- 실무에서는 데이터 특성에 따라 관계형과 비관계형을 조합하여 사용하는 것이 일반적이다

## 8.2 Amazon RDS

> 💡 **한 줄 요약**: Amazon RDS는 AWS가 운영 부담을 대신 처리해주는 관리형 관계형 데이터베이스 서비스로, 6가지 인기 데이터베이스 엔진을 지원한다.

### 직관적 이해

직접 집을 지어 관리하는 것(EC2에 DB 설치)과 관리비를 내고 아파트에 사는 것(RDS 사용)을 비교해 봅시다. 아파트에 살면 건물 유지보수, 보안, 주차장 관리 등은 관리사무소가 해줍니다. 입주자는 **인테리어(데이터베이스 설계와 쿼리 최적화)에만 집중**하면 됩니다. Amazon RDS가 바로 이 "관리사무소" 역할을 합니다.

왜 이것이 필요할까요? Section 8.1에서 살펴본 것처럼, 데이터베이스를 직접 운영하려면 설치, 패치, 백업, 장애 복구, 확장 등 수많은 운영 작업이 필요합니다. RDS는 이 모든 것을 자동화하여, 개발자가 **데이터와 애플리케이션 로직**에만 집중할 수 있게 합니다.

### 핵심 개념 설명

#### Amazon RDS란?

> 📝 **Amazon RDS(Relational Database Service)**: AWS에서 제공하는 관리형 관계형 데이터베이스 서비스이다. 하드웨어 프로비저닝, 데이터베이스 설정, 패치, 백업 등의 운영 작업을 AWS가 자동으로 처리한다.

RDS가 지원하는 **6가지 데이터베이스 엔진**:

| 엔진 | 특징 | 대표 사용 사례 |
|------|------|---------------|
| **MySQL** | 가장 널리 사용되는 오픈소스 DB | 웹 애플리케이션, CMS |
| **PostgreSQL** | 고급 기능이 풍부한 오픈소스 DB | GIS, 복잡한 쿼리, 분석 |
| **MariaDB** | MySQL에서 포크된 오픈소스 DB | MySQL 대안, 호환성 필요 시 |
| **Oracle** | 엔터프라이즈 상용 DB | 대기업 ERP, 레거시 시스템 |
| **Microsoft SQL Server** | Windows 환경의 상용 DB | .NET 애플리케이션, 기업 환경 |
| **Amazon Aurora** | AWS 자체 개발 클라우드 네이티브 DB | 고성능·고가용성 요구 환경 |

> 💬 **참고**: Aurora는 RDS 콘솔에서 생성할 수 있지만, 아키텍처가 다른 엔진들과 크게 다르므로 Section 8.3에서 별도로 자세히 다룹니다.

#### RDS가 관리해주는 것 vs 사용자가 관리하는 것

RDS가 "관리형"이라고 해서 모든 것을 AWS가 해주는 것은 아닙니다:

**AWS가 관리하는 영역:**
- 기본 인프라 (서버, 스토리지, 네트워크)
- 데이터베이스 엔진 설치 및 패치
- 자동 백업 및 특정 시점 복원(Point-in-Time Recovery)
- Multi-AZ 배포 시 자동 장애 조치(Failover)
- 모니터링 메트릭 수집 (CloudWatch 연동)

**사용자가 관리하는 영역:**
- 데이터베이스 스키마 설계 (테이블 구조)
- 쿼리 작성 및 최적화
- 사용자 계정 및 권한 관리 (DB 내부 사용자)
- 인스턴스 사양 선택 (CPU, 메모리 크기)
- 파라미터 그룹 설정 (DB 엔진 세부 설정)

> ⚠️ **주의**: RDS에서는 기본 운영체제(OS)에 직접 SSH로 접속할 수 없습니다. 이것이 EC2에 DB를 직접 설치하는 것과의 중요한 차이점입니다. OS 레벨의 커스터마이징이 꼭 필요하다면 EC2에 DB를 직접 설치하는 방식을 고려해야 합니다.

#### Multi-AZ 배포 — 고가용성

> 📝 **Multi-AZ 배포**: RDS 인스턴스를 두 개의 가용 영역(Availability Zone)에 걸쳐 배포하는 방식이다. 기본(Primary) 인스턴스와 대기(Standby) 인스턴스가 동기적으로 데이터를 복제하며, 기본 인스턴스에 장애가 발생하면 자동으로 대기 인스턴스로 전환(Failover)된다.

> 🔗 **연결**: Chapter 3에서 가용 영역(AZ)이 하나의 리전 내에서 물리적으로 분리된 데이터 센터 그룹임을 배웠습니다. Multi-AZ는 이 AZ 분리를 활용하여 하나의 데이터 센터에 문제가 생겨도 서비스가 중단되지 않도록 합니다.

```
                    ┌─────────────────────────────────────────┐
                    │              리전 (Region)                │
                    │                                           │
                    │   AZ-a                    AZ-b            │
                    │  ┌──────────┐          ┌──────────┐      │
                    │  │ Primary  │  동기식  │ Standby  │      │
  애플리케이션 ────→│  │   DB     │ ──복제─→ │   DB     │      │
                    │  │ (읽기+   │          │ (대기)   │      │
                    │  │  쓰기)   │          │          │      │
                    │  └──────────┘          └──────────┘      │
                    │       ↓ 장애 발생 시 자동 전환 (Failover)  │
                    └─────────────────────────────────────────┘
```

Multi-AZ의 핵심 특성:
- **목적**: **고가용성(High Availability)**. 장애 시 자동 복구
- **복제 방식**: **동기식** — Primary에 쓴 데이터가 Standby에도 즉시 복제됨
- **Standby 역할**: 읽기/쓰기 모두 불가. 오직 **장애 대기** 용도
- **장애 전환 시간**: 일반적으로 60~120초
- **DNS 변경**: 장애 전환 시 RDS의 DNS 엔드포인트가 자동으로 Standby를 가리키므로, 애플리케이션 코드 변경 불필요

#### 읽기 전용 복제본 (Read Replica) — 성능 확장

> 📝 **읽기 전용 복제본(Read Replica)**: Primary 인스턴스의 데이터를 비동기적으로 복제하는 별도의 인스턴스이다. 읽기(SELECT) 쿼리를 분산하여 전체 데이터베이스의 읽기 성능을 향상시킨다.

```
                          ┌──────────────┐
              쓰기 ──────→│   Primary    │
                          │     DB       │
                          └──────┬───────┘
                                 │ 비동기식 복제
                    ┌────────────┼────────────┐
                    ▼            ▼             ▼
              ┌──────────┐ ┌──────────┐ ┌──────────┐
   읽기 ────→│ Replica 1│ │ Replica 2│ │ Replica 3│
              └──────────┘ └──────────┘ └──────────┘
```

Read Replica의 핵심 특성:
- **목적**: **읽기 성능 확장(Read Scaling)**
- **복제 방식**: **비동기식** — Primary보다 약간의 지연(lag)이 있을 수 있음
- **읽기 가능**: 읽기 전용 쿼리를 처리할 수 있음 (쓰기 불가)
- **최대 개수**: RDS 엔진에 따라 **최대 5~15개** 생성 가능
- **리전 간 복제**: 다른 리전에 Read Replica를 생성하여 글로벌 읽기 성능 향상 가능

#### Multi-AZ vs Read Replica — 핵심 비교

| 비교 항목 | Multi-AZ | Read Replica |
|-----------|----------|--------------|
| **주요 목적** | 고가용성 (장애 대비) | 읽기 성능 확장 |
| **복제 방식** | 동기식 | 비동기식 |
| **읽기 가능 여부** | ❌ Standby는 읽기 불가 | ✅ 읽기 전용 쿼리 가능 |
| **장애 시 동작** | 자동 전환 (Failover) | 수동으로 Primary로 승격 가능 |
| **리전** | 같은 리전 내 다른 AZ | 같은 리전 또는 다른 리전 |
| **비용** | 2배 (Primary + Standby) | 추가 인스턴스 비용 |

> ⚠️ **주의**: Multi-AZ와 Read Replica를 혼동하는 것은 AWS 시험에서 가장 흔한 실수 중 하나입니다.
> - **"고가용성", "장애 자동 복구", "다운타임 최소화"** → Multi-AZ
> - **"읽기 성능 향상", "보고서 쿼리 분산", "글로벌 읽기"** → Read Replica
> - 둘은 동시에 사용할 수 있습니다! Multi-AZ로 가용성을 확보하면서 Read Replica로 읽기 성능도 확장하는 구성이 일반적입니다.

#### RDS 스토리지와 백업

RDS는 내부적으로 **Amazon EBS(Elastic Block Store)** 볼륨을 스토리지로 사용합니다.

> 🔗 **연결**: Chapter 6에서 배운 EBS는 EC2에 연결하는 블록 스토리지였습니다. RDS도 마찬가지로 EBS를 사용하지만, 스토리지 관리는 AWS가 자동으로 처리합니다.

**자동 백업 기능:**
- 매일 자동으로 전체 백업(스냅샷) 생성
- 5분 간격으로 트랜잭션 로그 백업
- **특정 시점 복원(Point-in-Time Recovery)**: 보관 기간(1~35일) 내 **아무 시점**으로 데이터베이스를 복원 가능
- 수동 스냅샷: 사용자가 원하는 시점에 수동으로 스냅샷을 생성할 수도 있으며, 이 스냅샷은 삭제할 때까지 영구 보관

### 상세 예시

> 🔍 **예시 8.2.1: 전자상거래 사이트의 RDS 구성**
>
> **상황**: 하루 주문 건수 10만 건인 전자상거래 사이트의 데이터베이스를 구성해야 합니다. 주문 처리가 절대 중단되면 안 되고, 데이터 분석팀이 매일 대량의 읽기 쿼리를 실행합니다.
> **풀이/설명**:
> 1. **엔진 선택**: 주문 데이터는 ACID 트랜잭션이 필수이므로 관계형 DB → MySQL 또는 PostgreSQL
> 2. **Multi-AZ 활성화**: 장애 시에도 주문 처리가 중단되지 않도록 고가용성 확보
> 3. **Read Replica 생성**: 분석팀의 대량 읽기 쿼리를 Read Replica로 분산하여, Primary 인스턴스가 주문 처리(쓰기)에 집중하도록 함
> 4. **자동 백업 설정**: 보관 기간 7일로 설정하여, 필요 시 7일 이내 아무 시점으로 복원 가능
>
> **핵심 포인트**: Multi-AZ는 "장애 대비", Read Replica는 "읽기 부하 분산"이라는 서로 다른 목적으로 동시에 사용합니다.

> 🔍 **예시 8.2.2: RDS 엔진 선택 가이드**
>
> **상황**: 다양한 프로젝트에서 어떤 RDS 엔진을 선택해야 할지 결정해야 합니다.
> **풀이/설명**:
> - **기존에 Oracle DB를 사용하던 기업이 AWS로 이전**: Oracle 엔진 선택 (호환성 최우선)
> - **비용을 최소화하면서 웹 서비스 구축**: MySQL 또는 PostgreSQL (오픈소스, 라이선스 비용 없음)
> - **.NET 기반 사내 시스템을 AWS로 이전**: Microsoft SQL Server
> - **최고 성능과 가용성이 필요하되, 특정 DB에 종속되지 않으려는 경우**: Amazon Aurora (Section 8.3에서 자세히 설명)
>
> **핵심 포인트**: 기존 시스템과의 호환성, 라이선스 비용, 팀의 기술 숙련도를 종합적으로 고려하여 엔진을 선택합니다.

### 미니 실습

> 🧪 **Mini Lab 8.2: RDS MySQL 인스턴스 생성하기**
>
> **목표**: AWS 콘솔에서 RDS MySQL 인스턴스를 생성하고, 데이터베이스 엔드포인트를 확인한다.
> **예상 소요 시간**: 15분 (인스턴스 생성에 약 5~10분 대기 포함)
> **비용**: Free Tier 범위 내 (db.t3.micro, 단일 AZ, 범용 SSD 20GB)
>
> **Step 1**: AWS 콘솔에 로그인한 뒤, 상단 검색창에 `RDS`를 입력하고 **Amazon RDS** 서비스를 클릭합니다.
>
> **Step 2**: 왼쪽 메뉴에서 **데이터베이스(Databases)**를 클릭한 뒤, 오른쪽 상단의 **데이터베이스 생성(Create database)** 버튼을 클릭합니다.
>
> **Step 3**: 데이터베이스 생성 방법에서 **표준 생성(Standard create)**을 선택합니다.
>
> **Step 4**: 엔진 옵션에서 다음을 선택합니다:
> - **엔진 유형**: MySQL
> - **에디션**: MySQL Community
> - **엔진 버전**: 기본값(최신 안정 버전) 유지
>
> **Step 5**: 템플릿에서 **프리 티어(Free tier)**를 선택합니다. 이렇게 하면 Multi-AZ 등 비용이 발생하는 옵션이 자동으로 비활성화됩니다.
>
> **Step 6**: 설정(Settings) 섹션에서 다음을 입력합니다:
> - **DB 인스턴스 식별자**: `my-first-rds`
> - **마스터 사용자 이름**: `admin`
> - **마스터 암호**: 기억할 수 있는 안전한 비밀번호 입력 (예: `MyPassword123!`)
>
> **Step 7**: 인스턴스 구성을 확인합니다:
> - **DB 인스턴스 클래스**: db.t3.micro (Free Tier에 해당) — 프리 티어 템플릿 선택 시 자동 설정됨
>
> **Step 8**: 스토리지 섹션을 확인합니다:
> - **스토리지 유형**: 범용 SSD (gp2)
> - **할당된 스토리지**: 20 GiB
> - **스토리지 자동 조정**: 체크 해제 (Free Tier 범위 유지를 위해)
>
> **Step 9**: 연결(Connectivity) 섹션에서:
> - **컴퓨팅 리소스**: EC2 컴퓨팅 리소스에 연결 안 함(Don't connect to an EC2 compute resource) 선택
> - **퍼블릭 액세스**: 아니요(No) — 실습 확인만 할 것이므로
> - 나머지는 기본값 유지
>
> **Step 10**: 페이지 하단의 **데이터베이스 생성(Create database)** 버튼을 클릭합니다.
>
> **Step 11**: 데이터베이스 목록에서 `my-first-rds`의 상태가 **생성 중(Creating)**에서 **사용 가능(Available)**으로 변할 때까지 기다립니다 (약 5~10분).
>
> **확인**:
> - 상태가 **사용 가능(Available)**으로 표시됨
> - `my-first-rds`를 클릭하면 상세 정보가 나타남
> - **연결 및 보안(Connectivity & security)** 탭에서 **엔드포인트(Endpoint)** 주소 확인 (예: `my-first-rds.xxxxxxxxxxxx.ap-northeast-2.rds.amazonaws.com`)
> - **구성(Configuration)** 탭에서 엔진 버전, 인스턴스 클래스(db.t3.micro), 스토리지 등을 확인
> - **유지 관리 및 백업(Maintenance & backups)** 탭에서 자동 백업이 활성화되어 있는 것을 확인
>
> **🧹 정리**: 이 인스턴스는 통합 실습(Part C-2)에서 한 번 더 확인하므로, 지금은 유지합니다. Part C-2 완료 후 삭제 안내를 따르세요. 만약 바로 삭제하려면:
> 1. 데이터베이스 목록에서 `my-first-rds`를 선택
> 2. **작업(Actions)** → **삭제(Delete)** 클릭
> 3. "최종 스냅샷을 생성하시겠습니까?"에서 **아니요(No)** 선택
> 4. "모든 자동 백업을 삭제합니다" 체크
> 5. 확인 문구 `delete me`를 입력하고 **삭제** 클릭

### 흔한 실수와 주의사항

> ⚠️ **주의**:
> - **Free Tier 초과 주의**: RDS Free Tier는 **db.t3.micro + 단일 AZ + 20GB SSD**에 한정됩니다. Multi-AZ를 선택하면 Free Tier 대상이 아닙니다. 반드시 "프리 티어" 템플릿을 선택하세요.
> - **RDS는 SSH 접속 불가**: RDS 인스턴스의 운영체제에 직접 접근할 수 없습니다. DB 접속은 MySQL 클라이언트, pgAdmin 등 데이터베이스 클라이언트를 통해 엔드포인트로 연결합니다.
> - **엔드포인트 주소 기억**: RDS는 IP 주소가 아닌 **DNS 엔드포인트**를 통해 접속합니다. 이 엔드포인트는 Multi-AZ 장애 전환 시에도 변하지 않으므로 애플리케이션에서 IP를 하드코딩하지 않아야 합니다.

### Section 요약

- **Amazon RDS**는 6가지 엔진(MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Aurora)을 지원하는 관리형 관계형 DB 서비스이다
- AWS가 인프라, 패치, 백업, 장애 복구를 관리하고, 사용자는 스키마 설계와 쿼리 최적화에 집중한다
- **Multi-AZ**는 동기식 복제로 고가용성을 제공하고, **Read Replica**는 비동기식 복제로 읽기 성능을 확장한다
- 자동 백업과 특정 시점 복원(PITR)으로 데이터 손실을 방지한다
- RDS는 OS에 직접 접근할 수 없으며, DNS 엔드포인트로 접속한다

## 8.3 Amazon Aurora

> 💡 **한 줄 요약**: Amazon Aurora는 AWS가 클라우드 환경에 최적화하여 자체 개발한 관계형 데이터베이스로, 상용 DB 수준의 성능을 오픈소스 DB 수준의 비용으로 제공한다.

### 직관적 이해

RDS가 기존 아파트에 입주하는 것이라면, Aurora는 AWS가 **클라우드 환경에 처음부터 설계하여 새로 지은 최신 아파트**에 입주하는 것과 같습니다. 기존 아파트(일반 RDS)도 충분히 좋지만, 신축 아파트(Aurora)는 최신 설계로 에너지 효율(성능), 내진 설계(가용성), 스마트홈 시스템(자동 관리) 등이 처음부터 내장되어 있습니다.

Aurora는 왜 만들어졌을까요? 기존 관계형 데이터베이스는 원래 단일 서버에서 동작하도록 설계되었습니다. 이것을 클라우드의 분산 환경에 맞추려면 많은 비효율이 생깁니다. AWS는 "클라우드에 처음부터 맞게 설계된 관계형 DB"를 만들기로 결정했고, 그 결과가 Aurora입니다.

### 핵심 개념 설명

#### Amazon Aurora란?

> 📝 **Amazon Aurora**: AWS가 클라우드 환경에 최적화하여 자체 개발한 관계형 데이터베이스 엔진이다. MySQL 및 PostgreSQL과 호환되며, 표준 MySQL보다 최대 5배, 표준 PostgreSQL보다 최대 3배 높은 처리량을 제공한다.

Aurora는 RDS 콘솔에서 생성하지만, 내부 아키텍처는 다른 RDS 엔진과 크게 다릅니다:

#### Aurora의 스토리지 아키텍처

일반 RDS는 인스턴스에 EBS 볼륨이 직접 연결되지만, Aurora는 **스토리지 계층이 완전히 분리**되어 있습니다:

```
    일반 RDS:                          Aurora:
    ┌──────────┐                     ┌──────────┐
    │ DB 인스턴스│                    │ DB 인스턴스│
    │    +     │                     └────┬─────┘
    │ EBS 볼륨  │                          │
    └──────────┘                     ┌────▼──────────────────────┐
                                     │    Aurora 공유 스토리지     │
                                     │  (3개 AZ × 2카피 = 6카피)  │
                                     │  자동 확장: 10GB → 128TB   │
                                     └───────────────────────────┘
```

Aurora 스토리지의 핵심 특성:
- **6개 데이터 복제본**: 3개의 가용 영역에 각 2개씩, 총 6개의 데이터 복사본을 자동으로 유지
- **자동 복구**: 6개 중 2개가 손실되어도 쓰기 가능, 3개가 손실되어도 읽기 가능
- **자동 확장**: 10GB에서 시작하여 최대 128TB까지 자동으로 확장 — 사용자가 스토리지를 미리 할당할 필요 없음
- **연속 백업**: Amazon S3에 지속적으로 백업되어 백업으로 인한 성능 저하 없음

#### Aurora vs 일반 RDS — 무엇이 다른가?

| 비교 항목 | 일반 RDS (MySQL/PostgreSQL 등) | Amazon Aurora |
|-----------|-------------------------------|---------------|
| **성능** | 표준 수준 | MySQL 대비 최대 5배, PostgreSQL 대비 3배 |
| **스토리지** | EBS (최대 64TB) | 공유 분산 스토리지 (최대 128TB, 자동 확장) |
| **데이터 복제** | Multi-AZ: 2개 AZ | 자동으로 3개 AZ에 6개 복사본 |
| **장애 복구 시간** | 60~120초 | 일반적으로 30초 이내 |
| **Read Replica** | 최대 5~15개 | 최대 15개 (장애 시 자동 승격 가능) |
| **비용** | 엔진별 표준 요금 | 일반 RDS 대비 약 20% 높음 |
| **호환성** | 각 엔진 고유 | MySQL 또는 PostgreSQL과 호환 |

#### Aurora Serverless

> 📝 **Aurora Serverless**: Aurora의 온디맨드 자동 확장 구성이다. 사용량에 따라 데이터베이스의 컴퓨팅 용량이 자동으로 확장·축소되며, 사용한 만큼만 비용을 지불한다.

일반 Aurora(또는 RDS)는 인스턴스 크기를 미리 선택해야 합니다. 하지만 트래픽이 불규칙한 서비스라면 어떨까요?

```
   트래픽이 불규칙한 애플리케이션:

   일반 Aurora:  ████████████████  (고정 용량, 남는 부분은 낭비)
   실제 트래픽:   █ █████ ██ █████
   Aurora Serverless: █ █████ ██ █████  (사용량에 맞춰 자동 조절)
```

Aurora Serverless가 적합한 경우:
- 트래픽이 예측 불가능한 신규 애플리케이션
- 개발/테스트 환경 (업무 시간에만 사용)
- 간헐적으로 실행되는 배치 작업

Aurora Serverless가 적합하지 않은 경우:
- 트래픽이 예측 가능하고 일정한 프로덕션 워크로드 (일반 Aurora가 비용 효율적)

### 상세 예시

> 🔍 **예시 8.3.1: Aurora를 선택해야 하는 상황**
>
> **상황**: 중견 전자상거래 기업이 MySQL로 운영하던 주문 시스템을 AWS로 이전하려 합니다. 요구사항은 다음과 같습니다: (1) MySQL 호환성 필요, (2) 다운타임 최소화, (3) 읽기 부하가 높아 Read Replica 활용 필요, (4) 블랙프라이데이 시즌에 트래픽 5배 증가
> **풀이/설명**:
> - MySQL 호환 → Aurora MySQL 선택
> - 다운타임 최소화 → Aurora의 자동 장애 복구 (30초 이내)
> - 읽기 부하 → Aurora Read Replica 최대 15개 지원, 자동 승격 가능
> - 시즌 트래픽 → Aurora Auto Scaling으로 Read Replica를 자동 추가/제거
>
> **핵심 포인트**: Aurora는 MySQL/PostgreSQL과 호환되므로 기존 애플리케이션의 코드 변경을 최소화하면서 성능과 가용성을 크게 향상시킬 수 있습니다.

> 🔍 **예시 8.3.2: Aurora Serverless의 활용**
>
> **상황**: 스타트업이 신규 모바일 앱을 출시했습니다. 사용자가 언제 급증할지 예측할 수 없고, 야간에는 트래픽이 거의 없습니다. 예산이 제한적이어서 고정 인스턴스 비용을 줄이고 싶습니다.
> **풀이/설명**: Aurora Serverless를 선택하면:
> - 야간 트래픽이 없을 때 → 최소 용량으로 자동 축소 (비용 절감)
> - 갑자기 바이럴되어 트래픽 폭증 시 → 자동으로 용량 확장 (서비스 안정)
> - 사용한 용량만큼만 과금 → 예산 효율적
>
> **핵심 포인트**: Aurora Serverless는 "예측 불가능한 트래픽 + 비용 최적화"가 핵심 요구사항일 때 적합합니다.

### 깊이 있는 이해

> 🔬 **Deep Dive**: Aurora의 쿼럼(Quorum) 기반 복제
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> Aurora가 6개의 복사본을 유지하는 이유는 **쿼럼(Quorum)** 프로토콜 때문입니다. 쿼럼이란 "과반수의 동의"를 의미합니다.
>
> - **쓰기**: 6개 중 4개 노드에 성공적으로 쓰면 쓰기 완료로 간주 (4/6 쿼럼)
> - **읽기**: 6개 중 3개 노드에서 읽으면 최신 데이터 확인 가능 (3/6 쿼럼)
>
> 이 설계 덕분에:
> - 하나의 AZ 전체가 장애나도 (2개 복사본 손실) → 남은 4개로 쓰기 가능
> - 2개 노드 추가 장애 (총 4개 손실) → 남은 2개로 읽기는 가능
>
> 이것이 Aurora가 일반 RDS 대비 더 빠른 장애 복구와 높은 내구성을 제공하는 핵심 원리입니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
> - **"Aurora가 항상 RDS보다 좋다"는 오해**: Aurora는 약 20% 정도 비용이 높습니다. 소규모 워크로드나 Free Tier 활용 시에는 일반 RDS가 더 적합할 수 있습니다.
> - **Aurora ≠ 새로운 DB 언어**: Aurora는 MySQL 또는 PostgreSQL과 **호환**됩니다. 기존 MySQL/PostgreSQL 코드를 거의 수정 없이 사용할 수 있습니다.
> - **Aurora Serverless와 Lambda 혼동 주의**: 둘 다 "서버리스"이지만, Aurora Serverless는 데이터베이스 서비스이고 Lambda는 컴퓨팅 서비스입니다.
> - **시험 팁**: "클라우드 네이티브 관계형 데이터베이스", "MySQL/PostgreSQL 호환", "상용 DB 성능을 오픈소스 비용으로" 같은 키워드가 나오면 Aurora를 떠올리세요.

### Section 요약

- **Amazon Aurora**는 AWS가 클라우드에 최적화하여 설계한 관계형 DB로, MySQL/PostgreSQL과 호환된다
- 3개 AZ에 6개 데이터 복사본을 자동으로 유지하여 뛰어난 내구성과 가용성을 제공한다
- 일반 MySQL 대비 최대 5배, PostgreSQL 대비 3배의 성능을 제공한다
- **Aurora Serverless**는 트래픽에 따라 자동 확장/축소되어 불규칙한 워크로드에 적합하다
- 일반 RDS보다 약 20% 비용이 높지만, 고성능·고가용성이 필요한 환경에서 비용 대비 효과가 뛰어나다

## 8.4 Amazon DynamoDB

> 💡 **한 줄 요약**: Amazon DynamoDB는 어떤 규모에서도 한 자릿수 밀리초의 응답 시간을 제공하는 완전 관리형 NoSQL 데이터베이스이다.

### 직관적 이해

도서관 비유를 다시 떠올려 봅시다. Section 8.1에서 "서류 봉투" 방식을 기억하시나요? DynamoDB는 이 봉투 방식을 **극한까지 최적화한 시스템**입니다.

초대형 물류 창고를 상상해 보세요. 수억 개의 물건이 보관되어 있지만, 각 물건에는 고유한 바코드(키)가 붙어 있습니다. 바코드를 스캔하면 **밀리초 만에** 해당 물건의 위치를 찾아줍니다. 물건의 크기나 모양은 제각각이어도 상관없습니다 — 바코드만 있으면 됩니다. 그리고 창고 관리(서버 운영, 확장, 백업)는 모두 물류 회사(AWS)가 처리합니다.

DynamoDB가 탄생한 배경은 Amazon.com의 실제 경험에서 비롯됩니다. 2004년 Amazon의 연말 쇼핑 시즌에 기존 관계형 데이터베이스가 폭증하는 트래픽을 감당하지 못했습니다. 이 경험을 바탕으로 Amazon은 대규모 트래픽에서도 일관된 성능을 보장하는 데이터베이스를 자체 개발했고, 이것이 DynamoDB의 시초가 되었습니다.

### 핵심 개념 설명

#### Amazon DynamoDB란?

> 📝 **Amazon DynamoDB**: AWS에서 제공하는 완전 관리형 NoSQL 데이터베이스 서비스이다. **키-값(key-value)** 및 **문서(document)** 데이터 모델을 지원하며, 어떤 규모에서도 한 자릿수 밀리초의 응답 시간을 제공한다.

"**완전 관리형(Fully Managed)**"이라는 표현에 주목하세요. RDS는 "관리형"이었지만, 사용자가 인스턴스 크기를 선택하고 확장을 직접 결정해야 했습니다. DynamoDB는 여기서 한 단계 더 나아가 **서버 프로비저닝, 패치, 확장 모두를 AWS가 자동으로 처리**합니다. 사용자가 관리할 서버 자체가 없습니다.

> 📝 **키-값 스토어(Key-Value Store)**: 데이터를 고유한 키(Key)와 그에 대응하는 값(Value)의 쌍으로 저장하는 데이터베이스 모델이다. 프로그래밍에서의 사전(dictionary)이나 해시맵(hashmap)과 유사한 개념이다.

#### DynamoDB의 데이터 구조

DynamoDB의 기본 구조는 다음과 같습니다:

```
DynamoDB 테이블: "OnlineShop-Orders"

┌────────────────┬─────────────┬──────────────────────────────────┐
│ Partition Key  │ Sort Key    │ Attributes (속성 — 자유 형식)     │
│ (파티션 키)     │ (정렬 키)    │                                  │
├────────────────┼─────────────┼──────────────────────────────────┤
│ user-001       │ order-2024-1│ {상품: "노트북", 금액: 1200000}   │
│ user-001       │ order-2024-2│ {상품: "마우스", 금액: 35000,      │
│                │             │  색상: "검정"}                     │
│ user-002       │ order-2024-1│ {상품: "키보드", 금액: 89000}      │
└────────────────┴─────────────┴──────────────────────────────────┘
```

> 📝 **파티션 키(Partition Key)**: DynamoDB 테이블에서 각 항목(item)을 고유하게 식별하는 기본 키이다. DynamoDB는 이 키를 기반으로 데이터를 여러 파티션에 분산 저장한다.

> 📝 **정렬 키(Sort Key)**: 파티션 키와 함께 복합 기본 키를 구성하는 선택적 키이다. 같은 파티션 키를 가진 항목들을 정렬하여 저장하며, 범위 검색을 가능하게 한다.

핵심 특성:
- **테이블(Table)**: 항목(Item)들의 모음
- **항목(Item)**: 하나의 데이터 레코드 — 관계형 DB의 "행(row)"에 해당
- **속성(Attribute)**: 항목의 개별 데이터 필드 — 관계형 DB의 "열(column)"에 해당
- **유연한 스키마**: 같은 테이블 안에서도 항목마다 속성의 종류와 수가 다를 수 있음 (위 예시에서 두 번째 주문에만 "색상" 속성이 있음)

#### DynamoDB의 핵심 특성

**1. 무한한 확장성**
- 저장할 수 있는 데이터 양에 제한이 없음
- 트래픽 증가 시 자동으로 파티션을 추가하여 부하 분산
- 하루에 10조 건 이상의 요청 처리 가능

**2. 한 자릿수 밀리초 응답 시간**
- 데이터 크기나 테이블 크기에 관계없이 일관된 성능
- **DAX(DynamoDB Accelerator)**를 사용하면 마이크로초(µs) 수준까지 가능

> 📝 **DAX(DynamoDB Accelerator)**: DynamoDB 전용 인메모리 캐시 서비스이다. 자주 읽는 데이터를 메모리에 캐싱하여 읽기 응답 시간을 밀리초에서 마이크로초 수준으로 단축한다.

**3. 완전 관리형 — 서버리스**
- 서버 프로비저닝, 패치, 확장 모두 AWS가 자동 처리
- 사용자는 테이블을 생성하고 데이터를 읽고 쓰기만 하면 됨
- 내장 암호화, 백업/복원, 특정 시점 복원(PITR) 지원

**4. 두 가지 용량 모드**

| 용량 모드 | 설명 | 적합한 경우 |
|-----------|------|------------|
| **온디맨드(On-Demand)** | 요청 건수만큼 자동 과금 | 트래픽 예측 불가, 신규 서비스 |
| **프로비저닝(Provisioned)** | 초당 읽기/쓰기 용량을 미리 설정 | 트래픽 예측 가능, 비용 최적화 |

**5. DynamoDB 글로벌 테이블**

> 📝 **글로벌 테이블(Global Tables)**: 여러 AWS 리전에 걸쳐 DynamoDB 테이블을 자동으로 복제하는 기능이다. 각 리전에서 읽기와 쓰기가 모두 가능하며(active-active), 리전 간 데이터가 자동으로 동기화된다.

#### DynamoDB vs RDS/Aurora — 언제 무엇을?

| 비교 항목 | RDS / Aurora | DynamoDB |
|-----------|-------------|----------|
| **데이터 모델** | 관계형 (테이블, SQL) | 키-값, 문서 (NoSQL) |
| **스키마** | 고정 스키마 필수 | 유연한 스키마 |
| **확장** | 수직 확장 위주, Read Replica | 자동 수평 확장 |
| **트랜잭션** | 완전한 ACID | DynamoDB Transactions로 ACID 지원 (제한적) |
| **쿼리** | 복잡한 JOIN, 집계 가능 | 기본 키 기반 조회에 최적화 |
| **응답 시간** | 쿼리 복잡도에 따라 다름 | 일관된 한 자릿수 밀리초 |
| **비용 모델** | 인스턴스 시간당 과금 | 요청 건수 또는 용량 단위 과금 |

### 상세 예시

> 🔍 **예시 8.4.1: 모바일 게임의 사용자 상태 관리**
>
> **상황**: 전 세계 1,000만 사용자가 플레이하는 모바일 게임에서, 각 사용자의 게임 상태(레벨, 아이템, 점수)를 저장해야 합니다. 사용자가 게임을 할 때마다 밀리초 단위로 상태를 읽고 업데이트해야 합니다.
> **풀이/설명**:
> - **데이터 구조**: 사용자마다 속성이 다를 수 있음 (보유 아이템, 업적 등) → 유연한 스키마 → DynamoDB
> - **파티션 키**: `user-id` (각 사용자를 고유하게 식별)
> - **성능**: 한 자릿수 밀리초 응답 → DynamoDB의 핵심 장점
> - **확장성**: 사용자가 급증해도 자동 확장 → DynamoDB 자동 스케일링
> - **글로벌**: 전 세계 사용자 → 글로벌 테이블로 각 리전에서 로컬 읽기/쓰기
>
> **핵심 포인트**: "대규모 사용자 + 밀리초 응답 + 유연한 데이터 구조"는 DynamoDB의 대표적인 사용 사례입니다.

> 🔍 **예시 8.4.2: DynamoDB가 부적합한 경우**
>
> **상황**: 회계 시스템에서 "이번 달 부서별 매출 합계를 구하고, 매출 상위 10개 거래처의 거래 내역을 조인하여 보고서를 생성"해야 합니다.
> **풀이/설명**:
> - 여러 테이블의 복잡한 **JOIN**, **GROUP BY**, **집계(SUM, AVG)** 필요
> - DynamoDB는 기본 키 기반의 단순 조회에 최적화되어 있어, 이런 복잡한 분석 쿼리는 비효율적
> - 이 경우 **RDS 또는 Aurora**(관계형 DB)가 적합
>
> **핵심 포인트**: DynamoDB는 "키로 빠르게 찾는 것"에 특화되어 있습니다. 복잡한 관계 분석이나 집계가 필요하면 관계형 DB를 사용하세요.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
> - **"DynamoDB는 SQL을 전혀 지원하지 않는다"는 오해**: DynamoDB는 **PartiQL**이라는 SQL 호환 쿼리 언어를 지원합니다. 다만, JOIN이나 복잡한 집계는 지원하지 않습니다.
> - **파티션 키 설계가 매우 중요**: 파티션 키가 고르게 분포하지 않으면 "핫 파티션" 문제가 생겨 성능이 저하됩니다. 예를 들어, `날짜`를 파티션 키로 쓰면 오늘 데이터에만 트래픽이 집중됩니다.
> - **시험 팁**: "서버리스 NoSQL", "한 자릿수 밀리초", "키-값 스토어", "완전 관리형 NoSQL" → DynamoDB. "마이크로초 지연 시간 + DynamoDB" → DAX.

### Section 요약

- **DynamoDB**는 완전 관리형 NoSQL 데이터베이스로, 키-값 및 문서 모델을 지원한다
- 서버 관리가 필요 없으며(서버리스), 어떤 규모에서도 한 자릿수 밀리초 응답을 제공한다
- 유연한 스키마로 항목마다 다른 속성을 가질 수 있다
- **온디맨드**와 **프로비저닝** 두 가지 용량 모드를 제공한다
- **글로벌 테이블**로 멀티 리전 복제, **DAX**로 마이크로초 캐싱이 가능하다

## 8.5 캐싱과 특수 목적 데이터베이스

> 💡 **한 줄 요약**: AWS는 캐싱(ElastiCache), 데이터 웨어하우스(Redshift), 그래프(Neptune), 문서(DocumentDB), 원장(QLDB) 등 특수 목적에 최적화된 데이터베이스 서비스를 제공한다.

### 직관적 이해

주방을 떠올려 봅시다. 모든 식재료를 냉장고(일반 데이터베이스)에 보관하지만, 자주 쓰는 소금과 후추는 조리대 위(캐시)에 꺼내둡니다. 냉장고 문을 여는 것보다 조리대에서 집는 게 훨씬 빠르니까요. 이것이 **캐싱(Caching)**의 핵심 아이디어입니다.

마찬가지로, 세상의 모든 데이터를 하나의 범용 데이터베이스로 처리할 수 있지만, 특수한 용도에는 **특화된 도구**가 훨씬 효율적입니다. 못을 박을 때 스위스 아미 나이프(범용 DB)보다 망치(특수 목적 DB)가 낫듯이요.

### 핵심 개념 설명

#### Amazon ElastiCache — 인메모리 캐시

> 📝 **Amazon ElastiCache**: AWS에서 제공하는 관리형 인메모리 캐싱 서비스이다. 자주 조회되는 데이터를 메모리에 저장하여 데이터베이스 부하를 줄이고 응답 시간을 **마이크로초~밀리초 수준**으로 단축한다.

> 📝 **인메모리 캐시(In-Memory Cache)**: 데이터를 디스크가 아닌 **RAM(메모리)**에 저장하는 방식이다. 메모리는 디스크보다 수십~수백 배 빠르므로, 자주 접근하는 데이터를 메모리에 올려두면 극적인 성능 향상이 가능하다.

ElastiCache는 두 가지 엔진을 지원합니다:

| 엔진 | 특징 | 적합한 경우 |
|------|------|------------|
| **Redis** | 다양한 데이터 구조, 영속성 지원, Pub/Sub | 실시간 순위표, 세션 관리, 채팅 |
| **Memcached** | 단순 키-값 캐싱, 멀티스레드 | 단순 캐싱, DB 쿼리 결과 저장 |

ElastiCache의 전형적인 사용 패턴:

```
  [사용자 요청]
       │
       ▼
  [애플리케이션] ──→ 캐시에 있나? ──(YES)──→ [ElastiCache] → 즉시 응답 (마이크로초)
       │                                      (Redis/Memcached)
       (NO)
       │
       ▼
  [RDS/Aurora] ──→ DB에서 조회 ──→ 결과를 캐시에 저장 ──→ 응답 (밀리초)
```

> 💬 **참고**: ElastiCache는 독립된 서비스이지만, 주로 RDS/Aurora/DynamoDB 앞에 배치하여 자주 읽는 데이터의 응답 시간을 단축하는 용도로 사용합니다.

#### Amazon Redshift — 데이터 웨어하우스

> 📝 **Amazon Redshift**: 페타바이트(PB) 규모의 데이터를 빠르게 분석할 수 있는 완전 관리형 데이터 웨어하우스 서비스이다. SQL 기반으로 대규모 데이터에 대한 복잡한 분석 쿼리를 실행할 수 있다.

> 📝 **데이터 웨어하우스(Data Warehouse)**: 다양한 소스에서 수집한 대량의 데이터를 **분석** 목적으로 중앙에 모아둔 저장소이다. 일상적인 트랜잭션 처리(OLTP)가 아닌, 비즈니스 인사이트를 얻기 위한 분석 쿼리(OLAP)에 최적화되어 있다.

RDS/Aurora와 Redshift의 차이를 이해하려면, **OLTP와 OLAP**를 구분해야 합니다:

| 구분 | OLTP (온라인 트랜잭션 처리) | OLAP (온라인 분석 처리) |
|------|---------------------------|----------------------|
| **목적** | 일상적 데이터 처리 | 데이터 분석, 리포팅 |
| **쿼리 특성** | 간단한 읽기/쓰기 (1건씩) | 대량 데이터 집계, 복잡한 분석 |
| **예시** | 주문 접수, 결제 처리 | "지난 1년간 지역별 매출 추이" |
| **적합한 서비스** | RDS, Aurora, DynamoDB | **Redshift** |

Redshift가 분석에 특화된 이유:
- **열 기반(Columnar) 스토리지**: 행이 아닌 열 단위로 데이터를 저장하여, 특정 열의 집계가 매우 빠름
- **대규모 병렬 처리(MPP)**: 쿼리를 여러 노드에 분산하여 동시에 처리
- **데이터 압축**: 열 기반 저장 덕분에 높은 압축률 → 스토리지 비용 절감

> 💬 **참고**: Redshift는 Chapter 11(분석 서비스)에서 더 자세히 다루게 됩니다. 이 Section에서는 "분석 전용 데이터베이스"라는 역할을 이해하는 것이 핵심입니다.

#### Amazon Neptune — 그래프 데이터베이스

> 📝 **Amazon Neptune**: 관계(relationship) 중심의 데이터를 효율적으로 저장하고 쿼리하는 완전 관리형 그래프 데이터베이스 서비스이다. 수십억 개의 관계를 밀리초 이내에 탐색할 수 있다.

그래프 데이터베이스는 데이터 간의 **연결(관계)이 핵심**인 경우에 사용합니다:

```
  소셜 네트워크 예시:

     [철수] ──친구──→ [영희]
       │                │
     팔로우            좋아요
       ↓                ↓
     [민수] ──친구──→ [수진] ──작성──→ [게시물A]
```

관계형 DB에서 이런 관계를 탐색하려면 수많은 JOIN이 필요하지만, 그래프 DB는 관계 탐색이 **네이티브**입니다.

**Neptune의 대표 사용 사례**:
- 소셜 네트워크 (친구 추천, 영향력 분석)
- 사기 탐지 (의심스러운 거래 패턴 발견)
- 지식 그래프 (Wikipedia 같은 개념 간 관계 표현)
- 추천 엔진 ("이 상품을 구매한 사람들이 함께 구매한 상품")

#### Amazon DocumentDB — 문서 데이터베이스

> 📝 **Amazon DocumentDB**: MongoDB와 호환되는 완전 관리형 문서 데이터베이스 서비스이다. JSON과 유사한 문서 형태로 데이터를 저장하며, 유연한 스키마와 풍부한 쿼리 기능을 제공한다.

DocumentDB는 DynamoDB와 달리 **문서 내부를 검색하는 복잡한 쿼리**가 가능합니다:

- DynamoDB: 키(Primary Key)로 빠르게 찾는 데 최적화
- DocumentDB: 문서 내부의 중첩된 필드까지 검색, 집계 가능

**DocumentDB의 대표 사용 사례**: 콘텐츠 관리 시스템(CMS), 사용자 프로필 관리, 카탈로그 관리

#### Amazon QLDB — 원장 데이터베이스

> 📝 **Amazon QLDB(Quantum Ledger Database)**: 모든 데이터 변경 이력을 **변경 불가능(immutable)**하고 **암호화 검증 가능(cryptographically verifiable)**한 형태로 기록하는 완전 관리형 원장(Ledger) 데이터베이스이다.

"원장(Ledger)"은 회계에서 쓰는 장부를 의미합니다. 한 번 기록한 내용을 수정하거나 삭제할 수 없으며, 누가 언제 무엇을 변경했는지 완전한 이력이 남습니다.

**QLDB의 대표 사용 사례**:
- 금융 거래 기록 (변경 이력의 무결성 보장)
- 공급망 추적 (제품의 이동 경로를 변조 불가능하게 기록)
- 규제 준수 (감사 추적 기록이 필요한 경우)

#### AWS의 목적별 데이터베이스 전략 — 한눈에 보기

| 서비스 | 유형 | 핵심 특징 | "이것이 필요하면" |
|--------|------|-----------|-----------------|
| **RDS** | 관계형 (관리형) | 6개 엔진, Multi-AZ, Read Replica | 일반적인 관계형 DB 필요 |
| **Aurora** | 관계형 (클라우드 네이티브) | 고성능, 고가용성, MySQL/PostgreSQL 호환 | 상용 DB 수준 성능이 필요 |
| **DynamoDB** | 키-값 / 문서 (NoSQL) | 서버리스, 밀리초 응답, 무한 확장 | 대규모 NoSQL, 빠른 키-값 조회 |
| **ElastiCache** | 인메모리 캐시 | Redis/Memcached, 마이크로초 응답 | DB 앞단 캐싱, 실시간 데이터 |
| **Redshift** | 데이터 웨어하우스 | 열 기반, 페타바이트 분석 | 대규모 데이터 분석 (OLAP) |
| **Neptune** | 그래프 DB | 관계 탐색 최적화 | 소셜 네트워크, 추천 엔진 |
| **DocumentDB** | 문서 DB | MongoDB 호환 | 유연한 문서, 복잡한 문서 쿼리 |
| **QLDB** | 원장 DB | 변경 불가, 암호화 검증 | 변조 불가능한 이력 관리 |

### 상세 예시

> 🔍 **예시 8.5.1: ElastiCache로 웹사이트 성능 개선**
>
> **상황**: 뉴스 웹사이트의 메인 페이지를 열 때마다 RDS에서 "인기 기사 Top 10"을 쿼리합니다. 하루 방문자 100만 명이 매번 같은 쿼리를 실행하여 DB에 과부하가 걸립니다.
> **풀이/설명**:
> 1. ElastiCache(Redis)를 도입하여 "인기 기사 Top 10" 결과를 캐시에 저장
> 2. 캐시 만료 시간(TTL)을 5분으로 설정 — 5분마다 최신 데이터로 갱신
> 3. 100만 명의 요청 중 99.9%가 캐시에서 처리되어, RDS로 가는 쿼리가 하루 288건(5분 × 288 = 24시간)으로 감소
>
> **핵심 포인트**: 캐싱은 "같은 데이터를 반복적으로 읽는" 패턴에서 극적인 효과를 발휘합니다. DB 부하 감소 + 응답 시간 단축이라는 두 가지 이점을 동시에 얻습니다.

> 🔍 **예시 8.5.2: 목적에 맞는 데이터베이스 선택**
>
> **상황**: 다음 각 요구사항에 가장 적합한 AWS 데이터베이스 서비스를 선택하세요.
> **풀이/설명**:
> - "지난 3년간의 판매 데이터를 분석하여 분기별 매출 추이 보고서를 만들고 싶다" → **Redshift** (대규모 분석 쿼리, OLAP)
> - "소셜 미디어에서 사용자 간의 친구 관계를 기반으로 친구 추천을 하고 싶다" → **Neptune** (관계 탐색)
> - "금융 거래의 모든 변경 이력을 변조 불가능하게 기록해야 한다" → **QLDB** (변경 불가 원장)
> - "웹 애플리케이션의 세션 데이터를 밀리초 이내로 읽고 써야 한다" → **ElastiCache** (인메모리 캐시)
>
> **핵심 포인트**: AWS CCP 시험에서는 "이 상황에 가장 적합한 서비스는?"이라는 형태의 문제가 자주 출제됩니다. 각 서비스의 **핵심 키워드**를 기억하면 빠르게 답을 찾을 수 있습니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
> - **ElastiCache vs DAX 혼동**: 둘 다 캐싱 서비스이지만, DAX는 **DynamoDB 전용** 캐시이고, ElastiCache는 **범용** 인메모리 캐시입니다. RDS/Aurora 앞단 캐싱에는 ElastiCache를 사용합니다.
> - **Redshift vs RDS 혼동**: 둘 다 SQL을 사용하지만, Redshift는 **분석(OLAP)**에, RDS는 **트랜잭션(OLTP)**에 최적화되어 있습니다. "일일 주문 처리" → RDS, "연간 매출 분석 리포트" → Redshift.
> - **DocumentDB vs DynamoDB 혼동**: 둘 다 NoSQL이지만, DynamoDB는 키-값 기반 초고속 조회에, DocumentDB는 문서 내부의 복잡한 쿼리와 MongoDB 호환성에 초점을 맞춥니다.

### Section 요약

- **ElastiCache**(Redis/Memcached)는 인메모리 캐싱으로 DB 부하를 줄이고 응답 시간을 마이크로초까지 단축한다
- **Redshift**는 페타바이트 규모의 데이터 분석(OLAP)에 특화된 데이터 웨어하우스이다
- **Neptune**은 관계 중심 데이터를 다루는 그래프 데이터베이스이다
- **DocumentDB**는 MongoDB 호환 문서 데이터베이스이다
- **QLDB**는 변경 불가능한 이력 관리를 위한 원장 데이터베이스이다
- AWS는 "올바른 작업에 올바른 도구"를 선택하는 **목적별 데이터베이스(Purpose-Built Database)** 전략을 강조한다

## 8.6 데이터베이스 마이그레이션

> 💡 **한 줄 요약**: AWS DMS는 기존 데이터베이스를 AWS로 쉽게 이전할 수 있게 해주는 서비스이며, AWS SCT는 이기종 DB 간 스키마 변환을 지원한다.

### 직관적 이해

이사를 상상해 봅시다. 오래된 집(온프레미스 데이터베이스)에서 새 아파트(AWS 데이터베이스)로 이사하려 합니다. 이때 두 가지 서비스가 필요합니다:

1. **이사 업체 (AWS DMS)**: 짐(데이터)을 옛날 집에서 새 집으로 안전하게 운반해 줍니다. 놀라운 점은, 옛날 집에서 살면서 동시에 이사할 수 있다는 것입니다 — 즉, 기존 서비스를 중단하지 않고 데이터를 이전합니다.
2. **인테리어 컨설턴트 (AWS SCT)**: 옛날 집은 한옥이고 새 집은 아파트라면, 가구(스키마)를 새 집 구조에 맞게 변환해야 합니다. SCT가 이 변환 작업을 도와줍니다.

왜 마이그레이션이 중요할까요? 많은 기업이 기존에 온프레미스(자체 데이터 센터)에서 운영하던 데이터베이스를 AWS로 이전하려 합니다. 하지만 데이터베이스 이전은 매우 까다로운 작업입니다 — 서비스 중단을 최소화하면서 데이터의 무결성을 보장해야 하기 때문입니다.

### 핵심 개념 설명

#### AWS DMS (Database Migration Service)

> 📝 **AWS DMS(Database Migration Service)**: 데이터베이스를 AWS로 빠르고 안전하게 마이그레이션하는 서비스이다. 소스 데이터베이스를 마이그레이션하는 동안 **소스 데이터베이스가 정상 운영 상태를 유지**하므로 서비스 다운타임이 최소화된다.

DMS의 작동 방식:

```
  ┌──────────────────┐          ┌──────────┐          ┌──────────────────┐
  │ 소스 데이터베이스  │ ──────→ │ DMS 복제  │ ──────→ │ 타깃 데이터베이스  │
  │ (온프레미스 또는   │   데이터  │  인스턴스  │   데이터  │ (AWS RDS, Aurora, │
  │  다른 클라우드)    │   추출   │           │   적재   │  DynamoDB 등)     │
  └──────────────────┘          └──────────┘          └──────────────────┘
        ↑ 정상 운영 유지              ↑ AWS에서 실행
```

DMS의 핵심 기능:

**1. 동종 마이그레이션 (Homogeneous Migration)**
- 같은 엔진 간 이전: MySQL → Amazon RDS MySQL, Oracle → Amazon RDS Oracle
- 스키마 변환 불필요 — DMS만으로 완료

**2. 이종 마이그레이션 (Heterogeneous Migration)**
- 다른 엔진으로 이전: Oracle → Amazon Aurora PostgreSQL, SQL Server → MySQL
- **SCT(Schema Conversion Tool)**로 먼저 스키마를 변환한 뒤, DMS로 데이터를 이전

**3. 지속적 복제 (Ongoing Replication / CDC)**

> 📝 **CDC(Change Data Capture)**: 소스 데이터베이스에서 발생하는 변경 사항(삽입, 수정, 삭제)을 실시간으로 감지하여 타깃에 반영하는 기술이다.

DMS는 초기 데이터를 한 번에 복사(전체 로드)한 뒤, 이후 변경 사항을 CDC로 지속적으로 동기화할 수 있습니다. 이를 통해:
- 마이그레이션 중에도 소스 DB 정상 운영
- 전환 시점에 소스와 타깃의 데이터 일치 보장
- 다운타임을 분 단위로 최소화

#### AWS SCT (Schema Conversion Tool)

> 📝 **AWS SCT(Schema Conversion Tool)**: 소스 데이터베이스의 스키마(테이블 구조, 뷰, 저장 프로시저 등)를 타깃 데이터베이스 엔진에 맞게 자동으로 변환하는 도구이다.

SCT는 이종 마이그레이션에서 필수적입니다:

```
  이종 마이그레이션 절차:

  [Oracle DB] ──(1) SCT: 스키마 변환──→ [Aurora PostgreSQL 스키마]
      │                                        │
      └──────(2) DMS: 데이터 이전──────────────→│
                                                ▼
                                    [Aurora PostgreSQL — 완성]
```

SCT의 주요 기능:
- 스키마 자동 변환 (테이블, 인덱스, 뷰 등)
- 변환 불가능한 부분에 대한 **평가 보고서** 생성 (수동 작업이 필요한 부분 식별)
- 저장 프로시저, 함수 등의 코드 변환 지원

#### 마이그레이션 시나리오 정리

| 시나리오 | 필요한 도구 | 예시 |
|---------|------------|------|
| **동종 마이그레이션** | DMS만 | MySQL → RDS MySQL |
| **이종 마이그레이션** | SCT + DMS | Oracle → Aurora PostgreSQL |
| **온프레미스 → AWS** | DMS (+ SCT) | 자체 서버 DB → RDS/Aurora |
| **AWS 내 이전** | DMS | RDS MySQL → Aurora MySQL |
| **지속적 동기화** | DMS (CDC) | 온프레미스 DB ↔ AWS 실시간 복제 |

### 상세 예시

> 🔍 **예시 8.6.1: 은행의 데이터베이스 클라우드 이전**
>
> **상황**: 은행이 온프레미스 Oracle 데이터베이스를 AWS Aurora PostgreSQL로 이전하려 합니다. 서비스 중단은 최대 30분만 허용됩니다.
> **풀이/설명**:
> 1. **SCT**로 Oracle 스키마를 PostgreSQL 형식으로 변환. 평가 보고서에서 수동 변환이 필요한 저장 프로시저 20개를 식별
> 2. 수동 변환 작업 완료 후, 변환된 스키마를 Aurora PostgreSQL에 적용
> 3. **DMS**로 전체 데이터 로드(Initial Load) 실행 — Oracle을 정상 운영하면서 데이터 복사
> 4. 전체 로드 완료 후, **CDC(지속적 복제)** 활성화 — 이후 변경사항 실시간 동기화
> 5. 소스와 타깃의 데이터 일치를 검증
> 6. 계획된 시점에 애플리케이션의 접속 대상을 Oracle → Aurora로 전환 (실제 다운타임: 수 분)
>
> **핵심 포인트**: 이종 마이그레이션은 SCT(스키마 변환) + DMS(데이터 이전)의 조합으로 수행합니다. CDC를 활용하면 전환 시점의 다운타임을 최소화할 수 있습니다.

> 🔍 **예시 8.6.2: 동종 마이그레이션의 단순함**
>
> **상황**: 온프레미스 MySQL 서버를 Amazon RDS MySQL로 이전하려 합니다.
> **풀이/설명**:
> - 같은 엔진(MySQL → MySQL)이므로 스키마 변환 불필요 → SCT 불필요
> - **DMS만으로** 데이터를 이전하면 됨
> - DMS가 소스 MySQL의 데이터를 읽어 타깃 RDS MySQL에 적재
> - CDC로 지속적 동기화하다가 전환 시점에 애플리케이션 연결 변경
>
> **핵심 포인트**: 동종 마이그레이션(같은 엔진)은 DMS만으로 간단하게 수행할 수 있습니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**:
> - **DMS vs SCT 역할 혼동**: DMS는 **데이터를 이동**하고, SCT는 **스키마를 변환**합니다. 둘은 별개의 도구이며, 이종 마이그레이션에서만 SCT가 필요합니다.
> - **시험 팁**: "데이터베이스를 AWS로 마이그레이션" → DMS. "서로 다른 엔진 간 마이그레이션" → SCT + DMS. "마이그레이션 중 다운타임 최소화" → DMS의 CDC 기능.
> - **DMS는 마이그레이션 후에도 활용 가능**: 일회성 이전뿐 아니라, 온프레미스와 AWS 간의 **지속적인 데이터 동기화**에도 사용할 수 있습니다 (하이브리드 환경 등).

### Section 요약

- **AWS DMS**는 소스 DB를 정상 운영하면서 AWS로 데이터를 이전하는 마이그레이션 서비스이다
- **동종 마이그레이션**(같은 엔진)은 DMS만으로, **이종 마이그레이션**(다른 엔진)은 SCT + DMS로 수행한다
- **CDC(Change Data Capture)**로 지속적 복제가 가능하여 다운타임을 최소화한다
- **AWS SCT**는 소스 DB의 스키마를 타깃 DB 엔진에 맞게 자동 변환하며, 변환 불가능한 부분의 평가 보고서를 제공한다

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 AWS의 데이터베이스 서비스를 체계적으로 학습했습니다.

- **8.1 데이터베이스 기초**: 관계형 DB(SQL, ACID, 고정 스키마)와 비관계형 DB(NoSQL, 유연한 스키마, 수평 확장)의 차이, 그리고 관리형 서비스의 이점을 이해했습니다.
- **8.2 Amazon RDS**: 6가지 엔진을 지원하는 관리형 관계형 DB 서비스로, Multi-AZ(고가용성)와 Read Replica(읽기 성능 확장)를 제공합니다.
- **8.3 Amazon Aurora**: AWS가 클라우드에 최적화하여 설계한 관계형 DB로, MySQL/PostgreSQL과 호환되며 최대 5배 성능과 뛰어난 가용성을 제공합니다. Aurora Serverless로 자동 확장도 가능합니다.
- **8.4 Amazon DynamoDB**: 완전 관리형 NoSQL 데이터베이스로, 한 자릿수 밀리초 응답과 무한한 확장성을 제공합니다. 키-값 및 문서 모델을 지원합니다.
- **8.5 캐싱과 특수 목적 DB**: ElastiCache(인메모리 캐시), Redshift(데이터 웨어하우스), Neptune(그래프), DocumentDB(문서), QLDB(원장) 등 목적별로 최적화된 서비스를 제공합니다.
- **8.6 데이터베이스 마이그레이션**: DMS로 데이터를 이전하고, SCT로 이기종 간 스키마를 변환하여, 기존 DB를 AWS로 안전하게 이전할 수 있습니다.

**Q1.** O/X: Amazon RDS에서는 운영체제(OS)에 SSH로 직접 접속하여 관리할 수 있다.

<details>
<summary>정답 확인</summary>

**X** — RDS는 관리형 서비스이므로 기본 OS에 직접 접근할 수 없습니다. OS 레벨의 커스터마이징이 필요하면 EC2에 DB를 직접 설치해야 합니다.
</details>

**Q2.** Multi-AZ 배포의 주요 목적은 무엇인가? (a) 읽기 성능 향상 (b) 고가용성 (c) 비용 절감 (d) 스토리지 확장

<details>
<summary>정답 확인</summary>

**(b) 고가용성** — Multi-AZ는 동기식 복제를 통한 장애 자동 전환(Failover)으로 고가용성을 보장합니다. 읽기 성능 향상은 Read Replica의 목적입니다.
</details>

**Q3.** O/X: Amazon Aurora는 MySQL 및 PostgreSQL과 호환되며, 완전히 새로운 쿼리 언어를 사용한다.

<details>
<summary>정답 확인</summary>

**X** — Aurora는 MySQL 및 PostgreSQL과 **호환**되므로, 기존 MySQL/PostgreSQL에서 사용하던 SQL 코드와 도구를 거의 그대로 사용할 수 있습니다.
</details>

**Q4.** DynamoDB의 특성으로 올바르지 **않은** 것은? (a) 완전 관리형 (b) 한 자릿수 밀리초 응답 (c) 복잡한 JOIN 쿼리에 최적화 (d) 키-값 데이터 모델

<details>
<summary>정답 확인</summary>

**(c)** — DynamoDB는 기본 키 기반의 빠른 조회에 최적화되어 있으며, 복잡한 JOIN이나 집계 쿼리에는 적합하지 않습니다. 복잡한 JOIN이 필요하면 RDS/Aurora를 사용합니다.
</details>

**Q5.** 온프레미스 Oracle 데이터베이스를 Amazon Aurora PostgreSQL로 이전할 때 필요한 도구의 조합은?

<details>
<summary>정답 확인</summary>

**AWS SCT + AWS DMS** — 서로 다른 엔진(Oracle → PostgreSQL) 간의 이종 마이그레이션이므로, SCT로 스키마를 변환하고 DMS로 데이터를 이전합니다.
</details>

**Q6.** O/X: Amazon ElastiCache와 DynamoDB Accelerator(DAX)는 동일한 용도로 사용할 수 있다.

<details>
<summary>정답 확인</summary>

**X** — DAX는 **DynamoDB 전용** 인메모리 캐시이고, ElastiCache는 **범용** 인메모리 캐싱 서비스입니다. RDS/Aurora 앞단 캐싱에는 ElastiCache를, DynamoDB 캐싱에는 DAX를 사용합니다.
</details>

**Q7.** "지난 5년간의 판매 데이터를 분석하여 분기별 트렌드 보고서를 생성"하는 데 가장 적합한 AWS 서비스는?

<details>
<summary>정답 확인</summary>

**Amazon Redshift** — 대규모 과거 데이터에 대한 복잡한 분석 쿼리(OLAP)에 최적화된 데이터 웨어하우스입니다. 일상적인 트랜잭션 처리(OLTP)용인 RDS/Aurora와 구분하세요.
</details>

> - **새 속성 추가** → **부울(Boolean)**: `isGift`, 값 `true` ← **또 다른 새 속성!**
>
> **Step 3**: 항목 목록을 확인합니다. 세 항목이 모두 다른 속성 조합을 가지고 있음을 확인합니다:
> - `user-001 / 2026-04-01`: productName, price
> - `user-001 / 2026-04-05`: productName, price, color
> - `user-003 / 2026-04-07`: productName, price, brand, isGift
>
> **Step 4**: 이 유연함이 관계형 DB와 어떻게 다른지 생각해 봅니다:
> - RDS(MySQL)였다면 → 모든 행에 같은 열이 있어야 함 → `brand`, `isGift`, `color` 열을 미리 정의하고, 해당 없는 행은 NULL로 채워야 함
> - DynamoDB → 각 항목이 필요한 속성만 가지면 됨 → 스키마 변경 없이 새 속성을 자유롭게 추가
>
> **✅ 체크포인트**: DynamoDB에서 항목마다 다른 속성을 가질 수 있음을 직접 확인했습니다.

> **Phase 4: RDS 주요 기능 탐색** (활용 개념: Section 8.2)
>
> **Step 1**: RDS 콘솔에서 `my-first-rds`를 클릭합니다.
>
> **Step 2**: **유지 관리 및 백업(Maintenance & backups)** 탭을 확인합니다:
> - 자동 백업이 활성화되어 있는지 확인
> - 백업 보관 기간을 확인 (기본 7일)
> - "자동 백업이 매일 생성됩니다"는 것은 관리형 서비스의 이점
>
> **Step 3**: **구성(Configuration)** 탭을 확인합니다:
> - Multi-AZ: 아니요 → Free Tier이므로 단일 AZ
> - 프로덕션 환경에서는 Multi-AZ를 활성화하여 고가용성을 확보
>
> **Step 4**: **모니터링(Monitoring)** 탭을 확인합니다:
> - CPU 사용률, DB 연결 수, 읽기/쓰기 지연 시간 등의 메트릭을 확인
> - 이 메트릭들이 자동으로 수집되는 것도 관리형 서비스의 이점
>
> **✅ 체크포인트**: RDS의 자동 백업, Multi-AZ 옵션, 모니터링 기능을 확인했습니다.

### 트러블슈팅 (Troubleshooting)

> **Q: RDS 인스턴스 생성 시 "프리 티어(Free Tier)" 템플릿이 보이지 않아요.**
> A: AWS 계정 생성 후 12개월이 지나면 RDS Free Tier가 만료됩니다. 이 경우 비용이 발생할 수 있으므로, 실습 후 반드시 인스턴스를 삭제하세요. 또는 "dev/test" 템플릿을 선택하고 db.t3.micro를 수동으로 지정하세요.

> **Q: RDS 인스턴스가 "생성 중" 상태에서 오래 걸려요.**
> A: RDS 인스턴스 생성은 일반적으로 5~15분이 소요됩니다. 20분 이상 걸리면 페이지를 새로고침하여 상태를 확인하세요.

> **Q: DynamoDB에서 "항목 생성" 시 속성 추가 버튼이 안 보여요.**
> A: 항목 생성 화면에서 **JSON 보기**가 아닌 **폼 보기**인지 확인하세요. 폼 보기에서 **새 속성 추가(Add new attribute)** 버튼이 표시됩니다.

> **Q: RDS 인스턴스를 삭제했는데 목록에서 사라지지 않아요.**
> A: "삭제 중(Deleting)" 상태가 수 분간 표시될 수 있습니다. 잠시 기다린 후 새로고침하면 목록에서 사라집니다.

### D2. Exercise (응용 문제) — 개념 적용 및 통합

> **Exercise 8.1**: 온라인 쇼핑몰의 백엔드 아키텍처를 설계하고 있습니다. 다음 데이터 유형 각각에 대해 가장 적합한 AWS 데이터베이스 서비스를 선택하고, 그 이유를 설명하세요.
>
> (a) 주문 및 결제 정보 (주문번호, 상품, 금액, 결제 상태)
> (b) 상품 카탈로그 (상품마다 속성이 다름 — 의류: 사이즈/색상, 전자제품: 사양/무게)
> (c) 사용자 세션 데이터 (로그인 상태, 장바구니 — 빈번한 읽기/쓰기, 밀리초 응답 필요)
> (d) 지난 3년간 판매 데이터의 분기별 매출 분석 보고서
>
> **힌트**: 각 데이터의 특성(구조, 일관성 요구, 접근 패턴, 규모)을 먼저 분석하세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 데이터 특성에 따라 목적별 데이터베이스(purpose-built database)를 선택합니다.
>
> **풀이 과정**:
>
> **(a) 주문 및 결제 정보 → Amazon RDS 또는 Aurora**
> - 정형 데이터(고정된 스키마) + ACID 트랜잭션 필수(결제는 원자적이어야 함) → 관계형 DB
> - 성능·가용성 요구가 높으면 Aurora, 비용 우선이면 RDS MySQL/PostgreSQL
>
> **(b) 상품 카탈로그 → Amazon DynamoDB**
> - 상품마다 속성이 다름 → 유연한 스키마 필요 → NoSQL
> - 빠른 키 기반 조회(상품 ID로 검색) → DynamoDB의 밀리초 응답
> - 대안: 문서 내부의 복잡한 검색이 많으면 Amazon DocumentDB도 고려
>
> **(c) 사용자 세션 데이터 → Amazon ElastiCache (Redis)**
> - 빈번한 읽기/쓰기 + 밀리초(또는 마이크로초) 응답 필요 → 인메모리 캐시
> - 세션은 일시적 데이터이므로 TTL 설정으로 자동 만료 가능
> - Redis는 다양한 데이터 구조를 지원하여 세션 관리에 적합
>
> **(d) 3년간 판매 데이터 분석 → Amazon Redshift**
> - 대량의 과거 데이터에 대한 복잡한 집계 쿼리(OLAP) → 데이터 웨어하우스
> - 열 기반 스토리지와 대규모 병렬 처리로 분석 쿼리 가속
>
> **보충 설명**: 이처럼 하나의 애플리케이션에서 여러 데이터베이스를 조합하는 것이 AWS의 "목적별 데이터베이스" 전략입니다. 모든 데이터를 하나의 DB에 넣는 것보다 각 데이터 특성에 맞는 서비스를 선택하는 것이 성능, 비용, 관리 측면에서 유리합니다.
> </details>

> **Exercise 8.2**: 중견 기업에서 운영 중인 데이터베이스의 요구사항이 다음과 같습니다. Multi-AZ와 Read Replica를 어떻게 구성할지 설명하세요.
>
> - 요구사항 1: 데이터베이스 장애 시 2분 이내에 자동 복구되어야 한다
> - 요구사항 2: 영업팀이 매시간 대량의 보고서 쿼리를 실행하는데, 이 때문에 주문 처리 성능이 저하되어서는 안 된다
> - 요구사항 3: 일본 지사에서 데이터를 빠르게 읽을 수 있어야 한다 (현재 리전: 서울)
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 요구사항을 Multi-AZ와 Read Replica의 특성에 매칭합니다.
>
> **풀이 과정**:
>
> **요구사항 1 → Multi-AZ 배포 활성화**
> - Multi-AZ는 동기식 복제로 Standby 인스턴스를 유지하며, 장애 시 60~120초 내에 자동 전환(Failover) → 2분 이내 복구 요건 충족
>
> **요구사항 2 → 서울 리전에 Read Replica 생성**
> - 영업팀의 보고서 쿼리를 Read Replica로 분산
> - Primary 인스턴스는 주문 처리(쓰기)에 집중하여 성능 저하 방지
>
> **요구사항 3 → 도쿄 리전(ap-northeast-1)에 크로스 리전 Read Replica 생성**
> - 일본 지사는 도쿄 리전의 Read Replica에서 데이터를 읽어 지연 시간 감소
> - 비동기식 복제이므로 약간의 데이터 지연(lag)이 있을 수 있으나, 읽기 전용 용도로는 충분
>
> **최종 구성**:
> ```
> 서울 리전 (ap-northeast-2):
>   Primary DB (AZ-a) ←── Multi-AZ ──→ Standby (AZ-b)   [요구사항 1]
>       │
>       ├── Read Replica (서울) → 영업팀 보고서용          [요구사항 2]
>       │
>       └── Cross-Region Read Replica (도쿄) → 일본 지사용 [요구사항 3]
> ```
>
> **보충 설명**: Multi-AZ와 Read Replica는 서로 다른 목적(가용성 vs 성능)을 위한 것이므로 동시에 활용할 수 있습니다. 비용은 각 인스턴스별로 발생하므로 예산과 함께 고려해야 합니다.
> </details>

> **Exercise 8.3**: 회사에서 온프레미스 Oracle 데이터베이스를 AWS로 이전하기로 결정했습니다. 다음 두 가지 옵션을 비교하고, 각각의 장단점과 적합한 상황을 설명하세요.
>
> - 옵션 A: Amazon RDS for Oracle로 이전 (동종 마이그레이션)
> - 옵션 B: Amazon Aurora PostgreSQL로 이전 (이종 마이그레이션)
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 기술적 복잡도, 비용, 장기적 효과를 기준으로 비교합니다.
>
> **풀이 과정**:
>
> | 비교 항목 | 옵션 A: RDS Oracle | 옵션 B: Aurora PostgreSQL |
> |-----------|-------------------|-------------------------|
> | **마이그레이션 복잡도** | 낮음 — DMS만 사용 (동종) | 높음 — SCT + DMS 사용 (이종) |
> | **코드 변경** | 최소 (같은 엔진) | 상당함 (Oracle 고유 기능 변환 필요) |
> | **라이선스 비용** | Oracle 라이선스 비용 포함 | 오픈소스 — 라이선스 비용 없음 |
> | **장기 운영 비용** | 높음 (상용 라이선스) | 낮음 (오픈소스 기반) |
> | **성능** | 표준 RDS 수준 | Aurora의 고성능 스토리지 활용 |
> | **벤더 종속성** | Oracle에 종속 유지 | 오픈소스 생태계 활용 가능 |
>
> **적합한 상황**:
> - **옵션 A**: 빠른 이전이 우선이고, Oracle 고유 기능(PL/SQL 등)에 대한 의존도가 높은 경우. 일단 AWS로 빠르게 이전("Lift and Shift")하고, 나중에 엔진 전환을 검토하는 단계적 전략에 적합.
> - **옵션 B**: 장기적인 비용 절감과 벤더 독립성이 중요하고, 충분한 마이그레이션 시간과 리소스가 확보된 경우. 초기 투자(스키마 변환, 코드 수정)가 크지만 장기적으로 운영 비용이 낮아짐.
>
> **보충 설명**: 실무에서는 "옵션 A로 먼저 AWS로 이전한 뒤, 점진적으로 옵션 B로 전환"하는 2단계 접근도 흔히 사용됩니다.
> </details>

> **Exercise 8.4**: 다음 각 시나리오에서 가장 적합한 AWS 데이터베이스 서비스를 하나씩 선택하고, 선택 이유를 간략히 설명하세요.
>
> (a) 은행 거래 기록을 변조 불가능하게 저장하고, 규제 감사 시 모든 변경 이력을 제출해야 한다
> (b) 전 세계 사용자가 접속하는 모바일 게임에서, 각 리전에서 빠르게 사용자 데이터를 읽고 쓰고 싶다
> (c) 기존 MongoDB를 사용하던 콘텐츠 관리 시스템을 AWS로 이전하고 싶다
>
> <details>
> <summary>모범 답안</summary>
>
> **(a) → Amazon QLDB**
> - QLDB는 모든 데이터 변경을 변경 불가능(immutable)하고 암호화 검증 가능한 형태로 기록하는 원장(Ledger) 데이터베이스입니다.
> - 금융 규제의 감사 추적(audit trail) 요구사항에 정확히 부합합니다.
>
> **(b) → Amazon DynamoDB (글로벌 테이블)**
> - DynamoDB 글로벌 테이블은 여러 리전에 데이터를 자동 복제하며, 각 리전에서 읽기·쓰기가 모두 가능합니다(active-active).
> - 전 세계 사용자가 로컬 리전에서 밀리초 응답으로 데이터를 읽고 쓸 수 있습니다.
>
> **(c) → Amazon DocumentDB**
> - DocumentDB는 MongoDB와 호환되는 문서 데이터베이스입니다.
> - 기존 MongoDB 코드와 드라이버를 대부분 그대로 사용할 수 있어 이전이 용이합니다.
> </details>

### D3. Problem (심화 문제) — 깊이 있는 사고와 창의적 문제 해결

> **Problem 8.1**: 빠르게 성장하는 전자상거래 스타트업의 CTO로서, 현재 단일 RDS MySQL 인스턴스(db.r5.large)로 운영되는 데이터베이스가 다음 문제를 동시에 겪고 있습니다:
>
> - 문제 1: 피크 시간에 쓰기 지연이 증가하여 주문 처리가 느려짐
> - 문제 2: 마케팅팀의 분석 쿼리가 실행될 때 전체 DB 성능이 저하됨
> - 문제 3: 지난주 AZ 장애로 2시간 동안 서비스가 중단됨
> - 문제 4: 상품 페이지 로딩 시 같은 상품 정보를 반복적으로 DB에서 조회하여 불필요한 부하 발생
>
> 각 문제에 대한 해결 방안을 제시하고, 이들을 하나의 통합된 아키텍처로 설계하세요. 비용과 복잡성 사이의 트레이드오프도 고려하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 하나의 해결책으로 모든 문제를 풀기보다, 각 문제에 맞는 AWS 서비스를 조합하여 전체 아키텍처를 설계합니다.
>
> **풀이 전략**: 각 문제를 독립적으로 분석한 뒤, 통합합니다.
>
> **상세 풀이**:
>
> **문제 1 (쓰기 지연)**: 인스턴스 스케일업 — db.r5.large → db.r5.xlarge 또는 db.r5.2xlarge로 업그레이드. 또는 Aurora MySQL로 마이그레이션하여 더 높은 쓰기 처리량 확보.
>
> **문제 2 (분석 쿼리로 인한 성능 저하)**: Read Replica를 생성하여 마케팅팀의 분석 쿼리를 Replica로 분산. 장기적으로는 분석 데이터를 Amazon Redshift로 분리하는 것도 고려.
>
> **문제 3 (AZ 장애 시 서비스 중단)**: Multi-AZ 배포를 활성화하여 자동 장애 전환(Failover)을 보장. 향후 Aurora로 전환하면 장애 복구 시간이 더 단축됨 (30초 이내).
>
> **문제 4 (반복 조회로 인한 부하)**: Amazon ElastiCache(Redis)를 도입하여 자주 조회되는 상품 정보를 캐싱. DB 부하를 크게 줄이고 응답 시간도 단축.
>
> **통합 아키텍처**:
> ```
>  [사용자] → [애플리케이션]
>                 │
>                 ├─ 상품 조회 ──→ [ElastiCache Redis] ──(miss)──→ [Primary DB]
>                 │                   (캐싱, 문제4)
>                 ├─ 주문 처리 ──→ [Primary DB — Aurora MySQL (AZ-a)]
>                 │                    │           (문제1: 성능 향상)
>                 │                    ├─ Multi-AZ ──→ [Standby (AZ-b)]
>                 │                    │                 (문제3: 고가용성)
>                 │                    └─ Replica ──→ [Read Replica]
>                 │                                    (문제2: 분석 분산)
>                 └─ 마케팅 분석 ──→ [Read Replica] (또는 Redshift)
> ```
>
> **트레이드오프**:
> - **단계적 접근**: 모든 것을 한 번에 바꾸면 위험. 우선순위를 정하여 1) Multi-AZ 활성화 (즉각적, 가용성), 2) ElastiCache 도입 (상대적으로 간단), 3) Read Replica 추가, 4) Aurora 마이그레이션 (가장 큰 변경) 순서로 진행
> - **비용 증가**: Multi-AZ(약 2배), Read Replica(추가 인스턴스), ElastiCache(추가 서비스), Aurora(20% 증가) — 전체적으로 상당한 비용 증가
> - **복잡성**: 서비스가 늘어날수록 운영 복잡성 증가 → 모니터링, 장애 대응 포인트 증가
>
> **확장 생각**: 사용자 기반이 글로벌로 확장되면, Aurora 글로벌 데이터베이스나 DynamoDB 글로벌 테이블을 고려할 수 있습니다. 또한, 자주 변경되지 않는 상품 카탈로그를 DynamoDB로 분리하는 것도 장기적인 아키텍처 개선 방향입니다.
> </details>

> **Problem 8.2**: 당신은 대형 병원 그룹의 클라우드 아키텍트입니다. 다음 데이터를 모두 관리해야 하는 시스템을 설계해야 합니다:
>
> - (A) 환자 기본 정보 및 진료 기록 — 정확성과 일관성이 가장 중요, 법적 보관 의무 10년
> - (B) 의료 영상(X-ray, MRI 등) — 대용량 파일, 저장 후 변경 없음
> - (C) 환자 간 가족 관계, 의사-환자 관계, 전문의 의뢰 네트워크 — 복잡한 관계 탐색 필요
> - (D) 약물 처방 이력 — 모든 변경을 변조 불가능하게 기록해야 함 (규제 요구사항)
> - (E) 응급실 대기 현황판 — 수백 번/초 갱신, 실시간 조회 필요
>
> 각 데이터에 적합한 AWS 서비스를 선택하고, 전체 아키텍처를 설계하세요. Chapter 6(스토리지)에서 배운 서비스도 활용할 수 있습니다.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 의료 시스템은 데이터 특성이 극도로 다양하며, 각 데이터에 최적화된 서비스를 선택하되 보안과 규정 준수가 최우선입니다.
>
> **풀이 전략**: 각 데이터의 핵심 요구사항(일관성, 규모, 접근 패턴, 규제)을 분석하여 서비스를 매칭합니다.
>
> **상세 풀이**:
>
> **(A) 환자 기본 정보 및 진료 기록 → Amazon Aurora (Multi-AZ)**
> - ACID 트랜잭션 필수 (환자 정보의 정확성) → 관계형 DB
> - 10년 보관 → 자동 백업 + 수동 스냅샷을 S3에 장기 보관
> - Aurora의 높은 가용성(3 AZ, 6 복사본)이 의료 시스템의 무중단 요구에 적합
>
> **(B) 의료 영상 → Amazon S3**
> - 대용량 파일, 저장 후 변경 없음 → 객체 스토리지 (Chapter 6에서 학습)
> - S3 Glacier로 장기 보관 시 비용 절감
> - 데이터베이스가 아닌 스토리지 서비스가 적합한 경우!
>
> **(C) 복잡한 관계 탐색 → Amazon Neptune**
> - 환자-의사-가족 간의 복잡한 관계 탐색 → 그래프 데이터베이스
> - "환자 A의 담당 의사와 같은 전문 분야의 다른 의사를 찾아줘" 같은 쿼리에 최적
>
> **(D) 약물 처방 이력 → Amazon QLDB**
> - 변조 불가능한 기록 + 규제 요구사항 → 원장 데이터베이스
> - 모든 처방 변경, 수정, 취소 이력이 암호화 검증 가능한 형태로 영구 보존
>
> **(E) 응급실 대기 현황판 → Amazon ElastiCache (Redis)**
> - 초당 수백 번 갱신 + 실시간 조회 → 인메모리 캐시
> - Redis의 Pub/Sub 기능으로 현황판 실시간 업데이트 가능
> - 영속적 저장이 필요한 데이터는 DynamoDB에 백업
>
> **전체 아키텍처**:
> ```
>  [의료 정보 시스템]
>       │
>       ├── 환자 정보/진료 기록 ──→ [Aurora — Multi-AZ]
>       │
>       ├── 의료 영상 ────────────→ [S3 → S3 Glacier (장기보관)]
>       │
>       ├── 관계 네트워크 ────────→ [Neptune]
>       │
>       ├── 처방 이력 ────────────→ [QLDB]
>       │
>       └── 응급실 현황 ──────────→ [ElastiCache Redis] + [DynamoDB (백업)]
> ```
>
> **확장 생각**: 이 아키텍처에서 Chapter 12(보안)에서 다루는 암호화, 접근 제어가 매우 중요해집니다. 의료 데이터(PHI — Protected Health Information)는 전송 중 암호화와 저장 시 암호화가 모두 필수이며, IAM(Chapter 13)을 통한 세밀한 접근 제어가 필요합니다.
> </details>

> **Problem 8.3**: AWS는 "올바른 작업에 올바른 도구(purpose-built database)"를 강조합니다. 그러나 어떤 기업은 "모든 데이터를 DynamoDB 하나로 통일하자"거나 "모든 것을 RDS에 넣자"는 접근을 취하기도 합니다. 이 두 극단적 접근의 장단점을 분석하고, 현실적으로 언제 "단일 DB 전략"이 합리적인지, 언제 "다중 DB 전략"이 필수적인지 논하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 기술적 최적성과 운영 복잡성 사이의 트레이드오프를 이해하는 것이 핵심입니다.
>
> **풀이 전략**: 각 접근의 장단점을 운영, 비용, 성능, 팀 역량 측면에서 분석합니다.
>
> **상세 풀이**:
>
> **단일 DB 전략의 장점**:
> - 운영 복잡성 최소화 — 하나의 서비스만 모니터링, 관리, 장애 대응
> - 팀 학습 부담 감소 — 하나의 기술 스택에 집중
> - 데이터 간 JOIN/관계가 간단 — 모든 데이터가 한 곳에 있으므로
> - 일관된 백업/복원 전략
>
> **단일 DB 전략의 단점**:
> - 성능 비효율 — 분석 쿼리와 트랜잭션이 같은 자원을 경쟁
> - 확장 한계 — 한 DB의 확장 한계가 전체 시스템의 병목
> - 비용 비효율 — 모든 데이터에 동일한(비싼) 서비스를 적용
> - 적합하지 않은 데이터 모델 강제 — 그래프 데이터를 테이블에 억지로 넣는 등
>
> **다중 DB 전략의 장점**:
> - 각 데이터에 최적화된 성능과 비용
> - 독립적 확장 가능 — 병목이 되는 서비스만 확장
> - 장애 격리 — 한 서비스의 장애가 다른 서비스에 영향을 주지 않음
>
> **다중 DB 전략의 단점**:
> - 운영 복잡성 증가 — 여러 서비스 관리, 모니터링, 장애 대응
> - 데이터 일관성 관리 어려움 — 서비스 간 데이터 동기화 필요
> - 팀의 다양한 기술 역량 요구
>
> **현실적 가이드라인**:
>
> | 상황 | 권장 접근 |
> |------|----------|
> | 소규모 스타트업, MVP 단계, 5인 이하 팀 | 단일 DB (RDS 또는 DynamoDB) |
> | 데이터 특성이 명확히 다른 경우 (예: 트랜잭션 + 캐싱) | 다중 DB |
> | 규모가 커져 성능 병목이 발생할 때 | 점진적으로 다중 DB로 전환 |
> | 규제 요구사항이 특정 DB를 요구할 때 (예: QLDB) | 해당 데이터에 특수 DB 도입 |
>
> **확장 생각**: 실무에서는 "시작은 단순하게, 필요할 때 분리"하는 접근이 가장 현실적입니다. 처음부터 7가지 DB를 도입하는 것은 과도한 엔지니어링(over-engineering)이 될 수 있습니다. AWS의 "purpose-built database" 전략은 규모와 요구사항이 충분히 커진 이후에 그 가치를 발휘합니다.
> </details>
