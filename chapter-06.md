## 6.1 스토리지 유형 비교 — 객체, 블록, 파일 스토리지의 개념과 차이

> 💡 **한 줄 요약**: 데이터를 저장하는 방식에는 객체, 블록, 파일 스토리지 세 가지가 있으며, 각각 용도와 특성이 다릅니다.

### 직관적 이해 (Intuitive Understanding)

여러분의 방에 있는 물건을 정리하는 세 가지 방법을 떠올려 봅시다.

- **파일 캐비닛(파일 스토리지)**: 폴더 안에 폴더를 만들고, 그 안에 문서를 넣습니다. "서랍 → 고객 → 2024년 → 계약서.pdf"처럼 계층적인 경로로 찾습니다. 우리가 컴퓨터에서 폴더를 만들어 파일을 정리하는 것과 같습니다.
- **창고의 선반(객체 스토리지)**: 물건마다 고유한 태그(이름표)를 붙여서 거대한 창고에 넣습니다. 물건을 찾을 때는 태그를 검색합니다. 폴더 구조 없이 평면적으로 관리하지만, 태그에 다양한 정보(메타데이터)를 붙일 수 있습니다.
- **레고 블록(블록 스토리지)**: 데이터를 일정한 크기의 블록으로 쪼개어 저장합니다. 마치 레고 블록을 조립하듯, 필요한 블록들을 모아 하나의 파일을 구성합니다. 컴퓨터의 하드디스크가 바로 이 방식으로 동작합니다.

왜 세 가지나 필요할까요? 모든 상황에 완벽한 하나의 저장 방식은 존재하지 않기 때문입니다. 사진 수백만 장을 저장하는 데는 객체 스토리지가 효율적이고, 데이터베이스처럼 빠른 읽기/쓰기가 필요한 곳에는 블록 스토리지가 적합하며, 여러 사람이 동시에 같은 문서를 편집해야 한다면 파일 스토리지가 필요합니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **객체 스토리지(Object Storage)**: 데이터를 "객체(object)"라는 단위로 저장하는 방식입니다. 각 객체는 데이터 본체, 메타데이터(데이터에 대한 설명 정보), 고유 식별자(키)로 구성됩니다. 폴더 계층 구조 없이 평면적(flat)으로 저장합니다.

> 📝 **블록 스토리지(Block Storage)**: 데이터를 고정된 크기의 "블록"으로 나누어 저장하는 방식입니다. 운영체제가 이 블록들을 직접 관리하며, 컴퓨터에 직접 연결된 하드디스크나 SSD처럼 동작합니다.

> 📝 **파일 스토리지(File Storage)**: 데이터를 계층적인 디렉터리(폴더) 구조로 저장하는 방식입니다. 여러 사용자나 시스템이 네트워크를 통해 동일한 파일 시스템에 동시에 접근할 수 있습니다.

세 가지 스토리지 유형의 핵심 차이를 표로 정리하면 다음과 같습니다:

| 특성 | 객체 스토리지 | 블록 스토리지 | 파일 스토리지 |
|------|-------------|-------------|-------------|
| **데이터 단위** | 객체 (데이터 + 메타데이터 + 키) | 고정 크기 블록 | 파일 (디렉터리 계층) |
| **접근 방식** | HTTP/HTTPS (API) | 운영체제가 직접 관리 | 네트워크 파일 프로토콜 (NFS, SMB) |
| **수정 방식** | 전체 객체를 교체 | 개별 블록만 수정 가능 | 파일 내 부분 수정 가능 |
| **확장성** | 사실상 무제한 | 볼륨 크기 내로 제한 | 파일 시스템 크기 내로 제한 |
| **성능** | 높은 처리량, 상대적으로 높은 지연 | 낮은 지연, 높은 IOPS | 중간 수준 |
| **대표 사례** | 사진, 동영상, 백업, 로그 | 데이터베이스, 운영체제 | 공유 문서, 홈 디렉터리 |
| **AWS 서비스** | Amazon S3 | Amazon EBS | Amazon EFS, FSx |

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.1.1: 스마트폰 사진 백업 — 객체 스토리지**
>
> **상황**: 스마트폰에 저장된 사진 10만 장을 클라우드에 백업하려고 합니다.
> **풀이/설명**: 사진 한 장 한 장이 독립적인 파일이며, 한번 저장되면 수정할 일이 거의 없습니다. 사진을 찾을 때는 날짜, 위치 같은 메타데이터로 검색합니다. 이런 경우 객체 스토리지가 가장 적합합니다 — 개수에 제한 없이 저장할 수 있고, 메타데이터를 풍부하게 활용할 수 있습니다.
> **핵심 포인트**: 객체 스토리지는 대량의 비정형 데이터(이미지, 동영상, 문서)를 저장하고 검색하는 데 최적화되어 있습니다.

> 🔍 **예시 6.1.2: 회사 공유 드라이브 — 파일 스토리지**
>
> **상황**: 디자인팀 10명이 동시에 같은 프로젝트 폴더에서 파일을 열어 작업해야 합니다.
> **풀이/설명**: 여러 사용자가 "프로젝트 → 2024 → 시안" 같은 폴더 구조를 통해 파일에 접근하고, 동시에 읽고 쓸 수 있어야 합니다. 이것은 전통적인 네트워크 파일 공유(회사에서 흔히 쓰는 공유 드라이브)와 같은 방식이며, 파일 스토리지가 적합합니다.
> **핵심 포인트**: 파일 스토리지는 여러 사용자/시스템이 동시에 접근해야 하는 공유 데이터에 적합합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "객체 스토리지에도 폴더가 있는 것처럼 보이는데요?"
>
> Amazon S3 같은 객체 스토리지에서 `photos/2024/vacation.jpg`처럼 슬래시(`/`)로 구분된 키를 사용하면 콘솔에서 폴더처럼 보입니다. 하지만 이것은 **실제 폴더가 아닙니다**. 슬래시는 키 이름의 일부일 뿐이며, S3는 내부적으로 모든 객체를 평면적으로 저장합니다. 이 "가상 폴더"는 보기 편하게 해주는 콘솔의 표시 방식일 뿐입니다.

> ⚠️ **주의**: "블록 스토리지가 가장 빠르니까 항상 블록 스토리지를 쓰면 되지 않나요?"
>
> 블록 스토리지는 낮은 지연 시간과 높은 성능을 제공하지만, 확장성에 한계가 있고 비용이 상대적으로 높습니다. 사진 100만 장을 블록 스토리지에 저장하면 불필요하게 비싼 비용을 지불하게 됩니다. **용도에 맞는 스토리지를 선택하는 것**이 클라우드 비용 최적화의 핵심입니다.

### Section 요약 (Section Summary)

- 클라우드 스토리지는 **객체, 블록, 파일** 세 가지 유형으로 구분됩니다.
- **객체 스토리지**는 대량의 비정형 데이터를 저장하는 데 적합하며, 사실상 무제한 확장이 가능합니다.
- **블록 스토리지**는 데이터베이스나 운영체제처럼 빠른 읽기/쓰기가 필요한 곳에 적합합니다.
- **파일 스토리지**는 여러 사용자가 동시에 접근하는 공유 파일 시스템에 적합합니다.
- 최적의 스토리지 선택은 데이터의 특성, 접근 패턴, 비용을 종합적으로 고려해야 합니다.

## 6.2 Amazon S3 기초 — 버킷, 객체, 키, 무제한 확장성

> 💡 **한 줄 요약**: Amazon S3는 인터넷 어디서든 원하는 양의 데이터를 저장하고 꺼내올 수 있는 객체 스토리지 서비스입니다.

### 직관적 이해 (Intuitive Understanding)

Amazon S3를 이해하려면 **무한히 커지는 창고**를 상상해 보세요.

이 창고에는 다음과 같은 특징이 있습니다:
- 물건(데이터)을 얼마든지 넣을 수 있고, 창고는 알아서 커집니다.
- 물건마다 고유한 이름표(키)를 붙입니다.
- 물건을 넣을 때 "이 물건은 깨지기 쉬우니 주의"와 같은 설명(메타데이터)을 함께 적을 수 있습니다.
- 인터넷이 연결된 곳이면 어디서든 물건을 넣거나 꺼낼 수 있습니다.
- 물건을 잃어버릴 확률이 거의 0에 가깝습니다 (99.999999999%의 내구성).

Amazon S3는 AWS에서 **가장 먼저 출시된 서비스 중 하나**(2006년)이자, 가장 널리 사용되는 서비스입니다. 웹사이트의 이미지부터 기업의 백업 데이터, 빅데이터 분석용 데이터 레이크까지, S3는 AWS 생태계의 중심에 있습니다.

> 🔗 **연결**: Chapter 3에서 AWS가 2006년에 S3와 EC2를 최초의 서비스로 출시했다고 배웠습니다. S3가 그만큼 AWS의 핵심 서비스라는 의미입니다.

### 핵심 개념 설명 (Core Explanation)

#### 버킷(Bucket)

> 📝 **버킷(Bucket)**: S3에서 객체를 담는 최상위 컨테이너입니다. 모든 객체는 반드시 하나의 버킷 안에 존재해야 합니다. 버킷은 컴퓨터의 "드라이브(C:, D:)"와 비슷한 역할을 합니다.

버킷의 핵심 규칙:
- **이름이 전 세계에서 고유**해야 합니다. 다른 AWS 사용자가 이미 쓰고 있는 이름은 사용할 수 없습니다.
- 이름은 3~63자, 소문자, 숫자, 하이픈만 사용 가능합니다.
- 버킷은 **특정 리전에 생성**됩니다. 데이터는 명시적으로 복제하지 않는 한 해당 리전을 벗어나지 않습니다.
- 계정당 기본 100개의 버킷을 생성할 수 있습니다 (요청 시 증가 가능).

> 🔗 **연결**: Chapter 3에서 배운 것처럼, 리전 선택은 데이터 규정 준수(데이터 주권), 지연 시간, 비용에 영향을 미칩니다. S3 버킷의 리전도 이 기준에 따라 선택해야 합니다.

#### 객체(Object)

> 📝 **객체(Object)**: S3에 저장되는 데이터의 기본 단위입니다. 파일과 해당 파일에 대한 정보(메타데이터)를 합친 것이라고 생각할 수 있습니다.

객체는 세 가지 요소로 구성됩니다:

1. **키(Key)**: 버킷 내에서 객체를 고유하게 식별하는 이름입니다. 예: `reports/2024/sales-q1.pdf`
2. **값(Value)**: 실제 데이터(파일 내용)입니다. 하나의 객체는 최대 **5TB**까지 저장할 수 있습니다.
3. **메타데이터(Metadata)**: 객체에 대한 부가 정보입니다. 콘텐츠 유형(예: `image/jpeg`), 생성 일시, 사용자 정의 태그 등이 포함됩니다.

#### S3의 핵심 특성

- **무제한 확장성**: 버킷에 저장할 수 있는 객체의 수와 전체 용량에 제한이 없습니다.
- **높은 내구성**: S3 Standard는 **99.999999999%(11 9's)**의 내구성을 제공합니다. 이는 1,000만 개의 객체를 저장하면 평균적으로 1만 년에 1개를 잃을 수 있다는 의미입니다.
- **높은 가용성**: S3 Standard는 **99.99%**의 가용성을 제공합니다. 즉, 1년 중 약 52분을 제외하고 항상 접근 가능합니다.
- **HTTP/HTTPS 접근**: S3의 모든 객체는 고유한 URL을 가지며, 웹 브라우저나 API를 통해 접근할 수 있습니다.

S3 객체의 URL 구조:
```
https://[버킷이름].s3.[리전].amazonaws.com/[키]
예: https://my-photo-bucket.s3.ap-northeast-2.amazonaws.com/photos/sunset.jpg
```

#### S3 버전 관리(Versioning)

> 📝 **S3 버전 관리(Versioning)**: 같은 키(이름)로 객체를 업로드할 때 이전 버전을 덮어쓰지 않고, 모든 버전을 보존하는 기능입니다.

버전 관리를 활성화하면:
- 실수로 파일을 덮어써도 이전 버전으로 복구할 수 있습니다.
- 삭제 시에도 "삭제 마커"만 추가되며, 실제 데이터는 보존됩니다.
- 모든 버전이 저장되므로 스토리지 비용이 증가할 수 있습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.2.1: 개인 블로그의 이미지 저장소**
>
> **상황**: 개인 블로그를 운영하면서, 블로그 글에 삽입하는 이미지를 S3에 저장하려고 합니다.
> **풀이/설명**:
> 1. 버킷 생성: `my-blog-images-2024`라는 이름으로 서울 리전(ap-northeast-2)에 버킷을 만듭니다.
> 2. 이미지 업로드: `posts/hello-world/cover.jpg`라는 키로 이미지를 업로드합니다.
> 3. 블로그에서 사용: 이미지의 S3 URL을 블로그 글에 삽입합니다.
>
> **핵심 포인트**: S3는 웹 기반 접근이 가능하므로, 웹사이트의 정적 자산(이미지, CSS, JS 파일)을 저장하는 데 널리 사용됩니다.

> 🔍 **예시 6.2.2: 기업의 데이터 백업**
>
> **상황**: 매일 밤 회사 서버의 데이터베이스 백업 파일(약 50GB)을 안전하게 보관해야 합니다.
> **풀이/설명**:
> 1. 버킷 생성: `company-db-backups`라는 이름으로 버킷을 만듭니다.
> 2. 버전 관리 활성화: 실수로 백업 파일을 덮어써도 복구할 수 있도록 합니다.
> 3. 키 구조 설계: `backups/2024/01/15/db-full-backup.sql.gz`처럼 날짜별 키를 사용합니다.
> 4. S3의 11 9's 내구성 덕분에 백업 데이터를 잃을 걱정이 거의 없습니다.
>
> **핵심 포인트**: S3의 높은 내구성과 무제한 용량은 기업 백업 전략의 핵심 요소입니다. 버전 관리를 통해 추가적인 데이터 보호가 가능합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "버킷 이름을 나중에 바꿀 수 있나요?"
>
> 아닙니다. S3 버킷의 이름은 **생성 후 변경할 수 없습니다**. 이름을 바꾸려면 새 버킷을 만들고, 데이터를 옮긴 후 기존 버킷을 삭제해야 합니다. 처음 만들 때 이름을 신중하게 정하세요.

> ⚠️ **주의**: "S3에 저장하면 자동으로 전 세계에서 접근 가능한가요?"
>
> 아닙니다. S3 객체는 기본적으로 **프라이빗(비공개)** 상태입니다. 버킷을 만든 계정의 소유자만 접근할 수 있습니다. 다른 사용자나 퍼블릭에 공유하려면 명시적으로 권한을 설정해야 합니다 (Section 6.4 참고).

### Section 요약 (Section Summary)

- Amazon S3는 무제한 확장 가능한 **객체 스토리지** 서비스입니다.
- 데이터는 **버킷** 안에 **객체**로 저장되며, 각 객체는 **키(이름)**, **값(데이터)**, **메타데이터**로 구성됩니다.
- 버킷 이름은 전 세계에서 고유해야 하며, 특정 리전에 생성됩니다.
- S3 Standard는 99.999999999%의 내구성과 99.99%의 가용성을 제공합니다.
- **버전 관리**를 활성화하면 객체의 모든 버전을 보존하여 실수로 인한 데이터 손실을 방지할 수 있습니다.

## 6.3 S3 스토리지 클래스와 수명 주기 — 비용과 접근 빈도에 따른 클래스 선택 전략

> 💡 **한 줄 요약**: S3는 데이터의 접근 빈도에 따라 여러 스토리지 클래스를 제공하며, 수명 주기 정책으로 자동 전환하여 비용을 최적화할 수 있습니다.

### 직관적 이해 (Intuitive Understanding)

집에 있는 물건들을 생각해 봅시다. 매일 쓰는 식기는 싱크대 위 선반에, 가끔 쓰는 명절 그릇은 높은 선반에, 거의 안 쓰지만 버리지 못하는 추억의 물건은 창고 깊숙한 곳에 보관합니다. 싱크대 위 선반은 접근이 편하지만 공간이 비싸고(주방 공간), 창고는 싸지만 꺼내려면 시간이 걸립니다.

S3 스토리지 클래스도 같은 원리입니다. **자주 접근하는 데이터**에는 비싸지만 빠른 클래스를, **거의 접근하지 않는 데이터**에는 저렴하지만 꺼내는 데 시간이 걸리는 클래스를 사용합니다. 이것이 클라우드 스토리지 비용 최적화의 핵심 전략입니다.

### 핵심 개념 설명 (Core Explanation)

S3는 총 6가지 주요 스토리지 클래스를 제공합니다:

#### 1. S3 Standard (표준)
- **용도**: 자주 접근하는 데이터
- **특징**: 높은 내구성(11 9's), 높은 가용성(99.99%), 빠른 접근
- **비용**: 저장 비용이 가장 높지만, 데이터 검색 비용 없음
- **적합한 데이터**: 웹사이트 콘텐츠, 모바일 앱 데이터, 활발히 사용 중인 데이터

#### 2. S3 Intelligent-Tiering (지능형 계층화)
- **용도**: 접근 패턴을 예측할 수 없는 데이터
- **특징**: S3가 **자동으로** 접근 빈도를 모니터링하여 데이터를 적절한 계층으로 이동
- **비용**: 소액의 월별 모니터링 비용이 발생하지만, 데이터 검색 비용 없음
- **적합한 데이터**: 접근 패턴이 불규칙하거나 변화하는 데이터

#### 3. S3 Standard-IA (Infrequent Access, 비빈번 접근)
- **용도**: 자주 접근하지 않지만, 필요할 때 빠르게 꺼내야 하는 데이터
- **특징**: Standard와 같은 내구성·가용성, 저장 비용은 저렴하지만 데이터 검색 시 비용 발생
- **비용**: 저장 비용 ↓, 검색 비용 ↑
- **적합한 데이터**: 백업 데이터, 재해 복구용 파일, 오래된 로그 파일

#### 4. S3 One Zone-IA (단일 가용 영역 비빈번 접근)
- **용도**: 자주 접근하지 않으며, 가용 영역 장애 시 데이터 손실을 감수할 수 있는 데이터
- **특징**: 단일 가용 영역(AZ)에만 저장 — 해당 AZ에 물리적 장애가 발생하면 데이터 손실 가능
- **비용**: Standard-IA보다 약 20% 저렴
- **적합한 데이터**: 다시 생성할 수 있는 데이터, 다른 리전에 원본이 있는 복제 데이터

> 🔗 **연결**: Chapter 3에서 배운 가용 영역(AZ) 개념을 떠올려 봅시다. S3 Standard는 데이터를 최소 3개의 AZ에 자동으로 복제하지만, One Zone-IA는 1개의 AZ에만 저장하므로 더 저렴하지만 내구성이 상대적으로 낮습니다.

#### 5. S3 Glacier Flexible Retrieval (글레이셔 유연한 검색)

> 📝 **Glacier**: "빙하"라는 뜻으로, 오랫동안 얼려두듯이 장기 보관하는 데이터를 위한 스토리지 클래스입니다.

- **용도**: 장기 아카이브 — 1년에 1~2번 접근하는 데이터
- **특징**: 매우 저렴한 저장 비용, 데이터 검색에 **수 분~수 시간** 소요
- **검색 옵션**: 긴급(Expedited, 1~5분), 표준(Standard, 3~5시간), 대량(Bulk, 5~12시간)
- **적합한 데이터**: 규정 준수용 장기 보관 데이터, 미디어 아카이브

#### 6. S3 Glacier Deep Archive (글레이셔 딥 아카이브)
- **용도**: 가장 저렴한 장기 보관 — 7~10년 이상 보관하며 거의 접근하지 않는 데이터
- **특징**: S3에서 가장 저렴한 스토리지 클래스, 데이터 검색에 **12~48시간** 소요
- **적합한 데이터**: 법적 규정에 따라 반드시 보관해야 하는 데이터, 금융 기록, 의료 기록

#### 스토리지 클래스 비교표

| 스토리지 클래스 | 접근 빈도 | 저장 비용 | 검색 비용 | 검색 시간 | AZ 복제 |
|----------------|----------|----------|----------|----------|---------|
| Standard | 빈번 | 높음 | 없음 | 즉시 | 3+ AZ |
| Intelligent-Tiering | 불규칙 | 자동 최적화 | 없음 | 즉시 | 3+ AZ |
| Standard-IA | 비빈번 | 중간 | 있음 | 즉시 | 3+ AZ |
| One Zone-IA | 비빈번 | 중간-낮음 | 있음 | 즉시 | 1 AZ |
| Glacier Flexible Retrieval | 아카이브 | 낮음 | 있음 | 분~시간 | 3+ AZ |
| Glacier Deep Archive | 장기 아카이브 | 가장 낮음 | 있음 | 12~48시간 | 3+ AZ |

#### S3 수명 주기 정책 (Lifecycle Policy)

> 📝 **수명 주기 정책(Lifecycle Policy)**: 객체의 나이(생성 후 경과 일수)에 따라 자동으로 스토리지 클래스를 전환하거나, 객체를 삭제하는 규칙입니다.

예를 들어, 다음과 같은 규칙을 설정할 수 있습니다:
1. 생성 후 0~30일: **S3 Standard** (활발히 사용)
2. 30일 후: **S3 Standard-IA**로 전환 (접근 빈도 감소)
3. 90일 후: **S3 Glacier Flexible Retrieval**로 전환 (아카이브)
4. 365일 후: **객체 삭제** (보관 기간 만료)

이 과정이 모두 자동으로 이루어지므로, 관리자가 수동으로 데이터를 옮길 필요가 없습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.3.1: 뉴스 웹사이트의 이미지 관리**
>
> **상황**: 뉴스 웹사이트에서 매일 수백 장의 뉴스 기사 이미지를 업로드합니다. 최신 기사의 이미지는 자주 조회되지만, 오래된 기사의 이미지는 거의 조회되지 않습니다.
> **풀이/설명**:
> - 업로드 직후: **S3 Standard** — 새 기사는 조회수가 높으므로 빠른 접근 필요
> - 30일 후: **S3 Standard-IA** — 한 달 지난 기사는 가끔 접근
> - 1년 후: **S3 Glacier Flexible Retrieval** — 오래된 기사는 거의 접근하지 않지만 삭제하면 안 됨
> - 수명 주기 정책을 설정하면 이 전환이 자동으로 이루어집니다.
>
> **핵심 포인트**: 수명 주기 정책은 "시간이 지남에 따라 접근 빈도가 자연스럽게 줄어드는 데이터"에 이상적인 비용 최적화 전략입니다.

> 🔍 **예시 6.3.2: 접근 패턴을 모르는 신규 프로젝트**
>
> **상황**: 새로 시작하는 프로젝트에서 S3에 데이터를 저장하려고 합니다. 어떤 데이터가 자주 사용될지 아직 알 수 없습니다.
> **풀이/설명**:
> - 접근 패턴을 예측할 수 없으므로 **S3 Intelligent-Tiering**을 선택합니다.
> - S3가 자동으로 각 객체의 접근 빈도를 모니터링합니다.
> - 30일간 접근이 없으면 자동으로 비빈번 접근 계층으로, 90일간 접근이 없으면 아카이브 계층으로 이동합니다.
> - 다시 접근하면 자동으로 빈번 접근 계층으로 복귀합니다.
>
> **핵심 포인트**: Intelligent-Tiering은 "잘 모르겠다"가 답인 상황에서 최적의 선택입니다. 자동으로 비용을 최적화해 주므로 수명 주기 정책을 직접 설계할 필요가 없습니다.

>
> **목표**: Mini Lab 6.2에서 생성한 버킷에 수명 주기 규칙을 설정하여, 비용 최적화 자동화를 체험합니다.
> **예상 소요 시간**: 5분
> **비용**: Free Tier 범위 내
>
> **Step 1**: S3 콘솔에서 Mini Lab 6.2에서 만든 버킷을 클릭합니다.
>
> **Step 2**: **관리(Management)** 탭을 클릭합니다.
>
> **Step 3**: **수명 주기 규칙 생성(Create lifecycle rule)** 버튼을 클릭합니다.
>
> **Step 4**: 규칙을 설정합니다:
> - **수명 주기 규칙 이름**: `move-to-ia-after-30-days`
> - **규칙 범위 선택**: "버킷의 모든 객체에 적용"을 선택하고, 확인 체크박스를 체크합니다.
> - **수명 주기 규칙 작업**: "스토리지 클래스 간에 객체의 현재 버전 전환(Transition current versions of objects between storage classes)"을 체크합니다.
> - **전환 설정**: 스토리지 클래스 **Standard-IA**를 선택하고, 객체 생성 후 일수에 **30**을 입력합니다.
>
> **Step 5**: **규칙 생성(Create rule)** 버튼을 클릭합니다.
>
> **확인**: 관리(Management) 탭에 `move-to-ia-after-30-days` 규칙이 "활성화됨(Enabled)" 상태로 표시되면 성공입니다.
>
> 💬 **참고**: 이 규칙은 실제로 30일이 지나야 동작하므로, 즉각적인 변화는 보이지 않습니다. 규칙이 정상적으로 생성되었는지만 확인하면 됩니다.
>
> 🧹 **정리**: 이 수명 주기 규칙은 Chapter 6 통합 실습 후 버킷과 함께 삭제합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "Glacier에 넣으면 비용이 적게 드니까 모든 데이터를 Glacier에 넣자!"
>
> Glacier는 저장 비용은 매우 저렴하지만, **검색 비용이 높고 시간이 오래 걸립니다**. 자주 접근해야 하는 데이터를 Glacier에 넣으면 오히려 총 비용이 높아지고, 사용자 경험도 나빠집니다. 반드시 접근 빈도에 맞는 클래스를 선택하세요.

> ⚠️ **주의**: "Standard-IA에는 최소 저장 기간(30일)과 최소 객체 크기(128KB) 과금이 있습니다."
>
> Standard-IA에 저장된 객체는 30일 이전에 삭제하더라도 30일분의 저장 비용이 청구됩니다. 또한 128KB보다 작은 객체도 128KB로 과금됩니다. 작은 파일이나 단기간 저장하는 데이터에는 Standard가 오히려 경제적일 수 있습니다.

### Section 요약 (Section Summary)

- S3는 **6가지 스토리지 클래스**를 제공하며, 접근 빈도에 따라 비용이 달라집니다.
- **Standard**는 빈번한 접근, **Standard-IA/One Zone-IA**는 비빈번 접근, **Glacier/Deep Archive**는 장기 아카이브에 적합합니다.
- **Intelligent-Tiering**은 접근 패턴을 예측할 수 없을 때 자동으로 비용을 최적화합니다.
- **수명 주기 정책**을 사용하면 시간 경과에 따라 자동으로 스토리지 클래스를 전환하거나 객체를 삭제할 수 있습니다.
- 각 클래스의 저장 비용, 검색 비용, 검색 시간, 최소 저장 기간을 종합적으로 고려하여 선택해야 합니다.

## 6.4 S3 보안과 접근 제어 — 버킷 정책, ACL, 퍼블릭 접근 차단

> 💡 **한 줄 요약**: S3는 기본적으로 모든 데이터를 프라이빗으로 유지하며, 버킷 정책과 접근 제어 기능으로 누가 데이터에 접근할 수 있는지를 세밀하게 제어합니다.

### 직관적 이해 (Intuitive Understanding)

S3 버킷을 은행의 **금고**라고 생각해 봅시다. 금고는 기본적으로 잠겨 있으며, 금고 주인(계정 소유자)만 열 수 있습니다. 다른 사람이 접근하려면 명시적으로 "이 사람에게 접근 권한을 줄게"라는 허가가 필요합니다.

S3의 보안도 같은 원리입니다. **기본적으로 모든 것이 차단**되어 있고, 필요한 만큼만 문을 열어줍니다. 이것을 **"기본 거부(Default Deny)"** 원칙이라고 합니다.

왜 이렇게 엄격할까요? 2017년, 어떤 기업이 S3 버킷의 접근 설정을 잘못하여 수백만 명의 고객 데이터가 공개된 사건이 있었습니다. 이후 AWS는 S3의 보안 기본값을 더욱 강화했으며, 현재는 **퍼블릭 접근 차단이 기본값**으로 설정됩니다.

### 핵심 개념 설명 (Core Explanation)

#### 버킷 정책 (Bucket Policy)

> 📝 **버킷 정책(Bucket Policy)**: 버킷 수준에서 접근 권한을 정의하는 JSON 형식의 규칙입니다. "누가, 어떤 동작을, 어떤 리소스에 대해, 허용 또는 거부할지"를 명시합니다.

버킷 정책의 기본 구조:
```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow 또는 Deny",
            "Principal": "누구에게 (예: 모든 사용자, 특정 계정)",
            "Action": "어떤 동작 (예: s3:GetObject, s3:PutObject)",
            "Resource": "어떤 리소스 (예: 버킷 내 특정 경로)"
        }
    ]
}
```

주요 용도:
- 특정 사용자나 다른 AWS 계정에게 접근 허용
- 버킷의 특정 폴더(접두사)만 공개
- 특정 IP 주소에서만 접근 허용
- 웹사이트 호스팅을 위해 객체를 공개 읽기로 설정

#### S3 퍼블릭 접근 차단 (Block Public Access)

AWS는 실수로 데이터를 공개하는 것을 방지하기 위해, **퍼블릭 접근 차단** 기능을 제공합니다. 이 설정은 **계정 수준**과 **버킷 수준** 모두에서 적용할 수 있으며, 기본적으로 활성화되어 있습니다.

4가지 차단 옵션:
1. 새로운 퍼블릭 ACL을 통한 접근 차단
2. 기존 퍼블릭 ACL을 통한 접근 차단
3. 새로운 퍼블릭 버킷 정책을 통한 접근 차단
4. 기존 퍼블릭 버킷 정책을 통한 접근 차단

> 💬 **참고**: 대부분의 경우 **4개 모두 활성화**(기본값)하는 것을 권장합니다. 정적 웹사이트 호스팅처럼 의도적으로 공개해야 하는 경우에만 선별적으로 비활성화합니다.

#### ACL (Access Control List)

> 📝 **ACL(Access Control List)**: 개별 버킷이나 객체에 대해 다른 AWS 계정이나 미리 정의된 그룹에게 읽기/쓰기 권한을 부여하는 방식입니다.

AWS는 현재 **ACL 대신 버킷 정책 사용을 권장**합니다. 새로 생성하는 버킷은 기본적으로 "ACL 비활성화"로 설정됩니다. ACL은 레거시(과거 호환용) 기능으로 이해하면 됩니다.

#### S3 암호화

S3는 데이터를 보호하기 위해 **암호화**를 기본으로 제공합니다:

- **서버 측 암호화(SSE, Server-Side Encryption)**: S3가 데이터를 저장할 때 자동으로 암호화하고, 읽을 때 자동으로 복호화합니다. 2023년부터 **모든 새 객체는 기본적으로 SSE-S3로 암호화**됩니다.
- **클라이언트 측 암호화**: 데이터를 S3에 업로드하기 전에 사용자가 직접 암호화합니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.4.1: 회사 보고서를 특정 부서에만 공개**
>
> **상황**: S3 버킷에 저장된 월간 보고서를 마케팅 부서의 AWS 계정에서만 읽을 수 있게 하고 싶습니다.
> **풀이/설명**: 버킷 정책을 사용합니다:
> - **Effect**: Allow (허용)
> - **Principal**: 마케팅 부서의 AWS 계정 ID
> - **Action**: `s3:GetObject` (읽기만 허용)
> - **Resource**: `arn:aws:s3:::company-reports/marketing/*` (marketing 폴더 내 객체만)
>
> 이렇게 하면 마케팅 부서는 `marketing/` 폴더의 보고서만 읽을 수 있고, 다른 폴더에는 접근할 수 없습니다.
>
> **핵심 포인트**: 버킷 정책은 "최소 권한 원칙"에 따라 필요한 만큼만 접근을 허용해야 합니다.

> 🔍 **예시 6.4.2: 정적 웹사이트 호스팅**
>
> **상황**: S3 버킷에 HTML, CSS, 이미지 파일을 올려 간단한 웹사이트를 만들고 싶습니다.
> **풀이/설명**:
> 1. 버킷의 퍼블릭 접근 차단 설정을 비활성화합니다 (의도적 공개).
> 2. 버킷 정책에서 모든 사용자(`"Principal": "*"`)에게 `s3:GetObject` 권한을 부여합니다.
> 3. 버킷의 "정적 웹사이트 호스팅" 기능을 활성화합니다.
> 4. 이제 누구든 웹 브라우저에서 버킷 URL로 접속하면 웹사이트가 표시됩니다.
>
> **핵심 포인트**: 퍼블릭 접근은 의도적인 경우(웹사이트 호스팅 등)에만 허용하고, 실수로 공개하지 않도록 주의해야 합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "S3 버킷을 퍼블릭으로 설정하면 AWS 콘솔에서 경고가 표시됩니다."
>
> AWS는 퍼블릭 버킷 옆에 주황색 **"공개적으로 액세스 가능(Publicly accessible)"** 경고 배지를 표시합니다. 이 경고가 보이는데 의도한 것이 아니라면, 즉시 퍼블릭 접근 차단을 다시 활성화하세요. 민감한 데이터가 공개되면 돌이킬 수 없는 피해가 발생할 수 있습니다.

> ⚠️ **주의**: "버킷 정책의 JSON 문법 오류"
>
> 버킷 정책은 JSON 형식이므로, 중괄호나 쉼표 하나가 빠져도 정책 전체가 적용되지 않습니다. AWS 콘솔의 정책 편집기에서 문법 오류를 자동으로 감지해주므로, 저장 전 오류 메시지를 확인하세요.

### Section 요약 (Section Summary)

- S3는 **기본적으로 모든 데이터를 프라이빗**으로 유지합니다 (기본 거부 원칙).
- **버킷 정책**은 JSON 형식으로 "누가, 무엇을, 어디에" 접근할 수 있는지를 정의합니다.
- **퍼블릭 접근 차단**이 기본으로 활성화되어 있어, 실수로 데이터를 공개하는 것을 방지합니다.
- **ACL**은 레거시 기능이며, 현재는 버킷 정책 사용이 권장됩니다.
- **서버 측 암호화(SSE)**가 모든 새 객체에 기본 적용되어 저장 데이터를 보호합니다.

## 6.5 Amazon EBS — EC2에 연결되는 블록 스토리지, 볼륨 유형, 스냅샷

> 💡 **한 줄 요약**: Amazon EBS는 클라우드 서버에 연결하는 가상 하드디스크로, 운영체제와 데이터베이스처럼 빠른 읽기/쓰기가 필요한 데이터를 저장합니다.

### 직관적 이해 (Intuitive Understanding)

여러분의 컴퓨터에는 하드디스크(또는 SSD)가 장착되어 있습니다. 운영체제(Windows, macOS)가 설치되어 있고, 프로그램과 파일이 저장되는 곳이죠. 이 하드디스크 없이는 컴퓨터가 동작하지 않습니다.

**Amazon EBS(Elastic Block Store)**는 이 하드디스크의 클라우드 버전입니다. 클라우드 서버에 연결하여 운영체제를 설치하고, 데이터베이스를 실행하고, 애플리케이션 파일을 저장하는 데 사용합니다.

> 💬 **참고**: EBS는 AWS의 클라우드 서버 서비스인 **EC2(Elastic Compute Cloud)** 인스턴스에 연결하여 사용합니다. EC2는 Chapter 5에서 자세히 다루는 컴퓨팅 서비스입니다. 이 Section에서는 EBS의 스토리지 관점에 집중하며, EC2에 대해서는 "클라우드 서버"로 이해하시면 됩니다.

EBS가 일반 하드디스크와 다른 핵심적인 차이점은:
- **네트워크로 연결**: 물리적으로 꽂는 것이 아니라, 네트워크를 통해 서버에 연결됩니다. 따라서 한 서버에서 분리하여 다른 서버에 연결할 수 있습니다.
- **크기 변경 가능**: 용량이 부족하면 실행 중에도 크기를 늘릴 수 있습니다.
- **스냅샷**: 특정 시점의 전체 상태를 사진 찍듯이 백업할 수 있습니다.

### 핵심 개념 설명 (Core Explanation)

#### EBS 볼륨의 핵심 특성

> 📝 **EBS 볼륨(Volume)**: EC2 인스턴스에 연결할 수 있는 네트워크 기반의 블록 스토리지 장치입니다. 컴퓨터에 장착하는 하드디스크와 같은 역할을 합니다.

- **가용 영역(AZ) 고정**: EBS 볼륨은 특정 AZ에 생성되며, **같은 AZ의 EC2 인스턴스에만** 연결할 수 있습니다.
- **1:1 연결 (기본)**: 하나의 EBS 볼륨은 기본적으로 하나의 EC2 인스턴스에만 연결됩니다 (일부 볼륨 유형은 Multi-Attach 지원).
- **독립적 수명**: EC2 인스턴스를 종료해도 EBS 볼륨은 유지할 수 있습니다 (설정에 따라 다름).

#### EBS 볼륨 유형

EBS는 용도에 따라 크게 **SSD 기반**과 **HDD 기반** 볼륨을 제공합니다:

**SSD 기반 볼륨 (빠른 응답 속도가 필요한 경우)**

| 볼륨 유형 | 이름 | 주요 용도 | IOPS | 특징 |
|----------|------|----------|------|------|
| **gp3** | 범용 SSD | 대부분의 워크로드 | 최대 16,000 | 가장 일반적, 비용 대비 성능 우수 |
| **gp2** | 범용 SSD (이전 세대) | 대부분의 워크로드 | 최대 16,000 | gp3 이전의 기본 볼륨 |
| **io2/io2 Block Express** | 프로비저닝된 IOPS SSD | 고성능 데이터베이스 | 최대 256,000 | 최고 성능, 비용 높음 |

> 📝 **IOPS(Input/Output Operations Per Second)**: 초당 입출력 작업 수로, 스토리지의 성능을 측정하는 단위입니다. IOPS가 높을수록 더 빠르게 데이터를 읽고 쓸 수 있습니다.

**HDD 기반 볼륨 (높은 처리량이 필요한 경우)**

| 볼륨 유형 | 이름 | 주요 용도 | 처리량 | 특징 |
|----------|------|----------|--------|------|
| **st1** | 처리량 최적화 HDD | 빅데이터, 로그 처리 | 최대 500 MB/s | 순차 읽기/쓰기에 최적화 |
| **sc1** | 콜드 HDD | 자주 접근하지 않는 데이터 | 최대 250 MB/s | 가장 저렴한 EBS 볼륨 |

> 💬 **참고**: 운영체제가 설치되는 **부트 볼륨**으로는 SSD 기반(gp3, gp2, io2)만 사용할 수 있습니다. HDD 볼륨(st1, sc1)은 부트 볼륨으로 사용할 수 없습니다.

#### EBS 스냅샷 (Snapshot)

> 📝 **EBS 스냅샷(Snapshot)**: EBS 볼륨의 특정 시점 상태를 복사(백업)한 것입니다. 스냅샷은 Amazon S3에 저장되어 높은 내구성을 가지며, 다른 AZ나 리전에서 새 볼륨을 생성하는 데 사용할 수 있습니다.

스냅샷의 주요 특징:
- **증분식(Incremental)**: 첫 스냅샷은 전체 볼륨을 복사하지만, 이후 스냅샷은 변경된 부분만 저장합니다. 따라서 저장 비용과 시간이 절약됩니다.
- **리전 간 복사**: 스냅샷을 다른 리전으로 복사하여 재해 복구(DR)에 활용할 수 있습니다.
- **새 볼륨 생성**: 스냅샷에서 새로운 EBS 볼륨을 만들 수 있습니다. 이를 통해 다른 AZ에 동일한 데이터를 가진 볼륨을 생성할 수 있습니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.5.1: 웹 서버의 기본 스토리지 구성**
>
> **상황**: 소규모 웹 서버를 클라우드에 만들려고 합니다. 운영체제를 설치하고 웹 애플리케이션 파일을 저장할 스토리지가 필요합니다.
> **풀이/설명**: **gp3** 볼륨을 사용합니다:
> - 운영체제 설치와 웹 애플리케이션 실행에 적합한 범용 성능
> - 비용 대비 성능이 우수하여 대부분의 워크로드에 기본으로 사용
> - 필요에 따라 IOPS와 처리량을 독립적으로 조정 가능
>
> **핵심 포인트**: "어떤 볼륨 유형을 쓸지 모르겠다"면 **gp3**를 선택하세요. 대부분의 일반적인 워크로드에 적합합니다.

> 🔍 **예시 6.5.2: 재해 복구를 위한 스냅샷 활용**
>
> **상황**: 서울 리전에서 운영 중인 서버의 데이터를 도쿄 리전에도 백업해 두고 싶습니다.
> **풀이/설명**:
> 1. 서울 리전의 EBS 볼륨에서 스냅샷을 생성합니다.
> 2. 해당 스냅샷을 도쿄 리전으로 복사합니다.
> 3. 서울 리전에 장애가 발생하면, 도쿄 리전에서 스냅샷으로 새 EBS 볼륨을 생성하고 서버를 복구합니다.
>
> **핵심 포인트**: 스냅샷의 리전 간 복사 기능은 재해 복구 전략의 핵심 요소입니다. 스냅샷은 S3에 저장되므로 높은 내구성이 보장됩니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "EBS 볼륨은 같은 AZ에서만 연결할 수 있습니다."
>
> 서울 리전의 `ap-northeast-2a` AZ에 생성한 EBS 볼륨을 `ap-northeast-2b` AZ의 EC2 인스턴스에 직접 연결할 수 없습니다. 다른 AZ로 데이터를 옮기려면 스냅샷을 생성한 후 원하는 AZ에서 새 볼륨을 생성해야 합니다.

> ⚠️ **주의**: "EC2 인스턴스를 종료(Terminate)하면 기본적으로 루트 EBS 볼륨도 함께 삭제됩니다."
>
> EC2 인스턴스의 루트 볼륨(운영체제가 설치된 볼륨)은 기본 설정상 인스턴스 종료 시 함께 삭제됩니다. 중요한 데이터는 종료 전에 스냅샷을 생성하거나, "종료 시 삭제" 옵션을 비활성화해야 합니다.

> ⚠️ **주의**: "사용하지 않는 EBS 볼륨도 비용이 발생합니다."
>
> EBS 볼륨은 EC2 인스턴스에 연결되어 있지 않아도 **프로비저닝된 용량에 대해 과금**됩니다. 사용하지 않는 볼륨은 스냅샷을 생성한 후 삭제하여 불필요한 비용을 줄이세요.

### Section 요약 (Section Summary)

- **Amazon EBS**는 EC2 인스턴스에 연결하는 네트워크 기반 블록 스토리지입니다.
- **gp3**(범용 SSD)가 가장 일반적으로 사용되며, 고성능이 필요하면 **io2**, 대용량 순차 처리에는 **st1/sc1**을 선택합니다.
- EBS 볼륨은 **특정 AZ에 고정**되며, 같은 AZ의 EC2 인스턴스에만 연결할 수 있습니다.
- **EBS 스냅샷**은 볼륨의 시점 백업으로, 증분식으로 저장되며 리전 간 복사가 가능합니다.
- 사용하지 않는 EBS 볼륨과 스냅샷도 비용이 발생하므로 주기적으로 정리해야 합니다.

## 6.6 Amazon EFS와 FSx — 공유 파일 시스템, 관리형 파일 스토리지

> 💡 **한 줄 요약**: Amazon EFS는 여러 서버가 동시에 접근할 수 있는 공유 파일 시스템이며, FSx는 Windows나 고성능 컴퓨팅 환경에 특화된 관리형 파일 스토리지입니다.

### 직관적 이해 (Intuitive Understanding)

사무실에서 여러 직원이 하나의 네트워크 드라이브(공유 폴더)에 접근하는 상황을 떠올려 보세요. 디자이너 A가 올린 시안을 디자이너 B가 바로 확인하고, 기획자 C도 같은 폴더에서 문서를 참고합니다.

**Amazon EBS**는 컴퓨터 한 대에 연결된 개인 USB 드라이브와 같습니다 — 해당 컴퓨터만 사용할 수 있습니다. 반면 **Amazon EFS**는 회사의 공유 네트워크 드라이브와 같습니다 — 여러 컴퓨터가 동시에 접근할 수 있습니다.

| | EBS | EFS |
|---|---|---|
| **비유** | 개인 USB 드라이브 | 회사 공유 드라이브 |
| **연결** | 하나의 EC2에 연결 | 여러 EC2에서 동시 접근 |
| **용량** | 미리 지정 (예: 100GB) | 자동 확장/축소 |
| **AZ 제한** | 특정 AZ에 고정 | 리전 내 여러 AZ에서 접근 가능 |

### 핵심 개념 설명 (Core Explanation)

#### Amazon EFS (Elastic File System)

> 📝 **Amazon EFS(Elastic File System)**: 여러 EC2 인스턴스가 동시에 접근할 수 있는 완전관리형 파일 스토리지 서비스입니다. Linux 기반의 NFS(Network File System) 프로토콜을 사용합니다.

EFS의 핵심 특징:
- **다중 접근**: 수백~수천 개의 EC2 인스턴스가 동시에 같은 파일 시스템에 접근 가능
- **자동 확장**: 파일을 추가하면 자동으로 용량이 늘어나고, 삭제하면 줄어듭니다. 용량을 미리 지정할 필요가 없습니다.
- **리전 내 가용**: 리전 내 여러 AZ에서 접근 가능 (데이터가 여러 AZ에 자동 복제)
- **Linux 전용**: NFS 프로토콜을 사용하므로, Linux 기반 EC2 인스턴스에서 사용합니다 (Windows는 지원하지 않음).

EFS의 스토리지 클래스:
- **Standard**: 자주 접근하는 데이터 — 빠른 응답
- **Infrequent Access (IA)**: 비빈번 접근 데이터 — 저장 비용 절감
- **수명 주기 관리**: S3와 유사하게, 일정 기간 접근하지 않은 파일을 자동으로 IA 클래스로 전환 가능

#### Amazon FSx

> 📝 **Amazon FSx**: 특정 파일 시스템에 최적화된 완전관리형 파일 스토리지 서비스입니다. 여러 종류가 있지만, CCP 시험에서 주로 다루는 것은 FSx for Windows File Server와 FSx for Lustre입니다.

**FSx for Windows File Server**
- Windows 환경의 공유 파일 시스템 (SMB 프로토콜)
- Active Directory 통합 지원
- Windows 기반 EC2 인스턴스에서 사용
- **EFS가 Linux용이라면, FSx for Windows는 Windows용 공유 파일 시스템**

**FSx for Lustre**
- 고성능 컴퓨팅(HPC), 머신 러닝, 미디어 처리 등 **대규모 데이터 처리**에 특화
- 초당 수백 기가바이트의 처리량과 수백만 IOPS 제공
- S3와 통합하여 데이터를 자동으로 가져오고 내보낼 수 있음

#### 스토리지 서비스 선택 가이드

| 요구사항 | 추천 서비스 |
|---------|-----------|
| 단일 서버의 디스크 스토리지 | **EBS** |
| 여러 Linux 서버의 공유 파일 시스템 | **EFS** |
| 여러 Windows 서버의 공유 파일 시스템 | **FSx for Windows** |
| 고성능 컴퓨팅(HPC)용 파일 시스템 | **FSx for Lustre** |
| 대량의 비정형 데이터(이미지, 동영상, 백업) | **S3** |

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.6.1: 웹 서버 팜의 공유 콘텐츠 — EFS**
>
> **상황**: 트래픽이 많은 웹사이트를 운영하기 위해 여러 대의 웹 서버를 동시에 가동하고 있습니다. 모든 서버가 동일한 웹 페이지 파일과 사용자 업로드 파일에 접근해야 합니다.
> **풀이/설명**: Amazon EFS를 사용합니다:
> - 모든 웹 서버(EC2 인스턴스)가 같은 EFS 파일 시스템을 마운트합니다.
> - 사용자가 서버 A에 파일을 업로드하면, 서버 B, C에서도 즉시 해당 파일에 접근할 수 있습니다.
> - 서버가 추가되어도 새 서버에서 같은 EFS를 마운트하면 됩니다.
>
> **핵심 포인트**: "여러 서버가 같은 파일을 동시에 읽고 써야 한다"면 EFS가 정답입니다.

> 🔍 **예시 6.6.2: 기존 Windows 서버 환경의 클라우드 이전 — FSx for Windows**
>
> **상황**: 회사에서 기존에 Windows 서버의 공유 폴더를 통해 문서를 관리하고 있었는데, 이 환경을 AWS로 옮기려고 합니다.
> **풀이/설명**: FSx for Windows File Server를 사용합니다:
> - 기존 Windows 환경에서 사용하던 SMB 프로토콜을 그대로 지원합니다.
> - Active Directory와 통합되어, 기존 사용자 인증 체계를 그대로 유지할 수 있습니다.
> - 사용자 입장에서는 기존과 동일하게 네트워크 드라이브로 접근합니다.
>
> **핵심 포인트**: Windows 환경의 파일 공유에는 EFS가 아닌 FSx for Windows를 사용합니다. EFS는 Linux(NFS) 전용입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "EFS는 Linux 전용, FSx for Windows는 Windows 전용"
>
> 시험에서 자주 출제되는 포인트입니다. 공유 파일 시스템이 필요한데 운영체제가 Linux라면 EFS, Windows라면 FSx for Windows를 선택해야 합니다. 두 서비스를 혼동하지 마세요.

> ⚠️ **주의**: "EFS는 사용한 만큼만 과금되지만, 비용이 EBS보다 높습니다."
>
> EFS는 용량을 미리 지정하지 않아 편리하지만, GB당 비용이 EBS의 gp3보다 높습니다. 단일 서버에서만 사용할 데이터라면 EBS가 더 경제적입니다. **공유가 필요한 경우에만** EFS를 사용하세요.

### Section 요약 (Section Summary)

- **Amazon EFS**는 여러 Linux EC2 인스턴스가 동시에 접근하는 공유 파일 시스템이며, 용량이 자동으로 확장/축소됩니다.
- **FSx for Windows**는 Windows 환경(SMB, Active Directory)에 최적화된 파일 스토리지입니다.
- **FSx for Lustre**는 고성능 컴퓨팅(HPC)에 특화된 초고성능 파일 시스템입니다.
- 스토리지 선택은 **접근 패턴(단일 vs 공유), 운영체제(Linux vs Windows), 성능 요구사항**에 따라 결정합니다.

## 6.7 AWS Backup과 데이터 보호 — 중앙 집중식 백업 관리, 백업 정책

> 💡 **한 줄 요약**: AWS Backup은 여러 AWS 서비스의 백업을 하나의 중앙 콘솔에서 통합 관리하는 완전관리형 백업 서비스입니다.

### 직관적 이해 (Intuitive Understanding)

집에 귀중품이 여러 곳에 흩어져 있다고 상상해 봅시다. 금고에는 현금, 서랍에는 보석, 책상에는 중요 서류가 있습니다. 각각을 별도로 관리하면 하나쯤은 빠뜨리기 쉽겠죠. 대신 **하나의 보안 업체에 위탁**하여 "매주 금요일에 모든 귀중품을 백업해 주세요"라고 한 번만 설정하면 편리합니다.

AWS Backup이 바로 그 "보안 업체" 역할입니다. S3, EBS, EFS 등 여러 AWS 서비스의 데이터를 **하나의 백업 정책**으로 통합 관리합니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS Backup의 핵심 기능

> 📝 **AWS Backup**: EBS, EFS, S3, RDS 등 다양한 AWS 서비스의 백업을 중앙에서 관리할 수 있는 완전관리형 서비스입니다.

- **백업 계획(Backup Plan)**: "언제, 얼마나 자주, 얼마나 오래 보관할지"를 정의하는 규칙입니다.
  - 예: "매일 새벽 2시에 백업, 30일간 보관, 이후 삭제"
- **백업 볼트(Backup Vault)**: 백업 데이터가 저장되는 암호화된 컨테이너입니다.
- **지원 서비스**: EBS, EFS, S3, RDS, DynamoDB, FSx 등 15개 이상의 AWS 서비스
- **리전 간 백업**: 다른 리전으로 백업을 자동 복사하여 재해 복구에 활용
- **규정 준수**: 감사 추적(audit trail)과 백업 보고서를 생성하여 규정 준수를 증명

#### 데이터 보호 전략의 핵심 원칙

> 📝 **3-2-1 백업 규칙**: 중요한 데이터는 **3개의 복사본**을 만들고, **2가지 다른 미디어 유형**에 저장하며, **1개는 오프사이트(원격지)**에 보관하는 전략입니다.

AWS에서 이 원칙을 적용하면:
- **3개 복사본**: 원본 데이터 + EBS 스냅샷 + 다른 리전에 복사된 스냅샷
- **2가지 미디어**: EBS(블록 스토리지) + S3(객체 스토리지, 스냅샷 저장)
- **1개 원격지**: 다른 리전에 저장된 백업 복사본

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.7.1: 스타트업의 간단한 백업 전략**
>
> **상황**: 스타트업에서 웹 서버(EC2+EBS)와 데이터베이스를 운영하고 있으며, 데이터 손실에 대비하여 백업을 설정하려고 합니다.
> **풀이/설명**: AWS Backup으로 백업 계획을 생성합니다:
> - **백업 주기**: 매일 새벽 3시 (서비스 이용이 적은 시간)
> - **보관 기간**: 30일 (최근 30일의 백업을 유지)
> - **대상 리소스**: 웹 서버의 EBS 볼륨 + 데이터베이스 인스턴스
> - **리전 간 복사**: 서울 리전의 백업을 도쿄 리전에도 자동 복사
>
> **핵심 포인트**: AWS Backup을 사용하면 여러 서비스의 백업을 하나의 정책으로 통합 관리하여, 백업 누락을 방지하고 일관된 보호를 보장합니다.

> 🔍 **예시 6.7.2: 규정 준수를 위한 장기 보관**
>
> **상황**: 금융 회사에서 법적으로 거래 기록을 7년간 보관해야 합니다.
> **풀이/설명**:
> - AWS Backup에서 보관 기간을 7년(2,555일)으로 설정합니다.
> - 백업 볼트에 **볼트 잠금(Vault Lock)** 을 설정하여, 관리자도 보관 기간 내에 백업을 삭제할 수 없게 합니다.
> - 백업 보고서를 생성하여 감사 시 규정 준수를 증명합니다.
>
> **핵심 포인트**: AWS Backup의 볼트 잠금 기능은 법적 규정에 따라 데이터를 보호해야 하는 환경에서 핵심적인 역할을 합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "백업만 하고 복구 테스트를 하지 않는 것"
>
> 백업이 존재하더라도 실제로 복구가 되는지 확인하지 않으면 무용지물입니다. 주기적으로 백업에서 리소스를 복구하는 테스트를 수행하여, 실제 장애 시 복구가 가능한지 검증해야 합니다.

> ⚠️ **주의**: "백업도 비용이 발생합니다."
>
> 백업 데이터는 별도의 스토리지를 사용하므로 비용이 발생합니다. 불필요하게 긴 보관 기간이나 과도한 백업 빈도는 비용을 증가시킵니다. 비즈니스 요구사항에 맞는 적정 수준의 백업 정책을 설계하세요.

### Section 요약 (Section Summary)

- **AWS Backup**은 여러 AWS 서비스의 백업을 중앙에서 통합 관리하는 완전관리형 서비스입니다.
- **백업 계획**에서 주기, 보관 기간, 대상 리소스, 리전 간 복사를 설정합니다.
- **3-2-1 백업 규칙**은 데이터 보호의 기본 원칙으로, AWS에서도 이를 적용할 수 있습니다.
- 백업 설정 후에는 반드시 **복구 테스트**를 수행하여 실효성을 검증해야 합니다.

## 6.8 하이브리드 스토리지와 데이터 전송 — Storage Gateway, Snow Family 개요

> 💡 **한 줄 요약**: AWS Storage Gateway는 온프레미스 환경과 AWS 클라우드 스토리지를 연결하는 하이브리드 서비스이며, Snow Family는 대용량 데이터를 물리적으로 전송하는 디바이스입니다.

### 직관적 이해 (Intuitive Understanding)

회사가 10년간 쌓아온 서버실의 데이터를 클라우드로 옮기려고 합니다. 두 가지 도전이 있습니다:

**도전 1: 점진적 전환**
"클라우드로 한 번에 옮길 수는 없으니, 기존 서버실과 클라우드를 함께 사용하면서 천천히 옮기자." → 이것이 **AWS Storage Gateway**의 역할입니다. 기존 서버에서는 지금처럼 사용하면서, 뒤에서는 데이터가 자동으로 클라우드에 동기화됩니다.

**도전 2: 대량 데이터 전송**
"서버실에 100TB의 데이터가 있는데, 인터넷으로 옮기면 한 달이 걸린다!" → 이것이 **Snow Family**의 역할입니다. AWS가 물리적인 저장 장치를 배송해주면, 데이터를 복사하여 다시 보내는 방식입니다. 마치 택배로 하드디스크를 보내는 것과 같습니다.

> 🔗 **연결**: Chapter 1에서 배운 클라우드 컴퓨팅의 이점 중 하나인 "민첩성"을 떠올려 봅시다. 하지만 기존 온프레미스 환경의 데이터를 클라우드로 전환하는 과정은 신중한 계획이 필요합니다. Storage Gateway와 Snow Family는 이 전환을 돕는 서비스입니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS Storage Gateway

> 📝 **AWS Storage Gateway**: 온프레미스(회사 자체 서버) 환경과 AWS 클라우드 스토리지를 연결하는 하이브리드 스토리지 서비스입니다. 온프레미스 애플리케이션은 기존 방식대로 데이터에 접근하면서, 실제 데이터는 AWS에 저장됩니다.

> 📝 **온프레미스(On-premises)**: 기업이 자체적으로 소유하고 관리하는 물리적 서버나 데이터 센터를 의미합니다. 클라우드의 반대 개념입니다.

Storage Gateway의 주요 유형:

| 유형 | 프로토콜 | 주요 용도 |
|------|---------|----------|
| **S3 File Gateway** | NFS/SMB | 온프레미스에서 파일 형태로 접근, 실제 데이터는 S3에 저장 |
| **FSx File Gateway** | SMB | 온프레미스에서 FSx for Windows에 빠르게 접근 |
| **Volume Gateway** | iSCSI | 온프레미스의 블록 스토리지를 클라우드에 백업 |
| **Tape Gateway** | iSCSI/VTL | 기존 테이프 백업 환경을 S3/Glacier로 대체 |

#### AWS Snow Family

> 📝 **AWS Snow Family**: 네트워크 전송이 비현실적일 정도로 대량의 데이터를 AWS로(또는 AWS에서) 물리적으로 이동시키는 디바이스 서비스입니다.

| 디바이스 | 용량 | 주요 용도 |
|---------|------|----------|
| **Snowcone** | 8~14TB | 가장 작고 가벼움, 드론·차량 등 엣지 환경 |
| **Snowball Edge** | 80~210TB | 가장 일반적, 대용량 데이터 마이그레이션 |
| **Snowmobile** | 100PB | 트럭 크기의 컨테이너, 초대용량 마이그레이션 |

Snow Family 사용 과정:
1. AWS 콘솔에서 Snow 디바이스를 주문합니다.
2. AWS가 디바이스를 배송합니다.
3. 디바이스에 데이터를 복사합니다.
4. 디바이스를 AWS로 반송합니다.
5. AWS가 데이터를 S3에 업로드합니다.

> 💬 **참고**: Snow Family의 마이그레이션 관점에서의 심화 내용은 Chapter 20에서 자세히 다룹니다. 이 Section에서는 스토리지 전송 수단으로서의 기본 개념을 이해하는 데 집중합니다.

### 상세 예시 (Detailed Examples)

> 🔍 **예시 6.8.1: 온프레미스 파일 서버의 클라우드 확장 — Storage Gateway**
>
> **상황**: 회사의 파일 서버 용량이 부족해지고 있지만, 직원들은 기존 방식(폴더 열기)으로 계속 접근해야 합니다.
> **풀이/설명**: S3 File Gateway를 설치합니다:
> - 직원들은 기존과 동일하게 네트워크 드라이브(공유 폴더)로 파일에 접근합니다.
> - Gateway는 자주 접근하는 파일을 로컬에 캐시(임시 저장)하여 빠른 접근을 보장합니다.
> - 실제 파일은 S3에 저장되므로 용량 제한이 없습니다.
>
> **핵심 포인트**: Storage Gateway는 "기존 사용 방식은 바꾸지 않으면서, 저장소만 클라우드로 확장"하고 싶을 때 적합합니다.

> 🔍 **예시 6.8.2: 200TB 데이터 센터 이전 — Snowball Edge**
>
> **상황**: 회사가 데이터 센터를 폐쇄하면서 200TB의 데이터를 AWS로 옮겨야 합니다. 회사의 인터넷 속도는 1Gbps입니다.
> **풀이/설명**:
> - 1Gbps로 200TB를 전송하면 약 **18일**이 걸립니다 (네트워크 오버헤드 고려 시 더 오래 걸림).
> - Snowball Edge 3대를 주문하면 (각 80TB), 물리적으로 데이터를 복사하고 반송하여 약 **1주일** 만에 완료할 수 있습니다.
> - 네트워크 대역폭도 소모하지 않으므로 기존 업무에 영향이 없습니다.
>
> **핵심 포인트**: 일반적으로 **10TB 이상**의 데이터를 전송할 때 Snow Family가 네트워크 전송보다 빠르고 경제적입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "Storage Gateway는 마이그레이션 도구가 아닙니다."
>
> Storage Gateway는 온프레미스와 클라우드를 **지속적으로 연결**하는 하이브리드 서비스입니다. 데이터를 한 번에 대량으로 옮기는 "마이그레이션"에는 Snow Family가 더 적합합니다. 두 서비스의 용도를 혼동하지 마세요.

> ⚠️ **주의**: "인터넷이 빠르면 Snow Family가 필요 없다?"
>
> 속도만이 아니라 **비용, 보안, 네트워크 대역폭 점유** 등도 고려해야 합니다. 200TB를 인터넷으로 전송하면 네트워크 비용과 대역폭 점유로 인해 기존 업무에 영향을 줄 수 있습니다.

### Section 요약 (Section Summary)

- **AWS Storage Gateway**는 온프레미스 환경과 AWS 클라우드 스토리지를 연결하는 하이브리드 서비스입니다.
- Storage Gateway의 주요 유형: S3 File Gateway, FSx File Gateway, Volume Gateway, Tape Gateway
- **Snow Family**는 대량 데이터를 물리적으로 전송하는 디바이스 서비스입니다 (Snowcone, Snowball Edge, Snowmobile).
- 10TB 이상의 데이터 전송에는 Snow Family가 네트워크 전송보다 빠르고 경제적일 수 있습니다.
- Storage Gateway는 "지속적 연결", Snow Family는 "일회성 대량 전송"으로 구분하여 기억하세요.

## 🧪 통합 실습: 나의 첫 번째 S3 데이터 관리 시스템 구축

> **시나리오**: 당신은 소규모 스타트업의 첫 번째 클라우드 담당자입니다. 회사에서 사용하는 문서와 이미지를 체계적으로 관리하고, 비용을 최적화하며, 보안을 설정하는 임무를 맡았습니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | S3 버킷을 생성하고, 폴더 구조를 설계하고, 보안 설정과 수명 주기 정책을 적용하는 전체 과정을 수행합니다 |
| **사용 서비스** | Amazon S3 |
| **활용 개념** | Section 6.2 (버킷, 객체), Section 6.3 (스토리지 클래스, 수명 주기), Section 6.4 (보안, 접근 제어) |
| **선수 실습** | Mini Lab 6.2에서 생성한 버킷을 사용합니다 (없으면 새로 생성) |
| **예상 소요 시간** | 20분 |
| **비용** | Free Tier 범위 내 |

### 아키텍처 다이어그램

```
[S3 버킷: my-company-storage-xxx]
├── documents/          ← 업무 문서 (Standard)
│   ├── reports/
│   └── contracts/
├── images/             ← 이미지 파일 (Standard)
│   └── logos/
└── archives/           ← 오래된 파일 (수명 주기 → Standard-IA → Glacier)

[보안 설정]
├── 퍼블릭 접근 차단: 활성화 (기본값 유지)
├── 버전 관리: 활성화
└── 서버 측 암호화: SSE-S3 (기본 적용)

[비용 최적화]
└── 수명 주기 정책: 30일 후 Standard-IA, 90일 후 Glacier
```

### 실습 단계

> **Phase 1: 버킷 확인 및 폴더 구조 생성** (활용 개념: Section 6.2)
>
> **Step 1**: S3 콘솔에서 Mini Lab 6.2에서 만든 버킷을 클릭합니다. (없다면 Mini Lab 6.2의 Step 1~4를 따라 새로 생성합니다.)
>
> **Step 2**: **폴더 만들기(Create folder)** 버튼을 클릭합니다.
> - **폴더 이름**: `documents`
> - **서버 측 암호화**: 기본값 유지
> - **폴더 만들기(Create folder)** 를 클릭합니다.
>
> **Step 3**: 같은 방법으로 `images`와 `archives` 폴더도 생성합니다.
>
> **Step 4**: `documents` 폴더를 클릭하여 들어간 후, 그 안에 `reports`와 `contracts` 하위 폴더를 생성합니다.
>
> **Step 5**: 각 폴더에 테스트 파일을 1개씩 업로드합니다 (아무 텍스트 파일이나 이미지 파일).
>
> **✅ 체크포인트**: 버킷 내에 `documents/`, `images/`, `archives/` 폴더가 보이고, `documents/` 안에 `reports/`와 `contracts/` 하위 폴더가 있으며, 각 폴더에 파일이 1개씩 업로드되어 있어야 합니다.

> **Phase 2: 버전 관리 활성화** (활용 개념: Section 6.2)
>
> **Step 1**: 버킷의 최상위 화면에서 **속성(Properties)** 탭을 클릭합니다.
>
> **Step 2**: **버킷 버전 관리(Bucket Versioning)** 섹션에서 **편집(Edit)** 을 클릭합니다.
>
> **Step 3**: **활성화(Enable)** 를 선택하고, **변경 사항 저장(Save changes)** 을 클릭합니다.
>
> **Step 4**: 버전 관리가 동작하는지 확인합니다:
> - `documents/reports/` 폴더에 이미 업로드한 파일과 **같은 이름**의 파일을 다시 업로드합니다 (내용이 다른 파일).
> - 해당 파일을 클릭하고 **버전(Versions)** 탭을 확인합니다.
>
> **✅ 체크포인트**: 버전(Versions) 탭에 같은 파일이 2개의 버전(Version ID가 다른)으로 표시되면 성공입니다.

> **Phase 3: 수명 주기 정책 설정** (활용 개념: Section 6.3)
>
> **Step 1**: **관리(Management)** 탭을 클릭합니다.
>
> **Step 2**: Mini Lab 6.3에서 이미 수명 주기 규칙을 만들었다면 이를 확인합니다. 없다면 **수명 주기 규칙 생성(Create lifecycle rule)** 을 클릭합니다.
>
> **Step 3**: 새로운 규칙을 추가합니다 (또는 기존 규칙을 수정합니다):
> - **규칙 이름**: `archive-old-data`
> - **규칙 범위**: "하나 이상의 필터를 사용하여 이 규칙의 범위 제한"을 선택
> - **접두사(Prefix)**: `archives/`
> - **작업**: "스토리지 클래스 간에 객체의 현재 버전 전환"을 체크
> - **전환 1**: Standard-IA → 30일 후
> - **전환 추가**를 클릭하여 **전환 2**: Glacier Flexible Retrieval → 90일 후
>
> **Step 4**: **규칙 생성(Create rule)** 을 클릭합니다.
>
> **✅ 체크포인트**: 관리 탭에 `archive-old-data` 규칙이 활성화 상태로 표시되고, `archives/` 접두사에 대해 30일 후 Standard-IA, 90일 후 Glacier로 전환하는 설정이 확인되면 성공입니다.

> **Phase 4: 보안 설정 확인** (활용 개념: Section 6.4)
>
> **Step 1**: **권한(Permissions)** 탭을 클릭합니다.
>
> **Step 2**: **퍼블릭 액세스 차단(Block public access)** 섹션을 확인합니다.
> - 4개 항목이 모두 **"켜짐(On)"** 으로 표시되어 있는지 확인합니다.
> - 버킷 옆에 "공개적으로 액세스 가능" 경고 배지가 **없어야** 합니다.
>
> **Step 3**: **속성(Properties)** 탭에서 **기본 암호화(Default encryption)** 섹션을 확인합니다.
> - **서버 측 암호화**: "Amazon S3 관리형 키(SSE-S3)"가 설정되어 있는지 확인합니다.
>
> **✅ 체크포인트**: 퍼블릭 접근이 완전히 차단되어 있고, 기본 암호화가 SSE-S3로 설정되어 있으면 성공입니다.

### 트러블슈팅 (Troubleshooting)

> **Q: 버킷 생성 시 "Bucket name already exists" 오류가 발생합니다.**
> A: S3 버킷 이름은 전 세계에서 고유해야 합니다. 이름 뒤에 본인 이름, 날짜, 또는 임의의 숫자를 추가하세요. 예: `my-company-storage-kim-20260407`

> **Q: 수명 주기 규칙에서 "Standard-IA requires a minimum 128KB object size" 경고가 나옵니다.**
> A: Standard-IA는 128KB 미만의 객체도 128KB로 과금합니다. 이 경고는 정보 제공용이며, 규칙 생성에는 영향을 주지 않습니다. 실제 운영 시에는 작은 파일이 많으면 Standard를 유지하는 것이 경제적입니다.

> **Q: 버킷 삭제 시 "Bucket is not empty" 오류가 발생합니다.**
> A: 버전 관리가 활성화된 버킷에서는 "현재 버전"을 삭제해도 이전 버전과 삭제 마커가 남아 있습니다. "버전 표시"를 켜고 모든 버전을 삭제하거나, **빈 버킷(Empty bucket)** 기능을 사용하세요.

> **Practice 6.1**: 다음 중 Amazon S3의 특성으로 **올바른** 것은?
>
> A) S3는 블록 스토리지 서비스이다

> **해설**: S3는 **객체** 스토리지 서비스(A 오류)이며, 버킷 이름은 **전 세계에서 고유**해야 합니다(B 오류). 총 저장 용량에 제한이 없지만(C 정답), 개별 객체의 최대 크기는 **5TB**입니다(D 오류).

> **Practice 6.2**: 1년에 1~2번만 접근하는 규정 준수용 문서를 가장 경제적으로 저장할 수 있는 S3 스토리지 클래스는?

> A) S3 Standard
> B) S3 Standard-IA
> C) S3 Glacier Flexible Retrieval
> D) S3 One Zone-IA

> B) EBS 스냅샷은 증분식으로 저장된다
> C) HDD 볼륨(st1, sc1)을 부트 볼륨으로 사용할 수 있다

> </details>

> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 데이터의 접근 패턴, 성능 요구사항, 공유 필요성, 비용 요건을 분석합니다.
>
> **풀이 과정**:
>
> 1. **운영체제와 애플리케이션 코드** → **Amazon EBS (gp3)**
>    - 이유: 운영체제 설치와 애플리케이션 실행에는 빠른 읽기/쓰기가 필요한 블록 스토리지가 적합합니다. gp3는 범용 워크로드에 적합하고 비용 효율적입니다.
>

>    - 이유: 대량의 비정형 데이터(이미지)이므로 객체 스토리지가 적합합니다. 매일 수천 건 접근하므로 빈번한 접근에 적합한 S3 Standard를 선택합니다. HTTP/HTTPS로 직접 접근 가능하여 웹 서비스에 적합합니다.
>
> 3. **3년 전 활동 로그** → **S3 Glacier Flexible Retrieval 또는 Glacier Deep Archive**
>    - 이유: 법적으로 보관은 필요하지만 거의 접근하지 않으므로, 가장 저렴한 아카이브 클래스가 적합합니다. 아직 2년 더 보관해야 하고 급히 꺼낼 일이 거의 없으므로 Glacier Deep Archive가 가장 경제적입니다. 수명 주기 정책으로 5년 후 자동 삭제를 설정할 수 있습니다.
>
> 4. **10대 서버의 공유 설정 파일** → **Amazon EFS**
>    - 이유: 여러 Linux 서버가 동시에 같은 파일에 접근해야 하므로 공유 파일 스토리지가 필요합니다. EFS는 다중 EC2 인스턴스의 동시 접근을 지원하며, Linux 환경에 적합합니다.
>
> **보충 설명**: 스토리지 선택의 핵심은 ① 데이터 유형(정형/비정형), ② 접근 빈도, ③ 성능 요구, ④ 공유 필요성, ⑤ 비용 제약을 종합적으로 고려하는 것입니다.
> </details>

> **Problem 6.2**: S3의 "11 9's" 내구성(99.999999999%)과 "4 9's" 가용성(99.99%)의 차이를 설명하고, 내구성이 높더라도 가용성이 100%가 아닌 이유를 논리적으로 설명하세요. 또한, 비즈니스 관점에서 내구성과 가용성 각각이 중요한 실제 시나리오를 하나씩 제시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 내구성은 "데이터가 손실되지 않을 확률", 가용성은 "데이터에 접근할 수 있는 시간 비율"입니다. 두 개념은 서로 다른 차원의 보장입니다.
>
> **풀이 전략**: 먼저 두 개념을 명확히 구분한 후, 차이가 발생하는 이유를 설명합니다.
>
> **상세 풀이**:
>
> **내구성 vs 가용성의 차이**:
> - **내구성(Durability)**: 저장한 데이터가 손실·손상되지 않을 확률. 11 9's란 1,000만 개의 객체를 저장하면 평균 1만 년에 1개를 잃을 수 있다는 의미.
> - **가용성(Availability)**: 요청 시 데이터에 접근할 수 있는 시간의 비율. 4 9's(99.99%)란 1년 중 약 52분의 접근 불가 시간이 있을 수 있다는 의미.
>
> **내구성이 높아도 가용성이 100%가 아닌 이유**:
> - 데이터 자체는 안전하게 보관되어 있지만, 접근 경로(네트워크, S3 서비스 자체, DNS 등)에 일시적 장애가 발생할 수 있습니다.
> - 비유: 은행 금고에 돈이 안전하게 있지만(내구성), 은행 문이 잠시 닫혀 있어 꺼내지 못하는 상황(가용성)과 같습니다.
> - S3는 내부적으로 데이터를 최소 3개 AZ에 복제하여 내구성을 보장하지만, 서비스 유지보수, 네트워크 장애, 일시적 과부하 등으로 짧은 시간 접근이 불가능할 수 있습니다.
>
> **비즈니스 시나리오**:
> - **내구성이 특히 중요한 시나리오**: 병원의 환자 의료 기록. 데이터가 손실되면 환자 안전에 직접적인 영향을 미치며, 법적 문제도 발생합니다. 데이터가 잠시 접근 불가여도(가용성) 복구 가능하지만, 데이터 자체가 사라지면(내구성) 되돌릴 수 없습니다.
> - **가용성이 특히 중요한 시나리오**: 실시간 주식 거래 플랫폼. 데이터가 존재하더라도 접근할 수 없는 52분 동안 수십억 원의 거래 기회를 놓칠 수 있습니다. 이 경우 99.99%보다 더 높은 가용성이 필요하며, 다중 리전 아키텍처를 고려해야 합니다.
>
> **확장 생각**: S3 One Zone-IA는 내구성이 11 9's이지만 단일 AZ에만 저장되므로, 해당 AZ의 물리적 파괴 시 데이터 손실이 가능합니다. 이는 내구성 수치가 "정상 운영 조건"에서의 보장이며, 극단적 재해는 별도로 고려해야 함을 보여줍니다.
> </details>
