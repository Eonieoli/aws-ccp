## 5.1 Amazon EC2 개요 — 가상 서버의 개념, 인스턴스 생명주기

> 💡 **한 줄 요약**: Amazon EC2는 클라우드에서 필요할 때 바로 생성하고, 필요 없으면 바로 없앨 수 있는 가상 컴퓨터(서버)입니다.

### 직관적 이해

컴퓨터가 한 대 필요하다고 상상해 봅시다. 전통적인 방식이라면 컴퓨터를 구매하고, 설치 장소를 마련하고, 전원과 네트워크를 연결하고, 운영 체제를 설치해야 합니다. 이 과정은 몇 주에서 몇 달이 걸릴 수 있습니다.

이것은 마치 **집을 사는 것**과 비슷합니다. 큰 초기 비용이 들고, 유지보수도 직접 해야 하며, 더 이상 필요 없어도 쉽게 처분하기 어렵습니다.

반면 Amazon EC2는 **호텔 방을 예약하는 것**과 같습니다. 필요할 때 체크인하고, 머무는 동안만 비용을 내고, 떠나면 더 이상 비용이 들지 않습니다. 방의 크기(싱글룸, 스위트룸)도 필요에 따라 고를 수 있고, 여러 방이 필요하면 추가로 예약하면 됩니다.

> 🔗 **연결**: Chapter 1에서 배운 클라우드 컴퓨팅의 핵심 이점 — *온디맨드 셀프서비스*, *신속한 탄력성*, *종량제 과금* — 을 기술적으로 구현한 대표 서비스가 바로 EC2입니다. Chapter 2에서 배운 서비스 모델 중 **IaaS(Infrastructure as a Service)**의 대표적인 예이기도 합니다.

**왜 EC2가 필요한가?**

기업이 웹사이트를 운영하려면 서버가 필요합니다. 새 프로젝트를 시작할 때마다 물리 서버를 구매하고 설치하는 것은 비용과 시간 면에서 비효율적입니다. EC2를 사용하면 몇 분 안에 원하는 사양의 서버를 시작하고, 프로젝트가 끝나면 즉시 종료할 수 있습니다. 이것이 클라우드 컴퓨팅이 말하는 **민첩성(Agility)**의 실제 모습입니다.

### 핵심 개념 설명

> 📝 **Amazon EC2 (Elastic Compute Cloud)**: AWS에서 제공하는 가상 서버 서비스입니다. 사용자가 원하는 운영 체제, CPU, 메모리, 스토리지 구성을 선택하여 가상 컴퓨터를 생성하고 사용할 수 있습니다. "Elastic"은 필요에 따라 자유롭게 늘리고 줄일 수 있다는 의미이고, "Compute"는 연산 능력, "Cloud"는 인터넷을 통해 접근한다는 의미입니다.

> 📝 **인스턴스 (Instance)**: EC2에서 실행 중인 하나의 가상 서버를 인스턴스라고 부릅니다. 물리 서버 한 대가 하나의 컴퓨터라면, 인스턴스는 그 물리 서버 위에서 돌아가는 가상의 컴퓨터입니다. 하나의 물리 서버에서 여러 개의 인스턴스가 독립적으로 실행될 수 있습니다.

**EC2의 핵심 특징**:

- **몇 분 만에 시작**: 클릭 몇 번으로 전 세계 어디에서든 서버를 시작할 수 있습니다
- **다양한 구성 선택**: 운영 체제(Linux, Windows 등), CPU 수, 메모리 크기, 스토리지 용량 등을 자유롭게 선택합니다
- **종량제 과금**: 인스턴스가 실행 중인 시간(초 단위)만큼만 비용을 지불합니다
- **탄력적 확장**: 트래픽이 많아지면 인스턴스를 추가하고, 줄어들면 인스턴스를 제거할 수 있습니다

**인스턴스 생명주기 (Instance Lifecycle)**

EC2 인스턴스는 다음과 같은 상태를 거칩니다:

```
[시작 요청] → pending → running ←→ stopping → stopped
                  │                        │
                  │                        ▼
                  ├──→ shutting-down → terminated
                  │
                  └──→ rebooting → running
```

| 상태 | 설명 | 과금 여부 |
|------|------|-----------|
| **pending** | 인스턴스가 시작 준비 중 | 과금 없음 |
| **running** | 인스턴스가 실행 중이며 사용 가능 | **과금 발생** |
| **stopping** | 인스턴스가 중지되는 중 | 과금 없음 |
| **stopped** | 인스턴스가 중지됨 (데이터 보존, 재시작 가능) | 인스턴스 과금 없음, **EBS 볼륨 과금은 발생** |
| **shutting-down** | 인스턴스가 종료되는 중 | 과금 없음 |
| **terminated** | 인스턴스가 완전히 삭제됨 (복구 불가) | 과금 없음 |
| **rebooting** | 인스턴스가 재부팅 중 | **과금 발생** (running 상태 유지) |

> ⚠️ **주의**: **중지(Stop)**와 **종료(Terminate)**는 완전히 다른 동작입니다!
>
> - **중지(Stop)**: 컴퓨터 전원을 끄는 것과 같습니다. 데이터가 보존되고, 나중에 다시 시작(Start)할 수 있습니다. 단, 연결된 EBS 스토리지 비용은 계속 발생합니다.
> - **종료(Terminate)**: 컴퓨터를 완전히 폐기하는 것입니다. 인스턴스와 기본 스토리지가 삭제되며, 복구할 수 없습니다.
>
> 실습 후에는 반드시 **종료(Terminate)**하여 불필요한 비용을 방지하세요.

### 상세 예시

> 🔍 **예시 5.1.1: 스타트업의 웹 서비스 런칭**
>
> **상황**: 3인 스타트업이 새로운 웹 서비스를 개발했습니다. 아직 사용자 수를 예측할 수 없어 서버 투자에 신중합니다.
>
> **풀이/설명**: 전통적 방식이라면 서버를 구매하고 데이터센터에 설치해야 합니다. 예상 사용자가 100명인지 10만 명인지 모르니, 적게 사면 서비스가 느려지고 많이 사면 돈을 낭비합니다.
> EC2를 사용하면 작은 인스턴스 하나로 시작하여, 사용자가 늘어남에 따라 인스턴스를 추가하거나 더 큰 인스턴스로 교체할 수 있습니다. 사용자가 줄면 다시 축소합니다.
>
> **핵심 포인트**: EC2는 "필요한 만큼만, 필요한 때에" 컴퓨팅 자원을 사용할 수 있게 해줍니다. Chapter 1에서 배운 "CapEx에서 OpEx로의 전환"이 실현되는 대표적 사례입니다.

> 🔍 **예시 5.1.2: 인스턴스 생명주기 — 이벤트 기간 서버 운영**
>
> **상황**: 온라인 쇼핑몰이 블랙 프라이데이 세일을 3일간 진행합니다. 평소보다 10배 많은 트래픽이 예상됩니다.
>
> **풀이/설명**:
> 1. 세일 시작 전: 추가 EC2 인스턴스 9대를 **시작(Start)** → `pending` → `running`
> 2. 세일 기간: 10대의 인스턴스가 `running` 상태로 트래픽을 처리
> 3. 세일 종료 후: 추가 9대를 **종료(Terminate)** → `shutting-down` → `terminated`
> 4. 결과: 3일간의 추가 인스턴스 비용만 발생. 물리 서버 9대를 구매했다면 수천만 원의 비용이 들었을 것
>
> **핵심 포인트**: EC2의 생명주기를 활용하면 일시적인 수요 급증에도 효율적으로 대응할 수 있습니다.

### Section 요약

- Amazon EC2는 AWS의 핵심 **IaaS** 서비스로, 클라우드에서 가상 서버(인스턴스)를 생성·운영할 수 있게 해줍니다
- 인스턴스는 몇 분 만에 시작할 수 있으며, 사용한 시간만큼만 비용을 지불합니다
- 인스턴스 생명주기에서 **중지(Stop)**와 **종료(Terminate)**의 차이를 반드시 이해해야 합니다
- EC2는 클라우드의 핵심 이점인 탄력성, 종량제, 민첩성을 기술적으로 구현한 서비스입니다

## 5.2 EC2 인스턴스 유형과 선택 — 범용, 컴퓨팅 최적화, 메모리 최적화 등 패밀리 분류

> 💡 **한 줄 요약**: EC2 인스턴스 유형은 용도에 따라 최적화된 "컴퓨터 사양 패키지"이며, 워크로드에 맞는 유형을 선택하는 것이 비용과 성능의 핵심입니다.

### 직관적 이해

자동차를 생각해 봅시다. 모든 자동차가 같은 용도로 쓰이지 않습니다:
- **세단**: 일상 출퇴근에 적합한 범용 차량
- **스포츠카**: 속도가 중요한 상황에 최적화
- **SUV/트럭**: 무거운 짐을 실어야 할 때 적합
- **미니밴**: 많은 인원을 태워야 할 때 최적

EC2 인스턴스 유형도 마찬가지입니다. 모든 컴퓨터 작업이 같은 자원을 필요로 하지 않습니다. 웹 서버는 CPU와 메모리가 균형 잡힌 구성이 좋고, 영상 편집은 강력한 CPU가 필요하며, 대용량 데이터 분석은 메모리가 넉넉해야 합니다.

**왜 인스턴스 유형이 다양해야 하는가?**

만약 모든 인스턴스가 동일한 사양이라면, 메모리가 많이 필요한 작업을 위해 CPU도 과도하게 큰 인스턴스를 사용해야 합니다. 이는 불필요한 비용 낭비로 이어집니다. 다양한 인스턴스 유형이 존재하는 이유는 **필요한 자원만 정확히 선택하여 비용을 최적화**하기 위해서입니다.

### 핵심 개념 설명

> 📝 **인스턴스 유형 (Instance Type)**: EC2 인스턴스의 CPU, 메모리, 스토리지, 네트워크 성능 조합을 정의하는 하드웨어 사양 템플릿입니다. 예를 들어 `t3.micro`는 2개의 vCPU와 1GiB 메모리를 제공합니다.

> 📝 **인스턴스 패밀리 (Instance Family)**: 유사한 용도로 설계된 인스턴스 유형들의 그룹입니다. 같은 패밀리 내에서 크기(size)만 다른 인스턴스들이 존재합니다.

**인스턴스 유형 명명 규칙**

EC2 인스턴스 유형 이름은 일정한 패턴을 따릅니다:

```
예: m5.xlarge

m    →  패밀리 (범용 - General Purpose)
5    →  세대 (5세대 — 숫자가 클수록 최신)
.    →  구분자
xlarge → 크기 (nano < micro < small < medium < large < xlarge < 2xlarge ...)
```

**주요 인스턴스 패밀리**

| 패밀리 | 대표 접두사 | 특징 | 비유 | 적합한 워크로드 |
|--------|-------------|------|------|-----------------|
| **범용 (General Purpose)** | T, M | CPU, 메모리, 네트워크가 균형 잡힌 구성 | 세단 — 일상적 용도에 두루 적합 | 웹 서버, 소규모 DB, 개발 환경 |
| **컴퓨팅 최적화 (Compute Optimized)** | C | 높은 CPU 성능에 특화 | 스포츠카 — 속도(연산)에 집중 | 과학 계산, 게임 서버, 미디어 인코딩 |
| **메모리 최적화 (Memory Optimized)** | R, X | 대용량 메모리에 특화 | 대형 트럭 — 많은 짐(데이터)을 한꺼번에 | 대규모 인메모리 DB, 빅데이터 분석 |
| **스토리지 최적화 (Storage Optimized)** | I, D, H | 높은 디스크 I/O에 특화 | 물류 창고 — 저장·입출력에 최적화 | 데이터 웨어하우스, 로그 처리 |
| **가속 컴퓨팅 (Accelerated Computing)** | P, G, Inf | GPU 또는 전용 하드웨어 탑재 | 레이싱카 + 터보 — 특수 가속 장치 장착 | 머신러닝 훈련, 그래픽 렌더링 |

> 💬 **참고**: CCP 시험에서는 각 패밀리의 구체적인 인스턴스 이름(예: r5.2xlarge)을 암기할 필요는 없습니다. **어떤 패밀리가 어떤 용도에 적합한지** 구분할 수 있으면 충분합니다.

**T 패밀리와 버스트(Burst) 성능**

범용 패밀리 중 **T 시리즈** (t2, t3, t3a 등)는 특별한 과금 방식을 가지고 있습니다:

> 📝 **버스트 가능 인스턴스 (Burstable Instance)**: 평소에는 기본 수준의 CPU 성능을 제공하다가, 순간적으로 높은 CPU가 필요할 때 일시적으로 성능을 높일 수 있는(버스트) 인스턴스입니다. **CPU 크레딧**이라는 개념으로 관리됩니다.

쉽게 말해, T 인스턴스는 "절약형 자동차"입니다. 평소에는 연비 좋게 달리다가, 필요할 때 잠깐 가속할 수 있습니다. 웹 서버처럼 평소에는 트래픽이 적다가 가끔 몰리는 워크로드에 매우 경제적입니다.

> 💬 **참고**: `t2.micro`와 `t3.micro`는 **AWS Free Tier** 대상 인스턴스 유형입니다. 신규 계정 생성 후 12개월 동안 월 750시간까지 무료로 사용할 수 있어, 학습용으로 가장 적합합니다.

### 상세 예시

> 🔍 **예시 5.2.1: 인스턴스 유형 이름 읽기**
>
> **상황**: 다음 인스턴스 유형 이름의 의미를 해석해 봅시다: `c5.2xlarge`
>
> **풀이/설명**:
> - `c` → **컴퓨팅 최적화** 패밀리 (CPU 성능에 특화)
> - `5` → 5세대 (최신에 가까운 세대)
> - `2xlarge` → xlarge의 2배 크기 (vCPU 8개, 메모리 16GiB)
>
> **핵심 포인트**: 인스턴스 이름만 보고도 대략적인 용도와 크기를 파악할 수 있습니다. 'c'로 시작하면 연산 집약적 작업에 적합한 인스턴스입니다.

> 🔍 **예시 5.2.2: 워크로드에 맞는 인스턴스 선택**
>
> **상황**: 당신은 세 가지 서비스를 AWS에서 운영해야 합니다:
> - (A) 하루 방문자 500명인 회사 홈페이지
> - (B) 4K 영상을 실시간으로 인코딩하는 미디어 서버
> - (C) 100GB 데이터셋을 메모리에 올려 분석하는 데이터 분석 서버
>
> **풀이/설명**:
> - (A) 회사 홈페이지 → **T 패밀리 (범용, 버스트)**: 트래픽이 적고, 가끔 몰릴 때만 성능이 필요 → `t3.small` 정도면 충분
> - (B) 영상 인코딩 → **C 패밀리 (컴퓨팅 최적화)**: CPU를 지속적으로 많이 사용 → `c5.xlarge` 이상
> - (C) 데이터 분석 → **R 패밀리 (메모리 최적화)**: 대량의 데이터를 메모리에 유지 → `r5.4xlarge` 이상
>
> **핵심 포인트**: 워크로드의 특성(CPU 집약 vs 메모리 집약 vs 균형)을 파악하면, 적합한 인스턴스 패밀리를 선택할 수 있습니다. 잘못된 유형을 선택하면 성능 부족 또는 비용 낭비가 발생합니다.

> ⚠️ **주의**: 인스턴스 유형 선택에서 흔한 실수
>
> - **무조건 큰 인스턴스 선택**: "혹시 모르니 크게" 고르면 비용이 불필요하게 증가합니다. 작게 시작하고 모니터링 후 필요하면 키우는 것이 클라우드의 모범 사례입니다.
> - **패밀리 무시**: 메모리가 많이 필요한데 범용(M) 패밀리의 큰 크기를 선택하면, CPU 비용까지 불필요하게 지불하게 됩니다. 메모리 최적화(R) 패밀리를 선택하는 것이 더 경제적입니다.
> - **세대 무시**: 같은 패밀리라도 최신 세대가 더 좋은 성능과 낮은 가격을 제공합니다. 예를 들어 `m5`는 `m4`보다 가격 대비 성능이 더 좋습니다.

### Section 요약

- EC2 인스턴스 유형은 CPU, 메모리, 스토리지, 네트워크 성능의 조합으로 정의됩니다
- 주요 패밀리: **범용(T, M)**, **컴퓨팅 최적화(C)**, **메모리 최적화(R, X)**, **스토리지 최적화(I, D)**, **가속 컴퓨팅(P, G)**
- 인스턴스 이름은 `패밀리 + 세대 + 크기` 규칙을 따릅니다 (예: `m5.xlarge`)
- Free Tier에서는 `t2.micro` 또는 `t3.micro`를 사용할 수 있습니다

## 5.3 AMI와 인스턴스 시작 — Amazon Machine Image의 개념과 인스턴스 구성

> 💡 **한 줄 요약**: AMI는 EC2 인스턴스를 생성하기 위한 "설계도(템플릿)"로, 운영 체제와 소프트웨어 구성이 미리 포함되어 있습니다.

### 직관적 이해

새 스마트폰을 구매하면 공장 초기 상태로 옵니다. 여기에 앱을 설치하고, 설정을 변경하고, 데이터를 옮기는 데 시간이 걸립니다. 만약 동일한 설정의 폰을 10대 준비해야 한다면? 한 대씩 수동 설정하는 것은 매우 비효율적입니다.

AMI는 **스마트폰의 백업 이미지**와 같습니다. 한 번 완벽하게 설정한 후 "스냅샷(사진)"을 찍어두면, 그 이미지로 동일한 설정의 스마트폰을 몇 대든 빠르게 복제할 수 있습니다.

**왜 AMI가 필요한가?**

서버를 매번 처음부터 설정하는 것은 시간 낭비이고 실수의 여지가 큽니다. AMI를 사용하면 한 번 검증된 서버 구성을 재사용하여, 동일한 환경의 서버를 빠르고 일관성 있게 배포할 수 있습니다.

### 핵심 개념 설명

> 📝 **AMI (Amazon Machine Image)**: EC2 인스턴스를 시작하는 데 필요한 모든 정보를 담은 템플릿입니다. 운영 체제, 애플리케이션 서버, 애플리케이션 코드, 설정 등이 포함됩니다. 하나의 AMI로 동일한 구성의 인스턴스를 여러 개 생성할 수 있습니다.

**AMI에 포함되는 정보**:

1. **루트 볼륨 템플릿**: 운영 체제(Linux, Windows 등)와 설치된 소프트웨어
2. **시작 권한**: 어떤 AWS 계정이 이 AMI를 사용할 수 있는지
3. **블록 디바이스 매핑**: 인스턴스에 연결할 스토리지 볼륨의 구성 정보

**AMI의 종류**:

| 종류 | 설명 | 예시 |
|------|------|------|
| **AWS 제공 AMI** | AWS가 공식적으로 관리하는 기본 이미지 | Amazon Linux 2023, Ubuntu, Windows Server |
| **AWS Marketplace AMI** | 서드파티 벤더가 소프트웨어를 미리 설치하여 판매하는 이미지 | WordPress가 설치된 AMI, 데이터베이스가 구성된 AMI |
| **커뮤니티 AMI** | 다른 AWS 사용자가 공유한 이미지 | 다양한 오픈소스 프로젝트 AMI |
| **사용자 정의(Custom) AMI** | 사용자가 직접 구성하여 만든 이미지 | 회사 표준 서버 환경이 설정된 AMI |

**인스턴스 시작 시 필요한 구성 요소**:

EC2 인스턴스를 시작할 때, 사용자는 다음 항목들을 결정합니다:

1. **AMI 선택**: 어떤 운영 체제와 소프트웨어를 사용할 것인가
2. **인스턴스 유형 선택**: 어떤 하드웨어 사양을 사용할 것인가 (Section 5.2에서 학습)
3. **네트워크 설정**: 어느 네트워크(VPC/서브넷)에 배치할 것인가
4. **스토리지 구성**: 얼마나 큰 디스크를 연결할 것인가
5. **보안 그룹**: 어떤 네트워크 트래픽을 허용할 것인가
6. **키 페어**: SSH/RDP 접속을 위한 인증 키

> 💬 **참고**: 네트워크(VPC/서브넷)와 보안 그룹은 Chapter 7에서 자세히 다룹니다. 지금은 "인스턴스를 시작할 때 네트워크와 보안도 함께 설정해야 한다"는 점만 이해하면 됩니다.

> 📝 **키 페어 (Key Pair)**: EC2 인스턴스에 안전하게 접속하기 위한 암호화 키 쌍입니다. **퍼블릭 키**는 인스턴스에 저장되고, **프라이빗 키**는 사용자가 보관합니다. 집 열쇠와 비슷하게, 프라이빗 키를 가진 사람만 인스턴스에 접속할 수 있습니다.

### 상세 예시

> 🔍 **예시 5.3.1: AMI를 활용한 환경 복제**
>
> **상황**: 개발팀이 동일한 환경의 테스트 서버 5대가 필요합니다. 서버에는 Python 3.11, Node.js 18, 그리고 팀이 사용하는 모니터링 에이전트가 설치되어 있어야 합니다.
>
> **풀이/설명**:
> 1. EC2 인스턴스 1대를 시작하고 필요한 소프트웨어를 모두 설치합니다
> 2. 이 인스턴스로부터 **사용자 정의 AMI**를 생성합니다
> 3. 이 AMI를 사용하여 동일한 구성의 인스턴스 4대를 추가로 시작합니다
> 4. 총 5대의 인스턴스가 완전히 동일한 환경으로 준비됩니다
>
> **핵심 포인트**: AMI를 사용하면 수동 설정의 반복을 없애고, 환경의 일관성을 보장할 수 있습니다.

> 🔍 **예시 5.3.2: AWS Marketplace AMI로 빠른 시작**
>
> **상황**: 소규모 사업자가 WordPress 블로그를 빠르게 시작하고 싶지만, 서버 설정에 대한 기술 지식이 부족합니다.
>
> **풀이/설명**: AWS Marketplace에서 "WordPress" AMI를 검색하면, WordPress + 웹 서버(Apache/Nginx) + 데이터베이스(MySQL)가 모두 미리 설치되고 구성된 AMI를 찾을 수 있습니다. 이 AMI로 인스턴스를 시작하면, 별도의 소프트웨어 설치 없이 바로 WordPress 사이트를 사용할 수 있습니다.
>
> **핵심 포인트**: Marketplace AMI를 활용하면 전문 지식 없이도 복잡한 소프트웨어 스택을 빠르게 구성할 수 있습니다. 일부 Marketplace AMI는 추가 라이선스 비용이 발생할 수 있으니 가격을 확인하세요.

### Mini Lab

> 🧪 **Mini Lab 5.3: EC2 인스턴스 시작하기**
>
> **목표**: AWS 콘솔에서 EC2 인스턴스를 직접 시작하고, 실행 상태를 확인한 뒤, 종료하는 전체 과정을 경험합니다.
> **예상 소요 시간**: 15분
> **비용**: Free Tier 범위 내 (t2.micro 또는 t3.micro)
>
> **Step 1**: AWS Management Console에 로그인합니다. 화면 상단의 리전 선택에서 **아시아 태평양 (서울) ap-northeast-2**가 선택되어 있는지 확인합니다.
>
> **Step 2**: 상단 검색창에 `EC2`를 입력하고 **EC2** 서비스를 클릭합니다. EC2 대시보드가 표시됩니다.
>
> **Step 3**: 좌측 메뉴에서 **Instances (인스턴스)**를 클릭하고, **Launch instances (인스턴스 시작)** 버튼을 클릭합니다.
>
> **Step 4**: 인스턴스 시작 화면에서 다음을 설정합니다:
> - **Name (이름)**: `my-first-ec2` 입력
> - **Application and OS Images (AMI)**: **Amazon Linux 2023 AMI** 선택 (Free tier eligible 표시 확인)
> - **Instance type (인스턴스 유형)**: **t2.micro** 선택 (Free tier eligible 표시 확인)
> - **Key pair**: **Proceed without a key pair** 선택 (이번 실습에서는 인스턴스에 직접 접속하지 않으므로)
>
> **Step 5**: 나머지 설정은 기본값을 유지하고, 페이지 우측의 **Launch instance (인스턴스 시작)** 버튼을 클릭합니다.
>
> **Step 6**: "Successfully initiated launch of instance" 메시지가 표시됩니다. 인스턴스 ID 링크를 클릭하여 인스턴스 목록으로 이동합니다.
>
> **Step 7**: 인스턴스 상태가 `Pending` → `Running`으로 변경되는 것을 확인합니다 (1~2분 소요). 하단의 **Details (세부 정보)** 탭에서 인스턴스 유형, AMI ID, 가용 영역 등의 정보를 확인합니다.
>
> **확인**: 인스턴스 상태가 **Running (실행 중)**이고, 상태 검사가 **2/2 checks passed**로 표시되면 성공입니다.
>
> **🧹 정리**: 실습이 끝났으므로 인스턴스를 **반드시 종료**합니다.
> 1. 인스턴스를 선택(체크박스)합니다
> 2. 상단의 **Instance state (인스턴스 상태)** 드롭다운을 클릭합니다
> 3. **Terminate instance (인스턴스 종료)**를 선택합니다
> 4. 확인 팝업에서 **Terminate**를 클릭합니다
> 5. 인스턴스 상태가 `Shutting-down` → `Terminated`로 변경되는 것을 확인합니다

> ⚠️ **주의**: 키 페어를 생성한 경우, 프라이빗 키 파일(`.pem`)은 **다운로드 기회가 한 번뿐**입니다. 분실하면 해당 키로는 인스턴스에 접속할 수 없으니, 안전한 곳에 보관하세요. 이번 실습에서는 키 페어를 생성하지 않았으므로 이 주의사항은 향후 실습을 위해 기억해 두세요.

### Section 요약

- AMI는 EC2 인스턴스의 "설계도"로, 운영 체제와 소프트웨어 구성을 포함합니다
- AMI 종류: AWS 제공, Marketplace, 커뮤니티, 사용자 정의
- 인스턴스 시작 시 AMI, 인스턴스 유형, 네트워크, 스토리지, 보안 그룹, 키 페어를 설정합니다
- 사용자 정의 AMI를 생성하면 동일한 환경을 빠르고 일관성 있게 복제할 수 있습니다

## 5.4 EC2 구매 옵션 — 온디맨드, 예약, 스팟, Savings Plans, 전용 호스트 비교

> 💡 **한 줄 요약**: EC2는 같은 인스턴스라도 "어떻게 구매하느냐"에 따라 비용이 최대 90%까지 차이 나며, 워크로드 특성에 맞는 구매 옵션 선택이 비용 최적화의 핵심입니다.

### 직관적 이해

호텔 방 예약을 다시 떠올려 봅시다:

- **당일 예약 (Walk-in)**: 아무 때나 빈 방에 바로 들어갑니다. 가장 편리하지만 가장 비쌉니다.
- **연간 계약**: "1년간 매달 이 방을 쓸게요"라고 약속하면, 할인을 받을 수 있습니다.
- **마감 할인**: 체크아웃 직전 비어 있는 방을 70~90% 할인된 가격에 제공합니다. 대신 갑자기 다른 손님이 와서 정가를 내겠다고 하면, 방을 빼야 할 수도 있습니다.
- **건물 전체 임대**: 특수한 이유(보안, 규정 등)로 건물 전체를 독점 사용합니다.

EC2의 구매 옵션도 이와 같은 원리입니다. 약속의 강도와 유연성의 트레이드오프에 따라 비용이 달라집니다.

### 핵심 개념 설명

#### 1) 온디맨드 인스턴스 (On-Demand Instances)

> 📝 **온디맨드 인스턴스**: 약정 없이 사용한 만큼만 지불하는 기본 구매 방식입니다. 초 단위(Linux) 또는 시간 단위(Windows)로 과금됩니다.

- **장점**: 약정 없음, 언제든 시작/종료 가능, 가장 유연함
- **단점**: 가장 높은 단가
- **적합한 경우**: 예측 불가능한 워크로드, 단기 프로젝트, 처음 시작하는 애플리케이션, 테스트/개발 환경

#### 2) 예약 인스턴스 (Reserved Instances, RI)

> 📝 **예약 인스턴스**: 1년 또는 3년 기간 약정을 통해 온디맨드 대비 최대 약 72% 할인을 받는 구매 방식입니다. 특정 인스턴스 유형, 리전, 운영 체제를 지정하여 예약합니다.

- **약정 기간**: 1년 또는 3년 (3년이 더 큰 할인)
- **결제 옵션**:
  - **전체 선결제 (All Upfront)**: 가장 큰 할인
  - **부분 선결제 (Partial Upfront)**: 중간 수준의 할인
  - **선결제 없음 (No Upfront)**: 가장 작은 할인, 월별 지불
- **적합한 경우**: 24시간 365일 안정적으로 실행되는 서비스 (웹 서버, 데이터베이스 등)

> 🔗 **연결**: Chapter 1에서 배운 CapEx vs OpEx 개념과 연결하면, 예약 인스턴스의 전체 선결제 방식은 클라우드 환경에서의 일종의 "CapEx적 접근"이라고 볼 수 있습니다. 큰 금액을 미리 지불하는 대신 장기적으로 더 저렴하게 사용합니다.

#### 3) Savings Plans

> 📝 **Savings Plans**: 특정 인스턴스 유형이 아니라, **시간당 사용 금액**(예: 시간당 $10 이상 EC2를 사용하겠다)을 1년 또는 3년 약정하는 유연한 할인 모델입니다.

- **Compute Savings Plans**: EC2, Lambda, Fargate에 걸쳐 적용. 인스턴스 패밀리, 리전, OS 변경 가능. 가장 유연함
- **EC2 Instance Savings Plans**: 특정 리전의 특정 인스턴스 패밀리에 대해 적용. 크기와 OS는 변경 가능. 더 큰 할인
- **적합한 경우**: 장기적으로 일정 수준의 컴퓨팅을 사용하지만, 인스턴스 유형을 변경할 가능성이 있는 경우

**예약 인스턴스 vs Savings Plans**:

| 비교 항목 | 예약 인스턴스 | Savings Plans |
|-----------|--------------|---------------|
| 약정 대상 | 특정 인스턴스 유형 | 시간당 사용 금액 |
| 유연성 | 낮음 (유형 고정) | 높음 (유형 변경 가능) |
| 할인율 | 최대 ~72% | 최대 ~72% |
| 적용 범위 | EC2만 | EC2, Lambda, Fargate |

#### 4) 스팟 인스턴스 (Spot Instances)

> 📝 **스팟 인스턴스**: AWS가 보유한 여유 컴퓨팅 용량을 온디맨드 대비 최대 약 90% 할인된 가격에 사용하는 방식입니다. 단, AWS가 해당 용량을 다시 필요로 하면 **2분 전 알림 후 인스턴스가 회수(중단)**될 수 있습니다.

- **장점**: 가장 저렴한 가격 (최대 90% 할인)
- **단점**: 언제든 중단될 수 있음 (안정성 보장 없음)
- **적합한 경우**: 중단되어도 괜찮은 워크로드 — 배치 처리, 데이터 분석, 이미지 렌더링, CI/CD 빌드
- **부적합한 경우**: 데이터베이스 서버, 핵심 웹 서비스 등 중단이 치명적인 워크로드

#### 5) 전용 호스트 (Dedicated Hosts)

> 📝 **전용 호스트**: 사용자 전용의 물리적 EC2 서버입니다. 다른 AWS 고객과 물리 하드웨어를 공유하지 않습니다.

- **적합한 경우**:
  - 소프트웨어 라이선스 조건상 물리 서버 단위 라이선스가 필요한 경우 (예: Windows Server, Oracle DB 등의 BYOL — Bring Your Own License)
  - 규정 준수(Compliance) 요구사항으로 물리적 격리가 필요한 경우
- **비용**: 가장 비싼 옵션

#### 6) 전용 인스턴스 (Dedicated Instances)

- 전용 호스트와 유사하게 물리적 격리를 제공하지만, 특정 물리 서버를 지정할 수 없습니다
- 전용 호스트보다 저렴하며, BYOL이 필요하지 않은 경우에 적합합니다

### 구매 옵션 종합 비교

| 구매 옵션 | 할인율 | 약정 | 중단 위험 | 핵심 포인트 |
|-----------|--------|------|-----------|-------------|
| **온디맨드** | 0% (기준 가격) | 없음 | 없음 | 가장 유연, 가장 비쌈 |
| **예약 인스턴스** | 최대 ~72% | 1년/3년 | 없음 | 안정적 워크로드에 최적 |
| **Savings Plans** | 최대 ~72% | 1년/3년 | 없음 | 예약보다 유연한 약정 |
| **스팟** | 최대 ~90% | 없음 | **있음** | 중단 가능 워크로드에 최적 |
| **전용 호스트** | 온디맨드/예약 가능 | 없음/있음 | 없음 | 라이선스, 규정 준수용 |

### 상세 예시

> 🔍 **예시 5.4.1: 구매 옵션 조합 — 온라인 쇼핑몰**
>
> **상황**: 온라인 쇼핑몰의 인프라를 설계합니다. 평균적으로 웹 서버 4대가 필요하고, 세일 기간에는 최대 10대까지 필요합니다. 또한 매일 밤 상품 추천 알고리즘을 위한 데이터 분석 배치 작업을 수행합니다.
>
> **풀이/설명**:
> - **기본 4대** (24/7 상시 운영): **예약 인스턴스 또는 Savings Plans** → 안정적이고 예측 가능하므로 약정 할인 적용
> - **추가 6대** (세일 기간만): **온디맨드** → 언제 필요할지 정확히 예측하기 어렵고 단기간 사용
> - **데이터 분석 배치 작업**: **스팟 인스턴스** → 중단되면 처음부터 다시 돌리면 되므로, 최대 할인 활용
>
> **핵심 포인트**: 하나의 구매 옵션만 사용할 필요 없습니다. 워크로드 특성에 따라 여러 구매 옵션을 **조합**하는 것이 비용 최적화의 핵심입니다.

> 🔍 **예시 5.4.2: 스팟 인스턴스의 적합/부적합 판단**
>
> **상황**: 다음 중 스팟 인스턴스에 적합한 워크로드는?
> - (A) 회사의 메인 데이터베이스 서버
> - (B) 10만 장의 이미지 썸네일 생성 작업
> - (C) 실시간 주식 거래 시스템
> - (D) 머신러닝 모델 훈련 (체크포인트 저장 기능 포함)
>
> **풀이/설명**:
> - (A) ❌ 부적합 — DB 서버가 중단되면 서비스 전체가 마비됩니다
> - (B) ✅ 적합 — 중단되어도 남은 이미지만 다시 처리하면 됩니다
> - (C) ❌ 부적합 — 실시간 거래에서의 2분 중단은 큰 손실을 유발합니다
> - (D) ✅ 적합 — 체크포인트에서 이어서 훈련할 수 있으므로, 중단의 영향이 적습니다
>
> **핵심 포인트**: 스팟 인스턴스 적합 여부 판단의 핵심 질문은 "이 작업이 갑자기 중단되면 어떻게 되는가?"입니다. 중단 후 복구가 쉬운 작업일수록 스팟에 적합합니다.

> ⚠️ **주의**: CCP 시험에서 자주 출제되는 구매 옵션 관련 함정
>
> - **"가장 비용 효율적인(cost-effective) 옵션"**이라는 질문에서, 단순히 가장 저렴한 것(스팟)을 고르면 안 됩니다. 워크로드의 **요구사항(안정성, 지속성)을 먼저** 고려해야 합니다.
> - **Savings Plans와 예약 인스턴스의 차이**: 둘 다 약정을 통해 할인을 받지만, Savings Plans가 더 유연합니다. 시험에서는 "유연성"을 키워드로 Savings Plans를, "특정 인스턴스 유형에 대한 할인"을 키워드로 예약 인스턴스를 구분하는 문제가 출제됩니다.

### Section 요약

- EC2 구매 옵션은 **유연성과 비용의 트레이드오프**로 구분됩니다
- **온디맨드**: 유연하지만 비쌈, **예약/Savings Plans**: 약정으로 할인, **스팟**: 최대 할인이지만 중단 위험
- **전용 호스트/인스턴스**: 물리적 격리가 필요한 특수한 경우에 사용
- 실무에서는 여러 구매 옵션을 조합하여 비용을 최적화합니다

## 5.5 AWS Lambda와 서버리스 컴퓨팅 — 서버 관리 없는 코드 실행, 이벤트 기반 아키텍처

> 💡 **한 줄 요약**: AWS Lambda는 서버를 관리하지 않고 코드만 업로드하면, 이벤트가 발생할 때마다 자동으로 실행되는 서버리스 컴퓨팅 서비스입니다.

### 직관적 이해

EC2가 "호텔 방을 빌리는 것"이라면, Lambda는 **"회의실을 시간 단위로 빌리는 것"**에 비유할 수 있습니다.

호텔 방(EC2)은 체크인 후 체크아웃할 때까지 비어 있든 사용 중이든 비용이 발생합니다. 에어컨, 청소, 유지보수도 직접 관리해야 합니다.

반면 회의실(Lambda)은 회의가 있을 때만 사용하고, 회의가 없으면 비용이 전혀 발생하지 않습니다. 청소, 유지보수, 냉난방은 건물 관리자(AWS)가 알아서 처리합니다. 사용자는 **회의 내용(코드)**에만 집중하면 됩니다.

**왜 서버리스가 필요한가?**

EC2를 사용하면 서버의 운영 체제 업데이트, 보안 패치, 용량 관리 등을 사용자가 직접 해야 합니다. 많은 경우 개발자는 이런 인프라 관리보다 **비즈니스 로직(코드)에 집중**하고 싶어합니다. Lambda를 사용하면 인프라에 대한 걱정 없이 코드 실행에만 집중할 수 있습니다.

> 🔗 **연결**: Chapter 2에서 배운 서비스 모델 중, EC2가 **IaaS**의 대표라면, Lambda는 그보다 추상화 수준이 높은 서비스입니다. 사용자는 인프라(서버)를 전혀 관리하지 않고, 코드만 제공합니다.

### 핵심 개념 설명

> 📝 **서버리스 (Serverless)**: 서버가 "없다"는 뜻이 아니라, 사용자가 서버를 **관리할 필요가 없다**는 뜻입니다. 서버는 여전히 존재하지만, AWS가 서버의 프로비저닝, 패치, 확장, 가용성 관리를 모두 담당합니다.

> 📝 **AWS Lambda**: AWS의 대표적인 서버리스 컴퓨팅 서비스입니다. 사용자가 업로드한 코드(함수)를 이벤트에 의해 자동으로 실행합니다. 실행된 시간과 요청 횟수에 대해서만 과금됩니다.

**Lambda의 핵심 특징**:

1. **서버 관리 불필요**: OS 패치, 용량 계획, 서버 유지보수 등을 AWS가 처리
2. **이벤트 기반 실행**: 특정 이벤트(파일 업로드, API 호출, 스케줄 등)가 발생하면 자동 실행
3. **자동 확장**: 요청이 1건이든 100만 건이든, Lambda가 자동으로 필요한 만큼 확장
4. **초 단위 과금**: 코드가 실행된 시간(밀리초 단위)에 대해서만 비용 발생. 실행되지 않으면 비용 0원
5. **Free Tier**: 월 100만 건의 요청과 40만 GB-초의 컴퓨팅 시간이 무료 (12개월 제한 없이 영구 무료)

> 📝 **이벤트 기반 실행 (Event-driven Execution)**: 사용자가 직접 프로그램을 실행하는 것이 아니라, 특정 **이벤트(사건)**가 발생했을 때 자동으로 코드가 실행되는 방식입니다.

**Lambda를 트리거(촉발)할 수 있는 이벤트 예시**:

- 사용자가 API를 호출할 때 (API Gateway와 연동)
- S3 버킷에 파일이 업로드될 때
- 데이터베이스의 데이터가 변경될 때
- 정해진 시간이 되었을 때 (예: 매일 오전 9시)

> 💬 **참고**: S3(스토리지 서비스)는 Chapter 6에서 자세히 다룹니다. 여기서는 "파일을 저장하는 AWS 서비스"라는 정도만 이해하면 됩니다.

**Lambda의 제한 사항**:

| 항목 | 제한 |
|------|------|
| 최대 실행 시간 | 15분 |
| 메모리 | 128MB ~ 10,240MB |
| 배포 패키지 크기 | 50MB (압축) / 250MB (비압축) |

**EC2 vs Lambda 비교**:

| 비교 항목 | EC2 | Lambda |
|-----------|-----|--------|
| 서버 관리 | 사용자가 관리 | AWS가 관리 |
| 과금 방식 | 실행 시간(초/시간) | 요청 수 + 실행 시간(ms) |
| 실행 시간 제한 | 제한 없음 | 최대 15분 |
| 확장 | 수동 또는 Auto Scaling 설정 | 자동 |
| 적합한 워크로드 | 장시간 실행, 상시 운영 | 짧고 빈번한 작업, 이벤트 처리 |
| 제어 수준 | 높음 (OS, 네트워크 등) | 낮음 (코드에만 집중) |

### 상세 예시

> 🔍 **예시 5.5.1: Lambda로 이미지 자동 리사이징**
>
> **상황**: 사진 공유 앱에서 사용자가 원본 사진을 업로드하면, 자동으로 썸네일(작은 크기) 이미지를 생성하고 싶습니다.
>
> **풀이/설명**:
> 1. 사용자가 사진을 S3 버킷에 업로드합니다 (이벤트 발생)
> 2. S3 업로드 이벤트가 Lambda 함수를 자동으로 트리거합니다
> 3. Lambda 함수가 원본 이미지를 읽어 썸네일을 생성합니다
> 4. 생성된 썸네일을 다른 S3 버킷에 저장합니다
>
> 이 구성의 장점:
> - 사진 업로드가 없으면 Lambda 비용이 0원
> - 갑자기 1000장이 동시에 업로드되어도 Lambda가 자동으로 1000개의 인스턴스로 확장
> - 서버를 관리할 필요 없음
>
> **핵심 포인트**: Lambda는 "이벤트가 발생할 때만 실행"되므로, 간헐적이고 불규칙한 작업에 매우 비용 효율적입니다.

> 🔍 **예시 5.5.2: EC2 vs Lambda 선택 판단**
>
> **상황**: 다음 각 워크로드에 EC2와 Lambda 중 무엇이 더 적합할까요?
> - (A) 24시간 운영되는 온라인 게임 서버
> - (B) 사용자가 문의 양식을 제출할 때 이메일을 보내는 기능
> - (C) 2시간 동안 실행되는 대규모 데이터 처리 작업
>
> **풀이/설명**:
> - (A) **EC2** — 24시간 상시 운영이 필요하고, 게임 서버는 지속적인 연결과 높은 커스터마이징이 필요함
> - (B) **Lambda** — 이벤트 기반(양식 제출), 실행 시간이 짧음(수 초), 빈도가 불규칙함
> - (C) **EC2** — Lambda의 최대 실행 시간은 15분이므로, 2시간 작업은 EC2가 적합
>
> **핵심 포인트**: Lambda 선택의 핵심 기준은 ① 실행 시간 15분 이내, ② 이벤트 기반, ③ 서버 관리 불필요 입니다. 이 조건을 벗어나면 EC2가 더 적합합니다.

### Mini Lab

> 🧪 **Mini Lab 5.5: Lambda 함수 만들어 보기**
>
> **목표**: AWS 콘솔에서 간단한 Lambda 함수를 생성하고 테스트 실행하여, 서버리스 실행 모델을 직접 경험합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내 (Lambda 월 100만 건 무료)
>
> **Step 1**: AWS Management Console 상단 검색창에 `Lambda`를 입력하고 **Lambda** 서비스를 클릭합니다.
>
> **Step 2**: **Create function (함수 생성)** 버튼을 클릭합니다.
>
> **Step 3**: 다음과 같이 설정합니다:
> - **Author from scratch (처음부터 작성)** 선택
> - **Function name (함수 이름)**: `my-first-lambda` 입력
> - **Runtime (런타임)**: **Python 3.12** 선택 (또는 최신 Python 버전)
> - **Architecture**: **x86_64** 유지
> - 나머지는 기본값 유지
>
> **Step 4**: **Create function (함수 생성)** 버튼을 클릭합니다.
>
> **Step 5**: 함수 생성 후 코드 편집기가 표시됩니다. 기본 코드가 이미 작성되어 있습니다:
> ```python
> def lambda_handler(event, context):
>     return {
>         'statusCode': 200,
>         'body': 'Hello from Lambda!'
>     }
> ```
> 이 코드는 호출될 때마다 "Hello from Lambda!"라는 메시지를 반환하는 간단한 함수입니다.
>
> **Step 6**: **Test (테스트)** 탭을 클릭합니다. **Test event (테스트 이벤트)** 설정에서:
> - **Event name (이벤트 이름)**: `test-event` 입력
> - 나머지는 기본값 유지
> - **Save (저장)** 클릭 후 **Test (테스트)** 버튼을 클릭합니다.
>
> **Step 7**: 실행 결과를 확인합니다. **Execution result: succeeded** 메시지와 함께 반환값, 실행 시간(Duration), 사용 메모리(Memory Used) 등이 표시됩니다.
>
> **확인**: 실행 결과에서 `statusCode: 200`과 `body: "Hello from Lambda!"`가 표시되면 성공입니다. 실행 시간이 몇 밀리초 수준인 것도 확인해 보세요 — 이것이 실제로 과금되는 시간입니다.
>
> **🧹 정리**: Lambda 함수를 삭제합니다.
> 1. Lambda 콘솔의 **Functions (함수)** 목록으로 돌아갑니다
> 2. `my-first-lambda` 함수를 선택합니다
> 3. **Actions (작업)** → **Delete (삭제)**를 클릭합니다
> 4. 확인 입력란에 `delete`를 입력하고 **Delete**를 클릭합니다

> ⚠️ **주의**: Lambda에서 흔히 하는 실수
>
> - **"서버리스 = 서버가 없다"라는 오해**: 서버는 존재하지만 AWS가 관리합니다. 사용자가 서버를 "인식하지 않아도 된다"는 의미입니다.
> - **Lambda를 모든 곳에 사용하려는 실수**: 15분 이상 실행되는 작업, 상시 운영되는 서비스에는 EC2가 더 적합합니다. 적재적소에 맞는 서비스를 선택하세요.

### Section 요약

- AWS Lambda는 서버를 관리하지 않고 코드를 실행하는 **서버리스 컴퓨팅** 서비스입니다
- **이벤트 기반**으로 동작하며, 실행된 시간과 요청 수에 대해서만 과금됩니다
- 최대 실행 시간 15분 제한이 있으며, 짧고 빈번한 작업에 적합합니다
- EC2와 Lambda는 경쟁이 아니라 **보완 관계**이며, 워크로드 특성에 따라 선택합니다

## 5.6 기타 컴퓨팅 서비스 — Elastic Beanstalk, Lightsail, AWS Batch, Outposts 개요

> 💡 **한 줄 요약**: AWS는 EC2와 Lambda 외에도 다양한 컴퓨팅 서비스를 제공하며, 각각은 특정 사용 시나리오에 최적화되어 있습니다.

### 직관적 이해

음식을 먹는 방법에 비유해 봅시다:

- **EC2**: 재료를 사서 직접 요리하는 것 — 자유도가 높지만 시간과 노력이 많이 듦
- **Lambda**: 배달 앱으로 한 끼만 주문하는 것 — 편리하지만 메뉴와 양이 제한적
- **Elastic Beanstalk**: 밀키트(Meal Kit) — 재료와 레시피가 준비되어 있고, 조립만 하면 됨
- **Lightsail**: 편의점 도시락 — 가장 간단하고 예측 가능한 가격, 선택지는 적음
- **AWS Batch**: 급식소 — 대량의 음식을 효율적으로 한꺼번에 조리

**왜 이렇게 많은 컴퓨팅 서비스가 존재하는가?**

모든 사용자의 기술 수준과 요구사항이 다르기 때문입니다. 숙련된 엔지니어는 EC2의 세밀한 제어를 원하지만, 애플리케이션 개발자는 인프라보다 코드에 집중하고 싶습니다. AWS는 다양한 추상화 수준의 서비스를 제공하여, 각자의 필요에 맞는 서비스를 선택할 수 있게 합니다.

> 🔗 **연결**: Chapter 2에서 배운 **IaaS, PaaS, SaaS** 모델을 다시 떠올려 봅시다. EC2는 IaaS이고, Elastic Beanstalk은 PaaS에 해당하며, Lightsail은 IaaS와 PaaS의 중간 정도의 추상화를 제공합니다.

### 핵심 개념 설명

#### 1) AWS Elastic Beanstalk

> 📝 **AWS Elastic Beanstalk**: 웹 애플리케이션을 쉽게 배포하고 관리할 수 있는 **PaaS** 서비스입니다. 코드를 업로드하면 Beanstalk이 자동으로 EC2 인스턴스, 로드 밸런서, Auto Scaling 등을 구성합니다.

- **핵심 가치**: 개발자가 **코드에만 집중**할 수 있게 인프라 관리를 자동화
- **내부 동작**: 실제로는 EC2, Auto Scaling, Elastic Load Balancing 등의 AWS 서비스를 사용합니다 (비용은 이 리소스들에 대해서만 발생, Beanstalk 자체는 무료)
- **지원 플랫폼**: Java, Python, Node.js, PHP, Ruby, .NET, Go, Docker 등
- **적합한 경우**: 인프라 관리 부담 없이 웹 애플리케이션을 빠르게 배포하고 싶을 때
- **제어 수준**: 필요하면 내부 EC2 인스턴스 등에 직접 접근하여 세부 설정도 가능

> 💬 **참고**: Elastic Beanstalk은 "오케스트레이션 서비스"라고도 불립니다. 여러 AWS 리소스를 자동으로 조합하고 관리해주는 역할을 합니다. Auto Scaling과 로드 밸런싱은 이 Chapter의 Section 5.7과 Chapter 7.6에서 각각 다룹니다.

#### 2) Amazon Lightsail

> 📝 **Amazon Lightsail**: 간단한 웹사이트, 블로그, 소규모 애플리케이션을 위한 올인원 서비스입니다. 서버, 스토리지, 네트워크가 패키지로 제공되며, **월 고정 요금제**로 비용을 예측할 수 있습니다.

- **핵심 가치**: AWS를 처음 사용하는 사람도 쉽게 시작할 수 있는 **단순성과 예측 가능한 비용**
- **월 요금**: $3.50/월부터 시작 (서버 + 스토리지 + 데이터 전송 포함)
- **적합한 경우**: 개인 블로그, WordPress 사이트, 소규모 비즈니스 웹사이트, 개발/테스트 환경
- **부적합한 경우**: 대규모 엔터프라이즈 애플리케이션, 복잡한 아키텍처

**EC2 vs Elastic Beanstalk vs Lightsail 비교**:

| 비교 항목 | EC2 | Elastic Beanstalk | Lightsail |
|-----------|-----|-------------------|-----------|
| 추상화 수준 | 낮음 (IaaS) | 높음 (PaaS) | 중간 |
| 제어 수준 | 매우 높음 | 중간 (세부 조정 가능) | 낮음 |
| 복잡도 | 높음 | 중간 | 매우 낮음 |
| 과금 방식 | 사용량 기반 | 사용한 리소스 기반 | 월 고정 요금 |
| 대상 사용자 | 숙련된 엔지니어 | 애플리케이션 개발자 | 초보자, 소규모 프로젝트 |

#### 3) AWS Batch

> 📝 **AWS Batch**: 대규모 **배치 컴퓨팅 작업**을 효율적으로 실행하는 관리형 서비스입니다. 수천 개의 컴퓨팅 작업을 자동으로 스케줄링하고, 필요한 컴퓨팅 리소스를 자동으로 프로비저닝합니다.

- **배치 컴퓨팅이란**: 대량의 데이터를 한꺼번에 처리하는 작업을 말합니다. 예: 영화 렌더링, 유전자 분석, 재무 위험 분석
- **핵심 가치**: 사용자는 실행할 작업만 제출하면, 컴퓨팅 자원의 관리와 최적화는 AWS Batch가 처리
- **적합한 경우**: 대량 데이터 처리, 과학 시뮬레이션, 영상 렌더링

#### 4) AWS Outposts

> 📝 **AWS Outposts**: AWS 인프라와 서비스를 **사용자의 온프레미스 데이터센터**에 설치하여 운영할 수 있게 하는 서비스입니다. AWS가 하드웨어를 고객의 데이터센터에 배송·설치·관리합니다.

- **핵심 가치**: 데이터가 반드시 특정 장소에 있어야 하는 규제 요건이나, 매우 낮은 지연 시간이 필요한 경우에 클라우드와 동일한 AWS 환경을 온프레미스에서 사용
- **적합한 경우**: 데이터 주권 요구사항이 있는 경우, 하이브리드 클라우드 아키텍처

> 🔗 **연결**: Chapter 2에서 배운 **하이브리드 클라우드** 배포 모델을 기억하시나요? Outposts는 하이브리드 클라우드를 구현하는 구체적인 AWS 서비스 중 하나입니다. 또한 Chapter 3에서 배운 **AWS Outposts**가 리전 외부로 인프라를 확장하는 옵션이었음을 기억해 봅시다.

### 상세 예시

> 🔍 **예시 5.6.1: 적합한 컴퓨팅 서비스 선택**
>
> **상황**: 다음 각 시나리오에 가장 적합한 AWS 컴퓨팅 서비스는?
> - (A) 프리랜서 웹 디자이너가 포트폴리오 웹사이트를 만들고 싶음
> - (B) 스타트업이 Python Flask 웹 앱을 빠르게 배포하고 싶지만, 인프라 관리에 시간을 쓰고 싶지 않음
> - (C) 제약회사가 신약 후보 물질 100만 개를 시뮬레이션해야 함
> - (D) 금융 기관이 규제상 데이터를 자체 데이터센터에 보관하면서도 AWS 서비스를 사용하고 싶음
>
> **풀이/설명**:
> - (A) **Amazon Lightsail** — 간단하고 저렴한 월정액, 기술 지식이 적어도 시작 가능
> - (B) **Elastic Beanstalk** — 코드만 업로드하면 인프라가 자동 구성, Flask 지원
> - (C) **AWS Batch** — 대규모 배치 컴퓨팅 작업에 최적화
> - (D) **AWS Outposts** — 자체 데이터센터에서 AWS 인프라를 운영
>
> **핵심 포인트**: AWS는 "한 가지 서비스로 모든 것을 해결"하는 접근이 아니라, 각 상황에 최적화된 서비스를 제공합니다. 요구사항에 맞는 서비스를 선택하는 것이 핵심입니다.

> 🔍 **예시 5.6.2: Elastic Beanstalk의 내부 구조 이해**
>
> **상황**: Elastic Beanstalk에 Python 웹 앱을 배포하면 내부적으로 어떤 일이 일어나는지 이해해 봅시다.
>
> **풀이/설명**:
> 1. 개발자가 Python 코드를 Beanstalk에 업로드합니다
> 2. Beanstalk이 자동으로 **EC2 인스턴스**를 생성하고 Python 런타임을 설치합니다
> 3. **로드 밸런서**(부하 분산기)를 구성합니다
> 4. **Auto Scaling 그룹**을 설정하여 트래픽에 따라 인스턴스 수를 자동 조절합니다
> 5. 헬스 체크(상태 확인)를 구성하여 문제가 생기면 자동으로 복구합니다
>
> **핵심 포인트**: Beanstalk은 **마법이 아닙니다**. EC2, 로드 밸런서 등 기존 AWS 서비스를 자동으로 구성해주는 "편의 서비스"입니다. Beanstalk 자체에는 추가 비용이 없고, 사용된 리소스(EC2, EBS 등)에 대해서만 과금됩니다.

> ⚠️ **주의**: 기타 컴퓨팅 서비스에서 흔한 혼동
>
> - **Elastic Beanstalk ≠ Lambda**: Beanstalk은 EC2 기반의 PaaS이고, Lambda는 서버리스입니다. Beanstalk은 상시 운영 서버가 필요한 웹 앱에, Lambda는 이벤트 기반의 짧은 작업에 적합합니다.
> - **Lightsail ≠ 단순한 EC2**: Lightsail은 EC2보다 설정이 간단하고 월정액이지만, 대규모 확장이나 세밀한 제어에는 한계가 있습니다. 프로젝트가 성장하면 EC2로 마이그레이션해야 할 수 있습니다.

### Section 요약

- **Elastic Beanstalk**: 코드 업로드만으로 웹 앱을 배포하는 PaaS (내부적으로 EC2 등 사용, 자체 비용 없음)
- **Lightsail**: 간단한 웹사이트를 위한 올인원 서비스 (월 고정 요금, 초보자 친화적)
- **AWS Batch**: 대규모 배치 컴퓨팅 작업의 자동 스케줄링 및 리소스 관리
- **AWS Outposts**: 온프레미스 데이터센터에서 AWS 인프라를 운영하는 하이브리드 솔루션

## 5.7 Auto Scaling 기초 — 수요에 따른 자동 확장/축소 개념

> 💡 **한 줄 요약**: Auto Scaling은 트래픽 변화에 따라 EC2 인스턴스 수를 자동으로 늘리거나 줄여, 성능과 비용을 동시에 최적화하는 서비스입니다.

### 직관적 이해

대형 마트의 계산대를 떠올려 봅시다. 평일 오전에는 손님이 적어 계산대 2개만 운영합니다. 점심 시간이 되어 손님이 몰리면 계산대를 5개로 늘립니다. 저녁에 다시 손님이 줄면 3개로 줄입니다.

만약 계산대가 항상 2개만 운영된다면? 바쁜 시간에 긴 줄이 생기고 손님이 불만을 갖습니다. 반대로 항상 10개를 운영한다면? 한가한 시간에 직원 인건비가 낭비됩니다.

Auto Scaling은 바로 이 **"필요할 때 늘리고, 필요 없을 때 줄이는"** 것을 자동으로 해주는 서비스입니다. 이것이 클라우드의 핵심 이점인 **탄력성(Elasticity)**의 실제 구현입니다.

> 🔗 **연결**: Chapter 1에서 배운 클라우드의 이점 중 **신속한 탄력성**과 **종량제 과금**을 동시에 실현하는 서비스가 Auto Scaling입니다.

### 핵심 개념 설명

> 📝 **Amazon EC2 Auto Scaling**: EC2 인스턴스의 수를 자동으로 조정하여, 애플리케이션에 필요한 컴퓨팅 용량을 유지하는 서비스입니다. 수요가 급증하면 인스턴스를 추가하고(Scale Out), 수요가 감소하면 인스턴스를 제거합니다(Scale In).

> 📝 **스케일 업/다운 vs 스케일 아웃/인**:
> - **스케일 업 (Scale Up)**: 기존 인스턴스를 더 큰 것으로 교체 (예: t3.small → t3.large). "수직 확장"이라고도 합니다.
> - **스케일 아웃 (Scale Out)**: 동일한 인스턴스를 더 많이 추가 (예: 2대 → 5대). "수평 확장"이라고도 합니다.
> - Auto Scaling은 주로 **스케일 아웃/인(수평 확장/축소)**을 자동으로 처리합니다.

**Auto Scaling의 핵심 구성 요소**:

> 📝 **Auto Scaling 그룹 (ASG, Auto Scaling Group)**: Auto Scaling이 관리하는 EC2 인스턴스들의 논리적 그룹입니다. 이 그룹에 대해 최소, 최대, 희망(Desired) 인스턴스 수를 설정합니다.

```
Auto Scaling 그룹 설정 예시:

최소 용량 (Minimum): 2대  ← 아무리 줄어도 이 이하로는 안 줄임
희망 용량 (Desired):  3대  ← 평상시 유지하는 인스턴스 수
최대 용량 (Maximum): 6대  ← 아무리 늘어도 이 이상으로는 안 늘림

     [최소: 2]──────[희망: 3]──────────────[최대: 6]
      ├──┤           ├──┤                  ├──┤
   인스턴스      인스턴스 3대              최대 6대까지
   2대 이상     (평상시 유지)              확장 가능
```

**Auto Scaling 정책 유형**:

| 정책 유형 | 설명 | 예시 |
|-----------|------|------|
| **대상 추적 (Target Tracking)** | 특정 지표를 목표 값으로 유지 | "CPU 사용률을 50%로 유지" |
| **단계별 (Step Scaling)** | 지표 변화 정도에 따라 다른 크기로 확장/축소 | "CPU 70% → 2대 추가, 90% → 4대 추가" |
| **예약 (Scheduled)** | 미리 정해진 시간에 확장/축소 | "매주 월요일 09:00에 5대로 확장" |

**Auto Scaling의 이점**:

1. **고가용성**: 인스턴스에 장애가 발생하면 자동으로 새 인스턴스를 시작하여 최소 용량을 유지
2. **비용 최적화**: 필요 이상의 인스턴스를 유지하지 않아 비용 절감
3. **자동화**: 수동으로 인스턴스를 추가/제거하는 번거로움 해소

> 💬 **참고**: Auto Scaling은 **트래픽 분산(Load Balancing)**과 함께 사용할 때 가장 효과적입니다. Auto Scaling이 인스턴스를 추가하면, 로드 밸런서가 새 인스턴스에도 트래픽을 분배합니다. ELB(Elastic Load Balancing)와의 연동은 Chapter 7.6에서 자세히 다룹니다.

### 상세 예시

> 🔍 **예시 5.7.1: Auto Scaling이 필요한 상황**
>
> **상황**: 뉴스 웹사이트를 운영합니다. 평소에는 서버 2대로 충분하지만, 속보가 터지면 트래픽이 10배로 급증합니다. 속보의 타이밍은 예측할 수 없습니다.
>
> **풀이/설명**:
> - Auto Scaling 그룹을 구성합니다: 최소 2대, 최대 20대
> - 정책: CPU 사용률이 60% 이상이면 인스턴스 추가, 30% 이하이면 인스턴스 제거
> - 속보 발생 시: 트래픽 급증 → CPU 사용률 상승 → Auto Scaling이 자동으로 인스턴스 추가 → 서비스 안정 유지
> - 트래픽 정상화 시: CPU 사용률 하락 → 추가 인스턴스 자동 제거 → 비용 절감
>
> **핵심 포인트**: Auto Scaling 없이 항상 20대를 운영하면, 평소에 18대분의 비용이 낭비됩니다. Auto Scaling 없이 2대만 운영하면, 속보 시 서비스가 다운됩니다. Auto Scaling은 이 딜레마를 해결합니다.

> 🔍 **예시 5.7.2: Auto Scaling 그룹 설정의 의미**
>
> **상황**: 다음과 같은 Auto Scaling 그룹을 설정했습니다:
> - 최소: 1대 / 희망: 2대 / 최대: 4대
> - 현재 2대 운영 중, 1대에 장애 발생
>
> **풀이/설명**:
> 1. 장애 발생: 2대 중 1대가 비정상 → 현재 정상 인스턴스 1대
> 2. Auto Scaling 감지: 희망 용량(2대)보다 적은 1대만 실행 중
> 3. 자동 복구: 새 인스턴스 1대를 자동으로 시작 → 다시 2대 유지
> 4. 결과: 관리자가 개입하지 않아도 서비스 안정성이 유지됨
>
> **핵심 포인트**: Auto Scaling은 성능 확장뿐 아니라 **장애 복구(자가 치유)**에도 중요한 역할을 합니다. 인스턴스에 문제가 생기면 자동으로 교체합니다.

> 🔬 **Deep Dive**: 스케일 업 vs 스케일 아웃 — 언제 무엇을 선택할까?
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> **스케일 업 (수직 확장)**:
> - 한 인스턴스의 성능을 올리는 것 (예: 4GB 메모리 → 16GB)
> - 장점: 애플리케이션 코드 변경이 불필요
> - 단점: 물리적 한계 존재 (세상에서 가장 큰 서버에도 상한이 있음), 확장 시 다운타임 발생 가능
>
> **스케일 아웃 (수평 확장)**:
> - 동일한 인스턴스를 여러 대 추가하는 것 (예: 2대 → 10대)
> - 장점: 이론적으로 무한 확장 가능, 한 대가 죽어도 나머지가 처리 (고가용성)
> - 단점: 애플리케이션이 여러 인스턴스에서 동작할 수 있도록 설계되어야 함
>
> 클라우드에서는 일반적으로 **스케일 아웃**이 권장됩니다. Auto Scaling은 이 스케일 아웃을 자동화하는 서비스입니다.

> ⚠️ **주의**: Auto Scaling 관련 주의사항
>
> - **Auto Scaling ≠ 무한 확장**: 최대 용량을 설정하는 이유는 비용 폭발을 방지하기 위해서입니다. 최대 용량을 너무 높게 설정하면 예상치 못한 비용이 발생할 수 있습니다.
> - **Auto Scaling 자체는 무료**입니다: 추가되는 EC2 인스턴스에 대한 비용만 발생합니다.

### Section 요약

- Auto Scaling은 수요에 따라 EC2 인스턴스 수를 **자동으로 확장(Scale Out)/축소(Scale In)**합니다
- Auto Scaling 그룹에서 **최소, 희망, 최대** 인스턴스 수를 설정합니다
- 정책 유형: 대상 추적, 단계별, 예약
- Auto Scaling은 **성능 유지, 비용 최적화, 장애 복구** 세 가지 이점을 동시에 제공합니다

## 자기 점검 질문 (Self-Check Questions)

**Q1.** EC2 인스턴스를 "중지(Stop)"하면 인스턴스 비용은 발생하지 않지만, EBS 볼륨 비용은 계속 발생한다. (O/X)

<details>
<summary>정답 확인</summary>

**O (맞음)** — 중지(Stop)는 인스턴스를 꺼두는 것이므로 EC2 인스턴스 과금은 멈추지만, 연결된 EBS 스토리지는 여전히 존재하므로 볼륨 비용이 발생합니다. 비용을 완전히 없애려면 종료(Terminate)해야 합니다.

</details>

**Q2.** 머신러닝 모델 훈련처럼 CPU를 많이 사용하는 작업에 가장 적합한 EC2 인스턴스 패밀리는?

<details>
<summary>정답 확인</summary>

**C (컴퓨팅 최적화) 패밀리** — 또는 GPU가 필요한 경우 P/G (가속 컴퓨팅) 패밀리. C 패밀리는 높은 CPU 성능에 최적화되어 있어 과학 계산, 미디어 인코딩, 머신러닝 추론 등에 적합합니다. 머신러닝 "훈련"의 경우 GPU가 필요할 수 있어 P 패밀리가 더 적합할 수도 있습니다.

</details>

**Q3.** AWS Lambda의 최대 실행 시간은 얼마인가?

<details>
<summary>정답 확인</summary>

**15분** — Lambda 함수 하나의 최대 실행 시간(Timeout)은 15분(900초)입니다. 이보다 오래 걸리는 작업은 EC2나 다른 서비스를 사용해야 합니다.

</details>

**Q4.** "서버리스(Serverless)"란 서버가 물리적으로 존재하지 않는다는 뜻이다. (O/X)

<details>
<summary>정답 확인</summary>

**X (틀림)** — 서버리스는 서버가 없다는 뜻이 아니라, 사용자가 서버를 관리할 필요가 없다는 뜻입니다. AWS가 서버의 프로비저닝, 패치, 확장, 유지보수를 모두 담당합니다.

</details>

**Q5.** 스팟 인스턴스는 온디맨드 대비 최대 약 몇 %까지 할인받을 수 있는가?

<details>
<summary>정답 확인</summary>

**최대 약 90%** — 스팟 인스턴스는 AWS의 여유 컴퓨팅 용량을 사용하는 것이므로 가장 저렴하지만, AWS가 해당 용량을 다시 필요로 하면 2분 전 알림 후 인스턴스가 회수될 수 있습니다.

</details>

**Q6.** Elastic Beanstalk 자체의 사용 비용은 얼마인가?

<details>
<summary>정답 확인</summary>

**무료** — Elastic Beanstalk 서비스 자체에는 비용이 없습니다. Beanstalk이 내부적으로 생성하는 EC2 인스턴스, EBS 볼륨, 로드 밸런서 등 실제 사용된 AWS 리소스에 대한 비용만 발생합니다.

</details>

**Q7.** Auto Scaling 그룹에서 "최소 용량(Minimum)"을 2로 설정한 경우, 인스턴스 하나에 장애가 발생하면 어떻게 되는가?

<details>
<summary>정답 확인</summary>

**Auto Scaling이 자동으로 새 인스턴스를 시작하여 최소 2대를 유지합니다.** Auto Scaling은 설정된 최소 용량 이하로 인스턴스 수가 줄어드는 것을 허용하지 않습니다. 장애가 발생한 인스턴스를 대체하는 새 인스턴스가 자동으로 시작됩니다.

</details>

## 🧪 통합 실습: 나의 첫 번째 클라우드 서버 구축과 운영

> **시나리오**: 당신은 소규모 스타트업의 클라우드 엔지니어입니다. 회사의 첫 번째 웹 서비스를 AWS에 배포하기 위해, EC2 인스턴스를 생성하고 웹 서버를 구동하며, Lambda를 활용한 간단한 자동화까지 구성해 봅니다.

### 실습 개요

| 항목 | 내용 |
|------|------|
| **목표** | EC2 인스턴스에 웹 서버를 구동하고, Lambda 함수를 생성하여 두 서비스의 차이를 체감 |
| **사용 서비스** | Amazon EC2, AWS Lambda |
| **활용 개념** | Section 5.1 EC2 개요, Section 5.2 인스턴스 유형, Section 5.3 AMI와 인스턴스 시작, Section 5.5 Lambda |
| **선수 실습** | Chapter 4에서 AWS 계정 생성 및 콘솔 탐색 완료 |
| **예상 소요 시간** | 30분 |
| **비용** | Free Tier 범위 내 |

### 아키텍처 다이어그램

```
[사용자의 브라우저]
       │
       ▼
[EC2 인스턴스 - t2.micro]
  └── Amazon Linux 2023
  └── Apache 웹 서버
  └── 간단한 HTML 페이지

[별도 구성]
[이벤트 (테스트)] → [Lambda 함수] → [응답 반환]
```

### 실습 단계

> **Phase 1: EC2 인스턴스 생성 및 웹 서버 구동** (활용 개념: Section 5.1, 5.2, 5.3)
>
> **Step 1**: AWS Management Console에 로그인하고, 리전을 **ap-northeast-2 (서울)**로 설정합니다.
>
> **Step 2**: 상단 검색창에 `EC2`를 입력하고 EC2 서비스로 이동합니다. **Launch instances (인스턴스 시작)**를 클릭합니다.
>
> **Step 3**: 다음과 같이 구성합니다:
> - **Name**: `my-web-server`
> - **AMI**: **Amazon Linux 2023 AMI** (Free tier eligible)
> - **Instance type**: **t2.micro** (Free tier eligible)
> - **Key pair**: **Proceed without a key pair** (이 실습에서는 User Data로 웹 서버를 구성하므로 SSH 접속 불필요)
>
> **Step 4**: **Network settings (네트워크 설정)**에서 **Edit (편집)**을 클릭합니다:
> - **Auto-assign public IP**: **Enable** (활성화) — 브라우저에서 접속하기 위해 퍼블릭 IP가 필요합니다
> - **Security group**: **Create security group** 선택
>   - **Security group name**: `my-web-sg`
>   - **Description**: `Allow HTTP access`
>   - 기본으로 있는 SSH 규칙(포트 22)은 **Remove (제거)** 합니다
>   - **Add security group rule (보안 그룹 규칙 추가)** 클릭:
>     - **Type**: **HTTP**
>     - **Source type**: **Anywhere** (0.0.0.0/0)
>
> **Step 5**: **Advanced details (고급 세부 정보)**를 펼치고, 맨 아래의 **User data** 필드에 다음 스크립트를 입력합니다:
> ```bash
> #!/bin/bash
> yum update -y
> yum install -y httpd
> systemctl start httpd
> systemctl enable httpd
> echo "<html><body><h1>Hello from my first EC2!</h1><p>Instance is running in $(curl -s http://169.254.169.254/latest/meta-data/placement/availability-zone)</p></body></html>" > /var/www/html/index.html
> ```
> 이 스크립트는 인스턴스가 시작될 때 자동으로 Apache 웹 서버를 설치하고, 간단한 HTML 페이지를 생성합니다.
>
> **Step 6**: **Launch instance (인스턴스 시작)**를 클릭합니다.
>
> **Step 7**: 인스턴스 목록에서 `my-web-server`의 상태가 **Running**이 되고 **Status check**가 **2/2 checks passed**가 될 때까지 기다립니다 (약 2~3분).
>
> **Step 8**: 인스턴스를 클릭하고, 하단 **Details** 탭에서 **Public IPv4 address**를 복사합니다.
>
> **Step 9**: 브라우저 주소창에 `http://[복사한 IP 주소]`를 입력합니다.
>
> **✅ 체크포인트**: 브라우저에 **"Hello from my first EC2!"** 메시지와 가용 영역 정보가 표시되면 Phase 1 성공입니다!

> **Phase 2: Lambda 함수 생성 및 비교** (활용 개념: Section 5.5)
>
> **Step 1**: 상단 검색창에 `Lambda`를 입력하고 Lambda 서비스로 이동합니다.
>
> **Step 2**: **Create function (함수 생성)**을 클릭합니다:
> - **Author from scratch** 선택
> - **Function name**: `my-web-response`
> - **Runtime**: **Python 3.12**
> - **Create function** 클릭
>
> **Step 3**: 코드 편집기에서 기본 코드를 다음으로 교체합니다:
> ```python
> import json
> from datetime import datetime
>
> def lambda_handler(event, context):
>     return {
>         'statusCode': 200,
>         'body': json.dumps({
>             'message': 'Hello from Lambda!',
>             'timestamp': datetime.now().isoformat(),
>             'note': 'This function runs without any server management!'
>         })
>     }
> ```
>
> **Step 4**: **Deploy** 버튼을 클릭하여 코드를 배포합니다.
>
> **Step 5**: **Test** 탭으로 이동하여 테스트 이벤트를 생성하고 **Test** 버튼을 클릭합니다.
>
> **✅ 체크포인트**: 실행 결과에서 `"message": "Hello from Lambda!"`가 표시되고, 실행 시간(Duration)이 수 밀리초 수준이면 Phase 2 성공입니다!

> **Phase 3: EC2와 Lambda 비교 관찰** (활용 개념: Section 5.1, 5.5)
>
> **Step 1**: EC2 콘솔로 돌아가서 `my-web-server` 인스턴스의 **Monitoring** 탭을 클릭합니다. CPU 사용률, 네트워크 트래픽 등의 지표를 관찰합니다. 웹 서버가 실행 중이지만, 접속이 없으면 자원이 거의 사용되지 않는 것을 확인합니다.
>
> **Step 2**: 다음 관점에서 EC2와 Lambda를 비교해 봅시다:
>
> | 관점 | EC2 (Phase 1) | Lambda (Phase 2) |
> |------|---------------|-------------------|
> | 서버 설정 | OS 선택, 보안 그룹 구성, User Data 작성 필요 | 코드만 작성하면 됨 |
> | 서버 관리 | 패치, 모니터링 등 사용자 책임 | AWS가 모두 관리 |
> | 접속 방식 | 퍼블릭 IP로 직접 접속 | 이벤트(API Gateway 등)로 호출 |
> | 과금 | 실행 중인 시간 전체 | 호출된 횟수 + 실행 시간만 |
> | 항시 가동 여부 | 네 (Stop/Terminate 전까지) | 아니오 (호출 시에만 실행) |
>
> **✅ 체크포인트**: 위 비교를 통해 "언제 EC2를, 언제 Lambda를 선택해야 하는지" 직접 느낄 수 있었다면 성공입니다!

### 트러블슈팅 (Troubleshooting)

> **Q: EC2 인스턴스의 퍼블릭 IP로 접속했는데 페이지가 뜨지 않아요.**
> A: 다음을 확인하세요:
> 1. 인스턴스 상태가 **Running**이고 상태 검사가 **2/2 checks passed**인지 확인
> 2. 보안 그룹에서 **HTTP (포트 80)**이 **0.0.0.0/0**에서 허용되어 있는지 확인
> 3. `https://`가 아닌 `http://`로 접속하고 있는지 확인 (이 실습에서는 HTTPS를 설정하지 않았습니다)
> 4. User Data 스크립트가 정확히 입력되었는지 확인 — 인스턴스를 종료하고 다시 생성해 볼 수 있습니다

> **Q: 보안 그룹을 삭제할 수 없어요.**
> A: 해당 보안 그룹이 아직 인스턴스에 연결되어 있기 때문입니다. 인스턴스가 완전히 종료(Terminated)된 후 몇 분 기다리면 삭제할 수 있습니다.

> **Q: Lambda 함수 테스트에서 오류가 발생해요.**
> A: 코드의 들여쓰기(indentation)가 정확한지 확인하세요. Python은 들여쓰기에 민감합니다. 코드를 복사-붙여넣기할 때 공백이 깨질 수 있습니다.

> B. Beanstalk 서비스 자체는 무료이며, 내부적으로 사용하는 리소스에 대해서만 과금된다

> D. Beanstalk은 월 고정 요금제로 운영된다

> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 시나리오의 기술 수준, 관리 부담, 트래픽 패턴, 실행 시간 등을 종합 고려합니다.

> **Exercise 5.3**: Auto Scaling 그룹을 구성하여 온라인 쇼핑몰의 웹 서버를 운영하려 합니다. 다음 조건을 고려하여 Auto Scaling 설정을 설계하세요.
>
> - 평소 트래픽을 처리하는 데 서버 3대가 필요합니다

> - 최대 트래픽(세일 기간)에는 서버 10대까지 필요할 수 있습니다
> - CPU 사용률이 70%를 넘으면 서버를 추가해야 합니다
>
> (1) 최소, 희망, 최대 용량을 설정하세요.
> (2) 적절한 Auto Scaling 정책 유형을 선택하고 이유를 설명하세요.
> (3) 이 구성에서 인스턴스 1대에 장애가 발생하면 어떻게 되는지 설명하세요.
>
> **힌트**: 대상 추적(Target Tracking) 정책과 예약(Scheduled) 정책을 함께 사용할 수도 있습니다.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 조건에서 최소/희망/최대 요구사항을 도출하고, 트래픽 패턴에 맞는 정책을 선택합니다.
>
> **(1) Auto Scaling 그룹 설정**:
> - **최소 용량**: **2대** — "최소 2대는 항상 운영" 요건
> - **희망 용량**: **3대** — "평소 트래픽 처리에 서버 3대 필요"
> - **최대 용량**: **10대** — "최대 서버 10대까지 필요"
>
> **(2) Auto Scaling 정책**:
> - **대상 추적 정책 (Target Tracking)** — CPU 사용률 목표를 70%로 설정
>   - 이유: "CPU 70% 넘으면 추가"라는 조건은 특정 지표를 목표 값으로 유지하는 대상 추적 정책에 가장 적합
>   - CPU 사용률이 70%를 초과하면 인스턴스가 자동 추가되어 70% 수준으로 낮춤
>   - CPU 사용률이 70%보다 훨씬 낮으면 인스턴스를 제거하여 비용 절감
> - 보완적으로, 세일 기간이 예정되어 있다면 **예약 정책 (Scheduled)**을 추가하여 세일 시작 전에 미리 인스턴스를 늘려놓을 수 있음
>
> **(3) 장애 발생 시**:
> - 현재 3대 중 1대에 장애 발생 → 정상 인스턴스 2대
> - Auto Scaling이 감지: 현재 2대 < 희망 용량 3대
> - 자동으로 새 인스턴스 1대를 시작하여 3대로 복구
> - 만약 2대에 동시 장애 시에도, 최소 용량 2대를 유지하기 위해 새 인스턴스 2대를 시작
> - 사용자 개입 없이 서비스 연속성 보장
>
> **보충 설명**: 실제 운영에서는 Auto Scaling 그룹의 인스턴스를 **여러 가용 영역(AZ)에 분산** 배치하여, 특정 AZ의 장애에도 서비스가 유지되도록 설계합니다. 이는 Chapter 3에서 배운 가용 영역의 고가용성 개념과 직결됩니다.
> </details>

>
> - **서비스 A**: 글로벌 뉴스 웹사이트 (24/7 운영, 속보 시 트래픽 급증, 예측 불가)

> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 각 서비스의 특성(지속성, 자원 요구, 중단 내성)을 분석하여 가장 적합한 조합을 선택합니다.

>
> **상세 풀이**:
>
> **서비스 A: 글로벌 뉴스 웹사이트**
> - (1) **EC2 + Auto Scaling**: 24/7 상시 운영, 트래픽 급증에 자동 대응 필요
> - (2) **범용 M 패밀리** (예: m5.large): 웹 서버는 CPU와 메모리가 균형 잡힌 구성이 적합
> - (3) **기본 인스턴스: 예약 인스턴스/Savings Plans** (상시 운영분) + **추가 인스턴스: 온디맨드** (Auto Scaling 확장분)

> - (1) **AWS Lambda** 또는 **EC2**: 영상당 5~10분이면 Lambda 15분 제한 내이므로 Lambda 가능. 다만 CPU 집약적이므로 Lambda의 최대 메모리(10GB)로 충분한지 확인 필요
>   - Lambda를 선택한다면: 파일 업로드 이벤트 기반으로 자동 트리거, 불규칙한 빈도에 비용 효율적
>   - EC2를 선택한다면: C 패밀리(컴퓨팅 최적화)가 영상 인코딩에 적합
> - (2) Lambda 사용 시: 메모리 최대 할당, 타임아웃 10분 설정 / EC2 사용 시: **C 패밀리** (c5.xlarge 이상)
> - (3) Lambda: 사용량 기반 과금 / EC2: 스팟 인스턴스 가능 (인코딩 실패 시 재시도 가능)
>
> **서비스 C: 야간 배치 분석**
> - (1) **EC2**: 45분 작업이므로 Lambda 15분 제한 초과 → EC2가 적합
> - (2) **메모리 최적화 R 패밀리** (예: r5.2xlarge): "대량의 데이터를 메모리에 올려 분석"하므로 메모리 특화 필요
> - (3) **스팟 인스턴스**: 매일 정해진 시간에 실행, 실패 시 다음 날 재실행 가능 (또는 배치 특성상 재시도 가능)
>   - 보완: 스팟이 확보 안 될 경우를 대비해 온디맨드 폴백 설정
>
> **(4) 클라우드 개념 연결**:
> - **Chapter 1 — 탄력성**: Auto Scaling이 서비스 A의 트래픽 변동에 자동 대응

> - **Chapter 3 — 리전/AZ**: EC2 인스턴스를 여러 AZ에 분산하여 서비스 A의 고가용성 확보
> - **Chapter 4 — Free Tier 이해**: 개발/테스트 단계에서 Free Tier를 활용하여 비용 최소화
>
> **확장 생각**: 서비스 B의 인코딩 결과를 어디에 저장할 것인가? → Chapter 6(스토리지)에서 배울 S3가 적합. 서비스 A의 글로벌 사용자에게 빠른 응답을 제공하려면? → Chapter 7(네트워킹)에서 배울 CloudFront(CDN)가 필요.
> </details>

> **Problem 5.2**: EC2 구매 옵션에 대한 심층 분석 문제입니다.
>
> 한 회사가 12개월간 다음과 같은 EC2 사용 패턴을 보였습니다:
> - **서버 A** (웹 서버): 12개월간 24/7 연속 실행 (m5.large, 서울 리전)
> - **서버 B** (개발 서버): 주중 09:00~18:00만 실행 (주당 약 45시간), 12개월 사용 (t3.medium)
> - **서버 C** (데이터 분석): 월 1회, 약 20시간 사용, 중단 시 처음부터 재실행 필요 (r5.xlarge)
>
> (1) 각 서버가 "온디맨드"만 사용했을 때의 비용 구조적 문제점을 분석하세요.
> (2) 각 서버에 최적의 구매 옵션을 추천하고, 예상 절감 효과를 정성적으로 설명하세요.
> (3) "Savings Plans와 예약 인스턴스 중 무엇이 이 회사에 더 적합한가?"에 대한 당신의 의견을 근거와 함께 제시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 사용 패턴(상시 vs 정기 vs 간헐적)과 각 구매 옵션의 약정 구조를 매칭합니다.
>
> **풀이 전략**: 각 서버의 월간 사용 시간을 계산하고, 구매 옵션별 할인 구조와 대조합니다.
>
> **상세 풀이**:
>
> **(1) 온디맨드만 사용 시 문제점**:
> - **서버 A**: 24/7 × 12개월 = 약 8,760시간. 상시 운영인데 온디맨드 정가를 지불하면, 예약 대비 최대 72% 비용을 더 내는 것. 가장 큰 비용 낭비 구간
> - **서버 B**: 주 45시간 × 52주 = 약 2,340시간. 사용하지 않는 시간(주 123시간)에는 과금 없으므로 온디맨드도 나쁘지 않으나, 12개월 지속이므로 약정 할인 여지가 있음
> - **서버 C**: 월 20시간 × 12개월 = 약 240시간. 사용량이 적으므로 온디맨드 비용 자체는 크지 않음. 그러나 중단 가능한 작업인데 비싼 가격을 지불하고 있음
>
> **(2) 최적 구매 옵션 추천**:
> - **서버 A** → **1년 예약 인스턴스 (전체 또는 부분 선결제)** 또는 **EC2 Instance Savings Plans**
>   - 24/7 상시 운영으로 가장 높은 할인 효과 (최대 ~72% 절감)
>   - 가장 큰 비용 항목이므로 절감 효과가 극대화
> - **서버 B** → **온디맨드 유지** 또는 **Compute Savings Plans (부분 적용)**
>   - 주중 업무시간만 사용하므로 전체 예약은 비효율적
>   - Savings Plans를 사용하면, B의 사용 시간에 대해서도 할인 가능
>   - 또는 인스턴스 스케줄러(EventBridge + Lambda)로 자동 시작/중지하여 온디맨드 비용을 최적화
> - **서버 C** → **스팟 인스턴스**
>   - 월 1회, 짧은 사용, 중단 내성 있음 (재실행 가능) — 스팟의 이상적 사용 사례
>   - 최대 90% 할인 가능
>   - 단, "중단 시 처음부터 재실행"이므로 인스턴스 회수 시 시간·비용 낭비 가능 → 그래도 온디맨드 대비 훨씬 저렴
>
> **(3) Savings Plans vs 예약 인스턴스**:
> - 이 회사에는 **Compute Savings Plans**가 더 적합할 수 있음:
>   - 서버 A의 비용을 주로 커버하되, 서버 B의 사용 시간에도 할인 적용 가능
>   - 향후 인스턴스 유형이나 리전을 변경할 유연성 확보
>   - 만약 서버 A의 인스턴스 유형(m5.large)을 확실히 유지할 것이라면, 예약 인스턴스가 약간 더 큰 할인을 제공할 수 있음
>   - 결론: 유연성을 중시하면 Savings Plans, 최대 할인을 중시하면 예약 인스턴스
>
> **확장 생각**: Chapter 21(요금)에서는 이러한 비용 최적화 전략을 더 체계적으로 다루며, AWS Cost Explorer 같은 도구를 활용하여 실제 절감 효과를 분석하는 방법을 배웁니다.
> </details>
