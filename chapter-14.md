## 14.1 네트워크 보안 계층 — 보안 그룹과 네트워크 ACL의 비교

> 💡 **한 줄 요약**: AWS 네트워크 보안은 "인스턴스 문 앞의 경비원(보안 그룹)"과 "단지 입구의 출입 게이트(네트워크 ACL)" 두 겹으로 트래픽을 걸러냅니다.

### 직관적 이해 (Intuitive Understanding)

아파트 단지에 살고 있다고 상상해 봅시다. 외부인이 여러분의 집에 도달하려면 **두 번의 보안 검문**을 통과해야 합니다.

**첫 번째 검문 — 단지 입구 게이트(네트워크 ACL)**: 단지 입구에는 모든 방문자가 지나야 하는 게이트가 있습니다. 이 게이트에는 "출입 허용 목록"과 "출입 거부 목록"이 적힌 규칙표가 있습니다. 경비원은 이 목록을 위에서부터 순서대로 확인하며, 첫 번째로 일치하는 규칙에 따라 허용하거나 거부합니다. 중요한 점은, 들어올 때 검문을 통과했더라도 **나갈 때 다시 한번 별도로 검문**받아야 한다는 것입니다.

**두 번째 검문 — 각 세대 현관문(보안 그룹)**: 단지 게이트를 통과하더라도, 각 집의 현관문에는 별도의 잠금장치가 있습니다. 이 잠금장치는 "허용된 사람만 들어올 수 있다"는 **허용 목록만** 관리합니다. 그리고 한 번 들어온 사람이 나갈 때는 **자동으로 문이 열립니다** — 별도로 나가는 허가를 받을 필요가 없습니다.

이 비유가 바로 AWS 네트워크 보안의 핵심 구조입니다:

- **보안 그룹(Security Group)** = 각 세대 현관문 → **인스턴스(ENI) 수준**, **상태 유지(Stateful)**
- **네트워크 ACL(Network ACL)** = 단지 입구 게이트 → **서브넷 수준**, **상태 비유지(Stateless)**

> 🔗 **연결**: Chapter 7에서 VPC, 서브넷, 인터넷 게이트웨이의 구조를 배웠습니다. 보안 그룹과 네트워크 ACL은 이 네트워크 구조 위에서 트래픽을 필터링하는 보안 계층입니다.

왜 두 겹의 보안이 필요할까요? 이것은 Chapter 12에서 배운 **심층 방어(Defense in Depth)** 원칙의 실제 적용입니다. 한 계층이 뚫리더라도 다른 계층이 방어선을 유지합니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **상태 유지 방화벽(Stateful Firewall)**: 이전 트래픽의 상태를 기억하는 방화벽입니다. 허용된 인바운드 요청에 대한 응답 트래픽(아웃바운드)은 규칙에 관계없이 자동으로 허용됩니다.

> 📝 **상태 비유지 방화벽(Stateless Firewall)**: 각 패킷을 독립적으로 평가하는 방화벽입니다. 인바운드로 허용된 트래픽이라도, 아웃바운드 응답은 별도의 규칙에 의해 명시적으로 허용되어야 합니다.

> 📝 **인바운드(Inbound)**: 외부에서 내부로 들어오는 트래픽입니다. 예를 들어, 사용자가 웹 서버에 접속하는 요청입니다.

> 📝 **아웃바운드(Outbound)**: 내부에서 외부로 나가는 트래픽입니다. 예를 들어, 웹 서버가 사용자에게 응답을 보내는 것입니다.

이 두 보안 계층의 핵심 차이를 정리하면 다음과 같습니다:

| 비교 항목 | 보안 그룹 (Security Group) | 네트워크 ACL (Network ACL) |
|---|---|---|
| **적용 수준** | 인스턴스(ENI) 수준 | 서브넷 수준 |
| **상태** | 상태 유지 (Stateful) | 상태 비유지 (Stateless) |
| **규칙 유형** | 허용 규칙만 (Allow only) | 허용 + 거부 규칙 (Allow & Deny) |
| **규칙 평가** | 모든 규칙을 종합 평가 | 번호 순서대로 평가 (첫 일치 적용) |
| **기본 동작** | 모든 인바운드 거부, 모든 아웃바운드 허용 | 기본 ACL: 모든 트래픽 허용 |
| **적용 방식** | 인스턴스에 직접 연결 | 서브넷의 모든 인스턴스에 자동 적용 |
| **응답 트래픽** | 자동 허용 | 명시적 규칙 필요 |

### 상세 예시

> 🔍 **예시 14.1.1: 웹 서버에 대한 두 겹의 보안**
>
> **상황**: VPC의 퍼블릭 서브넷에 웹 서버(EC2 인스턴스)가 있습니다. 외부 사용자가 웹 페이지를 요청(HTTP, 포트 80)합니다.
>
> **트래픽 흐름**:
> 1. 사용자의 요청이 인터넷 게이트웨이를 통해 VPC에 도착합니다.
> 2. **1차 검문 — 네트워크 ACL**: 서브넷 입구에서 규칙을 확인합니다. "포트 80 인바운드 허용" 규칙이 있으므로 통과합니다.
> 3. **2차 검문 — 보안 그룹**: 인스턴스 수준에서 규칙을 확인합니다. "포트 80 인바운드 허용" 규칙이 있으므로 통과합니다.
> 4. 웹 서버가 응답을 보냅니다.
> 5. **보안 그룹(아웃바운드)**: 상태 유지이므로, 인바운드 요청에 대한 응답은 **자동 허용**됩니다. 별도 규칙 불필요!
> 6. **네트워크 ACL(아웃바운드)**: 상태 비유지이므로, 아웃바운드 규칙에서 **명시적으로 허용**되어야 합니다. "임시 포트(Ephemeral Port, 1024-65535) 아웃바운드 허용" 규칙이 필요합니다.
>
> **핵심 포인트**: 보안 그룹은 응답을 자동 허용하지만, 네트워크 ACL은 응답 트래픽도 별도 규칙이 필요합니다. 이것이 Stateful vs Stateless의 실제 차이입니다.

> 🔍 **예시 14.1.2: 일상 비유 — 도서관 출입과 체육관 출입**
>
> **상황**: 두 종류의 출입 관리 시스템을 비교해 봅시다.
>
> **도서관(보안 그룹처럼 Stateful)**: 입구에서 회원증을 보여주고 들어오면, 나갈 때는 그냥 문을 밀고 나갑니다. 도서관은 "이 사람이 정상적으로 들어왔다"는 사실을 기억하고 있기 때문입니다.
>
> **체육관(네트워크 ACL처럼 Stateless)**: 입구에서 회원증을 찍어야 들어갈 수 있고, 나갈 때도 회원증을 다시 찍어야 합니다. 체육관은 "이 사람이 들어왔었는지"를 기억하지 않으므로, 나갈 때도 별도로 신원을 확인합니다.
>
> **핵심 포인트**: "기억하느냐(Stateful), 기억하지 않느냐(Stateless)"가 핵심 차이입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "보안 그룹만 설정하면 충분하다"고 생각하는 것은 위험합니다. 네트워크 ACL은 서브넷 전체에 적용되는 추가 방어선입니다. 특히, **특정 IP를 명시적으로 차단**해야 할 때는 보안 그룹으로는 불가능합니다(허용 규칙만 있으므로). 이때 네트워크 ACL의 거부 규칙이 필요합니다.

> ⚠️ **주의**: 네트워크 ACL에서 인바운드만 허용하고 아웃바운드를 설정하지 않으면, 요청은 들어오지만 응답이 나가지 못합니다. Stateless이므로 양방향 규칙을 모두 설정해야 합니다.

### Section 요약

- AWS 네트워크 보안은 **보안 그룹(인스턴스 수준)**과 **네트워크 ACL(서브넷 수준)** 두 계층으로 구성됩니다.
- 보안 그룹은 **Stateful**(상태 유지)로 응답 트래픽을 자동 허용하며, **허용 규칙만** 설정합니다.
- 네트워크 ACL은 **Stateless**(상태 비유지)로 양방향 규칙이 모두 필요하며, **허용·거부 규칙 모두** 설정합니다.
- 두 계층을 함께 사용하는 것은 **심층 방어(Defense in Depth)** 원칙의 실제 적용입니다.
- 특정 IP를 차단해야 할 때는 네트워크 ACL의 거부 규칙을 사용합니다.

## 14.2 보안 그룹 심화 — 인바운드/아웃바운드 규칙, 보안 그룹 참조

> 💡 **한 줄 요약**: 보안 그룹은 "허용할 트래픽만 명시"하는 화이트리스트 방식의 방화벽이며, 다른 보안 그룹을 참조하여 동적으로 접근을 제어할 수 있습니다.

### 직관적 이해 (Intuitive Understanding)

보안 그룹을 좀 더 깊이 이해하기 위해, "초대장 시스템"으로 비유해 봅시다.

여러분이 파티를 열 때, 문 앞에 **"이 사람들만 들어올 수 있습니다"라는 허용 목록**을 붙여 놓습니다. 목록에 없는 사람은 자동으로 거부됩니다. 이것이 보안 그룹의 작동 방식입니다 — **명시적으로 허용하지 않은 모든 트래픽은 거부**됩니다.

그런데 재미있는 기능이 하나 있습니다. 목록에 특정 개인 이름이 아니라, **"VIP 클럽 회원증을 가진 사람은 모두 허용"**이라고 쓸 수 있습니다. VIP 클럽에 새 멤버가 추가되면, 별도로 목록을 수정하지 않아도 자동으로 파티에 입장할 수 있습니다. 이것이 바로 **보안 그룹 참조(Security Group Reference)** 기능입니다 — IP 주소 대신 다른 보안 그룹을 소스로 지정할 수 있습니다.

### 핵심 개념 설명 (Core Explanation)

**보안 그룹 규칙의 구성 요소**:

| 구성 요소 | 설명 | 예시 |
|---|---|---|
| **유형(Type)** | 프로토콜 종류 | HTTP, HTTPS, SSH, Custom TCP 등 |
| **프로토콜(Protocol)** | 통신 프로토콜 | TCP, UDP, ICMP |
| **포트 범위(Port Range)** | 허용할 포트 | 80, 443, 22, 3306 등 |
| **소스/대상(Source/Destination)** | 트래픽의 출발지(인바운드) 또는 목적지(아웃바운드) | IP 주소, CIDR 블록, 또는 **보안 그룹 ID** |

> 📝 **보안 그룹 참조(Security Group Reference)**: 보안 그룹의 소스(인바운드) 또는 대상(아웃바운드)에 IP 주소 대신 **다른 보안 그룹의 ID**를 지정하는 기능입니다. 해당 보안 그룹에 연결된 모든 인스턴스로부터의 트래픽을 허용합니다.

보안 그룹의 주요 특성을 정리하면:

1. **기본 규칙**: 새로 생성된 보안 그룹은 모든 인바운드 트래픽을 거부하고, 모든 아웃바운드 트래픽을 허용합니다.
2. **허용만 가능**: "거부" 규칙을 만들 수 없습니다. 허용 목록에 없으면 자동 거부됩니다.
3. **다중 연결**: 하나의 인스턴스에 **최대 5개의 보안 그룹**을 연결할 수 있으며, 모든 규칙이 합산되어 평가됩니다.
4. **상태 유지**: 인바운드로 허용된 트래픽의 응답은 아웃바운드 규칙에 관계없이 자동 허용됩니다.
5. **실시간 적용**: 규칙 변경 시 즉시 적용됩니다 (인스턴스 재시작 불필요).

### 상세 예시

> 🔍 **예시 14.2.1: 3계층 아키텍처의 보안 그룹 설계**
>
> **상황**: 웹 서버(프론트엔드), 애플리케이션 서버(백엔드), 데이터베이스 서버의 3계층 구조를 운영합니다. 각 계층은 바로 인접한 계층과만 통신해야 합니다.
>
> **설계**:
> - **SG-Web** (웹 서버용): 인바운드에 포트 80/443을 `0.0.0.0/0`(모든 IP)에서 허용
> - **SG-App** (앱 서버용): 인바운드에 포트 8080을 **SG-Web**에서 허용 (보안 그룹 참조!)
> - **SG-DB** (DB 서버용): 인바운드에 포트 3306을 **SG-App**에서 허용 (보안 그룹 참조!)
>
> ```
> [인터넷] → [SG-Web: 80/443 허용] → [SG-App: 8080, 소스=SG-Web] → [SG-DB: 3306, 소스=SG-App]
> ```
>
> **핵심 포인트**: 보안 그룹 참조를 사용하면 IP 주소를 하드코딩하지 않아도 됩니다. 웹 서버 인스턴스가 추가·삭제되어도, SG-Web에 연결되어 있기만 하면 앱 서버에 자동으로 접근할 수 있습니다. 또한, 외부에서 직접 데이터베이스에 접근하는 것은 불가능합니다 — SG-App에 연결된 인스턴스만 접근할 수 있습니다.

> 🔍 **예시 14.2.2: SSH 접근 제한**
>
> **상황**: EC2 인스턴스에 SSH로 접속해야 하지만, 전 세계 어디서든 접속 가능하게 하면 보안 위험이 큽니다.
>
> **잘못된 설정**: 인바운드 — SSH(포트 22) 소스: `0.0.0.0/0` (전 세계 모든 IP 허용)
> **올바른 설정**: 인바운드 — SSH(포트 22) 소스: `203.0.113.50/32` (내 IP 주소만 허용)
>
> **핵심 포인트**: `/32`는 정확히 하나의 IP 주소만 의미합니다. SSH 접근은 반드시 필요한 IP만 허용하는 것이 보안 모범 사례입니다. 이것은 Chapter 13에서 배운 **최소 권한 원칙**을 네트워크 수준에서 적용하는 것입니다.

> 🔗 **연결**: Chapter 13에서 배운 최소 권한 원칙은 IAM 정책뿐 아니라, 보안 그룹 규칙에서도 동일하게 적용됩니다. "필요한 포트만, 필요한 소스에서만" 허용하는 것이 핵심입니다.

### 미니 실습 (Hands-on Mini Lab)

> 🧪 **Mini Lab 14.2: 보안 그룹 규칙 확인 및 수정**
>
> **목표**: 기본 VPC의 기본 보안 그룹 규칙을 확인하고, 웹 서버용 보안 그룹을 새로 생성합니다.
> **예상 소요 시간**: 10분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS 콘솔에 로그인한 후, 상단 검색창에 "VPC"를 입력하고 **VPC** 서비스를 클릭합니다.
>
> **Step 2**: 좌측 메뉴에서 **보안 그룹(Security Groups)**을 클릭합니다. 기본 VPC에 이미 존재하는 **default** 보안 그룹을 확인합니다.
>
> **Step 3**: default 보안 그룹을 클릭하고, **인바운드 규칙(Inbound rules)** 탭을 확인합니다. 기본적으로 "같은 보안 그룹에서 오는 모든 트래픽"이 허용되어 있는 것을 볼 수 있습니다.
>
> **Step 4**: 이제 새 보안 그룹을 만들어 봅시다. 상단의 **보안 그룹 생성(Create security group)** 버튼을 클릭합니다.
> - **보안 그룹 이름**: `my-web-sg`
> - **설명**: `Allow HTTP and HTTPS from anywhere`
> - **VPC**: 기본 VPC 선택
>
> **Step 5**: **인바운드 규칙 추가**를 클릭하고 다음 규칙을 추가합니다:
> - 규칙 1: 유형 `HTTP`, 소스 `Anywhere-IPv4` (0.0.0.0/0)
> - 규칙 2: 유형 `HTTPS`, 소스 `Anywhere-IPv4` (0.0.0.0/0)
>
> **Step 6**: 아웃바운드 규칙은 기본값(모든 트래픽 허용)을 유지합니다. **보안 그룹 생성** 버튼을 클릭합니다.
>
> **확인**: 보안 그룹 목록에 `my-web-sg`가 나타나고, 인바운드 규칙에 HTTP(80)와 HTTPS(443)가 표시됩니다.
>
> 🧹 **정리**: 이 보안 그룹은 통합 실습에서 사용할 수 있으므로 당분간 유지합니다. 모든 실습이 끝나면 삭제합니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: 보안 그룹의 내부 동작**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 보안 그룹은 내부적으로 **연결 추적(Connection Tracking)** 메커니즘을 사용합니다. 인바운드 요청이 허용되면, 해당 연결의 정보(소스 IP, 소스 포트, 대상 IP, 대상 포트, 프로토콜)가 연결 추적 테이블에 기록됩니다. 응답 트래픽이 나갈 때, 이 테이블을 조회하여 "이미 허용된 연결의 응답"임을 확인하고 자동으로 통과시킵니다. 이것이 "Stateful"의 실제 구현입니다.
>
> 이 연결 추적에는 한계가 있습니다. 인스턴스당 추적 가능한 연결 수에 제한이 있으며(기본 약 25만 개), 이를 초과하면 새 연결이 거부될 수 있습니다. 대규모 트래픽을 처리하는 인스턴스에서는 이 제한을 인지해야 합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: 보안 그룹에서 SSH(포트 22) 소스를 `0.0.0.0/0`으로 설정하는 것은 매우 위험합니다. 전 세계에서 SSH 접속 시도가 가능해지며, 무차별 대입 공격의 대상이 됩니다. 반드시 필요한 IP 범위로 제한하세요.

> ⚠️ **주의**: 보안 그룹의 아웃바운드 규칙을 무조건 "모든 트래픽 허용"으로 두는 것은 편리하지만, 보안 요구사항이 높은 환경에서는 아웃바운드도 필요한 트래픽만 허용하는 것이 좋습니다.

### Section 요약

- 보안 그룹은 **허용 규칙만** 설정 가능하며, 명시적으로 허용되지 않은 모든 트래픽은 자동 거부됩니다.
- 규칙의 소스/대상에 IP 주소 외에 **다른 보안 그룹 ID를 참조**할 수 있어, 동적이고 유연한 접근 제어가 가능합니다.
- 하나의 인스턴스에 여러 보안 그룹을 연결할 수 있으며, 모든 규칙이 합산됩니다.
- SSH(22), RDP(3389) 등 관리 포트는 반드시 **최소한의 IP 범위**로 제한해야 합니다.
- 규칙 변경은 즉시 적용되며 인스턴스 재시작이 필요 없습니다.

## 14.3 네트워크 ACL 심화 — 서브넷 수준 방화벽, 규칙 번호와 평가 순서

> 💡 **한 줄 요약**: 네트워크 ACL은 서브넷의 "출입 게이트"로, 번호 순서대로 규칙을 평가하며 명시적 거부가 가능한 방화벽입니다.

### 직관적 이해 (Intuitive Understanding)

공항 보안 검색대를 떠올려 봅시다. 보안 검색대에는 여러 개의 체크 항목이 **번호 순서대로** 적힌 매뉴얼이 있습니다:

- 규칙 #100: 유효한 항공권 소지자 → **통과 허용**
- 규칙 #200: 블랙리스트 국가 여권 소지자 → **통과 거부**
- 규칙 #300: VIP 라운지 회원 → **통과 허용**
- 규칙 * (마지막): 위 어디에도 해당하지 않으면 → **통과 거부**

검사관은 **위에서부터 순서대로** 확인하다가, 처음으로 일치하는 규칙에 따라 즉시 판단합니다. 나머지 규칙은 확인하지 않습니다. 그래서 규칙의 **순서(번호)**가 매우 중요합니다.

이것이 네트워크 ACL의 작동 방식입니다. 보안 그룹과 달리, 네트워크 ACL에서는:
- **허용과 거부** 모두 가능합니다.
- 규칙에 **번호**가 있고, **낮은 번호부터 순서대로 평가**합니다.
- **첫 번째 일치 규칙**에서 즉시 결정됩니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **네트워크 ACL(Network Access Control List)**: VPC의 서브넷 수준에서 인바운드·아웃바운드 트래픽을 제어하는 방화벽입니다. 각 서브넷에는 반드시 하나의 네트워크 ACL이 연결됩니다.

**네트워크 ACL 규칙의 구성 요소**:

| 구성 요소 | 설명 | 예시 |
|---|---|---|
| **규칙 번호(Rule Number)** | 평가 순서 (낮을수록 먼저) | 100, 200, 300 |
| **유형(Type)** | 프로토콜 종류 | HTTP, HTTPS, SSH, Custom TCP |
| **프로토콜** | TCP, UDP, ICMP 등 | TCP(6), UDP(17) |
| **포트 범위** | 허용/거부할 포트 | 80, 443, 1024-65535 |
| **소스/대상** | CIDR 블록 | 0.0.0.0/0, 10.0.1.0/24 |
| **허용/거부** | Allow 또는 Deny | ALLOW, DENY |

**기본 네트워크 ACL vs 사용자 정의 네트워크 ACL**:

- **기본 네트워크 ACL(Default NACL)**: VPC 생성 시 자동으로 생성됩니다. 기본적으로 **모든 인바운드·아웃바운드 트래픽을 허용**합니다. 즉, 방화벽 역할을 거의 하지 않습니다.
- **사용자 정의 네트워크 ACL(Custom NACL)**: 직접 생성한 네트워크 ACL입니다. 기본적으로 **모든 트래픽을 거부**하며, 필요한 규칙을 직접 추가해야 합니다.

> 📝 **임시 포트(Ephemeral Port)**: 클라이언트가 서버에 요청을 보낼 때, 응답을 받기 위해 임시로 사용하는 높은 번호의 포트(일반적으로 1024~65535)입니다. 네트워크 ACL에서 아웃바운드 응답을 허용하려면 이 범위의 포트를 열어야 합니다.

### 상세 예시

> 🔍 **예시 14.3.1: 규칙 순서의 중요성**
>
> **상황**: 특정 IP(192.168.1.100)에서의 접근을 차단하면서, 나머지 모든 IP에서의 HTTP 접근은 허용하고 싶습니다.
>
> **올바른 설정**:
> | 규칙 # | 유형 | 포트 | 소스 | 허용/거부 |
> |---|---|---|---|---|
> | 100 | HTTP | 80 | 192.168.1.100/32 | **DENY** |
> | 200 | HTTP | 80 | 0.0.0.0/0 | ALLOW |
> | * | 모든 트래픽 | 전체 | 0.0.0.0/0 | DENY |
>
> **잘못된 설정** (규칙 순서가 반대):
> | 규칙 # | 유형 | 포트 | 소스 | 허용/거부 |
> |---|---|---|---|---|
> | 100 | HTTP | 80 | 0.0.0.0/0 | ALLOW |
> | 200 | HTTP | 80 | 192.168.1.100/32 | **DENY** |
>
> **풀이**: 잘못된 설정에서는 규칙 #100에서 모든 IP의 HTTP를 허용하므로, 192.168.1.100도 규칙 #100에서 이미 허용됩니다. 규칙 #200의 거부는 절대 평가되지 않습니다!
>
> **핵심 포인트**: 네트워크 ACL에서 **특정 IP를 거부하는 규칙은 반드시 일반 허용 규칙보다 낮은 번호(높은 우선순위)**에 배치해야 합니다.

> 🔍 **예시 14.3.2: 네트워크 ACL의 양방향 규칙 설정**
>
> **상황**: 퍼블릭 서브넷에서 HTTP 웹 서비스를 운영하려고 합니다.
>
> **인바운드 규칙**:
> | 규칙 # | 유형 | 포트 | 소스 | 허용/거부 |
> |---|---|---|---|---|
> | 100 | HTTP | 80 | 0.0.0.0/0 | ALLOW |
> | 110 | HTTPS | 443 | 0.0.0.0/0 | ALLOW |
> | * | 모든 트래픽 | 전체 | 0.0.0.0/0 | DENY |
>
> **아웃바운드 규칙**:
> | 규칙 # | 유형 | 포트 | 대상 | 허용/거부 |
> |---|---|---|---|---|
> | 100 | Custom TCP | 1024-65535 | 0.0.0.0/0 | ALLOW |
> | * | 모든 트래픽 | 전체 | 0.0.0.0/0 | DENY |
>
> **핵심 포인트**: 아웃바운드에서 임시 포트(1024-65535)를 허용해야 클라이언트에게 응답을 보낼 수 있습니다. 이것이 Stateless 방화벽에서 반드시 기억해야 할 설정입니다.

### 미니 실습 (Hands-on Mini Lab)

> 🧪 **Mini Lab 14.3: 네트워크 ACL 규칙 확인**
>
> **목표**: 기본 VPC의 기본 네트워크 ACL 규칙을 확인하고, 규칙 구조를 이해합니다.
> **예상 소요 시간**: 5분
> **비용**: Free Tier 범위 내
>
> **Step 1**: AWS 콘솔 상단 검색창에 "VPC"를 입력하고 **VPC** 서비스를 클릭합니다.
>
> **Step 2**: 좌측 메뉴에서 **네트워크 ACL(Network ACLs)**을 클릭합니다.
>
> **Step 3**: 기본 VPC에 연결된 네트워크 ACL(Default 열이 "예"인 항목)을 클릭합니다.
>
> **Step 4**: **인바운드 규칙(Inbound rules)** 탭을 클릭합니다. 다음과 같은 규칙이 보여야 합니다:
> - 규칙 #100: 모든 트래픽 → ALLOW
> - 규칙 *: 모든 트래픽 → DENY
>
> **Step 5**: **아웃바운드 규칙(Outbound rules)** 탭도 확인합니다. 인바운드와 동일한 구조(#100 ALLOW, * DENY)가 보여야 합니다.
>
> **Step 6**: **서브넷 연결(Subnet associations)** 탭을 클릭하여, 이 기본 네트워크 ACL이 기본 VPC의 모든 서브넷에 연결되어 있는 것을 확인합니다.
>
> **확인**: 기본 네트워크 ACL은 규칙 #100에서 모든 트래픽을 허용하므로, 사실상 필터링을 하지 않는 상태입니다. 이것이 기본 동작입니다.
>
> 🧹 **정리**: 기본 네트워크 ACL은 시스템이 관리하는 리소스이므로 삭제할 필요가 없습니다. 규칙을 수정한 경우에만 원래대로 복구합니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: 규칙 번호 설계 모범 사례**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> AWS는 규칙 번호를 100 단위로 설정하는 것을 권장합니다(100, 200, 300...). 그 이유는 나중에 **기존 규칙 사이에 새 규칙을 삽입**할 여지를 남기기 위해서입니다. 예를 들어, 규칙 #100과 #200 사이에 급히 #150을 추가할 수 있습니다. 만약 1, 2, 3으로 번호를 매겼다면, 중간에 삽입이 불가능합니다.
>
> 또한, 거부 규칙은 허용 규칙보다 **낮은 번호(높은 우선순위)**에 배치하는 것이 일반적인 패턴입니다. 이는 "먼저 거부할 것을 거부하고, 나머지를 허용"하는 방식으로, 보안 관점에서 더 안전합니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: 사용자 정의 네트워크 ACL을 생성하면 **기본적으로 모든 트래픽이 거부**됩니다. 규칙을 추가하지 않고 서브넷에 연결하면, 해당 서브넷의 모든 통신이 차단됩니다. 반드시 필요한 허용 규칙을 먼저 추가한 후 연결하세요.

> ⚠️ **주의**: 네트워크 ACL의 마지막 규칙(규칙 번호 `*`)은 **수정하거나 삭제할 수 없습니다**. 이 규칙은 항상 모든 트래픽을 거부하며, 다른 어떤 규칙에도 일치하지 않는 트래픽의 "최종 방어선" 역할을 합니다.

### Section 요약

- 네트워크 ACL은 **서브넷 수준**에서 작동하며, 해당 서브넷의 **모든 인스턴스에 자동 적용**됩니다.
- 규칙은 **번호 순서대로 평가**되며, **첫 번째 일치 규칙에서 즉시 결정**됩니다.
- **허용(ALLOW)과 거부(DENY)** 모두 가능하여, 특정 IP 차단에 유용합니다.
- 기본 네트워크 ACL은 모든 트래픽을 허용하지만, 사용자 정의 ACL은 모든 트래픽을 거부합니다.
- Stateless이므로 **아웃바운드에 임시 포트(1024-65535) 허용 규칙**이 필요합니다.

## 14.4 데이터 암호화 기초 — 전송 중 암호화(SSL/TLS)와 저장 시 암호화

> 💡 **한 줄 요약**: 데이터 보호는 "이동 중(전송 중 암호화)"과 "보관 중(저장 시 암호화)" 두 상태에서 모두 이루어져야 합니다.

### 직관적 이해 (Intuitive Understanding)

귀중한 보석을 보관하고 운반하는 상황을 생각해 봅시다.

**보관 중 보호 (저장 시 암호화)**: 보석을 금고에 넣고 잠급니다. 금고 열쇠가 없는 사람은 보석을 꺼낼 수 없습니다. 누군가 금고를 통째로 훔쳐도, 열쇠 없이는 보석에 접근할 수 없습니다. 이것이 **저장 시 암호화(Encryption at Rest)** 입니다 — 데이터가 디스크(EBS, S3 등)에 저장될 때 암호화됩니다.

**이동 중 보호 (전송 중 암호화)**: 보석을 다른 장소로 운반할 때, 방탄 수송 차량을 사용합니다. 운반 중에 누군가 가로채려 해도, 방탄차를 뚫을 수 없습니다. 이것이 **전송 중 암호화(Encryption in Transit)** 입니다 — 데이터가 네트워크를 통해 이동할 때 암호화됩니다.

왜 두 가지가 모두 필요할까요?

- 전송 중에만 암호화하면: 목적지에 도착한 후 저장된 데이터는 평문이므로, 스토리지가 탈취되면 데이터가 노출됩니다.
- 저장 시에만 암호화하면: 네트워크를 통해 이동하는 동안에는 평문이므로, 중간에서 도청(Man-in-the-Middle)이 가능합니다.

따라서, 완전한 데이터 보호는 **두 상태 모두에서 암호화**가 필요합니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **전송 중 암호화(Encryption in Transit)**: 데이터가 네트워크를 통해 한 지점에서 다른 지점으로 이동하는 동안 암호화하는 것입니다. SSL/TLS 프로토콜이 가장 대표적입니다.

> 📝 **저장 시 암호화(Encryption at Rest)**: 데이터가 스토리지(하드 디스크, SSD, S3 버킷 등)에 저장되어 있는 동안 암호화하는 것입니다.

> 📝 **SSL/TLS(Secure Sockets Layer / Transport Layer Security)**: 네트워크 통신을 암호화하는 프로토콜입니다. SSL은 TLS의 이전 버전이며, 현재는 TLS가 사용되지만 관습적으로 "SSL"이라는 용어가 함께 사용됩니다. 웹 브라우저에서 `https://`로 시작하는 주소가 바로 TLS가 적용된 통신입니다.

**전송 중 암호화가 적용되는 AWS 사례**:

| 서비스/상황 | 전송 중 암호화 방법 |
|---|---|
| 사용자 → AWS 콘솔/API | HTTPS (TLS) |
| EC2 인스턴스 간 통신 | VPN 또는 TLS |
| S3 업로드/다운로드 | HTTPS 엔드포인트 사용 |
| RDS 데이터베이스 연결 | SSL/TLS 연결 옵션 |
| CloudFront 배포 | HTTPS 뷰어 프로토콜 정책 |

**저장 시 암호화가 적용되는 AWS 사례**:

| 서비스 | 저장 시 암호화 방법 |
|---|---|
| **S3** | SSE-S3, SSE-KMS, SSE-C (서버 측 암호화) |
| **EBS** | EBS 암호화 (AES-256) |
| **RDS** | RDS 암호화 옵션 활성화 |
| **DynamoDB** | 기본 암호화 (항상 활성화) |
| **EFS** | EFS 암호화 옵션 |

> 📝 **서버 측 암호화(Server-Side Encryption, SSE)**: AWS가 데이터를 저장하기 직전에 자동으로 암호화하고, 읽을 때 자동으로 복호화합니다. 사용자는 암호화/복호화 과정을 신경 쓸 필요가 없습니다.

> 📝 **클라이언트 측 암호화(Client-Side Encryption)**: 사용자가 데이터를 AWS로 보내기 전에 직접 암호화합니다. AWS는 이미 암호화된 데이터를 저장하므로, AWS조차도 원본 데이터를 볼 수 없습니다.

### 상세 예시

> 🔍 **예시 14.4.1: HTTPS와 HTTP의 차이**
>
> **상황**: 온라인 쇼핑몰에서 신용카드 번호를 입력하여 결제합니다.
>
> **HTTP(암호화 없음)**: 신용카드 번호가 평문 그대로 네트워크를 통해 전송됩니다. 마치 엽서에 신용카드 번호를 적어 보내는 것과 같습니다 — 배달 과정에서 누구나 읽을 수 있습니다.
>
> **HTTPS(TLS 암호화)**: 신용카드 번호가 암호화되어 전송됩니다. 마치 밀봉된 편지에 넣어 보내는 것과 같습니다 — 중간에 가로채도 내용을 알 수 없습니다.
>
> **핵심 포인트**: 웹 브라우저 주소창의 자물쇠 아이콘(🔒)은 TLS가 적용되어 전송 중 암호화가 활성화되어 있다는 의미입니다.

> 🔍 **예시 14.4.2: S3 버킷의 저장 시 암호화**
>
> **상황**: 고객 개인정보가 포함된 CSV 파일을 S3에 업로드해야 합니다.
>
> **암호화 없이 저장**: 파일이 S3의 물리적 디스크에 평문으로 저장됩니다. AWS 데이터센터의 디스크가 물리적으로 탈취되면(극히 드물지만) 데이터가 노출될 수 있습니다.
>
> **SSE-S3로 암호화하여 저장**: 파일이 S3에 도달하면, AWS가 자동으로 AES-256 알고리즘으로 암호화하여 디스크에 저장합니다. 읽을 때는 자동으로 복호화됩니다. 디스크가 탈취되어도 암호화된 데이터만 존재하므로 안전합니다.
>
> **핵심 포인트**: S3의 기본 암호화(Default Encryption) 설정을 활성화하면, 업로드되는 모든 객체가 자동으로 암호화됩니다. 2023년 1월부터 S3는 기본적으로 SSE-S3 암호화가 활성화되어 있습니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: "저장 시 암호화를 적용했으니 데이터가 안전하다"고 생각하는 것은 위험합니다. 저장 시 암호화는 물리적 디스크 탈취에 대한 보호이지, **접근 권한이 있는 사용자에 대한 보호가 아닙니다**. 권한이 있는 사용자는 암호화된 데이터도 정상적으로 복호화하여 읽을 수 있습니다. 따라서 암호화와 함께 Chapter 13에서 배운 **IAM 접근 제어**도 반드시 적용해야 합니다.

> ⚠️ **주의**: "암호화"와 "해싱"은 다릅니다. 암호화는 원본을 복구(복호화)할 수 있지만, 해싱은 일방향으로 원본 복구가 불가능합니다. AWS의 데이터 보호에서 말하는 암호화는 복호화가 가능한 **양방향 암호화**입니다.

### Section 요약

- 데이터 보호는 **전송 중 암호화**와 **저장 시 암호화** 두 가지를 모두 적용해야 완전합니다.
- 전송 중 암호화는 주로 **SSL/TLS(HTTPS)** 프로토콜을 사용합니다.
- 저장 시 암호화는 **서버 측 암호화(SSE)**와 **클라이언트 측 암호화**로 나뉩니다.
- S3는 2023년부터 기본적으로 SSE-S3 암호화가 활성화되어 있습니다.
- 암호화는 접근 제어(IAM)와 함께 사용해야 진정한 데이터 보호가 됩니다.

## 14.5 AWS KMS와 CloudHSM — 키 관리 서비스, 고객 관리 키, 하드웨어 보안 모듈

> 💡 **한 줄 요약**: AWS KMS는 암호화 키를 쉽고 안전하게 관리하는 서비스이고, CloudHSM은 전용 하드웨어에서 키를 관리하는 고보안 옵션입니다.

### 직관적 이해 (Intuitive Understanding)

Section 14.4에서 암호화에 대해 배웠습니다. 그런데 암호화에는 항상 **"열쇠(키)"**가 필요합니다. 금고에 보석을 넣으려면 금고 열쇠가 필요한 것처럼, 데이터를 암호화하려면 암호화 키가 필요합니다.

그렇다면 이 열쇠는 어디에 보관할까요? 열쇠를 금고 옆에 두면 의미가 없고, 열쇠를 잃어버리면 금고를 영영 열 수 없습니다. **암호화에서 가장 어려운 문제는 데이터 암호화 자체가 아니라, 암호화 키를 안전하게 관리하는 것**입니다.

이 문제를 해결하는 AWS 서비스가 바로:

- **AWS KMS(Key Management Service)**: "은행의 대여 금고" 같은 서비스입니다. 은행(AWS)이 금고 시설을 관리하고, 여러분은 금고(키)를 사용합니다. 편리하고 대부분의 상황에 적합합니다.
- **AWS CloudHSM**: "개인 전용 금고실"과 같습니다. 여러분만 접근할 수 있는 전용 하드웨어 장비에서 키를 관리합니다. AWS조차 키에 접근할 수 없습니다. 극도로 높은 보안이 필요한 경우에 사용합니다.
- **AWS Certificate Manager(ACM)**: "인증서 발급 대행" 서비스입니다. SSL/TLS 인증서를 무료로 발급하고 자동 갱신해 줍니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS KMS (Key Management Service)

> 📝 **AWS KMS(Key Management Service)**: 암호화 키의 생성, 저장, 관리, 사용을 중앙에서 관리하는 완전 관리형 서비스입니다. 대부분의 AWS 서비스(S3, EBS, RDS 등)와 통합되어 있습니다.

KMS에서 사용하는 키의 종류:

| 키 종류 | 설명 | 관리 주체 | 비용 |
|---|---|---|---|
| **AWS 소유 키(AWS Owned Key)** | AWS가 내부적으로 관리하는 키. 사용자에게 보이지 않음 | AWS | 무료 |
| **AWS 관리형 키(AWS Managed Key)** | AWS 서비스가 자동 생성·관리하는 키 (예: `aws/s3`, `aws/ebs`) | AWS (사용자 조회 가능) | 무료 |
| **고객 관리형 키(Customer Managed Key, CMK)** | 사용자가 직접 생성·관리하는 키. 키 정책, 교체 주기 등을 직접 제어 | 사용자 | 월 $1/키 |

> 📝 **고객 관리형 키(Customer Managed Key, CMK)**: 사용자가 직접 생성하고, 키 정책(누가 이 키를 사용할 수 있는지)과 키 교체 주기를 직접 제어하는 KMS 키입니다. 가장 세밀한 제어가 가능합니다.

KMS의 핵심 특징:

1. **중앙 집중 관리**: 모든 암호화 키를 한 곳에서 관리합니다.
2. **AWS 서비스 통합**: S3, EBS, RDS, Lambda 등 대부분의 AWS 서비스와 직접 연동됩니다.
3. **키 정책**: IAM 정책과 유사하게, 누가 키를 사용할 수 있는지 JSON으로 정의합니다.
4. **자동 키 교체**: 고객 관리형 키의 경우, 매년 자동으로 키를 교체하도록 설정할 수 있습니다.
5. **감사 추적**: 키 사용 이력이 AWS CloudTrail에 자동으로 기록됩니다.
6. **리전 한정**: KMS 키는 생성된 리전에서만 사용할 수 있습니다. 다른 리전에서 사용하려면 해당 리전에 새 키를 생성해야 합니다.

> 🔗 **연결**: Chapter 13에서 배운 IAM 정책은 "누가 어떤 AWS 리소스에 접근할 수 있는가"를 제어합니다. KMS 키 정책은 이와 유사하게 "누가 이 암호화 키를 사용할 수 있는가"를 제어합니다. 두 가지가 함께 작동하여, 데이터에 대한 접근과 암호화 키에 대한 접근을 이중으로 제어합니다.

#### AWS CloudHSM

> 📝 **AWS CloudHSM(Cloud Hardware Security Module)**: AWS 클라우드 내에서 고객 전용 하드웨어 보안 모듈(HSM)을 제공하는 서비스입니다. HSM은 암호화 키를 생성하고 저장하는 전용 하드웨어 장치입니다.

| 비교 항목 | KMS | CloudHSM |
|---|---|---|
| **키 관리** | AWS와 고객이 공유 관리 (키 유형에 따라) | 고객이 완전 단독 관리 |
| **하드웨어** | AWS 공유 인프라 | 고객 전용 HSM 장비 |
| **AWS의 키 접근** | AWS 관리형 키의 경우 가능 | **불가능** (고객만 접근) |
| **비용** | 키당 월 $1 + API 호출 비용 | 시간당 약 $1.45 (~월 $1,044) |
| **규정 준수** | 대부분의 요구사항 충족 | FIPS 140-2 Level 3 인증 |
| **사용 사례** | 일반적인 암호화 요구사항 | 금융, 의료 등 극도의 보안 요구 |

#### AWS Certificate Manager (ACM)

> 📝 **AWS Certificate Manager(ACM)**: SSL/TLS 인증서를 프로비저닝, 관리, 배포하는 서비스입니다. ACM이 발급한 인증서는 무료이며, 만료 전 자동으로 갱신됩니다.

ACM의 핵심 특징:

- **무료 인증서**: ACM에서 발급하는 공용 SSL/TLS 인증서는 무료입니다.
- **자동 갱신**: 인증서 만료 전 자동으로 갱신되어, 인증서 만료로 인한 장애를 방지합니다.
- **AWS 서비스 통합**: Elastic Load Balancer(ELB), CloudFront, API Gateway 등에 직접 연결할 수 있습니다.
- **EC2에 직접 설치 불가**: ACM 인증서는 ELB, CloudFront 등 AWS 관리형 서비스에서만 사용할 수 있으며, EC2 인스턴스에 직접 설치하는 것은 지원되지 않습니다.

### 상세 예시

> 🔍 **예시 14.5.1: KMS를 사용한 S3 데이터 암호화**
>
> **상황**: 회사의 고객 데이터를 S3에 저장할 때, 회사 자체 키로 암호화하고, 누가 키를 사용했는지 감사 로그를 남기고 싶습니다.
>
> **풀이**:
> 1. KMS에서 **고객 관리형 키(CMK)**를 생성합니다.
> 2. S3 버킷의 기본 암호화 설정에서 **SSE-KMS**를 선택하고, 방금 만든 CMK를 지정합니다.
> 3. 이제 이 버킷에 업로드되는 모든 객체는 해당 CMK로 자동 암호화됩니다.
> 4. 키 사용 이력은 CloudTrail에 자동 기록되어, "누가, 언제 데이터를 복호화했는지" 감사할 수 있습니다.
>
> **핵심 포인트**: 고객 관리형 키(CMK)를 사용하면 키 정책으로 "누가 이 키를 사용할 수 있는가"를 제어할 수 있습니다. IAM 정책으로 S3 버킷 접근 권한이 있더라도, KMS 키 사용 권한이 없으면 데이터를 복호화할 수 없습니다 — 이중 보호입니다.

> 🔍 **예시 14.5.2: ACM으로 웹사이트에 HTTPS 적용**
>
> **상황**: 회사 웹사이트(example.com)에 HTTPS를 적용하고 싶습니다.
>
> **풀이**:
> 1. ACM에서 `example.com` 도메인에 대한 SSL/TLS 인증서를 요청합니다 (무료).
> 2. 도메인 소유 확인을 완료하면 인증서가 발급됩니다.
> 3. Application Load Balancer(ALB)의 HTTPS 리스너에 이 인증서를 연결합니다.
> 4. 이제 사용자가 `https://example.com`으로 접속하면 TLS로 암호화된 통신이 이루어집니다.
> 5. 인증서 만료 전, ACM이 자동으로 갱신합니다.
>
> **핵심 포인트**: ACM을 사용하면 SSL/TLS 인증서 관리의 복잡함(구매, 설치, 갱신)을 모두 AWS에 맡길 수 있습니다. 비용도 무료입니다.

> 🧪 **Mini Lab 14.5: KMS 키 생성 및 확인**
>
> **목표**: KMS에서 고객 관리형 키(CMK)를 생성하고, 키의 구성 요소를 이해합니다.
> **예상 소요 시간**: 10분
> **비용**: ⚠️ 비용 발생 (월 $1/키). 실습 후 키 삭제 예약을 반드시 수행하세요.
>

>
> **Step 2**: 좌측 메뉴에서 **고객 관리형 키(Customer managed keys)**를 클릭합니다.
>

> - **키 유형**: 대칭(Symmetric) 선택
> - **키 사용**: 암호화 및 복호화(Encrypt and decrypt) 선택
> - **다음(Next)** 클릭
>
> **Step 4**: 키 별칭과 설명을 입력합니다.
> - **별칭(Alias)**: `my-test-key`
> - **설명**: `Test key for Chapter 14 lab`
> - **다음(Next)** 클릭
>
> **Step 5**: 키 관리자(Key administrators)를 설정합니다. 현재 로그인한 IAM 사용자를 선택하고 **다음** 클릭합니다.
>
> **Step 6**: 키 사용자(Key users)를 설정합니다. 동일한 IAM 사용자를 선택하고 **다음** 클릭합니다.
>
> **Step 7**: 키 정책(JSON)을 검토하고 **완료(Finish)** 버튼을 클릭합니다.
>
> **확인**: 키 목록에 `my-test-key`가 나타나고, 상태가 **활성(Enabled)**인 것을 확인합니다. 키를 클릭하면 키 ID, ARN, 생성 날짜, 키 정책 등을 확인할 수 있습니다.
>
> 🧹 **정리**: 비용 발생을 방지하기 위해 키 삭제를 예약합니다.
> 1. 생성한 키(`my-test-key`)를 선택합니다.
> 2. **키 작업(Key actions)** → **키 삭제 예약(Schedule key deletion)**을 클릭합니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: 봉투 암호화(Envelope Encryption)**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> KMS는 실제로 **봉투 암호화(Envelope Encryption)**라는 방식을 사용합니다. 직접 데이터를 KMS 키로 암호화하는 것이 아니라, 다음과 같은 2단계 과정을 거칩니다:
>
> 1. KMS에 **데이터 키(Data Key)**를 요청합니다.
> 2. KMS는 평문 데이터 키와 암호화된 데이터 키를 반환합니다.
> 3. 평문 데이터 키로 **실제 데이터를 암호화**합니다.
> 4. 평문 데이터 키는 메모리에서 삭제하고, **암호화된 데이터 키를 암호화된 데이터와 함께 저장**합니다.
>
> 복호화 시에는 암호화된 데이터 키를 KMS에 보내 평문 데이터 키를 얻고, 이를 사용해 데이터를 복호화합니다.
>
> 왜 이런 방식을 사용할까요? KMS API를 통해 직접 암호화할 수 있는 데이터 크기는 4KB로 제한됩니다. 봉투 암호화를 사용하면, KMS는 작은 데이터 키만 처리하고, 대용량 데이터의 암호화는 로컬에서 수행하므로 성능과 비용 면에서 효율적입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: KMS 키를 삭제하면, 해당 키로 암호화된 **모든 데이터를 영영 복호화할 수 없습니다**. 이것이 KMS 키 삭제에 최소 7일의 대기 기간이 있는 이유입니다. 키 삭제는 매우 신중하게 결정해야 합니다.

> ⚠️ **주의**: KMS 키는 **리전에 한정**됩니다. 서울 리전(ap-northeast-2)에서 만든 키를 미국 동부(us-east-1) 리전의 S3 버킷 암호화에 사용할 수 없습니다. 다중 리전 환경에서는 각 리전에 키를 생성하거나, KMS의 다중 리전 키(Multi-Region Key) 기능을 사용해야 합니다.

### Section 요약

- **AWS KMS**는 암호화 키를 중앙에서 관리하는 서비스로, 대부분의 AWS 서비스와 통합됩니다.
- KMS 키는 AWS 소유, AWS 관리형, **고객 관리형(CMK)** 세 종류이며, CMK가 가장 세밀한 제어를 제공합니다.
- **AWS CloudHSM**은 전용 하드웨어에서 키를 관리하며, AWS조차 키에 접근할 수 없는 최고 수준의 보안을 제공합니다.
- **AWS ACM**은 SSL/TLS 인증서를 무료로 발급·관리하며, ELB, CloudFront 등과 통합됩니다.
- KMS 키 삭제는 되돌릴 수 없으므로 매우 신중해야 합니다.

## 14.6 AWS WAF와 Shield — 웹 애플리케이션 방화벽, DDoS 보호 전략

> 💡 **한 줄 요약**: AWS WAF는 악성 웹 요청을 필터링하는 방화벽이고, AWS Shield는 DDoS 공격을 자동으로 방어하는 서비스입니다.

### 직관적 이해 (Intuitive Understanding)

유명 레스토랑을 운영한다고 상상해 봅시다. 두 가지 위협에 대비해야 합니다:

**첫 번째 위협 — 몰래 침입하는 도둑 (웹 공격)**: 손님으로 위장하여 들어온 후, 주방에 몰래 들어가거나 금고를 열려고 시도합니다. 이를 막기 위해 **보안 요원(AWS WAF)**을 배치합니다. 보안 요원은 "수상한 행동 패턴"을 감지합니다. 예를 들어, "주방 출입문을 반복적으로 두드리는 사람", "직원 전용 통로에 접근하려는 사람"을 식별하여 출입을 차단합니다.

**두 번째 위협 — 수천 명이 동시에 몰려오는 공격 (DDoS)**: 악의적인 경쟁자가 수천 명의 사람을 고용하여 레스토랑에 동시에 몰려오게 합니다. 아무것도 주문하지 않고 자리만 차지하여, 진짜 손님이 들어올 수 없게 만듭니다. 이를 막기 위해 **경비 시스템(AWS Shield)**을 설치합니다. 이 시스템은 "비정상적으로 많은 인원이 동시에 몰려오는 패턴"을 자동으로 감지하고 차단합니다.

> 📝 **DDoS(Distributed Denial of Service, 분산 서비스 거부 공격)**: 다수의 컴퓨터(봇넷)에서 동시에 대량의 트래픽을 보내, 서비스를 마비시키는 공격입니다. 정상적인 사용자가 서비스에 접근할 수 없게 만드는 것이 목적입니다.

### 핵심 개념 설명 (Core Explanation)

#### AWS WAF (Web Application Firewall)

> 📝 **AWS WAF(Web Application Firewall)**: 웹 애플리케이션으로 들어오는 HTTP/HTTPS 요청을 검사하고, 정의된 규칙에 따라 허용·차단·모니터링하는 방화벽 서비스입니다.

WAF가 방어하는 대표적인 웹 공격:

| 공격 유형 | 설명 | WAF 방어 방식 |
|---|---|---|
| **SQL Injection** | 데이터베이스 쿼리에 악성 SQL 코드를 삽입 | SQL Injection 패턴 탐지·차단 |
| **XSS (Cross-Site Scripting)** | 웹 페이지에 악성 스크립트를 삽입 | 스크립트 패턴 탐지·차단 |
| **악성 봇** | 자동화된 스크래핑, 무차별 로그인 시도 | 봇 제어 규칙, 속도 제한 |
| **특정 IP 차단** | 알려진 악성 IP에서의 접근 | IP 기반 차단 규칙 |
| **지역 기반 차단** | 특정 국가에서의 접근 차단 | Geo 매칭 규칙 |

WAF의 구성 요소:

- **Web ACL (Web Access Control List)**: WAF 규칙의 집합입니다. ALB, CloudFront, API Gateway 등에 연결합니다.
- **규칙(Rules)**: 각 요청을 검사하는 조건과 동작(허용/차단/카운트)을 정의합니다.
- **규칙 그룹(Rule Groups)**: 여러 규칙을 묶은 것입니다. AWS 관리형 규칙 그룹(무료 포함)과 AWS Marketplace의 서드파티 규칙 그룹을 사용할 수 있습니다.

> 💬 **참고**: AWS 관리형 규칙(AWS Managed Rules)은 AWS가 사전에 만들어 둔 규칙 세트로, SQL Injection, XSS 등 일반적인 웹 공격을 별도 설정 없이 바로 방어할 수 있습니다. 대부분 무료로 제공됩니다.

#### AWS Shield

> 📝 **AWS Shield**: AWS 리소스를 DDoS 공격으로부터 보호하는 관리형 서비스입니다.

AWS Shield는 두 가지 티어로 제공됩니다:

| 비교 항목 | Shield Standard | Shield Advanced |
|---|---|---|
| **비용** | **무료** (자동 활성화) | 월 $3,000 + 데이터 전송 비용 |
| **보호 범위** | Layer 3/4 (네트워크/전송 계층) DDoS 방어 | Layer 3/4/7 (애플리케이션 계층 포함) DDoS 방어 |
| **적용 대상** | 모든 AWS 고객 자동 적용 | CloudFront, Route 53, ELB, EC2, Global Accelerator |
| **DDoS 대응 팀** | 없음 | 24/7 AWS DDoS 대응 팀(SRT) 지원 |
| **비용 보호** | 없음 | DDoS로 인한 스케일링 비용 크레딧 |
| **실시간 모니터링** | 기본 | 상세 공격 진단 및 실시간 알림 |

핵심 포인트:
- **Shield Standard는 이미 여러분의 AWS 계정에 무료로 활성화되어 있습니다.** 별도 설정 없이 기본적인 DDoS 보호를 받고 있습니다.
- Shield Advanced는 대기업, 금융기관 등 높은 수준의 DDoS 보호가 필요한 경우에 사용합니다.

### 상세 예시

> 🔍 **예시 14.6.1: WAF로 SQL Injection 방어**
>
> **상황**: 웹사이트의 로그인 페이지에서 공격자가 다음과 같은 SQL Injection 시도를 합니다:
> - 사용자명 입력: `admin' OR '1'='1`
> - 이 입력이 서버에서 처리되면, 데이터베이스에서 모든 사용자의 정보가 유출될 수 있습니다.
>
> **WAF 방어**:
> 1. WAF의 Web ACL에 AWS 관리형 규칙 중 **"SQL Database" 규칙 그룹**을 추가합니다.
> 2. 이 규칙 그룹은 HTTP 요청의 본문, 헤더, 쿼리 문자열에서 SQL Injection 패턴을 자동으로 탐지합니다.
> 3. `admin' OR '1'='1` 패턴이 탐지되면 해당 요청을 **차단(Block)**합니다.
> 4. 정상 사용자의 로그인 요청은 영향 없이 통과합니다.
>
> **핵심 포인트**: AWS 관리형 규칙을 사용하면, 보안 전문가가 아니더라도 일반적인 웹 공격을 쉽게 방어할 수 있습니다.

> 🔍 **예시 14.6.2: DDoS 공격과 Shield의 방어**
>
> **상황**: 온라인 쇼핑몰이 블랙프라이데이 세일을 시작했습니다. 경쟁 업체가 봇넷을 이용하여 초당 수만 건의 가짜 요청을 보내 사이트를 마비시키려 합니다.
>
> **Shield Standard의 방어 (무료, 자동)**:
> - 네트워크 계층(Layer 3/4)의 비정상적 트래픽 패턴을 자동 감지합니다.
> - SYN Flood, UDP Reflection 등 일반적인 DDoS 공격을 자동으로 완화합니다.
> - 이 모든 것이 **별도 설정 없이 자동으로** 작동합니다.
>
> **핵심 포인트**: Shield Standard는 모든 AWS 고객에게 무료로 자동 적용되며, 일반적인 DDoS 공격의 대부분을 방어합니다. CCP 시험에서는 "Shield Standard는 무료이고 자동으로 활성화된다"는 점이 자주 출제됩니다.

### 미니 실습 (Hands-on Mini Lab)

> 🧪 **Mini Lab 14.6: AWS WAF Web ACL 살펴보기**
>
> **목표**: AWS WAF 콘솔에서 Web ACL의 구조와 AWS 관리형 규칙을 확인합니다.
> **예상 소요 시간**: 10분
> **비용**: ⚠️ Web ACL 생성 시 월 $5 비용 발생. 이 실습에서는 생성하지 않고 **콘솔 탐색만** 수행합니다.
>
> **Step 1**: AWS 콘솔 상단 검색창에 "WAF"를 입력하고 **WAF & Shield**를 클릭합니다.
>
> **Step 2**: 좌측 메뉴에서 **AWS WAF** 섹션을 확인합니다. **Web ACLs** 메뉴를 클릭합니다. (아직 생성한 Web ACL이 없으므로 빈 목록이 표시됩니다.)
>
> **Step 3**: **Web ACL 생성(Create web ACL)** 버튼을 클릭하여 생성 화면을 탐색합니다 (실제로 생성하지는 않습니다).
> - 이름, 설명, 리소스 유형(CloudFront 또는 Regional) 선택 화면을 확인합니다.
> - **다음(Next)**을 클릭하여 규칙 추가 화면으로 이동합니다.
>
> **Step 4**: **규칙 추가(Add rules)** → **AWS 관리형 규칙 그룹 추가(Add managed rule groups)**를 클릭합니다.
> - **AWS managed rule groups** 목록을 펼쳐봅니다.
> - "Core rule set", "SQL database", "Known bad inputs" 등 다양한 무료 규칙 그룹이 있는 것을 확인합니다.
> - 각 규칙 그룹의 설명을 읽어봅니다.
>
> **Step 5**: 좌측 메뉴에서 **AWS Shield**를 클릭합니다. Shield Standard가 이미 활성화되어 있다는 메시지를 확인합니다.
>
> **확인**: WAF의 Web ACL 구조(규칙 그룹 → 규칙 → 동작)를 이해하고, Shield Standard가 자동 활성화되어 있음을 확인했습니다.
>
> 🧹 **정리**: Web ACL을 실제로 생성하지 않았으므로 삭제할 리소스가 없습니다. 생성 화면에서 **취소(Cancel)**를 클릭하여 빠져나옵니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: WAF와 보안 그룹/네트워크 ACL을 혼동하지 마세요. **보안 그룹/네트워크 ACL**은 IP 주소, 포트 번호, 프로토콜 수준에서 트래픽을 필터링합니다(Layer 3/4). **WAF**는 HTTP/HTTPS 요청의 **내용**(URL, 헤더, 본문 등)을 검사합니다(Layer 7). 두 가지는 서로 다른 수준의 보호를 제공하며, 함께 사용해야 합니다.

> ⚠️ **주의**: Shield Advanced는 월 $3,000으로 매우 비쌉니다. CCP 시험에서 "DDoS 방어를 위한 무료 서비스는?"이라고 물으면, 답은 **Shield Standard**입니다. Shield Advanced를 혼동하지 마세요.

### Section 요약

- **AWS WAF**는 Layer 7(애플리케이션 계층)에서 HTTP/HTTPS 요청을 검사하여 SQL Injection, XSS 등 웹 공격을 방어합니다.
- WAF는 **Web ACL → 규칙 그룹 → 규칙** 구조로 구성되며, AWS 관리형 규칙을 사용하면 간편하게 보호할 수 있습니다.
- **AWS Shield Standard**는 **무료**이며 모든 AWS 고객에게 **자동 활성화**되어, 기본적인 DDoS(Layer 3/4) 방어를 제공합니다.
- **AWS Shield Advanced**는 유료(월 $3,000)이며, Layer 7 DDoS 방어, 24/7 대응 팀, 비용 보호 등을 추가로 제공합니다.
- WAF는 ALB, CloudFront, API Gateway 등에 연결하여 사용합니다.

## 14.7 AWS Firewall Manager와 네트워크 방화벽 — 중앙 집중식 방화벽 관리

> 💡 **한 줄 요약**: AWS Firewall Manager는 여러 계정·리소스에 걸친 방화벽 규칙을 한 곳에서 일괄 관리하는 서비스입니다.

### 직관적 이해 (Intuitive Understanding)

대형 프랜차이즈 기업을 운영한다고 상상해 봅시다. 전국에 100개의 지점이 있습니다. 각 지점마다 보안 시스템(CCTV, 잠금장치, 출입 규칙)을 설치해야 합니다.

**방법 1 — 개별 관리**: 각 지점의 점장이 독립적으로 보안 시스템을 설치하고 관리합니다. 문제는? 어떤 지점은 보안이 강하고, 어떤 지점은 약합니다. 한 지점에서 보안 사고가 나면, 나머지 99개 지점에도 같은 취약점이 있는지 하나하나 확인해야 합니다.

**방법 2 — 중앙 관리 (Firewall Manager)**: 본사의 보안팀이 "모든 지점에 적용할 표준 보안 규칙"을 정합니다. 이 규칙은 기존 지점뿐 아니라, **새로 오픈하는 지점에도 자동으로 적용**됩니다. 보안팀은 한 곳에서 모든 지점의 보안 상태를 모니터링할 수 있습니다.

이것이 바로 AWS Firewall Manager의 역할입니다.

### 핵심 개념 설명 (Core Explanation)

> 📝 **AWS Firewall Manager**: AWS Organizations와 통합하여, 여러 AWS 계정과 리소스에 걸쳐 방화벽 규칙(WAF, Shield Advanced, 보안 그룹, 네트워크 방화벽 등)을 중앙에서 관리하는 서비스입니다.

Firewall Manager의 핵심 기능:

1. **중앙 집중 관리**: 조직의 관리 계정에서 모든 멤버 계정의 방화벽 규칙을 일괄 설정합니다.
2. **자동 적용**: 새 계정이나 새 리소스가 추가되면, 정의된 보안 정책이 **자동으로 적용**됩니다.
3. **규정 준수 모니터링**: 각 계정/리소스가 보안 정책을 준수하고 있는지 모니터링하고, 비준수 항목을 알려줍니다.
4. **다양한 방화벽 지원**: WAF 규칙, Shield Advanced 보호, 보안 그룹 정책, AWS Network Firewall 정책 등을 모두 관리할 수 있습니다.

Firewall Manager가 관리할 수 있는 보안 서비스:

| 서비스 | Firewall Manager에서의 관리 |
|---|---|
| **AWS WAF** | Web ACL을 여러 계정의 ALB, CloudFront에 일괄 적용 |
| **AWS Shield Advanced** | 여러 계정의 리소스에 Shield Advanced 보호를 일괄 활성화 |
| **보안 그룹** | 여러 계정의 EC2, ENI에 공통 보안 그룹 규칙을 적용 |
| **AWS Network Firewall** | VPC에 대한 네트워크 방화벽 정책을 중앙에서 배포 |

> 📝 **AWS Network Firewall**: VPC 수준에서 작동하는 관리형 네트워크 방화벽 서비스입니다. 보안 그룹이나 네트워크 ACL보다 더 정교한 트래픽 검사(도메인 필터링, 침입 방지 시스템 등)를 제공합니다.

> 💬 **참고**: Firewall Manager를 사용하려면 **AWS Organizations**가 활성화되어 있어야 합니다. AWS Organizations는 여러 AWS 계정을 하나의 조직으로 묶어 관리하는 서비스로, Chapter 16에서 자세히 다룹니다.

### 상세 예시

> 🔍 **예시 14.7.1: 다중 계정 환경의 WAF 정책 중앙 관리**
>
> **상황**: 회사에 개발(Dev), 스테이징(Staging), 프로덕션(Prod) 3개의 AWS 계정이 있습니다. 모든 계정의 ALB에 동일한 WAF 규칙(SQL Injection 방어, XSS 방어)을 적용해야 합니다.
>
> **Firewall Manager 없이**:
> - 각 계정에 로그인하여 WAF Web ACL을 개별적으로 생성해야 합니다.
> - 규칙을 변경할 때마다 3개 계정에서 각각 수정해야 합니다.
> - 새 계정이 추가되면 수동으로 설정해야 하며, 빠뜨릴 수 있습니다.
>
> **Firewall Manager 사용 시**:
> 1. 관리 계정에서 "모든 계정의 ALB에 SQL Injection + XSS 방어 규칙 적용"이라는 정책을 한 번 생성합니다.
> 2. 3개 계정 모두에 자동으로 적용됩니다.
> 3. 나중에 4번째 계정(QA)이 추가되면, 이 정책이 **자동으로 적용**됩니다.
> 4. 관리 콘솔에서 모든 계정의 규정 준수 상태를 한눈에 확인할 수 있습니다.
>
> **핵심 포인트**: 계정 수가 많아질수록 Firewall Manager의 가치가 커집니다. 보안 정책의 일관성과 자동 적용이 핵심입니다.

> 🔍 **예시 14.7.2: 일상 비유 — 프랜차이즈의 위생 규정**
>
> **상황**: 프랜차이즈 본사에서 "모든 지점은 매일 영업 전 소독을 실시해야 한다"는 규정을 만듭니다.
>
> - 이 규정은 기존 100개 지점에 즉시 적용됩니다.
> - 신규 지점이 오픈하면, 자동으로 이 규정의 적용 대상이 됩니다.
> - 본사에서는 각 지점의 규정 준수 여부를 대시보드에서 확인할 수 있습니다.
> - 규정을 변경(예: "소독 횟수를 1일 2회로 변경")하면, 모든 지점에 일괄 반영됩니다.
>
> **핵심 포인트**: Firewall Manager = 보안 규칙의 "프랜차이즈 본사"입니다.

### 흔한 실수와 주의사항 (Common Pitfalls)

> ⚠️ **주의**: Firewall Manager 자체는 방화벽이 아닙니다. WAF, Shield, 보안 그룹 등 **기존 방화벽 서비스의 정책을 중앙에서 관리하는 도구**입니다. CCP 시험에서 "중앙 집중식 방화벽 관리"를 묻는 문제에서 WAF나 Shield가 아닌 **Firewall Manager**가 정답입니다.

> ⚠️ **주의**: Firewall Manager는 AWS Organizations가 필수 전제 조건입니다. 단일 계정 환경에서는 필요하지 않습니다.

### Section 요약

- **AWS Firewall Manager**는 여러 AWS 계정에 걸쳐 방화벽 규칙을 **중앙에서 일괄 관리**하는 서비스입니다.
- WAF, Shield Advanced, 보안 그룹, Network Firewall 정책을 모두 관리할 수 있습니다.
- 새 계정이나 새 리소스에 보안 정책이 **자동으로 적용**됩니다.
- **AWS Network Firewall**은 VPC 수준에서 정교한 트래픽 검사를 제공하는 관리형 방화벽 서비스입니다.
- Firewall Manager는 **AWS Organizations**가 활성화된 다중 계정 환경에서 사용합니다.

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 AWS의 네트워크 보안과 데이터 보호에 대해 학습했습니다.

- **14.1 네트워크 보안 계층**: 보안 그룹(Stateful, 인스턴스 수준)과 네트워크 ACL(Stateless, 서브넷 수준)이 심층 방어를 구성합니다.
- **14.2 보안 그룹 심화**: 허용 규칙만 설정 가능하며, 보안 그룹 참조 기능으로 동적 접근 제어가 가능합니다.
- **14.3 네트워크 ACL 심화**: 번호 순서대로 평가, 허용·거부 모두 가능, 양방향 규칙 필요(Stateless).
- **14.4 데이터 암호화 기초**: 전송 중 암호화(SSL/TLS)와 저장 시 암호화를 모두 적용해야 완전한 데이터 보호가 됩니다.
- **14.5 KMS와 CloudHSM**: KMS는 AWS 관리형 키 관리 서비스, CloudHSM은 전용 하드웨어 보안 모듈, ACM은 SSL/TLS 인증서 관리 서비스입니다.
- **14.6 WAF와 Shield**: WAF는 Layer 7 웹 공격 방어, Shield Standard는 무료 DDoS 방어, Shield Advanced는 유료 고급 DDoS 방어입니다.
- **14.7 Firewall Manager**: 여러 계정에 걸쳐 방화벽 규칙을 중앙에서 일괄 관리합니다.

## 핵심 용어 정리 (Glossary)

| 용어 | 정의 | 관련 Section |
|---|---|---|
| 보안 그룹 (Security Group) | 인스턴스(ENI) 수준의 상태 유지(Stateful) 방화벽. 허용 규칙만 설정 가능 | 14.1, 14.2 |
| 네트워크 ACL (Network ACL) | 서브넷 수준의 상태 비유지(Stateless) 방화벽. 허용·거부 모두 가능 | 14.1, 14.3 |
| 상태 유지 (Stateful) | 이전 트래픽 상태를 기억하여, 허용된 요청의 응답을 자동 허용하는 방식 | 14.1 |
| 상태 비유지 (Stateless) | 각 패킷을 독립적으로 평가하여, 양방향 규칙이 모두 필요한 방식 | 14.1 |
| 인바운드 (Inbound) | 외부에서 내부로 들어오는 트래픽 | 14.1 |
| 아웃바운드 (Outbound) | 내부에서 외부로 나가는 트래픽 | 14.1 |
| 보안 그룹 참조 | 보안 그룹의 소스/대상에 다른 보안 그룹 ID를 지정하는 기능 | 14.2 |
| 임시 포트 (Ephemeral Port) | 클라이언트가 응답 수신을 위해 사용하는 임시 포트 (1024-65535) | 14.3 |
| 전송 중 암호화 (Encryption in Transit) | 네트워크를 통해 이동하는 데이터를 암호화하는 것 (SSL/TLS) | 14.4 |
| 저장 시 암호화 (Encryption at Rest) | 스토리지에 저장된 데이터를 암호화하는 것 | 14.4 |
| SSL/TLS | 네트워크 통신을 암호화하는 프로토콜 | 14.4 |
| 서버 측 암호화 (SSE) | AWS가 데이터를 저장 시 자동으로 암호화·복호화하는 방식 | 14.4 |
| AWS KMS | 암호화 키를 중앙에서 관리하는 완전 관리형 서비스 | 14.5 |
| 고객 관리형 키 (CMK) | 사용자가 직접 생성·관리하는 KMS 키 | 14.5 |
| AWS CloudHSM | 전용 하드웨어 보안 모듈에서 키를 관리하는 서비스 | 14.5 |
| AWS ACM | SSL/TLS 인증서를 무료로 발급·관리하는 서비스 | 14.5 |
| DDoS | 다수의 컴퓨터에서 대량 트래픽을 보내 서비스를 마비시키는 공격 | 14.6 |
| AWS WAF | HTTP/HTTPS 요청을 검사하여 웹 공격을 방어하는 방화벽 | 14.6 |

**1. 보안 그룹은 상태 유지(Stateful) 방화벽이다. (O/X)**

<details>
<summary>정답 확인</summary>

**O** — 보안 그룹은 Stateful입니다. 허용된 인바운드 요청에 대한 응답(아웃바운드)은 아웃바운드 규칙에 관계없이 자동으로 허용됩니다.
</details>

**2. 네트워크 ACL에서 특정 IP를 차단하려면, 차단 규칙을 허용 규칙보다 높은 번호에 배치해야 한다. (O/X)**

<details>
<summary>정답 확인</summary>

**X** — 네트워크 ACL은 낮은 번호부터 평가합니다. 차단 규칙을 허용 규칙보다 **낮은 번호(높은 우선순위)**에 배치해야 합니다. 높은 번호에 배치하면 허용 규칙에 먼저 일치하여 차단이 적용되지 않습니다.
</details>

**3. AWS Shield Standard는 모든 AWS 고객에게 무료로 자동 활성화된다. (O/X)**

<details>
<summary>정답 확인</summary>

**O** — Shield Standard는 무료이며, 모든 AWS 고객에게 자동으로 활성화됩니다. 별도 설정이 필요 없습니다.
</details>

**4. "전송 중 암호화"와 "저장 시 암호화"의 차이를 한 문장으로 설명하세요.**

<details>
<summary>정답 확인</summary>

</details>

**5. AWS WAF가 방어할 수 있는 공격 유형 2가지를 말하세요.**

<details>

</details>

</details>

>
> **Step 3**: **인바운드 규칙** 탭을 확인합니다. 규칙 #100이 모든 트래픽을 ALLOW하는 것을 확인합니다.
>
> **Step 4**: **아웃바운드 규칙** 탭을 확인합니다. 역시 규칙 #100이 모든 트래픽을 ALLOW하는 것을 확인합니다.
>
> **Step 5**: **서브넷 연결** 탭에서 기본 VPC의 모든 서브넷이 이 ACL에 연결되어 있음을 확인합니다.
>

>
> **✅ 체크포인트**: 기본 네트워크 ACL의 인바운드·아웃바운드 규칙 구조를 확인했습니다.

> **Phase 3: KMS 키 생성 및 S3 버킷 암호화** (활용 개념: Section 14.5)
>
> **Step 1**: 상단 검색창에 "KMS"를 입력하고 **Key Management Service**로 이동합니다.
>
> **Step 2**: **고객 관리형 키** → **키 생성**을 클릭합니다.
> - 키 유형: 대칭(Symmetric)
> - 키 사용: 암호화 및 복호화
> - 별칭: `my-s3-encryption-key`
> - 설명: `Key for S3 bucket encryption lab`
> - 키 관리자·사용자: 현재 IAM 사용자 선택
> - **완료** 클릭
>
> **Step 3**: 상단 검색창에 "S3"를 입력하고 **S3** 서비스로 이동합니다.
>
> **Step 4**: **버킷 만들기(Create bucket)**를 클릭합니다.
> - **버킷 이름**: `my-encrypted-bucket-[고유한 숫자]` (예: `my-encrypted-bucket-20260408`)
> - **리전**: ap-northeast-2 (서울)
> - 나머지 설정은 기본값 유지
>
> **Step 5**: 기본 암호화 섹션에서:
> - **암호화 유형**: SSE-KMS 선택
> - **KMS 키**: **AWS KMS 키 중에서 선택**을 클릭하고, 방금 만든 `my-s3-encryption-key`를 선택합니다.
> - **버킷 키**: 활성화 (비용 절감)
>
> **Step 6**: **버킷 만들기** 버튼을 클릭합니다.
>
> **Step 7**: 생성된 버킷을 클릭하고, **속성(Properties)** 탭에서 기본 암호화가 SSE-KMS로 설정된 것을 확인합니다.
>
> **✅ 체크포인트**: S3 버킷의 속성에서 기본 암호화 유형이 "SSE-KMS"이고, 사용 키가 `my-s3-encryption-key`로 표시됩니다.

### 트러블슈팅 (Troubleshooting)

> **Q: S3 버킷 암호화 설정에서 KMS 키가 목록에 보이지 않습니다.**
> A: KMS 키와 S3 버킷이 **같은 리전**에 있는지 확인하세요. KMS 키는 리전에 한정되므로, 서울 리전에서 만든 키는 서울 리전의 S3 버킷에서만 사용할 수 있습니다.

> A: 다른 보안 그룹의 규칙에서 이 보안 그룹을 소스/대상으로 참조하고 있을 수 있습니다. 참조하는 규칙을 먼저 삭제한 후 보안 그룹을 삭제하세요.

> A: 아웃바운드 규칙에서 **임시 포트(1024-65535)**를 허용했는지 확인하세요. 네트워크 ACL은 Stateless이므로 양방향 규칙이 모두 필요합니다.

### D1. Practice (연습 문제) — 기본 개념 확인

> **Practice 14.1**: 보안 그룹(Security Group)에 대한 설명으로 **올바른** 것은?
>
> A. 서브넷 수준에서 작동한다
> B. 허용(Allow) 규칙과 거부(Deny) 규칙을 모두 설정할 수 있다
> C. 상태 유지(Stateful) 방화벽이다
> D. 규칙을 번호 순서대로 평가한다
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: C
> **해설**: 보안 그룹은 **인스턴스(ENI) 수준**(A 오류)에서 작동하며, **허용 규칙만** 설정할 수 있습니다(B 오류). **상태 유지(Stateful)** 방화벽이므로 허용된 인바운드 요청의 응답은 자동 허용됩니다(C 정답). 번호 순서대로 평가하는 것은 **네트워크 ACL**입니다(D 오류).
> </details>

> **Practice 14.2**: 네트워크 ACL에서 특정 IP(10.0.1.50)를 차단하면서 나머지 HTTP 트래픽은 허용하려고 합니다. 올바른 규칙 순서는?
>
> A. #100 ALLOW HTTP 0.0.0.0/0 → #200 DENY HTTP 10.0.1.50/32
> B. #100 DENY HTTP 10.0.1.50/32 → #200 ALLOW HTTP 0.0.0.0/0
> C. 순서는 상관없다
> D. 네트워크 ACL에서는 거부 규칙을 설정할 수 없다
>
> <details>
> <summary>정답 및 해설</summary>
>
> **정답**: B
> **해설**: 네트워크 ACL은 **낮은 번호부터 순서대로** 평가합니다. A처럼 허용을 먼저 두면, 10.0.1.50도 #100에서 이미 허용되므로 #200 거부가 적용되지 않습니다. B처럼 **거부를 먼저(낮은 번호)** 두어야 해당 IP가 차단됩니다. 네트워크 ACL은 거부 규칙을 설정할 수 있으므로 D도 오류입니다.
> </details>

> **Practice 14.3**: 전송 중 암호화(Encryption in Transit)에 해당하는 것은?
>
> A. S3 버킷에 SSE-S3를 적용하여 데이터를 저장한다
> B. EBS 볼륨 암호화를 활성화한다
> C. 웹사이트에 HTTPS(TLS)를 적용하여 통신을 암호화한다

> │      │                                │

>
> (a) IP 203.0.113.50에서 HTTP(포트 80) 요청이 오면 허용되는가, 거부되는가? 그 이유는?
> (b) IP 203.0.113.50에서 HTTPS(포트 443) 요청이 오면 허용되는가, 거부되는가? 그 이유는?
> (c) 이 규칙에 어떤 문제가 있는가? 의도가 "203.0.113.0/24에서 오는 모든 트래픽 차단"이었다면, 어떻게 수정해야 하는가?
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 네트워크 ACL은 낮은 번호부터 순서대로 평가하며, 첫 번째 일치 규칙에서 즉시 결정됩니다.
>
> **풀이 전략**:
>
> **(a) HTTP(포트 80) from 203.0.113.50**:
> - 규칙 #100: HTTP(80) from 0.0.0.0/0 → ALLOW — **일치!** 203.0.113.50은 0.0.0.0/0에 포함됩니다.
> - 결과: **허용됩니다.** 규칙 #100에서 이미 일치했으므로, #200의 DENY는 평가되지 않습니다.
>
> **(b) HTTPS(포트 443) from 203.0.113.50**:
> - 규칙 #100: HTTP(80) → 포트 불일치, 건너뜀
> - 규칙 #150: SSH(22) from 10.0.0.0/8 → 포트와 소스 모두 불일치, 건너뜀
> - 규칙 #200: 모든 트래픽 from 203.0.113.0/24 → DENY — **일치!** 203.0.113.50은 203.0.113.0/24에 포함됩니다.
> - 결과: **거부됩니다.** HTTPS(443)는 규칙 #200에서 먼저 거부됩니다.
>
> **(c) 문제점과 수정**:
> - **문제**: 203.0.113.0/24에서의 HTTP(80) 트래픽은 규칙 #100에서 이미 허용되므로, 규칙 #200의 DENY가 적용되지 않습니다. 즉, "모든 트래픽 차단"이라는 의도가 HTTP에 대해서는 달성되지 않습니다.
> - **수정**: 203.0.113.0/24 차단 규칙을 **모든 허용 규칙보다 낮은 번호**에 배치해야 합니다:
>
> | 규칙 # | 유형 | 포트 | 소스 | 허용/거부 |
> |---|---|---|---|---|
> | **50** | **모든 트래픽** | **전체** | **203.0.113.0/24** | **DENY** |
> | 100 | HTTP | 80 | 0.0.0.0/0 | ALLOW |
> | 150 | SSH | 22 | 10.0.0.0/8 | ALLOW |
> | 300 | HTTPS | 443 | 0.0.0.0/0 | ALLOW |
> | * | 모든 트래픽 | 전체 | 0.0.0.0/0 | DENY |
>
> **확장 생각**: 이 문제는 네트워크 ACL의 "순서 기반 평가"가 얼마나 중요한지를 보여줍니다. 규칙 설계 시 "거부할 것을 먼저 거부하고, 나머지를 허용"하는 패턴이 일반적인 모범 사례입니다.
> </details>

> **Problem 14.3**: 당신의 회사는 현재 단일 AWS 계정을 사용하고 있지만, 성장에 따라 개발(Dev), 스테이징(Staging), 프로덕션(Prod) 3개 계정으로 확장할 계획입니다. 현재는 각 서비스마다 보안 그룹과 WAF 규칙을 개별적으로 관리하고 있습니다.
>
> (a) 다중 계정 환경으로 전환했을 때, 현재 방식(개별 관리)의 보안 위험은 무엇인가?
> (b) 이 문제를 해결하기 위해 어떤 AWS 서비스를 사용해야 하는가?
> (c) 새 계정(QA)이 추가될 때, 보안 정책이 자동으로 적용되도록 하려면 어떻게 해야 하는가?
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 계정 수가 증가하면 개별 관리의 복잡성과 위험이 기하급수적으로 커지며, 중앙 집중 관리가 필수가 됩니다.
>
> **풀이 전략**:
>
> **(a) 개별 관리의 보안 위험**:
> - **정책 불일치**: 각 계정의 관리자가 독립적으로 규칙을 설정하므로, 프로덕션에는 WAF가 있지만 스테이징에는 없을 수 있습니다.
> - **설정 누락**: 새 리소스(ALB, EC2 등)를 생성할 때 보안 규칙 적용을 잊을 수 있습니다.
> - **변경 관리 어려움**: 보안 규칙을 업데이트할 때 3개 계정에 모두 적용해야 하며, 하나라도 빠뜨리면 보안 격차가 생깁니다.
> - **가시성 부족**: 전체 계정의 보안 상태를 한눈에 파악할 수 없습니다.
>
> **(b) 해결 서비스**:
> - **AWS Firewall Manager**: WAF 규칙, Shield 보호, 보안 그룹 정책을 모든 계정에 중앙에서 일괄 관리합니다.
> - **AWS Organizations**: Firewall Manager의 전제 조건으로, 3개 계정을 하나의 조직으로 묶습니다.
>
> **(c) 신규 계정 자동 적용**:
> - Firewall Manager에서 정의한 보안 정책은 **자동 적용 범위(Scope)**를 설정할 수 있습니다.
> - "이 조직의 모든 계정"으로 범위를 설정하면, 새 QA 계정이 조직에 추가되는 즉시 보안 정책이 자동으로 적용됩니다.
> - Firewall Manager는 정책 미준수 항목을 탐지하고, 자동으로 규정 준수 상태로 교정(Auto-remediation)할 수도 있습니다.
>
> **확장 생각**: 조직 규모가 커지면 보안의 "일관성"과 "자동화"가 점점 더 중요해집니다. 개별 관리에서 중앙 관리로의 전환은 보안 성숙도의 핵심 지표입니다. 이와 관련된 거버넌스 도구(AWS Config, CloudTrail 등)는 Chapter 16에서 자세히 다룹니다.
> </details>
